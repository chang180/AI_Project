1. 60 秒搞懂它在幹嘛

打開 IG、Threads、臉書，最上面那條一直往下刷、把你追蹤的人最新動態混在一起的牆——那就是 News Feed（動態牆）。 你追蹤了 200 個人，系統要在你一打開 App 的瞬間，把這 200 人最近發的東西，依時間排好端到你面前。

聽起來簡單？難就難在：你不是只有 200 人，是幾億人互相追蹤；而且大家刷的次數遠遠多過發文的次數（你一天發 1 篇，但刷了 50 次）。

🔑 一句話 News Feed = 把「你追蹤的人」的最新貼文，又快又新地合成一條牆。 全題的拉鋸就一個問題：這條牆要「事先幫你拼好」，還是「你來看時才現拼」？

2. 報紙：印好挨家投遞，還是你來報攤現買？

這題最核心的取捨，用一個你完全懂的生活經驗就能秒懂——報紙怎麼到你手上。有兩種做法：

📰 做法 A：印好挨家投遞（＝「推」） 報社半夜先把報紙印好、塞進每一戶的信箱。早上你起床，伸手到信箱馬上就拿到，不用等。 缺點是：報社半夜很忙，要印幾百萬份、跑幾百萬個信箱。
🏪 做法 B：你來報攤現買（＝「拉」） 報社不主動送，你想看的時候自己走去報攤，店員當場幫你把今天的內容湊一份給你。 報社很輕鬆（不用挨家送），但你每次都要等店員現湊，人多就排隊。

News Feed 一模一樣：

  • 推（fan-out on write）＝「印好投遞」：有人一發文，系統就立刻把這篇塞進他每個粉絲的信箱（feed）。你打開 App 時，牆早就拼好了，讀超快。代價：發文那一刻系統很忙（要塞進每個粉絲的信箱）。
  • 拉（fan-out on read）＝「報攤現買」：發文時什麼都不做，等你來看 feed，系統才現場去問「你追蹤了誰」、把他們的新文湊一份。發文超輕鬆，代價：每次讀都要現湊，讀很慢。

💭 停一下：一般人發一篇文，到底有幾個信箱要塞？這個數字大不大，會不會出事？

看答案 一般人大概 200 個粉絲，所以發 1 篇 → 塞 200 個信箱，還好。但如果是有 5000 萬粉絲的名人呢？發 1 篇 → 要塞 5000 萬個信箱！系統會被一篇貼文卡死。這就是下一段的主角——「名人問題」。

3. 把系統拆成 3 個關卡

別被「系統」嚇到。整件事其實就是闖 3 關，一關比一關有趣：

1推：發文時，先幫每個粉絲把貼文塞進信箱

預設大家都用「推」（印好投遞）。你一發文，系統就對你的每一個粉絲，把這篇貼文的編號放進他的信箱（feed 快取）。

📮 比喻就像報社半夜挨家投遞。好處是：粉絲早上打開信箱（讀 feed）東西已經在裡面了，零等待。這就是為什麼預設選推——因為讀的次數遠多於發文，把麻煩搬到「發文時」處理最划算。

2名人問題：千萬粉的人發文，信箱塞不完

推很美好，直到出現有 5000 萬粉絲的名人。他發一篇文，系統要瞬間塞 5000 萬個信箱——這一篇就能把系統拖垮。

🌟 比喻：報社遇到「全國都訂的頭版」 平常投遞還好，但如果有一份內容是全國 5000 萬戶都要，報社半夜根本印不完、送不完。 這時聰明的做法是：這份特別的內容「不主動送」，改放在報攤，誰想看自己來拿。

所以名人改用「拉」：名人發文時不塞任何信箱，只放在他自己的貼文庫。等你來看 feed，系統才現場去名人那裡拉他的最新貼文。名人人數少、又有快取攤平，這樣就不會被寫爆。

3Hybrid：一般人用推、名人用拉，讀時合在一起

業界真正的做法叫 Hybrid（混合），其實就是把前兩關合起來：

  • 你追蹤的一般人 → 他們的貼文早就推進你信箱了（直接拿，超快）。
  • 你追蹤的名人 → 讀的時候現場去拉他們的最新貼文。
  • 系統把這兩堆合併、依時間排好，就是你看到的那條牆。
🥗 比喻：自助餐 + 現點常吃的菜事先擺好（推）你直接夾；招牌大菜現點現做（拉）。最後同一個盤子上桌——你根本分不出哪道是先擺的、哪道是現做的。

這個「一篇貼文 → 分發給一群粉絲」的動作，工程上就叫 fan-out（扇出）。發文時就分發叫「on write（推）」，讀時才分發叫「on read（拉）」。

✅ 三關一句話 ① 一般人發文時先推進每個粉絲信箱（讀超快）→ ② 但名人會把信箱塞爆，改成讀時才拉 → ③ Hybrid：推來的＋拉名人的，讀時合併排序。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「讀太慢」 你打開 App，牆要立刻出現（慢半秒你就煩了）。所以預設用推——事先把牆拼好放信箱，讀的時候直接端出來，不用現算。讀快壓倒一切，因為讀的次數遠多於寫。
😱 怕「名人把寫端塞爆」 純推遇到千萬粉名人會「一篇發文 → 千萬次塞信箱」直接爆掉。所以名人改用拉，不主動塞。這就是純推唯一、但致命的例外。
😱 怕「白做工」 很多粉絲是殭屍粉、長期不上線，把貼文推進他們信箱根本沒人看，純粹浪費。所以對不活躍的人也改用拉（他哪天真的登入，再現場湊給他）。能省的寫入就省。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每天多少人在用2 億人規模很大，單機絕對不夠
發文（寫）有多頻繁每秒約 230 篇寫其實不多
刷牆（讀）有多頻繁每秒 2~7 萬次讀比寫多上百倍 → 要把讀做到極快
一般人發 1 篇要塞幾個信箱約 200 個（平均粉絲數）還可以接受，所以一般人用推
名人發 1 篇要塞幾個信箱最多 5000 萬個會爆！名人一定要改拉
⚖️ 關鍵洞察：讀寫天平嚴重偏一邊 讀比寫多上百倍。所以聰明的做法是把麻煩搬到「比較少發生」的寫端（發文時先把牆拼好）， 讓「超常發生」的讀端輕鬆到只要伸手拿信箱。但名人會讓寫端的成本爆炸——這就是整題的張力來源。

💭 停一下：既然推「讀很快」這麼好，為什麼不乾脆全部都用推，連名人也推？

看答案 因為名人粉絲太多。一個 5000 萬粉的名人發 1 篇，純推就要做 5000 萬次寫入，瞬間把系統塞爆、延遲飆高，還可能拖累其他人的發文。所以名人必須例外改拉——這正是 Hybrid 存在的理由：用粉絲數門檻把人分成「一般人（推）」和「名人（拉）」。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住三張「點餐單」：

① 追蹤 / 取消追蹤（改變「誰追誰」的關係）
   POST   /follow   給我：{ 誰(follower), 追誰(followee) }
   DELETE /follow   取消追蹤

這只是記下「人和人的追蹤關係」，是後面推、拉都要查的基礎名單。

② 發文（會觸發 fan-out 分發）
   POST /posts   給我：{ 作者, 內文 }
   我回：201 { 貼文編號, 這篇用 "推" 還是 "拉" }

系統收到後會看作者是不是名人，決定要「立刻推進每個粉絲信箱」還是「先放著、等人來拉」。真實系統會非同步處理推這件事——發文 API 先回「收到」，背景再慢慢塞信箱，不讓你卡著等。

③ 讀牆（最高頻、最該快的一張單）
   GET /feed?user_id=我&limit=20&cursor=上次刷到哪
   我回：{ 一串貼文, 每筆標了 來源:"推來的" 或 "即時拉的", 下一頁游標 }

正式系統不會回傳「來源」這欄；這裡刻意露出來，方便你眼睛看到「這筆是事先推好的、那筆是現場拉名人的」——這正是本題 demo 要呈現的重點。cursor 是「書籤」，記住你上次下拉到哪，再往下刷就從那裡接。

9. 系統要記住哪些事？（資料模型）

📒 比喻：四本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題只要四本：

👤使用者簿（誰是名人）

每個人一列：使用者ID、是不是名人。粉絲數超過門檻就自動標成名人——這欄直接決定他發文要用推還是拉。

🔗追蹤簿（誰追誰）

記 (追蹤者, 被追蹤者)。它要從兩個方向查：拉的時候問「我追了誰」、推的時候問「我有哪些粉絲」。所以兩個方向都要建索引，才查得快。

📝貼文庫（內文的唯一真相）

每篇貼文一列：貼文編號、作者、內文、發文時間。內文只存在這一本。

📬信箱簿（每個人的 feed 快取）

使用者 → [貼文編號, 貼文編號, ...]，只保留最新 N 筆。這是「推」寫入的地方，正式系統用 Redis 放，讀超快。

為什麼信箱只存「編號」不存內文？ 因為一篇名人貼文會出現在千萬個人的牆上。如果每個信箱都存一份內文，同一段話被複製千萬份，超浪費；而且改一個錯字要改千萬處。 只存編號，要顯示時再拿編號去貼文庫撈內文（叫 hydrate / 回填）——內文永遠只有一份、改一次就好。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
名人發文 → 要塞千萬個信箱名人改拉，根本不塞；他的貼文額外快取，讓所有讀者共用
發文 API 卡著等「塞完所有信箱」發文只寫貼文庫，塞信箱的工作丟給背景工人非同步做（晚幾秒沒關係）
重度追蹤者：追了上千人，純拉每次都要現湊一般人的部分早就推好了，只需現場拉「少數幾個名人」
信箱存全部歷史貼文 → 爆量只留最新 N 筆，更舊的退化成拉模型回補
把貼文推給長期不上線的殭屍粉對不活躍的人改拉，他登入時才現湊，省下大量白做工
同一篇名人貼文被超多牆引用內文單獨快取一份，每個信箱只存編號，再批次回填

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 推 vs 拉 推＝把成本搬到「發文時」、讀超快；拉＝發文超輕、但每次讀都要現湊。核心就是把讀寫不對稱往便宜的那一邊壓。因為讀遠多於寫，所以預設選推。
⚖️ 為什麼一定要 Hybrid 純推被名人寫放大擊穿；純拉被重度追蹤者讀放大拖垮。所以依粉絲數門檻分流：一般人推、名人拉，各取所長。
⚖️ 要不要等到「完全即時」 feed 可以最終一致——新貼文晚個幾秒才出現在你牆上，沒人會在意。換來的是更高吞吐與可用性（不用強求同步即時）。
💭 追問：使用者「剛追蹤一個名人」，怎麼立刻看到他以前的舊文？ 拉模型天然就解決了——讀牆時本來就會現場去名人那裡拉最新貼文，所以一追蹤馬上就看得到。若是推模型，反而還要特地回填，麻煩。
💭 追問：往下無限刷（分頁）怎麼做才不會錯亂？ 用 cursor（游標）記住「上次刷到哪一篇」，下一頁從那篇之後接著拿。不要用「第幾筆開始（offset）」，因為牆一直有新文插進來，用編號數會錯位、重複或漏掉。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「推 / 拉 / Hybrid」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8015 -t public
# 然後瀏覽器打開 http://localhost:8015
  1. 先建立追蹤關係，記得放一個高粉絲的名人進去。
  2. 讓幾個人發文（包含那個名人）。
  3. 挑一個使用者讀他的 feed，觀察：哪些貼文是「推來的」、哪些是「即時拉的」、合併排序後長怎樣。
玩的時候想一下把某個人的粉絲調到超過名人門檻，看他發文的策略是不是從「推」變成「拉」？這就是 Hybrid 的分流在運作。再看讀 feed 時，名人的貼文是不是讀的當下才被拉進來、和事先推好的混在一起依時間排好——那就是 k 路合併。

13. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「報紙」的比喻，解釋「推」和「拉」差在哪？ 推＝報社半夜印好挨家投遞，你早上開信箱馬上拿到（讀快，但發報時很忙）。拉＝報社不送，你想看自己去報攤，店員當場幫你湊一份（發報輕鬆，但你每次要等現湊）。News Feed 的推＝發文時就塞進每個粉絲信箱；拉＝讀時才現場聚合。
Q2：為什麼預設要用「推」而不是「拉」？ 因為讀的次數遠多於發文（常達 100:1）。用推把麻煩搬到「比較少發生的發文時」，讓「超常發生的讀」快到只要伸手拿信箱。把讀寫不對稱往便宜那邊壓。
Q3：「名人問題」是什麼？為什麼純推會爆？ 名人有千萬粉。純推時，他發 1 篇就要對每個粉絲塞信箱＝千萬次寫入，瞬間把寫端塞爆、延遲飆高。所以名人必須例外改用拉。
Q4：Hybrid 是怎麼同時用推和拉的？ 依粉絲數門檻分流：一般人用推（貼文早就在你信箱）、名人用拉（讀時才現場去拉）。讀牆時把「推來的」和「拉名人的」兩堆合併、依時間排序，端給你。
Q5（工程）：為什麼信箱（feed 快取）只存「貼文編號」，不存內文？ 因為一篇貼文會出現在千萬份牆上。存內文＝同一段話被複製千萬份，浪費儲存、改字要改千萬處。只存編號，內文統一放貼文庫一份，顯示時再用編號回填（hydrate）。
Q6（工程）：為什麼發文 API 要「先回收到、背景再慢慢塞信箱」？ 因為塞信箱（fan-out）可能要對很多粉絲做，同步做會拖慢發文。改成發文只寫貼文庫＋丟個事件，背景工人非同步把貼文推進各信箱。晚幾秒出現可接受（最終一致）。
Q7（工程）：為什麼分頁要用 cursor（游標）而不是 offset（第幾筆）？ 因為牆一直有新貼文插進來。用「第幾筆開始」會因為位置變動而錯位、重複或漏看。用 cursor 記住「上次那篇的編號／時間」，從它之後接著拿，才穩定。

14. 30 秒回顧

News Feed = 把你追蹤的人最新貼文，又快又新地合成一條牆。

核心比喻：報紙是「印好挨家投遞（推）」還是「你來報攤現買（拉）」。

3 個關卡：① 一般人發文時先推進粉絲信箱（讀快）→ ② 名人會把信箱塞爆，改成讀時才拉 → ③ Hybrid：推來的＋拉名人的，讀時合併排序。

3 個怕：怕讀慢（預設用推）、怕名人塞爆寫端（名人改拉）、怕白做工（不活躍的人也改拉）。

工程細節一句話：讀比寫多上百倍 → 用「點餐單(API)」溝通、發文先回收到再背景推 → 四本筆記本（使用者/追蹤/貼文庫/信箱）＋信箱只存編號 → 找出名人這個塞車環節，用 Hybrid 拓寬。
