⑮ News Feed 動態牆：推、拉與 Hybrid Fan-out

F 社交/Feed 建議 50 分鐘 Design a News Feed / Timeline

一句話定位 本質是「追蹤關係上的扇出（fan-out）分發」問題。產生內容很容易；面試真正考的是 發文時把貼文寫進每個追蹤者的 feed（推 / fan-out on write） 還是 讀 feed 時即時聚合所有追蹤對象（拉 / fan-out on read），以及名人（百萬粉絲）把推模型推爆後如何用 hybrid 解決。

1. 需求釐清

功能性需求

  • 使用者可追蹤（follow）其他使用者；發文（post）。
  • 讀取自己的 News Feed：所有追蹤對象的貼文，依時間（或排序分數）合併。
  • 分頁載入（無限下拉），通常只取最新 N 筆。
  • （進階）排序：時間序 → 演算法排序（互動分數、相關性）。

非功能性需求

  • 讀延遲低：打開 App 就要看到 feed，p99 < 200ms → feed 必須預先算好或可快速聚合。
  • 讀遠多於寫：刷 feed 的次數遠大於發文（讀寫比常達 100:1）。
  • 高可用：feed 可接受最終一致（晚幾秒看到新貼文沒關係），不需強一致。
  • 可擴展：使用者數與追蹤關係皆可達數億；單一名人粉絲可破千萬。
面試開場必問 「feed 是時間序還是演算法排序？」「要不要支援名人（super-user）？」這兩點決定整題走向。 時間序純合併最單純；一旦有百萬粉絲名人，推模型（fan-out on write）就會被「一次發文 → 千萬次寫入」推爆，整題核心就轉向 hybrid。

2. 規模估算（讀寫放大）

指標假設推算
日活（DAU）2 億—
發文（寫）每人每日 0.1 篇2000 萬/日 ≈ 230 寫 QPS
讀 feed（讀）每人每日刷 10 次20 億/日 ≈ 23k 讀 QPS（峰值 ×3 ≈ 70k）
平均追蹤者數每人 ~200 粉絲推模型：1 次發文 → ~200 次 feed 寫入（寫放大 200×）
名人粉絲數頂級帳號 5000 萬推模型：1 次發文 → 5000 萬次寫入（會推爆，必須改拉）
feed 快取容量每人存最新 ~800 筆貼文 id2億 × 800 × 8B ≈ 1.3 TB（純 id，放 Redis 列表）
關鍵洞察讀寫極不對稱 → 預設用推模型把成本搬到寫端（讀只需 O(1) 取自己 feed 快取）；但寫放大會隨追蹤者數爆炸，名人是這個策略的致命例外。整題的張力就在這條「寫放大 vs 讀放大」的天平上。

3. API 設計

# 追蹤 / 取消追蹤（寫，改變社交圖）
POST   /api/v1/follow      Body: { "follower_id": 1, "followee_id": 9 }
DELETE /api/v1/follow      Body: { "follower_id": 1, "followee_id": 9 }

# 發文（寫，觸發 fan-out）
POST   /api/v1/posts       Body: { "author_id": 9, "text": "..." }
→ 201 { "post_id": "p_123", "fanout": "push"|"pull" }   # 依作者是否名人決定

# 讀 News Feed（讀，超高頻，關鍵路徑）
GET    /api/v1/feed?user_id=1&limit=20&cursor=...
→ 200 { "items": [ {post_id, author_id, text, ts, source:"push"|"pull"} ],
        "next_cursor": "..." }
注意source 欄位在正式系統不會回傳，這裡刻意暴露「這筆是推來的還是即時拉的」，正是本題 demo 要呈現的核心。

4. 資料模型

users
  user_id      BIGINT PK
  is_celebrity BOOL            -- 粉絲數超過門檻自動標記（影響 fan-out 策略）

follows  (社交圖，雙向各建索引)
  follower_id  BIGINT          -- 我追蹤誰（拉模型讀此）
  followee_id  BIGINT          -- 我的粉絲是誰（推模型寫此）
  PRIMARY KEY (follower_id, followee_id)

posts  (貼文庫，真相來源)
  post_id      BIGINT PK
  author_id    BIGINT
  text         TEXT
  created_at   TIMESTAMP

feed_cache  (每使用者的預算 feed，推模型寫入處 → Redis 列表)
  user_id → [ post_id, post_id, ... ]   -- 只存 id，依 ts/score 排序，截斷保留最新 N 筆
為何 feed 只存 id貼文內容是單一真相來源（posts 庫）；feed 快取只存 post_id 列表，讀時再批次回填內容（hydrate）。這樣同一貼文不會在千萬份 feed 裡複製內文，省儲存、且改貼文只改一處。

5. 高階架構

🖼️ 開啟 Excalidraw 架構圖（diagram.excalidraw） 
 發文(寫) ─▶ Post Service ─▶ 貼文庫(posts)
                 │
                 ▼  查作者是否名人？
            ┌────────────┐
            │ Fan-out     │  一般作者：推 ─▶ 對每個粉絲寫 feed 快取
            │ Worker      │  名人作者：不推（標記，留待讀時拉）
            └────────────┘
                                          ┌─────────────┐
 讀 feed(讀) ─▶ Feed Service ─┬─(推來的)─▶│ Feed 快取    │ 取自己的預算 feed
                              │           └─────────────┘
                              └─(即時拉)─▶ 社交圖找出我追的名人 ─▶ 貼文庫取其最新貼文
                                          ─▶ 兩者合併、依時間排序、回傳
  1. Post Service：寫入貼文庫，發 fan-out 事件（真實系統走 Kafka 給 worker 非同步處理）。
  2. Fan-out Worker：一般作者 → 對每個粉絲的 feed 快取 push 該 post_id；名人 → 跳過（避免寫爆）。
  3. Feed Service：讀時把「已推進快取的內容」與「即時從名人貼文拉來的內容」合併排序後回傳。

6. 核心元件深入（推 / 拉 / hybrid）

Fan-out on write（推模型 / Push）

  • 發文當下，對作者的每個粉絲把 post_id 寫進其 feed 快取。
  • 讀超快：讀 feed = O(1) 取自己那份預算好的列表，不必聚合。
  • 缺點：寫放大。粉絲越多寫越多次；名人發一篇 → 千萬次寫入、延遲與成本爆炸；殭屍粉（不活躍者）的 feed 也白寫。

Fan-out on read（拉模型 / Pull）

  • 不預先寫；讀 feed 時才查「我追蹤了誰」，去各人貼文取最新，當場合併排序。
  • 寫超輕：發文只寫貼文庫一次，無寫放大。
  • 缺點：讀放大。追蹤上千人時，每次讀都要聚合上千個來源，讀延遲高、難快取。

Hybrid（混合，業界實際做法）

關鍵洞察 依作者類型分流：一般使用者用推（讀快，寫放大可控）；名人用拉（避免一次發文寫爆千萬份 feed）。 讀 feed 時＝已推進快取的內容 ＋ 即時拉取「我追的名人」的最新貼文，兩者合併依時間排序。名人數量少、其貼文可額外快取，拉的成本被攤平。

合併與排序

  • 每個來源各自已依時間有序 → 用 k 路歸併（k-way merge）取全域最新 N 筆，類似合併多個有序列表。
  • 排序鍵：純時間序最單純；演算法 feed 改用排序分數（時間衰減 × 互動率 × 親密度），但合併框架不變。

讀寫放大分析

策略寫成本（發 1 篇）讀成本（讀 1 次 feed）適合
純推O(粉絲數)　← 名人爆O(1)一般使用者
純拉O(1)O(追蹤數)　← 重度使用者慢名人 / 不活躍讀者
HybridO(非名人粉絲數)O(1 + 我追的名人數)正解

7. 擴展與瓶頸（名人問題）

階段 / 瓶頸問題對策
名人發文（寫熱點）純推 → 一次發文千萬寫入，fan-out 塞爆Hybrid：名人改拉，不推；其貼文額外快取讓讀端共用
Fan-out 延遲同步推會拖慢發文 API發文只寫貼文庫＋丟 Kafka；fan-out worker 非同步背景推，最終一致
讀放大（重度追蹤者）追上千人，純拉每次聚合太慢非名人部分走推（已預算好），只即時拉少量名人
feed 快取容量每人存全部歷史貼文 id 爆量只保留最新 N 筆（截斷）；更舊的退化成拉模型回補
不活躍使用者推給長期不上線者＝白寫對不活躍者改拉（登入時才聚合），省下大量無效寫入
熱點貼文 hydrate同一名人貼文被大量 feed 引用貼文內容單獨快取（內容快取），feed 只存 id 再批次回填

8. 取捨與面試重點

  • 推 vs 拉：推＝成本搬到寫端、讀 O(1)；拉＝寫端輕、讀要聚合。核心是把「讀寫不對稱」往便宜的一側壓。
  • 為何要 hybrid：純推被名人寫放大擊穿，純拉被重度追蹤者讀放大拖垮 → 依作者粉絲數門檻分流，取兩者之長。
  • 一致性：feed 可最終一致（非同步 fan-out、晚幾秒看到新貼文可接受），不需強一致 → 換來高可用與吞吐。
  • feed 存 id 還是存內容：存 id（避免內文複製千萬份、改貼文只改一處），讀時 hydrate；代價是多一次批次回填查詢。
  • 常見追問：
    • 名人門檻怎麼定？→ 以粉絲數（如 > 1 萬）或推一次的成本動態判定，可隨負載調整。
    • 使用者「剛追蹤名人」怎麼立刻看到舊文？→ 拉模型天然解決（讀時即時聚合），推模型則需回填。
    • 如何分頁？→ 用 cursor（以 post_id / ts 為游標），避免 offset 在變動列表上錯位。
    • 演算法排序與時間序差異？→ 合併框架相同，只換排序鍵；排序分數通常離線預算再寫入。

9. PHP Demo 說明

本題 demo 聚焦推 / 拉 / hybrid fan-out 的取捨（百萬級 feed 真正的考點），不是 CRUD。可用 PHP 內建伺服器跑起來：

  • src/SocialGraph.php　追蹤 / 粉絲關係（社交圖），並依粉絲數門檻判定名人。
  • src/FeedStore.php　每使用者的 feed 快取（推模型的寫入處），JSON 檔模擬 Redis 列表。
  • src/FeedService.php　發文 → 對「一般追蹤者」fan-out on write；名人貼文不推、讀時即時拉；讀 feed＝合併推來的＋拉名人的，依時間排序，並標記每筆來源。
  • public/index.php　路由 + 深色測試頁，可視覺化每位作者被判定推 or 拉、每筆 feed 的來源。
# 啟動
cd demo
php -S localhost:8015 -t public
# 瀏覽器開 http://localhost:8015
# 流程：建立追蹤關係（含一個高粉絲名人）→ 數人發文 → 看某使用者讀 feed
#       哪些是「推來的」哪些是「即時拉的」、合併排序結果
誠實聲明 Demo 用 JSON 檔模擬資料庫與 Redis、用同步函式呼叫模擬 Kafka 非同步 fan-out（標註於程式碼）；但推 / 拉判定、fan-out on write 寫入、讀時即時拉名人、k 路合併排序、來源標記等系統邏輯為真實可執行。要上正式環境，把 FeedStore 換成 Redis、把 fan-out 改丟訊息佇列即可，核心邏輯不變。
