1. 60 秒搞懂它在幹嘛

你打開 TikTok，沒追蹤任何人，但一支接一支的影片就是莫名好看，一不小心滑了一小時——這就是 For You（為你推薦）。

神奇的地方是：你沒告訴它你喜歡什麼，它卻越推越準。秘密在於它偷偷觀察你的行為：哪支看完、哪支秒滑、哪支按讚，然後不斷調整下一支推什麼。

🔑 一句話 For You = 一個會察言觀色的 DJ：先放幾首大眾金曲探你口味 → 看你跟著點頭還是皺眉 → 慢慢換成「你會喜歡」的歌，越放越懂你。它不需要你先說喜好，從你的反應就學會了。

2. 為什麼難的不是「播影片」？

很多人第一次看這題會想：「不就是一支一支播影片嗎？」——這想法漏掉了真正難的兩件事。

📚 比喻：在巨大圖書館幫你挑書 想像一座有50 億本書的圖書館，你走進去，館員要馬上遞給你「你會想看」的 10 本。 他不可能把 50 億本都翻一遍再挑（翻到天荒地老）。他得先飛快地縮到一小疊「大概有興趣的」，再從那一小疊裡細挑出最好的 10 本。

第一個難關就是這個：影片多到爆，要飛快地從海量縮到一小撮，再細挑。工程上叫「召回 → 排序」兩段式。

第二個難關更隱形：你沒說過你喜歡什麼。系統只能從你的行為偷學——看完代表喜歡、秒滑代表不愛——然後邊推邊學、越推越準。這個「推 → 看你反應 → 學起來 → 推得更準」的循環，就是第二個難關（叫回饋迴路）。

💭 停一下：為什麼館員不能「把 50 億本都評分一次再挑前 10」？

看答案 因為太慢、太貴。細評分一本書要花時間，50 億本逐一評你會等到睡著。 所以要先用便宜快速的方法（看書架分類）粗篩出幾百本「可能有興趣的」，只對這幾百本做仔細評分。這就是「召回（粗篩）→ 排序（細挑）」。

3. 把系統拆成 4 個關卡

別被「推薦系統」嚇到。整件事其實就是闖 4 關，一關一關來：

1召回：從海量飛快粗篩出一小撮

50 億支影片，先快速縮到幾百個「可能有興趣」的候選。這關要快、要廣，寧可多撈一點（後面還會再篩），不求精準。

🛒 比喻：逛超市先衝幾個區 你進超市要快速買晚餐，不會每個貨架都看。你直奔幾個區：常買的「熟食區」（你的興趣）、大家都在搶的「特價區」（熱門）、還有「跟你買過同款的人也愛逛的區」（協同）。 這幾區的東西先全丟進推車——這就是多路召回。

👉 為什麼要「多路」（好幾區一起逛）？因為只逛一區會有盲點：只看熟食區永遠吃一樣的、只看特價區又不合你口味。多區互補才撈得廣。

2排序：從那一小撮裡細挑出最好的

推車裡有幾百樣，但今晚只煮 10 道。要幫每樣打分，挑出最高分的。

⭐ 比喻：幫每道菜打分 每樣食材打三種分：「你有多愛」（相符你口味）＋「大家有多愛」（熱門）＋「新不新鮮」（新上架）。 三分加起來，挑最高的那 10 樣。「你有多愛」權重最大——畢竟是煮給你吃的。

真實的 TikTok 這一步用的是 AI 模型（預測「你會不會看完」）。但骨架一樣：給候選打分、挑最高。我們的小程式用「三種分加起來」來示範，邏輯位置一模一樣。

3去重 + 探索：別重複，也別困在同溫層

挑好了，發之前還有兩個小心機：

  • 去重：這一輪剛給你看過的，別再推一次（不然你會覺得「怎麼又是這支」）。
  • 探索：每次偷渡一兩支「你沒接觸過的類型」。萬一你愛上了呢？這也讓新影片有機會被看到。
🍣 比喻：吃迴轉壽司 你愛鮭魚，店家當然多給你鮭魚（利用）。但偶爾推一盤你沒吃過的（探索）——說不定你會發現新最愛。 如果只給鮭魚，你很快就膩了，而且其他好料永遠輪不到。

4回饋：看你反應，偷偷學起來

影片推出去後，戲才剛開始。系統盯著你的反應：看完了？按讚了？還是 0.5 秒就滑掉？

👀 比喻：會看臉色的服務生 好的服務生看你把某道菜吃光，下次就多上那類；看你動都沒動，就默默撤掉不再上。 系統也一樣：你看完/按讚的類型，分數加上去，下次召回和排序都更偏這類——越看越準。

這個「推 → 看反應 → 更新喜好 → 推得更準」轉個不停的循環，就是 回饋迴路，是整個 TikTok 的靈魂。

✅ 四關一句話 ① 召回（從海量粗篩一小撮）→ ② 排序（打分細挑最好的）→ ③ 去重 + 探索（別重複、偷渡新口味）→ ④ 回饋（看反應、學起來、下次更準）。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「慢 / 算不動」 50 億支影片，如果每支都仔細打分，打到天亮也打不完，你早就關 App 了。 所以要兩段式：召回先用便宜方法縮到幾百個，排序只對這幾百個細算。還要預載下一頁（先偷偷算好），你捲到底時瞬間就有。只細算必要的，別做白工。
😱 怕「越推越窄（同溫層）」 只推你已知愛看的，你會被困在同溫層，越看越膩；而且新影片永遠沒機會曝光。 所以要探索：每頁偷渡一點新口味，打破迴圈、給新片機會。大多給你愛的，但留一點驚喜。
😱 怕「冷啟動：新人/新片不知道怎麼推」 新使用者還沒留下任何行為，系統不知道他愛什麼；新影片還沒人看過，不知道好不好。 對策：新人先推熱門 + 探索（從前幾支反應快速學）；新片靠新鮮加分 + 探索名額拿到初始曝光。不認識你，先給安全牌再慢慢學。

白話導讀，分成「上傳」「取影片」「看影片」三條路：

  • 📤 上傳：影片上傳 → 轉碼（轉成各種畫質）→ 放 CDN（離你最近的倉庫）等著被播放。
  • 📲 取 For You：① 召回（多路粗篩一小撮）→ ② 排序（打分細挑）→ ③ 去重 + 探索 → 回一頁，順手算好下一頁先備著。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
有多少人在用每天 5 億人取影片這條路超高頻 → 要快、要能水平擴展
有多少影片累積 ~50 億支不可能逐一打分 → 一定要「召回先粗篩、排序再細挑」
取一頁要多快< 0.1 秒來不及細算 → 靠預載下一頁偷偷先算好
每天多少「行為」要消化每支看/滑/讚都算一筆，數千億筆/日不能即時硬算 → 走事件流後台慢慢聚合
💡 關鍵洞察：壓力在「海量挑選」和「海量行為」 難不在「播一支影片」（那是 CDN 的事），而在「從 50 億裡飛快挑出你會愛的 10 支」＋「消化你每天上百次的反應、即時更新喜好」。 所以「召回→排序」（快速挑）和「回饋迴路」（消化反應）才是設計核心。

💭 停一下：每天行為事件多到數千億筆，為什麼不「看一支就立刻把整套喜好重算一遍」？

看答案 因為太貴、會卡。每個動作都觸發一次大重算，伺服器瞬間爆掉。 所以把行為丟上一條輸送帶（事件流），後台慢慢聚合更新喜好。輕量的訊號（剛按的讚）可以近即時反映，重的計算則排隊批次做——這叫即時 + 批次混合。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」：

① 取一頁 For You（最常用）
   GET /api/v1/feed?user_id=1&limit=10
   我回：{ items:[ {影片id, 標籤, 分數, 來源:興趣/熱門/探索} ],
          next_page:[ ... ] }   ← 順便附上「先算好的下一頁」

注意回傳裡有 next_page（預載的下一頁）和每支的 來源標記——正式產品不會給你看來源，這裡故意秀出來讓你看懂「這支是因興趣、因熱門還是探索來的」。

② 看一支影片 → 回饋（量超大）
   POST /api/v1/watch
   給我：{ 哪個人, 哪支影片, 動作:看完/讚/滑掉, 看了幾秒 }
   我回：202（收到了！丟去後台慢慢消化，不卡你）

回 202（收到、稍後處理）而不是等它算完，是因為這條量太大，必須丟事件流非同步處理，不能讓你卡在那等。

③ 上傳影片（低頻但重）
   POST /api/v1/videos  → 觸發轉碼（轉各種畫質）→ 完成後才能被推薦

④ 查我的喜好（除錯/個人化頁）
   GET /api/v1/users/{我}/interests
   我回：{ dance:3.5, music:2.0, ... }   ← 系統偷學到的你的口味

9. 系統要記住哪些事？（資料模型）

📒 比喻：四本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題用四本，各管不同的事：

🎞️影片簿（每支影片的資料）

每支一列：影片id、標籤(分類)、熱度(多少人愛)、上架時間(算新鮮度)。 （影片本體不放這——轉碼後的檔放 CDN，這本只記「資料卡」。）

❤️口味簿（每個人的喜好向量）

每個人一份：{ 標籤 → 分數 }，例如 { dance:3.5, music:2.0 }。 這本是系統偷學來的你的口味，每次你看完/按讚就偷偷加分（回饋迴路改的就是這本）。

✅已看簿（這一輪看過哪些——去重用）

記「這一輪 session 你已經看過的影片id」。取下一頁時把這些剔掉，就不會重複推同一支。

📨行為簿（你做過的每個動作——回饋來源）

記每一筆 誰、看了哪支、動作(看完/讚/滑)、看幾秒、何時。 這本append-only（只加不改）、量超大，走事件流後台聚合，最後拿去更新「口味簿」和影片「熱度」。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
挑選：50 億影片逐一打分兩段式：召回先粗篩到幾百個（用「分類索引/相似搜尋」），排序只細算這幾百個
口味要即時更新，不然推不準行為走事件流聚合，輕訊號近即時、重計算批次（混合）
排序的 AI 模型太重、太慢模型服務多機並行 + 特徵快取，且只對召回後的小候選跑
取一頁來不及預載下一頁 + 提前把每人候選算好 + CDN 先把下一批影片送到邊緣
「已看簿」越記越大用Bloom filter（省記憶體的近似名單）+ session 過期就清掉
熱門影片同時被超多人看CDN 就近送 + 多畫質自適應（沿用 YouTube 那套分發）

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 個人化 vs 多樣性 全推你愛的→當下黏著最高，但越看越窄、新片餓死。留一點探索名額→犧牲一點即時黏著，換長期留存與新片曝光。拿捏這個比例就是核心藝術。
⚖️ 即時學 vs 批次學 看一支馬上變準（即時）體驗最好但很貴；晚上統一重算（批次）便宜但反應慢。實務混合：剛按的讚馬上反映、深度口味分析排隊慢慢做。
⚖️ 看完 vs 按讚，哪個訊號更準？ 按讚要動手指、很多人懶得按；「有沒有看完 / 停留多久」反而更誠實。所以「看完」通常權重比「讚」還高。
💭 追問：跟 News Feed（動態牆）到底差在哪？ News Feed 的內容來自你追蹤的人（社交圖），核心是「發文時推給粉絲、還是讀時拉」； 這題不靠追蹤，內容來自全站、靠系統猜你的興趣，核心是「召回→排序→回饋」。一個是「給你朋友的動態」，一個是「猜你會愛的內容」。
💭 追問：會不會越推越窄，最後同溫層出不來？ 會，如果只有「利用」沒有「探索」。所以靠三招防：① 探索名額（偷渡新口味）② 多樣性打散（同類不連發）③ 興趣分數衰減（久沒看的舊愛慢慢降溫）。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 VideoStore（影片簿 + 口味簿 + 已看簿）→ Recommender（召回→排序→去重→探索的大腦）→ FeedSession（把「取一頁 / 看一支 / 學起來」串起來）。

1召回：先逛幾個區，把候選丟進推車（Recommender）

第一關，從全部影片裡多路撈出候選——興趣區、熱門區、同好區，合併去重。

function 召回(使用者) {
    候選 = {}
    // (a) 興趣區：標籤命中我喜好的影片
    for (每支影片) if (影片標籤 ∩ 我的喜好標籤) 候選 += 影片
    // (b) 熱門區：全站最熱前幾支（也是冷啟動後備）
    候選 += 熱度最高的前 8 支
    // (c) 同好區：跟我看過的影片共享標籤的（簡化版協同）
    for (每支影片) if (影片標籤 ∩ 我看過的標籤) 候選 += 影片
    return 候選   // 幾百個，快、廣、可漏抓
}

三區合起來＝多路召回。重點是快又廣，多撈沒關係——下一關排序會再篩。

2排序：幫每支打三種分，挑最高的（Recommender）

對推車裡的候選打分：你有多愛 + 大家多愛 + 新不新鮮。

function 打分(影片, 我的喜好) {
    相符度 = Σ (影片每個標籤在「我的喜好」裡的分數)   // 你有多愛 ← 權重最大
    熱度   = 影片.熱度                                  // 大家多愛 ← 小配角
    新鮮度 = max(0, 1 - 影片.上架小時 / 48)             // 越新越高
    return 相符度×3.0 + 熱度×0.004 + 新鮮度×0.5
}
// 全部候選打完分 → 由高到低排好

真實 TikTok 這段是個 AI 模型（預測你會不會看完）；我們用「三種分加權相加」示範，位置和作用一模一樣。「你有多愛」權重(3.0)壓過「熱度」，所以才會個人化，而不是一味推熱門。

3去重 + 探索：剔掉看過的、偷渡一支新口味（Recommender）

排好序後，發之前做兩件事：

// 去重：本輪看過的剔掉
候選 = 候選裡「不在我的已看簿」的那些

// 取一頁：大多取高分(利用)，留 1 個名額給「我沒接觸過的標籤」(探索)
這頁 = 高分前 4 支
for (剩下的候選) if (它有我沒看過的新標籤) { 標記為探索; 這頁 += 它; break }

「剔掉已看」＝去重（不會又看到同一支）；「偷渡一支新標籤」＝探索（打破同溫層、給新片機會）。大規模時「已看簿」會換成省記憶體的 Bloom filter，邏輯一樣。

4回饋迴路：看完就偷偷加分（FeedSession）

這是全題的心臟。你看一支影片後，系統把它的標籤加進你的口味簿，下次就推得更準。

function 看一支(使用者, 影片, 動作) {
    記進「已看簿」(使用者, 影片)        // ← 下次取頁會被去重
    影片.熱度 += 加成[動作]             // ← 看完/讚會讓這支更熱(影響別人)

    權重 = { 讚:1.0, 看完:0.6, 滑掉:0.0 }[動作]   // ← 滑掉幾乎不加
    if (權重 > 0)
        for (影片每個標籤) 我的喜好[標籤] += 權重    // ← 偷偷學起來！
}

前面講的「推 → 看反應 → 更新喜好 → 推得更準」整套就是這段。看完(0.6) 是強訊號、讚(1.0) 加碼、滑掉(0) 不加——因為「看完」比「按讚」更誠實。改的就是「口味簿」，下次召回和排序都會用到它，所以越看越準。

💭 停一下：為什麼「滑掉」的權重是 0，而不是去推一支一樣的？

看答案 因為滑掉代表「不愛」，不該強化這個標籤（權重 0＝不加分）。但它仍被記進「已看簿」→ 下次去重不再推同一支。想更兇一點還能設成負分（主動降低該標籤），這就是把負回饋也納入學習。
🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 召回（粗篩一小撮）Recommender召回() 多路撈候選
② 排序（打分細挑）Recommender打分() 相符度+熱度+新鮮度
③ 去重 + 探索Recommender剔已看 + 留探索名額

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「圖書館館員」解釋，為什麼要「召回 → 排序」兩段？ 50 億本書不可能逐一細評（太慢）。館員先按分類飛快粗篩出幾百本「大概有興趣的」（召回），只對這幾百本仔細評分挑前 10（排序）。先粗後細，才挑得動海量。
Q2：什麼是回饋迴路？為什麼是整題的靈魂？ 推 → 看你反應（看完/讚/滑）→ 更新你的口味 → 下次推得更準，轉個不停。因為使用者沒說過喜好，系統只能從行為偷學，靠這條迴路「越看越準」，這正是 For You 神奇的來源。
Q3：「探索 vs 利用」在防什麼？用迴轉壽司比喻。 防「越推越窄的同溫層」+「新片永遠沒曝光」。你愛鮭魚就多給鮭魚（利用），但偶爾推一盤沒吃過的（探索）——說不定發現新最愛，也讓新料有機會。只給鮭魚會膩。
Q4：新使用者剛註冊、什麼都沒看過，怎麼推？（冷啟動） 沒口味訊號 → 先推大家都愛的熱門 + 偷渡幾支探索，從他對前幾支的反應快速學出口味，之後就轉成個人化。新影片同理：靠新鮮加分 + 探索名額拿初始曝光。
Q5（工程）：為什麼「看影片」的 API 回 202、丟事件流，而不是當場算完？ 因為行為事件量超大（每天數千億筆）。當場硬算會卡爆。丟上事件流（Kafka）非同步消化：輕訊號近即時更新、重計算批次做，使用者不用等。
Q6（工程）：跟 News Feed 動態牆最大的差別是什麼？ 內容來源不同：News Feed 來自你追蹤的人（社交圖，核心是推/拉扇出）；這題不靠追蹤，來自全站 + 系統猜興趣（核心是召回→排序→回饋）。
Q7（工程）：「看完」和「按讚」哪個訊號權重該更高？為什麼？ 通常「看完 / 停留時長」更高。因為按讚要動手指、很多人懶得按（漏訊號）；而「有沒有看完」是更誠實、更普遍的喜好證據。

15. 30 秒回顧

For You = 一個會察言觀色、越來越懂你的 DJ。不用你追蹤誰，它從你的反應偷學你的口味。

4 個關卡：① 召回（從海量粗篩一小撮）→ ② 排序（打分細挑最好的）→ ③ 去重 + 探索（別重複、偷渡新口味）→ ④ 回饋（看反應、學起來、下次更準）。

3 個怕：怕慢/算不動（兩段式 + 預載）、怕越推越窄（探索）、怕冷啟動（新人新片先靠熱門 + 探索）。

工程細節一句話：50 億影片不可能逐一打分 → 召回粗篩、排序細挑；行為量太大 → 走事件流非同步更新口味；靠「回饋迴路」越看越準；影片本身交給 CDN，推薦只管「推哪一支」。
