55 TikTok / Netflix 影片推薦流：召回 → 排序 → 回饋迴路

F 社交/Feed 建議 55 分鐘 Design TikTok / Netflix Video Feed

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「個人化推薦流（For You）＝ 召回 → 排序 → 去重 → 回饋迴路」的綜合題， 把 影片分發（轉碼 + CDN）、推薦排序、 Feed 分頁三題揉在一起。 與 News Feed 的關鍵差異：那題的內容來源是你關注的人（社交圖扇出）； 這題不靠關注，而是系統主動猜你會喜歡什麼（興趣推薦），來源是全站幾百萬支影片。

1. 需求釐清

功能性需求

  • 無限影片流（For You）：打開 App 就有一支接一支的影片，捲到底自動載入下一批。
  • 個人化：依使用者興趣推薦，不需要先追蹤任何人（與 News Feed 最大差異）。
  • 同一輪不重複：一次刷的過程中（session）不會重複推同一支。
  • 回饋學習：看完 / 按讚 / 滑掉等行為會回流，越看越準。
  • 影片上傳 → 轉碼成多碼率 → 經 CDN 播放（沿用 YouTube 那題的分發）。

非功能性需求

  • 黏著（engagement）是北極星指標：推得準 → 看得久 → 留得住。系統一切為它服務。
  • 低延遲：打開即播、捲動無感（p99 取一頁 < 200ms）→ 靠預載下一頁。
  • 讀遠多於寫：刷影片（讀）次數遠大於上傳（寫）；但「看影片產生的行為事件」寫入量極大。
  • 可擴展：影片庫達數十億、日活數億；召回必須能從海量裡快速縮到幾百個候選。
  • 新鮮度 / 多樣性：要給新片曝光、避免同溫層（探索）。
面試開場必問 「推薦靠關注關係還是興趣猜測？」這題是後者（興趣推薦）→ 整題走向就是 召回 → 排序 → 去重 → 回饋迴路，而不是 News Feed 的推/拉扇出。確認後再問「要不要回饋學習、要不要探索新片」。

2. 規模估算

指標假設推算
日活（DAU）5 億—
影片庫累積 ~50 億支召回必須把它縮到幾百個候選才排得動
每次刷取幾支一頁 ~10 支、一場 session 刷 5～10 頁每人每日數百支曝光
取一頁延遲召回 + 排序需 < 100ms靠離線預算候選 + 預載下一頁掩蓋延遲
行為事件（寫）每支曝光都產生看/滑/讚事件每日數千億事件 → 走串流（Kafka）批次聚合更新特徵
候選規模召回每次取 ~數百候選排序模型只精算這數百個，不碰 50 億全庫
關鍵洞察不可能對 50 億支影片逐一打分 → 必須兩段式：召回先用便宜手段把海量縮到幾百個（快、廣、可漏抓），排序再對這幾百個用貴的模型精算。這是所有大型推薦系統的骨架。

3. API 設計

# 取一頁 For You（讀，超高頻，關鍵路徑）
GET /api/v1/feed?user_id=1&limit=10
→ 200 { "items": [ {video_id, tags, score, source:"interest|popular|explore"} ],
        "next_page": [ ... ] }          # 順帶回傳預載的下一頁

# 看一支影片 → 回饋（寫，量極大）
POST /api/v1/watch
Body: { "user_id": 1, "video_id": "v_123",
        "action": "completed|liked|skipped", "watch_ms": 8200 }
→ 202 Accepted（非同步入事件流，不阻塞）

# 上傳影片（寫，低頻但重）
POST /api/v1/videos   → 觸發轉碼管線（多碼率 + 縮圖）→ 完成後可被召回

# 查使用者興趣狀態（除錯 / 個人化頁）
GET /api/v1/users/{id}/interests → { "dance": 3.5, "music": 2.0, ... }
注意source（interest / popular / explore）在正式系統不會回傳，這裡刻意暴露「這支是因興趣、因熱門還是探索來的」，正是本題 demo 要呈現的核心。

4. 資料模型

videos  (影片中繼資料，分片)
  video_id     PK
  tags         []         -- 標籤 / 類別（召回與相符度的依據）
  popularity   BIGINT     -- 累積熱度（看完/讚聚合而來）
  created_at   TIMESTAMP  -- 推算新鮮度
  -- 影片本體：轉碼後多碼率檔放物件儲存 / CDN，不進此表

user_interests  (使用者興趣向量，鍵值儲存)
  user_id → { tag: 分數, ... }   -- 由行為事件回流更新（回饋迴路）

watch_history  (觀看記錄，去重用)
  user_id, session_id → [ video_id, ... ]   -- 同一輪 session 已看 → 去重

candidate_pool  (召回結果，短暫快取)
  user_id → [ video_id, ... ]   -- 多路召回合併後的候選，供排序

watch_events  (行為事件，append-only，量極大)
  user_id, video_id, action, watch_ms, ts  -- 寫 Kafka → 聚合 → 更新興趣與熱度
為何分開影片中繼資料是可被多人共享的真相來源；興趣向量是每人一份且高頻更新；行為事件是高頻 append-only 走串流，不進主庫。三者的讀寫特性完全不同，所以分庫。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 上傳(寫) ─▶ 轉碼管線(多碼率+縮圖) ─▶ 物件儲存 ─▶ CDN ──(播放)──▶ 使用者
                                          │（沿用 YouTube ⑩ 分發）

 取 For You(讀) ─▶ Feed Service
        │
        ├─▶ 召回：興趣路 / 熱門路 / 協同路 ─▶ 候選池(幾百個)
        │
        ├─▶ 排序：對候選打分(相符度×熱度×新鮮度) ─▶ Top-N
        │
        ├─▶ 去重(剔除本輪已看) + 探索(留名額給新片)
        │
        └─▶ 回一頁 + 預載下一頁

 看一支(寫) ─▶ 行為事件 ─▶ Kafka ─▶ 聚合 ─▶ 更新興趣向量 & 影片熱度
                                              └────（回饋迴路）下次推得更準
  1. 分發層：上傳 → 轉碼 → CDN，跟 YouTube 那題相同；推薦只負責「推哪一支」，播放交給 CDN。
  2. 推薦層：召回（多路撈候選）→ 排序（特徵打分）→ 去重 + 探索 → Top-N，並預載下一頁降延遲。
  3. 回饋層：看影片的行為走 Kafka，聚合後更新興趣向量與影片熱度，形成回饋迴路。

6. 核心元件深入（召回 / 排序 / 去重 / 回饋）

召回（candidate generation）：多路撈候選

  • 為何要召回？50 億支影片不可能逐一打分 → 先用便宜手段縮到幾百個。召回要快、廣、可漏抓（寧可多撈，排序再篩）。
  • 興趣路：用使用者偏好標籤，從倒排索引撈相符影片。
  • 熱門路：全站 / 同地區熱門影片，涵蓋面廣，也是冷啟動的後備。
  • 協同路：「看過同片的人也愛看的」——協同過濾 / 向量近鄰（ANN）。demo 用標籤共現近似。
  • 多路結果合併去重成候選池。多路的意義：單一路會有盲點，多路互補才涵蓋得廣。

排序（ranking）：特徵打分

  • 對候選池每支算分：分數 = 相符度×w1 + 熱度×w2 + 新鮮度×w3。
  • 相符度：影片標籤命中使用者興趣的分數總和（個人化主訊號）。
  • 熱度：大家都愛看的通常不會太差（配角，權重要小，否則熱門片壓過個人化）。
  • 新鮮度：新片加分，避免老片霸榜。
  • 真實系統這一層是機器學習模型（GBDT / 深度模型，預估「看完率 / 互動率」）；可解釋的加權線性分數是它的簡化版，邏輯位置完全相同。

探索 vs 利用（explore / exploit）

關鍵洞察 只推「已知你愛看的」會把人困在同溫層，新片也永遠沒機會曝光。所以每頁保留少量探索名額給「你沒接觸過的標籤 / 新片」。 探索太多 → 推不準掉黏著；太少 → 同溫層 + 新片餓死。這是推薦系統的核心張力（多臂吃角子老虎 / bandit 問題）。

去重（同一輪不重複）

  • 記錄「本輪 session 已看」，取下一頁時剔除 → 不會重複推同一支。
  • 大規模下用 Bloom filter 做近似去重（省記憶體，可容極低誤判）。

預載降延遲

  • 回一頁時順手算好下一頁放 next_page，前端可先抓、CDN 預熱下一批影片 → 捲到底瞬間顯示，掩蓋召回+排序的延遲。

冷啟動（cold start）

  • 新使用者：沒興趣訊號 → 靠熱門路 + 探索撐起第一頁；快速從前幾支的行為學出興趣。
  • 新影片：沒熱度 → 靠新鮮度加分 + 探索名額取得初始曝光，再依表現決定是否擴大推送。

回饋迴路（feedback loop）

  • 看完 / 讚 → 把影片標籤加進使用者興趣向量（看完權重高、讚再加碼、滑掉幾乎不加），同時累積影片熱度。
  • 下次召回更貼、排序分更高 → 越看越準。這條迴路是整題（與整個 TikTok）的靈魂。

影片用 CDN 分發

  • 推薦只決定「推哪些 video_id」；真正的影片位元組由 CDN 就近送（沿用 YouTube 轉碼 + 多碼率 + 邊緣快取），與推薦解耦。

7. 擴展與瓶頸

階段 / 瓶頸問題對策
候選生成規模50 億影片無法逐一打分兩段式：召回先縮到幾百個（倒排索引 / ANN 向量近鄰），排序只算這幾百個
即時特徵興趣要隨行為即時更新，否則推不準行為走 Kafka → 串流聚合，興趣向量近即時更新（重特徵離線批次）
模型服務排序模型推論延遲高、QPS 大模型服務水平擴展 + 特徵快取 + 只對召回後的小候選集推論
取頁延遲召回+排序來不及預載下一頁 + 離線預算每人候選 + CDN 預熱影片
去重狀態爆量每人每輪已看清單膨脹Bloom filter 近似去重 + session 過期清掉
影片分發頻寬熱門影片同時被海量人看CDN 邊緣快取 + 多碼率自適應（沿用 YouTube 分發）

8. 取捨與面試重點

  • 個人化 vs 多樣性：全推已知興趣黏著最高但困在同溫層；保留探索名額犧牲一點即時黏著、換長期留存與新片曝光。核心是 explore/exploit 的平衡。
  • 即時 vs 批次興趣更新：即時更新（看一支馬上變準）體驗好但成本高；批次（離線重算）便宜但反應慢。實務混合：輕量訊號即時、重特徵批次。
  • 新片冷啟動：沒熱度的好片如何出頭？→ 新鮮度加分 + 探索名額給初始曝光，再依早期表現（看完率）決定擴大或收手。
  • 召回為何要多路：單路有盲點（純興趣 → 困同溫層、純熱門 → 不個人化）；多路互補涵蓋廣，排序再統一篩。
  • 與 News Feed 的差異：News Feed 內容來自社交圖（你關注誰），核心是推/拉扇出；這題內容來自全站興趣猜測，核心是召回→排序→回饋，不靠關注。
  • 常見追問：
    • 怎麼防回饋迴路把人越推越窄？→ 探索名額 + 多樣性打散 + 興趣分數衰減。
    • 看完率與按讚哪個訊號強？→ 看完（停留時長）通常比讚更真實，權重更高。
    • 怎麼分頁不重複？→ session 級去重（Bloom filter），下一頁剔除已看。
    • 影片很大怎麼秒開？→ 不是推薦的事，交給 CDN + 多碼率 + 預載。

9. PHP Demo 說明

本題 demo 聚焦推薦核心邏輯（綜合題真正的考點），不是 CRUD。可用 PHP 內建伺服器跑起來：

  • src/VideoStore.php　影片庫（標籤 / 熱度 / 新鮮度）+ 使用者興趣向量 + 本輪已看記錄（JSON 檔模擬 DB）。
  • src/Recommender.php　核心：多路召回 → 特徵排序打分 → 去重 → 探索/利用 → Top-N + 預載下一頁。
  • src/FeedSession.php　協調：取一頁 / 看一支（回饋迴路更新興趣與熱度）/ 看狀態 / 重新一輪。
  • public/index.php　路由 + 深色測試頁，可視覺化每支影片的分數、召回分路、探索標記、冷啟動。
# 啟動
cd demo
php -S localhost:8055 -t public
# 瀏覽器開 http://localhost:8055
# 流程：① 載入示範影片 → ② 取 alice 的 For You（dance/music 排前面）
#       → 對某支按「看完 / 讚」→ ③ 再取 For You：那支被去重、同類分數變更高（更貼興趣）
#       換 bob 看冷啟動（無興趣 → 靠熱門 + 探索）
誠實聲明 Demo 用 JSON 檔模擬影片庫與使用者特徵、用加權線性分數模擬排序模型；無真影片、無真 ML 模型、無真轉碼/CDN。 但多路召回、特徵排序打分、session 去重、探索 vs 利用、冷啟動後備、預載下一頁、看一支 → 興趣回饋迴路 → 推得更準，皆為真實可執行邏輯。 上正式環境：召回換倒排索引 + 向量近鄰、排序換 ML 模型服務、興趣更新走 Kafka、影片走轉碼 + CDN，召回→排序→去重→回饋的骨架不變。
