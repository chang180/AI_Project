㊶ 工作流引擎 (Temporal/Airflow)：把「長流程」變成可靠、可重試、可重放的程式

M 排程/工作流 建議 52 分鐘 Design a Workflow Orchestration Engine

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把跨服務、跨小時/天、會失敗會重試的長流程，變成可靠且可觀測的有狀態執行」。 難點不在「依序呼叫幾個 API」，而在 崩潰後如何續跑、失敗如何重試與補償、人工審批如何等待、重啟後狀態如何不丟——這正是 Temporal / Airflow / Step Functions 存在的理由。

1. 需求釐清

為什麼需要工作流引擎？

想像一個訂單履約流程：預留庫存 → 風控人工審批 → 扣款 → 出貨 → 通知客戶。若用一個普通的服務「依序呼叫」來做，會遇到一連串難題：

  • 長流程：審批可能等幾小時甚至幾天，程序不可能就這樣 hang 在那裡佔著執行緒。
  • 可靠（崩潰可續跑）：扣完款、機器在出貨前掛了——重啟後要從「出貨」續跑，而不是重頭把款再扣一次。
  • 可重試：出貨呼叫物流 API 偶發逾時，要能自動退避重試，不是一次失敗整單就死。
  • 可補償（saga）：出貨最終失敗，已經扣的款要退、已預留的庫存要釋放——要能反向回滾。
  • 可觀測：每一步現在在哪、重試幾次、卡在哪個審批，都要能查。

工作流引擎把這些「橫切關注點」（重試、持久化、計時、補償、審批等待）從業務碼裡抽出來，由引擎統一保證。代表系統：

  • Temporal / Cadence：durable execution——你用一般程式碼寫流程，引擎用事件溯源把每一步的結果記下來，崩潰後「重放」歷史把程式狀態還原到中斷點繼續跑。
  • Airflow：以 DAG 描述任務相依，排程器挑 ready 的任務丟給 worker，偏批次/資料管線。
  • AWS Step Functions：以狀態機（JSON/ASL）定義流程，託管執行。

功能性需求

  • 用 DAG 定義工作流：多個 step、step 間有相依（deps）。
  • 排程執行：相依都完成的 step 才 ready；引擎挑 ready step 執行、推進整張圖。
  • durable execution：每步結果持久化；崩潰可從中斷點續跑。
  • 重試 + 退避；step 最終失敗時依 saga 反向補償。
  • 人工審批 / 等待外部事件：執行到審批節點暫停，等外部 signal。

非功能性需求

  • 持久性（durability）：狀態一旦記錄就不可丟，是引擎的生命線。
  • exactly-once 效果：同一 step 不因重試/重放被執行兩次副作用。
  • 可水平擴展：能同時跑大量工作流實例、大量 step。
  • 確定性重放：重放歷史要得到一致結果（流程碼不可依賴非確定來源）。
面試開場必問 「流程定義是程式碼（Temporal，靈活、可分支迴圈）還是宣告式 DAG / 狀態機（Airflow / Step Functions，易可視化、易靜態分析）？」這決定了引擎如何保證持久性與確定性，也是整題分水嶺。

2. 規模估算

指標假設推算
並發工作流實例100 萬個長流程同時「在飛」多數時間在等待（審批/計時/外部）→ 不佔 CPU，只佔狀態
每流程 step 數平均 20 個 step2000 萬個 step 狀態需追蹤
狀態大小每流程 event log ~5 KB100 萬 × 5KB ≈ 5 GB 活躍狀態（+ 歷史歸檔）
step 執行吞吐每秒推進的 step由 worker 數決定；引擎本身只做排程與記錄
等待時長審批可達數天關鍵：等待不能佔用執行緒/記憶體常駐 → 狀態落盤，事件到才喚醒
關鍵洞察工作流的成本不在「算」而在「等與記」。100 萬個流程多半在等待外部事件，真正稀缺的是持久化狀態的容量與一致性，不是 CPU。所以引擎設計圍繞「狀態落盤 + 事件喚醒」，而非常駐執行緒。

3. API 設計

# 定義工作流（DAG：step + 相依 deps）
POST /api/v1/workflows
Body: { "name":"order_fulfillment",
        "steps":[ {"id":"reserve","deps":[],"compensate":"release_stock"},
                   {"id":"approve","type":"approval","deps":["reserve"]},
                   {"id":"pay","deps":["approve"],"compensate":"refund"}, ... ] }

# 啟動一個工作流實例
POST /api/v1/workflows/{name}/start   →  { "run_id":"..." }

# 對等待中的審批節點送 signal 放行（外部事件喚醒）
POST /api/v1/runs/{run_id}/signal     Body: { "step":"approve", "decision":"approve" }

# 查詢執行狀態（各 step 狀態、重試次數、event log）
GET  /api/v1/runs/{run_id}            →  { "status":"RUNNING", "steps":[...], "events":[...] }

# 取消 / 終止（觸發 saga 補償）
POST /api/v1/runs/{run_id}/terminate
注意真實 Temporal 的「推進」不是由 client 呼叫 tick，而是引擎內部由 task queue + worker 驅動；本筆記 demo 為了可觀察，把推進顯式化成 POST /api/run（步進 tick）。

4. 資料模型

workflow_def（工作流定義 / DAG）
  name, version
  steps  JSON  -- [{id, type:task|approval, deps:[...], compensate:"..."}]

workflow_run（實例）
  run_id  PK, name, status(RUNNING|COMPLETED|FAILED), started_at

event_log（事件溯源，append-only，每實例一條 log）
  run_id, seq, type, step, payload(result/attempt/backoff/action), ts
  -- type: WORKFLOW_DEFINED / STEP_COMPLETED / STEP_RETRY / STEP_FAILED
  --       STEP_BLOCKED / STEP_APPROVED / STEP_COMPENSATED / WORKFLOW_FAILED

-- 「目前狀態」不是一張可變表，而是 event_log 的 fold（重放推導）：
--   state = reduce(applyEvent, events, initialState)
為何用 event log 而非可變狀態表 把「狀態」存成一連串不可變事件（事件溯源），帶來三個關鍵能力：(1) 崩潰可續跑——重啟後重放 log 即還原進度；(2) 確定性重放——同樣的 log 永遠 fold 出同樣狀態；(3) 完整稽核——每一步何時做、重試幾次都留痕。這就是 durable execution 的底層。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 工作流定義(DAG) ──▶ 排程器(挑 ready step) ──▶ Worker(執行 step)
                          ▲                          │
                          │                          ▼
                    重放重建狀態 ◀──────────────  Event Log(事件溯源/durable)
                    (已完成不重跑)
                          │
        審批節點 BLOCKED ─┘  ← 收到 signal 放行          失敗──▶ 重試(指數退避)
                                                              │(超上限)
                                                              ▼
                                                  saga 反向補償(notify←ship←pay←…)
  1. 排程器：由 event log 重放出目前狀態 → 找出 deps 全完成的 ready step → 派發執行。
  2. Worker：執行 step 的實際工作（呼叫 API / 算東西），結果回報。
  3. Event Log：每個 step 的開始/完成/失敗/重試/補償都 append 進去，是唯一真相來源（持久化）。
  4. 重放：任何時刻「目前進度」都由重放 log 得到；崩潰重啟也靠它續跑、且已完成 step 不重跑。

6. 核心元件深入

① DAG 排程與相依

  • 工作流是有向無環圖：定義時要驗證 deps 都存在、且無環（Kahn 拓樸排序，環會導致永遠跑不完）。
  • ready 判定：一個 step 的所有 deps 都 DONE 且自己還 PENDING → ready。引擎每一輪挑出所有 ready step（可並行）推進。
  • 無分支的鏈是 DAG 的特例；真正的價值在「fan-out / fan-in」——多個前置完成才解鎖後續（如 reserve 與 charge_prep 都完成才能 approve）。

② Durable Execution / 事件溯源重放

  • 狀態 = fold(events)：引擎不維護一份可變的「目前狀態」，而是隨時重放 event log 推導。
  • 重放不重跑：重放時遇到 STEP_COMPLETED 就直接讀 log 裡記下的結果，不重新執行該 step 的副作用。所以崩潰後重啟，已扣的款不會再扣一次。
  • 這就是 Temporal 的招牌：你的流程碼看起來像普通函式呼叫，但每個「活動(activity)」結果都被記進歷史；重啟時引擎快速重放歷史把你的程式「快轉」回中斷點，再從那裡往下跑。
面試金句「Durable execution = 事件溯源 + 確定性重放。已完成步驟的副作用只發生一次（記在 log），重放只是把控制流還原，不重做副作用。」

③ 冪等與 exactly-once 效果

  • 嚴格的「exactly-once 投遞」在分散式下不可能；引擎做的是 at-least-once 執行 + 冪等 = exactly-once 效果。
  • 機制：每個 step 配唯一 id；執行前先查 log「這個 step 是否已 STEP_COMPLETED」，是則跳過。配合下游冪等鍵（如付款用 idempotency-key），重試/重放都安全。

④ 重試 + 指數退避

  • step 失敗（暫時性錯誤如逾時）→ 記 STEP_RETRY（含 attempt 與 backoff），退避後重試。
  • 指數退避：100ms → 200ms → 400ms…（真實系統再加 jitter 抖動避免重試風暴）。
  • 達 重試上限仍失敗 → 標記 STEP_FAILED，視為永久失敗，觸發補償。

⑤ Saga 補償

  • 長流程跨多服務，無法用單一 DB 交易回滾。saga 模式：每個會產生副作用的 step 配一個補償動作（pay→退款、reserve→釋放庫存）。
  • 某 step 最終失敗 → 對已完成的 step 依 反向順序執行補償（後完成的先補償）。例：notify ← ship ← pay ← approve ← reserve。
  • 補償動作本身也要冪等、也要記進 log（補償也可能失敗重試）。

⑥ 人工審批 / 等待外部事件

  • 審批節點執行到 → 進入 BLOCKED，狀態落盤後就釋放資源，不佔執行緒等待（這是長流程能擴展的關鍵）。
  • 外部送 signal（人按了核准 / 計時器到期 / webhook）→ 寫 STEP_APPROVED + STEP_COMPLETED，引擎在下一輪喚醒、繼續推進。
  • 同樣機制可實現「等計時器」「等子工作流」「等外部回呼」。

⑦ 確定性要求

  • 因為要「重放歷史還原狀態」，流程碼必須是確定性的：不可直接讀系統時鐘、亂數、外部 API（這些每次重放會不一樣，導致歷史對不上）。
  • 做法：所有非確定來源都走「activity / step」——它的結果記進 log，重放時拿 log 的值而非重新求值。Temporal 對此有嚴格的「workflow code 不可有副作用」約束。

7. 擴展與瓶頸

瓶頸問題對策
大量並發工作流百萬實例同時在飛實例依 run_id 分片(sharding)到多個歷史服務節點；無狀態排程器水平擴展
長時間等待的狀態保存審批等數天，不能常駐記憶體狀態落盤、釋放資源；用計時器服務 / 事件索引，到期或 signal 才喚醒重載
worker 擴展step 執行吞吐不足worker 與引擎解耦，透過 task queue 拉取；按佇列加 worker；長/短任務分流
event log 變長長壽流程歷史龐大、重放變慢continue-as-new（截斷歷史另起新 run）/ 週期性快照(snapshot) 縮短重放
歷史服務寫入熱點append log 的寫壓力依 run_id 分片；批次/順序寫；冷歷史歸檔到物件儲存

8. 取捨與面試重點

  • 編排(orchestration) vs 編舞(choreography)：本題是編排——有個中央引擎掌握全流程、好觀測與補償，但引擎是焦點與單點考量；編舞則靠服務間發事件彼此觸發，去中心、鬆耦合，但流程「散」在各處、難追蹤難補償。長交易/需補償的流程多選編排。
  • 同步 vs 非同步：同步呼叫鏈簡單但會被最慢一環卡死、不耐久等；工作流引擎本質是非同步、有狀態，用持久化換到「可等很久、可崩潰續跑」。
  • 狀態機 vs 程式碼定義流程：宣告式 DAG / 狀態機（Airflow、Step Functions）易可視化、易靜態檢查、好限制確定性；程式碼定義（Temporal）表達力強（任意分支、迴圈、區域變數），但要靠執行期約束保證確定性與可重放。
  • 常見追問：
    • 重放為什麼不會重複扣款？→ 已完成 step 從 log 拿結果，不重跑副作用。
    • 審批等三天，引擎掛了怎麼辦？→ 狀態在 log，重啟重放即還原 BLOCKED；signal 一到照常喚醒。
    • 補償也失敗怎麼辦？→ 補償亦記 log、可重試；最終可進「人工介入」死信狀態。
    • 怎麼保證流程碼確定性？→ 禁止流程碼直接讀時鐘/亂數/外部，一律走 activity 記結果。

9. PHP Demo 說明

本題 demo 著重於工作流引擎的核心邏輯（也是面試真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/WorkflowEngine.php　核心引擎：DAG 定義與驗證（無環）、tick 挑 ready step 推進、replay 事件溯源重建狀態（不重跑）、重試指數退避、saga 反向補償、審批 signal 放行。
  • public/index.php　路由（/api/workflow、/api/run、/api/signal、/api/fail、/api/state、/api/reset）+ 步進測試頁（DAG 狀態表 + event log）。
  • 內建範例 DAG：訂單履約（reserve, charge_prep → approve(審批) → pay → ship → notify）。
