㊷ 分散式排程器 / Cron：不重不漏地觸發百萬個定時任務

M 排程/工作流 建議 42 分鐘 Design a Distributed Scheduler / Cron

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「可靠地在對的時間、把對的任務、恰好觸發一次、並能水平擴展」。 單機 crontab 很簡單；面試真正考的是 多台機器 一起跑時，如何 不重複執行（去重）、不漏執行（misfire 補跑）、且負載能分散。

1. 需求釐清

功能性需求

  • 使用者註冊一個排程任務：固定間隔（每 5 分鐘）或 cron 表達式（每天 02:00、每週一）。
  • 到點時觸發任務：呼叫一個 webhook / 丟一條訊息到佇列 / 跑一段工作。
  • 能查詢執行歷史、下次觸發時間、失敗重試。
  • 可暫停 / 取消 / 修改排程。

非功能性需求

  • 不重複執行：一次觸發只能跑一次（或至少要冪等去重）。這是整題的核心。
  • 不漏執行：機器當掉、重啟也不能把該跑的任務吞掉（misfire 要能補）。
  • 水平擴展：從數百個 job 到上億個 job、每秒上萬次觸發。
  • 觸發準時：延遲可接受秒級，但不能無限漂移。
  • 高可用：沒有單點；單台排程器掛掉，其他要接手。
面試開場必問 「要 exactly-once 還是 at-least-once + 冪等？」
分散式下精確一次幾乎做不到（網路、當機隨時發生），務實答案是 at-least-once 觸發 + 任務本身冪等 / 用 fencing 去重。把這點講清楚，整題就穩了。

2. 規模估算

指標假設推算
排程 job 數1 億個 job每筆中繼資料 ~300B → ~30 GB（可進 DB / 分片）
觸發頻率平均每 job 每小時觸發 1 次1 億/3600 ≈ ~2.8 萬 觸發/秒（峰值 ×3 ≈ 8 萬）
掃描成本每秒掃「nextRun ≤ now」的 job需索引 nextRun；不能全表掃 1 億筆
worker 數單 worker 每秒處理 ~500 觸發2.8 萬 / 500 ≈ ~56 個 worker（峰值上百）
執行歷史每次觸發記一筆，留 30 天2.8萬×86400×30 ≈ 700 億筆 → 走時序庫 / 過期清理
關鍵洞察 瓶頸不在「執行」而在「掃描到點的 job」與「多 worker 搶同一個觸發時的去重」。 設計圍繞這兩點：高效找出到點 job（索引 / 時間輪） 與 claim 鎖確保只跑一次。

3. API 設計

# 建立排程（固定間隔或 cron）
POST /api/v1/jobs
Body: { "name": "send-report", "schedule": "*/5 * * * *",
        "target": { "type": "webhook", "url": "https://..." },
        "misfire_policy": "fire-once" }
→ 201 { "job_id": "j_abc", "next_run": "2026-06-01T02:05:00Z" }

# 修改 / 暫停 / 取消
PATCH  /api/v1/jobs/{id}   Body: { "paused": true }
DELETE /api/v1/jobs/{id}

# 查下次觸發與歷史
GET /api/v1/jobs/{id}              → { next_run, last_run, status }
GET /api/v1/jobs/{id}/runs        → [ { run_id, fired_at, worker, result } ]

# （內部）worker 認領一個到點觸發 —— 防重複的核心
POST /internal/claim   Body: { "fire_id": "...", "worker_id": "w1" }
→ 200 { "ok": true, "fence_token": 4711, "lease_until": "..." }   # 認領成功
→ 409 { "ok": false, "reason": "already-claimed" }                # 已被搶走（去重）

4. 資料模型

jobs                              -- 排程定義
  job_id        PK
  schedule      VARCHAR   -- cron 表達式或 interval
  next_run      TIMESTAMP -- ★建索引：掃描靠它
  misfire_policy ENUM     -- fire-once | skip | fire-all
  paused        BOOL
  shard         INT       -- 雜湊分片，決定哪個 worker 群負責

fires / claims                    -- 每次「到點」產生一筆，claim 去重的單位
  fire_id       PK
  job_id        FK
  scheduled_at  TIMESTAMP -- 邏輯上應觸發的時刻
  claimed_by    VARCHAR   -- 認領的 worker（NULL=未認領）
  fence_token   BIGINT    -- 單調遞增，fencing 用（拒絕殭屍寫入）
  lease_until   TIMESTAMP -- 租約到期；過期未完成 → 可被別人接管
  status        ENUM      -- pending | claimed | done | missed

runs                              -- 執行歷史（高頻 append → 走時序庫）
  run_id, fire_id, worker, fired_at, result, fence_token
為何要 fires 這張表 去重必須有一個「唯一的觸發實例」當鎖的對象。把 (job_id, scheduled_at) 做成唯一鍵， 多 worker 都想 insert/claim 同一筆時，DB 唯一約束 + 條件更新自然只讓一個成功 —— 這就是去重的根。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
             ┌──────────────┐        掃描 next_run ≤ now
 排程定義 ───▶│ Job Store(DB) │◀─────────┐
             └──────────────┘          │
                                  ┌─────┴──────┐  (leader 選舉：只一個在掃，避免重複)
                                  │  Scheduler  │──▶ 產生 fire（到點觸發）
                                  └─────┬──────┘
                                        ▼  分派（依 shard）
                        ┌────────────────────────────────┐
                        │   多個 Worker 競爭 claim 同一 fire │
                        │   鎖 + fencing token + 租約       │
                        │   → 只有一個成功 → 執行           │
                        └───────────┬─────────────────────┘
              worker 當掉           ▼ 租約過期未完成
              ─────────────▶  misfire / 接管：另一 worker 重新 claim（換更大 token）
  1. Scheduler（掃描 + 分派）：週期性查 next_run ≤ now 的 job，為每個到點 job 產生一個 fire；多台時用 leader 選舉 或 分片 避免重複掃描。
  2. Worker（認領 + 執行）：搶 claim 一個 fire，拿到 fencing token 後執行，完成寫 run；at-least-once + 去重。
  3. 協調儲存：DB / etcd / ZooKeeper 提供原子的 claim、租約、leader 選舉。

6. 核心元件深入

怎麼高效找出到點的 job？

  • 索引掃描（優先佇列）：對 next_run 建索引，每輪 WHERE next_run ≤ now ORDER BY next_run LIMIT N。簡單、好分片，億級 job 也可行。
  • 時間輪（Timing Wheel）：把任務掛到「未來時間槽」的環狀桶上，指針每 tick 前進一格只看當格 —— O(1) 觸發，Kafka / Netty 用此法。代價是長延遲任務要多層輪 + 重新入輪。
  • 取捨：索引掃描好理解、好水平分片（面試首選說法）；時間輪在超高頻、單機密集觸發時更省。

防重複執行：claim / lock + fencing（本題核心）

  • 每次到點是一個唯一 fire（job_id + scheduled_at）。多 worker 都想跑它，用原子的條件更新當鎖：UPDATE fires SET claimed_by=:me WHERE fire_id=:f AND claimed_by IS NULL —— 只有第一個影響到 1 列的 worker 算認領成功。
  • fencing token：認領時發一個單調遞增的 token。worker 帶 token 去寫結果 / 呼叫下游；下游只接受「比上次更大的 token」。這擋掉「殭屍 worker」：它卡住（GC、網路）租約過期被接管後才醒來，拿著舊 token 想寫入會被拒。
  • 租約（lease）：認領附帶到期時間。worker 須在租約內完成或續租；逾期未完成 → 視為當掉，鎖自動釋放，別人可重新 claim（拿更大 token）。避免「認領者死了任務永遠卡住」。
  • 結論：at-least-once 觸發 + fencing 去重，務實地逼近 exactly-once。

misfire 補跑策略

  • fire-once：不管錯過幾次，補跑一次就好（如「寄日報」漏了補一次即可）。
  • skip：直接跳過，等下一個正常週期（如「每分鐘清快取」，補跑無意義）。
  • fire-all：每個錯過的週期都補跑（如「每小時結算」，每筆都不能少）。
  • 實作：worker 復活 / 掃描時發現 next_run 已落後 now 多個週期，依策略決定產生幾個 fire，並把 next_run 推進到下一個未來時刻。

分片與 rebalance

  • job 依 hash(job_id) % N 分到 N 個 worker 群，各自只掃自己那片 → 掃描與執行負載分散，且天然降低 claim 競爭。
  • worker 增減時 rebalance（一致性雜湊 / 重新指派）；搬遷期短暫重複沒關係，靠 claim 去重兜底。

leader 選舉：避免重複掃描

若每台 scheduler 都掃全表，會重複產生 fire（雖然 claim 會去重，但浪費）。用 leader 選舉（etcd lease / ZooKeeper ephemeral node / DB advisory lock）讓「同一片」只有一個在掃；leader 掛掉，租約過期後其他人補位。

7. 擴展與瓶頸

瓶頸症狀對策
海量 job 掃描每秒掃 1 億筆找到點 jobnext_run 索引 + 分片 + 時間輪；只取 LIMIT N
觸發時間尖峰整點 / 02:00 大量 job 同時到點（thundering herd）抖動（jitter）分散觸發、令牌桶限流、預先攤平 next_run
長任務佔用某 worker 跑很久，租約過期被接管 → 重複跑長任務拆小 / 主動續租（heartbeat lease）/ 任務冪等
claim 競爭熱點同一批 fire 大量 worker 搶鎖分片降低競爭 + 批次認領（一次 claim 一段）
執行歷史寫爆每次觸發寫一筆 run批次寫 / 時序庫 / 過期清理

8. 取捨與面試重點

  • 精確觸發 vs 可靠觸發：要絕對準時很貴；務實是允許秒級延遲、保證最終一定觸發（不漏）。
  • at-least-once + 冪等 > exactly-once：別承諾 exactly-once。承諾「至少一次觸發 + 任務冪等 / fencing 去重」，這是分散式的標準答案。
  • 集中排程 vs 去中心：單一 leader 掃描簡單但有上限；分片 + 多 leader 可擴展但要處理 rebalance 與邊界重複。
  • 索引掃描 vs 時間輪：前者好分片好理解；後者超高頻省成本但長延遲任務複雜。
  • 常見追問：worker 在執行到一半當掉怎麼辦？→ 租約過期被接管 + fencing 擋殭屍 + 任務冪等。整點驚群怎麼解？→ jitter + 限流。怎麼保證不漏？→ misfire 策略 + 持久化 next_run。

9. PHP Demo 說明

本題 demo 著重於系統核心邏輯（也是分散式排程真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/Scheduler.php　排程定義（每 N tick 觸發 + nextRun）、tick 推進邏輯時鐘產生到點 fire、claim 防重複（fencing token + 租約到期）、misfire 策略（fire-once / skip / fire-all）、依雜湊分片指派 worker。
  • public/index.php　路由（加 job / tick / worker 上下線 / claim / complete / state / reset）+ 互動測試頁。
# 啟動
cd demo
php -S localhost:8042 -t public
# 瀏覽器開 http://localhost:8042
# 加 job → tick 推進看哪些 job 觸發 → 多 worker 但每次觸發只被一個認領
# → 把認領中的 worker 下線、tick 過租約，看 misfire 補跑與接管
誠實聲明 因 PHP 無跨請求常駐計時器，Demo 用手動 tick 推進邏輯時鐘做確定性模擬「多 worker 常駐排程」。 其中到點觸發、防重複 claim（fencing token + 租約到期）、misfire 補跑、分片指派皆為真實可執行邏輯。 真實系統改為：worker 常駐進程依牆鐘自掃、協調狀態放共用儲存（DB / etcd / ZooKeeper）、心跳偵測當掉、leader 選舉避免重複掃描 —— 核心去重與補跑模型一致。
