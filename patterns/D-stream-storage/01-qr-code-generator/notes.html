① QR Code 生成器：從零到百萬級請求

D 串流計算/儲存分發 建議 45 分鐘 Design a QR Code Generator

一句話定位 本質是「URL 短碼服務 + 圖片生成 + 大量讀取的快取/CDN 分發」的綜合題。 寫死的靜態 QR 很簡單；面試真正考的是 動態 QR（可改目標、可追蹤掃描）如何擴展到百萬級請求。

1. 需求釐清

功能性需求

  • 輸入一段文字/URL，產生對應的 QR Code 圖片（PNG / SVG）。
  • 動態 QR：QR 內嵌的是一個短網址 https://qr.io/abc123，掃描後 302 轉址到真實目標；目標可後續修改。
  • 掃描分析：記錄掃描次數、時間、地區、裝置。
  • 自訂：尺寸、容錯等級、可選 logo。

非功能性需求

  • 高可用：轉址路徑是關鍵路徑（掃描的人很多），需 99.99%。
  • 低延遲：轉址 < 50ms（p99）。
  • 讀遠多於寫：建立 QR 少、掃描多（讀寫比可達 100:1 以上）。
  • 可擴展：從零到百萬 QPS。
  • 短碼不可預測（避免被遍歷）、不可碰撞。
面試開場必問 「是靜態還是動態 QR？」決定了整題走向。靜態 = 純無狀態圖片生成；動態 = URL 短碼服務（本筆記主軸，因為它才有擴展挑戰）。

2. 規模估算

指標假設推算
建立 QR（寫）1000 萬/日~115 寫 QPS（峰值 ×3 ≈ 350）
掃描（讀）每碼平均 100 次掃描~11.5k 讀 QPS（峰值 ≈ 35k）
儲存（中繼資料）每筆 ~500 bytes，5 年1000萬×365×5×500B ≈ 9 TB
短碼空間base62，7 碼62⁷ ≈ 3.5 兆，足夠多年
QR 圖片生成後可快取/不存可由短碼即時重算，省儲存
關鍵洞察讀寫極不對稱 → 整個設計圍繞「讀路徑快取化」與「寫路徑可靠生成唯一短碼」。

3. API 設計

# 建立動態 QR（寫）
POST /api/v1/qrcodes
Body: { "target_url": "https://example.com/promo", "size": 512, "ecc": "M" }
→ 201 { "code": "abc123X", "short_url": "https://qr.io/abc123X",
        "image_url": "https://cdn.qr.io/abc123X.svg" }

# 更新目標（動態 QR 的賣點）
PATCH /api/v1/qrcodes/{code}   Body: { "target_url": "..." }

# 掃描轉址（讀，超高頻，關鍵路徑）
GET /{code}        → 302 Location:    （並非同步記錄一次掃描）

# 取得圖片
GET /api/v1/qrcodes/{code}/image?fmt=svg|png

# 掃描分析
GET /api/v1/qrcodes/{code}/stats → { "scans": 1234, "by_day": [...] }

4. 資料模型

qrcodes
  code         VARCHAR(7)  PRIMARY KEY   -- base62 短碼
  target_url   TEXT
  owner_id     BIGINT
  ecc          CHAR(1)                   -- L/M/Q/H 容錯等級
  created_at   TIMESTAMP
  updated_at   TIMESTAMP

scan_events (寫入量大 → 走串流/時序庫，不進主庫)
  code, ts, country, device, referrer        -- 寫入 Kafka → 聚合
scan_counters (聚合結果)
  code, total_scans, daily JSONB
為何分開掃描事件是 append-only 的高頻寫，放主庫會拖垮它；獨立到 Kafka + 時序/列式儲存（如 ClickHouse），主庫只保留可變的對映。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
            ┌─────────┐
 寫 ───────▶│  API GW  │──▶ Create Service ──▶ 短碼產生器(KGS) ──▶ Metadata DB(分片)
            └─────────┘                                              │
                                                                     ▼
 讀(掃描) ──▶ CDN ──▶ Redirect Service ──▶ Redis 快取 ──(miss)──▶ Metadata DB
                          │
                          └──▶ Kafka(掃描事件) ──▶ 聚合 ──▶ 分析庫
  1. Create Service：取得唯一短碼、寫入對映、回傳 short_url。
  2. Redirect Service：無狀態、可大量水平擴展；查快取→轉址→非同步發掃描事件。
  3. 圖片：由短碼即時生成 SVG（無狀態、可快取於 CDN），或生成後存物件儲存。

6. 核心元件深入

短碼如何唯一又不可預測？

  • 方案 A：雜湊截斷　hash(url+salt) 取前 7 碼 → 需處理碰撞（重試/加鹽）。同一 URL 可去重。
  • 方案 B：發號器(KGS, Key Generation Service)　預先產生大量唯一碼放池子，服務啟動時批次領取一個區段（如領 1000 個），用完再領。無碰撞、低延遲、易擴展，面試首選。
  • 不可預測：用全域遞增 ID 經 base62 + 打亂（如 Feistel/乘法雜湊），避免 abc124 緊鄰 abc123 被遍歷。

讀路徑快取

  • Redis 快取 code→url，命中率可 >95%（熱門碼集中）。
  • CDN 對轉址回應做短 TTL 快取需謹慎（動態 QR 可改目標）→ 通常 CDN 只快取圖片，不快取 302。

掃描計數不阻塞轉址

轉址先回 302，非同步把事件丟 Kafka；下游聚合。即使分析管線掛了，掃描體驗不受影響（關鍵路徑與分析解耦）。

7. 擴展與瓶頸（從零到百萬級請求）

階段瓶頸對策
單機CPU / 連線數無狀態服務 + 負載均衡，水平加機器
DB 讀爆掃描查詢打 DBRedis 快取 + 讀副本
DB 寫/容量單庫裝不下、寫熱點依 code 雜湊分片；KGS 去除寫時碰撞重試
全球延遲跨洲 RTT多區域部署 + CDN 就近轉址 + 快取就近
分析寫爆每次掃描寫一筆Kafka 緩衝 + 批次聚合 + 近似計數

8. 取捨與面試重點

  • 雜湊 vs 發號器：雜湊可天然去重但要處理碰撞；KGS 無碰撞但需額外服務與預生成。
  • 圖片即時生成 vs 預生成存檔：即時省儲存、靠 CDN 快取；預生成省 CPU 但耗儲存且改目標時 QR 圖不需變（QR 內是短碼，不含目標）→ 其實圖片可永久快取，只有對映會變，這是動態 QR 的精髓。
  • 強一致 vs 最終一致：對映更新需即時生效（讀自己的寫）→ 更新時主動失效 Redis；掃描計數可最終一致。
  • 常見追問：如何防惡意 URL（釣魚）？→ 黑名單/掃描安全檢查。如何防短碼被枚舉？→ 不可預測碼 + rate limit。

9. PHP Demo 說明

本題 demo 著重於系統核心邏輯（也是百萬級請求真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/ShortCode.php　base62 發號 + 乘法雜湊打亂（不可預測短碼）。
  • src/Store.php　對映儲存（JSON 檔模擬 DB）+ 行內快取，統計命中/未命中。
  • src/QrRenderer.php　把短網址渲染成 SVG（教學版方格圖；§註明如何換成 endroid/qr-code 產生可掃描 QR）。
  • public/index.php　路由：建立 / 轉址 / 統計，並附簡易測試頁。
# 啟動
cd demo
php -S localhost:8000 -t public
# 瀏覽器開 http://localhost:8000 建立 QR；掃描轉址走 GET /{code}
誠實聲明 Demo 的 QR 圖為教學用方格示意（非真正可掃描編碼）；短碼、儲存、快取、轉址、計數等系統邏輯為真實可執行。要產生可掃描 QR，依 demo/README.md 安裝 endroid/qr-code 即可無痛替換 QrRenderer。
