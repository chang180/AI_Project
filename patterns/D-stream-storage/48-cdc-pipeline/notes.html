㊽ CDC 資料管線：讓快取/搜尋/數倉跟著主庫一起變

D 串流計算/儲存分發 建議 45 分鐘 Design a CDC Data Pipeline (Debezium)

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把「主庫改一筆 → 下游怎麼跟著變」拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把資料庫的『變更』本身當成事件流」：讀主庫的 binlog，把每筆 insert/update/delete 變成有序事件，讓快取、搜尋索引、數倉等衍生資料自動跟著主庫變動，而不必在應用層做容易出錯的「雙寫」。

1. 需求釐清

問題背景

主庫（OLTP）是真相來源，但同一份資料常常要以不同形狀出現在很多地方：搜尋要進 Elasticsearch、 熱門查詢要進 Redis 快取、報表要進數倉（OLAP）。問題是：主庫一改，這些衍生副本怎麼跟著改？

最天真的做法：雙寫（dual write） 應用程式「寫完 DB，順手也寫一份去快取/ES」。聽起來簡單，但只要其中一步失敗（DB 成功、ES 失敗，或反之）， 兩邊就永久不一致；而且這是分散式無原子性的兩次寫，沒有交易能包住它。CDC 就是為了解掉這個。

功能性需求

  • 擷取來源表的每一筆變更（insert / update / delete），含變更前後（before/after）。
  • 把變更以有序事件流發布出去，供任意多個下游消費。
  • 下游能把變更套用到自己的衍生視圖（搜尋索引、去正規化快取、數倉）。
  • 支援初始快照（先把現有資料全量灌一次）+ 之後的增量。

非功能性需求

  • 有序：同一個鍵（主鍵）的變更必須依發生順序套用，否則新值被舊值覆蓋。
  • 不漏：寧可重送也不漏送（at-least-once）。
  • 冪等：重送的事件套第二次，結果要與套一次相同。
  • 低侵入：盡量不改應用程式碼、不拖慢主庫。
  • 最終一致：下游允許有秒級延遲（lag），但最終要追上主庫。

2. 規模估算

指標假設推算
主庫寫入5000 萬筆變更/日~580 變更 QPS（峰值 ×3 ≈ 1.7k）
每筆事件大小before+after ~1 KB日量 ~50 GB binlog 事件
下游數量ES + 快取 + 數倉 = 3 個消費群同一事件流被讀 3 次（fan-out 讀）
可接受 lag搜尋 < 5s、數倉可分鐘級決定消費者並行度與批次大小
初始快照主庫 10 億列全量回填需數小時 → 要可斷點續傳
關鍵洞察變更 QPS 本身不大，真正的工程難點是 有序 + 冪等 + 不漏，以及「初始快照如何與增量無縫接上」。

3. API / 介面設計

CDC 系統對外主要是「事件契約」與「消費介面」，而非 REST：

# 變更事件（Debezium 風格，精簡版）—— 這是整個系統的「契約」
{
  "lsn":    10231,            // log sequence number / offset，分區內單調遞增
  "op":     "u",              // c=create  u=update  d=delete  (r=快照讀)
  "table":  "products",
  "pk":     42,               // 主鍵 → 決定分區，保證同鍵有序
  "before": { "id":42, "price":100 },   // u/d 有；c 為 null
  "after":  { "id":42, "price":80  },   // c/u 有；d 為 null
  "ts":     "2026-06-01T06:33:49Z"
}

# 消費介面（拉模式）
GET  /changes?after_lsn=10000     → 回 LSN>10000 的有序事件
POST /consume                     → 下游套用一批；套完才提交 offset
GET  /offset                      → 目前消費位移（已套用到哪）

# 控制面
POST /snapshot/{table}            → 觸發初始快照（全量讀 → 以 op=r 灌入）
為何用 LSN 當鍵LSN（位移）既是排序依據，又是冪等去重鍵，還是斷點續傳的書籤——一個欄位身兼三職，是這題的核心。

4. 資料模型

-- ① 來源表（主庫，真相來源；應用程式照常 CRUD）
products
  id     BIGINT PRIMARY KEY
  name   TEXT
  price  INT
  ...                                  -- 變更會寫進 DB 自己的 WAL/binlog

-- ② 變更日誌（binlog → Kafka topic；append-only、依 LSN 有序）
change_log
  lsn    BIGINT   -- 單調遞增位移（排序 + 去重 + 書籤）
  op     CHAR(1)  -- c/u/d/r
  table  TEXT
  pk     BIGINT   -- 決定分區（同 pk 進同分區 → 同鍵有序）
  before JSON
  after  JSON
  ts     TIMESTAMP

-- ③ 下游衍生視圖（每個下游一份，形狀各自不同）
derived_view (例：去正規化搜尋文件)
  pk     BIGINT PRIMARY KEY
  ...攤平後的欄位...
  _lsn   BIGINT   -- 記住「這列由哪個 LSN 套出來的」→ 冪等比較用

-- ④ 消費位移（每個消費群一筆：已套用到哪個 LSN）
consumer_offset
  consumer_group TEXT
  applied_lsn    BIGINT
四件套來源表（真相）＋變更日誌（有序事件流）＋衍生視圖（下游副本）＋消費位移（書籤）。整個 CDC 就是這四者的舞蹈。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 應用 ──CRUD──▶ 主庫(MySQL/PG) ──寫入──▶ binlog/WAL
                                            │ 讀
                                            ▼
                                    Debezium(CDC connector)
                                            │ 轉成變更事件(op/before/after/LSN)
                                            ▼
                                    Kafka(topic, 依 pk 分區、分區內有序)
                          ┌─────────────────┼─────────────────┐
                          ▼                 ▼                 ▼
                 消費者A→ES索引       消費者B→Redis快取   消費者C→數倉
                 (依序冪等套用)       (依序冪等套用)     (批次套用)
  1. 主庫：應用照常寫，變更自然落進 binlog（不需改業務碼）。
  2. Debezium：讀 binlog，轉成標準化變更事件，記住讀到哪（LSN）可斷點續傳。
  3. Kafka：以主鍵分區 → 同一筆資料的變更永遠落在同一分區、分區內有序。
  4. 各下游消費者：獨立位移、各自把變更依序冪等套用到自己的視圖。

6. 核心元件深入

讀 binlog vs 應用層雙寫

  • 雙寫：DB 與下游是兩次獨立寫，沒有共同交易 → 任一失敗即永久不一致；還有順序競態（兩個並發更新到 ES 的順序可能反過來）。
  • 讀 binlog（CDC）：binlog 是 DB 交易已提交的單一真相，且天然有序。下游只是「重放」它 → 一致性問題從「協調兩次寫」降級成「重放一條有序日誌」，簡單得多。

有序與 LSN

同一主鍵的 price:100→80→120 若亂序套用，視圖可能停在 80。解法：依主鍵分區（同鍵進同分區）+ 分區內依 LSN 嚴格遞增套用。跨鍵不需全域有序，所以可水平擴展分區。

at-least-once + 冪等套用

消費者「套用完才提交 offset」：若套到一半崩潰、offset 還沒提交，重啟會從舊 offset 重拉 → 同一批事件被重送。要讓重送無害，靠冪等：

  • offset 去重：event.lsn <= applied_lsn 一律跳過。
  • upsert 語意：套用用「以主鍵覆寫」（insert-or-replace），重放同一筆結果不變。
  • 視圖記 _lsn：若收到比視圖現存 _lsn 還舊的事件，可選擇忽略（防亂序/重放套舊值）。

初始快照 + 增量

新接一個下游時，binlog 可能已被清掉舊段。流程：①記下目前 LSN →②全量讀主庫現有資料（op=r 灌入）→③從步驟①的 LSN 開始接增量。難點是快照期間主庫仍在變，需保證快照與增量的交界不漏不重（Debezium 用一致性快照 + 去重視窗處理）。

Schema 變更

來源表加欄位/改型別會反映在 binlog。事件需攜帶 schema 版本，下游要能容忍新欄位（向後相容），常配 Schema Registry 管理演進。

7. 擴展與瓶頸

瓶頸症狀對策
高變更量單一 connector / 分區吞不下依主鍵雜湊多分區並行；熱鍵單獨分區
下游落後（lag）消費追不上、視圖過舊多消費者並行（同 group）、批次套用、下游用更快的 upsert
回填 / 重建索引下游壞了要全量重來可重設 offset 重放整條日誌（冪等保證安全）+ 觸發快照
binlog 保留期消費太慢、舊段被清加大保留期或落 Kafka 長存；落後太多走快照重來
大事務 / 批次更新一筆交易產生海量事件突刺下游限流 + 背壓；分區攤平突刺

8. 取捨與面試重點

  • CDC vs 應用層雙寫：CDC 一致性強、低侵入，但要維運 connector + Kafka；雙寫架構簡單卻幾乎一定會不一致，面試明確不推薦。
  • 最終一致的延遲：下游永遠落後主庫一點（lag）。要接受「讀下游可能讀到舊值」；需強一致的查詢得讀主庫。
  • exactly-once 很難：分散式下真正的「恰好一次」代價高（需端到端交易/冪等寫）。務實做法＝at-least-once + 冪等套用，效果等同 exactly-once 的結果。
  • 查詢 binlog vs 查詢時間戳輪詢：「每隔幾秒撈 updated_at > X」的輪詢法會漏掉 delete、抓不到中間態、且拖主庫；CDC 讀 binlog 沒這些問題。
  • 常見追問：怎麼保證同鍵有序？（主鍵分區）重放會不會把資料弄壞？（冪等 upsert + LSN 去重）下游要全量重建怎麼辦？（重設 offset 重放 / 快照）。

9. PHP Demo 說明

本題 demo 著重 CDC 真正的系統核心——變更擷取、有序、冪等套用，可實際用 PHP 內建伺服器跑起來：

  • src/SourceTable.php　來源表（模擬主庫一張表）：apply(c/u/d) 回傳 before/after。
  • src/ChangeLog.php　變更日誌（模擬 binlog→topic）：append 事件 + 原子遞增 LSN + 依序讀取。
  • src/Consumer.php　下游消費者：依 LSN 依序套用 + 冪等去重 + 維護消費位移。
  • public/index.php　路由：改來源表 / 看變更事件流 / 下游套用 / 一致性比對 / 重放，附測試頁。
# 啟動
cd demo
php -S localhost:8048 -t public
# 瀏覽器開 http://localhost:8048
# ① 改來源表(c/u/d) → ② 看變更事件(op/before/after/LSN)
# → ③ 下游套用 → 兩邊一致 → ④ 重放全部 → 已套用的被 skip（冪等）
誠實聲明 Demo 用 JSON 檔模擬 binlog（真實系統是 Debezium 讀 MySQL/Postgres 的 WAL/binlog → Kafka）； 但變更擷取（op/before/after）、LSN 有序、下游依序冪等套用、最終一致比對等系統邏輯為真實可執行。 要接真 CDC，把 ChangeLog 換成 Kafka consumer、SourceTable 換成真資料庫即可，下游套用邏輯一字不用改。
