1. 60 秒搞懂它在幹嘛

想像一間公司有一本「總帳」（主資料庫），記著所有商品、價格、庫存。 但其他部門也想用這份資料：搜尋部門要拿去做「站內搜尋」、快取部門要放熱門資料讓網頁秒開、報表部門要做月報。

問題來了：總帳每天一直在改。那其他部門手上的副本，怎麼跟著一起變？難道每改一筆，就打電話通知三個部門各自手動改？這太容易出錯了。

🔑 一句話 CDC（變更資料擷取）＝幫總帳裝一台「自動廣播機」：總帳每改一筆（新增/修改/刪除）， 就自動廣播一則「我剛剛改了什麼」的通知，所有部門收到通知照著改自己的副本—— 永遠跟總帳同步，沒人需要手動抄。

2. 為什麼不能「順手也寫一份」就好？

很多人第一個念頭是：「改總帳的時候，順手也幫搜尋部門改一份不就好了？」這招叫雙寫，聽起來合理，其實是個大坑。

📌 比喻：寄兩封信，其中一封寄丟了 你要同時通知「銀行」和「房東」你搬家了，於是寄了兩封信。結果銀行那封寄到了，房東那封寄丟了—— 現在兩邊地址對不起來，而且你不會知道哪封丟了。雙寫就是這樣：改總帳成功、改副本失敗（或反過來），兩邊永久不一致，還沒人發現。

更糟的是順序會亂：兩個人幾乎同時改同一件商品的價格（先改成 80、再改成 120）， 但送到搜尋部門時順序顛倒，副本就停在 80——新的被舊的蓋掉了。

💭 停一下：為什麼「改完總帳，順手也改副本」這招這麼脆弱？

看答案 因為這是兩次獨立的動作，沒有任何東西保證「兩次都成功」或「兩次順序一致」。 只要其中一次失敗或順序亂掉，兩邊就對不起來，而且很難察覺。CDC 的聰明之處是：不要做兩次寫， 而是讓副本去「重播總帳已經發生的事」——總帳是唯一真相，副本只是照著它的紀錄抄。

3. 把系統拆成 4 個關卡

別被「資料管線」嚇到。整件事其實就是闖 4 關：

1擷取：總帳改了什麼，記下來

好消息：資料庫本來就會偷偷記自己改過什麼——這本內部小帳叫 binlog（變更日誌）。 我們不用改任何商業程式，只要派一個小幫手去讀這本小帳，就知道每一筆改動。

📹 比喻：監視器錄影 你不用叫店員「每做一件事都回報一次」（很煩、會漏）。店裡本來就有監視器把一切錄下來， 你只要看錄影就知道發生過什麼事。binlog 就是資料庫的監視器，CDC 就是看錄影的人。

每一筆改動會被整理成一則變更事件，內容包含：是新增/修改/刪除（op）、改之前長怎樣（before）、改之後長怎樣（after）。

2編號：每則事件貼一個「流水號」

每則變更事件都貼一個遞增的流水號（叫 LSN，可以想成「第幾號改動」）。這個小小的號碼，後面會立大功。

🎫 比喻：銀行抽號碼牌 銀行給每個人一張號碼牌（1、2、3…），照號碼順序叫號，誰都不會插隊。 變更事件也一樣：照流水號順序處理，就不會發生「新的被舊的蓋掉」。

3廣播：把事件流推給所有部門

整理好的變更事件，丟進一條廣播管道（工程上叫 Kafka）。任何想同步的部門都來訂閱這條管道，各看各的、互不干擾。

📰 比喻：訂報紙 報社印一份報紙，很多人各自訂閱。搜尋部門、快取部門、報表部門都訂同一份「變更日報」， 各自照自己的進度讀——有人讀得快、有人讀得慢，互不影響。

4套用：每個部門照事件改自己的副本

每個部門收到事件，就照流水號順序更新自己的副本：看到「新增」就加一筆、「修改」就覆寫、「刪除」就拿掉。改完，副本就跟總帳一樣了。

✍️ 比喻：照著食譜步驟做菜 食譜步驟有編號（1.切菜 2.下鍋 3.調味）。你照順序一步步做，最後一定做出同一道菜。 就算你不小心重做了某一步（手滑），只要那步是「把鹽加到剛好的量」這種做幾次都一樣的動作，結果也不會壞——這個「重做也不壞」的特性叫冪等。
✅ 四關一句話 ① 看監視器擷取總帳改了什麼 → ② 給每則改動貼流水號 → ③ 用報紙方式廣播給所有部門 → ④ 各部門照順序套用更新副本。就這樣。

4. 設計時最怕的 3 件事

這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「順序亂掉」 同一件商品的價格 80→120 若被反過來套用，副本就停在 80（新的被舊的蓋掉）。 所以才要流水號（LSN）+ 照順序套用，還要保證「同一件商品的改動永遠走同一條隊伍」。照號碼牌叫號，不准插隊。
😱 怕「漏掉一則」 如果有一則變更事件沒送到，副本就永遠少了那一筆，而且很難發現。 所以原則是寧可重送也不漏送（at-least-once）：寧願同一則送兩次，也絕不漏。少送是災難，多送可以補救。
😱 怕「重送把資料弄壞」 既然「寧可重送」，那同一則事件套兩次會不會出事？所以套用動作要設計成做幾次都一樣（冪等）： 用流水號當「這則我套過了嗎」的記號，套過的直接跳過。重做也不壞，才敢放心重送。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前先估「大概來幾個人」，才知道訂幾個便當。系統設計也一樣：先估數字，才知道要準備多大的「料」。
估什麼大概數字所以呢？
總帳每天改幾筆~5000 萬筆平均每秒幾百筆，其實不算多
每則事件多大before+after ≈ 1 KB一天的「變更日報」約 50 GB
幾個部門訂閱搜尋 + 快取 + 報表 = 3同一份事件被讀 3 次，互不影響
副本可以慢多久搜尋 < 5 秒、報表可分鐘級決定要派幾個工人、一次套幾筆
💡 關鍵洞察：難的不是「量」，是「正確」 每秒才幾百筆改動，機器很閒。真正難的是順序不能亂、一筆都不能漏、重送也不能弄壞—— 這三件事才是 CDC 的核心，不是流量。

💭 停一下：流量這麼小，為什麼還是個有難度的題目？

看答案 因為它考的是「資料一致性」，不是「扛流量」。要讓三份副本永遠跟總帳一樣， 必須同時做到有序 + 不漏 + 冪等，還要處理「新部門加入時怎麼先把舊資料補齊（初始快照）」。 這些都是邏輯上的難題，跟筆數多少無關。

8. 系統之間怎麼溝通？（事件契約）

🍔 比喻：標準格式的通知單 各部門收到的變更事件，要長成大家都看得懂的固定格式，這叫「契約」。 講好格式，誰來訂閱都能照著處理。

這題的「通知單」長這樣（看註解就懂）：

一則變更事件（CDC 的核心契約）
{
  "lsn":    10231,        // 流水號：第幾號改動（排序 + 去重用）
  "op":     "u",          // 動作：c=新增  u=修改  d=刪除
  "pk":     42,           // 哪一筆（主鍵）→ 同一筆永遠排同一隊
  "before": { 改之前長怎樣 },   // 修改/刪除才有
  "after":  { 改之後長怎樣 }    // 新增/修改才有
}

部門怎麼「訂閱」與「回報進度」：

GET  /changes?after_lsn=10000   給我：流水號 10000 之後的所有事件
POST /consume                   我套用了一批，請幫我記住「套到哪」
GET  /offset                    我現在套到第幾號了？（書籤）

9. 系統要記住哪些事？（資料模型）

📒 比喻：四本筆記本 這題剛好用四本筆記本，對應前面的故事：

📕總帳（來源表）

唯一的真相：商品編號、名稱、價格…。應用程式照常增刪改。

📰變更日報（變更日誌）

一則一則的改動紀錄，只增不改、照流水號排好：流水號、動作(c/u/d)、改前、改後。

📗各部門的副本（衍生視圖）

每個部門一份，形狀各自不同（搜尋文件 / 快取 / 報表）。每筆多記一個 _lsn＝「這筆是第幾號改動套出來的」，重送時可比對。

🔖各部門的書籤（消費位移）

記「這個部門已經套到第幾號了」。下次從這個號碼往後接著套就好。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時總有某個環節先塞車。一個一個看：
哪裡會塞車怎麼拓寬
改動太多，一條隊伍排不完分成多條隊伍（依商品編號分），但「同一筆」永遠排同一隊（保證有序）
某部門套得太慢、副本太舊多派幾個工人並行、一次批次套很多筆
部門的副本壞了要重建把書籤歸零、重放整份日報——因為冪等，重放不會弄壞
新部門剛加入，沒有舊資料先做初始快照（把總帳現有的全抄一次），再接後續改動
監視器錄影被清掉（保留期短）加大保留期，或落後太多就重做快照

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心。這裡補幾個常被面試官追問的：

⚖️ CDC（看監視器）vs 雙寫（順手也寫一份） CDC 一致性強、不用改商業程式，但要多維護「廣播管道」這套基礎建設；雙寫架構簡單卻幾乎一定會不一致——面試明確別選雙寫。
⚖️ 副本一定會「晚一點」（最終一致） 副本永遠落後總帳一點點。要接受「查副本可能查到舊值」；真的需要最新的，就直接查總帳。
⚖️ 「剛好一次」很難，所以用「至少一次 + 冪等」 分散式系統要做到「每則剛好處理一次」代價很高。務實做法：至少送一次（不漏）＋套用冪等（重送不壞）＝效果等同剛好一次。
💭 追問：為什麼不用「每隔幾秒撈一次『最近改過的』」就好，幹嘛讀 binlog？ 因為「撈最近改過的」會漏掉刪除（被刪的撈不到）、抓不到兩次撈之間的中間變化，還會一直去煩總帳拖慢它。 讀 binlog（監視器）能看到每一筆、含刪除、照順序，而且不打擾總帳。
💭 追問：怎麼保證「同一件商品」的改動不會亂序？ 把事件依商品編號分隊：同一個編號永遠進同一條隊伍，隊伍內照流水號處理。 不同商品之間不需要管順序，所以可以拆很多條隊伍同時跑（又快又有序）。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「改總帳 → 產生事件 → 副本跟著變 → 重放也不壞」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8048 -t public
# 然後瀏覽器打開 http://localhost:8048
  1. 改來源表：建立 id=1（填 name、price）→ 看「變更日誌」多了一筆 op:c（含 after、流水號）。
  2. 下游套用：按「下游套用」→ 副本（衍生視圖）跟著出現 id=1，狀態變「✔ 一致」。
  3. 再改：修改 id=1 的 price → 看新事件 op:u（before 舊、after 新）→ 再套用 → 又一致。
  4. 重放全部：把所有事件再餵一次。已套過的會被「skip」，結果完全不變——這就是冪等。
玩的時候想一下建一筆、套用、看到一致後，按「重放全部」——副本一點都沒變（全被 skip）。 這就證明了：就算上游因為「寧可重送」把同一則丟了好幾次，副本也不會被弄壞。這正是 CDC 敢「至少送一次」的底氣。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 SourceTable（總帳，改一筆就回報改前/改後）→ ChangeLog（變更日報，貼流水號、排好序）→ Consumer（部門工人，照順序冪等套用）。改一筆，就照這順序跑一遍。

1擷取：改總帳，回報改前/改後（SourceTable）

對總帳做一筆改動，它回傳「改之前」和「改之後」——這就是變更事件的原料。

function apply(動作, id, 欄位) {
    改之前 = 總帳裡這筆現在的樣子   // before（可能是 null＝本來不存在）
    if (動作 == 刪除)  從總帳拿掉這筆;   改之後 = null
    else               寫入/覆寫這筆;     改之後 = 新的樣子   // after
    return { before:改之前, after:改之後 }
}

關鍵：它同時給你 before 和 after。真實系統不用自己算，因為資料庫的 binlog 本來就把這兩個都錄下來了。

2貼流水號：排好序的變更日報（ChangeLog）

每收到一筆改動，就分配一個遞增流水號，連同 op/before/after 一起記下來。

function append(動作, id, before, after) {
    流水號 = 上一個流水號 + 1        // ← 單調遞增：第幾號改動
    事件 = { lsn:流水號, op:動作, pk:id, before, after }
    把事件加到日報最後面（只增不改）
    return 事件
}

這個「+1 的流水號」就是後面有序與去重的關鍵。真實系統的流水號來自資料庫 binlog 的位移，意思一樣。

3冪等套用：照順序、套過的跳過（Consumer）

這是全題的心臟。部門工人拿到一批事件，先照流水號排好，再一筆一筆套；套過的直接跳過。

把這批事件「照流水號由小到大」排好           // ← 保證有序

for (每則事件) {
    if (事件.流水號 <= 我已套到的流水號)
        跳過;                              // ← 冪等：套過的不再套（重送無害）
    if (事件.op == 刪除)  從副本拿掉這筆;
    else                  以主鍵覆寫這筆（順手記下 _lsn）;
    我已套到的流水號 = 事件.流水號          // ← 套完才推進書籤
}

三個怕全在這段化解：排序→不怕亂序；「套完才推進書籤」→崩潰重來會重拉（不漏）；「套過的跳過」+「以主鍵覆寫」→重送也不壞（冪等）。

💭 停一下：為什麼是「套完才推進書籤」，而不是「先推書籤再套」？

看答案 因為要寧可重做也不漏做。如果先推書籤、結果套到一半就崩潰，重啟後書籤已經往前了 → 那筆永遠不會再套＝漏了。 反過來「套完才推書籤」，崩潰時書籤還停在舊的 → 重啟會把那批重拉一次（最多重送，靠冪等擋掉），絕不漏。

關卡（前面講的）哪個檔關鍵那段
① 擷取（改前/改後）SourceTableapply() 回傳 before/after
② 貼流水號ChangeLogappend() 的 lsn+1

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「自動廣播機」解釋，CDC 在幹嘛？ 幫總帳（主庫）裝一台自動廣播機：每改一筆就廣播「我改了什麼」（含增/改/刪、改前、改後）， 所有訂閱的部門照著更新自己的副本，永遠跟總帳同步，沒人需要手動抄。
Q2：為什麼「改完總帳順手也改副本（雙寫）」是個坑？ 因為那是兩次獨立的寫，沒有東西保證兩次都成功、順序一致。任一邊失敗或順序亂掉，兩邊就永久不一致還難察覺。 CDC 改成「讓副本重播總帳已經發生的事」，總帳是唯一真相。
Q3：流水號（LSN）有什麼用？ 一號三用：① 排序（照號碼套用，不怕亂序）② 去重（套過的號碼直接跳過，重送無害）③ 書籤（記住套到第幾號，下次接著套）。
Q4：這個系統最怕的三件事是什麼？各自怎麼解？ 怕順序亂（→ 流水號 + 同一筆排同一隊）、怕漏掉一則（→ 寧可重送也不漏送 at-least-once）、怕重送弄壞（→ 套用冪等：套過跳過 + 以主鍵覆寫）。
Q5（工程）：為什麼「套完才推進書籤」而不是反過來？ 為了不漏。先推書籤再套，崩潰時那筆會被永遠跳過（漏了）。套完才推書籤，崩潰時會重拉那批（最多重送，靠冪等擋掉），絕不漏。
Q6（工程）：副本壞了要全部重建，怎麼做才安全？ 把書籤歸零、重放整份變更日報。因為套用是冪等的（以主鍵覆寫 + 流水號去重），重放整份的結果跟正常跑一次完全一樣，不會弄壞。
Q7（工程）：新部門剛加入，binlog 又只留最近一段，怎麼補齊舊資料？ 先做初始快照：記下目前流水號 → 把總帳現有的全部抄一次（當成一批「讀取」事件灌入）→ 再從剛記下的流水號開始接後續增量。難點是接縫處不漏不重。

15. 30 秒回顧

CDC 資料管線 = 幫總帳裝一台自動廣播機，讓所有副本自動跟著一起變。

4 個關卡：① 看監視器(binlog)擷取改動 → ② 給每則改動貼流水號(LSN) → ③ 用報紙方式廣播(Kafka) → ④ 各部門照順序冪等套用更新副本。

3 個怕：怕順序亂（流水號 + 同一筆排同一隊）、怕漏掉（寧可重送 at-least-once）、怕重送弄壞（套用冪等）。

工程細節一句話：別用雙寫（必不一致），改讓副本「重播總帳已發生的事」；流水號一號三用（排序/去重/書籤）；「套完才推書籤」保證不漏；副本要重建就歸零書籤重放整份日報（冪等保證安全）。
