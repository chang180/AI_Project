1. 60 秒搞懂它在幹嘛

想像你開了一間停車場。每台車進來都要在閘口抽一張票，月底你照「進場次數」跟客戶收停車費。 廣告點擊聚合就是這件事的網路版：每支廣告被點一下，就像一台車進場；系統要算每支廣告被點了幾次， 然後照次數跟廣告主結帳收錢。

聽起來不就是「數數字」嗎？難就難在——這個數字直接等於錢。多算一下，廣告主被多扣錢、會翻臉； 少算一下，平台自己虧。而且每天有幾十億次點擊湧進來，還有人專門派機器人來亂點。

🔑 一句話 廣告點擊聚合 = 一台超精準、防作弊的點擊收費機：海量點擊湧進 → 去掉重複的、擋掉作弊的 → 每小時結算一次帳 → 拿去收錢，一塊錢都不能算錯。

2. 為什麼「數點擊次數」沒那麼簡單？

很多人第一個反應是：「不就是 COUNT(*) 加一加？」這題真正的陷阱在一個你生活裡也遇過的問題——

📮 比喻：同一封信寄了兩三次 你網購下單，系統怕你沒收到通知，同一封確認信寄了兩三次給你。你收到三封，但其實只下了一張單。 如果倉庫照「收到幾封信」出貨，就會出三次貨——這就慘了。

網路傳訊息也一樣：為了「保證不漏掉」，系統寧可同一筆點擊送個兩三次（這叫「至少送一次」）。 問題是——同一次真實點擊，絕對不能算兩次錢。所以系統得有辦法認出「這封信我收過了」，重複的就丟掉。

再加上第二個麻煩：有人會派機器人狂點別人的廣告，把對手的廣告預算燒光。這種點擊雖然是「真的有人（機器）點了」， 但不能算錢。所以系統還要會「看點擊的行為，判斷哪些是作弊的」。

💭 停一下：「同一封信收兩次」跟「機器人狂點」，這兩個問題能用同一招解決嗎？

看答案 不行，要分兩關處理。「同一封信收兩次」是內容一模一樣的重複——每筆點擊帶一個獨一無二的編號，看到重複的編號就丟掉即可。 但機器人狂點時，每一下都是「不同的」點擊（編號都不一樣），騙得過去重那一關，所以要另外一關專門「看行為」：同一個人/IP 短時間點了幾百下，就判定作弊。所以是兩個獨立的關卡。

3. 把系統拆成 4 個關卡

別被「串流系統」嚇到。整件事其實就是讓每一筆點擊闖 4 關，一關一關來：

1收下：先把點擊「接住」，別讓人等

使用者點一下廣告，系統要馬上接住、立刻放行，不能讓使用者卡在那邊等「結算完成」。

🅿️ 比喻就像停車場閘口：車一來，「叭」一聲抬桿、丟一張票給你就放行，不會叫你停在閘口等收費員把帳算完。算帳是後面的事。

所以收點擊的服務只做一件事：把點擊丟進一條「待處理輸送帶」就回「收到」。這條輸送帶（叫 Kafka）能先把暴衝的車流囤著，後面慢慢處理。

2去重：同一筆點擊，只准算一次

輸送帶上難免有重複的點擊（同一封信寄了兩三次）。這一關要把重複的挑掉。

🎫 比喻：演唱會驗票 每張票有獨一無二的票號。驗票員手上有一本「已進場名單」，掃到沒看過的票號就放行並記下來； 掃到已經進過場的票號，就擋下來（同一張票不能讓兩個人進）。

系統一樣：每筆點擊在發生的當下就帶一個獨一無二的編號（叫 click_id）。去重關有一本「看過的編號」名單，重複的就丟掉。 這樣不管同一筆點擊被送來幾次，都只算一次——這就是花大力氣要做到的「exactly-once（剛好一次）」。

⚠️ 小細節：那本「看過的編號」名單不能無限長（會撐爆記憶體），所以只記最近一段時間（例如最近幾小時），過期就清掉。

3防作弊：擋掉機器人灌的點擊

剩下的點擊雖然都「不重複」了，但裡面藏著機器人狂點的假流量，不能拿去收錢。

🏪 比喻：店員看出來「這人不對勁」 一個人一分鐘衝進你店裡 300 次，店員一看就知道不正常。系統也是：盯著「同一個 IP / 同一台裝置」， 如果短時間內點擊次數爆量，就判定是作弊，這些點擊不算錢。

記住：去重 ≠ 防作弊。去重是擋「一模一樣的重複」；防作弊是擋「看起來很多筆、但行為很可疑」的點擊。兩件不同的事，所以是兩關。

4結算：把時間切成整段，分段算帳

剩下的就是「乾淨、可收錢」的點擊了。最後一關：把它們按時間分段加總。

⏰ 比喻：每小時結算一次帳 像便利商店每小時清點一次收銀機：8 點到 9 點一筆帳、9 點到 10 點一筆帳……每段時間不重疊、不漏接， 一筆點擊只會被算進它發生的那一段。這種「把時間切成整齊、不重疊的段」就叫 tumbling window（翻滾視窗）。

於是系統就能回答：「廣告 A 在 14:00～15:00 這段被點了 1832 次」。這些分段數字加起來，就是要跟廣告主收的錢。

✅ 四關一句話 ① 收下點擊立刻放行 → ② 去重（重複的票擋掉）→ ③ 防作弊（機器人灌的不算）→ ④ 分段結算（每小時一筆帳）。乾淨的數字拿去收錢。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「算錯錢」 這是頭號大忌。多算→廣告主被多扣、告你、不再投廣告；少算→平台自己虧錢。 所以同一筆點擊絕不能算兩次（去重做到 exactly-once），這是硬規定，沒得商量。
😱 怕「漏掉點擊」 點擊一旦在網路上弄丟，那筆錢就收不回來了。所以傳輸時寧可重送也不漏送（叫 at-least-once）。 重送造成的重複怎麼辦？交給第 2 關去重處理。寧可重複，不可遺失。
😱 怕「被作弊洗錢」 如果擋不住機器人灌點擊，廣告主的預算會被燒在假流量上，發現後對整個平台失去信任。 所以「把作弊點擊踢出帳單」跟「算對錢」一樣是底線。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每天多少次點擊50 億次／天 ≈ 每秒 5.8 萬筆（暴衝時 ×5）不可能逐筆慢慢寫進傳統資料庫，要用輸送帶削峰
原始點擊資料多大每筆約 200 bytes → 一天約 1 TB原始事件存便宜的倉庫，只把「算好的數字」放報表庫
「看過的票號」名單多大留 1 小時 ≈ 2 億筆 ≈ 3.2 GB會一直長大 → 一定要設保存期限（TTL），過期清掉
結算結果多少1000 萬支廣告 × 每分鐘一格 → 一天上百億列要用「壓縮得很好的報表庫（OLAP）」存
🌊 關鍵洞察：高速公路收費站 點擊像高速公路車流：尖峰瞬間暴衝、平常普通。如果收費站要求「每台車當場算完月帳才放行」，整條路會塞死。 所以做法是先抬桿放行＋丟張票（收下就回 202），把車流囤在輸送帶上，後面再慢慢結算。

💭 停一下：那本「看過的票號」名單，為什麼不能永遠留著？只留最近幾小時會不會出錯？

看答案 永遠留著會把記憶體撐爆（一天就 2 億筆 ×24）。但保存期限也不能太短——要比「上游最晚可能重送的時間」還長， 否則一筆遲到的重複點擊送來時，名單裡已經把它清掉了，就會被當成新的、多算一次錢。所以這是「省記憶體 vs 算對錢」的取捨。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住三張「點餐單」：

① 上報一筆點擊（最高頻）
   POST /clicks
   給我：{ 票號 click_id, 廣告 ad_id, 是誰 user_id, 從哪 ip, 點擊時間 ts }
   我回：202「收到了！」← 立刻回，不讓你等結算

為什麼「立刻回 202」？因為點擊的人不該卡著等我們去重、防作弊、結算。先收下、丟上輸送帶，後面慢慢處理。 注意票號(click_id)是「點擊發生的當下」就由前端產生——這樣同一筆點擊不管被重送幾次都帶同一個票號，去重才認得出來。

② 查某支廣告分時段的點擊數（給報表用）
   GET /ads/{ad_id}/clicks?from=...&to=...&granularity=1m
   我回：{ 每分鐘一格的可計費點擊數, 總計 }

這是「讀」的點餐單，直接從算好的報表庫撈，所以很快（不必當場去翻幾十億筆原始點擊）。

③ 查廣告主的帳單（即時近似 vs 對帳後精確）
   GET /advertisers/{id}/billing?date=2026-05-31
   我回：{ 即時近似金額, 對帳後精確金額 }

會給兩個數字：即時近似（馬上看得到，可能差一點點）和對帳後精確（日終重算過、可拿去真正收錢）。

9. 系統要記住哪些事？（資料模型）

📒 比喻：幾本不同用途的本子 系統處理的每一關，都各自記在自己的「本子」裡。為什麼要分開？因為每本本子的用途和壽命都不一樣—— 驗票名單看完就丟、帳本要長期保存。混在一起，一本爆了會拖垮全部。

🎫看過的票號（去重用，短命）

記「最近哪些 click_id 已經算過了」。鑰匙就是票號本身，看到重複的就丟。 這本只留最近幾小時（設 TTL），不然會無限長大。

🚫各 IP 的點擊紀錄（防作弊用，短命）

記「每個 IP / 裝置最近這陣子點了幾下」。用來抓「短時間爆量」的機器人。一樣只看近期一小段時間。

⏰各時段的結算數字（帳本，長存）

記 (廣告 ad_id, 哪個時段 window_start) → 這段算了幾次可計費點擊。 這就是最後拿去收錢的帳本，要長期保存，放進壓縮得很好的報表庫（OLAP）。

為什麼帳本要按「廣告＋時段」當鑰匙？ 因為收錢就是問「這支廣告、這段時間，被點幾次」。用這組當鑰匙，查帳和累加都直接對得上，不用再翻原始點擊。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
點擊暴衝，一台機器收不下收集服務不存任何狀態，多開幾台＋輸送帶(Kafka)先囤著削峰
「看過的票號」名單越長越大設保存期限(TTL)過期清掉＋按票號切成好幾份分開存
所有廣告都擠同一個結算器輸送帶按廣告分流，同一支廣告固定走同一個結算器，各算各的
有「超熱門廣告」獨自塞爆一台把熱門廣告再拆成好幾股分頭算，最後合併
報表查詢量太大打爆結算器把算好的數字放進報表庫(OLAP)＋快取熱門廣告，查詢不碰原始資料
即時數字跟真實值有一點點誤差日終重放原始點擊重算一次（對帳），把帳校到精確

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 至少送一次＋去重 vs 真正的「剛好一次傳輸」 讓網路傳輸「保證剛好送一次」極難又貴。業界務實做法：傳輸只保證不漏（會重送）→ 收端用票號去重，效果等同剛好一次。
⚖️ 票號名單留多久？ 留太短 → 遲到的重複點擊漏抓、多算錢；留太長 → 記憶體爆。原則：留得比上游最晚可能重送的時間還長一點。
⚖️ 翻滾視窗 vs 滑動視窗 收錢一定用翻滾視窗（每小時一段、不重疊，一筆點擊只算一段）。滑動視窗（會重疊）拿來算「近 5 分鐘平均」做監控就好，拿去收錢會重複算錢。
💭 追問：點擊「遲到」了（網路塞車晚到 10 分鐘）怎麼算？ 用「事件時間」歸到它真正發生的那一段，不是按收到的時間。系統用「水位線」判斷某段是否到齊才結算；遲到太久、那段已結算的，就丟到日終對帳時再補算。
💭 追問：結算器中途當機重啟，會不會把同一段重算、害人被多扣錢？ 不會。結算器會定期存檔（checkpoint）並記住處理到輸送帶的哪個位置；重啟後從上次位置重放，重放出來的重複點擊會被第 2 關去重擋掉，所以不會多算。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「去重 → 防作弊 → 結算」這條漏斗：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8022 -t public
# 然後瀏覽器打開 http://localhost:8022
  1. 按「跑範例批次」（等於餵一批點擊事件進管線）。
  2. 觀察那條漏斗：原始事件數 → 去重後剩多少 → 踢掉作弊後剩多少 → 最終可計費數。
  3. 再看分時段的聚合結果：每支廣告每個視窗被算了幾次。
玩的時候想一下把同一筆點擊（同一個 click_id）餵兩次，看「去重後」的數字會不會變多——不會，這就是 exactly-once 去重在運作。再把同一個 IP 連點很多下，看它怎麼被防作弊關踢出帳單。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那條漏斗到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的三件大事。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 ClickIngest（驗票員＝去重，認出重複的票號）→ FraudFilter（保全＝防作弊，看行為踢掉機器人）→ Aggregator（收銀機＝結算，每小時清點一次）。一筆點擊進來，就照這個順序闖三關。

1去重：同一張票只准進一次（ClickIngest）

第一關，先問一句「這張票我收過了嗎？」。手上有一本「已進場名單」（看過的票號），重複的直接擋掉。

function 收下(點擊) {
    票號 = 點擊.click_id
    if (看過名單 裡已經有 票號)
        return 丟掉;            // ← 重複的票 → 擋下，不重複計費
    看過名單[票號] = true        // 首次出現才記下來、放行
    return 接受;
}

就這一句 if，把上游「寧可重送也不漏送」造成的重複擋在門外。不管同一筆點擊被送來幾次，只有第一次會過——這就是把「至少一次」收斂成 exactly-once（剛好一次計費）。對應前面「演唱會驗票」那個比喻。

💭 停一下：那本「看過名單」如果永遠留著，會出什麼問題？

看答案 會無限長大，把記憶體撐爆（一天就 2 億筆）。所以真實系統會給它設保存期限（TTL），只留最近幾小時——但要留得比「上游最晚可能重送的時間」還長，否則遲到的重複會漏抓、多算錢。

2防作弊：短時間爆量就踢掉（FraudFilter）

過了去重的點擊都「不重複」了，但裡面藏著機器人狂點。這一關看行為：同一個 IP 在一小段時間內點太多下，就判作弊。

function 是不是正常的(點擊) {
    近期 = 這個 IP 在「最近 10 秒」內的點擊（舊的丟掉）  // ← 滑動視窗
    if (近期筆數 >= 門檻 5)
        return 作弊;            // ← 短時間爆量 → 剔除，不計費
    記下這次點擊時間
    return 正常;
}

注意：這裡的「最近 10 秒」是會跟著時間往前滑的滑動視窗（拿來抓行為，不是拿來算錢）。一旦同 IP 在窗內達門檻就剔除。對應「店員看出來這人一分鐘衝進來 300 次」的比喻。去重 ≠ 防作弊：去重擋一模一樣的票號，這關擋的是票號各不相同、但行為很可疑的洪水點擊。

3結算：把時間切成整段，分段加總（Aggregator）

最後剩下「乾淨、可收錢」的點擊。這關把它們按時間分段加起來。關鍵就一行：算出這筆點擊屬於哪一段。

function 歸到哪一段(點擊時間 ts) {
    return (ts ÷ 視窗長度) 取整 × 視窗長度   // 例：視窗 60 秒，ts=14:00:37 → 落到 14:00:00 那段
}
function 累加(點擊) {
    這一段 = 歸到哪一段(點擊.ts)
    帳本[廣告][這一段] += 1               // ← 同段同廣告，累加成一個數字
}

這就是 tumbling window（翻滾視窗）：時間被切成固定長度、不重疊的整段，一筆點擊只會落進它發生的那一段（對應「每小時清點一次收銀機」）。而且是用事件時間（點擊真正發生的 ts）歸段，所以就算事件亂序、晚到，也會落到正確的那一段。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
② 去重（exactly-once）ClickIngest收下() 裡「看過名單」的 if
③ 防作弊FraudFilter是不是正常的() 的爆量門檻
④ 分段結算Aggregator歸到哪一段() 的取整公式

👉 重點：真實系統會把「記憶體名單 + 迴圈」換成 Kafka（輸送帶）+ Flink（串流引擎，含 keyed-state / watermark）+ Redis/RocksDB（去重狀態）+ ClickHouse（報表庫）， 但上面這三段判斷邏輯一字都不用改：去重還是「看過就丟」、防作弊還是「窗內爆量就踢」、結算還是「ts 取整歸段累加」。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「同一封信收兩次」解釋，為什麼點擊計數不是單純 COUNT(*)？ 因為傳輸層為了不漏掉，會把同一筆點擊重送兩三次（像同一封確認信寄兩三次）。直接 COUNT 會把重複也算進去 → 多收錢。所以要先去重，同一筆只算一次。
Q2：去重和防作弊為什麼是「兩個不同的關卡」？ 去重擋的是「票號一模一樣的重複」；防作弊擋的是機器人灌的點擊——它每一下票號都不同，騙得過去重，所以要另一關專門「看行為」（同 IP 短時間爆量就剔除）。
Q3：tumbling window（翻滾視窗）是什麼？用「每小時結算」說說看。 把時間切成不重疊的整段（像便利商店每小時清點一次收銀機），每筆點擊只屬於它發生的那一段，加總起來就是帳。不重疊，所以不會重複算錢。
Q4：為什麼傳輸選「寧可重送也不漏送」，而不是想辦法剛好送一次？ 漏掉點擊那筆錢就收不回來了，而「保證剛好送一次」的傳輸極難又貴。務實做法：傳輸保證不漏（會重送），重複的交給收端的去重關處理，效果等同剛好一次。
Q5（工程）：收到點擊為什麼要「立刻回 202」而不是算完才回？ 因為點擊的人不該卡著等去重、防作弊、結算跑完。先接住、丟上輸送帶(Kafka)就回「收到」，後面非同步慢慢處理。像停車場閘口先抬桿放行，算帳是後面的事。
Q6（工程）：為什麼 click_id（票號）一定要在「點擊發生的當下」由前端產生？ 這樣同一筆點擊不管被重送幾次都帶同一個票號，去重才認得出是重複。如果等到伺服器才生編號，重送就會變成兩個不同編號，去重失效，會多算錢。
Q7（工程）：「看過的票號」名單為什麼要設保存期限，期限要設多長？ 不設會無限長大撐爆記憶體。但要設得比「上游最晚可能重送的時間」還長，否則一筆遲到的重複送來時名單已清掉它，就會被當成新的多算一次。這是省記憶體 vs 算對錢的取捨。

15. 30 秒回顧

廣告點擊聚合 = 一台超精準、防作弊的點擊收費機。

4 個關卡：① 收下點擊立刻放行 → ② 去重（重複票擋掉，做到 exactly-once）→ ③ 防作弊（機器人灌的不算）→ ④ 分段結算（每小時一筆帳，tumbling window）。

3 個怕：怕算錯錢（同一筆絕不算兩次）、怕漏掉點擊（寧可重送＋去重）、怕被作弊洗錢（假流量踢出帳單）。

工程細節一句話：先估「高速公路尖峰」般的暴衝 → 收到先回 202、票號上游產生 → 三本本子（看過的票號短命／IP 紀錄短命／結算帳本長存）＋名單設 TTL → 即時出近似、日終重算對帳到精確。
