㉑ 直播平台：即時低延遲串流與洪峰觀看 / 聊天 fan-out

D 串流計算/儲存分發 建議 44 分鐘 Design a Live Streaming Platform

一句話定位 本質是「即時攝入（RTMP/SRT/WebRTC）＋ 邊轉邊切的即時轉碼 ＋ 滑動視窗 HLS manifest ＋ CDN 全球分發 ＋ 聊天 fan-out」的綜合題。 與 VOD（⑩ YouTube）最大差別：影片不存在於未來 — 無法預轉碼、無完整 manifest，一切邊產生邊推送，觀眾只能看到「live edge（直播邊緣）」附近的幾秒。 面試真正考的是 延遲 vs 穩定的取捨、滑動視窗 manifest 的維護、以及洪峰觀看與聊天的 fan-out。

1. 需求釐清

功能性需求

  • 開播 / 收播：主播用 OBS / App 推流（RTMP / SRT / WebRTC），平台即時接收。
  • 即時轉碼：把單一來源流轉成多碼率（如 1080p / 720p / 480p / 360p），供不同頻寬觀眾選擇。
  • 低延遲播放：觀眾透過 HLS / LL-HLS 拉流觀看，延遲愈低愈好（互動直播需 < 3 秒）。
  • 聊天室：同一房間觀眾即時互發訊息，需 fan-out 給房內所有人。
  • 觀看人數：即時顯示同時在線人數（concurrent viewers）。
  • 可選：錄製轉 VOD：直播結束後把 segment 拼成回放（這時就退化成 VOD 題）。

非功能性需求

  • 低延遲：端到端（glass-to-glass）愈短愈好；互動場景需 LL-HLS / WebRTC 級（秒級或亞秒級）。
  • 高可用：主播這條攝入鏈一斷，整場直播就黑掉 → 攝入需備援。
  • 洪峰彈性：明星開播瞬間湧入數十萬～數百萬觀眾，必須 CDN 卸載 + 房間 fan-out 可橫向擴展。
  • 讀遠多於寫：一個主播（寫）對應百萬觀眾（讀）→ 設計圍繞讀路徑與 fan-out。
  • 無法回頭：直播是 append-only 的即時流，過期的 segment 直接丟棄，不像 VOD 永久保存。
面試開場必問 「要多低的延遲？」這決定整套技術選型：標準 HLS（~10～30s）夠用就好、互動直播要 LL-HLS（~2～5s）、連麥 / 視訊通話級要 WebRTC（亞秒）。延遲愈低，穩定性與成本愈難 — 先把這條軸講清楚再往下設計。

2. 規模估算

指標假設推算
同時直播數（寫）尖峰 5 萬場同時開播5 萬條攝入流 → 5 萬個即時轉碼工作（GPU/CPU 密集）
同時觀看（讀）熱門單場可達 100 萬同時在線單場峰值出向 = 100萬 × 5Mbps ≈ 5 Tbps → 必須 CDN 卸載
segment 大小2 秒一段、~1Mbps 一檔約 250KB每流每秒產 0.5 個 segment；滑動視窗只留最近 3～6 個
聊天訊息（fan-out）熱門房 1000 訊息/秒、100 萬觀眾fan-out 放大 = 1000 × 100萬 = 10 億 推送/秒 → 分房 + 分層 fan-out
儲存直播本身不存；只留最近視窗近乎 0（除非開錄製轉 VOD，才落物件儲存）
關鍵洞察兩個放大器決定成敗：①影片出向頻寬（Tbps 級 → CDN 卸載 99%+）；②聊天 fan-out 放大（訊息數 × 觀眾數 → 分房 + 階層式廣播）。儲存反而不是瓶頸（直播只留 live edge 附近數秒）。

→ 201 { "stream_id": "live_abc", "stream_key": "sk_xxx",
        "ingest_url": "rtmp://ingest.live.io/app",   # OBS 推這裡
        "playback_url": "https://cdn.live.io/live_abc/master.m3u8" }

# 2) 主播推流（協定層，非 HTTP）：OBS → RTMP/SRT/WebRTC → 攝入伺服器
#    平台收到後即時轉碼 → 切片 → 更新滑動視窗 manifest

# 3) 觀眾取直播 manifest（HLS，內容隨時間滑動）
GET /api/v1/streams/{stream_id}/master.m3u8     → 各碼率子清單
GET /api/v1/streams/{stream_id}/720p/index.m3u8 → 只列「最近 N 個」segment + EXT-X-MEDIA-SEQUENCE

# 4) 聊天（WebSocket / 長連線，這裡用 HTTP 示意）

# 5) 觀看人數 / 加入退出
POST /api/v1/streams/{stream_id}/join   → { "viewers": 12345 }
POST /api/v1/streams/{stream_id}/leave

# 6) 收播（可選觸發錄製轉 VOD）
POST /api/v1/streams/{stream_id}/stop   → { "status": "ENDED", "vod_id": "vid_999"? }

4. 資料模型

streams （直播中繼資料，存 RDB / KV；影片本體不在這）
  stream_id    VARCHAR PRIMARY KEY
  user_id      BIGINT
  title        TEXT
  stream_key   VARCHAR        -- 推流鑑權用（保密）
  status       ENUM(LIVE, ENDED)
  started_at   TIMESTAMP
  viewers      INT            -- 同時在線（高頻 → 走快取 / 近似計數）

# segment 本體 → 物件儲存 / 邊緣記憶體，且「只保留滑動視窗內最近 N 個」，過期即刪
# 直播 manifest（.m3u8）→ 每產一個新 segment 就重算：
#    EXT-X-MEDIA-SEQUENCE++ ，移除最舊一段，加入最新一段（無 ENDLIST）

chat_messages （聊天，append-only；按房間分區）
  stream_id, seq, user, text, ts        -- 真實走 WS 推播 + Kafka，不一定落久存
viewer_presence （在線狀態，TTL 過期自動退出）
  stream_id, viewer_id, last_heartbeat  -- 心跳續命，掉線自動移除
為何直播不像 VOD 永久保存VOD 的 manifest 是完整、固定、可預轉碼的；直播的 manifest 是滑動視窗、不斷變動、邊產生邊丟棄的。直播只需 live edge 附近數秒的 segment 即可服務觀眾，過期 segment 立刻清掉（除非另開錄製）。這就是「無法預轉碼」的根本原因 — 內容此刻才產生。

5. 高階架構

🖼️ 開啟 Excalidraw 架構圖（diagram.excalidraw） 
 主播 ──RTMP/SRT/WebRTC 推流──▶ 攝入(Ingest) ──▶ 即時轉碼(多碼率) ──▶ 打包(HLS 切片)
                                                                      │ 更新滑動視窗 manifest
                                                                      ▼
                                                               CDN 邊緣（快取 segment）
                                                                      │
 觀眾 ◀──就近拉 master.m3u8 → 子清單 → 拉最近 N 個 segment──────────────┘
   │
   ├──聊天──▶ Chat 服務 ──▶ fan-out（分房廣播）──▶ 房內所有觀眾
   └──join/leave──▶ ViewerCounter（同時在線，近似計數）
  1. 攝入（Ingest）：接 RTMP / SRT / WebRTC 單一來源流，做備援（主備攝入點），一斷就切換。
  2. 即時轉碼：邊收邊轉成多碼率（ABR ladder），不能像 VOD 那樣事後批次轉 — 內容此刻才到。
  3. 打包（Packager）：把轉碼後的流切成小 segment（2s），維護滑動視窗 manifest：只列最近 N 段、EXT-X-MEDIA-SEQUENCE 遞增、無 ENDLIST。
  4. CDN 分發：segment 推到邊緣快取，百萬觀眾就近拉 → 卸載 99%+ 出向頻寬；源站只服務回源 miss。
  5. 聊天 fan-out：依 stream_id 分房，訊息廣播給房內連線；熱門房用階層式 fan-out（前端聚合節點扇出）。

6. 核心元件深入

攝入協定：RTMP vs SRT vs WebRTC

  • RTMP：最通用（OBS 預設），TCP-based、相容性最好，但延遲較高、本身已老。
  • SRT：基於 UDP，抗丟包、適合不穩定網路的遠端攝入，延遲可控。
  • WebRTC：亞秒級延遲，適合連麥 / 互動，但擴展到百萬觀眾成本高（通常只用在攝入端與小範圍互動）。

即時轉碼：為什麼不能預轉碼？

  • VOD 的原片完整存在，可離線批次轉所有解析度；直播的內容此刻才產生，只能邊收邊轉。
  • 轉碼是關鍵路徑且即時 → 算力（GPU/CPU）必須跟上輸入速率，跟不上就掉幀 / 卡頓。
  • ABR ladder（多碼率）讓不同頻寬觀眾各取所需，但每多一檔碼率就多一份即時轉碼成本。

滑動視窗 HLS manifest（本題精髓）

  • 直播 media playlist 不含 #EXT-X-ENDLIST（代表「還沒結束，繼續來抓」）。
  • 每產一個新 segment：移除最舊一段、加入最新一段，視窗長度固定（如保留 3～6 段）。
  • #EXT-X-MEDIA-SEQUENCE 標示「目前清單第一段是全域第幾段」，隨舊段滾掉而遞增 — 播放器靠它判斷連續性與是否落後。
  • 觀眾因此只能看到 live edge 附近的內容，無法倒帶到開播時（除非另存 VOD）。
#EXTM3U                          ← 720p/index.m3u8（直播，某一刻）
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:2
#EXT-X-MEDIA-SEQUENCE:104         ← 第一段是全域第 104 段（前 104 段已滾掉）
#EXTINF:2.000,
seg_104.ts
#EXTINF:2.000,
seg_105.ts
#EXTINF:2.000,
seg_106.ts                       ← 沒有 #EXT-X-ENDLIST → 直播持續中
# 下一刻：seg_104 滾掉、加入 seg_107，MEDIA-SEQUENCE 變 105

低延遲手段：LL-HLS

  • 把 segment 再切成更小的 partial segment（如 200ms），邊產邊推，不必等整段 2s 才能播。
  • 用 blocking playlist reload（伺服器 hold 住請求直到新片就緒）省掉輪詢空轉。
  • 代價：manifest 更新更頻繁、實作更複雜、對 CDN 與播放器要求更高。

聊天 fan-out 與觀看人數

  • 分房（room sharding）：依 stream_id 把連線分到不同 chat 節點，一則訊息只廣播給該房連線。
  • 階層式 fan-out：百萬人房間單節點扇不動 → 上游節點扇給多個邊緣聚合節點，再各自扇給觀眾（樹狀放大）。
  • 觀看人數：精確計數成本高 → 用心跳 + TTL 維護 presence，顯示時走近似計數（HyperLogLog / 抽樣），不阻塞播放。

7. 擴展與瓶頸（百萬同時觀看）

階段瓶頸對策
攝入單點主播這條鏈一斷整場黑掉主備攝入點 + SRT 抗丟包 + 自動切換
即時轉碼算力GPU/CPU 跟不上輸入速率 → 卡頓每流獨立轉碼 worker + GPU 加速 + 動態調整 ABR 檔數
出向頻寬單場 Tbps 級、源站撐不住CDN 卸載 99%+，segment 邊緣快取（短 TTL）；源站只服務回源 miss
聊天 fan-out訊息數 × 觀眾數 爆炸放大分房 + 階層式廣播 + 訊息合批 / 限流 / 抽樣顯示
同時在線計數百萬連線精確計數昂貴心跳 + TTL presence + 近似計數（HLL）
全球延遲跨洲 RTT 進關鍵路徑多區域邊緣攝入 + 就近轉碼 + CDN 就近拉流

8. 取捨與面試重點

  • 延遲 vs 穩定（最核心取捨）：低延遲（小 segment、小緩衝）→ 觀眾互動好，但抗網路抖動差、易卡頓；高延遲（大 segment、大緩衝）→ 穩定流暢但互動延遲大。互動直播選低延遲，賽事 / 演唱會可容忍稍高換穩定。
  • HLS vs WebRTC：HLS / LL-HLS 靠 CDN 天然可擴展到百萬觀眾，延遲秒級；WebRTC 亞秒但點對點擴展成本高 → 大型直播用 HLS，連麥 / 小範圍互動用 WebRTC。
  • 直播 vs VOD（對照 ⑩ YouTube）：VOD 可預轉碼、完整 manifest、永久保存、播放可任意 seek；直播邊轉邊切、滑動視窗 manifest、只留 live edge、無法倒帶。同一套 CDN/HLS 基建，但管線方向相反。
  • segment 長度：短段（1～2s）延遲低、切換快，但 manifest 更新頻繁、overhead 高；長段（6～10s）反之。直播偏短段，LL-HLS 再切 partial。
  • 強一致 vs 最終一致：manifest / live edge 需即時推進（落後就追不上）；觀看人數、聊天歷史可最終一致 / 近似。
  • 常見追問：
    • 觀眾網路慢怎麼辦？→ ABR 自動降碼率；落後 live edge 太多就跳回邊緣（追直播）。
    • 主播斷線怎麼辦？→ 攝入備援 + 重連寬限；超時則收播。
    • 百萬人聊天室怎麼扛？→ 分房 + 階層 fan-out + 訊息抽樣 / 限流（不是每則都推給每個人）。
    • 要不要存回放？→ 開錄製把 segment 拼成完整 manifest 落物件儲存 → 退化成 VOD（⑩ 題）。
    • 怎麼降啟播延遲？→ 觀眾加入時直接從 live edge 前數段開始拉，預連線 + 首段預熱。

9. PHP Demo 說明

本題 demo 聚焦低延遲 HLS 滑動視窗 manifest ＋ 聊天 fan-out（面試真正的考點，不是 CRUD），可實際用 PHP 內建伺服器跑起來：

  • src/IngestService.php　開始 / 結束直播；模擬攝入時不斷「產生新 segment」，每段有遞增的全域 sequence（對應主播推流被切片）。
  • src/HlsPackager.php　維護直播 manifest：滑動視窗只保留最近 N 個 segment、EXT-X-MEDIA-SEQUENCE 隨舊段滾掉而遞增、無 ENDLIST（產出 .m3u8 文字）→ 呈現觀眾只能看到 live edge 附近。
  • src/ChatRoom.php　聊天訊息 fan-out 給房間觀眾（依 stream_id 分房）；內含 ViewerCounter 維護同時在線人數。
  • public/index.php　路由：開播 / 產生 segment / 取 manifest / 加入 / 送聊天（fan-out）/ 收播，並附深色測試頁。
# 啟動
cd demo
php -S localhost:8021 -t public
# 瀏覽器開 http://localhost:8021
# 流程：開始直播 → 多次「產生 segment」→ 取直播 manifest（只列最近 N 個、media sequence 遞增、舊的滾掉）
#       → 觀眾加入看人數 → 送聊天訊息 fan-out → 結束直播
誠實聲明 Demo 不做真正的 RTMP 攝入或 ffmpeg 編碼：segment 內容是佔位文字而非真正的 H.264 切片，也沒有真的 CDN。 但滑動視窗 manifest（只留最近 N 段、MEDIA-SEQUENCE 遞增、無 ENDLIST）與聊天 fan-out（分房廣播 + 同時在線計數）是真實可執行的系統邏輯。 真實系統用 RTMP/SRT/WebRTC 攝入、ffmpeg 即時轉碼切片、CDN 全球分發、WebSocket 做聊天推播 — 詳見 demo/README.md。
