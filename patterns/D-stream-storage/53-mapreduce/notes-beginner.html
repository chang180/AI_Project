1. 60 秒搞懂它在幹嘛

想像你有一本一億頁的書，老師要你算「每個英文單字各出現幾次」。你一個人從第一頁數到最後一頁，數到天荒地老都數不完。

聰明的做法：把書撕成一千疊，找一千個同學同時數，每人只數自己那疊。數完再想辦法把大家的結果合起來。這就是 MapReduce——一套「把超大工作切碎、找一大群機器同時做、再把結果合起來」的標準流程。

🔑 一句話 MapReduce = 一大群人一起數錢的標準流程：每人先數自己手上的一疊（map）→ 把同面額的鈔票集中到同一個人那裡（shuffle）→ 每個人把自己那種面額加總（reduce）。一個人做不完的事，一群人分頭做就快了。

2. 為什麼不能「一個人從頭數到尾」？

因為資料大到誇張。Google 要數整個網路有哪些網頁連到哪些網頁、臉書要算每天幾十億則貼文的統計——這種資料一台電腦的硬碟都裝不下，更別說算完。

📌 比喻：全班一起數班費 全班收了一大箱零錢要算總額。老師不會自己一個人數（數到下課也數不完），而是：每排同學先數自己這排收到的，再把同樣面額的硬幣集中給負責那個面額的人加總。 人多、分工、同時做——這就是「分散式運算」的精神。

MapReduce 厲害的地方是：你不用懂怎麼分工。你只要說清楚兩件事——「每一小份要怎麼處理」和「同一類的結果怎麼合起來」——剩下的切片、找機器、分配、合併、誰掛了重做，框架全部幫你搞定。

💭 停一下：為什麼「找一千個人同時數」會比「一個人數」快這麼多？

看答案 因為這些工作彼此不相干——A 數他那疊、B 數他那疊，誰也不用等誰。可以真正同時進行。 只要工作能切成「互不干擾的小份」，人（機器）越多就越快。MapReduce 就是專門用來處理這種「可以拆開、各做各的」的大工作。

3. 把工作拆成 3 個關卡

不管資料多大，MapReduce 永遠就這 3 關，順序固定：

1Map：每個人先數自己手上那一疊

先把大資料切成很多小片（叫 split），每片交給一個工人。工人對自己那片做最簡單的動作：看到一個詞，就記一筆「這個詞 +1」。

💵 比喻：數鈔票的第一步 你拿到一疊雜亂的鈔票，不急著加總，先一張一張喊出面額：「一百、一百、五百、一百、五百…」。每喊一張就是 map 吐出的一筆 (面額, 1)。

重點：每個工人只看自己那片，誰也不管誰，所以可以一千個人同時做。map 的產物是一大堆 (詞, 1) 這種小紙條。

2Shuffle：把同一種的集中到同一個人

現在桌上散落著一大堆 (詞, 1) 小紙條，來自所有工人。下一步：把「同一個詞」的紙條全部收集到同一個人手上。

🗂️ 比喻：把硬幣按面額分籃子 大家數完，桌上一堆「一元、五元、十元」的紀錄混在一起。你拿幾個籃子，所有「十元」丟同一籃、所有「五元」丟另一籃。 這樣負責「十元」的人，才能把全部的十元一次加總，不會漏掉散在別人那邊的。

怎麼決定「哪個詞給哪個人」？用一個固定規則：把詞算個雜湊值，再除以人數取餘數。同一個詞算出來的答案永遠一樣，所以同一個詞一定進同一個籃子——這超重要，不然加總會分散、算錯。

3Reduce：每個人把自己那一種加總

分好籃子後，每個人拿著「分給我的那些詞 + 一堆 1」，做最後動作：把同一個詞的 1 全部加起來。

🧮 比喻：數錢的最後一步 負責十元的人，把手上所有「十元」的紀錄一加：「喔，總共 37 個十元」。負責一百元的人加出「52 張一百」。 每個人各報各的，合起來就是全部的總額——而且因為同面額都在同一個人手上，不會重複也不會漏。

把所有人的結果擺在一起，就是最終答案：the 出現 4 次、fox 出現 3 次、dog 出現 3 次…。一億頁的書，就這樣被一群機器幾分鐘數完了。

✅ 三關一句話 ① Map：每人數自己那疊，吐出 (詞,1) → ② Shuffle：把同一個詞集中到同一個人（按固定規則分籃子）→ ③ Reduce：每人把自己那種加總。就這樣，永遠這三關。

4. 設計時最怕的 3 件事

把抽象的「取捨」換成「最怕三件事」，每個設計都是為了避開某個「怕」。

😱 怕「搬運太慢」（shuffle 塞爆） 第 2 關「把同類集中到同一個人」聽起來簡單，但資料超大時，這是把幾乎全部的紙條跨整個機房搬來搬去——最慢、最花錢的一步。 所以有個聰明招：每個工人數完先自己先小計一下（同一疊裡的「十元」先自己加成「十元 ×30」再交出去），要搬的紙條就少很多。這招叫 combiner。
😱 怕「分配不均」（資料傾斜） 如果某個詞超級熱門（像英文的 "the"），負責它的那個人會拿到幾億張紙條累死，其他人早就下班了——整個工作要等這個最慘的人。 這叫資料傾斜，解法是把超熱門的那種再拆給好幾個人分擔。
😱 怕「有人中途落跑」（機器掛掉） 一千台機器同時跑，總有幾台會當機。好在每個工人做的事互不相干、又沒有副作用，所以誰掛了，就把他那份重發給別人重做一次就好，結果一模一樣，整個工作不受影響。 還有人特別慢（硬碟快壞了），就多找一個人做同一份，誰先做完算誰（叫 speculative，賽馬備援）。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 動手前先估數字，才知道要切幾片、找幾台機器。重點不是背數字，是感受它的「形狀」。
估什麼大概數字所以呢？
資料有多大~1 PB（一百萬 GB）一台裝不下 → 一定要切片、分散
切成幾片（= map 工人數）每片 128 MB → ~800 萬片map 工人數由「資料大小 ÷ 片大小」自動決定
reduce 工人數 (R)由人設定，常數百~數千太少 → 容易塞車、傾斜；太多 → 一堆零碎小檔
shuffle 要搬多少接近 1 PB（跟輸入差不多）這就是最花時間 / 頻寬的一步 → 想辦法用 combiner 壓小
💡 關鍵洞察：最貴的不是「數」，是「搬」 每個工人「數自己那疊」其實很快。真正的瓶頸是中間那步 shuffle——把幾乎全部的資料跨整個機房搬去重新分組。所以省錢省時間的關鍵，就是讓要搬的東西變少（combiner 先小計）。

💭 停一下：map 工人要幾個是「算出來」的，reduce 工人要幾個卻是「人決定」的，為什麼？

看答案 map 是「每片一個工人」，而片數＝資料大小 ÷ 每片大小，所以資料一給定，map 數就定了。 reduce 是「最後分幾籃」，這由你決定：分太少籃，每籃太擠（傾斜、單檔太大）；分太多籃，輸出一堆零碎小檔、管理麻煩。所以 R 要自己權衡。

8. 系統怎麼用？（程式介面）

🍔 比喻：你只要填兩張表 MapReduce 最貼心的地方：你不用會分散式，只要填兩張「表單」告訴它怎麼做，剩下全自動。 
① 第一張表：每一小份怎麼處理？（map）
   給一行字 → 把每個詞吐成 (詞, 1)
   map(一行字):
       對每個詞: 吐出 (詞, 1)

② 第二張表：同一種怎麼合？（reduce）
   給一個詞 + 一堆 1 → 加總
   reduce(詞, [1,1,1,...]):
       吐出 (詞, 總數)

就這兩張表！「怎麼切片、找幾台機器、誰負責哪個詞、誰掛了重做」——你完全不用管，框架自動處理。這就是 MapReduce 把「分散式」這件難事變簡單的魔法。

（可選）第三張表：交出前先自己小計（combiner）—— 通常跟 reduce 一樣
（可選）第四張表：自訂「哪個詞給哪個人」的規則（partitioner）—— 預設就是 hash(詞) % 人數

本題的 demo 把這套包成網頁：你貼一段文字，它就跑一遍 map→shuffle→reduce，把三階段攤開給你看。

9. 資料長什麼樣？（資料模型）

📒 比喻：資料的四個樣貌 從頭到尾，資料會變身四次：

📦輸入：一個超大檔

躺在分散式硬碟（HDFS / S3）上的大檔。框架自動把它切成很多片（split），每片約 128 MB。

🏷️中間產物：一堆 (詞, 1) 小紙條

map 吐出的東西，例：(fox,1) (the,1) (the,1) (dog,1)…。先暫存在工人自己的硬碟，等著被收集。

🗂️分組後：每個 reducer 的籃子

shuffle 把同詞集中後，每個 reducer 手上是：{ fox:[1,1,1], the:[1,1,1,1] }。同一個詞的所有 1 都在這了。

📄輸出：(詞, 總數)

每個 reducer 寫一個結果檔：(fox,3) (the,4)…。全部合起來就是最終答案。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時總有某環節先塞車，工程師的工作就是找出最會塞的、把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
Shuffle：把幾乎全部資料跨機房搬去重新分組（頭號瓶頸）用 combiner 交出前先小計、壓縮中間資料、減少要搬的量
資料傾斜：某熱門詞（如 "the"）把一個 reducer 累死把熱門詞加鹽打散分給好幾個 reducer、自訂分區規則
小檔太多：海量小檔，每個都派一個工人，光排程就累死先把小檔合併成大檔再處理
有人特別慢：最慢的工人拖垮整個工作speculative：多派一個人做同份，誰先完成算誰
反覆讀寫硬碟：每一輪都把中間結果寫回硬碟，迭代慢改用 Spark：中間結果儘量留在記憶體

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 批次(MapReduce/Spark) vs 即時串流(Flink) 批次＝資料整批算、慢但穩、適合「每天結算一次的大報表」；串流＝一筆筆即時算、快但複雜、適合「即時儀表板」。兩者是互補的兩半，不是誰取代誰。
⚖️ MapReduce vs Spark MapReduce 每步都寫硬碟，超穩、適合一次性超大批次；Spark 中間結果留記憶體、快很多，適合要反覆迭代的工作（如機器學習）。新專案多選 Spark，但吃記憶體。
⚖️ combiner 能不能用？ 只有「先合一部分再合起來，答案不變」的計算才能用（加總、找最大可以）。算平均就不行——要改成同時記「總和」和「個數」，最後再相除。
💭 追問：為什麼「同一個詞一定要進同一個 reducer」這麼重要？ 因為 reduce 是「把同詞的數字加總」。如果 "fox" 一部分在 reducer A、一部分在 reducer B，兩邊各加各的，就會得出兩個半截的數字，永遠湊不出正確的總數。靠 hash(詞) % 人數 這個固定規則保證同詞同人，才不會算錯。
💭 追問：一台機器跑到一半當機，為什麼可以直接重做、不怕出錯？ 因為每個 map / reduce 工作都是「給一樣的輸入就吐一樣的輸出、而且不會去動別人」（純函式、沒有副作用）。所以同一份工作重做幾次，結果都一樣，直接重發給別的機器即可，不會汙染整體結果。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那三關到底是哪幾段程式做的」。 好消息：核心剛好三個小檔對應三關，加一個工頭把它們串起來。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三關 Mapper（每人數自己那疊）→ Shuffler（按面額分籃子）→ Reducer（每籃加總）。價值一來，就照這順序跑一遍，Job 這個工頭負責串起來。

1Map：每個詞吐一張 (詞, 1)（Mapper）

第一關超簡單：把一行字斷成一個個詞，每個詞吐一張「這個詞 +1」的紙條。

function map(一片文字) {
    紙條 = []
    對每一行:
        對行裡每個詞:
            紙條.加入( (詞, 1) )       // ← 看到一個詞就記一筆 +1
    return 紙條
}

就這樣。一千個工人各跑各的這段，互不干擾，所以可以同時進行。產物是一大堆 (詞,1)。

2Shuffle：同一個詞丟同一個 reducer（Shuffler）

第二關是全題最關鍵的一行：用固定規則決定「這個詞該丟給哪個 reducer」。

function 該給哪個reducer(詞, 人數R) {
    return hash(詞) % R            // ← 同一個詞算出來永遠一樣 → 一定進同一個 reducer
}

function shuffle(全部紙條, R) {
    籃子 = {}
    對每張 (詞, 1):
        r = 該給哪個reducer(詞, R)
        籃子[r][詞].加入(1)         // 把同詞的 1 收集到同一個籃子
    return 籃子
}

關鍵就是那行 hash(詞) % R：因為同一個詞每次算出的答案都一樣，所以不管哪個 mapper 產生的 "fox"，都會進同一個 reducer。少了這個保證，加總就會分散、算錯。

3Reduce：把同詞的 1 加起來（Reducer）

第三關也很單純：每個 reducer 拿著自己籃子裡的詞，把每個詞的一堆 1 加總。

function reduce(一個籃子) {     // 籃子 = { 詞 => [1,1,1,...] }
    結果 = {}
    對籃子裡每個 (詞 => 數字清單):
        結果[詞] = 數字清單.加總()   // ← (fox,[1,1,1]) → (fox,3)
    return 結果
}

想換成別的工作（算平均、找 Top-N）？只要改這一行加總，map 和 shuffle 完全不用動——這就是 MapReduce 通用的精髓。

4combiner：交出前先自己小計（可選的省錢招）

還記得「最怕搬運太慢」嗎？這就是解法：工人把紙條交出去之前，先自己把同詞小計一下。

// 沒 combiner：吐出 (the,1)(the,1)(the,1)... 一百張要搬
// 有 combiner：自己先加 → 只吐 (the,100) 一張要搬
本地小計 = {}
對我這片的每個詞: 本地小計[詞] += 1
吐出本地小計裡的 (詞, 小計)        // ← 要搬的紙條變超少

demo 裡勾選 combiner，就會看到「中間 (詞,值) 對數」明顯變少——這就是它幫 shuffle 省下的搬運量。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「全班一起數錢」解釋 MapReduce 的三關。 ① Map：每排同學先數自己收到的，喊出一筆筆 (面額,1)；② Shuffle：把同面額的集中給負責那個面額的人（按固定規則分籃子）；③ Reduce：每個人把自己那種面額加總。合起來就是總額。
Q2：為什麼「同一個詞一定要進同一個 reducer」？怎麼保證？ 因為 reduce 要把同詞的數字加總，若同個詞散在不同 reducer，會各加各的、湊不出正確總數。用 hash(詞) % R 這個固定規則保證：同一個詞算出的 reducer 編號永遠一樣，必進同一個。
Q3：三關裡哪一關最花時間？為什麼？怎麼省？ Shuffle 最花時間——它要把幾乎全部的中間資料跨整個機房搬去重新分組。省法：用 combiner 在 map 端「交出前先小計」，把要搬的量壓小。
Q4：什麼是資料傾斜？舉例並說解法。 某個 key 超熱門（如英文 "the"），負責它的 reducer 拿到海量資料累死，其他 reducer 閒著，整個 job 被它拖住。解法：把熱門 key 加鹽打散分給好幾個 reducer，或自訂分區規則。
Q5：一台機器跑到一半當機，框架怎麼處理？為什麼不怕算錯？ 工頭(master)發現它心跳沒了，就把它那份 task 重發給別的機器重做。因為 map/reduce 是「同輸入同輸出、沒副作用」的純函式，重做結果一樣，不會汙染整體。
Q6：批次（MapReduce）和串流（Flink）差在哪？什麼時候用哪個？ 批次＝資料整批一次算、延遲高但穩，適合每天結算的大報表；串流＝一筆筆即時算、延遲低但複雜，適合即時儀表板。兩者互補，常一起用。
Q7：combiner 是什麼？什麼計算不能用它？ combiner＝工人交出前先本地小計，減少 shuffle 搬運量。只有「先合一部分再合起來答案不變」的計算能用（加總、最大值）；算平均不行（要拆成總和+個數，最後再除）。

15. 30 秒回顧

MapReduce = 一大群機器一起數錢的標準流程，把一個人做不完的超大工作切碎、分頭做、再合起來。

3 個關卡：① Map（每人數自己那疊，吐 (詞,1)）→ ② Shuffle（用 hash(詞)%R 把同詞集中到同一個 reducer）→ ③ Reduce（每人把自己那種加總）。

3 個怕：怕搬運太慢（shuffle 是瓶頸 → combiner 先小計）、怕分配不均（資料傾斜 → 熱門 key 打散）、怕有人落跑（機器掛 → 純函式直接重跑、太慢就 speculative 備援）。

工程細節一句話：map 工人數由資料大小自動定、reduce 數 (R) 自己權衡；最貴的是中間的 shuffle（跨機房搬資料）；它是串流的另一半——離線、整批、吞吐優先；進化版是把中間結果留記憶體的 Spark。
