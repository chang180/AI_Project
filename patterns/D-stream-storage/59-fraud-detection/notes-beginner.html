1. 60 秒搞懂它在幹嘛

你刷卡買杯咖啡，一秒內就過了。但如果你的卡號被偷，小偷在 30 秒內連刷 10 筆、還買了一支高價手機——銀行得在每一筆「刷下去的當下」判斷：這筆放行、還是攔下來、還是請人來看一眼？這就是即時詐欺偵測。

聽起來像「比對黑名單」對吧？難的是：壞人不會乖乖待在黑名單裡。系統得在毫秒內、從一堆線索（這張卡最近刷幾筆？金額正常嗎？這 IP 是不是在掃號？）算出一個「風險分數」，再決定怎麼處理——而且千萬別把正常人的卡誤擋，不然客人氣到換銀行。

🔑 一句話 詐欺偵測 = 每筆交易的瞬間保全：用幾條線索算出「這筆有多可疑」的分數 → 分數低就放行、中間就請人工看、太高就擋下。又快又不能冤枉好人。

2. 為什麼難點不是「比對黑名單」？

很多人第一次看這題會想：「不就是把壞卡號列個清單，刷到就擋嗎？」——這想法漏掉了兩個真正的難關。

📌 比喻：機場安檢 安檢不是只比對「通緝犯名單」。多數壞人不在名單上。安檢看的是行為訊號：一個人短時間反覆進出、帶超量現金、神色異常……每個訊號加一點可疑度，加起來夠高才攔下盤查。

詐欺偵測的第一個難關就是這個：壞人多半不在名單上，得靠「行為訊號加總成分數」來抓。其中最有力的訊號是速度——同一張卡短時間內刷太多筆，幾乎一定有鬼（小偷在試卡）。

第二個難關更要命：不能冤枉好人。你出國旅遊刷卡、或雙 11 連下好幾單，行為很像「異常」，但你是真的。擋錯一筆正常交易，輕則客訴、重則客人直接換銀行。所以系統寧可設一個「中間地帶」交給人工看，也不要全部一刀切。

💭 停一下：為什麼「只比對黑名單」抓不到大部分詐欺？

看答案 因為盜刷用的卡號 / IP 多半是第一次出現，根本不在任何名單裡。真正能抓到的是行為——尤其「同一張卡短時間內爆量交易」這種速度訊號。黑名單只是眾多訊號中最確定的那一條，不是全部。

3. 把系統拆成 4 個關卡

別被「系統」嚇到。一筆交易進來，其實就闖 4 關：

1算線索：這筆交易現在「看起來」怎樣？

交易一進來，先算出幾個即時線索。最重要的是速度：這張卡 / 這個 IP 最近一小段時間（例如 60 秒）刷了幾筆？

⏱️ 比喻：看「最近一分鐘」的計步器 想知道一個人是不是在跑步，不看他一整天走幾步，只看「最近一分鐘」動了幾下。 系統也一樣：只盯最近 60 秒這張卡刷了幾筆——這個「最近一小段時間的計數」就叫滑動視窗，是抓盜刷最強的線索。

👉 這一關跟同一組的 ㉒ 廣告點擊聚合是同一套招：那題用「最近一段時間同 IP 點幾次」抓刷廣告，這題用「最近一段時間同卡刷幾筆」抓盜刷。

2評分：把每條線索的可疑度加起來

有了線索，就像安檢累加可疑度：每一條「規則」命中就加一些分，全部加起來＝這筆交易的風險總分。

🎯 比喻：考試扣分制 速度太快？+40 分可疑。金額破天荒大？+30。這張卡在黑名單裡？+80。一個「模型」覺得整體怪怪的？再+一些。 把這些「可疑分」加起來，就知道這筆到底多可疑。

這些規則有兩種來源：人寫的規則（清楚、好解釋，像「金額 > 1000 就加分」）和機器學的模型（抓人想不到的細微壞模式）。兩者的分數加在一起，後面會講它們各自的長短。

3三分決策：放行 / 人工審 / 拒絕

有了總分，最後一步是切三段（不是只有「過」跟「不過」）：

🚦 比喻：紅黃綠燈 分數低 → 綠燈，放行（allow）。 分數中間 → 黃燈，請人工看一眼（review，別急著擋，怕冤枉好人）。 分數很高 → 紅燈，直接擋（deny）。

為什麼要有「黃燈」？因為怕冤枉好人。雙 11 連下好幾單的你，分數可能落在中間——與其直接擋（惹怒你），不如丟給人工確認。這個「黃燈中間地帶」是整個系統調鬆調緊的旋鈕。

4回饋：事後對答案，讓它越來越準

交易處理完還沒結束。過幾天，如果使用者來申訴「這筆不是我刷的」（這叫 chargeback），系統就知道「啊，那筆其實是詐欺」——這就是答案。

📚 比喻：考完對答案 考完試對答案，才知道哪題做錯、下次怎麼改進。系統把這些「事後確認的答案」餵回去重新訓練模型、補新規則、把壞卡加進黑名單——下次就更準。

這個「處理 → 拿到真實結果 → 改進 → 再處理」的循環，就叫回饋迴路。只是答案常常要等好幾天（申訴沒那麼快），所以即時那關還是得靠規則先擋。

✅ 四關一句話 ① 算線索（最近一段時間刷幾筆＝速度）→ ② 評分（每條規則命中加分，加成總分）→ ③ 三分決策（紅黃綠燈：放行/人工審/拒絕）→ ④ 回饋（事後對答案，越來越準）。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「冤枉好人」（誤殺） 這是詐欺系統最致命的怕。擋錯一筆正常交易，客人當場刷不過、超丟臉，回家就換銀行。 所以才要三分決策的「黃燈」把不確定的丟給人工，門檻也偏保守。寧可漏抓一點，也別亂擋好人。
😱 怕「慢」 判斷發生在你刷卡的當下，多等 1 秒體驗就崩。系統只有幾十毫秒把線索查完、規則跑完、分數算完。 所以「最近一段時間刷幾筆」這種線索得事先放在超快的記憶體（Redis），不能臨時去翻大資料庫。慢＝不可用。
😱 怕「線索不新鮮」 小偷在 10 秒內連刷 8 筆，如果系統「剛剛那筆」還沒算進速度計數，第 8 筆來的時候它以為「才刷 1 筆」，就漏了。 所以「剛剛那筆一定要立刻反映」進線索裡——這叫特徵新鮮度。線索過時，等於沒看。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。系統設計也一樣：先估數字，才知道要準備多大的「料」。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每天多少筆交易~2 億筆 ≈ 平均每秒 2 千多筆，尖峰近 1 萬量不算誇張，但每筆都要「即時」判斷
每筆能花多久判斷幾十毫秒（不能拖慢付款）線索要放超快記憶體，不能臨時翻大資料庫
有幾條規則數十到數百條每筆都要全跑完 → 規則本身得很快
多久才知道「答案」申訴/拒付常拖好幾天即時靠規則先擋，模型只能用「確認過」的舊答案再訓練
💡 關鍵洞察：壓力不在「筆數」，在「每筆都要又快又準」 每秒幾千筆聽起來還好。真正難的是每一筆都得在幾十毫秒內查到「最近刷幾筆」這種即時線索、跑完所有規則、還要不誤殺。所以設計重點是「線索放超快記憶體」+「規則跑得飛快」。

💭 停一下：為什麼「最近一段時間刷幾筆」這種線索不能在判斷時才去大資料庫算？

看答案 因為判斷只有幾十毫秒，去大資料庫翻「這張卡過去所有交易再數一遍」太慢了。所以要事先把「最近 60 秒刷幾筆」維護在超快的記憶體（Redis）裡，每來一筆就順手更新，判斷時一查就有。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你給我哪些欄位、我回你什麼。講好格式，雙方就能合作。

這題只要記住幾張「點餐單」：

① 評估一筆交易（最核心、要超快）
   POST /api/v1/evaluate
   給我：{ 交易編號, 卡號, IP, 裝置, 金額, 時間 }
   我回：{ 決策: allow/review/deny, 分數: 62, 觸發了哪些規則:[...] }

回傳要附「觸發了哪些規則」，這樣被擋的人才能解釋為什麼被擋（合規、客訴都需要）。

② 回饋答案（事後對答案）
   POST /api/v1/feedback   給我：{ 交易編號, 答案: 詐欺/正常, 來源: 申訴/人工 }
③ 調門檻（即時生效，不用重啟）
   POST /api/v1/config   給我：{ 同卡速度門檻, review 分界, deny 分界 }
④ 加名單
   POST /api/v1/blacklist   給我：{ 值, 類型: 卡/IP/裝置 }

③ 很重要：詐欺手法天天變，要能當場調鬆調緊，不能每次都重新部署。

9. 系統要記住哪些事？（資料模型）

📒 比喻：幾本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題主要幾本：

⏱️速度本（最近刷幾筆——核心！放超快記憶體）

每個鍵（卡號 / IP）記最近一段時間的交易時間，好隨時回答「最近 60 秒刷幾筆」。 這本放在Redis（超快），而且設自動過期，舊資料自己消失，不會無限長大。

📐規則本（每條規則怎麼加分）

每條規則一列：類型(速度/金額/黑名單/模型)、參數(視窗多久、門檻多少)、加幾分、開不開。可以隨時改，不用重啟。

🚫名單本（確定壞的 / 確定好的）

黑名單（直接擋）和白名單（保護高價值客戶不被誤殺）。命中就是最高優先、最好解釋的判斷。

📑決策本 + 答案本（解釋 + 再訓練）

決策本記「每筆的分數、觸發哪些規則、最後決策」（供申訴解釋）；答案本記「事後確認這筆是詐欺還是正常」（餵回去再訓練）。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
每筆都要查「最近刷幾筆」（讀太多）放 Redis，照卡號/IP 分片分散壓力；同一張卡固定走同一台，保證「剛剛那筆」算得到
規則太多，每筆全跑很慢先跑便宜的規則，能早點下結論就早退；服務多開幾台一起跑
模型推論慢用輕量模型、把線索先準備好、模型服務獨立擴展
線索不新鮮會漏抓同一張卡路由到同一台，保證「先寫進去、再讀出來」
換新模型怕出包影子模式（新模型先只算不擋、跟舊的比）+ 灰度上線 + 可快速回滾
風控服務掛了事先決定 fail-open（放行保營收）還是 fail-close（保守擋保安全）

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 誤殺 vs 漏抓（最核心） 門檻調嚴 → 少漏抓，但多冤枉好人（誤殺）；調鬆 → 不冤枉人，但放過更多詐欺。 「黃燈」人工審就是調這個平衡的旋鈕：不確定的別一刀切，丟給人看。
⚖️ 人寫規則 vs 機器學模型 規則好解釋、改得快（被擋的人問為什麼，你答得出來），但抓不到沒見過的招； 模型準、抓得到細微壞模式，但黑箱、要訓練資料、上線更謹慎。實務是兩個一起用。
⚖️ 即時擋 vs 事後批次抓 即時擋得住「刷卡當下」的盜刷，但只有幾十毫秒；事後可以慢慢用大資料做更重的分析、補抓漏網的。兩者互補，不是二選一。
💭 追問：小偷用 8 個不同卡號、但同一個 IP 連續刷，黑名單和「同卡速度」都抓不到，怎麼辦？ 換個鍵看速度：除了「同卡速度」，再加一條「同 IP 速度」——同一個 IP 短時間內出現一堆不同卡的交易，一樣可疑。這就是為什麼要對多個鍵（卡、IP、裝置）都算速度。
💭 追問：詐欺的「答案」要好幾天才知道，模型怎麼學？ 即時那關靠規則先擋（不用等答案）；模型用的是已經確認過的舊交易（成熟標籤）來再訓練。新手法先用規則補上，等累積夠多答案再進模型。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「速度線索 + 加分 + 三分決策」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8059 -t public
# 然後瀏覽器打開 http://localhost:8059
  1. 先按「送 5 筆正常交易」：不同卡、小額 → 看每筆分數都很低、決策都是 allow。
  2. 再按「同卡連送 8 筆」：同一張卡狂刷 → 看前幾筆還 allow，刷到第 4 筆速度規則被觸發、分數跳高、決策變 review 甚至 deny。
  3. 調門檻：把「同卡速度門檻」調小、或把 deny 分界調低，再跑一次，看決策整體往「擋」的方向移動。
玩的時候想一下同樣 8 筆交易，只是把門檻調一下，決策就從多半放行變成多半攔下——這就是誤殺 vs 漏抓那個旋鈕在你手上轉。回傳裡的 triggered 告訴你「為什麼」被加分，這就是可解釋。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有兩個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 兩個檔 = 兩個小幫手 SlidingWindow（算「最近刷幾筆」的速度計）→ RuleEngine（每條規則加分、加成總分、切三段決策）。交易一來，就照這順序跑一遍。

1速度計：只看「最近一段時間」刷幾筆（SlidingWindow）

每來一筆，先丟掉太舊的紀錄（超出視窗的不算），再把這一筆加進去，剩下的筆數就是現在的速度。

function 記一筆並回傳速度(這個鍵, 現在時間) {
    最近的 = 把舊紀錄裡「還在最近 60 秒內」的留下來   // ← 這就是「滑動」：舊的自動掉出去
    最近的.加入(現在時間)                              // 把這一筆算進去
    記錄[這個鍵] = 最近的
    return 最近的.筆數     // ← 視窗內筆數 = 此刻的速度
}

「鍵」可以是卡號，也可以是 IP。同一招換個鍵就能同時抓「同卡爆量」和「同 IP 爆量」。這段跟 ㉒ 廣告聚合的同 IP 洪水偵測一模一樣，只是這裡把筆數當「線索」交給下一關評分。

2評分大腦：每條規則命中就加分（RuleEngine）

這是全題的心臟。把幾條規則一條條檢查，命中就加上各自的分數，順便記下「觸發了哪條」好解釋：

分數 = 0;  觸發清單 = []
卡速度 = 速度計.記一筆並回傳速度("卡:"+卡號, 時間)   // 線索①
IP速度 = 速度計.記一筆並回傳速度("IP:"+IP,  時間)   // 線索②

if (卡速度 > 同卡門檻)   { 分數 += 40; 觸發清單.加("velocity_card") }   // 同卡刷太快
if (IP速度 > 同IP門檻)   { 分數 += 25; 觸發清單.加("velocity_ip")   }   // 同IP刷太快
if (金額   > 金額門檻)   { 分數 += 30; 觸發清單.加("amount")        }   // 大額
if (卡號或IP在黑名單)    { 分數 += 80; 觸發清單.加("blacklist")     }   // 確定壞的
分數 += 模型示意分(金額, 速度)                                          // 模型再補一點

每條規則就一行 if，命中加分。分數的大小（40/25/30/80）就是「這條訊號有多重」。黑名單加最多（80），因為它最確定。

3切三段：紅黃綠燈（RuleEngine 的最後一步）

分數算完，用兩條分界線切成三段——就是前面講的三分決策：

if (分數 >= deny門檻)        decision = "deny"     // 紅燈：擋
else if (分數 >= review門檻) decision = "review"   // 黃燈：人工看
else                        decision = "allow"    // 綠燈：放行

兩條門檻 review門檻、deny門檻 就是你在 demo 裡能調的旋鈕。調低 → 更多交易往「擋」走（少漏抓、多誤殺）；調高 → 反之。誤殺 vs 漏抓的平衡，全在這兩個數字。

💭 停一下：為什麼要「黃燈 review」這一段，不直接「過 / 擋」二選一？

看答案 因為怕冤枉好人。很多交易分數落在「有點可疑但不確定」的中間，直接擋會誤殺正常客戶，直接放又怕漏。丟給人工審是最安全的折衷——這就是三分決策比二分決策聰明的地方。
🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 算線索（速度）SlidingWindow記一筆並回傳速度()
② 評分（規則加分）RuleEngine一串 if … 分數 +=
③ 三分決策RuleEngine兩條門檻切 allow/review/deny
④ 回饋（真實系統）事後標籤回流再訓練（demo 以調門檻示意）

👉 重點：真實系統會把「JSON 檔 + 迴圈」換成「Redis 速度計 + 線上模型服務 + Kafka 串流」， 但上面這幾段判斷邏輯一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。 唯一誠實聲明：demo 裡的「模型分」是加權示意（金額/速度越高分越高），不是真的訓練出來的 ML 模型；速度計和規則引擎都是真的。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「機場安檢」解釋，這題為什麼不能只靠黑名單？ 因為大部分壞人不在通緝名單上。要靠行為訊號加總：反覆進出、帶超量現金……每個加一點可疑度，加起來夠高才攔。對應到交易就是「速度太快、金額異常」等線索加成風險分，黑名單只是其中最確定的一條。
Q2：抓盜刷最強的線索是什麼？為什麼？ 速度——同一張卡（或同 IP）在最近一小段時間內刷了幾筆。盜刷常是短時間連續試卡/爆刷，所以「最近 60 秒筆數暴增」是極強訊號。靠滑動視窗計數即時算出。
Q3：為什麼要「三分決策」（放行/人工審/拒絕），不直接二選一？ 因為怕誤殺。中間「有點可疑但不確定」的交易，直接擋會冤枉好人、直接放又怕漏，丟給人工審最安全。這個中間地帶也是調「誤殺 vs 漏抓」平衡的旋鈕。
Q4：為什麼「最近刷幾筆」這種線索要放 Redis，不在判斷時去大資料庫算？ 因為判斷只有幾十毫秒。去大資料庫翻整段歷史再數太慢。所以事先把「最近 60 秒筆數」維護在超快記憶體（Redis）並設自動過期，判斷時一查就有。這也保住了低延遲。
Q5（工程）：「特徵新鮮度」是什麼？不顧它會怎樣？ 指「剛剛那筆」要立刻反映進速度計數。若沒做到，小偷 10 秒連刷 8 筆時，第 8 筆來時系統以為「才刷 1 筆」就漏抓。所以同一張卡要路由到同一台、保證先寫後讀。
Q6（工程）：規則和 ML 模型各自的長短？為什麼兩個一起用？ 規則好解釋、改得快，但抓不到沒見過的招；模型準、抓得到細微壞模式，但黑箱、要訓練資料。一起用：規則擋明確壞模式 + 模型補細緻分數，分數加總。
Q7（工程）：詐欺的「答案」要好幾天才知道，這對系統有什麼影響？ 即時那關不能等答案，只能靠規則先擋；模型只能用「已經確認過的舊交易」（成熟標籤）再訓練。新手法先用規則補，累積夠多答案後再進模型。

15. 30 秒回顧

即時詐欺偵測 = 每筆交易的瞬間保全：算分數，紅黃綠燈決定放行/人工審/拒絕。

4 個關卡：① 算線索（最近一段時間刷幾筆＝速度，跟 ㉒ 同一招）→ ② 評分（每條規則命中加分，加成總分）→ ③ 三分決策（紅黃綠燈）→ ④ 回饋（事後對答案越來越準）。

3 個怕：怕冤枉好人（誤殺最致命 → 三分決策的黃燈）、怕慢（線索放 Redis、幾十毫秒搞定）、怕線索不新鮮（剛剛那筆要立刻反映）。

工程細節一句話：壓力不在筆數、在「每筆都要又快又準」→ 速度線索放超快記憶體並分片、規則跑得飛快、模型補細緻分數 → 兩條門檻切三段決策（調它就是調誤殺 vs 漏抓）→ 事後答案回流再訓練形成閉環。
