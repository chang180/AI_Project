⑳ Dropbox / Google Drive 檔案同步：分塊去重與增量同步

D 串流計算/儲存分發 建議 46 分鐘 Design a File Sync / Storage Service

一句話定位 本質是「內容定址的分塊儲存 + 增量同步 + 多裝置一致性」的綜合題。 把整個檔案上傳很簡單；面試真正考的是 只同步「變更的那一小塊」、相同內容只存一份， 以及多裝置同時改同一檔案時如何偵測並解決衝突。

1. 需求釐清

功能性需求

  • 使用者在任一裝置新增 / 修改 / 刪除檔案，其他裝置自動同步到最新狀態。
  • 增量同步（delta sync）：改一個大檔的一小段，只上傳變更的區塊，而非整檔重傳。
  • 去重（dedup）：相同內容的區塊在全系統只存一份（同使用者、甚至跨使用者）。
  • 多裝置同步通知：某裝置上傳新版本後，其他在線裝置即時收到「該拉取」的通知。
  • 斷點續傳：上傳大檔中途斷線，重連後從未完成的區塊接續，不從頭來。
  • 版本歷史與還原；衝突時保留兩方資料（不靜默覆蓋）。

非功能性需求

  • 耐久性最高優先：使用者資料不可遺失（11 個 9 等級），物件儲存多副本 / EC 編碼。
  • 頻寬效率：同步成本應正比於「變更量」而非「檔案大小」。
  • 最終一致：各裝置最終收斂到相同狀態；同步延遲秒級可接受。
  • 可擴展：億級檔案、PB 級資料、千萬裝置長連線。
  • 讀寫不對稱：metadata 操作頻繁且小，blob 傳輸量大但相對低頻。
面試開場必問 「同步的最小單位是整檔還是區塊？」決定整題走向。整檔同步 = 簡單但頻寬爆炸； 區塊級同步（分塊 + 去重 + delta）才是 Dropbox 的精髓，也是這題唯一值得深入的方向。

2. 規模估算

指標假設推算
使用者5 億註冊、5000 萬日活平均每人多裝置（手機 + 電腦）
檔案數每人 5000 檔~2.5 兆檔案 → metadata 必須分片
分塊大小固定 4 MB / chunk1 GB 檔 ≈ 256 個 chunk

去重效益跨使用者重複率 30%~50%實體儲存可省下近半 → 直接省成本

關鍵洞察儲存與頻寬是主要成本 → 設計圍繞「內容定址去重」省儲存、「delta 增量」省頻寬、「metadata 與 blob 分離」讓兩者各自擴展。

3. API 設計

# 1) 上傳前先問：哪些 chunk 伺服器已經有了？（去重的關鍵一步）
POST /api/v1/files/{fileId}/chunks:check
Body: { "hashes": ["sha256:ab12...", "sha256:cd34...", ...] }
→ 200 { "missing": ["sha256:cd34..."] }   # 只有 missing 的需要上傳

# 2) 上傳缺少的 chunk（內容定址，可平行、可斷點續傳）
PUT /api/v1/chunks/{hash}          Body: <binary chunk>
→ 201（已存在則 200，不重複存 = 去重）

# 3) 提交新版本：送出有序 chunk 清單 + 基底版本號
POST /api/v1/files/{fileId}/commit
Body: { "baseVersion": 7, "chunks": ["sha256:ab12...", "sha256:cd34..."] }
→ 200 { "version": 8 }
→ 409 Conflict { "serverVersion": 9, "conflictCopy": "report (裝置B 的衝突副本).docx" }

# 4) 下載：先取 metadata（有序 hash 清單），再依需要抓缺少的 chunk
GET /api/v1/files/{fileId}/metadata → { "version": 8, "chunks": [...] }
GET /api/v1/chunks/{hash}           → <binary chunk>

# 5) 變更通知（長連線 / 推播）：其他裝置收到「該拉取」訊號
WS  /api/v1/notify   ← { "event": "changed", "fileId": "...", "version": 8 }
為何先 check 再 PUT「先問缺哪些、只傳缺的」是去重與省頻寬的核心動作；伺服器以 chunk hash 為主鍵，PUT 同一 hash 第二次是 no-op。

4. 資料模型

-- metadata DB（關聯式、可分片，存「指標」不存內容）
files
  file_id      BIGINT PRIMARY KEY
  owner_id     BIGINT
  path         TEXT
  latest_ver   INT
  is_deleted   BOOL

file_versions（不可變、append-only，支援版本歷史與還原）
  file_id, version  PRIMARY KEY
  chunk_list   JSONB        -- 有序的 chunk hash 清單（檔案 = chunk 的序列）
  size         BIGINT
  device_id    TEXT         -- 哪個裝置提交的
  vector_clock JSONB        -- 版本向量，用於衝突偵測
  created_at   TIMESTAMP

chunks（內容定址索引；blob 本體在物件儲存）
  hash         CHAR(64) PRIMARY KEY   -- sha256(內容)，相同內容 → 同 key → 去重
  size         INT
  ref_count    BIGINT                 -- 被幾個版本引用，歸零才可 GC
  blob_url     TEXT                   -- 指向 S3 / GCS 的物件位置

-- blob 本體：物件儲存（S3 / GCS），key = chunk hash，與 metadata 完全分離
為何 metadata 與 blob 分離metadata 小、操作頻繁、需交易與索引 → 放分片式關聯庫；blob 大、不可變、需極高耐久 → 放物件儲存。兩者特性相反，分開才能各自最佳化與擴展。

5. 高階架構

🖼️ 開啟 Excalidraw 架構圖（diagram.excalidraw） 
          ┌──────────┐  ①check ②PUT chunk ③commit
 寫(上傳)──▶│ 同步服務  │──▶ 分塊/去重 ──▶ Blob 物件儲存(S3)
          └──────────┘        │                 ▲
                              ▼                 │ ④拉取缺少的 chunk
                        Metadata DB(分片) ───────┘
                              │
                              ▼ 版本提交事件
                        變更通知服務(長連線) ──▶ 其他裝置(讀/同步=綠)
 讀(同步)──▶ 其他裝置 ──取 metadata──▶ Metadata DB ──抓 chunk──▶ Blob 儲存
  1. 同步服務：接收 chunk check / 上傳 / commit；無狀態，可水平擴展。
  2. 分塊 / 去重：客戶端先把檔案切塊算 hash；伺服器以 hash 判重，相同 chunk 只落地一次。
  3. Blob 物件儲存：存不可變 chunk 本體，key 即 hash，多副本 / EC 保證耐久。
  4. Metadata DB：存檔案 → 版本 → 有序 chunk 清單；commit 是交易邊界。
  5. 變更通知服務：版本提交後推播給該使用者其他在線裝置，告知「該拉取」。

6. 核心元件深入

檔案分塊：固定大小 vs 內容定義分塊（CDC）

  • 固定大小分塊（如每 4 MB）：實作簡單，但在檔案開頭插入 1 byte 會讓後面所有 chunk 邊界位移，導致幾乎全部重傳（boundary shift 問題）。
  • 內容定義分塊（CDC，如 Rabin 滾動雜湊）：依內容特徵決定切點，插入 / 刪除只影響鄰近 1~2 個 chunk，其餘 chunk 的 hash 不變 → 去重與 delta 效果遠優於固定分塊。面試講出 CDC 是加分點。

內容定址儲存與去重

  • chunk 的 key = sha256(內容)。相同內容 → 相同 hash → 同一 key → 天然去重。
  • 上傳前用 chunks:check 問伺服器缺哪些，只傳 missing → 同時省頻寬與儲存。
  • ref_count（引用計數）：一個 chunk 可能被多版本 / 多檔引用；刪除版本時遞減，歸零才能垃圾回收，否則會誤刪別人還在用的內容。

增量同步（delta sync）

客戶端保存「上次同步的 chunk 清單」。再次上傳時，把新版本切塊，新清單 − 舊清單 = 新出現的 chunk，只上傳這個差集；commit 時送完整有序清單讓伺服器組裝新版本。改大檔的一小段，頻寬成本只正比於那一小段。

衝突解決（版本向量 / 衝突副本）

  • commit 時帶 baseVersion（或版本向量）。若伺服器當前版本 > baseVersion，表示有人搶先提交 → 衝突。
  • 策略：不靜默覆蓋。較晚者另存為「衝突副本（conflicted copy）」，如 report (裝置B 的衝突副本).docx，兩份都保留交由使用者處理。
  • 版本向量可區分「真衝突」與「單純落後」：若一方因果上包含另一方，則只是落後，直接快轉即可，無須產生副本。

多裝置同步通知與斷點續傳

  • 通知：裝置與通知服務維持長連線（WebSocket / long-poll）；某裝置 commit 後，服務推「changed + version」給同帳號其他裝置，避免輪詢浪費。
  • 斷點續傳：上傳以 chunk 為單位且內容定址，斷線重連後重跑 chunks:check，已上傳的 chunk 會被回報為「已存在」，只續傳剩餘的，天然支援續傳。

7. 擴展與瓶頸（億級檔案）

階段瓶頸對策
單庫 metadata兆級檔案裝不下、寫熱點依 owner_id / file_id 分片；同帳號資料同片以利交易
blob 容量 / 耐久PB~EB 級、不可遺失物件儲存多副本 / EC 編碼 + 跨區複寫；冷資料降溫到低成本層
頻寬成本大檔整傳塞滿上行CDC 分塊 + delta 同步 + 去重，成本只正比於變更量
通知扇出千萬長連線、熱帳號扇出長連線閘道分層；以使用者為單位扇出，事件僅帶「該拉取」訊號
chunk GC誤刪共享 chunkref_count 引用計數 + 延遲回收（標記後寬限期再刪）

8. 取捨與面試重點

  • 固定分塊 vs CDC：固定簡單但怕邊界位移；CDC 抗位移、去重佳但需滾動雜湊、運算稍重。大檔頻繁小改的場景 CDC 完勝。
  • 客戶端分塊 vs 伺服器分塊：客戶端切塊可在上傳前就 check + 去重，省上行頻寬（Dropbox 採此）；伺服器切塊則客戶端簡單但得先把整檔傳上去，失去頻寬效益。
  • 強一致 vs 最終一致：同一檔的版本提交在單一 metadata 分片內可做交易（強一致，用於衝突偵測）；裝置間同步是最終一致（靠通知 + 拉取收斂）。
  • 跨使用者去重的隱憂：可省大量儲存，但帶來「存在性側信道」（猜中 hash 即可秒傳，洩漏他人是否擁有某檔）與加密衝突；隱私敏感系統常只在同使用者內去重。
  • 常見追問：
    • 大檔上傳到一半斷線怎辦？→ chunk 級內容定址，重跑 check 只續傳缺的。
    • 兩裝置離線各自改同檔，上線後？→ 版本向量判真衝突 → 產生 conflicted copy，不覆蓋。
    • 刪檔後 chunk 何時真正釋放？→ ref_count 歸零 + 寬限期後 GC。
    • 如何端到端加密又能去重？→ 收斂加密（convergent encryption），但有已知側信道取捨。

9. PHP Demo 說明

本題 demo 聚焦分塊去重 + 增量同步 + 衝突偵測三大核心（而非 CRUD），可用 PHP 內建伺服器跑起來：

  • src/Chunker.php　把內容切成固定大小 chunk，逐 chunk 算 sha256（以位元組切，對二進位內容亦然）。
  • src/BlobStore.php　內容定址儲存：以 hash 為 key，相同 chunk 只存一份，統計去重節省的位元組。
  • src/FileIndex.php　檔案 → 有序 chunk hash 清單 + 版本號（模擬 metadata DB 的 file_versions）。
  • src/SyncService.php　上傳新版本時比對舊清單只上傳差集（delta）；兩裝置基於同版本各自修改 → 偵測衝突並產生 conflicted copy。
  • public/index.php　路由 + 深色測試頁，可直接貼入文字內容當作「檔案」操作。
# 啟動
cd demo
php -S localhost:8020 -t public
# 瀏覽器開 http://localhost:8020
# 1) 貼入內容上傳 → 看切成幾個 chunk、各 hash
# 2) 小改內容再上傳 → 只有變更的 chunk 被新增（顯示 delta 與去重率）
# 3) 模擬兩裝置基於同版本並發修改 → 偵測衝突，產生衝突副本
誠實聲明 為了在本機零依賴展示，Demo 用固定大小分塊（真實 Dropbox 用 CDC 內容定義分塊以抗邊界位移），以 JSON 檔模擬 metadata DB、以本機檔案模擬 blob 物件儲存。 本機 PHP 未載入 mbstring，故以位元組（byte）切塊——這其實與真實系統一致：分塊面對的是二進位內容，本就不該以字元為單位。 分塊、sha256 內容定址、去重統計、delta 差集、版本向量衝突偵測等系統邏輯為真實可執行。
