1. 60 秒搞懂它在幹嘛

你在公司電腦改了一份報告，回到家打開筆電，發現報告自動就是最新版了——你什麼都沒做。 這就是 Dropbox（或 Google Drive、iCloud）在幫你做的事：讓你每一台裝置上的檔案，永遠長一樣。

聽起來很簡單對吧？「不就是把檔案傳上雲端，再傳下來」。但魔鬼藏在三個地方：你的檔案可能幾百 GB、 你只改了一行字、而且你可能兩台電腦同時在改同一份檔。把這三件事做好，才是這題真正的難點。

🔑 一句話 Dropbox = 把檔案切成小塊的聰明同步：相同的塊只存一份（省空間）、只傳改動的塊（省頻寬）、 兩邊同時改就兩份都留（不偷偷蓋掉你的心血）。

2. 為什麼不能「整個檔案重傳」就好？

很多人第一個念頭是：「檔案改了，就把整個檔重新傳上去不就好了？」我們用一個你天天在做的事來看為什麼不行——

📄 寄一份 300 頁的報告 你寫了一份 300 頁的報告寄給同事。後來你只改了第 5 頁的一個錯字。 這時你會把整本 300 頁重新印、重新寄一次嗎？太蠢了。你只會把改過的那一頁，重寄給他就好。

檔案同步一模一樣。一個 1 GB 的影片你只剪掉幾秒，若每次都重傳整個 1 GB，你的網路會被塞爆，雲端也存了一堆幾乎一樣的東西。

所以這題的核心精神就是：「不要重來、只處理有變的那一小部分」。為了做到這件事，系統先把每個檔案切成很多小塊 （像把報告拆成一頁一頁），之後所有的聰明操作，都是建立在「以小塊為單位」之上。

💭 停一下：如果系統是「以整個檔案」為單位，而不是「以小塊」為單位，前面說的「只寄改動那頁」還辦得到嗎？

看答案 辦不到。如果單位是「整個檔案」，那系統眼中只有「檔案變了 / 沒變」兩種狀態，一變就只能整檔重傳。 只有先把檔案切成小塊，系統才看得見「哪幾塊變了、哪幾塊沒變」，才有辦法只傳那幾塊。切塊是這整題的地基。

3. 把系統拆成 4 個關卡

別被「系統」嚇到。整件事其實就是闖 4 關，每一關都對應一個你已經懂的生活直覺：

1分塊：把檔案切成一頁一頁

第一步，把檔案切成固定大小的小塊（例如每塊 4 MB）。一個 1 GB 的檔案就被切成大約 256 塊。

📚 比喻就像把一本厚厚的書拆成一頁一頁。拆開後，每一頁都可以單獨被檢查、單獨被處理——這是後面所有魔法的前提。

每一塊都會算出一個「指紋」（一段叫 sha256 的代碼）。內容一樣的塊，指紋就一樣；內容只要差一個字，指紋就完全不同。記住這個「指紋」，下一關馬上要用。

2去重：一樣的塊，只存一份

切好塊、算好指紋後，系統會發現：很多塊的內容其實一模一樣。可能是同一份檔案的重複段落，也可能是不同人都存了同一個熱門影片。

✍️ 比喻：相同段落不重抄 抄筆記時，如果有一段話跟前面寫過的完全一樣，你不會笨笨地再抄一遍，你會寫「同上」。 系統也一樣：看到指紋已經存過的塊，就不再存第二份，只記一筆「我用的是那塊」。

這招叫 去重（dedup）。因為「指紋一樣 = 內容一樣」，系統上傳前會先問雲端：「這幾個指紋你有沒有？」 雲端只回「我缺這幾塊」，你就只傳缺的。光這一招，就能省下接近一半的儲存空間。

3增量同步：只寄改動的那幾頁（delta）

這一關就是把「寄報告只寄改動頁」變成真的。檔案改過後，系統把新版本重新切塊，跟上次的塊清單比一比， 找出哪幾塊是新的，就只傳這幾塊。

📑 比喻：改報告只寄修改頁 300 頁報告只改了第 5 頁 → 第 5 頁的指紋變了，其他 299 頁的指紋都沒變。 系統一比對就知道「只有第 5 頁是新的」，於是只傳第 5 頁。頻寬成本只跟「你改了多少」有關，跟「檔案多大」無關。

這招叫 增量同步（delta sync）。改 1 GB 大檔的一小段，可能只傳幾 MB。

4衝突處理：兩邊同時改，兩份都留

最棘手的一關。你在公司電腦改了報告，同事在家裡也改了同一份（兩人都還沒同步到對方的版本）。 現在雲端收到兩個都說「我是基於版本 7 改的」的新版本——該聽誰的？

🍴 比喻：別偷偷把對方的菜倒掉 兩個人各自煮了一鍋湯要放進同一個鍋位。系統不會偷偷把先到的那鍋倒掉， 而是兩鍋都留下來，其中一鍋標上「（衝突副本）」，讓你自己決定要哪個、還是兩個都要。

這叫衝突副本（conflicted copy）。系統靠每個版本附帶的「我是基於哪一版改的」資訊， 判斷出「有人搶先提交了」，於是把較晚那份另存一份，例如 report (裝置B 的衝突副本).docx。 核心原則：寧可留兩份讓你煩，也絕不偷偷蓋掉你的心血。

✅ 四關一句話 ① 切塊（拆成一頁頁）→ ② 去重（一樣的塊只存一份）→ ③ 增量同步（只傳改動的塊）→ ④ 衝突就留兩份。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「弄丟你的檔案」 照片、論文一旦弄丟就回不來了，這是最不可原諒的事。所以耐久性壓倒一切：每塊內容都存好幾份在不同地方， 機房失火都不能掉。這就是為什麼承諾「資料幾乎不可能遺失」（11 個 9 的等級）。可以慢，不能丟。
😱 怕「偷偷蓋掉你的心血」 兩邊同時改，如果系統自作主張只留一份、把另一份默默丟掉，使用者會崩潰（「我寫一整晚的東西不見了！」）。 所以衝突時寧可留兩份讓你自己選，絕不靜默覆蓋。寧可多一份，不可少一份。
😱 怕「把網路與硬碟塞爆」 如果每次小改都重傳整檔、每份重複內容都存一遍，頻寬和儲存成本會爆炸到付不起。 所以才要分塊 + 去重 + 增量同步，讓成本只正比於「真正變動的量」，而不是「檔案的大小」。只處理有變的，別重來。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
有多少使用者5 億註冊、5000 萬天天用每人多裝置（手機 + 電腦）要同步
總共多少檔案每人 5000 檔 → 約 2.5 兆檔帳本一台機器裝不下，必須分到很多台
要存多少資料每人 50 GB → 共約 25 EB天文數字，所以「省空間」是錢的問題
去重能省多少不同人重複率約 30%~50%實體空間可省近一半 → 直接省一半的錢
💰 關鍵洞察：成本花在「存」和「傳」 這系統最燒錢的地方，是存那 25 EB 的資料和傳輸的網路頻寬。 所以整個設計都圍著省這兩樣轉：用去重省儲存、用增量同步省頻寬。省下來的，全是真金白銀。

💭 停一下：為什麼「去重」對 Dropbox 來說，不只是技術問題，更是「錢」的問題？

看答案 因為要存的資料是 EB 等級的天文數字，存愈多花愈多錢。去重讓「不同人存的同一個熱門影片只存一份」， 重複率 30%~50% 意味著實體儲存能省下近一半——這直接砍掉接近一半的硬碟成本。在這種規模下，省空間 = 省錢。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住三張「點餐單」，剛好對應前面的關卡：

① 上傳前先問：這些塊你有沒有？（去重的關鍵一步）
   POST /files/{檔案}/chunks:check
   給我：{ 一串塊的指紋 }
   我回：{ 我缺這幾塊 } ← 你只要傳缺的就好

這就是「同上、不重抄」的精神：先問清楚雲端缺什麼，只傳缺的，同時省下頻寬和儲存。

② 上傳缺的塊（用「指紋」當地址）
   PUT /chunks/{指紋}   Body: 塊的內容
   → 已經有了就「不重複存」（這就是去重）

為什麼用指紋當地址？因為內容一樣指紋就一樣，同一個指紋傳第二次系統直接當「已有」忽略——天然去重，還順便支援斷點續傳（斷線後重問一次缺哪些，只補沒傳完的）。

③ 提交新版本：送出「這版是哪串塊 + 我基於第幾版改的」
   POST /files/{檔案}/commit
   給我：{ 基底版本=7, 塊清單=[指紋1, 指紋2, ...] }
   我回：{ 新版本=8 }
   或回：409 衝突 { 伺服器已是第9版, 幫你存了「衝突副本」 }

關鍵在那個「基底版本=7」：如果雲端發現自己已經是第 9 版（有人搶先提交了），就知道發生衝突，於是不硬蓋，改成幫你存一份衝突副本。

9. 系統要記住哪些事？（資料模型）

📒 比喻：帳本與倉庫分家 系統把資料分成兩種東西：一本薄薄的「帳本」（記每個檔案有哪些版本、每版是哪串塊）， 和一座巨大的「倉庫」（存塊的內容本體）。這兩樣分開放，是這題最重要的設計之一。

📒帳本（Metadata）：記檔案的目錄與版本

每個檔案記：檔案ID、擁有者、路徑、最新版本；每個版本記：這版是「哪串塊（有順序的指紋清單）」、誰提交的、基於哪版。

為什麼帳本只存「指紋清單」不存內容？ 因為一個檔案 = 一串塊照順序拼起來。帳本只要記「第 1 塊用指紋A、第 2 塊用指紋B……」這串目錄， 要還原檔案時照著清單去倉庫一塊塊拿來拼就好。帳本小、查得快；內容大、放倉庫。

📦倉庫（Blob）：存塊的內容本體

每塊記：指紋（當地址）、大小、被幾個版本用著(ref_count)、實際存在哪。塊的內容本體放在「物件儲存」，每塊存好幾份保命。

為什麼要記「被幾個版本用著」(ref_count)？ 同一塊可能被很多檔案、很多版本共用（去重的結果）。所以刪檔時不能馬上把塊清掉， 要先看「還有沒有別人在用」——這個計數歸零了，才能安全回收，否則會誤刪別人還在用的內容。
為什麼帳本與倉庫一定要分家？ 帳本小、改得頻繁、要能精準查詢（適合放關聯式資料庫）；倉庫大、塊一旦存了就不再改、要超級耐久（適合放物件儲存）。 兩者個性相反，分開放才能各自用最適合的方式擴展與最佳化。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
帳本太大：兆級檔案一台機器裝不下依擁有者把帳本拆到很多台機器（分片）；同帳號資料放同一台好交易
倉庫要存 EB 級、還不能弄丟每塊存好幾份在不同地點；很少用的冷資料搬到便宜倉庫
大檔整傳塞滿上行頻寬分塊 + 去重 + 增量同步，成本只正比於變動量
千萬裝置都掛著等通知通知用長連線而非一直輪詢；以使用者為單位推「該拉取」訊號
刪檔時誤刪了還有人在用的共用塊用引用計數，歸零再加一段寬限期才真的清掉

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心。這裡補幾個常被面試官追問的：

⚖️ 怎麼切塊？固定大小 vs 看內容切 固定大小切（每 4 MB 一刀）最簡單，但在檔案開頭插一個字，後面所有刀口全位移，導致幾乎整檔都變「新塊」要重傳。 進階作法是看內容特徵決定切點（CDC），插字只影響鄰近一兩塊，去重效果好很多。（面試講得出 CDC 是加分點。）
⚖️ 跨使用者去重：省錢 vs 隱私 讓「不同人的相同塊也只存一份」最省空間，但有隱私隱憂：別人只要猜中指紋就能「秒傳」，等於洩漏「某人是否擁有某檔」。 所以隱私敏感的系統，常只在同一個使用者內去重。
⚖️ 強一致 vs 最終一致 同一檔的版本提交，在單一帳本分片內可以做到強一致（用來精準偵測衝突）；但裝置之間的同步是最終一致—— 靠「通知 + 各自拉取」慢慢收斂，延遲個幾秒可以接受。
💭 追問：大檔上傳到一半斷線了，怎麼辦？ 因為上傳是以塊為單位、用指紋當地址，重連後只要重跑一次 chunks:check， 已經傳上去的塊會被回報為「已存在」，系統就只續傳剩下沒傳完的塊，不用從頭來。這就是斷點續傳，是分塊的天然好處。
💭 追問：兩台裝置都離線、各自改了同一檔，上線後會怎樣？ 兩邊都帶著「我基於第 7 版改的」上線。系統用版本向量判斷出這是真衝突（不是單純某邊落後）， 於是不覆蓋，把較晚那份另存成「衝突副本」，兩份都保留給使用者決定。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「分塊 + 去重 + 增量 + 衝突」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8020 -t public
# 然後瀏覽器打開 http://localhost:8020
  1. 貼一段文字當作「檔案」上傳 → 看它被切成幾塊、每塊的指紋。
  2. 小改一下內容再上傳 → 觀察：只有變動的塊被當成「新塊」（顯示 delta 差集與去重率）。
  3. 模擬兩台裝置基於同一版各自修改 → 看系統偵測到衝突、產生衝突副本。
玩的時候想一下同一段內容上傳兩次，第二次是不是「沒有新增任何塊」？這就是去重。 只改一個字再傳，是不是只多了一兩塊、其他塊都沿用？這就是增量同步。兩邊基於同版各改一次，是不是兩份都被留下？這就是衝突副本。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有四個小檔，剛好對應前面講的四關。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 四個檔 = 四個小幫手 Chunker（把檔案切塊、算指紋的切紙機）→ BlobStore（倉庫：一樣的塊只收一份）→ FileIndex（帳本：記每版是哪串塊）→ SyncService（總管：串起去重、增量、衝突）。一次上傳，就照這個順序跑一遍。

1分塊 + 指紋：把報告拆成一頁一頁（Chunker）

第一關，把檔案切成固定大小的小塊，並幫每一塊算一個指紋（hash）。指紋就是後面去重、比對的鑰匙。

function 切塊(內容) {
    塊清單 = []
    一塊一塊往後切（每塊固定大小，例如 16 個位元組）:
        這塊 = 內容的這一段
        塊清單.加入({ 指紋: 算指紋(這塊), 內容: 這塊 })
    return 塊清單
}
function 算指紋(資料) {
    return "sha256:" + sha256(資料)   // ← 內容一樣 → 指紋一定一樣
}

關鍵就是算指紋那一行：相同內容必得相同指紋。這個「同內容 → 同指紋」的特性，正是下一關「指紋一樣就不再存」能成立的地基。

2去重：指紋一樣就不再存（BlobStore）

要把一塊存進倉庫前，先看「這個指紋我有沒有？」。有了就不再存第二份，只把「被幾個人用著」的計數加一。這就是「相同段落寫『同上』」。

function 存一塊(指紋, 資料) {
    if (倉庫裡已經有這個指紋) {
        引用計數[指紋] += 1     // ← 同上！不重抄，只記「我也用這塊」
        累計省下的空間 += 資料大小
        return 去重命中          // 沒有真的多存
    }
    倉庫[指紋] = 資料            // 第一次見到，才真的落地一份
    引用計數[指紋] = 1
    return 新存了一塊
}

就這個 if：指紋一樣 = 內容一樣 → 直接跳過不存。光這招就能省下接近一半的硬碟。引用計數 是為了刪檔時知道「這塊還有沒有別人在用」，歸零才能安全清掉。

3增量同步：只算出缺的那幾塊（SyncService）

上傳一個檔案，不是整檔硬塞，而是先切塊算指紋，再一個一個問倉庫「這塊你有嗎？」，挑出缺的（missing）才傳。這就是「改報告只寄修改頁」。

function 上傳(檔案, 內容, 裝置, 基底版本) {
    塊清單 = 切塊(內容)                    // ← 第 1 關

    缺的塊 = []
    for 每一塊 in 塊清單:
        if (倉庫沒有這塊的指紋)
            缺的塊.加入(這塊)             // ← delta：只挑出新出現的塊

    // ……（先做衝突偵測，見第 4 關）……

    只把「缺的塊」存進倉庫             // ← 省頻寬 + 去重
    沿用的舊塊只把引用計數 +1
    新版本 = 帳本.提交(檔案, 塊清單的指紋, 裝置)
    return { 共幾塊, 缺幾塊, 沿用幾塊 }   // 沿用越多 = 省越多
}

重點是那個 if (倉庫沒有這塊)：只有沒見過的塊才進 缺的塊。改一個字的大檔，只有變動那一兩塊算「缺」，其餘整批沿用——傳輸量只跟「你改了多少」有關。

💭 停一下：為什麼是「一塊一塊問倉庫缺不缺」，而不是「比對上一版的塊清單」？兩種想法會得到一樣的結果嗎？

看答案 結果幾乎一樣，但「問倉庫」更強：它連別人傳過的相同塊也算「已有」（跨檔去重），不只是「跟我上一版比」。所以 demo 直接問倉庫——沒見過才傳，自然同時做到「只傳改動」和「跨檔去重」兩件事。

4衝突副本：兩邊都改，留兩份（SyncService）

提交新版本前，先看一句「我基於的版本，是不是已經落後了？」如果有人搶先提交過，就不硬蓋，把較晚這份另存成衝突副本，兩份都留。

伺服器版本 = 帳本.最新版本(檔案)
有衝突 = (伺服器版本 > 0 且 基底版本 < 伺服器版本)
//        ↑ 我以為是第 7 版，但伺服器已經是第 9 版 → 有人搶先了

if (有衝突) {
    // 不偷偷覆蓋！另存一份，名字標清楚是誰的衝突副本
    衝突檔名 = 檔案 + "（" + 裝置 + " 的衝突副本）"
    存缺的塊；帳本.提交(衝突檔名, 塊清單, 裝置)
    return { 狀態: "conflict", 衝突副本: 衝突檔名 }   // ← 兩份都在
}
// 沒衝突，才正常提交成新版本

核心就那行 基底版本 < 伺服器版本：一旦成立就代表「我改的時候，別人已經先改過了」。系統絕不靜默覆蓋，寧可留兩份讓你自己選，這就是report（裝置B 的衝突副本）.docx 的由來。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 切塊 + 算指紋Chunker切塊() + 算指紋()
② 去重（同上不重抄）BlobStore存一塊() 的那個 if
③ 增量同步（只傳改動）SyncService挑出 缺的塊 的迴圈
④ 衝突就留兩份SyncService基底版本 < 伺服器版本 的判斷

👉 重點：真實系統會把「JSON 檔 + 一塊塊問」換成內容定址儲存（S3/物件儲存）+ 分片帳本 + Redis， 切塊也會從「固定大小」升級成「看內容切（CDC）」，但分塊去重、delta 增量、衝突留兩份這幾段判斷邏輯一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「寄 300 頁報告」解釋，為什麼不能每次都整檔重傳？ 報告只改了第 5 頁，沒人會把整本 300 頁重印重寄，只會重寄改過那一頁。檔案同步一樣：先把檔案切成小塊，只傳變動的塊，頻寬成本只跟「改了多少」有關，跟「檔案多大」無關。
Q2：什麼是「去重」？用「抄筆記」的比喻說說看。 抄筆記遇到跟前面一模一樣的段落，會寫「同上」而不重抄。系統靠每塊的「指紋」判斷內容是否相同，指紋已存在的塊就不再存第二份，只記一筆「我用那塊」。相同內容只存一份。
Q3：增量同步（delta）是怎麼做到「只傳改動」的？ 新版本重新切塊後，跟上次的塊清單比對，只有指紋變了的塊才是「新塊」，系統就只傳這些新塊。沒變的塊指紋一樣，雲端早就有了，不必再傳。
Q4：兩台裝置同時改同一檔，系統為什麼「不直接覆蓋」？ 因為偷偷蓋掉會讓使用者弄丟心血（最不可原諒的事）。所以系統靠「我基於哪一版改的」判斷出衝突，把較晚那份另存成「衝突副本」，兩份都留，讓使用者自己決定。
Q5（工程）：上傳前為什麼要先做 chunks:check「問雲端缺哪些」？ 因為很多塊雲端可能已經有（同檔重複段落、或別人傳過的相同內容）。先問清楚只傳缺的（missing），同時省下頻寬和儲存——這是去重與省頻寬的核心動作。
Q6（工程）：為什麼「帳本（metadata）」和「倉庫（blob）」要分開放？ 帳本小、改得頻繁、要能精準查詢，適合放關聯式資料庫；倉庫大、塊存了就不再改、要超級耐久，適合放物件儲存。兩者個性相反，分家才能各自用最適合的方式擴展。
Q7（工程）：刪檔時為什麼不能馬上把它的塊清掉？怎麼處理？ 因為去重的關係，同一塊可能被很多檔案、很多版本共用。要用「引用計數（ref_count）」記「還有幾個人在用」，歸零後再加一段寬限期才真的回收，否則會誤刪別人還在用的內容。

15. 30 秒回顧

Dropbox 同步 = 把檔案切成小塊的聰明同步。

4 個關卡：① 切塊（拆成一頁頁）→ ② 去重（相同塊只存一份，像「同上不重抄」）→ ③ 增量同步（只傳改動的塊，像「改報告只寄修改頁」）→ ④ 衝突就留兩份（不偷偷覆蓋）。

3 個怕：怕弄丟檔（多副本保命）、怕偷偷蓋掉心血（衝突留兩份）、怕塞爆網路與硬碟（只處理變動量）。

工程細節一句話：成本花在「存」與「傳」→ 用「點餐單(API)」先 check 再傳，只傳缺的塊 → 帳本(metadata)與倉庫(blob)分家、各自擴展 → 引用計數管何時能安全清塊。
