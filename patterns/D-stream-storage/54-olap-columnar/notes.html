54 OLAP / 列式儲存：分析查詢為何只掃幾欄就夠

D 串流計算/儲存分發 建議 45 分鐘 Design an OLAP / Columnar Store

🛠️ 完整版（你在這） 🌱 初學版（看故事就懂）
🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把「列式 vs 行式」「只掃需要欄」講清楚，邊讀邊自我測驗。
一句話定位 本質是「把同一欄的值連續存在一起，讓分析查詢只掃用到的幾欄、其他欄一個 byte 都不讀」。 OLTP（交易）一次處理一筆完整的列；OLAP（分析）一次掃大量列、但只取幾個欄做聚合——儲存方向相反，這就是列式儲存存在的理由。

1. 需求釐清

先分清楚兩種工作負載

OLTP（線上交易）OLAP（線上分析）
典型查詢「取出 id=42 這筆訂單的全部欄位」「更新這筆」「過去 90 天各地區的營收總和」
觸碰的列少數幾列掃大量列（百萬～數十億）
觸碰的欄整列所有欄只用其中幾個欄
讀寫大量逐筆讀寫、要強一致大量讀、批次寫、可最終一致
最佳儲存行式（row-store）列式（column-store）

功能性需求

  • 支援分析型聚合查詢：SELECT SUM/AVG/COUNT/MIN/MAX(欄) ... GROUP BY 欄。
  • 掃大量歷史列，但每個查詢通常只用到少數欄。
  • 批次入庫（ETL / CDC 把 OLTP 的列灌進來，轉成列式）。

非功能性需求

  • 大掃描吞吐高：聚合要在秒級～分鐘級掃完數十億列。
  • I/O 最小化：只讀用到的欄；用壓縮再砍掉一輪 I/O。
  • 可水平擴展：MPP（大規模平行處理），多節點分頭掃。
  • 不要求逐筆即時更新（這交給 OLTP）。
面試開場必問 「這是分析型還是交易型負載？」決定走列式還是行式。一句話：逐筆讀寫整列 → 行式；掃海量列只取幾欄聚合 → 列式。

2. 規模估算

指標假設推算
事實表列數每日 5 億筆事件，存 1 年~1800 億列
欄數寬表 ~50 欄單一聚合查詢常只用 2～4 欄
單欄原始大小1800 億 × 8 bytes~1.4 TB / 欄（未壓縮）
整表大小50 欄 × 1.4 TB~70 TB（行式得全讀）
列式只掃 2 欄2 / 50I/O 降為 ~1/25，再乘壓縮可達 1/100+
關鍵洞察分析查詢的成本主要是掃描的 I/O。列式的兩大省法：① 只掃用到的欄（欄裁剪）② 同欄好壓（壓縮後 byte 更少）。兩者相乘，常見 10～100× 的差距。

3. 查詢介面

# 批次入庫（ETL / CDC）
POST /api/load     Body: { rows: 2000 }     # 列資料 → 拆成各欄連續陣列

# 分析聚合查詢
GET /api/query?select=amount&groupby=region&agg=sum
→ {
    "groups": [ { "key":"TW", "value":1237761.7, "rows":500 }, ... ],
    "scan":  { "columns_scanned":2, "total_columns":4,
               "columnar_cells":4000, "rowstore_cells":8000,
               "rowstore_saving_x":2.0 }      # 行式要多掃幾倍
  }

# 狀態 + 各欄壓縮統計
GET /api/state → { row_count, schema, columns:[{encoding, ratio}], overall_ratio }

真實系統用 SQL（ClickHouse / BigQuery）；介面本質都是「選哪幾欄 + 怎麼聚合 + 怎麼分組」，引擎據此只讀需要的欄。

4. 資料模型（列式 vs 行式）

同一份資料（4 欄）：
  ts        region  device   amount
  1700000001 TW     ios      120.50
  1700000002 JP     web      88.00
  1700000003 TW     android  301.20

行式（OLTP，每「列」連續存）：
  [1700000001,"TW","ios",120.50][1700000002,"JP","web",88.00][...]
  ↑ 要算 SUM(amount) 也得把 ts/region/device 一起搬進來

列式（OLAP，每「欄」連續存）：
  ts     = [1700000001, 1700000002, 1700000003, ...]   ← delta 編碼
  region = ["TW","JP","TW", ...]                        ← 字典編碼
  device = ["ios","web","android", ...]                 ← 字典編碼
  amount = [120.50, 88.00, 301.20, ...]
  ↑ SUM(amount) GROUP BY region 只讀 amount + region 兩個陣列
為何這樣存 分析查詢的存取型態是「沿著欄縱向掃」而非「沿著列橫向取」。把同欄放一起 → 一次連續讀就拿到整欄、不夾雜無關欄；又因同欄型別一致、值分佈相近 → 壓縮率極高。

5. 高階架構（資料倉儲管線）

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 OLTP 來源(行式) ──ETL/CDC(行轉列)──▶ 列式儲存(每欄一連續陣列, 壓縮+分區)
                                              │
                            向量化查詢引擎 ◀──┘  只讀用到的欄
                                  │            (欄裁剪 + min/max 跳塊 + MPP)
                                  ▼
                              BI / 報表(聚合結果)
  1. ETL / CDC 入庫：把交易庫的列抽出、清洗，行轉列寫入倉儲（批次或近即時 CDC）。
  2. 列式儲存：每欄獨立檔／區塊，套欄壓縮，按時間/鍵分區並記每塊的 min/max。
  3. 向量化查詢引擎：解析查詢 → 只挑需要的欄 → 用 min/max 跳過不可能命中的塊 → 對整欄批次運算 → 多節點 MPP 平行掃。

代表系統：ClickHouse、BigQuery、Snowflake、Redshift、Druid、Apache Parquet/ORC。

6. 核心元件深入

① 只掃需要的欄（欄裁剪，column pruning）

列式的第一性原理：欄各自連續存，所以「不在 SELECT/GROUP BY/WHERE 出現的欄」完全不必讀。寬表 50 欄、查詢用 2 欄 → I/O 立刻砍到 4%。行式做不到——它的最小讀取單位是「整列」。

② 行式 vs 列式 I/O

掃描格子數（列 × 欄）：
  列式：row_count × 用到的欄數          (本 demo: 2000 × 2 = 4000)
  行式：row_count × 全部欄數            (本 demo: 2000 × 4 = 8000)
  → 行式得多掃 (全部欄 / 用到欄) 倍；欄越多差距越大

③ 欄壓縮（同欄值相近 → 超好壓）

  • 字典編碼（Dictionary）：低基數字串（region、device）抽字典，欄改存小整數索引。例 ["TW","TW","JP"]→字典["TW","JP"]+[0,0,1]。
  • RLE（Run-Length）：連續重複值存成 (值, 次數)，排序後的欄效果驚人。
  • Delta：單調遞增欄（時間戳、自增 id）只存差值，差值小、好打包。
  • 之上再疊 LZ4 / ZSTD 通用壓縮二次壓。

④ 向量化執行（vectorized execution）

對整欄陣列做批次運算（一次處理一批值），而非「一列一列、逐筆 if」。連續記憶體對 CPU cache 友善、可用 SIMD，吞吐遠勝逐列。

⑤ 分區與 min/max 跳塊（zone map / data skipping）

把欄切成塊，每塊記 min/max。查 WHERE ts > X 時，min/max 不可能命中的塊整塊跳過，連讀都不讀——欄裁剪之外的第二層省 I/O。

⑥ MPP 平行

資料按分區散到多節點，每節點掃自己那份做局部聚合，最後合併。掃描量水平攤開 → 近線性加速。

7. 擴展與瓶頸

瓶頸成因對策
大掃描慢聚合要掃數十億列欄裁剪 + 壓縮 + min/max 跳塊 + MPP 平行
JOIN 昂貴大表 JOIN 要重分佈資料反正規化成寬表、預聚合/物化視圖、廣播小維表
即時性差批次 ETL 有延遲近即時 CDC、流式入庫（如 ClickHouse MergeTree、Druid 即時段）
逐筆更新貴改一欄一值要重寫整塊列式不做逐筆 update；用 append + 背景合併/重寫（LSM 式）
高基數 GROUP BY分組鍵太多、雜湊表爆記憶體分區聚合 + 溢寫磁碟 + 近似（HyperLogLog 估 distinct）

8. 取捨與面試重點

  • 列式利分析、不利逐筆操作：取一整列要把每個欄各讀一次再拼回來；逐筆更新要重寫塊。所以列式不拿來當 OLTP 主庫。
  • OLTP 與 OLAP 分離：交易走行式庫（強一致、低延遲寫），分析另建列式倉儲，中間靠 ETL/CDC 同步。這是業界標準分工，也是這題最重要的一句結論。
  • 批次 ETL 延遲 vs 即時：批次簡單但資料有滯後；要近即時就上 CDC/流式，複雜度與成本上升。
  • 寬表反正規化 vs JOIN：列式偏好把資料攤成寬表避免 JOIN，代價是冗餘與寫入放大。
  • 常見追問：為什麼列式壓縮率高？（同欄型別一致、值相近）為什麼分析不該打 OLTP 庫？（大掃描拖垮交易、且行式掃整列浪費 I/O）count(distinct) 怎麼省？（HyperLogLog 近似）。

9. PHP Demo 說明

本題 demo 著重列式儲存的系統核心邏輯，可實際用 PHP 內建伺服器跑起來：

  • src/ColumnStore.php　列式儲存：每欄一個連續陣列；入庫時把列拆成欄；查詢只抓需要的欄。
  • src/Compressor.php　欄壓縮：字典（低基數字串）／RLE（連續重複）／Delta（單調時間戳），真實算出壓縮率。
  • src/QueryEngine.php　向量化聚合 + 掃描量對比（列式只掃 N 欄 vs 行式被迫讀整列全部欄）。
  • public/index.php　路由：載入 / 查詢 / 狀態，附測試頁。
# 啟動
cd demo
php -S localhost:8054 -t public
# 瀏覽器開 http://localhost:8054 → 載入樣本 → 看 SUM ... GROUP BY 的「掃描欄數 vs 行式」對比
誠實聲明 Demo 用記憶體 + JSON 檔模擬列式儲存（非磁碟上的 Parquet 格式）；但「只掃用到的欄、欄壓縮率、向量化聚合、列式 vs 行式掃描量對比」皆為真實計算。真實系統會再加上磁碟列式檔、LZ4/ZSTD、min/max 跳塊、MPP——查詢邏輯一致，只是把儲存層換成記憶體陣列。
