1. 60 秒搞懂它在幹嘛

想像一張超大的試算表：幾十億列（每列一筆交易），幾十個欄（時間、地區、裝置、金額……）。 老闆問你：「過去一年，每個地區的總營收是多少？」

你只需要兩個欄：地區和金額。其他幾十欄你看都不用看。 問題是——如果資料是「一列一列」存的，你要拿到「金額」這一欄，就得把每一列整列搬出來，連你根本不用的時間、裝置都一起搬。幾十億列 × 幾十欄，全搬一遍，慢死。

🔑 一句話 列式儲存 = 把試算表「直著存」：同一欄的值收在一起。 這樣老闆只問地區和金額時，你只翻這兩欄、其他欄碰都不碰——這就是 OLAP（分析型）資料庫飛快的祕密。

2. 為什麼難點不是「把資料存起來」？

存資料誰都會。難的是：同一份資料，要怎麼「擺」，分析才會快？

📌 比喻：倉庫怎麼放貨 一間倉庫有兩種放法。放法 A（行式）：每個「訂單」的所有東西打包成一箱（鞋＋襪＋帽放一起）。 要找「全部的鞋」做盤點時，你得每一箱都打開翻出鞋——其他東西全被你翻一遍，白費力。 放法 B（列式）：所有鞋放一櫃、所有襪放一櫃。盤點鞋？只開鞋櫃，一次拿完，其他櫃看都不看。

分析查詢（像盤點）的特性就是：掃超多列，但每次只關心少數幾欄。所以「同欄放一起」（放法 B）才對。 反過來，如果你的工作是「把第 42 號訂單整筆叫出來、改一下」（這叫 OLTP，交易型），那「一箱一訂單」（放法 A）反而比較快——一次就拿到整筆。

💭 停一下：為什麼「算各地區營收」時，用「一列一列存」會很浪費？

看答案 因為一列一列存時，最小的讀取單位是「一整列」。你只想要「金額」和「地區」兩欄，卻被迫把每一列的所有欄都搬進記憶體，再丟掉用不到的。幾十億列全搬一遍，I/O 全浪費在你根本不看的欄上。列式（直著存）就沒這問題：只讀那兩欄。

3. 把概念拆成 4 個關卡

列式儲存為什麼快又省，其實就靠 4 招，一招一招看：

1只開需要的櫃子（只掃用到的欄）

查詢用到哪幾欄，就只讀哪幾欄，其他欄一個 byte 都不碰。這是列式最核心的省法。

🗄️ 比喻：只開鞋櫃 盤點鞋只開鞋櫃。一張 50 欄的表，查詢只用 2 欄 → 你只讀了 2/50 ≈ 4% 的資料就得到答案。欄越多、用到的越少，省得越誇張。

2同櫃東西長得像，超好打包（欄壓縮）

同一欄的值型別一樣、長得像（地區欄就那幾個國家、時間欄一直往上加）。長得像的東西特別好壓縮。

📦 比喻：整理行李 把襪子全捲一起、衣服全疊一起，比「鞋襪衣混一袋」好壓縮得多。三個常用招：
  • 字典編碼：地區只有 TW/JP/US/SG 幾種 → 編號成 0/1/2/3，欄改存小數字。
  • RLE（連續重複）：TW,TW,TW,JP,JP → 記成「TW×3, JP×2」。
  • Delta（只記差距）：時間 1000,1004,1009 → 記成 1000,+4,+5，數字小好存。

壓得越小 = 要從硬碟搬的 byte 越少 = 又快一輪。

3一整櫃一起算（向量化）

因為同欄收在一起、排得整整齊齊，電腦可以一次抓一大批一起算，而不是「一列一列、慢慢加」。

🧮 比喻：用整箱秤，不用一顆一顆秤 要算一整櫃蘋果多重，把整箱搬上磅秤一次秤完，比「一顆一顆拿起來秤再加」快太多。電腦對「排好的連續資料」批次運算，又快又省力。

4明顯不對的整櫃直接跳過（min/max 跳塊）

每一櫃（資料塊）門口貼張小卡：「這櫃最小～最大值」。查「金額 > 1000」時，門口寫「最大才 500」的櫃連開都不開。

🏷️ 比喻：看門口標籤決定要不要進去翻 找「2024 年的資料」，門口貼「這櫃全是 2019 年」的，你掃一眼標籤就走人，不浪費時間進去翻。這叫「跳塊」。
✅ 四關一句話 ① 只開需要的櫃（只掃用到的欄）→ ② 同櫃好打包（欄壓縮）→ ③ 整櫃一起算（向量化）→ ④ 明顯不對的整櫃跳過（min/max 跳塊）。就靠這四招，分析快上 10～100 倍。

4. 設計時最怕的 3 件事

😱 怕「用錯地方」 列式擅長「掃海量列、只取幾欄聚合」（分析）。但你若拿它做「取出第 42 號訂單整筆」或「改一筆」（交易），反而慢——要把每個欄各讀一次拼回整列，改一格還得重寫一整塊。 所以鐵律：交易用行式庫、分析用列式庫，兩邊分開。
😱 怕「資料不夠新」 分析庫的資料通常是從交易庫搬過來的（叫 ETL／CDC）。搬得不夠勤，報表就有滯後（昨天的資料今天才到）。 要更即時就得上「近即時同步」，但複雜度和成本都上升。新鮮度 vs 簡單，要選。
😱 怕「JOIN 拖垮」 分析常要把好幾張大表拼起來（JOIN），這很貴。 對策是事先攤成一張寬表（把要用的欄都放進來，雖然有點冗餘）或預先算好結果（物化視圖），查的時候就不用臨時拼。用空間換時間。

工程術語白話解釋
OLAP（分析型）「掃海量列、只取幾欄做加總平均」的工作，像盤點、做報表
OLTP（交易型）「取一筆、改一筆」的日常操作，像下訂單、改地址
列式儲存 / 行式儲存同一欄收一起（直著存）／同一列收一起（橫著存）
欄裁剪（只掃用到的欄）查詢用哪幾欄就只讀哪幾欄，其他不碰
字典編碼少數幾種值（地區）→ 編號成小數字存，省空間
RLE（連續重複編碼）TW,TW,TW → 記成「TW×3」
Delta 編碼只記跟上一筆的差距（時間戳超好用）
向量化一次抓一大批一起算，不是一筆一筆慢慢算
min/max 跳塊（zone map）每櫃門口貼最小最大值，明顯不對的整櫃跳過
ETL / CDC把交易庫的資料搬進分析庫（順手行轉列）
MPP（大規模平行）把資料分給很多台機器，各掃各的再合併
寬表 / 反正規化事先把要用的欄都塞進一張表，避免臨時 JOIN
掃描量這次查詢總共要讀多少格資料（列 × 欄），越少越快

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 先估數字，才知道列式到底省多少。重點是感受形狀，數字不用背。
估什麼大概數字所以呢？
有多少列每天 5 億筆，存一年 ≈ 1800 億列掃全表行式會死，列式只掃需要的欄
有多少欄寬表約 50 欄但一個查詢通常只用 2～4 欄
只掃 2 / 50 欄讀取量降到 ~4%I/O 直接砍到 1/25
再乘上壓縮同欄好壓，再砍幾倍合起來常見快 10～100 倍
💡 關鍵洞察：成本＝你「讀了多少資料」 分析查詢慢，幾乎都慢在「從硬碟搬資料」。列式同時從兩個方向砍搬運量：只搬用到的欄 ×（再）壓得更小。兩個一相乘，就是它海電行式的原因。

💭 停一下：同樣一個「算各地區營收」，列式和行式差在哪一個數字？

看答案 差在「讀了幾個欄」。列式只讀 2 欄（地區、金額）；行式被迫讀整列全部 50 欄。同樣的列數，行式要多搬 25 倍的資料才得到一模一樣的答案。這就是 demo 裡那個「掃描量對比」要給你看的數字。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是講好的「點餐單格式」：你說要哪幾欄、怎麼加總、怎麼分組，引擎照辦。 
① 把資料灌進來（行轉列入庫）
   POST /api/load   給我：{ 要產生幾筆 }
   我做：把一列一列拆成一欄一欄存好

② 跑分析查詢（最重要）
   GET /api/query?select=amount&groupby=region&agg=sum
   白話：把「金額」加總，照「地區」分組
   我回：各地區的營收 + 一個「掃描量對比」（這次只掃幾欄 vs 行式要掃幾欄）

注意第 ② 張回傳裡有個 scan 區塊，直接告訴你「列式掃了 4000 格、行式要掃 8000 格、行式多掃 2 倍」——這就是這題的考點被做成可以親眼看到的數字。

③ 看狀態 + 壓縮效果
   GET /api/state
   我回：目前幾列、每欄用什麼編碼壓、壓縮率多少

9. 系統怎麼擺資料？（資料模型）

📒 比喻：同一張表，兩種擺法 重點不是「存什麼欄」，而是「怎麼擺」。同一份 4 欄資料： 
行式（OLTP，一列一箱）：
  [時間1,地區1,裝置1,金額1] [時間2,地區2,裝置2,金額2] ...
  → 算金額也得把整箱搬出來

列式（OLAP，一欄一櫃）：
  時間櫃 = [時間1, 時間2, 時間3, ...]    ← 用 Delta 壓
  地區櫃 = [地區1, 地區2, 地區3, ...]    ← 用字典壓
  裝置櫃 = [裝置1, 裝置2, 裝置3, ...]
  金額櫃 = [金額1, 金額2, 金額3, ...]
  → 算「各地區營收」只開地區櫃 + 金額櫃

🗄️每欄一個「櫃子」（連續陣列）

列式的全部祕密就在這：把同一欄的值連續收在一起。其他三招（只開需要的櫃、壓縮、跳塊）全都是建立在這個擺法上才做得到。

🏷️每個櫃再貼上「最小～最大」標籤

把長長的欄切成一塊塊，每塊記 min/max。查條件時，標籤明顯不符的整塊跳過，連讀都省了。

10. 哪裡會塞車、怎麼拓寬？（擴展）

哪裡會塞車怎麼拓寬
掃太多資料慢只開需要的欄 + 壓縮 + min/max 跳塊 + 多機平行(MPP)
大表 JOIN 很貴事先攤成寬表 / 預先算好結果（物化視圖）
資料不夠新從「每天搬一次」改成「近即時同步(CDC)」
想逐筆改資料列式不擅長改 → 改成「只追加、背景再合併重寫」
分組鍵太多吃爆記憶體分區聚合 + 必要時用「估算」(如算大概有幾種)

11. 還有哪些「魚與熊掌」？（取捨）

⚖️ 列式 vs 行式 列式利「掃海量、取幾欄」（分析）；行式利「取一筆、改一筆」（交易）。所以業界兩個都用：交易走行式，分析另開列式倉儲，中間搬資料同步。
⚖️ 即時 vs 簡單 每天批次搬最簡單，但資料有滯後；要即時就上 CDC，複雜也貴。看業務多需要「最新」。
⚖️ 寬表(冗餘) vs JOIN 把欄都塞進一張寬表 → 查得快但占空間、寫得重；保持分表 → 省空間但查時要 JOIN。
💭 追問：為什麼「同一欄」特別好壓縮？ 因為同一欄的值型別一致、分佈相近（地區就那幾國、時間一直遞增）。長得像的資料壓縮率天生就高。行式一塊裡混著數字、字串、時間，雜亂難壓。
💭 追問：為什麼不該拿分析查詢直接打交易庫？ 兩個原因：① 一個大掃描會吃光資源，拖垮正在收訂單的交易庫；② 交易庫是行式，掃整列拿一欄很浪費。所以分析要分到專門的列式倉儲去跑。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看「那四招到底是哪幾段程式做的」。核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，看註解就懂。

🔧 三個檔 = 三個小幫手 ColumnStore（把資料一欄一櫃擺好）→ Compressor（把每櫃打包壓縮）→ QueryEngine（只開需要的櫃、整櫃一起算，還順手算出「比行式省幾倍」）。

1一欄一櫃擺好：行轉列（ColumnStore）

入庫時，把「一列一列」的資料拆開，同一欄的值丟進同一個櫃（陣列）。

function 入庫(一堆列) {
    foreach (每一列 as 列) {
        欄櫃["時間"][] = 列.時間      // 時間全收進時間櫃
        欄櫃["地區"][] = 列.地區      // 地區全收進地區櫃
        欄櫃["金額"][] = 列.金額      // 金額全收進金額櫃
        ...
    }
}

就這個迴圈，把「橫著的列」翻成「直著的欄」。之後要某一欄，直接拿那個櫃就好，完全不碰別的櫃。

2打包壓縮：同櫃長得像（Compressor）

每個櫃挑最適合的壓法。字串櫃用字典、時間櫃用 Delta：

// 字典：少數幾種值 → 編號
function 字典壓(地區櫃) {
    字典 = 不重複的值清單           // ["TW","JP","US","SG"]
    欄改存 = 每個值換成它在字典的編號  // [0,1,0,2,...] ← 變成小數字，超省
}

// Delta：只記跟上一筆的差距（時間戳超好用）
function delta壓(時間櫃) {
    存 = [第一筆, 之後每筆只記「比上一筆多多少」]  // 1000,+4,+5...
}

同一櫃的值型別一致、長得像，所以壓縮率很高。demo 會真的算出每欄壓縮前後的 bytes 給你看。

3只開需要的櫃 + 整櫃一起算（QueryEngine）

這是全題的心臟。先算出「這個查詢到底用到哪幾欄」，只抓那幾櫃，然後一次跑完：

function 查詢(要加總的欄, 分組欄, 怎麼加) {
    用到的欄 = [分組欄, 要加總的欄]      // ← 只有這幾欄會被讀
    分組櫃 = 拿(分組欄)                  // 只開這兩櫃
    數值櫃 = 拿(要加總的欄)              // 其他櫃碰都不碰

    foreach (每一列 i) {                 // 向量化：在連續陣列上一路掃
        key = 分組櫃[i]
        累加[key] += 數值櫃[i]           // 邊掃邊加總
    }
    return 累加
}

關鍵就是 用到的欄 = [分組欄, 要加總的欄] 這一行——查詢只會去抓這幾櫃，其他幾十櫃一個 byte 都不讀。這就是「只掃需要欄」的程式長相。

4順手算出「比行式省幾倍」（QueryEngine）

最後它還幫你把考點算成數字：列式只掃幾格、行式要掃幾格。

列式掃的格子 = 列數 × 用到的欄數        // 例：2000 × 2 = 4000
行式掃的格子 = 列數 × 全部欄數          // 例：2000 × 4 = 8000（被迫讀整列）
行式多掃 = 行式 / 列式                  // = 2 倍

這段不是真的去搬資料，而是把「如果用行式會被迫多讀多少」算給你看。欄越多、查詢用到的越少，這個倍數就越大——這正是面試要你講清楚的核心。

14. 考考自己

Q1：用「倉庫放貨」解釋列式和行式差在哪？ 行式＝每張訂單的東西打包成一箱（橫著存），盤點某類貨要每箱都翻。列式＝同類貨放同一櫃（直著存），盤點只開那一櫃。分析（盤點）適合列式，交易（取整筆訂單）適合行式。
Q2：為什麼「算各地區營收」在列式會快很多？ 因為只用到「地區」「金額」兩欄，列式只讀這兩櫃，其他幾十欄一個 byte 都不碰；行式被迫把整列搬出來再丟掉用不到的欄。讀取量差好幾十倍。
Q3：列式有哪三種常見壓縮，各適合什麼欄？ 字典（低基數字串，如地區/裝置）、RLE（連續重複的欄）、Delta（單調遞增，如時間戳/自增 id）。共通點：同欄型別一致、值相近，所以好壓。
Q4：為什麼分析查詢不該直接打交易（OLTP）庫？ ① 大掃描會吃光資源拖垮交易；② 交易庫是行式，掃整列拿一欄很浪費。所以要分到專門的列式倉儲跑，中間用 ETL/CDC 同步資料。
Q5（工程）：min/max 跳塊（zone map）怎麼省 I/O？ 把欄切成塊、每塊記 min/max。查條件時，範圍不可能命中的塊整塊跳過，連讀都不讀。是「只掃需要欄」之外的第二層省法。
Q6（工程）：那個「行式多掃幾倍」的倍數，取決於什麼？ 取決於「全部欄數 ÷ 查詢用到的欄數」，跟列數無關。50 欄表用 2 欄 → 行式多掃 25 倍。欄越多、用到越少，差距越大。
Q7（工程）：為什麼列式不適合逐筆更新？ 因為一筆的各欄分散在不同櫃，改一筆要動好幾個櫃；而且欄是壓縮成塊存的，改一個值得把整塊解壓、改、再壓回寫入。所以列式偏好「只追加、背景合併」，逐筆更新交給行式 OLTP 庫。

15. 30 秒回顧

列式儲存 = 把試算表「直著存」，讓分析只開需要的櫃。

4 招：① 只掃用到的欄（只開需要的櫃）→ ② 欄壓縮（同櫃好打包：字典/RLE/Delta）→ ③ 向量化（整櫃一起算）→ ④ min/max 跳塊（明顯不對的整櫃跳過）。合起來快 10～100 倍。

3 個怕：怕用錯地方（交易用行式、分析用列式）、怕資料不夠新（ETL/CDC 同步延遲）、怕 JOIN 拖垮（攤寬表/預聚合）。

工程細節一句話：分析成本＝讀了多少資料；列式從「只搬用到的欄 × 壓得更小」兩個方向砍 I/O → 行轉列入庫 → 一欄一櫃 + 壓縮 + min/max 標籤 → 查詢只開需要的櫃、整櫃向量化算、多機 MPP 平行。
