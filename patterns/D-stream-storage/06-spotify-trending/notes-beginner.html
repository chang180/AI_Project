1. 60 秒搞懂它在幹嘛

想像你是 Spotify 的店長。全世界每秒有幾十萬人按下播放鍵。老闆要你隨時掛一塊牌子在牆上： 「現在最紅的 100 首歌」，而且要幾秒鐘就更新一次。

Spotify 趨勢榜就是在做這件事。它的難，難在三點：播放事件多到數不完、要幾乎即時排出名次、而且榜單要反映「現在紅」而不是「歷史總紅」。

🔑 一句話 趨勢榜 = 對「數不完的播放」估個大概，挑出現在最紅的前 100 名：不追求分毫不差，只要名次對、夠快、夠新就好。

2. 為什麼可以「估個大概」就好？

很多人第一個反應是：「播放幾次，老老實實一筆一筆數不就好了？」問題在於——量大到你數不動。先用一個生活經驗錨定：

🍚 量杯估米，不數整袋 你要知道一大袋米大概幾粒，不會把幾百萬粒倒出來一粒一粒數（數到天荒地老）。 你會舀一小量杯、數那杯有幾粒，再乘回去估個大概。雖然不是分毫不差，但「夠用」。

趨勢榜也一樣：要精確記住「每一首歌被播了幾次」，得替上億首歌各留一個計數器，記憶體會直接爆掉。 但榜單根本不需要分毫不差——你只想知道「誰是前 100 名」，第 3 名播了 183 萬還是 183 萬零 5 次，沒人在乎。

所以這題的核心心法就是：用一個「小小的、固定大小的量杯」去估每首歌大概播幾次， 換來「記憶體小到不可思議、還能跑超快」。這個神奇量杯有個名字，叫 Count-Min Sketch（近似計數草圖）。

💭 停一下：為什麼「排行榜」這種題目特別適合用「估個大概」，但「銀行帳戶餘額」就絕對不行？

看答案 因為排行榜只在乎「相對名次」，每首歌都被估得稍微偏多一點點，名次幾乎不受影響，誤差可以容忍。 但銀行餘額在乎「絕對數字」——少你一塊錢都不行，必須精確。能不能用近似，看的是「你在乎相對還是絕對」。

3. 把系統拆成 4 個關卡

別被「即時串流排行系統」這串字嚇到。整件事其實就是闖 4 關，一關一關來：

1收事件：海量播放湧進來

全世界每秒幾十萬人按播放，每按一次就產生一筆播放事件（哪首歌、誰、什麼時候、哪個地區）。 第一關只做一件事：趕快把事件收下、排進隊伍，立刻說「收到」，不要讓使用者卡著等。

🎫 比喻：演唱會驗票口幾萬人同時湧入，驗票員不會當場跟你聊天——掃一下票、放行、馬上換下一個。事件收集層就是這個只負責「快速收下、放進隊伍」的驗票口。

2估計數：用小量杯估每首歌大概幾次

事件從隊伍裡一筆筆出來。每來一筆「歌 A 被播了」，系統就用那個神奇量杯（Count-Min Sketch）替歌 A 的計數 +1。

🍚 比喻：固定格子的計數板 想像一塊只有固定幾排幾格的計數板。每首歌用名字算出「該記在哪幾格」，播一次就在那幾格各 +1。 不同的歌偶爾會擠到同一格（撞格子），所以某格的數字可能被別人灌水、偏多一點。

查某首歌大概播幾次時，就看它對應的那幾格、取裡面最小的那個數字——因為被灌水只會讓數字變大， 取最小的那格，最接近真相。重點是：這塊板子大小固定，不管你有 1 億首還是 10 億首歌，記憶體都一樣大。

3挑前 K 名：誰能擠進排行榜？

量杯只能「給一首歌估個數字」，但它不會自己吐出「前 100 名是誰」。所以還要一個小小的排行榜（只放 100 格）。

🏆 比喻：只有 100 個座位的 VIP 席 VIP 席只有 100 個位子，門口站著保鏢盯著「目前席內分數最低的那位」。 每來一首歌，比一下它的估計分數：比那位最低的高，就把最低的請出去、讓新人坐進來；不夠高就擋在門外。

這就是「近似量杯 + 100 格排行榜」的經典組合：量杯負責估分，排行榜負責隨時維護「目前最紅的前 K 名」。

4讓榜「退燒」：舊播放慢慢不算數

最後一關最關鍵，也最容易被忽略。如果只是「累計總播放」，那榜上永遠是幾年來的老神曲—— 新歌再爆紅也擠不上去。但「趨勢榜」要的是現在正在紅的歌。

📰 比喻：新聞熱度會退燒 一則新聞剛爆出來時洗版，過幾天就沒人提了，熱度自己退掉。趨勢榜也要這樣： 定期把所有歌的分數打個折（例如每輪乘以 0.9），舊播放的影響一輪一輪淡掉， 新播放權重最大。於是最近狂飆的新歌就能浮上來。

這個「定期把分數打折讓舊的淡出」的動作叫 時間衰減。它就是「總量榜」和「趨勢榜」的分水嶺。

✅ 四關一句話 ① 驗票口快速收事件 → ② 用小量杯估每首歌大概幾次 → ③ 100 格 VIP 席維護前 K 名 → ④ 定期打折讓舊播放退燒。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「記憶體爆掉」 上億首歌如果各留一個精確計數器，要好幾 GB，再乘上「多個時間視窗 × 多個地區」就直接爆。 所以寧可接受一點點誤差，改用大小固定的小量杯（Count-Min Sketch）。記憶體可控 > 分毫不差。
😱 怕「太慢、卡到使用者」 全世界每秒幾十萬次播放，如果每按一次都要當場算完整個榜單再回應，使用者會等到天荒地老。 所以播放事件先收下就好（立刻回 202），榜單在後面非同步慢慢算、定期刷新。先收下，後面慢慢算。
😱 怕「榜單過時 / 不夠新」 趨勢榜的價值在「現在紅」。如果榜上全是陳年老歌（沒退燒），或榜慢半小時才更新，使用者會覺得「這榜假的」。 所以要時間衰減讓舊播放淡出，並讓榜秒級～分鐘級刷新。夠新，比夠精確重要。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每秒幾筆播放（寫）5 億人每天聽 30 次 ≈ 每秒 17 萬筆（尖峰 ×3 ≈ 50 萬）量大到不能「精確一筆筆數」
有多少首歌曲庫約 1 億首各留計數器 → 記憶體會爆
精確計數要多少記憶體1 億首 × 每首約 16 位元組 ≈ 1.6 GB，再 ×多視窗×多地區這就是「為什麼要估」的原因
小量杯要多少記憶體固定一塊板子 ≈ 5.4 MB（跟歌曲數量無關！）用一點誤差，換 300 倍記憶體省下來
看榜的查詢（讀）幾萬 QPS，全部走公告欄（快取）讀很快、不打擾後面算的人
🔑 關鍵洞察：記憶體「形狀」完全不同 精確計數的記憶體會隨歌曲數量一路長大（歌越多越爆、不可控）； 小量杯的記憶體是固定的一塊板子（1 億首和 10 億首一樣大）。 這個「固定大小」就是大規模排行榜願意接受一點誤差的根本原因。

💭 停一下：「寫」（每秒幾十萬筆播放）遠多於「讀」（看榜）。這跟一般網站「讀多寫少」相反，那設計重點該擺哪？

看答案 重點要擺在「扛住海量寫入」：用 Kafka 大水管把爆量攤平、用串流邊流邊算、用固定大小的小量杯省記憶體。 至於「讀」（看榜）量小又要快，把算好的榜貼到公告欄（Redis）讓大家看就好，不必每次重算。哪邊壓力大，料就往哪邊堆。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住三張「點餐單」：

① 上報一筆播放事件（超高頻，先收下就好）
   POST /api/v1/events
   給我：{ 歌ID, 使用者ID, 時間, 地區 }
   我回：202「收到了！」← 立刻回，不讓你等榜算完

為什麼「立刻回 202」？因為按播放的使用者不該卡著等我們把整個榜算完。先收下、丟進大水管（Kafka），後面非同步慢慢聚合。就像餐廳收單給號碼牌，不讓你站在櫃台等餐。

② 看排行榜（只看公告欄，超快）
   GET /api/v1/trending?window=1h®ion=global&k=100
   我回：200 { 視窗:"1h", 更新時間:"...", 前K名:[
            {名次:1, 歌ID:"s_8821", 分數:1832450}, ... ] }

可以挑時間視窗（過去 1 小時 / 24 小時 / 7 天）和地區（全球榜 / 台灣榜）。這些榜都是事先算好貼在公告欄的，所以查起來飛快。

③ 查單一首歌「大概」播幾次
   GET /api/v1/songs/{歌ID}/score?window=1h
   我回：200 { 歌ID:"s_8821", 估計值:1832450, 視窗:"1h" }

注意回的是「估計值」，不是精確值——因為這是小量杯估出來的（只會偏多一點點）。對排行榜來說，這個精度綽綽有餘。

9. 系統要記住哪些事？（資料模型）

📒 比喻：播放事件「不進帳本」，只進輸送帶 一般資料會老老實實寫進「帳本」（資料庫主庫）。但播放事件太高頻，硬寫主庫會把它壓垮。 所以播放事件改走輸送帶（Kafka），邊流邊算，最後只把「算好的榜」貼到公告欄。

📮播放事件流（Kafka，只進不改）

每筆播放一列：歌ID、使用者ID、時間、地區、裝置。它是只往後加、不修改的串流， 依歌ID 分流——這樣同一首歌的事件都落到同一個工人手上，計數不用跨工人合併。

🍚小量杯 + VIP 席（工人腦中的狀態）

這不是傳統資料表，而是工人邊算邊記在腦中的東西：一塊固定大小的計數板（Count-Min Sketch）估每首歌頻率， 加一個 100 格的 VIP 席（min-heap）維護目前前 K 名。每個「時間視窗 × 地區」各有一份。

📋排行榜公告欄（Redis 快取）

把算好的前 K 名序列化貼到公告欄，鑰匙是 (視窗 + 地區)，設個短短的有效期（TTL，幾秒～幾分）。 看榜的人全部只看這張公告欄——讀超快，也不會去吵後面忙著算的工人。

💡 為什麼原始事件還是要另外存一份？ 串流邊流邊算是近似的。若哪天要「精確對帳」或重算歷史，可把原始事件另存到一個擅長大量分析的倉庫（如 ClickHouse），事後慢慢精算。即時看近似、對帳用精確，兩者分工。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
瞬間爆量的播放事件衝垮系統用 Kafka 大水管「攤平」尖峰，後面慢慢消化（削峰）
1 億首歌精確計數 → 記憶體爆改用固定大小的小量杯（Count-Min Sketch）
一個工人算不完全世界的歌依歌ID 分流給多個工人並行算，各自的量杯再逐格相加合併
看榜的查詢去吵忙著算的工人把榜貼到公告欄（Redis），看榜全走公告欄
多個時間視窗 × 多個地區，組合爆炸把時間切成小桶，多個視窗共用相加；地區榜各自分流聚合
🧩 小知識：量杯為什麼能「合併」？ 因為每個工人的小量杯大小、格子規則完全一樣，要合併時只要把兩塊板子對應的格子相加就好——天生適合分給很多工人一起算，最後拼起來。

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 精確 vs 近似 精確計數記憶體隨歌曲數爆且不可控；小量杯記憶體固定，代價是計數只會偏多的有界誤差。對「排名」影響很小，值得換。
⚖️ 指數衰減 vs 分桶滑動視窗 「每輪打折」實作簡單、記憶體固定，但說不清「剛好過去 1 小時」；「把時間切成小桶」邊界精準，但桶越多越吃記憶體、過期桶要管理。看你多在乎視窗邊界的精準度。
⚖️ 每筆都重算 vs 定期刷新 每來一筆事件就重算整個榜，太貴；改成每隔幾秒物化一次到公告欄，省算力但榜稍舊（最終一致）。一般選後者。
💭 追問：常見誤區——「光有小量杯（Count-Min Sketch）就能出排行榜」對嗎？ 不對。小量杯只能「給你一首歌、估出它大概幾次」，但它不會自己列出「前 100 名是誰」。 所以一定要額外搭一個 100 格的 VIP 席（min-heap）來維護前 K 名。這是這題最常被考倒的點：量杯估分數，VIP 席排名次，兩個缺一不可。
💭 追問：有人開外掛狂刷某首歌的播放，怎麼辦？ 可以做去重，或改成「有幾個不同的人聽」而不是「被播幾次」（同一個人狂播只算一次）。算「幾個不同的人」也有專門的省記憶體估計工具（HyperLogLog），概念跟小量杯一樣是「用近似換省」。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 CountMinSketch（估每首歌大概幾次的「小量杯」，含退燒）→ TopK（只放 K 格的「VIP 席」＝排行榜）→ EventStore（把工人腦中的狀態存起來，讓多次看榜能接著算）。一筆播放事件來了，就照這個順序跑一遍。

1小量杯：用固定幾格估每首歌幾次（CountMinSketch）

核心就是一塊固定 d 排 × w 格的計數板。每首歌用名字算出「該記在哪幾格」，播一次就在那幾格各 +1：

// 計數板：d 排、每排 w 格，全部從 0 開始
// 不管 1 億首還是 10 億首歌，板子大小固定不變 ← 記憶體可控的關鍵

function 哪一格(第幾排, 歌名) {
    雜湊 = crc32(歌名 + 這排專屬種子)   // 每排用不同種子 → 算出不同位置
    return 雜湊 % w                     // 落在 0 ~ w-1 的某一格
}

function 計入(歌名) {
    for (第幾排 = 0 .. d-1)
        板子[第幾排][ 哪一格(第幾排, 歌名) ] += 1   // 每排都 +1（共 d 格）
}

這就是「舀一小量杯估個大概」的板子。不同的歌偶爾會擠到同一格（撞格子），所以某格可能被別人灌水、偏多一點——但板子大小固定，這正是它省記憶體的本錢。

查一首歌大概幾次時，看它對應的那 d 格、取最小的那個：

function 估計(歌名) {
    最小值 = 無限大
    for (第幾排 = 0 .. d-1)
        最小值 = min( 最小值, 板子[第幾排][ 哪一格(第幾排, 歌名) ] )
    return 最小值    // ← 被灌水只會變大，取最小那格最接近真相
}

為什麼取 min？因為撞格子只會讓數字變大、不會變小，所以那幾格裡最小的那個，最接近真值。這也是 demo 頁面上「估計值永遠 ≥ 真值」的由來。

💭 停一下：為什麼不取那幾格的「平均」或「最大」，偏偏要取「最小」？

看答案 因為每一格都可能被別的歌灌水（只會偏多）、絕不會偏少。所以這幾格全都是「真值 + 一些別人的灌水」。灌水最少的那一格＝數字最小的那一格，自然最貼近真值。取平均或最大反而把別人的灌水也算進來，誤差更大。

2退燒：定期把整塊板子打折（CountMinSketch.decay）

這是「總量榜」變「趨勢榜」的分水嶺。每隔一輪，把整塊板子的每一格都乘上一個衰減係數（例如 0.9）：

function 衰減(λ) {              // λ 例如 0.9
    for (每一格 in 板子)
        每一格 = floor(每一格 × λ)   // 全部打 9 折，舊播放一輪一輪淡掉
}

就像新聞熱度會退燒：每打一次折，舊播放的影響就淡一截，而新播放是用「+1」剛加進來的、幾乎沒被打折。連打幾輪，最近狂飆的新歌自然浮上來、陳年老歌往下沉——這就是趨勢榜的靈魂。

3VIP 席：只放 K 個位子的排行榜（TopK）

小量杯只能「給一首歌估個數字」，不會自己吐出前 K 名是誰。所以另外擺一個只有 K 格的 VIP 席。每來一筆播放，先讓量杯 +1、估出最新分數，再嘗試擠進席位：

function 餵入(歌名) {
    量杯.計入(歌名)              // ① 先讓小量杯 +1
    分數 = 量杯.估計(歌名)        // ② 估出這首歌最新分數

    if (歌名 已在席內)  { 更新它的分數; return }   // 已在榜上 → 刷新分數
    if (席內還沒坐滿 K 個) { 收進來; return }       // 還有空位 → 直接坐進來

    // 席滿了：找出「目前席內分數最低的那位」
    最低者 = 席內分數最小的歌
    if (分數 > 最低者的分數) {     // ③ 新人比最低的高 → 換人
        踢掉 最低者
        收進 歌名
    }
}

這就是「100 格 VIP 席 + 門口保鏢盯著最低分那位」。量杯估分數、VIP 席排名次，兩個缺一不可——這是這題最常被考倒的點。demo 為了好懂用一個迴圈找最低者；真實系統把它換成 min-heap（小頂堆），堆頂永遠是最低分，換人是 O(log K)——邏輯一模一樣，只是換更快的工具。

4衰減後刷新榜 + 存狀態（TopK.decay + EventStore）

打完折別忘了：席內每首歌的分數都變小了，要用量杯重新估一次、刷新席位分數，舊歌才會真的往下掉：

function 席位衰減(λ) {
    量杯.衰減(λ)                       // 整塊板子先打折（退燒）
    for (歌名 in 席內每一位)
        席內[歌名] = 量杯.估計(歌名)    // 用打折後的板子重估，刷新席位分數
}

這樣「最近沒人播」的老歌分數會一路下滑、可能跌出榜，反映趨勢變化。

最後，因為 PHP 內建伺服器每次請求是從零開始的，要把工人腦中的板子與席位存進一個 JSON 檔，下次請求才能接著累積：

function 存檔(狀態) {                  // 狀態 = { 板子, 席位, 精確值, 事件數 }
    寫入 state.json（json 格式, 加檔案鎖避免並行寫壞）
}
function 讀檔() {
    if (檔案不存在) return 空狀態      // 空狀態裡放一塊全 0 的新板子
    return 從 state.json 還原回來
}

demo 還順手存了一份「精確計數」，純粹是為了在頁面上跟量杯估計值對照，讓你親眼看到「估計值 ≥ 真值」的誤差。真實系統不會這樣存，這份精確值只是教學用的對照組。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
② 用小量杯估每首歌幾次CountMinSketch計入() + 估計() 取 min

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「量杯估米」解釋，為什麼趨勢榜不需要精確計數？ 因為榜只在乎「相對名次」，每首歌都被估得稍微偏多一點，名次幾乎不變，誤差可容忍。就像估一袋米幾粒，舀一小杯數完乘回去估個大概就夠了，不必一粒粒數。精確計數要替上億首歌各留計數器、記憶體會爆。
Q2：把系統 4 個關卡用一句話串起來。 ① 驗票口快速收播放事件 → ② 用小量杯估每首歌大概幾次 → ③ 100 格 VIP 席挑出前 K 名 → ④ 定期打折讓舊播放退燒，才能反映「現在紅」。
Q3：「總量榜」和「趨勢榜」差在哪？靠什麼機制做出趨勢榜？ 總量榜是「歷年累計最紅」，永遠是老神曲；趨勢榜要「現在紅」。靠「時間衰減」——定期把所有分數打折，舊播放像新聞一樣退燒，新播放權重最大，最近竄紅的歌就能浮上來。
Q4：小量杯查一首歌時為什麼要「取最小那格」？ 因為不同的歌偶爾會擠到同一格（撞格子），讓某格被灌水、數字偏大。被灌水只會讓數字變大、不會變小，所以一首歌對應的那幾格裡，取最小的那格最接近真相。
Q5（工程）：上報播放事件為什麼要「立刻回 202」、而不是算完榜再回？ 因為按播放的使用者不該卡著等整個榜算完。先把事件收下、丟進 Kafka 大水管，立刻回「收到」，後面非同步慢慢聚合、定期刷新榜。先收下，後面慢慢算。
Q6（工程）：「光有小量杯就能出排行榜」為什麼是錯的？要再加什麼？ 小量杯只能「給一首歌估個數字」，不會自己列出前 K 名。所以要再加一個大小為 K 的 min-heap（100 格 VIP 席）：每來一首歌比一下分數，比席內最低的高就擠進去。量杯估分、VIP 席排名，缺一不可。
Q7（工程）：這題「寫遠多於讀」，跟一般網站相反，設計重點該擺哪？ 重點擺在扛住海量寫入：用 Kafka 攤平爆量、串流邊流邊算、固定大小小量杯省記憶體、依歌ID 分流給多個工人並行。讀（看榜）量小要快，把算好的榜貼到 Redis 公告欄即可。哪邊壓力大，料就往哪堆。

15. 30 秒回顧

Spotify 趨勢榜 = 對數不完的播放估個大概，挑出現在最紅的前 K 名。

4 個關卡：① 驗票口收事件 → ② 小量杯（Count-Min Sketch）估每首歌大概幾次 → ③ 100 格 VIP 席（min-heap）挑前 K 名 → ④ 時間衰減讓舊播放退燒。

3 個怕：怕記憶體爆（用近似換固定大小）、怕太慢卡使用者（先收下、非同步算）、怕榜過時（衰減+秒級刷新）。

工程細節一句話：先估「每秒幾十萬筆」的爆量 → 用「點餐單(API)」溝通、收到事件先回 202 → 事件走 Kafka 大水管、依歌ID 分流給多工人 → 小量杯只會偏多取 min、量杯估分 VIP 席排名 → 算好的榜貼到 Redis 公告欄供快讀。
