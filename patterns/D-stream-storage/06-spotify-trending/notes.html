⑥ Spotify 趨勢榜：即時 Top-K 排行系統

D 串流計算/儲存分發 建議 41 分鐘 Design a Real-time Trending / Top-K System

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「海量播放事件串流 → 近似頻率計數 → 取 Top-K 排行 → 快取分發」的串流計算題。 精確計數每首歌的播放量在百億級事件下記憶體會爆；面試真正考的是 用近似演算法（Count-Min Sketch）+ 時間衰減/滑動視窗，以可控誤差換取吞吐與記憶體。

1. 需求釐清

功能性需求

  • 使用者播放歌曲 → 產生海量「播放事件」串流（song_id, user_id, ts, region）。
  • 即時 Top-K 排行：回傳「目前最熱門的 K 首歌」（如 Top 100），近乎即時更新（秒～分鐘級）。
  • 多維度榜單：全球榜、地區榜、不同時間視窗（過去 1 小時 / 24 小時 / 7 天）。
  • 查詢任一首歌的近似播放頻率（趨勢值）。
  • 新歌竄升偵測（爆發性成長 → 趨勢榜，不只總量榜）。

非功能性需求

  • 高吞吐寫：事件寫入量巨大（百萬 events/sec 級），必須能水平擴展、不丟事件（至少一次）。
  • 低延遲讀：榜單查詢 < 50ms（p99），由快取直接服務。
  • 近似可接受：榜單不要求精確到個位數，排名正確 + 誤差有界即可 → 用近似演算法。
  • 新鮮度：榜單可落後幾秒～幾分鐘（最終一致），不需強一致。
  • 讀遠多於寫？相反：寫（播放事件）量極大，讀（看榜）相對少但要快 → 寫路徑靠串流聚合攤平。
面試開場必問 「要精確還是近似？」「總量榜還是趨勢/竄升榜？」「時間視窗多長？」 這三題決定演算法：精確總量 → 排序聚合（重）；近似 + 滑動視窗 → Count-Min Sketch + 衰減（本筆記主軸，因為它才是大規模考點）。

2. 規模估算

指標假設推算
播放事件（寫）5 億日活，每人每日 30 次播放150 億/日 ≈ 17.4 萬 events/sec（峰值 ×3 ≈ 52 萬/s）
歌曲基數曲庫總量~1 億首（distinct key 很大）
精確計數記憶體1 億首 × (8B key + 8B count)≈ 1.6 GB / 視窗 / 節點，再乘多視窗多地區 → 爆
Count-Min Sketch寬度 w=272k、深度 d=5（誤差 ε≈e/w, 信心 1−δ=1−1/e⁵）272k × 5 × 4B ≈ 5.4 MB 固定（與歌曲數無關！）
榜單讀看榜遠少於播放~數萬讀 QPS，全由快取（Redis）服務
關鍵洞察精確 hash map 隨 distinct key 線性成長（GB 級且不可控）；Count-Min Sketch 記憶體固定（只看 w×d），這就是大規模 Top-K 願意接受誤差的根本原因。

3. API 設計

# 寫：上報播放事件（高頻，進 Kafka，不同步回應業務）
POST /api/v1/events
Body: { "song_id": "s_8821", "user_id": 421, "ts": 1717200000, "region": "TW" }
→ 202 Accepted   （非同步：只負責入列，不等聚合）

# 讀：取得 Top-K 榜單（由快取直接服務）
GET /api/v1/trending?window=1h®ion=global&k=100
→ 200 { "window":"1h", "updated_at":"...", "items":[
        { "rank":1, "song_id":"s_8821", "score":1832450 }, ... ] }

# 讀：查單曲近似趨勢值
GET /api/v1/songs/{song_id}/score?window=1h
→ 200 { "song_id":"s_8821", "estimate":1832450, "window":"1h" }

4. 資料模型

play_events （Kafka topic，append-only，依 song_id 分區）
  song_id, user_id, ts, region, device      -- 高頻寫入串流，不進 OLTP 主庫

# 串流處理器（Flink）狀態，非傳統資料表：
count_min_sketch[window][region]            -- 固定大小 w×d 的二維整數陣列（近似頻率）
heavy_hitters_topk[window][region]          -- 維護候選熱門集合（min-heap，大小 K）

# 對外服務的物化結果（最終一致，定期刷新）：
trending_cache  (key = window:region)
  → 序列化的 Top-K 榜單（JSON / sorted set），TTL 數秒~數分
為何不進主庫播放事件是極高頻 append-only，寫 OLTP 會被壓垮；走 Kafka 緩衝 + 串流聚合，最終只把「已算好的 Top-K」物化進 Redis 供讀。原始事件可另存列式/時序庫（ClickHouse）供離線精算對帳。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 播放(寫) ─▶ 事件收集 ─▶ Kafka(依 song_id 分區) ─▶ 串流聚合(Flink)
                                                        │  Count-Min Sketch 估頻率
                                                        │  + 滑動視窗 / 時間衰減
                                                        ▼
                                                   Top-K 堆(Heavy Hitters)
                                                        │  定期物化
                                                        ▼
 看榜(讀) ◀────────────────────────────────  排行快取(Redis sorted set)
  1. 事件收集層：無狀態，接收上報後立刻丟 Kafka 並回 202；播放體驗不被聚合阻塞。
  2. Kafka：依 song_id 分區，保證同一首歌的事件落到同一個串流算子，計數不需跨節點合併（或合併 sketch）。
  3. 串流聚合（Flink）：每個算子維護 Count-Min Sketch 估頻率 + 滑動視窗衰減，並用大小為 K 的 min-heap 維護 Top-K 候選。
  4. 排行快取：聚合結果定期（秒級）物化進 Redis（sorted set / 序列化 JSON），讀路徑只打快取。

6. 核心元件深入

Count-Min Sketch：固定記憶體的近似頻率計數

  • 結構：d 列 × w 欄的整數陣列，搭配 d 個獨立雜湊。
  • add(key)：對每一列用該列雜湊定位一格，+1（共 d 次遞增）。
  • estimate(key)：取 d 格中的最小值（碰撞只會讓計數偏大，取 min 最接近真值）。
  • 誤差保證：估計值 ≥ 真值，且超出量以高機率 ≤ ε·N（N=總事件數，ε≈e/w）。記憶體 O(w·d) 與 distinct key 數無關。
  • 合併：多節點各自的 sketch 可逐格相加直接合併（同 w、d、雜湊）→ 天然可分散式聚合。

Top-K：Heavy Hitters（光有 sketch 不夠）

Sketch 只能「給 key 估頻率」，無法列舉「誰是前 K 名」。所以額外維護一個大小為 K 的 min-heap（堆頂是目前榜內最小者）：每次事件用 sketch 估出該歌的最新估計值，若大於堆頂就嘗試入堆，藉此追蹤候選熱門集合。這就是 Count-Min Sketch + min-heap = 近似 Top-K（Heavy Hitters） 的經典組合。

時間衰減 / 滑動視窗：榜單要反映「現在熱」

  • 指數衰減：每一輪把所有計數乘上衰減係數 λ（如 0.9），新事件權重大、舊事件指數式淡出。實作簡單、記憶體固定，本 demo 採用。
  • 分桶滑動視窗：把時間切成固定長度的桶（如每分鐘一桶），每桶一份 sketch；查 1 小時榜就把最近 60 桶的 sketch 相加，過期桶整桶丟棄。視窗邊界更精確，但記憶體 ×桶數。
  • 趨勢 vs 總量：衰減後的分數天然偏向「近期竄升」的歌，正好對應趨勢榜；要竄升偵測可比較「短視窗估計 / 長視窗估計」的比值。

榜單更新頻率與快取

不必每個事件都重算榜單（成本高）。聚合器定期（如每 5～10 秒）把目前 Top-K 物化進 Redis；讀全部走快取，與聚合解耦。即使聚合短暫落後，使用者看到的榜只是稍舊，可接受（最終一致）。

7. 擴展與瓶頸（從零到百億級事件）

階段瓶頸對策
單機事件吞吐 / CPU無狀態收集層 + 負載均衡，水平加機器，Kafka 緩衝削峰
精確計數爆記憶體1 億 distinct key 的 hash map 不可控改用 Count-Min Sketch，記憶體固定 O(w·d)
單算子算不完所有歌擠一個聚合器Kafka 依 song_id 分區，多算子並行；各自 sketch 再合併
榜單讀爆看榜查詢打聚合器物化 Top-K 進 Redis，讀全走快取（TTL 秒級）
多視窗多地區視窗×地區組合爆炸分桶 sketch 可被多視窗共用相加；地區榜分區聚合

8. 取捨與面試重點

  • 精確 vs 近似：精確 hash map 記憶體隨 distinct key 線性成長且不可控；Count-Min Sketch 記憶體固定，代價是計數只會偏大的有界誤差 → 對「排名」影響小，值得。
  • 指數衰減 vs 分桶滑窗：衰減實作簡單、記憶體固定，但視窗邊界模糊（無法精準回答「剛好過去 1 小時」）；分桶精確但記憶體 ×桶數、過期桶要管理。
  • 每事件更新 vs 定期物化：每事件重算榜成本高；定期物化省算力但榜稍舊（最終一致）→ 通常選後者。
  • Sketch 需搭 heap：常見誤區是以為 Count-Min Sketch 自己就能出 Top-K。它只能估頻率，列舉前 K 名仍需 min-heap / Heavy Hitters 結構。
  • 常見追問：如何防刷榜（同一 user 灌量）？→ 事件去重 / 按 distinct user 計數（HyperLogLog）。如何做地區榜？→ 依 region 分區各自聚合。sketch 誤差怎麼估？→ ε≈e/w、信心 1−δ=1−e^(−d)，調 w、d 換精度。

9. PHP Demo 說明

本題 demo 聚焦大規模 Top-K 真正的考點：Count-Min Sketch 近似計數 + Top-K + 時間衰減，可實際用 PHP 內建伺服器跑：

  • src/CountMinSketch.php　真實實作 Count-Min Sketch（d 列雜湊、寬度 w，add/estimate），示範近似計數的固定記憶體。
  • src/TopK.php　用 min-heap 維護 Top-K 候選；用 sketch 估頻率；支援每輪指數decay（時間衰減）。
  • src/EventStore.php　持久化 sketch / 精確計數 / heap 狀態（JSON 檔模擬串流算子狀態）。
  • public/index.php　路由：批次餵播放事件 / 查單曲估計 vs 精確 / 看 Top-K 榜 / 套用衰減，並附深色測試頁。
# 啟動
cd demo
php -S localhost:8006 -t public
# 瀏覽器開 http://localhost:8006
# 模擬一批播放 → 查某歌近似計數(對比真實看誤差) → 看 Top-K → 套用衰減看排名變化
誠實聲明 Demo 為單機單行程，用 JSON 檔保存串流算子狀態以示意，並非真的接 Kafka / Flink 叢集；但 Count-Min Sketch 的雜湊計數與取 min 估計、Top-K 的 min-heap 維護、指數時間衰減皆為真實可執行的演算法， 且頁面會把 sketch 估計值對比精確計數，讓你直接看到近似誤差（估計值 ≥ 真值）。
