㉙ 遊戲排行榜 Leaderboard：百萬玩家的即時排名

D 串流計算/儲存分發 建議 38 分鐘 Design a Gaming Leaderboard

一句話定位 本質是「有序集合（sorted set）的即時維護 + 海量查詢分發」題。 難點不在「存分數」，而在每次更新都要知道全域名次：用 SQL ORDER BY ... LIMIT + COUNT 每次 O(N) 會爆掉； 正解是 Redis ZSET（skiplist）把 zadd / zrank 壓到 O(log N)。

1. 需求釐清

功能性需求

  • 提交分數：玩家一局結束回報分數，分數可累加或取最高（先問清楚是「最新值」「最高值」還是「累計值」）。
  • 即時排名：提交後馬上知道自己排第幾名（全域名次）。
  • Top-K 榜：顯示前 10 / 前 100 名。
  • 查自己名次 + 鄰居：「你在第 1234 名」，並顯示上下各幾名（玩家最愛看的「我離前一名差多少」）。
  • 分窗榜：日榜 / 週榜 / 總榜（賽季榜），各自獨立、可過期重置。

非功能性需求

  • 低延遲：提交與查榜 p99 < 100ms（遊戲內 UI 即時更新）。
  • 高寫入：熱門遊戲每秒大量對局結束 → 大量 zadd。
  • 讀多於寫：每次查榜都讀，Top-K 是熱點。
  • 可擴展：百萬甚至千萬活躍玩家。
  • 準確：名次要正確（但「精確 vs 近似」在超大規模可放寬，見 §7）。
面試開場必問 「分數語意是什麼？」（最高分 / 累計 / 最新）、「需要精確名次還是近似就好？」、「分窗榜怎麼重置（跨日/跨週）？」——這三題決定整題走向。

2. 規模估算

指標假設推算
活躍玩家1000 萬 DAU單一榜 ZSET 約 1000 萬成員
提交分數（寫）每人每日 10 局1 億次/日 ≈ ~1.2k zadd QPS（峰值 ×5 ≈ 6k）
查榜（讀）每人每日看榜 20 次2 億次/日 ≈ ~2.3k QPS（峰值更高，Top-K 是熱點）
ZSET 記憶體每成員 member(~16B)+score(8B)+skiplist 指標Redis 實測約 60～100 bytes/成員 → 1000 萬 ≈ 0.6～1 GB/榜
分窗榜數量日 + 週 + 總×3 榜（日/週榜會過期回收，總榜常駐）
歷史明細每局一筆落 DB1 億/日 → 走批次/時序庫，不進 ZSET
關鍵洞察單榜記憶體 ~1GB 完全放得進 Redis；真正的挑戰是更新時要維持全域有序（O(log N)）與熱榜單點（Top-K 查詢集中），而非容量。

3. API 設計

# 提交 / 更新分數（寫，高頻）
POST /api/v1/scores
Body: { "player": "alice", "score": 1500, "board": "daily" }
→ 200 { "rank": 42, "score": 1500, "total": 9_823_117 }   # 回更新後的全域名次

# Top-K（讀，熱點）
GET /api/v1/leaderboard/top?k=100&board=weekly
→ 200 { "top": [ { "rank":1, "player":"zoe", "score":99999 }, ... ] }

# 我的名次 + 上下鄰居
GET /api/v1/leaderboard/around/alice?n=5&board=daily
→ 200 { "rank":1234, "total":9_823_117,
        "window":[ {"rank":1229,...}, ..., {"rank":1234,"player":"alice","me":true}, ... ] }

# 查單一玩家分數
GET /api/v1/scores/alice?board=all → { "score":1500, "rank":42 }
對應 Redis 指令 POST /scores → ZADD；Top-K → ZREVRANGE 0 k-1 WITHSCORES；名次 → ZREVRANK；鄰居 → 先 ZREVRANK 取 r，再 ZREVRANGE r-n r+n。

4. 資料模型

# Redis ZSET：一個榜一個 key，member=玩家、score=分數
lb:daily:20260601    → ZSET { alice:1500, bob:1800, ... }   # 跨日換新 key / 設 EXPIRE
lb:weekly:2026W22    → ZSET { ... }                          # 跨週換新 key
lb:all               → ZSET { ... }                          # 賽季/總榜，常駐

# 明細落 DB（高頻 append-only，不進 ZSET）—— 走時序/列式庫或 Kafka→批次
score_events(player, score, board, ts)   # 補榜/審計/防作弊用，可重算 ZSET
為何 ZSET 與明細分開 ZSET 只保留「目前每位玩家的當前分數」（可變、要排序）；每局的歷史事件是 append-only 高頻寫，放 ZSET 沒意義，落獨立明細庫供重算 / 防作弊 / 審計。分窗 = 不同 key，跨日/跨週直接換 key 或設過期，乾淨重置。

5. 高階架構

🖼️ 開啟 Excalidraw 架構圖（diagram.excalidraw） 
            zadd 更新分數
 遊戲服務 ───────────────▶ Redis ZSET (skiplist, O(log N))
   │                          │  ├─ lb:daily   (日榜 key)
   │  每局明細                 │  ├─ lb:weekly  (週榜 key)
   ▼                          │  └─ lb:all     (總榜 key)
 明細 DB / Kafka              │
 (歷史·防作弊·可重算)          ▼
                       查 Top-K / 我的名次±鄰居 ──▶ 玩家
  1. 遊戲服務：對局結束 → zadd 同時更新日/週/總三個榜（同一玩家一局得分計入多榜）。
  2. Redis ZSET：唯一真相源（排名）；skiplist 維護全域有序，zadd / zrank O(log N)。
  3. 查詢路徑：Top-K（ZREVRANGE）、名次（ZREVRANK）、鄰居視窗；Top-K 結果可再加一層短 TTL 快取扛熱點。
  4. 明細 DB：每局事件落地，供補榜、防作弊、需要時重算 ZSET。

6. 核心元件深入

為什麼不用 SQL ORDER BY score LIMIT + COUNT？

  • Top-K：ORDER BY score DESC LIMIT 100 需掃描/排序，加索引後尚可，但每次仍是 O(N) 等級的維護成本。
  • 致命的是「我的名次」：SELECT COUNT(*) WHERE score > ? 每次都要數一遍，1000 萬列 = 每次查名次 O(N)，百萬玩家同時查直接拖垮 DB。
  • 更新即重排：分數一改，所有人名次都可能變，SQL 沒有「便宜地拿到某列排名」的手段。

ZSET / skiplist 為何快

  • Redis ZSET = skiplist + hash table。hash 表 member→score 讓 ZSCORE O(1)；skiplist 維護有序，並在節點上記 span（跨度），讓 ZRANK 也能 O(log N) 算出排名——這正是 SQL 缺的關鍵能力。
  • ZADD 更新：在 skiplist 中刪舊位置、插新位置，O(log N)，更新後榜即有序。

Top-K

ZREVRANGE key 0 k-1 WITHSCORES：從 skiplist 頂端順著走 k 步，O(log N + k)。Top-100 幾乎免費。

我的名次 ± 鄰居

玩家最在意「我第幾名、離上一名差多少」。做法：ZREVRANK 取得我的 rank r（O(log N)），再 ZREVRANGE r-n r+n 取上下各 n 名的視窗。本題 demo 的 around() 就是實作這個。

同分 tie-break

  • 分數相同時必須有穩定且可預期的次序，否則名次會抖動。
  • Redis ZSET 對同分成員以 member 字典序排。常見業務做法是把「達標時間」編進分數（如 score.timestamp 複合鍵：先到者排前），避免「同分但先達成的人吃虧」。
  • 本 demo 採「先加入者排前面」（單調遞增 seq），語意等價於「先達標者優先」，已於程式碼註明與 Redis 字典序的差異。

分窗榜過期

日榜 / 週榜 = 不同 key（如 lb:daily:20260601）。跨日直接寫新 key，舊 key 設 EXPIRE 自動回收；總榜常駐。好處：重置乾淨、不必逐筆刪、歷史榜可保留查詢。

7. 擴展與瓶頸（百萬玩家）

瓶頸說明對策
熱榜單點單一榜 = 單一 ZSET key，落在單一 Redis 分片上，寫入與 Top-K 查詢都集中Top-K 結果加短 TTL 快取（多數人看同樣的前 100 名）；讀副本擴讀
分片困難名次需全域有序，無法像一般資料按 key 雜湊切——切了就算不出跨片的全域 rank近似分桶（見下）、或單榜垂直擴容、或按「分數區間」分片再合併
近似排名（分桶）百萬級「精確名次」未必必要用百分位 / 分數直方圖：維護每個分數區間的人數，查名次時累加比我高的桶 → 近似 rank，O(桶數) 且可分片
百萬級 = 分段 / 百分位玩家其實只在意「前 1%」「打贏多少人」回傳「你贏過 87% 的玩家」而非精確第幾名，計算更便宜
更新合併同一玩家短時間多次提交 → 大量 zadd本地合併後批次寫（只保留最高/最新），削寫入峰值
持久化 / 重建Redis 掛了榜不能丟AOF/RDB + 從明細庫重算重建 ZSET

8. 取捨與面試重點

  • 精確排名 vs 近似排名：小規模用 ZSET 精確 O(log N)；超大規模可改分桶/百分位近似，省算力也好分片。先確認業務要不要精確。
  • 即時 vs 批次：對局結束即時 zadd（即時榜）；或事件先進 Kafka，週期性批次重算（省 Redis 壓力，但榜有延遲）。電競即時、休閒可批次。
  • ZSET 記憶體 vs DB 持久化：ZSET 快但在記憶體、單榜 ~1GB；明細落 DB 持久但慢。兩者並存：ZSET 當熱排名、DB 當真相備援與重算來源。
  • 分數語意：最高分 / 累計 / 最新——zadd 時用 GT（只升不降）還是 INCRBY 還是直接覆寫，取決於語意。面試務必先釐清。
  • 常見追問：作弊分數怎麼防？→ 明細庫審計 + 異常偵測 + 可回滾重算。跨區玩家同榜？→ 全域 ZSET 或就近寫再合併。

9. PHP Demo 說明

本題 demo 著重排行榜的核心資料結構語意（也是面試真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/SortedSet.php　模擬 Redis ZSET：zadd（更新即重排）、zscore、zrevrank（名次 1 起算）、topK（Top-K）、around（我的名次±鄰居視窗）；同分以「先加入者在前」tie-break；以 flock 保護並落地 data/；每個榜（日/週/總）是不同 key / 不同檔。
  • public/index.php　路由：POST /api/score、GET /api/top、GET /api/around/{member}，並附測試頁（提交分數、看 Top-K、查名次+鄰居、切換日/週/總榜）。
# 啟動
cd demo
php -S localhost:8029 -t public
# 瀏覽器開 http://localhost:8029 提交分數、看榜、查自己名次±鄰居、切換榜別
誠實聲明 Demo 用陣列排序模擬 ZSET：語意、Top-K、我的名次±鄰居、同分 tie-break、日/週/總分窗榜、更新即重排皆為真實可執行； 但未達 Redis skiplist 的 O(log N)（每次查詢/更新都重排，為 O(N log N)），也無真正 skiplist，持久化用 JSON 檔（flock）而非 Redis。 換成真 Redis 時，SortedSet 各方法可一對一映射到 ZADD / ZSCORE / ZREVRANK / ZREVRANGE（見 demo/README.md）。
