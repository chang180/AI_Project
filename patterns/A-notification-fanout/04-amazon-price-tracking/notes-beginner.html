1. 60 秒搞懂它在幹嘛

你在 Amazon 上看到一支耳機要 3000 元，覺得太貴。你按下「降到 2400 就通知我」。 幾天後價格真的掉到 2400，你的手機「叮」一聲——這就是價格追蹤。

聽起來很簡單對吧？難的是：商店裡有好幾億件商品，盯著它們的人有好幾千萬， 每件商品價格隨時在變。系統要做到：價格一變，瞬間找出「誰想要這個價」並只通知對的人，而且不能因為價格上下抖動就把人吵到退訂。

🔑 一句話 價格追蹤 = 幫幾千萬人盯著降價的自動管家：價格一變 → 算出「誰等的就是這個價」→ 通知他們， 但同一件事只提醒一次，不會疲勞轟炸。

2. 為什麼這題的難點不是「存價格」？

很多人第一次看這題會想：「不就是把價格存起來，便宜了就寄信嗎？」——這想法漏掉了兩個會把系統壓垮的真正難關。

📌 比喻：徵人啟事配對 想像一個人力銀行：每天有上千萬人留下「我想要月薪 4 萬以上的工作」「我要 5 萬以上」。 今天一個新職缺開出「月薪 4 萬 5」——你要瞬間找出「所有開價 4 萬 5 以下都願意」的求職者通知他們。 你不可能把上千萬人從頭問一遍「你要的價錢到了沒？」（太慢）。

價格追蹤的核心難關就是這個：一筆價格變動，要飛快地對上「在等這個價的所有人」。這叫條件匹配。

第二個難關更隱形：價格常常在你的目標價附近上上下下抖（2400→2410→2399→2405…）。 如果你「每次低於 2400 就發一次」，使用者一個下午就被你吵幾十次，直接退訂。怎麼讓它只在「真的降到了」時響一次，就是第二個難關（叫去抖）。

💭 停一下：為什麼「逐一問每個人『你要的價到了沒』」在這題行不通？

看答案 因為盯著商品的人有上千萬，每件商品價格一變就把這千萬人從頭問一遍，計算量會爆炸。 我們需要的是「反過來」：事先把「誰要什麼價」整理成一張排好序的表，新價來時一翻就知道命中誰，只碰真正達標的人。這就是後面要講的「反向索引」。

3. 把系統拆成 4 個關卡

別被「系統」嚇到。整件事其實就是闖 4 關，一關一關來：

1拿價格：商品現在多少錢？

要知道降價，得先持續知道每件商品「現在多少錢」。但商品有好幾億件，不可能每件每秒都去問一次（伺服器會被你問爆，成本也炸）。

🏥 比喻：醫院看診的優先順序 護理師不會每個病人每分鐘都量一次血壓。重症的盯緊一點（常量）、穩定的久久看一次。 系統也一樣：很多人在盯、價格常變的熱門商品「常常查」；沒人理的冷門商品「久久查一次」。這叫分層輪詢。

👉 面試時先問一句：「價格是商店會主動通知我們，還是我們得自己去一直問？」答案會大大改變這一關怎麼做。

2匹配：這個新價，誰在等？

價格變了，馬上要回答：「降到這個價，剛好踩中誰設定的門檻？」這就是第 2 節說的「徵人啟事配對」難關。

📖 比喻：按價錢排好的名冊 把所有盯這件商品的人，照他們想要的目標價從低排到高列成一本名冊。 新價是 2400 時，你只要翻到「2400 這一頁」往後看——後面（目標價 ≥ 2400）的人全部達標，前面的人還沒到，看都不用看。

這本「照目標價排好、能一翻就命中」的名冊，工程上叫 反向索引（reverse index）。 它把「逐一問一千萬人」變成「翻到那一頁、往後撈」，速度天差地遠。

3去抖：別像門鈴被按住一直響

命中的人找到了，先別急著發！價格會在門檻附近抖來抖去，如果每次低於門檻都發，使用者會被轟炸。

🔔 比喻：門鈴按一次別連響 門鈴被按一下，響一聲就好。如果手指停在按鈕上微微顫抖，門鈴跟著「叮叮叮叮」連響，誰受得了？ 價格追蹤要做的，就是「降到了 → 響一聲 → 然後安靜一陣子，別再為同一件事吵你」。

系統靠一個三檔狀態來管這件事（像門鎖的三個位置）：

  • ARMED（待命）：還沒達標，隨時準備通知。像門鈴「裝好電池等著被按」。
  • TRIGGERED → COOLDOWN（響過了、冷靜中）：剛通知過你，進入冷卻期（例如 24 小時），這段時間就算價格還在門檻下也不再吵你。
  • 回到 ARMED（重新待命）：要等價格明顯回升（不只是漲一塊，要漲過一個「安全線」）才重新武裝。這條安全線避免「降一塊→升一塊→又降」的反覆觸發。

這個三檔狀態的循環就叫去抖狀態機，是這題面試最常被追問的核心。

4通知：同時送給好多人（fan-out）

熱門商品（像新款 iPhone）可能上百萬人同時在等降價。一旦降價，要同時通知這麼多人。

🧑‍🏫 比喻：老師通知全班 老師通知全班不會自己一個個打電話（慢），而是分給幾個小組長分頭傳（同時進行）。 系統也一樣：把名單切成很多份，丟給一大群「推播工人」同時送。

這個「一筆降價 → 炸開成上百萬則通知」的動作，就叫 fan-out（扇出），也是這一組（A 組）的共同主題。

✅ 四關一句話 ① 分層拿價格 → ② 用排好序的名冊匹配出誰在等 → ③ 去抖（響一聲就好、別連響）→ ④ 一群工人同時推播。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「吵到退訂」 這是價格追蹤最致命的怕。價格一抖就連發通知，使用者一氣之下把追蹤全關掉——以後真降價也收不到了。 所以才需要去抖狀態機 + 冷卻期 + 回升安全線，讓「同一件事只響一次」。寧可少吵，不可亂吵。
😱 怕「慢 / 算不動」 幾億商品 × 幾千萬個門檻，如果每筆變價都把所有人問一遍，機器直接跑不動。 所以要用反向索引（排好序的名冊）只碰達標的人，還要在拿價格那關用分層輪詢省下沒必要的查詢。只算必要的，不做白工。
😱 怕「漏發 / 重發」 使用者等的價真的到了卻沒收到通知（漏發）→ 信任崩盤；同一次達標卻收到十封（重發）→ 一樣惹人厭。 所以原則是寧可重送也不漏送（at-least-once），再用「同一次達標只算一次」的去重（冪等）把重送擋掉。不漏、也不重。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
有多少商品~5 億件不可能每件都常常去問價（成本爆炸）→ 要分層輪詢
有多少「追蹤」關係5000 萬人 × 平均 4 件 ≈ 2 億條匹配不能逐條掃 → 要反向索引
每天多少件變價5% 商品變價 ≈ 2500 萬/日 ≈ 每秒 ~290 筆事件量其實不大，真正的爆點在後面
一筆變價會命中多少人熱門商品單筆可命中數萬人通知量遠大於變價量 → fan-out 才是壓力來源
💡 關鍵洞察：壓力不在「變價」，在「命中後的通知」 每秒才幾百筆變價，聽起來很閒。但一筆熱門商品變價，可能瞬間炸出幾萬則通知。 所以系統的重點不是「處理變價」，而是「一筆變價要飛快匹配出誰在等（反向索引）+ 用去抖把實際發送量壓下來」。

💭 停一下：每天變價才 2500 萬筆（不算多），為什麼系統還是可能被壓垮？

看答案 因為一筆變價會「放大」。一支降價的熱門 iPhone 可能同時命中幾萬個在等的人，一筆事件瞬間變成幾萬則通知。 真正要扛的是這個「放大後的通知洪峰」，不是原始的變價筆數——所以反向索引（快速匹配）和去抖（壓低發送量）才是設計核心。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」：

① 使用者建立追蹤（最常用）
   POST /api/v1/watches
   給我：{ 哪個使用者, 哪件商品, 想要的目標價(或跌幅%) }
   我回：201「建好了！狀態：ARMED（待命）」

門檻有兩種：「降到絕對價」（如 2400 元）或「從基準跌幾 %」（如 -15%）。後面會看到，跌幅其實可以先換算成一個「絕對觸發價」，這樣兩種門檻就能塞進同一本名冊統一處理。

② 取消 / 改門檻
   DELETE /api/v1/watches/{追蹤編號}
   PATCH  /api/v1/watches/{追蹤編號}   給我：{ 新的目標價 }

改了目標價，記得做兩件事：更新名冊裡這個人的位置（排序變了），並把狀態重設回 ARMED（重新待命）。

③ 查我的追蹤清單與狀態
   GET /api/v1/users/{使用者}/watches
   我回：[{ 商品, 目標價, 目前狀態, 已發幾次提醒, 上次價格 }, ...]
④ 內部：價格擷取層送來新價格（不對外）
   POST /internal/prices   給我：{ 哪件商品, 新價格, 時間 }
   → 系統收到後做四件事：變更偵測 → 匹配 → 去抖 → 通知 fan-out

注意第 ④ 張是「內部」點餐單——只有自家的價格擷取器會用，不開放給外面，使用者碰不到。

9. 系統要記住哪些事？（資料模型）

📒 比喻：四本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題用四本（其中一本是那本「排好序的名冊」）：

🏷️商品簿（每件商品現在多少錢）

每件商品一列：商品編號、現價、更新時間、輪詢分層(熱/溫/冷)。 多存一欄「分層」，是為了讓拿價格那關知道「這件要常拿還是久久拿一次」。

👀追蹤簿（誰在等什麼價）

每個追蹤一列：追蹤編號、哪個使用者、哪件商品、門檻類型(絕對價/跌幅)、門檻值、狀態(ARMED/TRIGGERED/COOLDOWN)、冷卻到何時、已發幾次。 這本就是去抖狀態機的「記憶」——每個人現在是「待命」還是「冷卻中」都記在這。

📖名冊（反向索引——核心！）

給定一件商品，要能瞬間取出所有在等的人，而且照目標價排好序。 工程上常用 Redis 的 ZSET（排序集合）：以「目標價」當排序分數。

為什麼要「排好序」？ 因為匹配的本質是區間查詢：「新價 2400 來了，誰的目標價 ≥ 2400？」。 如果名冊沒排序，你得整本翻一遍比大小（一千萬人 → 慢）。排好序後，翻到 2400 那頁、往後一撈就全是達標的人——這就是把「逐條掃描」變成「翻書」的精髓。

✅提醒簿（發過沒——去重用）

記「哪個追蹤、哪次達標、發給誰、何時發」。 用 (追蹤編號 + 這次觸發的時段) 當鑰匙——同一次達標只算一次，就能做到「重送也不會讓你收到兩封」（這就是冪等）。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
拿價格：幾億商品每件都常去問分層輪詢（熱門常拿、冷門少拿）；能改成「商店主動通知」(CDC)更好
一堆「價格其實沒變」的事件灌爆下游變更偵測：跟上次比，沒變就直接丟掉，不打擾後面
匹配：一千萬人逐條比大小用排好序的名冊（反向索引），翻到那頁往後撈
超熱門商品：那本名冊太大、單點壓力把這本名冊再切成幾份分開放 + 本地快取 + 批次處理
通知洪峰：一筆變價瞬間命中幾萬人通知佇列削峰（先排隊慢慢送）+ 多工人並行 + 每人限流
價格抖動造成反覆觸發去抖狀態機 + 冷卻期 + 回升安全線 + 冪等鍵

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 商店主動通知(CDC) vs 我們一直問(輪詢) 商店主動通知最省（價格真變才動），但要對方願意發事件；自己去問最通用，但要分層控成本，而且兩次問之間的降價可能漏抓。
⚖️ 價格一來就主動算(推) vs 等使用者開 App 才查(拉) 價格追蹤要主動提醒，所以選「推」：價格來就立刻匹配推送。代價是寫入時要算（寫放大），但這正是使用者要的體驗。
⚖️ 去抖要多嚴？ 冷卻期太長 → 漏掉「跌得更深」值得再提醒的好機會；太短 → 又開始轟炸。 折衷做法：冷卻期內只有「跌破更低的價」才破例再發一次。
💭 追問：使用者把目標價從 2400 改成 2000，系統要做什麼？ 兩件事：① 更新名冊（反向索引）裡這個人的「排序分數」，因為他的位置變了；② 把他的狀態重設回 ARMED（待命），這樣新門檻才能正常觸發。
💭 追問：怎麼決定哪些商品算「熱門、要常拿價」？ 看兩個訊號：有多少人在追蹤（很多人盯 = 重要）+ 歷史上價格波動大不大（常變 = 要盯緊）。用這兩個動態調整它的輪詢分層（熱/溫/冷）。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「匹配 + 去抖狀態機」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8004 -t public
# 然後瀏覽器打開 http://localhost:8004
  1. 先建立追蹤：幫某商品設一個目標價（例如 2400）。
  2. 餵入幾筆價格：手動推一筆新價，或讓它隨機波動一輪。
  3. 觀察：哪些追蹤觸發了、每個追蹤現在是 ARMED / TRIGGERED / COOLDOWN 哪一檔、已發幾次提醒。
玩的時候想一下價格剛好降到門檻 → 觸發一次後，把價格再壓低一點點，看它會不會再發（不會！因為在冷卻中）——這就是去抖。再把價格拉高過「回升安全線」再壓低，看它是不是又能觸發了——這就是遲滯帶 re-arm。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 PriceFeed（拿價格的巡邏員，含變更偵測）→ Watchlist（那本排好序的名冊＝反向索引）→ AlertEngine（決定「要不要響」的去抖大腦）。價格一來，就照這個順序跑一遍。

1變更偵測：沒變就丟掉（PriceFeed）

新價格進來的第一關，先問一句「真的變了嗎？」。沒變就直接丟，不去吵後面的人。

function push(商品, 新價) {
    上次價 = 記錄裡這件商品的價格
    if (上次價 和 新價 幾乎一樣)
        return 沒事;          // ← 沒變 → 丟掉，下游完全不會被打擾
    記錄[商品] = 新價           // 真的變了才更新、往下送
    return { 商品, 新價, 上次價 }
}

就這一行 if，把「其實沒變」的雜訊擋在門外。少了它，一堆「價格沒動」的空事件會白白灌爆後面每一關。

2名冊匹配：翻到那頁往後撈（Watchlist）

先看一個小聰明：不管你設「降到 2400」還是「跌 15%」，建立追蹤時都先換算成同一個「觸發價」：

觸發價 = (門檻是跌幅%) ? 基準價 × (1 - 跌幅/100)   // 例：3000 跌15% → 2550
                       : 目標價                  // 例：直接就是 2400

兩種門檻統一成一個「絕對觸發價」後，就能塞進同一本名冊，用同一種比較處理——這是讓系統變簡單的關鍵一步。

然後匹配就是這麼一句：

function 找出誰在等(商品, 現價) {
    回傳所有「同一件商品 且 現價 <= 它的觸發價」的追蹤
}

這就是前面說的「翻到 2400 那頁、往後一撈」。demo 為了好懂用一個迴圈過濾；真實系統把它換成 Redis 的 ZSET（排好序的名冊），用 ZRANGEBYSCORE 一次 O(log N) 撈出所有達標的人——邏輯一模一樣，只是換更快的工具。

3去抖大腦：門鈴按一次別連響（AlertEngine）

這是全題的心臟。每個達標的人，先問「它現在是哪一檔？」再決定要不要真的發：

if (這個人達標了) {
    if (狀態 == ARMED) {           // ← 只有「待命」才會真的響
        狀態 = TRIGGERED
        冷卻到 = 現在 + 30 秒
        發出通知(...)              // 響一聲！
        狀態 = COOLDOWN            // ← 響完馬上進「冷卻中」
    }
    // 已經是 COOLDOWN？就算還在門檻下，也不發 → 抖動被吸收
} else {                          // 沒達標（價格回升了）
    回升安全線 = 觸發價 × 1.05     // ← 遲滯帶：要漲過這條線才算數
    if (狀態==COOLDOWN 且 現價 >= 回升安全線 且 冷卻時間已過)
        狀態 = ARMED              // 重新待命，下次降價才會再響
}

前面講的「ARMED → TRIGGERED → COOLDOWN → 回 ARMED」整套就是這段 if。三檔狀態 + 冷卻 30 秒 + 回升安全線（×1.05），全在這裡。這也是面試官最愛追問的一段。

💭 停一下：為什麼「響完」要馬上把狀態設成 COOLDOWN，而不是留在 TRIGGERED？

看答案 TRIGGERED 只是「正在響」的瞬間狀態；真正存起來的是 COOLDOWN（冷卻中）。這樣下一筆價格進來、看到這個人是 COOLDOWN，就知道「剛響過，別再吵」——抖動才會被吸收。

4冪等：同一次只發一封（AlertEngine 的發送）

真的要發了，最後一道保險：同一次達標，絕不發成兩封。

週期 = 現在時間 ÷ 30秒（取整）       // 同一個冷卻週期算「同一次」
鑰匙 = hash(追蹤編號 + 週期)
if (這把鑰匙發過了) return 跳過;     // ← 冪等：同一次只發一次
記下這把鑰匙;  真的送出通知

就算上游因為「寧可重送也不漏送」（at-least-once）重複丟了同一筆，投遞層一看「這把鑰匙已存在」就跳過。這就是把「至少一次」變成「剛好一次」的體驗。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 拿價格 + 變更偵測PriceFeedpush() 的那行 if
② 匹配誰在等Watchlist找出誰在等()（反向索引）
③ 去抖（響一聲就好）AlertEngineARMED / COOLDOWN 的 if
④ 不重發AlertEngine發送前的「鑰匙」檢查

👉 重點：真實系統會把「JSON 檔 + 迴圈過濾」換成「分片資料庫 + Redis ZSET + Kafka 佇列」， 但上面這幾段判斷邏輯一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「徵人啟事配對」解釋，這題的核心難關是什麼？ 一筆新價（職缺開價）來了，要瞬間從上千萬個「在等某個價」的人裡，找出「這個價剛好踩中門檻」的所有人。不能逐一問每個人，要用「反過來、排好序」的方式一翻就命中——這就是條件匹配 + 反向索引。
Q2：為什麼不能「每次價格低於門檻就發一次通知」？怎麼解決？ 因為價格會在門檻附近上下抖，這樣會把使用者轟炸到退訂。用去抖狀態機（ARMED→TRIGGERED→COOLDOWN）：響過進冷卻期不再吵，要等價格回升過「安全線」才重新待命。像門鈴按一次別連響。
Q3：fan-out 是什麼？用班級的比喻說說看。 fan-out = 一筆事件同時通知超多人。像老師通知全班不自己一個個打，而是分組長分頭傳；系統把命中名單切片，交給一大群推播工人同時送。熱門商品一降價可能命中上百萬人。
Q4：每天變價才 2500 萬筆不算多，為什麼系統還可能被壓垮？ 因為一筆變價會「放大」——一支熱門商品降價可能同時命中幾萬個在等的人，瞬間炸出幾萬則通知。真正要扛的是放大後的通知洪峰，所以反向索引（快速匹配）和去抖（壓低發送量）是核心。
Q5（工程）：為什麼那本「名冊（反向索引）」要照目標價排好序？ 因為匹配本質是區間查詢「新價 P 來了，誰的目標價 ≥ P？」。排好序後翻到 P 那頁往後一撈就全是達標的人（O(log N + 命中數)）；沒排序就得整本掃一遍比大小（O 全部 watch），千萬級下不可行。
Q6（工程）：「冪等去重」在這題防的是什麼？怎麼做？ 防的是「同一次達標卻發了好幾封」。原則是寧可重送不漏送（at-least-once），再用一把鑰匙 hash(追蹤編號 + 觸發時段) 記在提醒簿，同一次只算一次，把重送擋在投遞層。
Q7（工程）：使用者把目標價改掉，系統要更新哪兩個地方？ ① 反向索引（名冊）裡他的排序分數要改，因為位置變了；② 他的狀態要重設回 ARMED（待命），新門檻才能正常觸發。

15. 30 秒回顧

價格追蹤 = 幫幾千萬人盯著降價的自動管家。

4 個關卡：① 分層拿價格 → ② 用排好序的名冊（反向索引）匹配誰在等 → ③ 去抖（門鈴按一次別連響）→ ④ 一群工人同時推播（fan-out）。

3 個怕：怕吵到退訂（去抖最致命）、怕慢/算不動（反向索引 + 分層輪詢）、怕漏發/重發（at-least-once + 冪等去重）。

工程細節一句話：壓力不在變價筆數、在「命中後放大的通知洪峰」→ 用「點餐單(API)」溝通、新價先過變更偵測 → 四本筆記本（商品/追蹤/名冊/提醒）＋反向索引排序 → 去抖狀態機（冷卻＋回升安全線）壓低發送量 → 找出塞車環節各自拓寬。
