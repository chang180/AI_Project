1. 60 秒搞懂它在幹嘛

想像你是一個 5000 萬人 LINE 群組的管理員。突然偵測到「地震來了！」，你必須在幾秒鐘內， 只通知住在危險區域的人，叫他們「快躲！還有 15 秒」。

地震預警系統就是在做這件事。它的難，難在三點：群組有幾千萬人、只能通知對的人、而且你在跟地震波賽跑。

🔑 一句話 地震預警 = 跟地震賽跑的精準群組廣播：偵測 → 算出要通知誰 → 同時通知幾百萬人，全程要在幾秒內完成。

2. 為什麼地震「可以」被預警？

很多人第一個疑問是：「地震都發生了，怎麼可能『預先』警告？」關鍵在一個你早就懂的生活經驗——

⚡ 閃電與雷聲 打雷時，你會先看到閃電（快），過幾秒才聽到雷聲（慢）。 地震也發出兩種波：P 波跑得快但溫和（像閃電）、S 波跑得慢但具破壞性（像雷聲）。

震央附近的地震儀會先偵測到溫和的 P 波，馬上判斷「地震了！」，然後用光速的網路訊息 通知遠方的城市——趕在那個會搖垮房子的 S 波（雷聲）走到你家之前。

所以預警的本質是：用「光速的訊息」超車「地震波」。系統要做的，就是把「偵測＋通知」的總時間，壓到比地震波傳過來還短。

💭 停一下：為什麼住得離震央越遠的人，反而有越多逃命時間？

看答案 因為訊息走光速（瞬間到），地震波走每秒幾公里（很慢）。離震央越遠，地震波要花越久才到你家，而訊息早就到了——所以「訊息超車地震波」的領先幅度就越大。離震央太近的人（正下方）則幾乎來不及，這叫「預警盲區」。

3. 把系統拆成 3 個關卡

別被「系統」嚇到。整件事其實就是闖 3 關，一關一關來：

1偵測：「地震發生了！」

有人要先喊出「地震來了」。這通常由官方測震網（像中央氣象署、美國 USGS）負責——它們有滿地的地震儀， 幾秒內就能估出震央在哪、規模多大。

🏫 比喻就像學校的火災警報器：偵測器負責「發現火」並拉響鈴；我們的系統可以只負責「鈴響之後，通知對的人」。

👉 面試時先問一句：「偵測是我們做，還是別人給我們事件？」答案會大大改變題目範圍。多數情況我們只負責後面兩關。

2篩選：到底要通知「誰」？

地震只影響震央附近一圈人。問題是：怎麼從 5000 萬人裡，幾毫秒內挑出「住在這圈裡」的人？

📮 比喻：找附近的人，先看郵遞區號 你要找「住我家附近的人」，不會把全國電話簿從頭到尾、一個一個量距離（太慢）。 你會先看郵遞區號 / 地址前幾個字，先框出同區的人，再細看。

系統用的工具叫 geohash：把地球切成一格一格的小網格，每格一個短代碼，位置相近的代碼開頭一樣。 於是篩選變兩步：

  • 粗篩：用「震央附近那幾格」的代碼前綴，快速撈出候選的人。
  • 細篩：再對這小群候選人，精算「離震央幾公里」，留下真正在範圍內的。

⚠️ 小細節：人可能住在格子的邊緣，所以要連旁邊 8 格一起撈，免得漏掉邊界上的鄰居。寧可多撈，不可漏人。

3通知：同時送給幾百萬人（fan-out）

名單有了，現在要同時把警報送到幾百萬支手機。一台機器一個一個送，送到天亮地震都結束了。

🧑‍🏫 比喻：老師通知全班 老師要通知全班 30 人，不會自己一個一個打電話（慢）。而是分給幾個小組長分頭傳（同時進行）。 系統也一樣：把名單切成很多份，丟給一大群「推播工人」同時送。

這個「一個事件 → 炸開成幾百萬則通知」的動作，就叫 fan-out（扇出）。 而且地震一發生就要主動推給大家（你不會自己打開 App 去「拉」地震消息），所以是「事件一來就立刻展開」。

✅ 三關一句話 ① 偵測地震 → ② 用郵遞區號式的網格挑出附近的人 → ③ 一群工人同時推播。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「慢」 每慢 1 秒，破壞性的地震波就多走約 3.5 公里。所以「快」壓倒一切： 寧可先用少量資料趕快發警報，晚一點再用更完整資料發「修正」或「取消」。速度 > 完美。
😱 怕「漏掉人」 漏通知一個人，可能就是一條命。所以系統寧可重複送（叫 at-least-once，至少送一次）。 那重複送會不會吵到你？用「同一個警報只響一次」的去重機制處理（叫冪等）。寧可吵，不可漏。
😱 怕「吵錯人」 如果常常誤報、沒事也響，大家會直接把警報關掉——等到真地震時就沒人理了，反而更危險（這叫「誤報疲勞」）。 所以「範圍外的人不要打擾」跟「不能漏掉」一樣是硬規定。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
要照顧多少裝置全國 ~5000 萬支手機篩選不能慢慢一個個查
一次地震影響多少人都會區可達 500 萬～1000 萬要能「同時」通知這麼多人
有多少時間離震央 70 公里 ≈ 只有 20 秒，扣掉偵測剩 ~10 秒全部動作要在 10 秒內做完
要多快的推播速度1000 萬人 ÷ 10 秒 = 每秒 100 萬則一台機器不可能，要一大群一起送
🏟️ 關鍵洞察：演唱會散場 這系統平常幾乎沒事（0 流量），地震瞬間爆量——像空蕩蕩的體育館，演唱會散場那一刻幾萬人同時擠出口。 所以重點不是「平均多忙」，而是「能不能在那一瞬間，突然放大到百萬級」。

💭 停一下：如果平時流量幾乎是 0，為什麼不能「地震發生才開機器」？

看答案 因為開機器、暖機要時間（幾十秒到幾分鐘），但地震只給你 10 秒。所以要平時就留一小撮機器待命（小火種），地震時瞬間放大，不能從零開始臨時抱佛腳。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住三張「點餐單」：

① 偵測服務通報地震（最重要）
   POST /quake-events
   給我：{ 震央經緯度, 規模, 影響半徑, 偵測時間 }
   我回：202「收到了！」← 立刻回，不讓你等

為什麼「立刻回 202」？因為通報的人不該卡在那邊等我們慢慢通知幾百萬人。先收下，後面非同步慢慢展開。就像餐廳收單後給你號碼牌，不會讓你站在櫃台等餐。

② 修正 / 取消（用同一個地震編號）
   PATCH /quake-events/{編號}   → 規模改了、降級
   DELETE /quake-events/{編號}  → 誤報，發取消通知

用「同一個編號」很關鍵：這樣修正就是蓋掉原本那筆，而不是變成第二次地震（這就是「冪等」——同個編號做幾次結果都一樣）。

③ 使用者註冊裝置 + 位置（平常就做好）
   POST /devices
   給我：{ 推播 token, 經緯度, 想用哪些管道(push/簡訊) }

送到手機的內容長這樣：{ 強度:"5強", 還有幾秒:18, 該做什麼:"就地掩護" }。

9. 系統要記住哪些事？（資料模型）

📒 比喻：三本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題只要三本：

📱裝置簿（誰、在哪、怎麼通知）

每支手機一列：裝置ID、推播token、經緯度、geohash、想用的管道。

為什麼多存一欄 geohash？ 把每個人的「郵遞區號」事先算好寫進去，地震來時才能用前綴秒查附近的人。 如果地震當下才一個個算，就來不及了——這叫「預先算好，換查詢飛快」。

🌋地震簿（哪次地震）

每次地震一列：地震編號（主鍵）、震央、規模、影響半徑、狀態(新/修正/取消)。 用「地震編號」當主鍵，修正/取消才能精準蓋掉同一筆。

✅投遞簿（送了沒）

記「哪次地震、送給哪支手機、送到了沒、重試幾次」。 用 (地震編號 + 裝置ID) 當鑰匙——同一組只算一次，就能做到「重送也不會讓你手機響兩下」。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
篩選：5000 萬人一個個算距離用網格索引（郵遞區號法）先粗篩再細篩
地震消息被一堆普通通知卡在後面排隊給地震一條專屬 VIP 通道，永遠插隊最前面
一台推播工人送不完千萬則把名單切片，多請一大群工人同時送（水平擴展）
蘋果/Google 的推播服務對你限速用多個帳號/多管道分流 + 批次送
平時 0 流量，地震要瞬間爆量平時留小火種待命，能秒級放大（別從零開機）
地震可能直接震垮你的機房機器放好幾個不同地點，一個垮了還有別的

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 快 vs 準 等更多地震儀資料 → 判斷更準，但更慢。地震選快：先用少量資料發警報，再用後續資料發「修正」。
⚖️ 主動推 vs 等人來拉 一般 App 可以「等你打開才載入」（拉）。但地震一定要主動推——沒人會自己打開 App 去「查有沒有地震」。
⚖️ 網格畫多大？ 格子太大 → 撈出太多人、細篩慢；太小 → 邊界容易漏、要撈更多鄰格。取一個剛好蓋住典型影響範圍的大小，並一律多撈周圍 8 格保險。
💭 追問：使用者在「移動中」（開車、搭高鐵）怎麼辦？ 手機定時回報新位置（心跳）更新索引；或推播時用「粗一點的網格」容忍一些位置誤差，寧可多通知幾個人。
💭 追問：平常不發生地震，這系統怎麼測試？ 用「演練」——定期重放歷史地震事件或灌入假事件，量「從偵測到手機響」要幾秒，確保真地震時來得及。

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「篩選 + 通知」：

# 啟動（在這題的 demo 資料夾裡）
cd demo

# 然後瀏覽器打開 http://localhost:8002
  1. 先按「載入示範裝置」（等於在地圖上放幾個人）。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心就那四個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 四個檔 = 四個小幫手 GeoIndex（看郵遞區號挑出附近的人＝地理篩選）→ NotificationQueue（把一個地震「炸開」成一人一張的派工單＝fan-out）→ Dispatcher（真的把警報送出去的推播工人，負責去重與重試）→ Store（記帳本：誰送到了沒）。地震一來，就照這個順序跑一遍。

1地理篩選：先看郵遞區號，再量距離（GeoIndex）

對應關卡 2的「先看郵遞區號框出同區，再細看」。程式分兩步：粗篩（算出震央那格的代碼）＋細篩（精算每個人離震央幾公里）。

function 找出範圍內的人(全部裝置, 震央, 半徑) {
    // ── 粗篩：算出震央那一格的「郵遞區號」前綴 ──
    前綴集 = 算geohash(震央) 連同周圍 8 格   // ← 邊界的人也要撈，寧可多撈不漏人
    // 真實系統拿這串前綴去空間索引秒撈候選；demo 資料少，直接做細篩

    // ── 細篩：對候選逐一算「真正的距離」 ──
    命中 = []
    for (每個 裝置 in 全部裝置) {
        距離 = haversine(震央, 裝置座標)     // 地球兩點實際距離公式
        if (距離 <= 半徑)
            命中.加入(裝置 + { 距離 })         // 在圈內才留下
    }
    回傳 依距離排序(命中)                      // 近的先通知
}

這正是「郵遞區號先框、再精算距離」那兩步。haversine 就是名詞表裡那條「算地球兩點實際距離」的公式。注意粗篩一定要連周圍 8 格一起撈，否則住在格子邊緣的鄰居會被漏掉——寧可多撈，不可漏人。

💭 停一下：既然細篩一定會精算距離，為什麼還要先做「粗篩」這一步？

看答案 因為細篩要對「每個候選」算一次距離公式。如果不先粗篩，就得對全國 5000 萬支手機每支都算一次（太慢）。粗篩用前綴比對先把範圍縮到「震央附近那幾格」的少數人，細篩才只需精算這一小群——這就是把 O(全國) 變成 O(附近)。

2fan-out：一個地震炸成一人一張派工單（NotificationQueue）

對應關卡 3「老師通知全班，分組長分頭傳」。篩出名單後，要把「一個地震事件」展開成「一人一張派工單」排進佇列，工人才能分頭領。

function 展開(地震, 收件人名單) {
    for (每個 收件人 in 收件人名單) {
        抵達秒數 = 收件人.距離 ÷ 3.5         // S波每秒約3.5km，算「還有幾秒」
        派工單佇列.加入({
            冪等鍵: 地震.編號 + "|" + 收件人.裝置ID,  // ← 之後去重就靠這把鑰匙
            裝置ID: 收件人.裝置ID,
            內容: { 規模, 還有幾秒:抵達秒數, 該做什麼:"就地掩護，遠離窗戶" }
        })
    }
}

一個地震 → 名單上每個人各生一張派工單，這就是 fan-out（扇出）：一件事炸開成幾百萬則通知。注意每張派工單都帶一把冪等鍵＝地震編號|裝置ID，下一步去重就靠它。順手把「離震央幾公里」換算成「還有幾秒」塞進通知，這樣使用者收到的是「還有 18 秒」而不是冷冰冰的距離。

3冪等去重：同一個警報只響一次（Dispatcher + Store）

對應怕「漏掉人」的解法：原則是寧可重送也不漏送（at-least-once），但重送不能讓你手機響兩下。所以工人送之前先翻記帳本：

function 投遞(派工單) {
    // ── 冪等去重：這把鑰匙(地震|裝置)已經成功送過？跳過！──
    if (記帳本.已送達(地震編號, 裝置ID))
        return 跳過;                  // ← 同一警報只響一次，重送被擋在這

    結果 = 真的送出去(派工單)          // 真實系統這裡換成呼叫 APNs/FCM
    if (結果成功)
        記帳本.記下("已送達", 地震編號, 裝置ID)   // 寫進帳本，下次重送就會被擋
}

這就是名詞表裡的冪等：上游為了不漏人「至少送一次」（可能重送），投遞層一翻帳本「這把鑰匙已經是已送達」就直接跳過。把「至少一次」修成「剛好一次的體驗」——重送也不會讓你手機響兩下。

4退避重試：送不到先等一下再試，真不行丟回收桶（Dispatcher）

推播商偶爾會暫時送不到（網路抖動）。不能一失敗就放棄（漏人），也不能瘋狂猛重試（把對方打爆）。做法是每次失敗就等久一點再試：

嘗試 = 0
while (嘗試 < 3) {                    // 最多試 3 次
    嘗試++
    if (送出成功) { 標記成功; break }
    // ── 指數退避 + 抖動：等待時間一次比一次長 ──
    等待 = 50毫秒 × 2^(嘗試-1)         // 50 → 100 → 200…一次比一次久
    等待 += random(0, 50)             // 加一點隨機「抖動」，避免大家同時重試
    等一下(等待)
}
if (還是失敗) 丟進DLQ回收桶            // ← 試完還不行：地震時效太短，補送無意義，留著事後檢討

每次失敗等待時間翻倍（50→100→200 毫秒）叫指數退避；再加一點隨機抖動，是怕幾百萬個工人「同時失敗、同時重試」又把推播商打爆。試到上限還不行，就丟進 DLQ（回收桶）——地震警報過了幾秒就沒意義，補送沒用，留著事後檢討為什麼送不到。

關卡 / 怕的事（前面講的）哪個檔關鍵那段
② 挑出附近的人（地理篩選）GeoIndex粗篩前綴 + haversine 細篩

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「閃電與雷聲」解釋，為什麼地震能被預警？ 地震有快而溫和的 P 波（閃電）和慢而破壞的 S 波（雷聲）。震央附近的儀器先抓到 P 波就判定地震，再用光速的網路訊息通知遠方，趕在 S 波（雷聲）抵達前讓人避難。訊息超車地震波。
Q2：為什麼不能「一個一個量距離」找出受影響的人？該怎麼做？ 5000 萬人逐一算距離太慢。改用 geohash（位置郵遞區號）先粗篩出附近網格的人，再對這一小群細篩精算距離。記得連旁邊 8 格一起撈，避免漏掉邊界的人。
Q3：fan-out 是什麼？用班級的比喻說說看。 fan-out = 一個事件同時通知超多人。像老師通知全班不自己一個個打，而是分組長分頭傳；系統把名單切片，交給一大群推播工人同時送。
Q4：為什麼「寧可重送也不漏送」？重送會吵人怎麼辦？ 漏掉一個人可能出人命，所以選「至少送一次」。為了不吵人，用「冪等／去重」讓同一個警報在你手機只響一次。
Q5（工程）：偵測服務通報地震，為什麼我們要「立刻回 202」？ 因為通報的人不該卡著等我們把幾百萬則通知送完。先收下、給個「收到」，後面非同步慢慢展開。像餐廳收單給號碼牌，不讓你站在櫃台等。
Q6（工程）：為什麼裝置簿要「事先存好 geohash」這一欄？ 地震當下才一個個算位置代碼會來不及。平時就把每個人的「郵遞區號」算好寫進去，地震時直接用前綴秒查附近的人——預先算好，換查詢飛快。
Q7（工程）：平時流量幾乎 0，為什麼不能「地震才開機器」？ 開機器、暖機要幾十秒到幾分鐘，但地震只給 10 秒。要平時留小火種待命、瞬間放大，不能從零臨時開。

15. 30 秒回顧

地震預警 = 跟地震賽跑的精準群組廣播。

3 個關卡：① 偵測地震 → ② 用網格（geohash）挑出附近的人 → ③ 一群工人同時推播（fan-out）。
