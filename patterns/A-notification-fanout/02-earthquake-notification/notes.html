② 地震預警系統：偵測到推播的秒級競賽

A 通知/Fan-out 建議 48 分鐘 Design an Earthquake Early Warning System

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂）——用生活比喻把系統拆成 3 個關卡、邊讀邊自我測驗，看完再回來這份工程細節會輕鬆很多。
一句話定位 本質是「極致低延遲的事件偵測 + 地理範圍篩選 + 海量 fan-out 推播」的綜合題。 它跟一般通知系統最大的差別是：跟物理賽跑——破壞性的 S 波正以每秒數公里逼近， 從偵測到使用者手機響起的每一秒都直接換算成「多爭取到幾公里的逃生緩衝」。

1. 需求釐清

功能性需求

  • 從感測器網路（地震儀 / 手機加速度計）即時接收震動讀數，偵測地震發生並估計震央、規模。
  • 計算受影響地理範圍（震央周邊半徑、預估烈度等高線），只通知範圍內的使用者。
  • 對範圍內所有使用者大規模 fan-out 推播（手機警報、簡訊、廣播、IoT 設備）。
  • 推播內容含「預估抵達秒數」倒數，讓使用者就地避難。
  • 事後回放與校正：用更多測站資料修正震央/規模，發補充或取消通知。

非功能性需求

  • 極致低延遲：偵測 → 推播端到端 p99 < 數秒。這是全題的靈魂，所有設計都為它讓路。
  • 不可漏報（高 recall）：寧可偶爾誤報，也不能讓真地震漏掉任何一個受災者。
  • 高可用：天災時系統本身可能斷電斷網，需多區域、多通道冗餘。
  • 地理正確：範圍外的人不該被打擾（誤報疲勞會讓人關掉警報，反而致命）。
  • 突發海量：平時 0 流量，地震瞬間要對數百萬至千萬使用者同時推播（尖峰極陡）。
面試開場必問 「我們負責偵測，還是只負責分發？」——若偵測由官方測震網（如 USGS / 中央氣象署）提供， 我們系統的核心就純粹是「吃事件 → 地理篩選 → 海量 fan-out」（本筆記主軸）； 若連偵測都要做，還得加上感測器接入與信號處理。先釐清，否則範圍會爆炸。

2. 規模估算

指標假設推算
使用者總數全國 5000 萬台裝置地理索引需涵蓋 5000 萬個座標點
地震事件頻率有感地震 ~每日數次平均寫入極低，但尖峰瞬間極高
單次受影響人數都會區地震波及半徑 100km可達 500 萬～1000 萬人需同時推播
推播時間預算S 波 3.5 km/s，要在抵達前送達距震央 70km ≈ 20 秒緩衝，扣偵測耗時剩 ~10 秒
fan-out 吞吐1000 萬人 / 10 秒內送完需 ≥100 萬 推播/秒 的尖峰吞吐
地理篩選延遲從 5000 萬點篩出半徑內者暴力掃描不可行 → 必須空間索引（geohash / R-tree）
關鍵洞察 這題的數字呈現「平時近乎零、尖峰瞬間爆表」的脈衝形狀。設計重點不是穩態 QPS， 而是「冷啟動下的瞬間放大能力」：地理篩選要能毫秒級算出收件人集合，推播管線要能瞬間水平展開到百萬級。

3. API 設計

# ── 上游：偵測服務發布地震事件（內部，高優先級）──
POST /internal/v1/quake-events
Body: { "event_id": "eq_2026...", "epicenter": {"lat":24.15,"lng":121.65},
        "magnitude": 6.2, "depth_km": 12, "detected_at": "2026-05-31T...Z",
        "affected_radius_km": 120, "confidence": 0.93 }
→ 202 Accepted（立刻回，不阻塞；後續非同步 fan-out）

# ── 後續修正 / 取消（用同一 event_id，冪等覆蓋）──
PATCH  /internal/v1/quake-events/{event_id}   Body: { "magnitude": 5.8, "status": "revised" }
DELETE /internal/v1/quake-events/{event_id}   # 誤報 → 發取消通知

# ── 使用者端：註冊裝置與位置（寫，低頻）──
POST /api/v1/devices
Body: { "device_token": "...", "lat": 25.03, "lng": 121.56, "channels": ["push","sms"] }

# ── 使用者端：心跳更新位置（移動中）──
PUT  /api/v1/devices/{device_id}/location   Body: { "lat": ..., "lng": ... }

# ── 推播 payload（送到裝置）──
{ "event_id":"eq_2026...", "intensity":"5強", "eta_seconds": 18, "action":"就地掩護" }

4. 資料模型

devices  (使用者裝置與位置 → 走空間索引)
  device_id     UUID PRIMARY KEY
  device_token  TEXT          -- APNs / FCM 推播 token
  lat, lng      DOUBLE
  geohash       VARCHAR(9)    -- 預先算好，建立空間索引的鍵
  channels      JSONB         -- ["push","sms",...]
  updated_at    TIMESTAMP

quake_events  (地震事件，append + 可修正)
  event_id      VARCHAR PRIMARY KEY    -- 冪等鍵：修正/取消用同一 id
  epicenter     GEOGRAPHY(POINT)
  magnitude     DOUBLE
  affected_radius_km  DOUBLE
  status        ENUM(new, revised, cancelled)
  detected_at   TIMESTAMP

delivery_log  (投遞紀錄，at-least-once 去重用)
  event_id, device_id   COMPOSITE KEY  -- (event_id, device_id) = 冪等鍵
  status        ENUM(queued, sent, failed)
  attempts      INT
為何要 geohash 欄位 地理篩選不能在查詢時對 5000 萬列逐一算 haversine 距離。預先把每台裝置的座標編成 geohash 前綴， 就能用「字串前綴比對」快速框出震央附近的少數網格（候選集），再對候選集做精確距離過濾。空間查詢由 O(N) 降到 O(log N)。 生產上常用 PostGIS（GIST 索引）、Elasticsearch geo_point、或 Redis GEO 指令。

5. 高階架構

 感測器網路 ──▶ 偵測服務 ──▶ 事件匯流排(Kafka) ──▶ 地理篩選服務
 (地震儀/手機)   (估震央/規模)   高優先級 topic        │   ▲
                                                      │   │ 查空間索引
                                                      ▼   │
                                            ┌──────────────────┐
                                            │ 空間索引(geohash) │  ← 裝置位置(預載)
                                            └──────────────────┘
                                                      │ 收件人分區
                                                      ▼
            分區佇列(by region) ──▶ 推播閘道(水平擴展) ──▶ APNs/FCM/簡訊 ──▶ 使用者
            (NotificationQueue)      (Dispatcher)         失敗→退避重試→DLQ
  1. 偵測服務：聚合測站讀數，最快速度判定「地震發生」並估出震央/規模，搶在 S 波抵達前發事件。
  2. 事件匯流排：地震事件走專屬高優先級 topic，與其他通知物理隔離，避免被排隊拖慢。
  3. 地理篩選服務：拿事件查空間索引，毫秒級算出「震央半徑內的裝置集合」，這是 fan-out 的收件人名單。
  4. 分區佇列 + 推播閘道：依 region 分區並行投遞；推播閘道無狀態，可瞬間水平擴展到百萬級吞吐。

6. 核心元件深入

偵測：為什麼能「預警」？P 波 / S 波時間差

地震同時發出快但溫和的 P 波（~6 km/s）和慢但破壞性強的 S 波（~3.5 km/s）。 離震央越遠的測站越晚收到 P 波，但只要近震央的少數測站先偵測到 P 波並判定地震， 就能趕在 S 波抵達遠方城市之前把警報用「光速的網路」送過去——這就是預警能成立的物理基礎。 系統設計的天職，就是把「偵測 + 篩選 + 推播」的總耗時壓到比 S 波傳播還短。

地理篩選：從 5000 萬人毫秒級框出收件人

  • 不可暴力掃描：對全體裝置逐一算 haversine 距離是 O(N)，尖峰下完全不可行。
  • geohash 網格法：把座標編成前綴字串，地理相鄰的點前綴相同。用震央附近的幾個網格前綴撈出候選集（粗篩），再對候選集逐一算精確 haversine 距離細篩。
  • 邊界處理：geohash 在網格邊界會把鄰格的人漏掉，要連同 8 個鄰格一起撈。寧可多撈（recall 優先），細篩會把多餘的剔除。
  • 替代方案：R-tree / PostGIS GIST 索引、Uber H3 六邊形網格、S2 cell。原理相同：用空間索引把 O(N) 降到候選集大小。

海量 fan-out：一個事件炸成千萬則推播

  • 分區並行：收件人名單依 region / geohash 前綴切片，丟進多個分區佇列，由一群推播 worker 並行消費。
  • fan-out on write：地震是「寫一次、瞬間讀放大千萬倍」的極端讀放大場景，必須在事件進來時就主動展開推播（而非等使用者來拉）。
  • 背壓與優先級：地震 topic 永遠插隊在最前面；普通通知（如價格提醒）讓路。

投遞保證：at-least-once + 冪等 + 退避重試

  • at-least-once：寧可重送也不可漏送（漏報會死人）。用 (event_id, device_id) 當冪等鍵，裝置端或閘道去重，避免同一警報響兩次。
  • 重試：推播通道（APNs/FCM）暫時失敗 → 指數退避 + 抖動重試；但地震有時效性，重試視窗很短（過了 S 波抵達時間，補送已無意義，直接進 DLQ 記錄供事後分析）。
  • 多通道冗餘：push 失敗退簡訊、退電視/廣播插播。天災時網路可能壅塞，單一通道不可靠。

7. 擴展與瓶頸（海量 fan-out）

階段 / 環節瓶頸對策
地理篩選對 5000 萬裝置算距離geohash / PostGIS 空間索引，粗篩候選集再細篩
事件分發地震事件被一般通知排隊拖慢專屬高優先級 topic + 資源隔離，地震永遠插隊
推播吞吐單一 worker 送不完千萬則依 region 分區佇列 + 無狀態 worker 大量水平擴展
上游推播商限流APNs/FCM 對單一 app 有速率上限多帳號 / 多通道分流 + 批次推播 API
冷啟動平時 0 流量，地震瞬間要爆量常駐最小資源池 + 預熱 + 可秒級擴容（avoid scale-from-zero 延遲）
災害時自身存活地震本身可能打掉機房多區域部署，跨區冗餘，邊緣節點就近推播

8. 取捨與面試重點

  • 速度 vs 準確：等更多測站資料判得更準，但會慢；地震預警選速度——先用少量測站快速發警報，再用後續資料發「修正/取消」。誤報可接受，漏報不可接受。
  • at-least-once vs exactly-once：選 at-least-once（不漏），用冪等鍵在裝置端去重把「重複」壓到不擾民，而非追求昂貴的 exactly-once。
  • fan-out on write vs on read：本題是極端讀放大、且要主動推，必選 on write（事件來就展開）；on read 不適用（使用者不會主動來「拉地震」）。
  • geohash 精度取捨：網格太粗 → 撈太多人細篩慢；太細 → 邊界漏報風險高、要撈更多鄰格。通常取能覆蓋典型影響半徑的精度，並一律帶上 8 鄰格保 recall。
  • 誤報疲勞：地理篩選做不準 → 沒事也被吵 → 使用者關警報 → 真地震時致命。所以「範圍外的人不該收到」跟「不漏報」一樣是硬需求。
  • 常見追問：
    • 「使用者在移動（開車/搭高鐵）怎麼辦？」→ 心跳更新位置 + 索引近即時刷新；或推播時用粗網格容忍位置誤差。
    • 「怎麼測這種平時不發生的系統？」→ 影子流量 / 定期演練重放歷史地震事件，量端到端延遲。
    • 「如何避免修正通知造成第二次驚嚇？」→ 取消/降級用較溫和通道，警報升級才用強提醒。
    • 「跨國 / 跨時區？」→ 多區域獨立部署，事件按地理路由到對應區域的 fan-out 管線。

9. PHP Demo 說明

本題 demo 刻意不做 CRUD 表單，而是聚焦真正的考點——地理篩選 + fan-out 投遞，可實際用 PHP 內建伺服器跑起來：

  • src/GeoIndex.php　空間索引：用 haversine 距離實作「找出震央半徑 N 公里內的所有裝置」，並附 geohash 粗篩示意（對應真實系統的 PostGIS / Redis GEO）。
  • src/NotificationQueue.php　通知佇列：事件展開成逐裝置的投遞任務（fan-out on write），帶 (event_id, device_id) 冪等鍵入列。
  • src/Dispatcher.php　投遞器：消費佇列、模擬推播（含隨機延遲/失敗）、at-least-once + 指數退避重試 + 冪等去重，多次失敗記 DLQ。
  • src/Store.php　裝置 / 事件 / 投遞紀錄持久化（JSON 檔模擬 DB）。
  • public/index.php　路由：註冊裝置、觸發一次地震、檢視 fan-out 結果（人數、各自距離、投遞狀態與模擬延遲），並附深色測試頁。
# 啟動
cd demo
php -S localhost:8002 -t public
# 瀏覽器開 http://localhost:8002
# 1) 先註冊幾台裝置（或按「載入示範裝置」）
# 2) 設定震央(lat/lng)+規模+半徑 → 觸發地震
# 3) 觀察：誰在範圍內、各自距震央幾公里、投遞狀態與模擬延遲
誠實聲明 Demo 的「推播」是模擬投遞（隨機延遲與失敗，不會真的呼叫 APNs/FCM），偵測服務也以「手動觸發一次事件」代替真實信號處理； 但地理篩選（haversine 半徑查詢）、fan-out 展開、冪等去重、指數退避重試、DLQ 等系統邏輯為真實可執行。 要接真實推播，把 Dispatcher 的模擬投遞換成 APNs/FCM SDK 呼叫即可，其餘邏輯不變（見 demo/README.md）。
