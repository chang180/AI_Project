1. 60 秒搞懂它在幹嘛

想像你在網路上開了一間店，用「綠界 / Stripe」這類金流服務收款。客人刷卡成功的那一刻， 你的後台系統需要立刻知道：「這筆付款成功了！」好去出貨、寄收據。

問題是：金流公司怎麼「通知」你的系統？它不會打電話給你。它的做法是：你事先留一個網址給它， 事件一發生，它就主動打你那個網址，把消息送上門。這個「事件一發生就打你預留網址」的機制，就叫 Webhook。

而 Webhook 平台，就是專門幫一大堆公司做這件「打網址通知」的系統。難在哪？難在——你要通知的對象（別人的伺服器）你完全控制不了： 它可能正好當機、可能很慢、可能回你一個錯誤、可能同一通知收到兩次還搞混。

🔑 一句話 Webhook 平台 = 一個超可靠的「送信員」：事件一發生，就把消息送到對方門口； 對方不在家就過一陣子再來、再不在就放進待領郵箱，而且信封上蓋了防偽鋼印讓對方確認「這真的是我寄的」。

2. Webhook 跟「自己一直刷新」差在哪？

很多人第一個疑問是：「我自己每隔幾秒去問金流『付款好了沒？』不就好了，幹嘛要它來通知我？」這兩種做法有名字——

📦 等包裹：一直查 vs 等通知 你網購了一個包裹。做法 A：每 5 分鐘自己打開物流網站刷新一次「到了沒？到了沒？」——大部分時候都是白問（這叫輪詢 / polling）。 做法 B：留個電話，包裹到了，物流主動打給你——你不用一直盯著（這就是 Webhook）。

Webhook 的好處是即時又省力：事件發生「當下」你就知道，平常也不用一直空跑去問。 代價是：你必須留一個「電話號碼」（公開網址）讓人打得到，而且送信的責任全落在送信方—— 對方沒接到、要不要再打、打幾次，都是 Webhook 平台要扛的。

所以這題的本質不是「怎麼發一個 HTTP 請求」（那很簡單），而是：當對方是個你管不到、隨時會出狀況的人，你要怎麼「把信穩穩送到」？

💭 停一下：如果送信平台「打了一次對方沒回應，就放棄」，會出什麼事？

看答案 對方可能只是剛好重啟了 3 秒，或網路抖一下。如果打一次就放棄，這筆「付款成功」的通知就永遠遺失了——對方系統會以為錢沒付、不出貨，客人就客訴。 所以送信平台一定要會重試：第一次沒接到，過一陣子再打，給對方恢復的機會。這就是後面要講的「指數退避重試」。

3. 把系統拆成 4 個關卡

別被「平台」嚇到。整件事其實就是闖 4 關，一關一關來：

1登記：對方先留下「電話」和「暗號」

客戶（例如某間商店）要先來平台登記：「我的接收網址是 https://我家.com/hook，我想收『付款成功』這類事件。」 平台登記完，會發給對方一組只有你我知道的密碼（secret）。

🔐 比喻：辦門禁卡時約定的暗號 就像你跟管理員約好一個「只有我們倆知道的暗號」。以後我送東西來，會用這個暗號蓋個記號，你一看就知道「確實是他、不是冒充的」。

👉 重點：這組 secret 只在登記當下顯示一次，之後平台只留加密版。雙方各自保管，用來「蓋鋼印 + 驗鋼印」。

2炸開：一個事件 → 通知所有訂閱的人（fan-out）

平台內部發生一個事件（例如 payment.succeeded 付款成功）。可能不只一個客戶訂了這類事件—— 商店要收、會計系統要收、發票系統也要收。

📣 比喻：群組公告 → 拆成一張張個別待辦 你在班群公告一件事，但每個家長都要「個別確認收到」。系統把「一個事件」拆成一張張「送給某某某」的待辦單， 每張單各自獨立——A 送到了不代表 B 也送到了，各送各的、各自重試。

這個「一個事件 → 炸開成很多筆投遞任務」的動作，就叫 fan-out（扇出）。 關鍵是：拆開之後每筆任務命運獨立——這樣某一筆失敗，不會牽連其他筆。

3蓋鋼印：證明「這信真的是我發的」

送信時，對方怎麼知道這封「付款成功」不是壞人偽造的（假裝付款成功騙你出貨）？平台會在信封上「蓋鋼印」。

🏛️ 比喻：公文蓋鋼印 正式公文會壓一個鋼印。鋼印是用「公章 + 公文內容」一起壓出來的，所以：①只有有公章的人壓得出來（防偽造）； ②內容只要被改一個字，鋼印就對不上（防竄改）。

技術上這個鋼印叫 HMAC 簽章：平台拿「剛剛那組 secret 暗號 + 這封信的內容 + 時間」一起算出一串亂碼，貼在信封上。 對方收到後，用同一組暗號照算一遍，對得上 → 確定是你發的、沒被改過。沒有暗號的壞人，永遠算不出對的鋼印。

順帶解決一個壞招：壞人就算偷錄了一封真的信，想「重播」再寄一次騙你（例如重複觸發出貨）怎麼辦？信封上有時間戳， 對方規定「超過 5 分鐘的舊信一律不收」——就像支票有兌現期限，過期作廢。

4送達：對方不在就重試，真的不行就丟「待領郵箱」

送信員按了門鈴（發 HTTP），對方有三種反應：

  • ✅ 回 200「收到」 → 完成，這筆任務結束。
  • ❓ 沒回應 / 回錯誤碼（對方當機、太慢、回 500）→ 等一下再來一次。
  • ❌ 來了很多次都不行 → 別再死纏，把這封信丟進「待領郵箱」（DLQ），通知客戶自己來處理。
📞 比喻：打電話沒人接，間隔越拉越長 你打給朋友沒接，不會每 1 秒狂撥（很煩、也沒用）。你會等 1 分鐘再撥，還沒接等 2 分鐘，再 4 分鐘… 給對方喘息和恢復的時間。撥了很多次真的都沒接，就留言到語音信箱，不再一直撥。

這個「失敗就重試，而且間隔一次比一次久」的策略叫 指數退避（exponential backoff）； 那個放最終失敗訊息的「待領郵箱」叫 DLQ（死信佇列）。

✅ 四關一句話 ① 對方登記網址＋拿暗號 → ② 一個事件炸開成多筆獨立投遞（fan-out）→ ③ 每封信蓋鋼印（HMAC）防偽防竄改 → ④ 送達/重試/丟待領郵箱。就這樣。

4. 設計時最怕的 4 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕四件事，每個設計都是為了避開某個「怕」。

😱 怕「信被偽造或竄改」 如果壞人能假裝平台發「付款成功」，你就會幫他出貨——直接虧錢。所以每封信都要蓋鋼印（HMAC 簽章）， 對方驗不過就丟掉。沒蓋鋼印的信，一律當詐騙。
😱 怕「漏送一封信」 漏掉一筆「付款成功」，對方就以為沒收到錢、不出貨，客人客訴。所以系統寧可重複送（叫 at-least-once，至少送一次）。 那重複送、對方收到兩封一樣的怎麼辦？信上帶一個固定編號，對方看到重複編號就只處理一次（叫冪等去重）。寧可重送，不可漏送。
😱 怕「一個慢客戶拖垮所有人」 假設 A 客戶的伺服器超慢，每封信都要等 30 秒才回。如果送信員大家擠在同一條隊伍，那 A 一卡住，後面 B、C、D 的信全部跟著塞住、送不出去。 所以要每個客戶各排各的隊，A 慢只塞到 A 自己。慢的人請自己慢，別連累別人。
😱 怕「重試反而把對方打爆」 對方剛從當機恢復，結果你累積的幾萬封信瞬間全衝過去，又把它打掛了（叫「驚群」）。 所以重試要間隔越拉越長 + 加一點隨機（抖動），把信錯開、不要同一秒全到。救人別把人壓死。

工程術語白話解釋
Webhook「事件一發生，就主動打你預留的網址通知你」——像物流到貨主動打電話給你
輪詢 (polling)反過來：你自己一直去問「好了沒？」——像一直刷新物流網站
端點 (endpoint)對方留給你的「接收網址 / 電話號碼」
fan-out（扇出）一個事件 → 炸開成很多筆「送給某某某」的獨立任務
HMAC 簽章用「暗號 + 信件內容」壓出的鋼印：防偽造、防竄改
secret（密鑰）只有平台和對方知道的「暗號」，用來蓋鋼印 / 驗鋼印
防重放 (replay)信上帶時間戳，過期的舊信不收——像支票有兌現期限
at-least-once「至少送一次」：寧可重送也不漏送
冪等 / 去重同一個編號的信只處理一次，重複收到不會做兩遍
指數退避 + 抖動重試間隔一次比一次久（1s→2s→4s），再加點隨機錯開，避免同時打爆對方
DLQ（死信佇列）怎麼送都送不到的信，丟進的「待領郵箱」，事後人工/自動處理
端點隔離 / head-of-line blocking每個客戶各排各的隊，慢的客戶不拖垮其他人
熔斷器 (circuit breaker)對方一直掛，就先「跳電不打了」，過陣子再試探它好了沒
SSRF客戶亂填內網網址想騙平台幫他打內部系統——要擋掉
QPS每秒幾筆請求（衡量「多忙」的單位）

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每天多少事件1 億筆/日 ≈ 每秒 1000 多筆收件入口要扛得住、且要立刻回「收到」
fan-out 放大幾倍每事件平均 1.5 個訂閱者真正要「送出去」的信比事件還多
重試再放大約 1 成需重試、平均重 2 次送信工人實際工作量再 ×1.2 左右
要存多少投遞記錄每筆嘗試都記一行，存 30 天 ≈ 幾 TB要用「很會寫、會自動過期」的資料庫
🐢 關鍵洞察：瓶頸不是你，是「對方多慢」 這系統最特別的一點：拖垮你的不是你自己的 CPU，而是「對方的伺服器有多慢、多會掛」。 就像送信員本身跑很快，但收件人開門慢吞吞，整體速度被收件人決定。 所以整個設計的核心，就是「把這些不受控的『慢』和『掛』，隔離成一個個不會互相拖累、而且可以重試的小任務」。

💭 停一下：既然瓶頸是「等對方回應」，為什麼「一個送信工人一次只處理一封信、等它回完再拿下一封」會是大災難？

看答案 因為只要碰到一個慢端點（等 30 秒），這個工人就整整 30 秒只能乾等，什麼都做不了，後面排隊的信全卡住。 正解是：①很多工人併發同時送不同的信；②每個客戶分開排隊，慢的只塞自己那隊；③重試不要原地等，而是把「下次幾點再試」記下來、先去處理別的，時間到了再回來。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題有兩類「點餐單」：客戶來操作平台的，和平台送給對方的。

① 客戶登記端點（拿到 secret 暗號，只給一次）
   POST /api/v1/endpoints
   給我：{ 你的接收網址, 想訂哪些事件 }
   我回：{ 端點編號, secret暗號 }  ← 之後拿來蓋鋼印

secret 為什麼「只顯示一次」？因為它是暗號，平台之後只留加密版，不留明文——萬一平台被偷看資料庫，也偷不到能偽造鋼印的暗號。

② 內部發布事件（觸發 fan-out）
   POST /api/v1/events
   給我：{ 事件類型, 內容, idempotency_key 編號 }
   我回：202「收到了，我會去送」← 立刻回，不讓你等

為什麼「立刻回 202」？因為「產生事件」和「把信送到對方」是兩件事，要解耦。上游發完事件就該走人，不該卡著等我們慢慢送幾筆、還可能重試很久。先收下，後面非同步慢慢送。像餐廳收單給號碼牌，不讓你站在櫃台等餐。

③ 客戶查投遞記錄 / 手動重送
   GET  /api/v1/endpoints/{id}/deliveries  → 看每筆送了沒、重試幾次、對方回幾號
   POST /api/v1/deliveries/{id}/redeliver  → 把待領郵箱(DLQ)裡的信再送一次
④ 平台「送給對方」的信長這樣（對方要驗這個）
   POST https://對方.com/hook
   信封上貼：
     X-Webhook-Timestamp: 寄出時間（防重放）
     X-Webhook-Signature: 鋼印 = HMAC(暗號, "時間.內容")
     Idempotency-Key:     固定編號（對方據此去重）
   內容：{ 事件類型, 資料 }
   對方做：驗鋼印 + 檢查時間沒過期 + 看編號沒處理過 → 回 200

注意鋼印是用「時間 + 內容」一起算的：所以改時間或改內容，鋼印都會對不上——這同時擋掉了偽造、竄改、和重播舊信三種招數。

9. 系統要記住哪些事？（資料模型）

📒 比喻：三本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題只要三本：

🏪端點簿（誰、網址、暗號）

每個登記的客戶一列：端點編號、客戶、接收網址、secret暗號(加密存)、訂了哪些事件、狀態(啟用/停用)。

為什麼有「狀態」這欄？ 如果某個端點連續失敗太多次（網址早就廢了），平台可以把它自動標成「停用」，不再浪費力氣一直送，並通知客戶「你的網址壞了」。

📨投遞簿（每一封信送得如何）

fan-out 後，每封信一列：投遞編號、給哪個端點、事件編號、狀態(待送/送出中/已送達/重試中/死信)、重試幾次、對方最後回幾號、下次幾點再試。

這本其實是一台「狀態機」 一封信的命運就是在這幾個狀態間移動：待送 → 送出中 →（成功）已送達 ／（失敗）重試中 → …超過上限 → 死信。 把「下次幾點再試」也記下來，工人就不用原地乾等，時間到再回來拿。
⚠️ 這本寫得超頻繁（每次嘗試都記一行），所以要用「很會寫、能自動過期(TTL)」的資料庫，並批次寫入。

🗑️死信簿（DLQ：怎麼送都送不到的）

重試到上限還是失敗的信，整封丟進這裡：投遞編號、端點、事件編號、信的內容、失敗時間、失敗原因。 保留內容，供客戶事後手動重送或自動排程重送。

為何「事件」和「投遞」要分開記？ 一個事件 fan-out 成 N 封信，每封信獨立重試、獨立進死信——A 失敗不該牽連 B。 而那個貫穿全程不變的事件編號（idempotency_key），正是對方用來「同一封只處理一次」的鑰匙。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
大家擠同一條隊，一個慢客戶卡住全部每個客戶各排各的隊＋每個客戶限制同時送幾封（端點隔離）
一群端點同時恢復，重試瞬間全衝（驚群）重試間隔越拉越長 + 加隨機抖動，把信錯開
對方一直掛，還傻傻一直送裝熔斷器：先「跳電不打」，過陣子才試探它好了沒
一個送信工人送不完那麼多信多請一大群工人同時送（水平擴展）
投遞記錄寫到爆（每次嘗試一行）用「很會寫」的資料庫 + 批次寫 + 設定過期自動刪
客戶填了內網網址想騙平台幫他打內部系統(SSRF)送之前檢查網址，封鎖私有 IP / 走專用對外代理

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的四個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 至少一次 vs 剛好一次 網路上「已送達但對方的『收到』回應在路上弄丟了」，你分不出來到底送到沒，只好再送一次。所以實務上選「至少一次」（簡單可靠）， 「剛好一次」的責任交給對方用編號去重。別承諾你做不到的「剛好一次」。
⚖️ 保順序 vs 衝吞吐量 要保證「同一個客戶的事件按順序到」，就得一封一封排隊送，速度變慢。多數場景其實不用嚴格保序—— 讓信自帶時間戳/版本號，對方自己判斷新舊就好，這樣可以多人併發、更快。
⚖️ 原地等著重試 vs 排程重試 失敗了就原地 sleep 5 秒再試？這會把工人卡住 5 秒什麼都不做。正解：把「下次幾點再試」記到投遞簿，工人先去忙別的，時間到了再回來拿——工人永遠不空轉。
⚖️ 推（webhook）vs 拉（polling） Webhook 即時，但要求對方有公開網址、且送信責任全在我方。對特別不可靠的對象，可以額外提供「事件補拉 API」當後備：對方也能自己回來查漏掉的事件。
💭 追問：怎麼同時做到「不漏送」又「不讓對方收到重複的兩封」？ 平台這邊寧可重送（at-least-once）保證不漏；對方那邊靠每封信帶的固定編號（idempotency_key）去重——第一次處理完就把編號記起來，之後看到一樣的編號直接回 200、不再處理。兩邊配合，才能既不漏又不重複做。
💭 追問：對方的網址持續掛了好幾天，平台該怎麼辦？ ①重試到上限就丟DLQ 待領郵箱不再自動重試；②對持續失敗的端點開熔斷器，先不浪費力氣狂送，只偶爾試探；③連續失敗過多就自動把端點停用並通知客戶「你的網址壞了，修好再來開」。

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「簽章 + 重試退避 + DLQ」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8008 -t public
# 然後瀏覽器打開 http://localhost:8008
  1. 先註冊一個「會成功」的端點，再註冊一個「會失敗」的端點。
  2. 發布一個事件 → 看它自動 fan-out 成多筆投遞任務入列。
  3. 反覆按「觸發投遞」→ 觀察那個「會失敗」的端點：重試間隔越拉越長，最後超過上限進 DLQ。
玩的時候想一下看那筆失敗任務的 next_retry_at（下次幾點再試）是不是每次間隔都變長？這就是指數退避。 把端點指向內建的 /receive 測試接收端，它會驗你的鋼印（HMAC）才回 200——故意改一下 secret，看它是不是就驗不過、被擋掉了，這就是簽章防偽。
小提醒為了讓單執行緒的 PHP 內建伺服器不會「自己打自己」卡死，demo 預設用「模擬投遞結果」（依端點設定直接判定成功/失敗/慢）， 但狀態機、HMAC 簽章與驗證、指數退避+抖動、重試上限、DLQ 都是真的在跑。真實系統會用一群獨立的送信工人並發、非阻塞地送。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有四個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 四個檔 = 四個小幫手 EndpointStore（登記簿：誰訂了哪些事、各自的暗號 secret）→ DeliveryQueue（投遞簿＝那台會記「下次幾點再試」的狀態機，也兼待領郵箱 DLQ）→ Signer（蓋鋼印的人：算 HMAC 簽章、也負責驗章）→ Dispatcher（送信工人：取信、蓋章、送、依結果決定成功/重試/丟 DLQ）。事件一發布，就照這個順序跑一遍。

1蓋鋼印：HMAC 簽章（Signer）

每封信送出前都要蓋一個鋼印，對方才知道「這真的是平台寄的、而且路上沒被改」。鋼印是用「暗號(secret) ＋ 時間戳 ＋ 信件內容」一起壓出來的：

function 蓋章(暗號, 信件內容) {
    時間戳 = 現在時間
    // 關鍵：把「時間戳.信件內容」整串拿去壓鋼印
    鋼印 = HMAC_SHA256(暗號, 時間戳 + "." + 信件內容)
    return {
        "X-Webhook-Timestamp": 時間戳,
        "X-Webhook-Signature": "sha256=" + 鋼印,   // ← 隨信附上
    }
}

就像公文蓋鋼印：沒暗號的壞人壓不出對的鋼印（防偽造）；信件內容改一個字，鋼印就對不上（防竄改）。為什麼連時間戳也一起壓？這樣攔截到的舊信會「過期」，壞人沒辦法把它一直重放。

對方收到信後，用同一組暗號照算一遍，對得上才收：

function 驗章(暗號, 信件內容, 來的時間戳, 來的鋼印) {
    if (現在 - 來的時間戳 > 5 分鐘) return 拒收;   // ① 太舊 → 防重放
    應有鋼印 = "sha256=" + HMAC_SHA256(暗號, 來的時間戳 + "." + 信件內容)
    return 常數時間比較(應有鋼印, 來的鋼印)          // ② 對得上才收
}

注意「常數時間比較」（程式裡的 hash_equals）：不是普通的字串相等，而是不管對幾個字都花一樣久，避免壞人靠「猜得越對、回應越慢」一點一點試出鋼印（時序攻擊）。

💭 停一下：如果鋼印只壓信件內容、不壓時間戳，會怎樣？

看答案 壞人只要攔截到一次合法的信，就能把「內容＋鋼印」原封不動無限重放——因為鋼印永遠對得上。把時間戳一起壓進去後，舊信會被「5 分鐘視窗」擋下，重放就失效了。

2沒人接？間隔越拉越長再撥：指數退避 + 抖動（Dispatcher）

信沒送到（對方回 5xx 或逾時），不要馬上狂撥把剛恢復的對方又打掛。等待時間要一次比一次久，還要加一點隨機：

function 算等多久(第幾次嘗試) {
    基礎 = 1秒 × 2^(第幾次 - 1)     // 1、2、4、8、16… 一次比一次久
    抖動 = 0 到 基礎 之間的隨機數    // ← 打散「驚群」，別大家同一秒一起撥
    return 基礎 + 抖動
}

像打電話沒人接，就隔久一點再撥，越撥不通隔越久。抖動（隨機）是怕一大堆失敗的信「同一秒」全衝過去（驚群效應），把它們錯開到不同時間點。

算好之後，不是原地 sleep 等，而是把「下次幾點再試」記進投遞簿，工人先去忙別的：

等待 = 算等多久(這封信.嘗試次數)
這封信.狀態 = RETRY
這封信.下次幾點再試 = 現在 + 等待       // ← 只記下時間，工人不空等
存回投遞簿(這封信)

關鍵就是「記時間、不空等」：送信工人永遠不會被一封信卡住空轉，時間到了，那封信自然會被「現在可投」的掃描重新撈出來。

3撥到上限就丟待領郵箱：DLQ（Dispatcher + DeliveryQueue）

不能無限重試（對方網址可能早就廢了）。撥超過上限還送不到，就把信丟進待領郵箱（DLQ）保留起來，並順手停用這個一直掛的端點：

// 送出後拿到回應碼，依結果決定下一步（狀態機）
if (回應碼 是 2xx) {
    這封信.狀態 = DELIVERED            // 成功，收工
} else if (這封信.嘗試次數 >= 6) {     // ← 達重試上限
    這封信.狀態 = DEAD                 // 丟進 DLQ（待領郵箱）
    停用端點(這個端點)                  // 連續掛 → 自動停用，別再浪費資源
} else {
    這封信.狀態 = RETRY                // 還沒到上限 → 走上面第②關退避重試
}

DLQ 就是「待領郵箱」：送不到的信不直接丟掉，而是收進一個專門的箱子保留，供事後人工或自動重送。要拿全部死信，就是把投遞簿裡狀態是 DEAD 的撈出來：

function 看待領郵箱() {
    return 投遞簿裡「狀態 == DEAD」的所有信   // 這就是 DLQ
}

💭 停一下：為什麼要設「最多撥 6 次」這個上限，而不是一直撥到通為止？

看答案 因為對方網址可能早就廢了，一直撥只是白白浪費資源，還會擠佔正常任務的工人。設上限 → 丟 DLQ → 停用端點 → 通知客戶，比無止境空撥務實得多。

4各客戶各排各的隊：端點隔離（DeliveryQueue 的分區鍵）

一個慢客戶不該拖垮所有人。秘訣在入列時：每筆投遞都記住它是哪個端點的，這個 endpoint_id 就是分區鍵——讓「同一個端點的信走自己那一格」：

function 入列(事件編號, 事件內容, 端點編號) {
    新任務 = {
        投遞編號:   隨機產生,
        事件編號:   事件編號,        // = 冪等鑰匙（對方據此去重，重送也只算一次）
        端點編號:   端點編號,        // ← 分區鍵：同端點的信排同一格，彼此隔離
        狀態:       PENDING,
        嘗試次數:   0,
        下次幾點再試: 0,             // 0 = 立即可投
    }
    存進投遞簿(新任務)
}

有了「每個端點各排各的隊」，一個慢/掛的客戶只會塞住它自己那一格，不會卡到其他人的信——這就是把前面講的「隊頭阻塞」隔離掉。順帶一提：事件編號 全程不變，當作對方去重的冪等鑰匙，所以就算「寧可重送」送了兩次，對方也只當一次。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 蓋鋼印防偽防竄改Signer蓋章() / 驗章()（HMAC + 時間戳）
② 沒人接就隔久再撥Dispatcher算等多久()（退避 + 抖動）
③ 撥到上限丟待領郵箱Dispatcher / DeliveryQueue狀態機 → DEAD / 看待領郵箱()
④ 各客戶各排各隊DeliveryQueue入列() 的 端點編號（分區鍵）

👉 重點：真實系統會把「JSON 檔 + 迴圈掃描」換成 Kafka / SQS（依 endpoint 分區）＋ 延遲佇列承載 next_retry_at ＋ 一群獨立的送信工人並發地真正發 HTTP（demo 為了不讓單執行緒伺服器自己打自己卡死，預設用「模擬投遞結果」）。 但上面這幾段判斷邏輯——HMAC 簽章、指數退避、DLQ、端點隔離——一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「等包裹」的比喻，說說 Webhook 跟 polling（輪詢）差在哪？ polling 是你一直自己刷新「到了沒？」，大多白問又費力；Webhook 是包裹到了，物流主動打給你——即時又省力，代價是你要留一個打得到的公開網址，而送信責任全在送信方。
Q2：用「公文鋼印」解釋，HMAC 簽章怎麼同時防「偽造」和「竄改」？ 鋼印是用「暗號(secret) + 信件內容」一起壓出來的。沒暗號的壞人壓不出對的鋼印（防偽造）；內容只要改一個字，鋼印就對不上（防竄改）。對方用同一組暗號照算一遍，對得上才收。
Q3：為什麼重試要「間隔一次比一次久」，還要加隨機（抖動）？ 間隔拉長（指數退避）是給剛恢復的對方喘息，別狂撥把它又打掛；加隨機抖動是怕一大堆失敗的信「同一秒」全衝過去（驚群），把它們錯開、分散到不同時間點。
Q4：DLQ（死信佇列 / 待領郵箱）是幹嘛的？為什麼不要無限重試？ 重試到上限還送不到的信，丟進 DLQ 保留起來，供事後手動/自動重送。不無限重試是因為對方可能網址早就廢了，一直送只是浪費資源，還可能干擾正常任務；丟 DLQ 並通知客戶才務實。
Q5（工程）：發布事件時，為什麼平台要「立刻回 202」而不是等送完才回？ 因為「產生事件」和「把信送到對方」是兩件事，要解耦。送信可能很久還要重試，不該卡著上游等。先收下回 202，後面非同步慢慢 fan-out、慢慢送。像餐廳收單給號碼牌，不讓你站櫃台等。
Q6（工程）：一個慢客戶為什麼會拖垮所有人？怎麼解？這現象叫什麼？ 叫 head-of-line blocking（隊頭阻塞）。若所有信擠同一條隊/同一池工人，慢客戶佔住連線，後面所有人的信都卡住。解法：每客戶各排各的隊 + 每客戶限制併發 + 對持續掛的端點開熔斷器，把慢隔離在它自己那一格。
Q7（工程）：失敗重試，為什麼「原地 sleep 等一下再試」是壞做法？正解是什麼？ 原地 sleep 會把這個送信工人卡住空轉，那段時間什麼都不能做。正解是把「下次幾點再試（next_retry_at）」記進投遞簿，工人先去處理別的任務，時間到了再回來拿——工人永遠不浪費。

15. 30 秒回顧

Webhook 平台 = 一個超可靠的送信員：事件一發生就把消息送到對方門口，蓋了鋼印、對方不在就改天再來。

4 個關卡：① 對方登記網址＋拿暗號 → ② 一個事件炸開成多筆獨立投遞(fan-out) → ③ 每封信蓋鋼印(HMAC 防偽防竄改) → ④ 送達/重試/丟待領郵箱(指數退避＋DLQ)。

4 個怕：怕信被偽造（蓋鋼印）、怕漏送（至少一次＋對方去重）、怕一個慢客戶拖垮全部（每客戶各排各隊）、怕重試把對方打爆（退避＋抖動）。

工程細節一句話：瓶頸是「對方多慢」不是你 → 收到事件先回 202 解耦 → 三本筆記本（端點/投遞/死信），投遞那本是台狀態機 → 把不受控的慢與掛，隔離成一個個可重試、不互相拖累的小任務。
