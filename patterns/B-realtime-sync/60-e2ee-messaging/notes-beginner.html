1. 60 秒搞懂它在幹嘛

你用 WhatsApp / Signal 傳訊息給朋友。中間經過電信、伺服器、一堆網路設備。 「端到端加密」（End-to-End Encryption，E2EE）保證的就是：只有你和你朋友兩端能讀這則訊息， 中間經手的所有人（包含提供服務的公司自己）都看不到內容，只看得到一團亂碼。

🔑 一句話 端到端加密 = 把信鎖進只有你和收件人各有鑰匙的保險箱再交給郵差。 郵差（伺服器）負責送，但打不開箱子；就算他想偷看、被搶、被法院要求交出，拿到的也只是一個打不開的箱子。

2. 為什麼這題的難點不是「加密」本身？

很多人以為：「加密不就是把字變成亂碼嗎？很簡單啊。」——加密演算法本身確實是現成的。 真正難、面試真正考的是這兩件事：

📌 難關一：怎麼「不見面」就講好同一把鑰匙？ 你要鎖保險箱、朋友要能開，你們得用「同一把鑰匙」。可是你們從沒見過面， 中間只有一個你不信任的郵差能傳東西——你怎麼把鑰匙交給朋友，又不被郵差偷拷貝一份？ 這就是「金鑰交換」難題。答案很神奇：有一種數學魔術（Diffie-Hellman），讓你們各自公開喊一個數字， 就能各自算出「同一把只有你倆知道的鑰匙」，郵差全程在聽也算不出來。
📌 難關二：萬一鑰匙哪天被偷，以前的信也要保住 如果你和朋友永遠只用同一把鑰匙，那只要這把鑰匙哪天外洩，過去所有訊息就全被解光了。 好的設計要做到：每封信用一把不一樣的鑰匙，而且用完就「燒掉」、推進到下一把，舊的再也算不回來。 這樣就算今天的鑰匙被偷，昨天的信還是安全的。這個「鑰匙不斷往前推、舊的銷毀」的機制叫棘輪。

💭 停一下：為什麼「永遠用同一把鑰匙」很危險？

看答案 因為一旦那把鑰匙外洩（手機被駭、被搶、金鑰被截），攻擊者就能把過去到未來所有訊息一次全解開。 把鑰匙「一封一封換、用完就銷毀」之後，洩漏一把只會影響那一封，過去的信仍解不開——這叫「前向保密」。

3. 把系統拆成 4 個關卡

別被「密碼學」嚇到。整件事其實就是闖 4 關，一關一關來：

1交換鑰匙：不見面也能講好同一把鑰匙

你和朋友從沒見面，要靠不信任的郵差協商出「同一把只有你倆知道」的鑰匙。

🎨 比喻：混顏料 你和朋友各先選一個「祕密顏色」（私鑰，不告訴任何人）。大家公開一個「共同底色」。 你把底色＋自己的祕密色混好，公開寄給對方（這是公鑰，混過的顏色很難反推回原色）；對方也寄混色給你。 最後你拿「對方的混色」再加上「自己的祕密色」，朋友也對應地做一次——你倆會調出一模一樣的顏色， 但偷看的郵差只看到兩坨混色，調不出你們最後那個顏色。這就是 Diffie-Hellman 金鑰交換。

👉 算出共同顏色後，再「磨」一下（這步叫 KDF，把共享祕密整理成正式的鑰匙）就能拿來鎖箱子了。

2加密：把信鎖進保險箱

有了共同鑰匙，把訊息（明文）用它鎖成亂碼（密文）。

🔒 比喻：上鎖的箱子 明文「晚上 7 點老地方見」用鑰匙一鎖，變成一串亂碼「9f3a7c…」。 只有同一把鑰匙能解回原文；沒鑰匙的人看到的永遠是那串亂碼，毫無意義。

3棘輪：每封信換一把鑰匙、用完就燒

這是全題的心臟。不要永遠用同一把鑰匙，而是每傳一封就把鑰匙往前推一格、舊的銷毀。

⚙️ 比喻：棘輪（只能往前轉的齒輪） 工具裡的棘輪只能單方向轉、轉過去就回不來。鑰匙也一樣：每封信用完，就用一個「單向函式」把它推進成下一把， 從新鑰匙絕對推不回舊鑰匙。所以就算今天手機被偷、拿到現在的鑰匙，也解不開昨天那封（鑰匙早被燒掉了）。

這個「鑰匙一直往前推、舊的算不回來」就是前向保密。真實的 Signal 還會再疊一層（定期重新交換鑰匙），讓「萬一被偷後，未來也能恢復安全」——叫後向保密。兩層合起來叫 Double Ratchet（雙棘輪）。

4郵差只送箱子：伺服器看不到內容

加密好的密文交給伺服器，由它存轉（對方離線就先暫存、上線再給）。但伺服器只經手箱子，沒有鑰匙。

📮 比喻：不識內容的郵差 郵差負責把保險箱送到，途中可以暫放郵局（離線暫存）。但他沒有鑰匙， 打不開、也不知道裡面寫什麼。這跟一般聊天 App（伺服器看得到你打什麼）最大的不同。

👉 這一關沿用 ⑦ Messenger 的存轉與離線投遞——架構幾乎一樣，差別只在「送的是密文」。

✅ 四關一句話 ① 混顏料交換出同一把鑰匙（DH）→ ② 用鑰匙把信鎖成密文 → ③ 棘輪（每封換鑰匙、用完就燒 = 前向保密）→ ④ 郵差只送箱子（伺服器只見密文）。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「郵差被換成假的」（中間人） 最致命的怕。如果交換鑰匙時，郵差偷偷把你朋友的公鑰換成自己的，那你其實是在跟「冒牌朋友（郵差）」講好鑰匙， 郵差就能解開、看完、再轉給真朋友——你毫不知情。對策：見面或視訊時互相核對一串「安全碼」（公鑰指紋），對上了才確定中間沒被掉包。
😱 怕「一把鑰匙被偷、全部訊息陪葬」 如果永遠一把鑰匙，洩漏一次＝過去未來全洩。所以要用棘輪：一封一把、用完就燒， 讓「洩漏的損失只限那一封」。把雞蛋分到很多籃子，還隨手砸掉空籃子。
😱 怕「以為加密就全隱形了，其實還露餡」 加密只藏了內容。但「你幾點跟誰聊、聊多頻繁」這些中繼資料，伺服器還是看得到。 很多人以為 E2EE = 完全隱形，其實不是。面試講到這點會大大加分（誠實面對極限）。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前你會先估「大概來幾人」，才決定訂幾個便當、租多大場地。系統設計也一樣：先估數字，才知道要準備多大的「料」。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
有多少裝置2 億人 × 1.5 台 ≈ 3 億把鑰匙對每台都要管自己的私鑰與棘輪狀態（在裝置上，不在伺服器）
一次性鑰匙（prekey）每台先存 100 把 ≈ 300 億把讓對方離線也能先建鑰匙；用完要補
每天多少則訊息每人 40 則 ≈ 每秒 9 萬筆（峰值 28 萬）跟一般 Messenger 差不多；瓶頸還是「送」而非「加密」
加密多花多少每則多附一點點鑰匙資料密文比明文稍大，可接受；計算在裝置端，不壓伺服器
💡 關鍵洞察：累的是「鑰匙管理」，不是「加密運算」 加密本身很快、又發生在使用者的手機上，不會壓垮伺服器。E2EE 真正新增的負擔是 金鑰管理：身分鑰匙、一次性 prekey 的發放與補充、每則訊息的棘輪狀態。送密文那部分，沿用 Messenger 即可。

💭 停一下：為什麼說「加密不會壓垮伺服器」？

看答案 因為加解密都在使用者裝置端做（手機算自己的就好），伺服器只負責存轉密文， 跟存轉明文的成本幾乎一樣。E2EE 增加的成本主要在「鑰匙的發放、領取、補充」這種金鑰管理工作。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼、我回你什麼。講好格式雙方就能合作。

這題只要記住幾張「點餐單」：

① 上傳我的公鑰（裝置產生私鑰，只上傳公鑰）
   POST /api/v1/keys
   給我：{ 身分公鑰, 一批一次性公鑰(prekey) }
   我回：201「收到！這是你的安全碼（指紋），請和朋友核對」

重點：上傳的是公鑰，私鑰永遠留在手機裡。伺服器握有再多公鑰也解不開訊息。

② 領對方的公鑰（要開始聊天前）
   GET /api/v1/keys/{對方}
   我回：{ 對方身分公鑰, 一把一次性公鑰 }   ← 領走那把一次性的就用掉了
③ 送密文（伺服器只中繼，看不到內容）
   POST /api/v1/messages
   給我：{ 寄給誰, 棘輪序號 n, 密文, 鑰匙材料 header }
   我回：201（存著、對方上線就轉給他）
④ 收密文（自己手機解密）
   GET /api/v1/inbox?after_id=120
   我回：[ { 誰寄的, n, 密文, header }, ... ]   ← 手機端用棘輪鑰匙解開

注意 ③④ 經手的都是密文。伺服器這份「點餐單」永遠收不到明文。

9. 系統要記住哪些事？（資料模型）

📒 比喻：東西分兩個地方放 這題最關鍵的設計就是「東西分兩半放」：能公開的放伺服器，祕密的只放手機。

🖥️伺服器只放兩本（都是公開或亂碼）

公鑰目錄：每人的公鑰束（給人來領、算共同鑰匙用）。
密文信箱：每則只有 寄給誰、棘輪序號 n、密文——沒有明文、沒有鑰匙。 所以伺服器被駭 / 被法院要資料，交得出的只有一堆亂碼。

📱手機端才放祕密（永不上傳）

身分私鑰（自己的祕密色）＋棘輪狀態（每個對話的發送鏈鑰匙、接收鏈鑰匙、推到第幾格）。 只要這些不離開手機，伺服器無論如何都解不開訊息。

為什麼一定要分兩半？ 因為 E2EE 的整個安全保證，就建立在「私鑰只有你手機有」這一點上。 一旦私鑰上了伺服器，伺服器就能解密，E2EE 立刻破功。所以設計上把它物理隔離在裝置端。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
對方離線，沒辦法當場「混顏料」交換鑰匙一次性預付金鑰（prekey）：先存一批公鑰在伺服器，對方離線你也能領一把先把鑰匙建好
一次性鑰匙被領光了裝置上線時補一批新的上去；真的用完先退回用備援鑰匙
群組聊天：N 個人兩兩配對鑰匙會爆炸用「群組鑰匙（sender key）」：每人發一把給群裡所有人，群播只加密一次
多裝置：手機、平板、電腦鑰匙要對齊每台各有自己的鑰匙，送訊息時對每台各加密一份；新裝置要重新驗證
送密文的長連線/投遞量大直接沿用 ⑦ Messenger 那套（連線註冊表 + 離線佇列），只是內容換成密文

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 隱私 vs 伺服器端功能 加密後伺服器看不到內容 → 全文搜尋、垃圾訊息過濾、內容審核、防詐騙、雲端備份全都變難或做不到。 這是 E2EE 最大的代價：保護了隱私，就失去了很多「伺服器幫你做事」的功能。
⚖️ 換手機怎麼辦？備份 vs 安全 私鑰只在手機 → 手機丟了 / 換新機，舊訊息可能永遠解不開。要做「雲端鑰匙備份」會多一個被攻擊的入口； 不做則使用者體驗差。怎麼折衷（例如用一組強密碼加密備份）是個經典難題。
⚖️ 加密藏得了內容，藏不了「中繼資料」 「你幾點跟誰聊、多頻繁、從哪個 IP」這些伺服器還是看得到。要再藏需要額外設計（隱藏寄件人、混淆路由），成本很高。
💭 追問：伺服器被法院要求交出某人的對話，能交出什麼？ 只能交出密文和中繼資料（誰跟誰、何時聊），交不出明文——因為伺服器根本沒有鑰匙。 這正是 E2EE 的賣點，也是它常被監管討論的原因。
💭 追問：怎麼確定「對方公鑰」沒被郵差掉包（中間人）？ 雙方核對一串安全碼（公鑰指紋）——當面掃 QR 或唸數字對一遍。 對得上，代表你拿到的就是真朋友的公鑰，中間沒被換掉。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「金鑰交換 + 棘輪 + 伺服器只見密文」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8060 -t public
# 然後瀏覽器打開 http://localhost:8060
  1. 按「① A、B 建立加密 session」：看兩個裝置各自用 DH 算出一模一樣的共享祕密（畫面顯示 ✅ 兩邊一致）。
  2. 在 A 輸入一句話「加密送給 B」：看「伺服器視角」面板——它只收到一串密文，看不到你打的字。
  3. 在 B 按「收訊息並解密」：B 端用鑰匙解出原文。每送一則，棘輪序號就 +1（鑰匙往前推）。
  4. 按「② 示範前向保密」：試著用「已經推進過、被燒掉的舊鑰匙」去解舊訊息 → 解不開。
玩的時候想一下盯著「伺服器視角」面板：不管你打什麼，它永遠只有密文——這就是端到端加密。 再看「棘輪狀態」：每收發一則，send_n / recv_n 就 +1、鑰匙指紋變動——這就是「每封換一把鑰匙」。最後試前向保密，體會「舊鑰匙燒掉就救不回來」。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心邏輯很短，剛好對應前面講的關卡。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 對應前面的關卡 ① 混顏料交換鑰匙（DH）→ ② KDF 磨成鑰匙 → ③ 棘輪推進 + 加解密 → ④ 伺服器只存密文。

1混顏料：算出「同一把鑰匙」（DH）

公開一個底色 g、一個大數 p。每人選祕密色（私鑰），算出混色（公鑰）寄出去：

公鑰 = g 的 (私鑰) 次方，再對 p 取餘數     // 混色：很難反推回私鑰

// 雙方交換公鑰後，各自算共同鑰匙：
共享祕密 = (對方公鑰) 的 (自己私鑰) 次方，對 p 取餘數
// 神奇：A 和 B 算出來「一模一樣」，但偷聽的郵差只看到兩個公鑰，算不出共享祕密

這就是「混顏料魔術」。demo 用很小的數字（p=467）方便你看；真實系統用 2048 位元以上的大數或橢圓曲線，大到電腦窮舉到宇宙毀滅也算不出來。

2磨成鑰匙 + 鎖箱子（KDF + 加密）

共享祕密不直接當鑰匙，先用一個單向函式（hash）「磨」一下，再拿來鎖訊息：

根鑰匙 = hash("KDF" + 共享祕密)        // 把祕密整理成正式鑰匙
密文   = 明文 XOR 鑰匙                 // demo 用 XOR 示意「上鎖」（真實用 AES）

為什麼用 hash？因為 hash 是單向的：給輸入算得出輸出，給輸出反推不回輸入。這個「單向」性質，正是下一步棘輪能「燒掉舊鑰匙」的關鍵。

3棘輪：每封推進一格、舊鑰匙燒掉（前向保密）

這是全題的心臟。每發一則，就把鏈鑰匙用 hash 單向推進到下一把：

function 取下一把發送鑰匙() {
    這封的鑰匙 = hash("msg" + 目前鏈鑰匙)
    目前鏈鑰匙 = hash("ratchet" + 目前鏈鑰匙)   // ← 單向推進！舊的被覆蓋
    return 這封的鑰匙
}

關鍵在那行「目前鏈鑰匙 = hash(...)」：因為 hash 單向，從新鑰匙算不回舊鑰匙。所以舊鑰匙一旦被推進覆蓋，就永遠救不回來——這就是前向保密。收的一方有一條對應的「接收鏈」，同步往前推，剛好能解開每一封。

4郵差只碰密文：伺服器存的東西長這樣

送出去時，伺服器（信箱）只記下這些，完全沒有明文、沒有鑰匙：

信箱存：{ 寄件人, 收件人, 棘輪序號 n, 密文 }   // 就這些，沒有明文！
// 收件人下載後，用自己的接收鏈鑰匙(第 n 格)解開 → 還原明文

序號 n 是讓收件人知道「這封要用接收鏈推到第幾格的鑰匙來解」。伺服器看得到 n，但沒有鏈鑰匙，所以照樣解不開。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「保險箱 + 郵差」解釋什麼是端到端加密？ 把信鎖進只有你和收件人各有鑰匙的保險箱，再交給郵差送。郵差（伺服器）負責送，但沒鑰匙、打不開， 所以只有收發兩端能讀內容，中間所有人（含服務商）都只看到一個打不開的箱子。
Q2：用「混顏料」解釋，兩個不見面的人怎麼講好同一把鑰匙？ 各選一個祕密色（私鑰），混進公開底色後把混色（公鑰）寄給對方。最後各自把「對方混色 + 自己祕密色」再混一次， 會得到一模一樣的顏色（共享祕密），但偷看的郵差只看到兩坨混色，調不出最終色。這就是 Diffie-Hellman。
Q3：「前向保密」是什麼？為什麼要每封換一把鑰匙？ 前向保密＝就算今天的鑰匙被偷，以前的訊息也解不開。做法是用棘輪：每封信把鑰匙用單向函式推進一格、舊的燒掉， 從新鑰匙算不回舊鑰匙，所以洩漏一把只影響那一封，過去的信仍安全。
Q4：為什麼說「加密不會壓垮伺服器，累的是金鑰管理」？ 因為加解密都在使用者手機上做，伺服器只存轉密文（成本跟存明文差不多）。E2EE 新增的負擔是金鑰管理： 身分鑰匙、一次性 prekey 的發放與補充、每則的棘輪狀態。
Q5（工程）：為什麼一定要把「私鑰」和「棘輪狀態」放在裝置端、不上伺服器？ 因為 E2EE 的安全保證就建立在「私鑰只有你手機有」。一旦私鑰上了伺服器，伺服器就能解密，E2EE 立刻破功。 所以設計上把私鑰與棘輪狀態物理隔離在裝置端，伺服器只存公鑰與密文。
Q6（工程）：對方離線，沒辦法當場做 DH，怎麼還能先把鑰匙建好？ 靠「一次性預付金鑰（prekey）」：每個裝置事先存一批一次性公鑰在金鑰伺服器，要跟離線的人聊天時， 領一把他的 prekey 就能單向把 session 建起來（這就是 X3DH 的精神）。prekey 用完上線再補。
Q7（工程）：E2EE 藏得住內容，但藏不住什麼？面試講這點為什麼加分？ 藏不住「中繼資料」——誰幾點跟誰聊、多頻繁、從哪個 IP。講出這點代表你誠實面對 E2EE 的極限、 威脅模型想得清楚，而不是以為「加密＝完全隱形」。

15. 30 秒回顧

端到端加密 = 把信鎖進只有收發兩端有鑰匙的保險箱，郵差（伺服器）只送箱子、打不開。

4 個關卡：① 混顏料交換出同一把鑰匙（DH）→ ② 把信鎖成密文（KDF 導鑰匙 + 對稱加密）→ ③ 棘輪（每封換鑰匙、用完就燒 = 前向保密）→ ④ 伺服器只存密文（沿用 Messenger 存轉）。

3 個怕：怕中間人（郵差掉包公鑰 → 核對安全碼）、怕一把鑰匙陪葬全部（棘輪一封一把）、怕以為加密就全隱形（中繼資料藏不住）。

工程細節一句話：累的是金鑰管理不是加密運算；東西分兩半放（公鑰/密文在伺服器，私鑰/棘輪只在手機）；離線靠 prekey、群組靠 sender key；E2EE 換來隱私、犧牲了搜尋/審核/備份等伺服器端功能。
