60 端到端加密訊息：只有收發雙方能讀

B 即時連線/同步 建議 45 分鐘 Design End-to-End Encrypted Messaging (Signal)

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把金鑰交換與棘輪拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是在 ⑦ Messenger 的「中繼存轉」之上，疊一層「只有收發雙方持有金鑰」的密碼學： 伺服器退化成不識內容的郵差，只看得到密文。面試真正考的是 雙方怎麼在不信任伺服器的前提下協商出共享金鑰（DH）、 以及 怎麼讓每則訊息用不同金鑰、舊金鑰外洩也救不回舊訊息（棘輪 / 前向保密）。

1. 需求釐清

功能性需求

  • 端到端加密：訊息只有收發雙方能讀，伺服器與任何中間人看不到明文。
  • 金鑰協商：雙方在不互傳私鑰、不信任伺服器的前提下，協商出一把共享對稱金鑰。
  • 前向保密（Forward Secrecy）：就算某把金鑰日後外洩，先前的訊息仍解不開。
  • 後向保密 / 自我修復（Post-compromise Security）：金鑰一度外洩後，後續訊息能恢復安全。
  • 離線投遞（對方不在線時密文先存伺服器）、多裝置、防中間人（金鑰驗證）。

非功能性需求

  • 機密性 / 完整性：密文不可被解、不可被竄改而不被察覺（需帶認證的加密）。
  • 伺服器零信任：即使伺服器被入侵 / 被法院索取資料，也拿不到明文。
  • 低延遲、可離線：對方離線也能先送（需離線金鑰預發 prekey）。
  • 可擴展：億級裝置、海量金鑰與密文中繼。
面試開場必問 「E2EE 是必要還是加分？要不要前向保密、後向保密、多裝置？」 這決定要不要走 Signal 等級（X3DH + Double Ratchet）還是簡化版（單把對稱金鑰）。 也要釐清：E2EE 後伺服器端搜尋、內容審核、防濫用都做不到了（取捨見 §8）。

2. 規模估算

指標假設推算
使用者 / 裝置2 億 DAU，平均 1.5 裝置~3 億把身分金鑰對
預付金鑰 prekey每裝置預存 100 把一次性金鑰3 億 × 100 = 300 億把（用完要補）
訊息量（密文中繼）每人每日 40 則2億×40/86400 ≈ 9.3 萬 寫 QPS（峰值 ×3 ≈ 28 萬）
密文額外開銷每則附帶金鑰材料 / header密文比明文略大（~+數十 bytes/則），可接受
金鑰伺服器讀建 session 才查公鑰，可快取讀多於寫，公鑰目錄易快取/CDN
關鍵洞察密文中繼的「量」跟一般 Messenger 差不多（瓶頸仍是長連線與投遞）；E2EE 真正新增的負擔是金鑰管理：身分金鑰、一次性 prekey 的預發與補充、以及每則訊息的棘輪狀態。計算不在伺服器、而在裝置端。

3. API 設計

# 註冊 / 發布金鑰（裝置產生私鑰，只上傳公鑰）
POST /api/v1/keys
Body: { "identity_key": "<公鑰>", "signed_prekey": "...", "one_time_prekeys": ["...","..."] }
→ 201 { "fingerprint": "安全碼，供雙方比對防中間人" }

# 領取對方的公鑰束（建立 session 用，X3DH）
GET  /api/v1/keys/{user}
→ { "identity_key":"...", "signed_prekey":"...", "one_time_prekey":"..." }  # 一次性金鑰領走即用掉

# 送密文（伺服器只中繼，看不到明文）
POST /api/v1/messages
Body: { "to":"B", "n": 7, "ciphertext":"<密文>", "header":"<棘輪/DH 公鑰材料>" }
→ 201 { "message_id": 123 }

# 收密文（收件人裝置端解密）
GET  /api/v1/inbox?after_id=120
→ [ { "from":"A", "n":7, "ciphertext":"...", "header":"..." }, ... ]
為何上傳公鑰而非私鑰私鑰永遠不離開裝置。伺服器只是公鑰目錄 + 密文郵局；它握有再多資料也解不開訊息——這就是「零信任伺服器」的具體落地。

4. 資料模型

# 伺服器端（只存公鑰與密文，無明文、無私鑰）
key_directory
  user_id, identity_pubkey, signed_prekey, [one_time_prekeys]   -- 公鑰束
ciphertext_mailbox (依收件人，append-only)
  message_id, from, to, n(棘輪序號), header, ciphertext, put_at  -- 只有密文！

# 裝置端（本機安全儲存，永不上傳）
identity_keypair
  private_key, public_key                       -- 身分金鑰
session_state (每個對話一份)
  root_key, send_chain_key, recv_chain_key       -- 棘輪鏈金鑰
  send_n, recv_n                                 -- 已推進次數（對齊金鑰）
  skipped_message_keys[]                          -- 亂序到達時暫存的金鑰
為何切兩半把資料切成「伺服器端（公鑰 + 密文）」與「裝置端（私鑰 + 棘輪狀態）」是這題的核心。只要私鑰與棘輪狀態不離開裝置，伺服器無論如何都看不到明文。

5. 高階架構

                         ▲                                 │ 只看得到密文！                ▲
                         └──── 🔑金鑰伺服器(只發布公鑰) ◀──┴──────────────────────────────┘
              建 session：A、B 各領對方公鑰 → DH 算出「相同共享祕密」→ KDF 導根金鑰 → 開棘輪
  1. 裝置端（綠）：私鑰、共享祕密、棘輪狀態全留本機；負責加密與解密。密碼學計算發生在這裡。
  2. 伺服器（黃）：跟 ⑦ Messenger 一樣負責存轉 + 離線佇列，但只經手密文，是「不識內容的郵差」。
  3. 金鑰伺服器（藍）：發布公鑰束（含一次性 prekey），讓對方就算離線也能先建 session（X3DH）。只存公鑰，無私鑰。

6. 核心元件深入

① Diffie-Hellman 金鑰交換：不傳私鑰也能共享一把金鑰

公開參數 g, p。A 私鑰 a、公鑰 A = g^a mod p；B 私鑰 b、公鑰 B = g^b mod p。雙方交換公鑰後各自算： A 算 B^a = g^(ab)、B 算 A^b = g^(ab) → 得到相同的共享祕密。竊聽者只看到 A、B，要反推 a/b 是離散對數難題（大數下不可行）。

② KDF：把共享祕密導成對稱金鑰

共享祕密本身不直接當金鑰，而是經 KDF（金鑰導出函式，如 HKDF） 導出根金鑰與鏈金鑰。KDF 內部以 HMAC/hash 為基礎——單向性是後面棘輪能成立的關鍵。

③ 對稱加密：實際加密訊息

用導出的訊息金鑰對明文做帶認證的對稱加密（AES-GCM）：既保密又能驗證未被竄改。對稱加密快，適合每則訊息；DH 只用來協商金鑰。

④ Double Ratchet（雙棘輪）：前向保密 + 後向保密

  • 對稱鏈棘輪：每發 / 收一則，就把鏈金鑰用 KDF 單向推進一次（newKey = KDF(oldKey)）。每則訊息金鑰不同；推進後舊金鑰被覆蓋、算不回來 → 前向保密（舊金鑰外洩，先前訊息仍安全）。
  • DH 棘輪：雙方定期換新的 DH 金鑰對、重算共享祕密注入根金鑰 → 即使某輪金鑰被攻破，下一輪會「自我修復」 → 後向保密。
  • 兩條棘輪交織，就是 Signal 的 Double Ratchet。

⑤ 多裝置

同一帳號多台裝置，各有自己的身分金鑰；發訊息時對每台收件裝置各建一條 session、各自加密一份（sender 端 fan-out 到 N 台）。或用「sender key」群組金鑰優化群播。

⑥ 金鑰驗證：防中間人（MITM）

伺服器若偷換公鑰，可發動中間人攻擊。對策：雙方比對公鑰指紋 / 安全碼（safety number）（掃 QR 或唸數字），一致即確認對方公鑰沒被掉包。

7. 擴展與瓶頸

階段瓶頸對策
離線建 session對方離線，無法即時做 DH一次性預付金鑰（prekey）：離線金鑰預發到金鑰伺服器，對方領一把就能單向建 session（X3DH）
prekey 耗盡一次性金鑰被領光裝置上線時補充；耗盡時退回用 signed prekey（安全性略降）
群組金鑰N 人群組，兩兩配對 DH 爆炸sender key：每人發一把群組鏈金鑰給成員，群播只加密一次
多裝置同步同帳號多裝置金鑰 / 歷史一致每裝置獨立 session；新裝置需安全地同步或重新驗證
密文中繼量同 Messenger 的長連線 / 投遞沿用 ⑦ 的連線註冊表 + Pub/Sub + 離線佇列（架構不變，只是內容變密文）

8. 取捨與面試重點

  • E2EE vs 伺服器端功能：加密後伺服器看不到明文 → 全文搜尋、內容審核、防濫用、垃圾訊息過濾、雲端歷史備份都變難或做不到。這是 E2EE 最大的取捨：隱私 ↔ 功能。
  • 金鑰備份 vs 安全：使用者換機 / 丟失裝置，私鑰也沒了 → 舊訊息永遠解不開。提供雲端金鑰備份（如用密碼加密的 backup）會擴大攻擊面；不提供則使用者體驗差。
  • metadata 仍會外洩：E2EE 保護內容，但「誰在何時跟誰聊、頻率、IP」等中繼資料伺服器仍看得到。要進一步隱藏需 sealed sender / 混淆路由等額外設計。
  • 前向 vs 後向保密：對稱棘輪給前向保密（成本低）；後向保密需 DH 棘輪（多一次金鑰交換）。依威脅模型取捨。
  • 常見追問：伺服器被法院索取資料能交出什麼？→ 只有密文與中繼資料，交不出明文。如何防中間人？→ safety number 驗證。亂序 / 遲到的訊息怎麼解？→ 暫存 skipped message keys 以序號對齊。

9. PHP Demo 說明

本題 demo 著重於 E2EE 的系統核心邏輯（金鑰交換 + 棘輪 + 伺服器只見密文），可實際用 PHP 內建伺服器跑起來：

  • src/ToyCrypto.php　玩具版密碼學：DH 模冪、KDF（hash）、棘輪推進、XOR 對稱加密。
  • src/KeyServer.php　金鑰伺服器：發布 / 領取公鑰（只存公鑰）+ 指紋。
  • src/DeviceStore.php　裝置端：身分私鑰 + 每對話的棘輪狀態（send / recv 鏈，永不上伺服器）。
  • src/Mailbox.php　伺服器端密文信箱：只存 { from, to, n, cipher }，無明文。
  • src/E2EEService.php　流程編排：register → openSession(DH) → send(加密) → receive(解密) → 前向保密示範。
  • public/index.php　路由：建 session / 加密送 / 收取解密 / 伺服器視角 / 前向保密，並附 A、B 雙裝置測試頁。
# 啟動
cd demo
php -S localhost:8060 -t public
# 瀏覽器開 http://localhost:8060
# ① A、B 建 session（DH 算出「相同」共享祕密）→ A 加密送 B（伺服器只見密文）
# → B 端棘輪解出明文 → 每則金鑰推進 → ② 用舊金鑰解舊訊息（解不開 = 前向保密）
⚠️ 嚴正誠實聲明（務必閱讀） Demo 用 小數字 Diffie-Hellman（p=467）+ XOR 對稱加密，純為「看得懂」的教學示意， 絕對不是安全加密：小質數可被秒破、XOR 無完整性驗證可被竄改。 但底層概念為真、與 Signal Protocol 一致：DH 雙方算出相同共享祕密（竊聽者僅見公鑰算不出）、 KDF 導金鑰、棘輪每則單向推進（舊金鑰算不出新的）達成前向保密、伺服器只中繼密文。 真實系統用 X3DH 初始協商 + Double Ratchet + AES-GCM / Curve25519，把玩具密碼學替換掉後架構與流程不變。
