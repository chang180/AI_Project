1. 60 秒搞懂它在幹嘛

你每天用的 Zoom、Google Meet、Teams，背後在解一個問題：怎麼讓一群人同時看到、聽到彼此？

兩個人視訊很簡單——就像兩支對講機直接連起來。但人一多就麻煩了： 10 個人的會議，每個人都要看到另外 9 個人的畫面，影像要從哪裡來、走哪條路、誰負責搬運？這就是這題的核心。

🔑 一句話 視訊會議 = 先「講好怎麼連」（信令握手），再「找一個轉播中心幫大家轉影像」（SFU）， 而且網路差的人會自動收到比較糊的畫面（simulcast），正在說話的人會自動被放大（active speaker）。

2. 先看最簡單的：兩個人怎麼視訊？

在搞懂「一群人」之前，先看「兩個人」——這是一切的基礎。

🤝 比喻：兩個陌生人要見面通話 你想打電話給一個陌生人，但你們互相不知道對方的電話號碼。 於是你們先透過一個共同朋友幫忙：交換名片、講好「我用中文、你也聽得懂中文」、約好在哪見面。 講好之後，之後的通話就你們兩個直接講，那個朋友就退場了。

視訊也一樣，分成清楚的兩段：

  • 🤝 「講好怎麼連」這段（信令）：兩邊先交換「名片」——我支援哪種影像格式、我的網路位址在哪。 這段由一個中間的信令伺服器幫忙傳話，它只負責「牽線」，不碰影像本身。
  • 📹 「真的傳影像」這段（媒體）：講好之後，影像聲音就盡量兩邊直接傳，不繞遠路，這樣最快。

這個交換名片、握手講好怎麼連的過程，工程上叫做交換 SDP（我支援什麼格式）和 ICE（我可以從哪些路連到你）。名字先記個印象就好。

💭 停一下：信令伺服器明明幫忙「牽線」了，為什麼影像不也順便經過它，要讓兩邊直接傳？

看答案 因為「牽線」只是一開始講幾句話，資料量超小（幾 KB）；但影像是持續不斷的大流量。如果影像也全部繞經中間伺服器，那台伺服器頻寬馬上被塞爆，而且多繞一圈會更慢更卡。所以原則是：牽線歸牽線，影像盡量直送，能不繞就不繞。

3. 把系統拆成 4 個關卡

「兩個人」搞懂了，現在進入正題：一群人。整件事就是闖 4 關，一關一關來。

1握手：開會前先「交換名片」

每個人加入會議時，都要先跟其他人（或跟轉播中心）交換名片、講好怎麼連：我用哪種影像格式、我的網路位址在哪、走哪條路連得到我。

🤝 比喻就像開實體會議前，大家先互相交換名片、確認彼此聽得懂同一種語言、約好在哪間會議室碰面。講好了，才開始正式開會。

👉 重點：這一關「只在加入的那幾秒」忙碌，講好之後就閒下來了。所以信令伺服器很輕鬆，它從來不負責搬影像。

2選路：影像到底要怎麼走？（最關鍵的一關）

10 個人開會，每個人都要看到另外 9 個人。影像怎麼走？這裡有三種走法，選錯了系統就垮。

📺 比喻：班上要互相看到彼此的表演 想像全班 10 個人，每個人都表演一段，而且每個人都要看到其他 9 個人的表演。怎麼安排？
  • 走法 A：每個人各自連上其他每個人（全員互連 / Mesh）。 你要同時對著 9 台攝影機表演、上傳 9 份影像，網路頻寬瞬間爆掉。人一多，每個人的手機/電腦先撐不住。
  • 走法 B：找一個「轉播中心」（SFU）。每個人只上傳一份自己的影像給中心，中心再幫你轉發給其他 9 個人。 你只要上傳一次，輕鬆很多。這是主流做法。
  • 走法 C：把大家混成一張畫面（MCU）。中心收到所有人的影像後，合成一張九宮格大圖再送回給每個人。 你只收一份很省，但中心要做「剪輯合成」的重活，很吃力又會變慢。

大房間幾乎都選走法 B（SFU 轉播中心）：每人只上傳一份，中心只負責「轉發」不做「合成」，又快又省力。

3適應：網路差的人，自動給他糊一點的畫面（simulcast）

會議裡有人用光纖（網路超快），有人在捷運上（網路爛）。同一份高清影像送給捷運上的人，他一定卡到爆。怎麼辦？

📡 比喻：同一個節目，播高/中/低三種畫質 像 YouTube 一支影片可以選 1080p / 480p / 240p。 視訊也是：每個人同時上傳「高、中、低」三種畫質給轉播中心。 網路好的訂高清，網路爛的自動切到低畫質——畫面糊一點但不卡，比卡死好。

這個「同時上傳多種畫質、讓每個人各取所需」的招數叫 simulcast。 關鍵好處：一個網路爛的人，不會拖垮整場會議——他自己收糊的，別人照樣看高清。網路一回復，又自動升回高畫質。

4聚焦：誰在說話，就自動放大誰（active speaker）

10 個人的九宮格，眼睛該看哪？系統會幫你決定。

🎤 比喻：會議室的鏡頭自動對準發言人 高級會議室有「智慧攝影機」，誰拿起麥克風講話，鏡頭就自動轉過去拍他。 視訊也一樣：每個人的麥克風持續回報自己的音量，系統挑出當下音量最大的人＝主講者，通知大家把他的畫面放大。

這叫 active speaker（主講者偵測）。順帶一提：被放大的主講者，通常也會優先收到高畫質（反正大家在看他）。

✅ 四關一句話 ① 開會前交換名片講好怎麼連 → ② 找一個轉播中心（SFU）幫大家轉影像 → ③ 網路差的人自動收糊畫面 → ④ 正在說話的人自動被放大。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「卡頓 / 延遲」 講話一來一回，超過 0.2 秒就會明顯打斷對話節奏（你會一直跟對方「搶話」）。 所以影像要走最短的路、不做多餘的加工。也因此影像掉了就算了不重傳——一張過時的舊畫面補回來反而更亂。流暢 > 完美畫質。
😱 怕「一個人爆量拖垮全場」 最怕「全員互連」那種走法：人一多，每個人都要上傳好幾份影像，弱網的人直接陣亡，還可能拖累別人。 所以才要轉播中心（SFU）只讓每人上傳一份，再加 simulcast 讓弱網的人自己收糊的，把「一個人爆量」隔離開。
😱 怕「連不上」 很多人在公司防火牆、家用路由器後面，沒有對外的真實地址，兩邊想直連卻找不到路。 所以要先「問問自己在網路上長怎樣」（STUN），真的怎麼樣都直連不通時，就找一個中繼站幫忙轉（TURN）——能通就好，雖然繞遠路。

5. 一張圖看懂

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw）

白話導讀，分兩條線看：

  • 🤝 牽線那條（信令，量小）：每個人先連上信令伺服器交換名片（SDP/ICE），講好怎麼連。它只傳話，不碰影像。
  • 📹 傳影像那條（媒體，量大）：每個人上傳一份自己的影像給 SFU 轉播中心；中心查「轉發表」（誰要看誰），把影像轉發給該看的人。
  • 🎚️ 轉發時，中心會幫每個收看的人挑畫質（高/中/低），並標出誰是主講者。
  • 🌐 旁邊還有 STUN/TURN：幫忙解決「在防火牆後面連不上」的問題（先問位址，不行就找中繼站轉）。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂多大的會議室、拉多粗的網路。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

這題最該感受的，是「頻寬（網路水管粗細）」這個數字的形狀（數字不必背）：

估什麼大概數字所以呢？
一路高清影像多大720p 約 1.5 Mbps（每秒約 1.5 百萬位元）影像很「重」，是主要負擔
每個人上傳幾份用 SFU 只上傳 1 份（含高中低三層約 2.5 Mbps）上傳壓力固定，不會因人多而暴增
50 人會議，轉播中心要轉多少每人收 49 路 → 中心出口約 1 Gbps（每秒 10 億位元）瓶頸在「中心的出口水管」
牽線（信令）的量只有加入時幾 KB，之後幾乎是 0牽線超輕，可跟影像分開擴展
🚰 關鍵洞察：瓶頸不在牽線，在「出水量」 牽線（信令）只是開頭講幾句話，輕得很。真正會撐爆的是轉播中心往外送影像的總水量—— 50 個人每人要收 49 路影像，全部加起來就是一條巨大的水管。所以擴展的重點，永遠是「中心的出口頻寬」。

💭 停一下：用「全員互連」的話，為什麼 10 個人就會先讓使用者的電腦受不了，而不是伺服器？

看答案 因為全員互連沒有中心，每個人要自己對另外 9 個人各上傳一份影像，等於一個人要同時上傳 9 路（約 13 Mbps），家用網路上傳頻寬根本撐不住。改用 SFU 後，每人只上傳 1 份給中心，把使用者端的壓力從 9 份壓回 1 份，代價是換成中心的出口頻寬要夠大。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作。這題的 API 幾乎都是「牽線」用的，影像本身不走這些單子（影像直接走轉播中心）。

記住幾張關鍵「點餐單」：

① 加入會議
   POST /api/join   給我：{ 房間, 名字 }
   我回：{ 你的編號, 目前房裡有誰 }

就像走進會議室先報到，櫃台給你一個名牌，並告訴你「現在房裡有哪些人」。

② 交換名片（SDP / ICE）← 牽線的核心
   POST /api/sdp   給我：{ 我是誰、要給誰、我的影像格式 }
   POST /api/ice   給我：{ 我是誰、要給誰、我可以從哪條路被連到 }

信令伺服器只是幫你把名片轉交給對方，它不看內容、不碰影像，純粹「傳話」。

③ 回報「我網路只剩多少」→ 中心幫我挑畫質
   POST /api/bandwidth  給我：{ 我是誰, 我現在頻寬只剩 400 }
   我回：{ 幫你切到 "中畫質" }

這就是 simulcast 的自動降階：你網路變差，中心就把你那一路切到比較糊的層，先保不卡。

④ 回報「我現在音量多大」→ 中心選出主講者
   POST /api/audio  給我：{ 我是誰, 我音量 0.83 }
   我回：{ 現在主講者是你 }

大家都回報音量，中心挑最大聲的當主講者，再通知所有人去放大他。

9. 系統要記住哪些事？（資料模型）

📒 比喻：幾本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題主要記這幾本：

🏠房間簿 + 成員簿（誰在哪間房）

房間簿：每間會議室一列（房號、建立時間）。成員簿：每個參加者一列 編號、在哪間房、名字、目前頻寬、最近音量。

為什麼要記「頻寬」和「音量」？ 頻寬 → 決定幫你挑哪種畫質（simulcast 選層）；音量 → 決定誰是主講者。這兩欄就是系統「自動調整」的依據。

📨名片信箱（暫存還沒被取走的名片）

記「誰要傳給誰、是 SDP 還是 ICE、內容、時間」。 就像一個公司信箱：你的名片先放著，對方上線了再來取走。這就是「信令中繼」。

🔀轉發表（轉播中心的核心：誰的影像要轉給誰）

記 誰上傳的影像（publisher）、要轉給誰看（subscriber）、用哪一層畫質（高/中/低）。

轉發表就是 SFU 的大腦 10 個人的房間，這張表會有很多列：「A 的影像 → 轉給 B（高）、轉給 C（低）…」。 每一列都能挑不同畫質，因為每個收看的人網路狀況不同。表雖然長，但每個人仍然只上傳一份影像，這就是 SFU 省力的關鍵。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
小房間還架轉播中心，太浪費2~3 人直接兩邊互連，連中心都省了
轉播中心出口水管被塞爆（~Gbps 級）幫弱網的人降畫質少送一點、限制同時收的高清路數
萬人演講（webinar），影像流爆炸只訂主講者+少數可見的人的影像，其他人的先暫停不收
一台轉播中心撐不住一場大會多台中心串接：跨機只傳一份，到了當地再分送（級聯）
跨國開會，地球另一端很遠很慢每個人連離自己最近的那台中心，中心之間再互聯

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 轉播中心(SFU) vs 混流中心(MCU) SFU 只轉發影像，不加工，所以快又省 CPU，但每個收看的人都要拉一條（下行頻寬大）。 MCU 把大家合成一張，下行很省，但「合成」這個重活很吃 CPU 又會變慢。 大房間選 SFU；只有要錄影、或相容很老的設備時，才考慮 MCU。
⚖️ 全員互連 vs 轉播中心 小房間全員互連最省（不用伺服器、最快）；但人一多，每個人的上傳就爆，所以改用轉播中心把上傳壓回 1 份。 分界點大概就是「2~3 人 vs 更多人」。
⚖️ 牽線要可靠 vs 影像可以掉 牽線（信令）的名片一定要送到，掉了就連不上；但影像幀掉了就算了，不重傳—— 因為一張慢半拍補回來的舊畫面，反而會讓畫面更亂。影像追流暢，不追完整。
💭 追問：有人在公司防火牆後面，怎麼樣都連不上怎麼辦？ 先用 STUN「問自己在網路上長怎樣」嘗試打洞直連；真的不通（對稱型 NAT）就用 TURN 中繼站幫忙轉所有影像——能通就好，代價是繞遠路、多花頻寬、稍微慢一點。
💭 追問：上百人的大會議，畫面那麼多怎麼不卡？ 用「分頁訂閱」：你的螢幕一次只看得到幾個人，系統就只送你「看得到的那幾個人」+ 主講者的影像，其他人的暫停不送。再配合給弱網的人降畫質。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「牽線 + 轉播中心決策」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8030 -t public
# 然後瀏覽器打開 http://localhost:8030
  1. 先建一個房間，再讓幾個人加入（等於幾個人進會議室）。
  2. 貼一段 SDP / ICE，看它怎麼幫你「轉交名片」給對方。
  3. 觀察轉發表：誰的影像要轉給誰。
  4. 調某個人的頻寬，看他被切到哪一層畫質（降階）；回報某人音量，看誰被選成主講者。
誠實聲明 這個 demo 不傳真的影像（PHP 環境跑不了真實的 WebRTC 媒體）。它模擬的是「牽線」和「轉播中心的決策」—— 轉發表怎麼建、依頻寬怎麼挑畫質、怎麼選主講者。這些「決策邏輯」正是面試真正在考的東西，而且是真的可以跑、邏輯為真的。真實系統會把同樣的決策接到 mediasoup / Janus 等正式的 SFU 引擎上。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：這題的核心其實全擠在一個小檔（Sfu.php）裡，剛好對應前面講的四關工作。 不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 一個檔 = 一個轉播中心小幫手 這個 Sfu 小幫手身兼四職：幫人交換名片（信令中繼） → 建一張「誰轉給誰」的轉發表 → 依每個人的頻寬挑畫質（選層） → 挑出最大聲的當主講者。一個人忙完這四件，一場會議就轉起來了。

1信令中繼：幫你把名片轉交給對方

第一關，幫兩個人「交換名片」。中心不看內容、不碰影像，純粹把你的名片放進對方的信箱，等對方來取走。

function 轉交名片(房間, 從誰, 給誰, 種類, 內容) {
    if (從誰 或 給誰 不在這房間) 報錯;
    // 把名片丟進「給誰」的信箱，等他上線來取
    對方信箱[給誰].加入({ 從誰, 種類:'sdp'或'ice', 內容, 時間 })
    return 已投遞;
}

function 取走信箱(房間, 我是誰) {
    我的信件 = 信箱[我是誰]
    信箱[我是誰] = 空的       // ← 取走即清空，模擬「投遞出去」
    return 我的信件
}

這就是前面說的「交換名片、握手講好怎麼連」。SDP（我支援什麼格式）和 ICE（我可以從哪條路被連到）都是同一個轉交動作，只差「種類」欄不同。中心像公司收發室，只負責傳話，從不拆開影像。

2建轉發表：誰的影像要轉給誰（SFU 的大腦）

每當有人加入或頻寬改變，中心就重建一張轉發表：房裡每個上傳者的影像，要轉給其他每個人看（不轉給自己）。

function 重建轉發表(房間) {
    表 = 空的
    每個 上傳者 in 房裡的人:
        每個 觀看者 in 房裡的人:
            if (上傳者 == 觀看者) 跳過       // ← 不轉發給自己
            表.加入({
                上傳者, 觀看者,
                層: 依頻寬挑層(觀看者的頻寬)   // ← 每一列各挑各的畫質！
            })
    房間.轉發表 = 表
}

10 個人就會有很多列：「A 的影像 → 轉給 B（高）、轉給 C（低）…」。 重點：每一列可以是不同畫質，因為每個觀看者網路狀況不同。但不管表多長，每個人仍只上傳一份影像——這就是轉播中心（SFU）省力的精髓。

3依頻寬選層：網路差的人自動給糊一點的（simulcast）

這是 simulcast 的心臟。每個人同時上傳高/中/低三層，中心幫每個觀看者挑「他頻寬扛得住的最高那層」。

三層的需求頻寬 = { 高:1500, 中:500, 低:150 }   // kbps（由高到低）

function 依頻寬挑層(可用頻寬) {
    // 白話：依訂閱者頻寬，由高層往低層試，挑「頻寬扛得住的最高那層」
    if (可用頻寬 >= 1500) return '高';   // 頻寬夠 → 給高畫質
    if (可用頻寬 >= 500)  return '中';   // 中等 → 給中畫質
    return '低';                         // 再差也保底給低（總比全黑好）
}

就是這幾行 if：依訂閱者頻寬，頻寬 > 高層門檻給高、> 中層門檻給中、否則給低。 捷運上的人頻寬掉到 400 → 自動切「低」，畫面糊一點但不卡；網路一回復又自動升回「高」。 關鍵好處：一個弱網的人不會拖垮整場，他自己收糊的，別人照樣看高清。

💭 停一下：為什麼頻寬連「低」（150）都不到，還是要硬給他「低」，而不是乾脆不送？

看答案 因為糊畫面也比全黑好。如果連最低層都不給，那個人的畫面整片黑，使用者會以為「壞掉了」。保底給最低層，至少看得到對方還在、嘴巴有在動，體驗差很多。所以挑層的最後一步永遠是「再差也回低層」。

4選主講者：誰最大聲就放大誰（active speaker）

每個人的麥克風持續回報自己的音量，中心每次收到就重選一次主講者——挑當下音量最大的那個。

function 回報音量(房間, 我是誰, 音量) {
    我的音量 = 夾在 0~1 之間(音量)        // 防呆：超過範圍先夾住
    房間.主講者 = 選最大聲的(房間)
}

function 選最大聲的(房間) {
    最佳 = null;  最大音量 = 0
    每個 人 in 房裡的人:
        if (這人的音量 > 最大音量) {
            最大音量 = 這人的音量
            最佳 = 這人
        }
    return 最佳        // ← 全部都是 0（沒人說話）→ 回 null
}

就是一個「挑最大值」的迴圈。誰音量最大就是主講者，通知大家把他的畫面放大（通常也優先給他高畫質）。 全場靜音時回 null＝「沒人說話」，畫面就不特別放大誰。這正是「智慧會議室鏡頭自動對準發言人」那個比喻。

🧭 把程式對回前面的關卡
關卡（前面講的）哪段程式關鍵那段
① 交換名片（信令）轉交名片 / 取走信箱丟進對方信箱、取走即清空
② 轉播中心轉發重建轉發表每個上傳者 × 每個觀看者（不含自己）
③ 依頻寬選層（simulcast）依頻寬挑層≥1500 高 / ≥500 中 / 否則低
④ 主講者（active speaker）選最大聲的挑音量最大值的迴圈

👉 重點：這個 demo 不傳真的影像，它模擬的是「決策」——轉發表怎麼建、依頻寬怎麼挑層、怎麼選主講者。 真實系統會把「JSON 檔 + 迴圈」換成 WebRTC + 正式的 SFU 媒體伺服器（mediasoup / Janus）來真的搬運影像， 但上面這幾段判斷邏輯（信令中繼、轉發表、選層、選主講者）一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「交換名片」的比喻，說說信令在做什麼？為什麼影像不走信令？ 信令＝開會前交換名片、握手講好怎麼連（我支援什麼格式、我的網路位址在哪），由信令伺服器幫忙傳話。影像不走信令，是因為信令只是開頭幾句話、量很小，而影像是持續的大流量，繞經中間會塞爆又變慢，所以影像盡量直送或走轉播中心。
Q2：10 人會議，影像三種走法（全員互連 / SFU / MCU）差在哪？大房間選哪個？ 全員互連＝每人對每人直連，上傳會爆，只適合 2~3 人。SFU＝每人上傳一份給轉播中心，中心轉發給其他人，省力又快。MCU＝中心把大家合成一張九宮格再送回，下行省但很吃 CPU 又慢。大房間選 SFU。
Q3：simulcast 是什麼？用 YouTube 畫質選單的比喻說說看。 像 YouTube 同一支影片有 1080p/480p/240p。simulcast 就是每個人同時上傳高/中/低三種畫質，網路差的人自動收低畫質（糊但不卡），網路好的收高畫質。關鍵是一個弱網的人不會拖垮整場會議。
Q4：active speaker（主講者偵測）怎麼運作？ 每個人的麥克風持續回報自己的音量，系統挑出當下音量最大的人當主講者，通知大家把他的畫面放大（通常也優先給他高畫質）。就像智慧會議室鏡頭自動對準發言人。
Q5（工程）：為什麼說瓶頸不在「信令」而在「轉播中心的出口頻寬」？ 信令只在加入的那幾秒交換幾 KB 名片，之後幾乎是 0，超輕。真正的負擔是中心要把每個人的影像轉發給其他所有人——50 人房每人收 49 路，加總可達 ~1 Gbps，這條出口水管才是會撐爆的地方。
Q6（工程）：SFU 的「轉發表」裡記什麼？為什麼每一列可以是不同畫質？ 記「誰上傳的影像 → 要轉給誰看 → 用哪一層畫質（高/中/低）」。每一列可不同畫質，是因為每個收看的人網路狀況不同：網路好的那一列給高層，網路差的那一列給低層。但不管轉發表多長，每個人仍只上傳一份影像。
Q7（工程）：有人在防火牆後面連不上，STUN 和 TURN 各扮演什麼角色？ STUN 先幫你「問自己在公網上長怎樣」，多數情況能藉此打洞直連。如果是對稱型 NAT 怎麼樣都打不通，就用 TURN 當中繼站，把所有影像經它轉一手——能連通但繞遠路、耗頻寬、稍慢，是最後手段。

15. 30 秒回顧

視訊會議 = 先「講好怎麼連」，再「找轉播中心幫大家轉影像」。

4 個關卡：① 交換名片握手（信令 SDP/ICE）→ ② 找轉播中心 SFU 轉影像（每人只上傳一份）→ ③ 網路差的人自動收糊畫面（simulcast）→ ④ 正在說話的人自動放大（active speaker）。

3 個怕：怕卡頓（影像走最短路、掉了不重傳）、怕一個人爆量拖垮全場（用 SFU + simulcast 隔離）、怕連不上（STUN 打洞、不行就 TURN 中繼）。

工程細節一句話：瓶頸在「中心出口頻寬」不在信令 → 用「點餐單(API)」交換名片、回報頻寬挑畫質、回報音量選主講者 → 用「轉發表」決定誰的影像轉給誰、各挑畫質 → 大會議靠多台中心串接、分頁訂閱、就近連線來拓寬。
