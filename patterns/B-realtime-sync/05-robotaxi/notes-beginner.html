1. 60 秒搞懂它在幹嘛

你打開 App 按「叫車」，幾秒後螢幕上就出現「ABC-001 號車，2 分鐘後到」，還能看到那台車在地圖上一格一格往你這邊開過來。 RoboTaxi（自駕計程車）做的就是這件事。

聽起來不就是「按一下 → 配一台車」嗎？很簡單啊。記下一筆叫車單確實很簡單。難的是藏在背後的三件事，我們下一節就拆給你看。

🔑 一句話 RoboTaxi 媒合 = 在一張塞滿幾百萬台移動車子的活地圖上，瞬間幫你抓一台最近的空車，而且保證這台車不會同時被別人搶走。

2. 難的不是叫車，是這三件事

「按一下配一台車」之所以難，是因為這張地圖又大、又一直在動、還有別人在搶：

🗺️ 難點一：地圖超大，怎麼瞬間找到「附近」的車？ 全世界可能同時有 100 萬台車在路上。你叫車時，系統不可能把 100 萬台車一台一台量「離你多遠」，那要算到天荒地老。
🔄 難點二：每台車都在動，位置一直變 車子在開，位置每幾秒就變一次。100 萬台車同時回報「我現在在這」，這是海量的更新——系統要一邊吃進這些更新，一邊還要能馬上查。
🎫 難點三：同一台車，不能同時配給兩個人 你跟隔壁路口的人幾乎同一秒都叫車，附近最近的空車剛好是同一台。如果系統不小心，兩個人都收到「配到這台車」——那就糗了，一台車不可能載兩攤。

記住這三個難點，後面的設計全部都是在解這三件事。

3. 把系統拆成 3 個關卡

別被「系統」嚇到。一次叫車，其實就是闖 3 關，一關一關來：

1找附近：先用「郵遞區號」縮小範圍

你叫車，系統要先撈出「你附近的空車」。前面說了，不能把 100 萬台車一台台量距離。怎麼辦？

📮 比喻：找附近的人，先看郵遞區號 你要找「住我家附近的店」，不會把全國店家從頭翻一遍量距離。 你會先看郵遞區號 / 地址前幾個字，先框出「同一區」的店，再從這一小撮裡細看。

系統用的工具叫 geohash：把地球切成一格一格的小網格，每格一個短代碼，位置相近的代碼開頭一樣。 於是找附近變兩步：

  • 粗篩：用「你所在那幾格」的代碼前綴，秒撈出候選的車（可能 20 台，而不是 100 萬台）。
  • 細篩：再對這一小撮候選，精算「實際離你幾公里」，挑出最近的幾台。

⚠️ 小細節：你可能站在格子的邊緣，最近的車說不定在隔壁格。所以要連旁邊 8 格一起撈，免得漏掉就在你身後那台。

2搶車：原子媒合，只能一個人成功

找到候選車了，現在要指派一台給你。難點三來了：隔壁的人可能跟你搶同一台。

🎫 比喻：搶演唱會最後一張票 演唱會剩最後一張票，兩個人同時點「結帳」。售票系統不能讓兩個人都成功——它會在最後一刻檢查：「這張票還在嗎？在 → 給你、鎖定；已被搶 → 跟你說『手慢了，換下一場吧』。」只能有一個人結帳成功。

系統用的招式叫 CAS（條件更新），白話就是「搶之前先確認還沒被搶」：

  • 指派時不是直接寫「這台車配給你」，而是「如果這台車還是『空車』，才把它改成『已配』」。
  • 這個「確認＋改」是一個不可被打斷的動作（這就是「原子」的意思）。兩個請求同時來，只有一個會成功，另一個會發現「咦，已經不是空車了」。
  • 沒搶到的那位不會收到錯誤，系統直接幫他換候選清單裡的下一台（反正也是附近的近車）。

👉 這就是整題最關鍵的考點：怎麼保證「同一台車不會被配給兩個人」。答案就是「搶最後一張票」的那一招。

3追蹤：看著車在地圖上開過來

配到車了，你會想盯著它開過來，司機也想看到你在哪。這要靠「即時位置串流」。

🛵 比喻：外送 App 上的那台小機車 你點外送後，地圖上那台小機車會一格一格往你家移動，不用你一直按「重新整理」。 系統幫你跟司機開了一條「專線」，車子每動一下就把新位置主動推到你螢幕上。

這條專線叫 WebSocket（長連線）——像一通不掛斷的電話，雙方隨時能講話，不用每次都重撥（重撥＝一直輪詢，又慢又耗電）。 車子每幾秒回報的新位置，一邊更新地圖索引，一邊推給正在看的你。

✅ 三關一句話 ① 用郵遞區號式網格找附近的車 → ② 用「搶最後一張票」的方式原子指派一台 → ③ 開一條不掛斷的專線即時追蹤。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「一台車被搶兩次」 這是最嚴重的錯。一台車同時配給兩個乘客，一定有人撲空。所以「指派狀態」必須絕對精準（強一致）—— 用「搶最後一張票」的原子動作（CAS），只能一個人成功。這條是硬規定，不能妥協。
😱 怕「找車太慢」 你按了叫車，等超過 1～2 秒沒反應就會煩躁。但附近有 100 萬台車的資料，一台台量距離一定慢。 所以用「郵遞區號網格」先粗篩，把要算的車從 100 萬縮到 20 台。先縮範圍，再精算。
😱 怕「位置更新淹沒系統」 100 萬台車每幾秒回報一次位置，這是系統最大量的寫入。如果每筆都很慎重地存進正式資料庫，會被活活壓垮。 所以「車現在在哪」只放記憶體裡直接覆寫（只要最新的）；「車走過的歷史軌跡」才慢慢另存別處。當前位置求快，歷史軌跡求穩，兩者分開。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
同時在線的車全球 ~100 萬台要同時維持 100 萬條「不掛斷的電話」
位置上報（寫）每車每 4 秒一次100萬÷4 ≈ 每秒 25 萬筆寫入，超大量
叫車請求尖峰每分鐘 1 萬筆每筆都要「查附近 + 搶票指派」
一筆當前位置多大車號+經緯度+時間 ≈ 50 bytes100萬筆才 ~50 MB，整張地圖塞得進記憶體
歷史軌跡每車每天約 2 萬點每天 ~1TB，是「冷資料」，不能擠進查附近的熱路徑
🧊 關鍵洞察：熱資料 vs 冷資料分開放 「車現在在哪」每秒被蓋掉、只要最新的 → 放記憶體（快、直接覆寫，像白板擦了重寫）。 「車走過的軌跡」只進不出、堆成山 → 慢慢另存到倉庫（時序資料庫）。 把這兩種資料分開，是這題能撐住的關鍵。

💭 停一下：為什麼「車現在的位置」不存進正式資料庫，只放記憶體？

看答案 因為它每幾秒就變一次、而且只要最新的那一筆。記憶體覆寫又快又省（像白板擦掉重寫）；如果每筆都慎重寫進硬碟資料庫，每秒 25 萬筆會直接把資料庫壓垮。而且舊位置根本不需要留在熱路徑——歷史另外存就好。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題記住四張「點餐單」：

① 司機回報位置（最高頻，走「不掛斷的電話」）
   WS → { 司機:"d1", 緯度:25.0418, 經度:121.565, 時間:... }
   不用每次回，只要默默更新地圖索引

為什麼走 WebSocket 不走一般網頁請求？因為每幾秒就要傳一次，用「不掛斷的電話」最省；每次重撥（一般 HTTP）又慢又費。

② 乘客叫車（最關鍵的一張單）
   POST /api/v1/rides
   給我：{ 乘客, 上車點經緯度, 下車點 }
   我回：{ 行程編號, 司機:車牌, 距離, ETA幾分鐘, 加價倍率 }

系統收到這張單後，內部就跑「找附近 → 搶票指派」。回給你的是已經搶到的那台車的資訊。

③ 查附近有幾台車（給「附近 5 台車」這種畫面用）
   GET /api/v1/drivers/nearby?lat=..&lng=..&radius_km=3&k=5

這張單只「看」不「搶」，所以可以用「拉」的（你自己定時問），不一定要長連線。

④ 行程結束 → 把車放回「空車」
   POST /api/v1/rides/{編號}/complete

很重要：行程結束一定要把車的狀態改回「空車」，它才能再被別人搶到。忘了放回去，這台車就「卡死」不再接客了。

9. 系統要記住哪些事？（資料模型）

📒 比喻：三本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題只要三本，而且故意放在不同地方：

🚗車況簿（誰、在哪、是不是空車）— 放記憶體

每台車一列：車號、最新經緯度、geohash、狀態(空車/已配/載客中/離線)、更新時間。

為什麼多存一欄 geohash？ 把每台車的「郵遞區號」一算出來就寫進去，叫車時才能用前綴秒查附近的車。 這本是「熱資料」，每秒被覆寫無數次，所以放記憶體裡（像 Redis），不放硬碟資料庫。

🧾行程簿（哪次媒合）— 放正式資料庫

每筆行程一列：行程編號、乘客、司機、上車點、下車點、狀態(已配/載客中/已完成/取消)、ETA、加價。 這本要精準、可靠（牽涉錢跟責任），所以放正式資料庫。

🛤️軌跡簿（車走過哪些點）— 串流另存倉庫

記「哪台車、什麼時間、在哪個點」。這本只進不出、量超大，不能擠進「查附近」的熱路徑—— 用串流（像輸送帶）慢慢搬到時序資料庫 / 倉庫，事後分析用。

為什麼故意分三個地方？三本筆記本的「讀寫個性」完全不同：車況要快又一直覆寫、行程要準、軌跡只堆不查。硬塞在一起會互相拖垮，所以分開放，各用最適合的工具。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚦 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
找附近：100 萬台車一台台量距離用網格索引（郵遞區號法）先粗篩，再對少數細算距離
位置寫爆：每秒 25 萬筆打資料庫當前位置只在記憶體覆寫，歷史軌跡走輸送帶另存
搶車競爭：同一台車被狂搶、鎖卡住用 CAS「搶票術」+ 搶不到就換下一台候選，鎖只鎖單一台車不鎖全部
地理熱點：市中心一格塞太多車網格自動細分（密的地方切更細，像 QuadTree）
百萬條「不掛斷電話」一台機器撐不住多開幾台閘道機器分攤連線，連線狀態外放（水平擴展）
跨城市查詢沒意義又拖慢以城市為單位分區，台北的車不會跑去跟東京的比

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 強一致 vs 最終一致（要多精準？） 車的指派狀態必須強一致（絕不能被搶兩次，用搶票術）；但位置追蹤可以最終一致——晚 1 秒看到車的位置，沒人會在意。該嚴的地方嚴，能放鬆的地方放鬆。
⚖️ 即時媒合 vs 批次撮合 本題是即時型（你一叫，馬上找一台給你）：快、直觀。另一種是批次型（每隔幾秒集中一批一起配，像拼車），能做全局最佳（整體等最短），但你會多等一下、實作也複雜。
⚖️ 推 vs 拉 行程中的位置一定用推（不掛斷的電話）才即時又省電；「附近有幾台車」這種畫面可以用拉（自己定時問一下就好）。
💭 追問：司機手機突然斷線（進隧道、沒電）怎麼辦？ 用「心跳 + 過期時間（TTL）」：車要定時報平安，太久沒報就標記離線、從地圖索引移走，免得有人被配到一台早就連不上的車。
💭 追問：怎麼避免「鬼車」（位置卡住不動的殭屍車）？ 同樣靠過期時間：超過一段時間沒更新位置的車，自動從索引淘汰。沒在動、沒在報的，就當它不在線上。

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「找附近 + 搶票指派」：

# 啟動（在這題的 demo 資料夾裡）

# 然後瀏覽器打開 http://localhost:8005
  1. 先註冊幾台司機（等於在地圖上放幾台車）。
  2. 在某個點叫車 → 看媒合到哪一台、距離多遠、ETA 幾分鐘。

玩的時候想一下連續叫車，看同一台車會不會被配給兩個人——這就是關卡 2 的搶票術（CAS）在運作。把司機移到比較遠的地方再叫車，看它是不是不再被挑到——這就是關卡 1 的地理篩選。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心就三個小檔，剛好對應前面講的關卡。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 GeoIndex（地圖小幫手：用郵遞區號縮範圍 → 算實際距離找最近的車）→ Matcher（媒合小幫手：拿到一排近車後，逐台用「搶票術」搶一台）→ Store（保管小幫手：用一道鎖保證「搶確認＋改狀態」不會被打斷）。叫一次車，就照這個順序跑一遍。

1找附近：先用郵遞區號縮範圍（GeoIndex）

第一關，從一大堆車裡撈出「離你最近、而且是空車」的幾台。重點是：只看空車、算實際距離、由近到遠排好。

function 找附近的車(我的緯度, 我的經度, 半徑) {
    候選 = []
    for (每一台車) {
        if (這台車不是「空車」) continue;      // ← 載客中/離線的直接跳過
        距離 = 算地球兩點距離(我, 這台車)        // haversine 球面距離
        if (距離 <= 半徑)
            候選.加入({ 車, 距離 })
    }
    把候選「由近到遠」排好
    return 取前面幾台
}

這就是前面「郵遞區號縮範圍」的精算那半段。為了好懂，demo 直接掃一遍所有車算距離；但它另外備了一支 geohash() 把座標編成「位置郵遞區號」（開頭一樣 = 地理相近），真實系統就是先用這個前綴秒撈出同一區的車當候選，再對這一小撮算實際距離——邏輯一模一樣，只是先粗篩再細算。

媒合引擎要的不是「全部附近的車」，而是「最近的 K 台」當候選清單：

function 最近K台可用車(我, K) {
    候選 = 找附近的車(我, 放寬一點的半徑)   // 先大範圍撈
    return 取前 K 台                       // 已照距離排好，挑最近的幾台
}

為什麼要「一串候選而不是只要最近那一台？因為最近那台可能剛好被隔壁的人搶走——手上有一排備胎，搶不到就換下一台，下一關就靠這個。

2搶車：搶最後一張票，只能一個人成功（Matcher）

這是全題的心臟。拿到一排近車後，從最近的開始，逐台搶，搶到第一台就成交：

function 叫車(乘客, 我的位置) {
    候選 = 最近K台可用車(我的位置, K)          // 第 1 關給的一排近車

    for (每一台候選車 從近到遠) {
        // ★ 關鍵這行：搶最後一張票
        if (!原子搶車(這台車, 期望="空車", 改成="已配"))
            continue;                        // ← 沒搶到（被別人先搶走）→ 換下一台

        建立行程(乘客, 這台車, 距離, ETA...)    // 搶到了！這台歸我
        return 行程
    }
    return null;                             // 一排候選都被搶光 → 附近暫無車
}

注意那行 原子搶車(…) 回傳 false（沒搶到）時，程式不報錯，而是 continue 默默換下一台候選再搶——乘客通常不會察覺，只是配到第二近的車。這正是前面說的「搶不到就換下一張票」。

💭 停一下：為什麼搶車要「逐台試」，而不是先挑出最近那台、確定它空車、再寫「配給你」？

看答案 因為「確定它空車」和「寫配給你」中間有一個空檔——就在這一瞬間，隔壁的人可能把同一台搶走了，於是你又寫上去，一台車配給兩個人。解法是把「確認＋改」併成一個不可被打斷的動作（下一段的 CAS），失敗就換下一台。所以一定要「逐台試」配上「原子搶」。

3原子保證：把「確認＋改」鎖成一個動作（Store）

上一關那行 原子搶車(…)，真正的魔法在這裡。它叫 CAS（條件更新）：只有當車「還是空車」時，才把它改成「已配」，而且整段在一道鎖裡完成，不會被別的請求插隊：

function 原子搶車(車號, 期望狀態, 新狀態) {
    上鎖();                                  // ← 進互斥區，同時間只有一個請求能進來
    if (這台車.狀態 == 期望狀態) {            // 還是「空車」嗎？
        這台車.狀態 = 新狀態                  // 是 → 改成「已配」
        搶到 = true
    } else {
        搶到 = false                         // 已被別人改走 → 沒搶到
    }
    解鎖();
    return 搶到
}

關鍵是「確認還是空車」和「改成已配」這兩步被鎖在一起，中間插不進別人。兩個請求同時搶同一台，鎖讓他們排隊進來：第一個看到「空車」搶成功並改掉；第二個進來時已經不是空車，回 false。只能一個人成功——這就是「搶最後一張票」在程式裡的樣子。

4收尾：行程結束把車放回「空車」（Matcher）

最後一道別忘了：行程做完，要把車的狀態改回「空車」，它才能再被下一個人搶到。

function 完成行程(行程編號) {
    行程.狀態 = 已完成
    原子搶車(司機, 期望="已配", 改成="空車")   // 把車釋放回可被媒合
}

忘了這一步，這台車就永遠卡在「已配」、再也不會出現在任何人的候選清單裡——等於白白少一台車。狀態機要有借有還：空車 →（搶到）已配 →（完成）空車。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「郵遞區號」解釋，為什麼不用一台台量距離找附近的車？ 100 萬台車一台台量太慢。改用 geohash（位置郵遞區號）：把地圖切網格，附近的格子代碼開頭一樣。先用前綴粗篩出同區那一小撮車，再對這幾台細算實際距離。記得連旁邊 8 格一起撈，避免漏掉就在你身後、卻在隔壁格的車。
Q2：用「搶最後一張票」解釋，怎麼保證一台車不會配給兩個人？ 兩個人同時想要同一台車。系統不直接寫「配給你」，而是「如果這台還是空車，才改成已配」，而且這個「確認＋改」是一個不可被打斷的原子動作。所以只有一個人會成功，另一個會發現「已經不是空車了」，被自動換到下一台候選——就像售票系統只讓一個人結帳成功。
Q3：「即時追蹤車子」靠什麼？用外送 App 的比喻說說看。 靠 WebSocket（不掛斷的電話）：系統跟你開一條長連線，車子每動一下就把新位置主動推到你螢幕，不用你一直按重新整理——就像外送 App 上那台小機車自己一格一格往你家移動。
Q4：這系統「最怕」哪三件事？ 怕一台車被搶兩次（指派要強一致，用搶票術）、怕找車太慢（用網格先縮範圍再精算）、怕位置更新淹沒系統（當前位置只在記憶體覆寫，歷史另存）。
Q5（工程）：為什麼「車現在的位置」放記憶體、不放正式資料庫？ 因為它每幾秒變一次、只要最新那筆。記憶體覆寫又快又省（像白板擦了重寫）；每秒 25 萬筆若都慎重寫進硬碟資料庫會被壓垮。舊位置不需要留在熱路徑，歷史軌跡另外用串流存到倉庫即可。
Q6（工程）：搶車失敗（車剛被別人搶走）時，系統該怎麼做？ 不該直接回報錯誤給乘客。因為候選清單本來就按距離排好，搶不到就自動換下一台候選（也是附近的近車）再試一次。乘客通常不會察覺，只是配到第二近的車而已。
Q7（工程）：行程中「位置」可以晚 1 秒，但「車的指派狀態」不能錯一次，這是什麼取捨？ 強一致 vs 最終一致。指派狀態牽涉「車有沒有被搶走」，錯了會有人撲空，必須強一致（用原子操作）。位置追蹤晚 1 秒看到車在哪沒人在意，可以最終一致——把嚴格的成本只花在真正需要的地方。

15. 30 秒回顧

RoboTaxi 叫車 = 在一張塞滿百萬台移動車的活地圖上，瞬間抓一台最近的空車，且保證不被搶兩次。

3 個關卡：① 用網格（geohash）找附近的車 → ② 用「搶最後一張票」的原子指派（CAS）搶一台 → ③ 開一條不掛斷的電話（WebSocket）即時追蹤。

3 個怕：怕一台車被搶兩次（強一致、搶票術）、怕找車太慢（先縮範圍再精算）、怕位置更新淹沒系統（當前位置記憶體覆寫、歷史另存）。

工程細節一句話：先估「百萬連線、每秒 25 萬筆位置寫入」的形狀 → 用「點餐單(API)」溝通、位置走長連線 → 三本筆記本（車況放記憶體/行程放資料庫/軌跡另存倉庫）＋預存 geohash → 找出塞車環節各自拓寬。
