1. 60 秒搞懂它在幹嘛

你肚子餓，打開 App 點了一份牛肉麵。幾分鐘後你看到「餐廳已接單 → 製作中 → 外送員已取餐 → 正在送來」， 地圖上有個小機車圖示一路朝你移動，還告訴你「還有 8 分鐘到」。這就是外送平台。

它要同時服務三種人：點餐的食客、做菜的餐廳、跑腿的外送員。 系統像個總指揮：收到單 → 通知餐廳做 → 在附近找一位外送員去接 → 全程讓你看到進度。

🔑 一句話 外送平台 = 串起「食客 / 餐廳 / 外送員」三方的即時總指揮： 一張單進來，要找到最近的外送員（而且別把同一位派去兩張單）、盯著餐做好沒、再讓你即時看到餐到哪了、還要多久。

2. 為什麼這題的難點不是「記訂單」？

很多人第一次看會想：「不就是存一筆訂單、改改狀態嗎？」——這想法漏掉了三個真正的難關。

📌 比喻：餐廳門口的調度員 想像尖峰時段，一條街上有幾十位外送員在跑、幾百家餐廳同時出餐。 一張單做好了，你要瞬間在地圖上找出「離這家餐廳最近、現在有空」的外送員派過去。 你不可能把所有外送員從頭問一遍「你在哪、有空嗎」（太慢），也絕不能把同一位外送員同時派給兩張單（他分身乏術）。

所以三個難關是：

  • ① 在地圖上快速找最近的人（跟叫車一樣，叫「地理媒合」）。
  • ② 同一位外送員別被搶兩次（兩張單幾乎同時派，都看上同一位 → 要有「先搶先贏、搶到才算」的機制）。
  • ③ 多了「餐廳備餐」這一關：太早派外送員到了乾等做菜，太晚派你又多等外送員過來——抓派單的時機是外送獨有的頭痛事。

💭 停一下：為什麼「同一位外送員不能被派兩張單」這件事，需要特別設計？

看答案 因為兩張單可能幾乎同時要派，系統一查附近，兩張單都看到「外送員 c2 有空」，於是各自把 c2 派走 → c2 同時背兩單。 解法是「搶到才算」：派的時候做一個條件動作「只有當 c2 還是『有空』才把他改成『忙』」，誰先改成功誰拿到，另一張單發現改不動（已被搶）就自動換下一位。這叫 CAS（條件更新）。

3. 把系統拆成 4 個關卡

別被「系統」嚇到。整件事其實就是闖 4 關，一關一關來：

1下單 + 餐廳備餐：把單立起來

食客送出訂單，系統建一筆單，狀態先是「已下單」。接著餐廳「接受」、開始「備餐」。

🎫 比喻：取餐號碼牌 就像速食店給你一張號碼牌，上面的狀態會一格一格往前走：已點餐 → 製作中 → 可取餐。 外送多一層：可取餐之後還要有人幫你跑這一趟。

這串「一格一格往前走、不能跳格也不能倒退」的設計，工程上叫狀態機。

2派單：在地圖上找最近、有空的外送員

餐快做好了，要在餐廳附近找一位外送員去接。怎麼「快速找附近」？跟叫車一模一樣的招數。

🗺️ 比喻：把地圖切成方格找人 與其問遍全城外送員「你在哪」，不如把地圖切成一格一格的方格（geohash）。 要找餐廳附近的人，只看餐廳那格和周圍幾格就好，不用掃整張地圖。找到候選後，再用真實距離排個序，挑最近的。

挑到最近那位後，做一個「搶到才算」的動作把他標成「忙」——這就是第 2 節說的，防止同一位被派兩次。搶輸了就自動換下一位。

3取餐 → 送達：狀態繼續往前走

外送員到餐廳取餐、再送到你家。號碼牌繼續走：已取餐 → 已送達。送達後，這位外送員恢復成「有空」，可以接下一單。

🚦 比喻：紅綠燈只能照順序變 紅燈不會直接跳綠燈（中間要黃燈），也不會倒退。訂單狀態也是：只能照順序往前推， 系統每次推進前都檢查「現在這格對不對」，亂序或重複按都會被擋下來——這樣才不會出現「還沒取餐就顯示已送達」的怪事。

4即時追蹤：餐到哪了？還要多久？

全程你都想知道「外送員到哪了、還有幾分鐘」。外送員的手機每幾秒回報一次 GPS 位置，系統據此更新地圖上的小機車，並重新估「還要多久到」（ETA）。

📍 比喻：朋友傳即時位置給你 就像朋友開車來接你，一路把「即時位置」分享給你看，你看著那個點越來越近，心裡有底。 外送員越靠近，ETA 的數字就越小——這個「會隨位置變化的剩餘時間」就是動態 ETA。
✅ 四關一句話 ① 下單 + 餐廳備餐（號碼牌起步）→ ② 派單（地圖找最近、搶到才算）→ ③ 取餐送達（狀態照順序走）→ ④ 即時追蹤（看小機車移動 + 動態 ETA）。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「同一位外送員被派兩單」 兩張單幾乎同時派、都看上同一位有空的外送員 → 他被迫分身。 所以才需要「搶到才算（CAS 條件更新）」：誰先把他從「有空」改成「忙」誰拿到，另一張單發現改不動就換人。同一位，只能被一張單拿走。
😱 怕「派單時機抓不準」 太早派 → 外送員到餐廳乾等做菜（浪費他時間）；太晚派 → 餐做好了沒人接，食客多等。 所以要估餐廳大概多久出餐，反推「提前多久派」，讓外送員到達時餐剛好做好。剛剛好，不早不晚。
😱 怕「位置更新太多，把系統壓垮」 幾十萬外送員每幾秒回報一次位置，這個「寫入量」是全系統最大的負擔。 所以「當前位置」只放在記憶體裡覆寫（只留最新、查最近用），「歷史軌跡」另外慢慢存檔，兩者分開走。熱的放手邊，冷的另外收。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
同時上線外送員尖峰 ~50 萬位要維持幾十萬條「即時連線」聽他們回報位置
位置回報（寫）每人每 4 秒一次 → 約 12 萬+ 筆/秒這是全系統最大的「寫入」壓力 → 當前位置只放記憶體
下單（要派單）尖峰約每秒 100 筆每筆要「查附近 + 搶到才算 + 算兩段 ETA」
進行中訂單同時數十萬筆狀態一直在變 → 放可靠、可審計的主資料庫
💡 關鍵洞察：壓力不在「下單」，在「位置回報」 每秒才一百筆下單，聽起來很閒。但幾十萬外送員每幾秒回報一次位置，加起來是每秒十幾萬筆寫入——這才是大頭。 所以系統把「當前位置」放記憶體覆寫（只留最新），歷史另外慢慢存，這樣才扛得住。

💭 停一下：下單量不大，為什麼還要那麼在意「位置回報」？

看答案 因為位置回報的筆數是下單的上千倍：一筆訂單從頭到尾才幾次狀態變更，但外送員這段時間每幾秒就回報一次位置。 真正會把資料庫寫爆的是這股「位置洪流」，所以才要把它跟訂單分開、只放記憶體覆寫。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」：

① 食客下單（最常用）
   POST /api/order
   給我：{ 食客, 餐廳座標, 送達座標 }
   我回：201「建好了！狀態：placed（已下單）」
② 推進狀態（餐廳接單備餐 / 外送員取餐送達都走這支）
   POST /api/order/advance   給我：{ 訂單編號 }
   → 把號碼牌往前推一格（亂推會被擋）

所有狀態變更走同一支 advance，由系統檢查「現在這格能不能往前」——比每個動作各開一支 API 更不容易出錯。

③ 外送員上線 / 回報位置
   POST /api/courier/location   給我：{ 外送員 id, lat, lng }
   → 更新他在地圖上的位置（模擬每幾秒一次的 GPS 串流）
④ 派單（地圖找最近、搶到才算）
   POST /api/dispatch   給我：{ 訂單編號 }
   我回：派到哪位外送員、距餐廳多遠、ETA 幾分鐘
⑤ 追蹤（看餐到哪了、還要多久）
   GET /api/track/{訂單編號}
   我回：{ 狀態, 外送員目前位置, 即時 ETA }

9. 系統要記住哪些事？（資料模型）

📒 比喻：兩本筆記本 + 一張地圖 系統把資料記在「筆記本」（資料表）裡。這題主要兩本筆記本，加一張會動的地圖：

🛵外送員簿（誰在哪、有沒有空）

每位外送員一列：外送員編號、目前座標、狀態(有空/忙/離線)、目前在送哪張單。 「座標」會被高頻覆寫（一直更新成最新位置）；「狀態」是搶單時 CAS 要比對的關鍵欄位。

🧾訂單簿（每張單走到哪一格了）

每張單一列：訂單編號、食客、餐廳座標、送達座標、狀態(placed→accepted→preparing→picked_up→delivered)、指派到哪位外送員、ETA、每次轉移的時間戳。 多存「每次轉移的時間」，是為了事後查帳 / 客服（誰幾點接單、幾點取餐）。

🗺️地圖索引（找附近用——核心！）

給一個座標（餐廳），要能瞬間取出附近、有空的外送員。 工程上把外送員位置丟進geohash 方格（或更高級的 Redis GEO / S2），查附近只看那幾格。

為什麼不用「一個個算距離」？ 如果每次派單都把全城外送員都算一遍距離（幾十萬人 → 慢）。 切成方格後，只看餐廳那格和周圍幾格就把範圍縮到幾十人，再精算排序——這就是把「掃全城」變成「看附近幾格」的精髓。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
位置回報：幾十萬人每幾秒回報當前位置只放記憶體覆寫（只留最新），歷史另外慢慢存檔
派單搶人：同一位被兩張單搶搶到才算（CAS），搶輸就換下一位候選；鎖只鎖到「單一外送員」
找附近：全城外送員逐一算距離用地圖方格（geohash）只看附近幾格，再精算排序
用餐尖峰某商圈：單多人少動態加給 / 加價引導外送員過去 + 排隊派發 + 方格自動細分
派單算太久 / 挑太隨便分區並行計算、限時就退回挑最近、依備餐時間提前派
跨城查詢沒意義還拖慢以城市 / 區域分區隔離，就近部署

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 挑最近(貪婪) vs 順路批量 一單一派、挑最近最快最直觀；但每幾秒收一批一起算（甚至一位外送員順路帶多單）總路程更省、外送員賺更多，代價是派單稍微慢一點、實作複雜。
⚖️ 一下單就派 vs 等餐快好才派 一下單就派：外送員早早出發但可能到餐廳乾等做菜；等餐快好才派：外送員少等但要準確估出餐時間。折衷是用備餐預估反推何時派。
⚖️ 位置回報要多密？ 回報越密（每秒）→ 追蹤越準但寫入量爆、外送員耗電；越疏（每 10 秒）→ 省成本但 ETA 會抖。常見折衷：快到了才加密回報。
💭 追問：外送員接了單卻失聯 / 拒單，系統怎麼辦？ 把這張單退回待派、釋放外送員、重新媒合給下一位；對拒單 / 失聯設懲罰。位置太久沒更新的（殭屍）用「心跳 + 逾時淘汰」移出地圖索引。
💭 追問：為什麼訂單狀態一定要「照順序、不能跳格」？ 避免出現「還沒取餐就顯示已送達」這種錯亂，也防止並發下重複推進。每次推進前都檢查「現在這格對不對」（用 CAS），不對就擋下來——這就是狀態機的價值。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「地理派單 + 搶到才算 + 狀態機 + 即時 ETA」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8057 -t public
# 然後瀏覽器打開 http://localhost:8057
  1. 先讓幾位外送員上線報位置（不同座標，模擬路上的車）。
  2. 食客下單（填餐廳座標 + 送達座標）→ 把訂單推進到 accepted → preparing（模擬餐廳接單備餐）。
  3. 派單：看系統派到「離餐廳最近、有空」的那位，顯示距餐廳多遠、ETA 幾分鐘。
  4. 再下一張同餐廳的單、也派單 → 看它派給不同的人（同一位不會被搶兩次！）。
  5. 繼續推進 picked_up → delivered，送達後那位外送員變回有空。移動某位外送員的位置，回追蹤頁看 ETA 跟著變。
玩的時候想一下同一個餐廳連派兩張單，看兩張單是不是派到不同人（這就是「搶到才算」防重複）。 把外送員位置移近送達點再看追蹤，ETA 是不是變小了（這就是動態 ETA 跟著位置走）。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 GeoIndex（地圖索引：找附近最近的人）→ Dispatcher（派單大腦 + 訂單狀態機）→ Store（兩本筆記本 + 「搶到才算」的鎖）。

1找附近最近的人（GeoIndex）

在「餐廳座標」附近，把有空的外送員照距離由近到遠排出來：

function 找附近可用外送員(餐廳lat, 餐廳lng) {
    對每一位外送員:
        if (他不是「有空」) 跳過          // 忙 / 離線的不算
        算他到餐廳的直線距離(haversine)
    照距離由近到遠排序
    回傳前幾位                         // 這就是「候選名單」
}

demo 為了好懂用迴圈掃一遍；真實系統把它換成 Redis GEO（geohash 方格），只看附近幾格，一次就撈出來——邏輯一樣，只是換更快的工具。

2派單：挑最近 + 搶到才算（Dispatcher + Store）

這是防重複指派的心臟。拿到候選名單後，從最近的開始一個個試「搶」：

候選 = 找附近可用外送員(餐廳座標)
對每一位候選 c（由近到遠）:
    搶到了 = 把 c 從「有空」改成「忙」(只有他還有空才改得動)  // ← CAS
    if (搶到了) {
        把 c 綁到這張單; 算 ETA; 回傳這張單   // 成功！
    }
    // 沒搶到（已被別張單搶走）→ 自動換下一位候選
回傳「附近沒有可用外送員」

關鍵就是那句「只有他還有空才改得動」（CAS）。兩張單同時搶同一位，只有一個改得動，另一個自動跳下一位——同一位永遠不會被派兩次。這跟叫車那題（RoboTaxi）防止同一台車被搶兩次，是一模一樣的招數。

3訂單狀態機：只能照順序往前（Dispatcher）

每次推進，先查「現在這格的下一格是什麼」，而且只有現在這格沒被別人改過才推得動：

下一格表 = { placed→accepted, accepted→preparing,
            preparing→picked_up, picked_up→delivered }

function 推進(訂單) {
    現在 = 訂單.狀態
    if (現在沒有下一格) return 「已是終點，不能再推」
    下一格 = 下一格表[現在]
    if (現在=preparing 但還沒派外送員) return 「先派外送員才能取餐」
    推成功 = 把狀態從「現在」改成「下一格」(只有還在「現在」才改得動)  // ← 也是 CAS
    if (下一格 = delivered) 把外送員改回「有空」      // 送達 → 釋放
}

「只能照順序、不能跳格、不能重複」全靠這段。亂序或並發重複推進都會被「只有還在現在這格才改得動」擋下來。

4即時 ETA：外送員一動就重算（Dispatcher 追蹤）

追蹤時，依「現在走到哪一格」算剩餘時間：

function 追蹤(訂單) {
    if (還沒取餐)  ETA = 備餐剩餘 + (外送員→餐廳→食客) ÷ 平均車速
    if (已取餐)    ETA = (外送員→食客) ÷ 平均車速        // 只剩送這段
    if (已送達)    ETA = 0
    回傳 { 狀態, 外送員目前位置, ETA }
}

外送員每次回報新位置，下次追蹤算出來的 ETA 就跟著變小——這就是動態 ETA。真實系統把「直線距離 ÷ 平均車速」換成路網路由引擎（會考慮紅綠燈、路況），判斷邏輯不變。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
② 派單找最近GeoIndex找附近可用外送員()
② 搶到才算（防重複）Dispatcher + Store候選逐一 CAS 改「忙」
①③ 狀態機照順序走Dispatcher推進() 的下一格表 + CAS
④ 即時 ETADispatcher追蹤() 依進度分兩段算

👉 重點：真實系統會把「JSON 檔 + 迴圈」換成「Redis GEO + 分散式鎖 + 訊息佇列」， 但上面這幾段判斷邏輯一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：這題跟「叫車（RoboTaxi）」最大的不同是什麼？ 多了「餐廳備餐」這一環。叫車是人上車就走；外送要先等餐廳把餐做好，所以多了派單時機的取捨（太早外送員乾等、太晚食客多等），ETA 也變成「外送員→餐廳→食客」兩段路。
Q2：為什麼派單要「搶到才算（CAS）」？不這樣會怎樣？ 因為兩張單可能同時看上同一位有空的外送員。不用 CAS 的話兩張單各自把他派走 → 同一位背兩單。用 CAS「只有他還有空才改得動」，誰先改成功誰拿到，另一張自動換下一位。跟叫車防止同一台車被搶兩次同招。
Q3：用「切方格」解釋怎麼快速找附近外送員。 把地圖切成 geohash 方格，找餐廳附近只看「餐廳那格 + 周圍幾格」，把候選從全城幾十萬人縮到幾十人，再用真實距離（haversine）精算排序挑最近——不用掃整張地圖。
Q4：訂單狀態為什麼要做成「只能照順序走的狀態機」？ 避免「還沒取餐就顯示已送達」這種錯亂，也防並發下重複推進。每次推進前檢查「現在這格能不能往前」（用 CAS），亂序 / 重複都會被擋下。
Q5：下單量不大，為什麼系統的最大壓力是「位置回報」？ 因為位置回報筆數是下單的上千倍：幾十萬外送員每幾秒回報一次，加起來每秒十幾萬筆寫入。所以「當前位置」放記憶體覆寫、只留最新，歷史另外存，才扛得住。
Q6（工程）：「動態 ETA」是怎麼隨外送員位置變化的？ 外送員每次回報新位置，追蹤時就重算剩餘時間：還沒取餐算「外送員→餐廳→食客」，取餐後只算「外送員→食客」。外送員越靠近、ETA 越小。真實系統用路網路由取代直線距離，邏輯不變。
Q7（工程）：外送員接單後失聯，系統要做哪些事？ 把單退回待派、釋放該外送員、重新媒合給下一位；對失聯 / 拒單設懲罰；位置太久沒更新的用「心跳 + 逾時淘汰」移出地圖索引，避免被當成可用的殭屍。

15. 30 秒回顧

外送平台 = 串起「食客 / 餐廳 / 外送員」三方的即時總指揮。

4 個關卡：① 下單 + 餐廳備餐（號碼牌起步）→ ② 派單（地圖找最近、搶到才算）→ ③ 取餐送達（狀態照順序走）→ ④ 即時追蹤（看小機車移動 + 動態 ETA）。

3 個怕：怕同一位外送員被派兩單（CAS 搶到才算）、怕派單時機抓不準（用備餐預估反推何時派）、怕位置更新壓垮系統（當前位置放記憶體覆寫、歷史另存）。

工程細節一句話：它是叫車題的升級版（多了備餐這一環）→ 用「點餐單(API)」溝通、外送員簿 + 訂單簿 + 地圖索引三本記憶 → geohash 找附近、CAS 防重複、狀態機照順序、ETA 隨位置動 → 找出塞車環節各自拓寬，最大的塞點是位置回報量。
