⑨ Google Docs：字元級即時協作編輯器

B 即時連線/同步 建議 48 分鐘 Design a Collaborative Document Editor

一句話定位 本質是「多人同時對同一份文件做字元級編輯，且結果必須收斂一致」的即時同步題。 CRUD 存檔很簡單；面試真正考的是 並發衝突解決——當 A、B 基於同一版本同時打字，如何讓兩邊套用後文件收斂到相同內容、誰都不被覆蓋。核心就是 OT（Operational Transformation）vs CRDT。

1. 需求釐清

功能性需求

  • 字元級即時協作：多人同時編輯同一份文件，每個按鍵級的 insert/delete 操作近即時（< 200ms）同步給所有人。
  • 衝突收斂：並發編輯後，所有客戶端最終看到完全相同的文件內容（收斂性 / convergence）。
  • 游標與選取共享（presence）：看得到別人的游標位置、選取範圍、上線狀態。
  • 版本歷史與快照：可回溯任意時間點、還原版本、看誰改了什麼。
  • 離線編輯：斷線可繼續編輯，重連後與雲端合併。
  • 權限分享（檢視 / 留言 / 編輯）、留言批註。

非功能性需求

  • 低延遲：本地操作立即顯示（樂觀更新），同步延遲 p99 < 200ms。
  • 強最終一致：可暫時分歧，但必須收斂；絕不可永久分歧或丟字。
  • 高可用：協作伺服器需具備故障轉移；連線斷掉要能無痛重連續傳。
  • 持久性：已確認的操作不可遺失（操作日誌持久化）。
  • 可擴展：單文件數十並發編輯者、全站數百萬份活躍文件。
面試開場必問 「衝突解決要用 OT 還是 CRDT？要不要支援離線？」這兩問決定整題走向。需要中央權威排序、頻寬敏感 → OT（Google Docs 走這條）；需要去中心化、離線優先、P2P → CRDT（Figma、Apple Notes 傾向這條）。

2. 規模估算

指標假設推算
活躍文件同時編輯中 100 萬份需百萬條 WebSocket 長連線 → 連線層水平分片
單文件並發者平均 2~5、峰值數十單文件操作需序列化（單一 owner 節點）
操作頻率（寫）每人打字 ~5 ops/s單文件 5 人 ≈ 25 ops/s；廣播放大 ×5（N 個訂閱者）
單操作大小insert 1 字 + 中繼資料 ~100 B頻寬主要花在廣播放大而非單筆大小
操作日誌每文件每日數千 ops × 數百 B定期快照壓縮，否則日誌無限成長
儲存文件本體 + 快照 + 近期日誌快照 + 增量日誌，回放成本可控
關鍵洞察讀寫不是重點，並發寫的收斂才是。瓶頸在「單文件操作必須由單一權威節點序列化」與「廣播放大」——這決定了路由與分片策略。

3. API 設計

# 建立 / 開啟文件（HTTP）
POST /api/v1/docs                 → { "doc_id": "d_abc", "version": 0 }
GET  /api/v1/docs/{id}            → { "content": "...", "version": 42 }  # 載入快照

# 即時協作走 WebSocket（核心）
WS   /ws/docs/{id}

→ 客戶端送出操作（基於它目前看到的版本）
{ "type": "op", "doc": "d_abc", "baseVersion": 42,
  "op": { "kind": "insert", "pos": 5, "text": "X" } }

← 伺服器排序後回 ACK + 廣播（已 transform 過、附權威版本號）
{ "type": "ack",  "version": 43 }
{ "type": "op",   "version": 43, "op": { "kind":"insert","pos":5,"text":"X" }, "by": "userA" }

# presence：游標 / 選取（高頻、可丟、不進操作日誌）
{ "type": "presence", "user": "userB", "cursor": 12, "selection": [12,18] }

# 版本歷史
GET  /api/v1/docs/{id}/history    → [{version, ts, author}]
GET  /api/v1/docs/{id}/snapshot/{version}
為何 op 要帶 baseVersion伺服器靠它判斷這個操作「基於哪個版本」，才知道要對它做幾次 transform（追上權威版本）。這是 OT 的命脈。

4. 資料模型

documents
  doc_id        VARCHAR  PRIMARY KEY
  title         TEXT
  owner_id      BIGINT
  current_ver   BIGINT                 -- 權威版本號（單調遞增）
  created_at    TIMESTAMP

snapshots (定期壓縮，避免回放整條日誌)
  doc_id, version, content TEXT, created_at

operations (append-only 操作日誌 — OT 的真相來源)
  doc_id, version BIGINT,             -- 套用後的權威版本
  author_id, kind ('insert'|'delete'),
  pos INT, text TEXT, len INT, ts

  PRIMARY KEY (doc_id, version)        -- 同一文件版本嚴格有序

presence (記憶體 / Redis，不持久化)
  doc_id, user_id, cursor, selection, last_seen
為何分快照 + 日誌純靠日誌回放成本隨時間爆炸；純存內容又丟失歷史。快照 + 增量操作日誌：載入時取最近快照再回放少量後續操作，兼顧載入速度與完整歷史。presence 是高頻易逝資料，放記憶體/Redis，絕不進持久化日誌。

5. 高階架構

🖼️ 開啟 Excalidraw 架構圖（diagram.excalidraw） 
 Client A ──┐                         ┌─▶ 廣播 ─▶ Client A
 (本地樂觀) │   WebSocket            │           (套 transform 後)
            ├─▶ 連線層 ─▶ 協作伺服器 ─┤
 Client B ──┘   (Gateway) (單文件權威) ├─▶ 廣播 ─▶ Client B
 (本地樂觀)                  │  ▲      │
                            │  │      └─▶ 操作日誌(append-only)
                  排序 + OT transform        │
                            │                ▼
                            └──────▶ 文件快照儲存 / 版本歷史

 presence(游標/選取) ──▶ Pub-Sub(Redis) ──▶ 廣播給同文件其他人（不進日誌）
  1. 連線層 / Gateway：維持百萬 WebSocket 長連線，依 doc_id 把同一文件的客戶端路由到同一台協作伺服器（一致性雜湊）。
  2. 協作伺服器（單文件權威）：對某文件的所有操作做序列化排序，對落後版本的操作做 OT transform，賦予權威版本號後廣播 + 寫入操作日誌。
  3. 操作日誌：append-only，是收斂的真相來源；定期觸發快照壓縮。
  4. Pub-Sub：游標/選取等 presence 走 Redis pub-sub 廣播，與操作管線解耦（可丟、不影響收斂）。

6. 核心元件深入：OT vs CRDT（本題重點）

這段就是考點面試官要聽的不是「我選 X」，而是「我懂 OT 的 transform 與 CRDT 的無衝突合併各自如何達成收斂、代價在哪、誰用在哪」。下面逐一拆。

問題本質：並發操作的位置會錯位

文件目前是 "Hello"（版本 v0）。A 與 B 同時基於 v0 送出操作：

A: insert(pos=0, "X")   想得到 "XHello"
B: insert(pos=5, "!")   想得到 "Hello!"

伺服器先收到 A，套用得 "XHello"（v1）。接著收到 B 的操作，但 B 的 pos=5 是基於舊的 v0算的；此刻文件已因 A 在前面插了一個字而整體右移一格。若直接套 pos=5，"!" 會插在 "XHell|o" 而非結尾，結果發散、不收斂。這就是必須 transform 的根因。

方案 A：OT（Operational Transformation）

中央伺服器維護權威版本號 + 操作日誌。當一個操作的 baseVersion 落後權威版本時，伺服器把它依序對「中間已套用的每個操作」做 transform，調整其位置，再套用。核心是 transform(opB, opA)：在 opA 已先套用的前提下，回傳「修正後的 opB」。

// insert vs insert：A 先套用，調整 B 的位置
transform(B=insert(pos_b, text_b), A=insert(pos_a, text_a)):
    if pos_a <= pos_b:
        pos_b += len(text_a)      // A 插在 B 之前/同位 → B 右移
    return insert(pos_b, text_b)
// 例：B.pos 5 + len("X") 1 = 6 → "!" 插在 "XHello" 結尾 = "XHello!"  ✓收斂

// delete vs insert：A 插字 → 在插入點之後的刪除位置要右移
// insert vs delete：A 刪字 → B 的位置左移；若刪除範圍涵蓋 B → 夾到刪除點
// delete vs delete：需處理範圍重疊（避免重複刪 / 刪過頭）
  • 收斂保證：滿足 TP1 性質 apply(apply(s,A), T(B,A)) == apply(apply(s,B), T(A,B))，兩條套用順序殊途同歸。
  • 優點：操作小（一個 insert 幾 byte）、頻寬省；有中央權威，排序簡單；游標位置易維護。
  • 代價：transform 函式組合爆炸難寫對（insert/delete × 各情況），歷史上多家實作出過收斂 bug；強依賴中央伺服器序列化，不適合純 P2P。
  • 誰用：Google Docs、Etherpad。需要中央排序、頻寬敏感、富文本屬性多時的首選。

方案 B：CRDT（Conflict-free Replicated Data Type）

不靠中央 transform，而是讓資料結構天生可無衝突合併。文字常用 sequence CRDT（如 RGA、Logoot/LSEQ、Yjs 的 YATA）：每個字元給一個全域唯一、可比較且稠密的 ID（不是脆弱的整數位置）。插入只是「在某兩個字元 ID 之間放一個新 ID」，刪除是「打 tombstone 墓碑」。任意順序合併，結果都收斂。

"Hello" 的字元帶 ID： H(1) e(2) l(3) l(4) o(5)
A: 在 <起點, H(1)> 之間插 X，分得 ID 0.5  →  X(0.5) H e l l o
B: 在 <o(5), 終點> 之間插 !，分得 ID 6    →  H e l l o !(6)
合併：依 ID 全序排序 → X(0.5) H e l l o !(6) = "XHello!"  ✓收斂
（兩端不論先收到誰、收幾次，依 ID 排序結果恆等 → 冪等、可交換）
  • 收斂保證：合併滿足交換律 + 結合律 + 冪等（CmRDT/CvRDT 性質），不需中央協調即收斂，天生離線友善。
  • 優點：去中心化 / P2P 可行；離線編輯重連直接合併；無需寫複雜 transform。
  • 代價：每字元帶 ID 與 tombstone → 中繼資料肥大、記憶體/頻寬開銷高；墓碑需垃圾回收；游標映射較麻煩。現代實作（Yjs/Automerge）已大幅優化。
  • 誰用：Figma、Apple Notes、Yjs/Automerge 生態。離線優先、P2P、本地優先（local-first）場景首選。

一句話對比

維度OTCRDT
收斂方式中央 transform 調整位置結構天生可交換合併
需中央伺服器是（序列化排序）否（P2P 可行）
頻寬 / 中繼資料小（操作精簡）較大（每字元 ID + 墓碑）
離線優先弱（重依賴中央）強
實作難度transform 難寫對結構設計難，但合併簡單
代表Google Docs、EtherpadFigma、Yjs、Automerge

presence、版本歷史、離線

  • presence（游標/選取）：高頻、可丟、不收斂也無妨 → 走獨立 pub-sub，不進操作日誌；別人插字時本地把游標位置一起 transform 即可跟著移動。
  • 版本歷史 / 快照：操作日誌即天然歷史；定期把日誌壓成快照，回溯時取快照 + 回放後續操作。
  • 離線編輯合併：OT 重連時把離線期間累積的操作逐一對伺服器中間操作 transform 後補送；CRDT 直接把本地變更集合與雲端合併（這正是 CRDT 的主場）。

7. 擴展與瓶頸

階段瓶頸對策
連線量百萬 WebSocket 長連線連線層水平分片；依 doc_id 一致性雜湊把同文件導到同節點
單文件熱點單一權威節點序列化成瓶頸單文件並發通常有限；熱文件升配 + 操作合批；presence 與操作分流
廣播放大N 訂閱者 → 一筆操作放大 N 倍節點內本地 fan-out；跨節點才走 pub-sub，減少跨機流量
日誌無限成長回放成本爆炸、儲存膨脹定期快照壓縮 + 冷日誌歸檔；CRDT 需墓碑垃圾回收
節點故障權威節點掛掉操作日誌持久化 → 新節點回放快照+日誌接手；客戶端重連續傳

8. 取捨與面試重點

  • OT vs CRDT：要中央排序 + 省頻寬 + 富文本 → OT（Google Docs）；要離線優先 / P2P / local-first → CRDT（Figma、Yjs）。能講清「為何 Google 選 OT 而 Figma 選 CRDT」就贏一半。
  • 樂觀本地更新 vs 等伺服器確認：本地立即顯示（體驗好）→ 必須能在收到 ACK/他人操作時把本地未確認操作重新 transform（client 端也跑一套）。
  • 強一致 vs 最終一致：協作天生最終一致——允許短暫分歧，但收斂是硬底線，且操作不可丟。
  • presence 與操作解耦：游標可丟、操作不可丟，混在一起會拖垮收斂與持久化。
  • 常見追問：
    • transform 寫錯導致不收斂怎麼辦？→ TP1/TP2 性質測試、模糊測試大量隨機並發序列驗證收斂。
    • 離線一週回來怎麼合併？→ OT 補送 transform / CRDT 直接合併；過久可能需人工或版本分叉。
    • 富文本（粗體、樣式）？→ 操作要含屬性，transform 需處理屬性區間重疊。
    • 權威節點故障如何不丟操作？→ 已 ACK 必先落地操作日誌再回 ACK。

9. PHP Demo 說明

本題 demo 著重真正的考點——OT transform 讓並發編輯收斂，而非 CRUD 存檔。可實際用 PHP 內建伺服器跑起來：

  • src/OTEngine.php　最小可用 OT：insert/delete 操作、transform() 對同一基準版本的並發操作做位置調整；伺服器維護權威版本號與操作日誌。
  • src/DocStore.php　文件內容 + 版本號 + 操作歷史（JSON 檔模擬持久化）。
  • public/index.php　路由：載入文件 / 提交操作（自動 transform）/ 重設，並附深色測試頁（兩個編輯輸入模擬使用者 A、B）。
# 啟動
cd demo
php -S localhost:8009 -t public
# 瀏覽器開 http://localhost:8009
# 讓 A 在位置 0 插 "X"、B 在位置 0 插 "Y"（都基於同一版本）
# → 伺服器 transform 後兩者都套用、結果收斂一致、版本號遞增
誠實聲明 真實系統用 WebSocket 長連線 + 完整 OT（insert/delete 全情況 + TP1/TP2）或 CRDT，並有連線層分片、快照壓縮、presence pub-sub。本 demo 用同步 HTTP 模擬，OT transform 聚焦 insert/insert、delete vs insert、insert vs delete 的核心收斂邏輯——並發操作收斂、版本號遞增、操作日誌為真實可執行，足以展示「為何需要 transform、它如何讓兩個並發編輯收斂到同一結果」這一核心考點。
