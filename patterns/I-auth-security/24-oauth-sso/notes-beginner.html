1. 60 秒搞懂它在幹嘛

你一定看過這個畫面：到一個新網站（例如某個訂房 App），它不要你註冊，而是出現一顆 「用 Google 帳號登入」或「用 臉書 登入」的按鈕。你按下去、Google 問你「要把你的基本資料給這個 App 嗎？」， 你按「同意」，咻一下就登入成功了——而且你從頭到尾沒把 Google 的密碼交給那個 App。

OAuth 2.0 / SSO 就是在做這件事。它的厲害之處在於：讓你「授權」一個 App 代表你，卻不用把密碼交出去。

🔑 一句話 OAuth＝「不交鑰匙，只發臨時通行證」的代客登入：你向 Google 證明一次自己是誰，Google 發一張「有期限的識別證」給那個 App，App 拿著它去存取你的資料，全程碰不到你的密碼。

2. 為什麼不能直接把密碼給網站？

很多人第一個疑問是：「要登入，把帳號密碼給它不就好了？幹嘛搞這麼複雜？」想一個生活情境就懂了——

🚗 把車鑰匙交給代客泊車 你的車有一把主鑰匙，能開門、發動、還能打開後車廂跟手套箱（放著你的證件）。 代客泊車只需要「把車停好」這一件事，你會把能打開全部東西的主鑰匙交給他嗎？ 比較安全的做法，是給他一把「只能發動、開不了後車廂」的泊車鑰匙。

密碼就是那把「主鑰匙」。如果你把 Google 密碼交給某個訂房 App：

  • 它就能看你全部的信、改你的密碼、用你的身分到處跑——權限大到誇張，而它只需要「知道你 email」而已。
  • 它可能把你的密碼存起來（甚至外洩），你一個密碼全網通用，等於全部淪陷。
  • 你想收回授權時，唯一辦法是改密碼，連帶把其它正常的服務也一起踢掉。

所以 OAuth 的核心思想就是：把「證明你是誰」跟「允許 App 做某件事」拆開。 你只向「你信任的地方（Google）」證明身分，Google 再發一張權限有限、有期限的識別證給 App。 App 拿到的不是萬能主鑰匙，而是那把「只能發動的泊車鑰匙」。

💭 停一下：如果某個 App 把你的 Google「臨時識別證」弄丟了，會比把「密碼」弄丟好在哪？

看答案 識別證是有期限的（例如 15 分鐘就過期），就算被偷也只能用一下下；它的權限也有限（只能讀你同意的那幾項，不能改密碼）；而且你可以單獨撤銷這一張，不影響你的密碼和其它服務。密碼則是萬能又長期有效，弄丟＝整個帳號淪陷。

3. 把登入拆成 4 個關卡

別被流程圖嚇到。整件事其實就是闖 4 關，用「去遊樂園玩」的比喻一關一關走：

1你同意：「好，讓它代表我」

你在 App 按「用 Google 登入」，App 把你導去 Google 的頁面。Google 確認你已登入（或請你登入）， 然後問你：「這個 App 想讀你的姓名和 email，同意嗎？」你按「同意」。

🎡 比喻就像到遊樂園售票口，你親自到櫃台（Google）出示身分、說「我要進去」。App 沒有代你站在櫃台前——是你本人去的，所以你的密碼只給了 Google，沒給 App。

👉 關鍵：登入這一步一定發生在 Google 的網域上，App 全程看不到你輸入的密碼。

2拿到「一次性通行證」（授權碼）

你同意後，Google 不會馬上把正式識別證丟給瀏覽器，而是先發一張一次性的小紙條（授權碼）， 透過瀏覽器轉交給 App。這張紙條用過一次就作廢、而且只活幾十秒。

🎫 比喻：售票口給的「換票券」 售票口不會在大門口就把正式手環套你手上（怕在人擠人的門口被人順走）。 它先給你一張「換票券」，你拿著它到裡面的服務台，才換成真正的手環。 換票券就算掉了，撿到的人也只能換一次、而且馬上過期。

👉 為什麼要多這一步？因為這張紙條會短暫出現在瀏覽器的網址列（看得到、會被記錄）。 讓它「只能用一次、馬上過期」，就算被偷瞄到也沒用。

3換成「有期限的識別證」（access token）

App 的後端（不是瀏覽器）拿著那張一次性紙條，再附上只有自己知道的暗號（client_secret）， 偷偷去跟 Google 換一張正式識別證（access token）。這張識別證才是 App 真正用來存取你資料的東西。

🪪 比喻：到服務台換正式手環 到了裡面的服務台，工作人員核對你的換票券，外加確認 App「是不是它本人」（暗號），才把正式手環給它。 這個換的動作發生在後台、不在大門口，正式手環不會經過瀏覽器網址列，外洩風險小很多。

👉 重點：真正值錢的識別證，是在後端對後端悄悄換來的，不會在你看得到的網址裡晃。

4過期了，用「續命券」換新（refresh token）

那張識別證故意設計成很短命（例如 15 分鐘就過期），這樣就算被偷也只能用一下。 過期後 App 不會叫你重登一次，而是拿一張「續命券」（refresh token）去跟 Google 換一張新的識別證，你完全無感。

🏨 比喻：飯店房卡會「自動失效」 飯店房卡通常退房時間一到就刷不開門。但你還在住，怎麼辦？到櫃台「續住」一下，房卡就重新生效。 識別證短命、續命券去換新，就是這個道理——東西短命比較安全，但又不想一直煩你重登。
✅ 四關一句話 ① 你親自同意（密碼只給 Google）→ ② 拿到一次性通行證（授權碼）→ ③ 後端偷偷換成有期限識別證（access token）→ ④ 過期就用續命券（refresh token）換新。全程 App 碰不到你的密碼。

4. JWT＝蓋了鋼印的識別證

App 拿到識別證後，每次要存你的資料（例如讀你的姓名），都要把識別證亮給「資源伺服器」（管你資料的那台機器）看。 問題來了：資源伺服器怎麼確認這張識別證是真的、不是偽造的？

🛂 比喻：護照上的鋼印（鋼印章 / 浮水印） 機場海關看你的護照，不會打電話回你的國家問「這人是真的嗎」——那太慢了。 他直接看護照上的鋼印、浮水印、防偽線，這些東西只有官方做得出來、別人偽造不了， 一看就知道真偽。JWT 就是這種「自帶鋼印的識別證」。

JWT（讀作 "jot"）其實是一張紙條，上面寫著「你是誰、能做哪些事、什麼時候過期」，最後用 Google 的私章蓋了一個防偽簽章（鋼印）。 資源伺服器只要：

  • 驗鋼印：用 Google 公開的「驗章工具」確認簽章是真的（一改內容鋼印就對不上）。
  • 看有效期：紙條上寫的過期時間還沒到。

兩個都過就放行——完全不用打電話回 Google 問。這就是 JWT 最強的地方：驗證不必查資料庫、超快、能扛超大流量。

💭 停一下：既然 JWT 自帶鋼印、不用問 Google，那 Google 想「立刻收回」一張已經發出去的識別證，做得到嗎？

看答案 很難立刻收回。因為資源伺服器根本不會打電話問 Google，它只看鋼印和有效期。鋼印是真的、又還沒過期，它就放行——Google 想喊停也喊不到它。這就是 JWT「方便」帶來的代價（後面「最怕的事」會講怎麼補救：把識別證設得很短命＋黑名單）。

5. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「票被偷」 通行證、識別證在傳遞時都可能被偷看。所以全套設計都在縮小被偷的傷害： 一次性通行證用過即廢、幾十秒就過期；正式識別證故意短命（被偷也只能用一下）； 最值錢的識別證只在後端對後端交換，不經過瀏覽器網址列。每張票都「短命又限縮」。
😱 怕「收不回」 JWT 自帶鋼印、驗證不問 Google，好處是快，壞處是發出去就收不回，要等它自然過期。 萬一識別證被偷、或使用者按了「登出所有裝置」怎麼辦？ 解法：識別證設得很短命（撤銷最多等 15 分鐘就自動失效）＋把續命券列入黑名單（之後就換不到新的識別證，整條鏈自然枯萎）。短命＋黑名單，雙保險。
😱 怕「給錯權限 / 被冒充」 如果 App 能拿到「超過它需要」的權限，或壞人能假裝是你本人去按同意，就糟了。 所以：使用者同意時明列 App 能碰哪幾項（scope，最小權限）； 流程中夾帶一個對暗號（state），確保「回來的人」就是「出發的人」，防止有人半路插隊冒充（這叫防 CSRF）。只給剛好夠用的權限，並確認真的是你。

8. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

這題最關鍵的數字，是要分清楚兩種動作的頻率差很多：

估什麼大概數字所以呢？
發識別證（登入）1 億人，每天登 2 次 → 每秒約幾千筆量不算太大，可以慢一點、查資料庫沒關係
驗識別證（每次 API 都要驗）每人每天用 200 次 → 每秒幾十萬筆量爆大，絕對不能每次都打電話問 Google
一張識別證多大JWT 約 700~900 位元組每個請求都多揹一張，不能塞太多東西進去
🛂 關鍵洞察：海關蓋章 vs 打電話回國查 「驗票」比「發票」多了上百倍。如果每次驗都打電話回 Google 問（叫 introspection），Google 會被打爆。 所以才用 JWT（自帶鋼印）：資源伺服器當場看鋼印就放行，零查詢，像海關看護照鋼印而不是打電話回國。這就是 JWT 在大流量下無可取代的價值，也正是它「難以即時撤銷」這個取捨的來源。

💭 停一下：為什麼把「能不能放行」的判斷放進識別證自己身上（鋼印＋過期時間），比每次問 Google 更能撐大流量？

看答案 因為「每次問 Google」會讓所有流量擠到同一台中央機器，它就是瓶頸；而「當場看鋼印」每台資源伺服器各自就能判斷、互不打擾、沒有共用瓶頸，要扛更多就多加幾台機器即可（水平擴展）。把判斷分散出去，才擴得開。

9. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」，剛好對應前面四關：

① 去 Google 登入＋同意（走瀏覽器，關卡 1→2）
   GET /authorize?要哪些權限(scope)&回哪裡(redirect_uri)&對暗號(state)
   → 你登入並同意後，Google 把你導回 App，網址帶上「一次性通行證」(code)

這段走瀏覽器，因為要你本人登入。回來時只夾帶那張「一次性通行證」，不夾帶值錢的識別證，把外洩風險降到最低。

② 後端拿通行證去換識別證（後端對後端，關卡 3）
   POST /token  grant_type=授權碼 &code=通行證 &client_secret=暗號
   → { access_token: 識別證(JWT), refresh_token: 續命券, expires_in: 900(秒) }

這段後端對後端，要附上只有 App 知道的「暗號」。換來的識別證永遠不出現在瀏覽器網址列。

③ 識別證過期，用續命券換新（關卡 4）
   POST /token  grant_type=續命券 &refresh_token=續命券
   → 一張新的識別證（使用者完全無感）

④ 撤銷（把續命券列入黑名單）
   POST /revoke  token=續命券   → 一律回 200（不洩漏這張票存不存在）

⑤ 拿識別證去存資料（每次都做，量最大）
   GET /api/me   Authorization: Bearer 識別證
   → 200 你的資料   或   401「這張票無效/過期」

第 ⑤ 段就是「驗票」，每秒幾十萬次都走這裡——所以它只看鋼印＋有效期、不查資料庫。

10. 系統要記住哪些事？（資料模型）

📒 比喻：幾本筆記本 系統要把資料記在「筆記本」（資料表）裡。注意一件神奇的事：正式識別證（access token） 根本不用記下來！因為它自帶鋼印、能自己證明自己，系統不必存一份去對照。要記的只有這幾本：

🏢App 名冊（哪些 App 可以來要票）

每個合作的 App 一列：App編號、暗號(client_secret)、可導回的網址白名單、可申請的權限。 換票時會嚴格比對導回網址，免得壞人把通行證騙到自己的網站。

🎫通行證簿（一次性授權碼）

每張一次性通行證一列：通行證、屬於哪個 App、哪個使用者、過期時間、用過沒(used)。 「用過沒」這一欄是關鍵——用過就標記，再被拿來用就是攻擊訊號。這本天生短命，很適合放會自動過期的 Redis。

🔄續命券簿（refresh token）

每張續命券一列：續命券、哪個使用者、哪些權限、過期時間。 這本比較長命，要好好存。撤銷時就是把某張續命券丟進黑名單，之後它就換不到新識別證了。

11. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
驗票量爆（每秒幾十萬次）用自帶鋼印的 JWT當場驗、零查詢，每台資源伺服器各自判斷，要多快就多加機器
續命券簿查詢／寫入太頻繁把這本筆記切成好幾份分開放（依使用者切片）；通行證簿丟 Redis 自動過期

12. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 自帶鋼印(JWT) vs 每次問 Google JWT 驗證零查詢、超能擴展，但難即時撤銷、票較大、內容能被讀（別塞祕密進去）；每次問 Google 能即時撤銷，但量一大就把中央打爆。讀超多 → 選 JWT＋短命；強撤銷需求 → 摻一點黑名單查詢折衷。
⚖️ 識別證壽命：長 vs 短 壽命長 → 少換票、體驗順，但被偷／撤銷的空窗大；壽命短 → 撤銷快、安全，但要常常用續命券換新。常見折衷是 5~15 分鐘。
⚖️ 鋼印章：大家共用一把 vs 私章＋公開驗章 共用一把（HS256）簡單，但任一台機器被攻陷就能偽造票；改用「私章蓋、公開工具驗」（RS256），資源伺服器只拿得到驗章工具、偽造不了，還能輪替。對外開放／多服務 → 用私章公驗；內部信任圈才用共用章。
💭 追問：那個「對暗號(state)」到底防什麼？ 防有人在登入流程半路插隊冒充（CSRF）。App 出發時帶一個只有自己知道的暗號，回來時核對一樣，才確定「回來的人就是出發的那位」，不是被別人塞了一張假回應進來。
💭 追問：手機 App / 純前端網站沒辦法藏「暗號」怎麼辦？ 它們保管不了 client_secret。改用 PKCE：出發前自己產一個一次性的隨機祕密，先送出它的「指紋」，換票時再附上原本的祕密讓 Google 比對。等於把「固定暗號」換成「臨時暗號」，就算通行證被攔走，沒有那個臨時祕密也換不到票。

14. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那四關到底是哪幾段程式做的」。 好消息：核心只有兩個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 兩個檔 = 兩個小幫手 AuthServer（發票／換票／撤票的「售票服務台」：管一次性通行證、簽發/刷新/撤銷識別證）→ Jwt（蓋鋼印／看鋼印的「防偽工坊」：簽 JWT、驗 JWT）。走一次登入流程，就是這兩個檔輪流上場。

1一次性通行證：用過就作廢（AuthServer）

對應關卡 2→3：拿那張一次性通行證（授權碼）去換識別證。最關鍵的一句，是「用過沒？」的檢查——這就是「換票券撿到也只能換一次」那個比喻的本體。

function 用通行證換識別證(App, 暗號, 通行證, 導回網址) {
    if (App 不存在 或 暗號對不上)  return 拒絕;     // 先確認 App 是它本人
    這張 = 通行證簿[通行證]
    if (這張 不存在)              return 拒絕;
    if (這張.用過 == true)        return 拒絕「疑似重放攻擊」  // ← 一次性的核心！
    if (現在 >= 這張.過期時間)    return 拒絕;            // 只活 60 秒
    if (這張.App 或 導回網址 對不上) return 拒絕;          // 綁死當初發給誰

    通行證簿[通行證].用過 = true        // ← 先標記用過，再發票
    寫回通行證簿
    return 簽發識別證(...)               // 過了所有關才真的發票
}

就這一行 if (用過 == true)，把「同一張票換第二次」擋下來。注意順序：先把 used 標記成 true、再發票——就算發票過程出錯，這張票也已經作廢，撿到的人換不了第二次。demo 裡「同一張通行證換第二次被擋」演的就是這段。

2簽發 / 刷新 / 撤銷：三張票的一生（AuthServer）

對應關卡 3→4 的「發識別證 → 過期換新 → 收回」。重點是：access token（識別證）不入庫（它自帶鋼印能自證），只有 refresh token（續命券）要存起來，才撤得掉。

// 簽發：一次給兩張票
function 簽發識別證(App, 使用者, 權限) {
    識別證 = Jwt.簽(使用者, 權限, 存活=15分鐘)   // ← access：JWT，不入庫
    續命券 = 隨機字串("rt_")
    續命券簿[續命券] = { 使用者, 權限, 過期=14天 } // ← refresh：要存，之後才撤得掉
    return { 識別證, 續命券 }
}

// 刷新：續命券換新識別證（順便「輪替」舊券）
function 刷新(App, 暗號, 續命券) {
    if (續命券 在黑名單)  return 拒絕;          // 撤銷過就不給換
    if (續命券簿沒這張 或 已過期 或 不屬於此 App) return 拒絕;
    撤銷(舊續命券)                              // ← 輪替：舊的作廢、發一張新的
    return 簽發識別證(...)                       // 被盜重放會因為「舊券已撤」而露餡
}

// 撤銷：把續命券丟進黑名單
function 撤銷(續命券) { 黑名單[續命券] = 現在時間 }   // ← 之後它就換不到新識別證了

三個動作扣回前面的「飯店房卡」比喻：簽發＝發房卡＋一張續住券；刷新＝房卡過期拿續住券去櫃台換新（順便把舊續住券作廢，這叫輪替，被偷的舊券一用就露餡）；撤銷＝把續住券列黑名單，整條鏈自然枯萎。demo 裡「撤銷續命券後就換不到新識別證」就是這段。

💭 停一下：為什麼撤銷時是把「續命券」列黑名單，而不是把「識別證」列黑名單？

看答案 因為識別證（access token）是 JWT，自帶鋼印、資源伺服器當場驗、根本不查資料庫，列了黑名單它也不會去看。所以策略是：識別證設得很短命（最多十幾分鐘就自然失效）＋把續命券列黑名單（之後換不到新識別證）。短命＋黑名單雙保險，整條鏈枯萎。

3蓋鋼印：簽一張 JWT（Jwt）

對應第 4 節「蓋了鋼印的識別證」。一張 JWT 長這樣：header.payload.簽章，三段用點接起來。前兩段是「明文」（誰、能做什麼、何時過期），第三段的簽章就是那個鋼印。

function 簽(內容, 存活秒數) {
    header  = { 用哪種鋼印: "HS256" }
    payload = 內容 + { 簽發時間, 過期時間=現在+存活秒數, 發證單位 }

    h = base64url(header)            // 明文，編碼一下方便傳
    p = base64url(payload)           // 明文（注意：能被讀！別塞祕密進去）
    簽章 = base64url( hash_hmac("$h.$p", 密鑰) )   // ← 鋼印＝對「h.p」算 HMAC
    return "$h.$p.$簽章"             // 點接三段
}

鋼印的本體就是 hash_hmac(內容, 密鑰)：把「header.payload」這串內容＋一把只有發證方知道的密鑰，用 HMAC-SHA256 算出一段誰也偽造不出來的指紋。內容改一個字，算出來的指紋就完全不同——這就是「一改鋼印就對不上」。

4看鋼印：驗一張 JWT（Jwt）

資源伺服器收到識別證，要回答「這是真的、沒被改、沒過期嗎？」——而且完全不問 Google，當場看鋼印就好：

function 驗(token) {
    [h, p, 簽章] = token 用「.」拆三段
    if (不是三段)  return 失敗「格式錯」;

    期望簽章 = base64url( hash_hmac("$h.$p", 密鑰) )   // ← 自己照樣蓋一次鋼印
    if (簽章 != 期望簽章)  return 失敗「被竄改」;        // 對不上＝假票/被改

    payload = 解開(p)
    if (現在 >= payload.過期時間)  return 失敗「已過期」;  // 看有效期
    return payload               // 兩關都過 → 放行，全程零查詢
}

驗章的精髓：拿一樣的內容＋一樣的密鑰，自己再蓋一次鋼印，看算出來的跟票上的一不一樣。一樣＝沒被動過；不一樣＝有人改了內容（鋼印對不上）。真實程式比簽章時會用 hash_equals 做「常數時間比較」，避免壞人靠「比對要花多久」一點一點猜出鋼印（時序攻擊）。demo 裡「偷改識別證內容 → 驗章失敗」就是這一句 if 擋的。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
② 一次性通行證，用過作廢AuthServerexchangeCode() 的 used 檢查
③ 簽發識別證＋續命券AuthServerissueTokens()
④ 過期刷新＋撤銷（黑名單）AuthServerrefresh() / revoke()

👉 重點：真實系統會把「JSON 檔當資料庫」換成「分片資料庫＋Redis」，把鋼印從「大家共用一把密鑰（HS256）」換成「私章蓋、公開工具驗（RS256＋JWKS）」， 但上面這幾段邏輯一字都不用改：授權碼用過即廢、token 簽發/刷新/撤銷那三步、鋼印蓋一次／再蓋一次比對——完全一樣。所以讀懂這個小 demo，你就讀懂了這題的核心。

15. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「代客泊車」解釋，為什麼登入別的網站不該把 Google 密碼給它？ 密碼是能開全部東西的「主鑰匙」。代客泊車（App）只需要做一件小事，給它主鑰匙＝把信箱、改密碼的權限全交出去，太危險。OAuth 改發一把「權限有限、有期限」的泊車鑰匙（識別證），App 碰不到主鑰匙。
Q2：為什麼要先給「一次性通行證」，而不是直接給識別證？ 因為這張票會短暫出現在瀏覽器網址列（看得到、會被記錄）。讓它「只能用一次、幾十秒就過期」，就算被偷瞄也沒用；真正值錢的識別證留到後端對後端時才換，不經過瀏覽器。
Q3：JWT 為什麼像「蓋了鋼印的護照」？這帶來什麼好處？ JWT 上面寫著你是誰、能做什麼、何時過期，最後蓋上只有官方做得出的防偽簽章（鋼印）。資源伺服器看鋼印就知真偽，不必打電話回 Google 問——驗證零查詢、超快、能扛超大流量。
Q4：識別證為什麼要「短命」？過期了使用者要重登嗎？ 短命是為了「被偷也只能用一下」。過期不用重登：App 拿「續命券(refresh token)」去換一張新的，使用者無感，像飯店房卡到時間刷不開、櫃台續住一下就好。
Q5（工程）：驗證 QPS 比發 token QPS 大上百倍，這件事為什麼決定了要用 JWT？ 驗證量爆大，如果每次都回問 Google（introspection）會把中央伺服器打爆。JWT 自帶鋼印，資源伺服器當場驗、零查詢、沒有共用瓶頸，要扛更多就多加機器（水平擴展）。所以大量讀必選 JWT。
Q6（工程）：access token 不存進資料庫，refresh token 卻要存，為什麼？ access token 是自帶鋼印的 JWT，能自己證明自己，系統不必存一份對照（要存反而拖累驗證速度）。refresh token 是「續命券」，撤銷時要能把它列入黑名單、之後換不到新票，所以必須持久化記下來。
Q7（工程）：JWT「難以即時撤銷」是怎麼來的？實務上怎麼補救？ 因為資源伺服器只看鋼印＋有效期、不問 Google，發出去的票在自然過期前喊不停。補救：把 access 設得很短命（撤銷最多等十幾分鐘自動失效）＋撤銷時把 refresh 列黑名單（之後換不到新 access，整條鏈枯萎）；需要更即時就加一個小型黑名單快取折衷。

16. 30 秒回顧

OAuth／SSO ＝「不交密碼、只發臨時通行證」的代客授權。

4 個關卡：① 你親自同意（密碼只給 Google）→ ② 拿一次性通行證（授權碼）→ ③ 後端換成有期限識別證（access token）→ ④ 過期用續命券（refresh token）換新。

JWT ＝ 蓋鋼印的識別證：看鋼印＋有效期就放行，不必回問 Google → 驗證零查詢、超能擴展。

3 個怕：怕票被偷（每張票短命又限縮）、怕收不回（短命＋續命券黑名單）、怕給錯權限／被冒充（最小權限 scope ＋對暗號 state）。

工程細節一句話：驗票比發票多上百倍 → 用自帶鋼印的 JWT 當場驗 → access 不入庫、續命券要存還要能黑名單 → 找出塞車環節各自拓寬。
