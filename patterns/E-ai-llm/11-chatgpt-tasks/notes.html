⑪ ChatGPT Tasks：排程式 / 非同步 AI 任務系統

E AI/LLM 建議 48 分鐘 Design a Scheduled / Async AI Task System

一句話定位 本質是「排程器（cron-like）+ 延遲佇列 + 非同步 worker 跑 LLM」的工作流系統。 使用者排定週期或一次性任務（如「每天早上摘要新聞」），系統在到期時把任務丟進佇列，由 worker 非同步呼叫 LLM 執行。 面試真正考的是 排程準確性、冪等去重、失敗重試與規模化，而不是任務的 CRUD。

1. 需求釐清

功能性需求

  • 使用者建立任務：一次性（「明早 8 點幫我摘要這份報告」）或週期性（「每天早上 7 點摘要科技新聞」）。
  • 任務內容是一段給 LLM 的提示（prompt），到期時非同步執行並產生結果。
  • 結果投遞：完成後以通知（push / email / app 內收件匣）告知使用者。
  • 任務管理：暫停 / 恢復 / 刪除 / 查看歷史執行與結果。
  • 支援使用者時區（「每天早上 7 點」是使用者當地時間）。

非功能性需求

  • 排程準時：到期任務應在可接受誤差內被觸發（如 < 1 分鐘）。
  • 不重複執行：同一觸發只跑一次（LLM 呼叫昂貴，重複跑＝重複計費 + 重複通知）。
  • 可靠：worker / LLM 失敗要能重試；最終失敗要通知或進死信。
  • 可擴展：任務數與 worker 數可獨立水平擴展；LLM 慢任務不阻塞排程。
  • 讀寫特性：建立任務少、執行多（週期任務一次建立、長期反覆觸發）。
面試開場必問 「任務是週期還是一次性？要不要支援 cron 表達式與使用者時區？」這決定排程器的複雜度。 再問「同一任務若同時被觸發兩次，可以接受跑兩次嗎？」→ 答案幾乎都是不行，因此冪等是本題核心。

2. 規模估算

指標假設推算
活躍任務1000 萬使用者 × 每人 2 個週期任務~2000 萬個排程任務
每日觸發量多數每天 1 次~2000 萬次/日 ≈ 230 觸發 QPS（峰值集中早上 ×10 ≈ 2.3k）
排程掃描每分鐘掃一次到期掃描需走索引（next_run_at），不可全表掃描
LLM 執行每次 ~3 秒、~2k tokens慢任務 → 必須非同步 + worker 池；成本＝觸發量 × token 計費
結果儲存每筆 ~4KB，保留 90 天2000萬×90×4KB ≈ ~7 TB
關鍵洞察觸發量有強烈時間尖峰（大家都設「早上 7 點」）→ 排程需抹平尖峰（抖動 jitter / 分桶），worker 需能彈性擴縮。

3. API 設計

# 建立任務（寫，低頻）
POST /api/v1/tasks
Body: {
  "name": "每日科技新聞摘要",
  "schedule": { "type": "cron", "expr": "0 7 * * *", "tz": "Asia/Taipei" },
  "prompt": "摘要今天最重要的 3 則科技新聞",
  "notify": { "channel": "push" }
}
→ 201 { "task_id": "task_123", "next_run_at": "2026-06-01T07:00:00+08:00" }

# 一次性任務
POST /api/v1/tasks  Body: { "schedule": { "type": "once", "run_at": "..." }, ... }

# 管理
PATCH  /api/v1/tasks/{id}   Body: { "status": "paused" }   # 暫停 / 恢復
DELETE /api/v1/tasks/{id}

# 查執行歷史與結果（讀）
GET /api/v1/tasks/{id}/runs        → [{ "run_id", "status", "result", "attempt" }]
GET /api/v1/runs/{run_id}          → { "status": "DONE", "result": "..." }

4. 資料模型

tasks（可變的任務定義 + 排程狀態；next_run_at 建索引供掃描）
  task_id       PK
  owner_id      BIGINT
  type          ENUM(once, interval, cron)
  schedule_expr TEXT            -- cron 字串 / 間隔秒數
  tz            VARCHAR         -- 使用者時區（排程準點關鍵）
  prompt        TEXT
  status        ENUM(active, paused, deleted)
  next_run_at   TIMESTAMP  ◀── 建索引！scheduler 靠它掃出到期任務
  last_fired_at TIMESTAMP

task_runs（每次觸發一筆，狀態機，高頻寫）
  run_id          PK
  task_id         FK
  idempotency_key VARCHAR UNIQUE  ◀── = task_id + fire_at，唯一索引擋重複入列
  fire_at         TIMESTAMP       -- 本次觸發對應的排定時間
  status          ENUM(SCHEDULED, QUEUED, RUNNING, DONE, FAILED, RETRY)
  attempt         INT
  result          TEXT
  error           TEXT
  finished_at     TIMESTAMP
為何 task 與 run 分開task 是少量可變定義；run 是高頻 append 的執行紀錄。分開後 run 可走獨立表 / 時序庫，且冪等鍵建在 run 上，靠 DB 唯一約束從根本擋住重複觸發。

5. 高階架構

🖼️ 開啟 Excalidraw 架構圖（diagram.excalidraw） 
 建立 ─▶ Task API ─▶ 任務 DB(next_run_at 索引)
                            ▲  │
                            │  ▼
                      ┌──────────────┐
                      │  排程器(tick) │ 每分鐘掃 next_run_at ≤ now
                      └──────┬───────┘
                             ▼  (產生冪等鍵 task_id@fire_at，入列)
                       任務佇列(延遲/去重) ◀── 週期任務排定下次時間
                             ▼
        Worker(狀態機/重試) ─▶ LLM 推論 ─▶ 狀態/結果 DB ─▶ 通知投遞 ─▶ 使用者(收結果)
  1. Task API：建立 / 管理任務，寫入任務 DB 並算出首個 next_run_at。
  2. 排程器：週期性 tick，掃出到期任務 → 產生冪等鍵 → 入佇列；週期任務立刻排定下次時間。
  3. 佇列 + Worker：worker 競爭性消費、推進狀態機、呼叫 LLM、失敗重試、投遞通知。LLM 慢不阻塞排程（解耦）。

6. 核心元件深入

排程器：怎麼知道「到期了」？

  • 方案 A：DB 輪詢　cron worker 每分鐘 SELECT … WHERE next_run_at <= now() AND status='active'（next_run_at 建索引）。簡單、夠用到中等規模，面試可先講這個。
  • 方案 B：時間輪 / 延遲佇列　把到期時間當 key 放 Redis ZSET（score = timestamp），或用佇列原生延遲投遞（SQS DelaySeconds、RabbitMQ TTL+DLX）。規模大、精度高。
  • 週期任務　觸發後立刻依間隔 / cron 算出下一個 next_run_at 寫回（catch-up：若落後多次，對齊到未來的下一個點，避免瘋狂補跑）。
  • 時區　「每天早上 7 點」是使用者當地時間；DST（日光節約）切換時，cron 換算要用使用者時區規則，不能只存 UTC offset。

冪等與去重：避免同一觸發跑兩次

  • 每次觸發產生冪等鍵 = task_id + fire_at（該次排定時間）。入列前先檢查 / 靠 DB 唯一索引擋住。
  • 為何會重複？掃描器多副本同時掃到同一任務、訊息佇列「至少一次」語意重投、worker 逾時被判死又復活。冪等鍵讓這些情況都安全。

狀態機與失敗重試

SCHEDULED ──tick 到期──▶ QUEUED ──worker 取出──▶ RUNNING
   RUNNING ──成功──▶ DONE                 ──失敗&還可重試──▶ RETRY ──退避──▶ QUEUED
                                          ──失敗&重試用盡──▶ FAILED（死信 / 通知）

重試用指數退避 + 抖動，並區分可重試（LLM 逾時、429）與不可重試（prompt 違規）錯誤。通知投遞本身也要至少一次 + 冪等（同一 run 不重複通知）。

7. 擴展與瓶頸

階段瓶頸對策
單機排程單一 cron 掃全表、單點故障任務分片（依 task_id 雜湊），每分片獨立掃描；排程器多副本 + 租約（lease）防重複掃
早高峰尖峰大量任務同設 7:00 同時觸發加抖動（jitter）分散觸發；分桶；worker 自動擴縮
LLM 慢 / 限流worker 被慢任務佔滿、撞 LLM rate limitworker 池 + 佇列緩衝；per-tenant 配額；批次 / 快取相同提示
重複執行佇列至少一次、掃描器競爭冪等鍵 + DB 唯一索引；worker 租約 / 可見性逾時
失敗堆積下游故障導致大量重試指數退避 + 死信佇列（DLQ）+ 斷路器

8. 取捨與面試重點

  • DB 輪詢 vs 延遲佇列：輪詢簡單但有掃描延遲與全表壓力；延遲佇列精度高但多一層基建。面試先講輪詢，再說「規模大時換 Redis ZSET / 佇列原生延遲」。
  • at-least-once vs exactly-once：分散式下難真正 exactly-once；務實做法是at-least-once 投遞 + 冪等消費＝效果上的 exactly-once。這是本題最重要的一句話。
  • 排程與執行解耦：排程器只負責「何時觸發」，執行交給佇列 + worker。LLM 再慢也不拖累排程準點。
  • 時區 / DST：存使用者時區（IANA 名稱如 Asia/Taipei），每次計算 next_run_at 時換算，不可只存固定 offset。
  • 常見追問：任務暫停 / 刪除如何即時生效？→ 掃描時檢查 status，或入列後執行前再驗一次。如何防使用者建立海量任務？→ per-user 配額 + rate limit。長任務超過一次掃描週期怎麼辦？→ 租約續期 + 可見性逾時。

9. PHP Demo 說明

本題 demo 著重於排程 → 延遲佇列 → 狀態機 + 冪等 / 重試這條主線（也是面試真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/Scheduler.php　排程器：管理 next_run_at（一次性 + 週期 interval），tick() 以「邏輯時間」掃出到期任務入列，週期任務自動排定下次（含 catch-up）。
  • src/TaskQueue.php　延遲佇列 + 狀態機（QUEUED→RUNNING→DONE/RETRY/FAILED），冪等鍵 task_id@fire_at 防同一觸發重複入列。
  • src/Worker.php　佇列消費者：取出 run、呼叫假的 LLM 產生結果、推進狀態機、失敗重試、投遞通知。
  • src/Store.php　三個 JSON 檔模擬 DB（tasks / runs / notifications）+ 檔案鎖發號器。
  • public/index.php　路由 + 深色測試頁（建立任務 / tick / run / 檢視狀態）。
# 啟動
cd demo
php -S localhost:8011 -t public
# 瀏覽器開 http://localhost:8011
# 流程：建立週期+一次性任務 → ⏩tick(模擬時間前進，掃出到期入列) → ▶️run worker → 狀態轉 DONE、產生結果與通知
誠實聲明 Demo 的 LLM 呼叫為假呼叫（依 prompt 產生確定性文字，可勾選模擬失敗以示範重試）；而 排程的下次時間計算、延遲佇列、任務狀態機、冪等去重（防同一觸發跑兩次）、失敗重試等系統邏輯為真實可執行。 要接真實 LLM，只需把 Worker::fakeLlmCall() 換成實際 API 呼叫，其餘排程 / 佇列 / 狀態機完全不變。
