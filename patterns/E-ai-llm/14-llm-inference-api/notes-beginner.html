1. 60 秒搞懂它在幹嘛

想像你開了一家店，店裡只有一張超級貴的桌子（一張 GPU，租一小時就燒掉好幾百塊）。 客人（請求）一個個上門，每個人都想坐這張桌子，叫 AI 幫他「寫一段文字」。

LLM 推論 API 就是這家店的帶位系統：怎麼安排客人坐這張貴桌子，讓它一秒都不要空著、又不會讓客人等太久。 ChatGPT、各種 AI 對話服務的後台，本質都在做這件事。

🔑 一句話 LLM 推論 API = 把超貴的 GPU 當共享桌子來排隊、併桌、上菜：核心目標是「讓 GPU 一刻都不空轉」，而不是讓某一個客人最快吃完。

2. 為什麼這題的重點是「GPU 別空轉」？

很多人第一直覺是：「不就是呼叫一次 AI 模型、把答案傳回來嗎？有什麼難？」難就難在那張桌子實在太貴了。

💸 比喻：計程車的「空車跑」 計程車最怕的不是塞車，而是空車到處跑——沒載人還在燒油。GPU 就像一台貴到爆的計程車： 只要它在「開機卻沒在算東西」，你就在白白燒錢。

所以這題的一切設計，都是為了同一件事：讓這台貴車隨時都載滿客人。衡量做得好不好的標準，不是「單一客人多快到家」， 而是「每小時總共載了多少人」——工程上叫吞吐量（throughput）。

還有一個關鍵：AI 生成文字是一個字一個字慢慢吐的（不是一次想好整段）。它每吐一個字，都要回頭看一遍前面所有的字，再決定下一個字。 這個「一個字一個字、邊吐邊往前看」的特性，正是後面所有設計的根源。

💭 停一下：如果 GPU 這麼貴，為什麼不乾脆「一個客人一張桌子」，誰來就現開一張，省得排隊？

看答案 因為一張 GPU 一次其實能同時招呼好幾個客人（像一張大桌能併桌坐多人）。如果一人一桌，那張桌子大半時間都坐不滿、在空轉，等於把貴桌子的錢丟水裡。重點是把這張桌子塞滿人，而不是讓每個人獨佔。

3. 把系統拆成 4 個關卡

別被「推論服務」嚇到。一個請求從進門到吃飽離開，其實就闖 4 關：

1進門：門口先驗票、記帳（API Gateway）

客人一進門，門口的服務生先做三件事：看你有沒有票（API key 鑑權）、看你今天是不是點太多（限流）， 還有幫你記帳（用了多少字、之後好算錢）。

🎫 比喻就像遊樂園入口：先驗門票、看你買的是不是 VIP 快速通關、順手記下你進場——確認沒問題才放你進去排隊。

👉 這一關刻意做得很輕、很快，因為它不負責「算 AI」，只負責守門。守門的可以多請幾個（沒狀態，隨便複製）。

2排隊：照優先級排好隊（請求佇列）

進場後不是馬上有桌子，要排隊。但排隊有講究：付錢多的 VIP 排前面，「不急的批次任務」（例如半夜跑報表）排最後面墊底。

🏥 比喻：醫院的檢傷分類 急診室不是「先到先看」，而是照嚴重程度排：快不行的先進去，小擦傷的等一下。 系統也一樣，給每個請求一個優先級，避免一個慢吞吞的大訂單，把後面一堆快可以好的小訂單卡死。

3併桌：把客人湊在一起算（這關是全題核心）

輪到上桌了。重點來了：那張貴桌子一次可以坐好幾個人（GPU 一次能同時幫好幾個請求算）。 問題是——怎麼安排併桌，桌子才不會有空位？這就是本題的靈魂：動態批次（continuous batching）。

🚌 比喻：拼車 / 公車（最重要的一個比喻！） 笨方法是「包車」：湊滿 4 個人才開車，而且要等車上最慢的那個人到站，全車才能下、才能載下一批。 結果：早到站的人乾等，座位空著也不讓新乘客上——車子半空地跑，浪費。

聰明方法是「公車」：邊開邊上下客。哪個乘客到站了，立刻下車、空出座位，馬上讓站牌等候的新乘客補上來，不必等全車的人。 於是座位幾乎時時刻刻是滿的，同一台車一趟就能服務多很多人。

這就是「動態批次」：AI 每吐一個字（一個 step），系統就檢查一輪——誰生完了就讓出座位，立刻從排隊區拉一個新請求補進來，不用等整批一起結束。 效果驚人：同一張 GPU，吞吐量常常變成笨方法的 2～4 倍。

4上菜：一個字一個字端出去（串流回吐）

AI 算好的字，不會等整段寫完才一次給你，而是寫好一個字就馬上推給你一個字。這就是你在 ChatGPT 看到字「一個一個蹦出來」的原因。

🍣 比喻：迴轉壽司 vs 套餐全上 與其讓你枯等 10 分鐘才端出一整桌（套餐全上），不如做好一盤先送一盤（迴轉壽司）。 你很快就「有東西吃」，體感上等待短很多——這個「第一個字多快出現」叫 首字延遲（TTFT）。
✅ 四關一句話 ① 門口驗票記帳 → ② 照優先級排隊 → ③ 像公車一樣邊開邊上下客（動態批次）把桌子塞滿 → ④ 一個字一個字串流端出去。就這樣。

4. 設計時最怕的 4 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕四件事，每個設計都是為了避開某個「怕」。

😱 怕「桌子空著」 GPU 只要有一個 step 在空轉，就是在燒錢。所以動態批次存在的唯一理由就是：座位一空出來，立刻補人。 寧可排程邏輯寫得複雜，也不要讓貴桌子閒著。
😱 怕「記憶體爆掉」 AI 為了「一個字一個字往前看」，會把前面算過的東西記在 GPU 的記憶體裡不重算（這份「筆記」叫 KV cache）。 但記憶體有限，客人越多、聊得越長，這份筆記就越肥——記憶體常常先被塞爆，導致桌子併不了那麼多人。 這通常才是真正的瓶頸，不是算得不夠快。
😱 怕「大訂單餓死小訂單」 如果單純「誰先到誰先做、做完才換」，一個要寫 2000 字的超長請求，會把後面一堆「只要寫 10 個字」的小請求卡到天荒地老。 所以要有優先級排隊，還要限制單一請求最多寫幾個字，讓大家都有機會。
😱 怕「一個人吃垮全店」 如果某個客戶（租戶）的程式抓狂、瘋狂送請求，可能把整張桌子佔滿，害其他所有客戶都用不了。 所以要每個客戶各自限流、各有配額，把彼此隔開，一個出事不會拖垮全部。

  • 💎 GPU Worker（那張貴桌子）幫桌上每個人各算下一個字，旁邊放著大家的「筆記本」（KV cache）→
  • 📤 算好的字一個一個串流回吐給你（你看到字一個個蹦出來）。

右下角還有一句重點：同一組（排程器 + GPU）可以複製好幾組，忙的時候多開、閒的時候縮回——省錢。

7. 要準備多大？（規模估算）

🍱 比喻：開店前先估客流 開店前，你會先估「一天大概來幾個客人、每人坐多久」，才決定要擺幾張桌子、請幾個人。 系統設計也一樣：先估數字，才知道要租幾張貴 GPU。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
一天多少次生成每日 1000 萬次 ≈ 平均每秒 115 次，尖峰約 350 次要能撐住尖峰，不是平均
每次寫多長平均一次寫 300 個字（token）每秒總共要產出 ~3.5 萬個字
一張桌子多能產用動態批次，一張 GPU 約每秒 2500 字3.5 萬 ÷ 2500 ≈ 要 14 張 GPU（留點餘裕抓 ~20 張）
記憶體（KV cache）每個字、每一層都要記一點聊得越長、桌上人越多 → 記憶體先爆，限制了能併幾人
📌 關鍵洞察：瓶頸在「記憶體」和「桌子有沒有坐滿」 這題的瓶頸不是網路、不是 CPU，而是兩件事：GPU 的記憶體被 KV cache 吃光（併不了更多人）， 以及桌子有沒有一直坐滿（有沒有在空轉）。整個設計都繞著「別讓 GPU 任何一秒空著」打轉。

💭 停一下：流量有尖峰也有離峰，為什麼不能「永遠只租 20 張 GPU 開著」？

看答案 因為離峰時段（例如半夜）幾乎沒人用，20 張桌子大半空著還在燒錢。所以要自動擴縮：忙的時候多開幾組、閒的時候縮回最小數量。 但又不能縮到 0——因為模型權重載入要幾十秒（開店要時間），所以保留最少幾組常駐待命，避免客人一來卻要等開店。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住三張「點餐單」：

① 一次寫完（普通版）
   POST /v1/completions
   給我：{ 要用哪個模型, 你的提示文字, 最多寫幾個字, 創意程度 }
   我回：整段寫好的文字 + 用了多少字（好算錢）

「最多寫幾個字（max_tokens）」很關鍵：它順便防止某個人寫到天荒地老把桌子佔死，是限制大訂單的小韁繩。

② 一個字一個字回吐（串流版）← 線上服務主力
   POST /v1/completions  (stream=true)
   我回：寫好 "Hello" → 馬上推給你
         寫好 " world" → 馬上推給你
         ... → 最後推一個 [DONE] 表示寫完了

為什麼線上服務都用這個？因為生成是一個字一個字的，等整段寫完才回你要乾等好幾秒；邊寫邊推，第一個字幾百毫秒就出現，體感天差地別（迴轉壽司）。

③ 查用量 / 算錢
   GET /v1/usage?tenant=某客戶&from=...
   我回：{ 讀進去多少字, 寫出來多少字, 該收多少錢 }

計費按「讀進去的字 + 寫出來的字」算。而且記帳不擋在主要流程上——先把字產出去最重要，帳後面慢慢加總就好（這叫「與關鍵路徑解耦」）。

9. 系統要記住哪些事？（資料模型）

📒 比喻：兩本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題其實只要兩本：一本記「每個客人現在進行到哪」，一本記「每個客戶用了多少（算錢）」。

🧾請求進度簿（每個客人現在到哪一步）

每個請求一列，記它走到哪個階段：排隊中 → 讀題中(prefill) → 寫答案中(decode) → 寫完， 還有是哪個客戶、優先級多高、要寫幾個字、已經寫了幾個、進場是第幾步、完成是第幾步。

為什麼要記「進場第幾步、完成第幾步」？ 這樣才能算出每個人等了多久、花了多久，公車調度員（排程器）才知道誰該優先、哪裡塞車。

💰用量簿（每個客戶用了多少，算錢）

每個客戶一列，累加：讀進去多少字、寫出來多少字、總共幾次請求。

為什麼要分開記？ 因為計費跟「產出文字」是兩件事，混在一起會拖慢主要流程。真實系統會把用量當成一連串事件丟到旁邊慢慢加總，不擋住客人吃飯。
補充：那個吃記憶體的「KV cache」算資料嗎？ 算，但它活在 GPU 的記憶體裡、隨請求結束就丟掉，不是存進資料庫的長期資料。你只要記得「它很肥、會限制桌子能併幾人」就夠了。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
一張桌子有空位在空轉動態批次：座位一空就立刻補人（公車法）
記憶體被 KV cache 吃光，併不了更多人把筆記本分頁整齊管理（PagedAttention）＋壓縮（量化）省記憶體
一張桌子再怎麼塞也不夠用了多開好幾組（排程器+GPU）並排服務（水平擴展）
尖峰排隊、離峰桌子空著燒錢自動擴縮：依排隊長度、桌子忙不忙，自動加減組數
大訂單拖慢小訂單優先級排隊＋限制單筆最多寫幾字；必要時把「讀題」和「寫答案」分開部署
單一客戶暴衝佔滿全店給每個客戶各自的水龍頭（限流）＋配額，彼此隔開

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的四個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 吞吐 vs 延遲（最核心） 一次併越多人，桌子越滿、總產量越高（吞吐高）；但每個人要等的隊伍也越長，第一個字更慢出現（延遲差）。 實務做法：在「最慢也不能超過某個秒數」的約束下，盡量把桌子塞滿。
⚖️ 包車 vs 公車（靜態 vs 動態批次） 包車（靜態批次）寫起來簡單，但桌子常空轉；公車（動態批次）寫起來複雜，但吞吐翻倍——線上服務幾乎都選公車。
⚖️ 想多併人 vs 記憶體不夠 想一次招呼更多客人，卻被 KV cache 把記憶體吃光 → 用壓縮（量化）和分頁管理把記憶體省出來，換更多座位。
💭 追問：「讀你的問題」和「寫答案」為什麼有人要分開做？ 因為這兩件事個性完全不同：讀題（prefill）是一口氣讀完整段，很吃「算力」；寫答案（decode）是一個字一個字慢慢來，很吃「記憶體頻寬」。 混在同一張桌子會互相扯後腿，分開部署可以各自調到最省。
💭 追問：服務剛開、第一個客人來時為什麼特別慢？ 因為要先把幾十 GB 的「模型權重」載進 GPU（開店、進貨要時間，可能幾十秒）。對策是預熱＋保留最少幾組常駐，不要讓客人撞上「正在開店」。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「公車調度到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 Request（每位客人的點餐卡：要寫幾個字、現在在哪一檔）→ BatchScheduler（公車調度員＝本題的靈魂，管座位填補）→ InferenceModel（假裝是 GPU 的廚房，每呼叫一次吐一個字）。一個請求進門，就照這個順序被服務。

1排隊：照優先級插進隊伍（BatchScheduler.submit）

客人一進門，先丟進排隊區，並立刻照優先級排好——VIP（付費）排前面，半夜跑報表的 batch 任務墊底。

function 提交請求(客戶, 優先級, 提示文字, 最多寫幾字) {
    要寫幾字 = 模型.算出這題大概要寫幾字(提示文字, 最多寫幾字)
    新請求 = 建一張點餐卡(客戶, 優先級, 提示文字, 要寫幾字)
    排隊區.加進去(新請求)
    排隊區.重新排序()        // ← 小數字=高優先；同級先到先服務
}

排序的規則是「先比優先級，再比誰先到」。這就是 Part 1 講的「醫院檢傷分類」——不是先到先看，是照輕重排。排好之後，調度員每次都從隊伍最前面拉人補位。

2填補空位：座位一空就立刻補人（BatchScheduler.step 第一段）

這是全題的心臟。桌子有固定幾個座位（最大批次，受 GPU 記憶體 / KV cache 限制）。每推進一步，第一件事就是巡一圈座位，只要有空位、排隊區又有人，馬上把新客人拉上桌——不必等整桌的人吃完。

function 推進一步() {
    // ── 公車核心：邊開邊上客 ──
    for (每個座位 in 桌子) {
        if (座位是空的 且 排隊區還有人) {
            新客 = 排隊區.拉出最前面那位()
            新客.狀態 = 讀題中(prefill)     // 第一步先讀完他的問題
            座位 = 新客                     // ← 空位「當下」就被補上
        }
    }
    ... (下一段：幫桌上每個人寫一個字)
}

注意：填補發生在「幫大家寫字」之前。這一念之差就是公車（動態批次）和包車（靜態批次）的分水嶺——不等全車到站，哪個站牌有人、哪個座位空了，這一步就接上。

3寫一個字 + 寫完就讓座（BatchScheduler.step 第二段）

補完位，接著幫桌上每一位各寫一個字（一個 step ＝ 全桌各前進一個 token）。誰寫到目標字數了，當場宣布完成、立刻清空他的座位——讓下一步開頭的「填補」能馬上把新人放進來。

for (每個座位 in 桌子) {
    if (座位是空的) 跳過;
    這位客.狀態 = 寫答案中(decode)
    一個字 = 模型.寫下一個字(...)      // ← 真實系統這行＝GPU 算一次
    這位客.已寫字數 += 1

    if (這位客.已寫字數 >= 要寫幾字) {  // 寫夠了
        這位客.狀態 = 完成
        座位 = 空                       // ← 當場讓座，不必等整桌
    }
}
忙碌座位數 = 這一步真正有寫字的座位數
累計利用率(忙碌座位數 / 總座位數)        // ← 用來算「桌子坐多滿」

「當場讓座」這一行，就是公車「乘客到站立刻下車空出座位」。它和上一關的「立刻補人」一搭一唱，座位才會幾乎時時刻刻是滿的。每一步順手記下「幾個座位在忙」，就能算出整趟的平均利用率。

💭 停一下：為什麼「填補空位」要寫在「幫大家寫字」前面，而不是後面？順序真的有差嗎？

看答案 有差，而且是關鍵。先填補，這一步空出來的座位立刻能讓新客上桌、馬上開始算，這一步桌子就坐滿了。若放後面，新客要再等一整步才動工，桌子白白空一輪——這正是包車（靜態批次）空轉的來源。把「填補」放最前面，就是把空轉壓到最小。

4公車 vs 包車：算給你看坐多滿（continuous vs static 利用率）

怎麼證明公車真的比較好？demo 用同一批請求各算一次平均利用率：公車法（邊開邊上下客）vs 包車法（湊一組、等全組最慢的人才換）。

// 公車法：每一步「在忙的座位」加總 ÷「總座位」
公車利用率 = 累計忙碌座位 / 累計總座位

// 包車法：每 N 人切一組，整組要陪「組裡最長的人」跑完才放人
for (每一組 of 把所有人每N個切一組) {
    整組步數 = 這組裡最長那位要寫的字數   // ← 大家被最慢的人綁住
    總座位步 += 整組步數 × 每組座位數
    忙碌座位步 += 這組每個人各自的字數     // 短的人寫完就空轉、卻仍佔著組
}
包車利用率 = 忙碌座位步 / 總座位步         // ← 通常明顯低於公車

關鍵就在 整組步數 = 組裡最長那位：包車裡早寫完的人空出座位也不能讓新人上，得陪整組跑到最慢那位——空轉就是這樣來的。公車沒這限制，所以利用率（＝吞吐量）常常是包車的 2～4 倍。這就是你在 demo 畫面看到的那兩條對比數字。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「公車 vs 包車」解釋什麼是動態批次（continuous batching）？ 包車要湊滿才開、還得等全車最慢的人到站才能換下一批，座位常空著（靜態批次）。公車邊開邊上下客，誰到站就下、空位立刻補新乘客（動態批次），座位幾乎一直滿——同一張 GPU 吞吐常變 2～4 倍。
Q2：為什麼這題的目標是「GPU 別空轉」而不是「讓某個請求最快」？ 因為 GPU 超貴，空轉就是燒錢。一張 GPU 一次能同時招呼好幾個請求，所以重點是把它塞滿、最大化「每秒總產出（吞吐量）」，而不是讓單一客人獨佔、最快吃完。
Q3：KV cache 是什麼？為什麼它常常是瓶頸？ AI 一個字一個字寫，每寫一個字都要回頭看前面所有字；KV cache 就是把「前面算過的東西」記在 GPU 記憶體裡省得重算的筆記本。它隨聊天變長、人變多而變肥，常常先把記憶體塞爆，限制了桌子能併幾個人。
Q4：為什麼線上服務要「一個字一個字串流回吐」？ 生成是逐字進行的，等整段寫完才回要乾等好幾秒；邊寫邊推，第一個字幾百毫秒就出現（首字延遲 TTFT 短），體感像迴轉壽司——很快就有東西吃。
Q5（工程）：批次開越大越好嗎？要在什麼之間取捨？ 不是。批次越大、桌子越滿、吞吐越高，但隊伍越長、單請求要等越久、首字越慢。實務是「在 P99 延遲（最慢也不能超過某秒數）約束下，盡量把批次塞滿」。
Q6（工程）：怎麼避免「大訂單餓死小訂單」？至少講兩招。 ① 優先級排隊（VIP/短請求排前、batch 任務墊底）；② 限制單一請求最多寫幾個字（max_tokens）；③ 必要時把「讀題(prefill)」和「寫答案(decode)」分開部署，互不拖累。
Q7（工程）：離峰沒人用，為什麼不能把 GPU 全關掉省到底？ 因為模型權重載入要幾十秒（開店要時間）。全關掉的話，客人一來要乾等開店。對策是自動擴縮但保留最少幾組常駐＋預熱，平衡省錢與反應速度。

15. 30 秒回顧

LLM 推論 API = 把超貴的 GPU 當共享桌子，想辦法讓它一刻都不空轉。

4 個關卡：① 門口驗票記帳 → ② 照優先級排隊 → ③ 像公車一樣邊開邊上下客（動態批次）把桌子塞滿 → ④ 一個字一個字串流端出去。

4 個怕：怕桌子空著（動態批次補位）、怕記憶體爆（KV cache 是頭號瓶頸）、怕大訂單餓死小訂單（優先級＋限字數）、怕一個人吃垮全店（各自限流隔離）。

工程細節一句話：先估要租幾張 GPU（瓶頸在記憶體與利用率）→ 用「點餐單(API)」溝通、串流回吐讓首字快 → 兩本筆記本（請求進度／用量）→ 找出塞車環節各自拓寬，閒時自動縮、忙時自動加。
