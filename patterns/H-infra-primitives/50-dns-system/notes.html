㊿ DNS 系統：把名字翻成 IP，全球、要快、要就近

H 基礎設施 建議 40 分鐘 Design the DNS System

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把人類好記的網域名（www.example.com）翻成機器用的 IP，而且要全球、要快、要就近」的問題。 單機查一張對照表很簡單；面試真正考的是 階層式遞迴解析、多層快取與 TTL、以及 GeoDNS/anycast 如何讓全球使用者就近命中最近的伺服器。

1. 需求釐清

功能性需求

  • 輸入網域名（如 www.example.com）→ 回傳對應的 IP（A 記錄）。
  • 支援常見記錄型別：A（名字→IP）、CNAME（別名→另一個名字，要再跟一次）、NS（委派權威）。
  • 階層解析：透過 root → TLD → authoritative 三層逐級問出答案。
  • GeoDNS：同一個名字，依使用者所在區域回最近的 IP。
  • 網域擁有者可新增 / 修改記錄（並能控制變更多久生效 = TTL）。

非功能性需求

  • 低延遲：解析在每一次連線前的關鍵路徑上，p99 要 < 數十 ms；命中快取應 < 1ms。
  • 極高可用：DNS 掛了 = 全網「失明」，需 > 99.999%（anycast + 多副本）。
  • 讀遠多於寫：查詢（讀）量巨大、記錄變更（寫）罕見 → 整個設計圍繞「快取」。
  • 全球可擴展：使用者遍布全球，要就近服務 + 抗 DDoS。
  • 最終一致可接受：改記錄後靠 TTL 自然過期收斂，不追求即時全球一致。
面試開場必問 「設計的是 recursive resolver（幫使用者跑完整遞迴的那台，如 8.8.8.8）還是 authoritative（持有某網域權威資料的那台）？」與「TTL 大概多長（決定快取效率 vs 變更生效速度）？」這兩問決定整題走向。

2. 規模估算

指標假設推算
查詢量（讀）大型 public resolver~1 兆查詢/日 ≈ 千萬級 QPS（峰值更高）
快取命中率熱門網域高度集中resolver 快取命中可達 80~90%+ → 真正打到權威的只剩零頭
記錄變更（寫）網域記錄很少改讀寫比可達 百萬:1 → 寫不是瓶頸，讀才是
記錄資料量每網域數筆記錄 × 數億網域數十 GB 級，輕鬆放進記憶體
封包大小UDP/53 單包 ~512B（EDNS 更大）小封包、無連線 → 適合 anycast 廣布
關鍵洞察查詢量天文數字但高度可快取。設計重點不是「算得多快」，而是「讓絕大多數查詢在快取就回掉（多層快取 + TTL）」與「就近命中（anycast/GeoDNS）」。

3. API / 查詢介面

# DNS 沒有「業務 REST API」，它的介面是 DNS 協定本身（UDP/TCP port 53）：
# 一次查詢 = (QNAME, QTYPE, QCLASS)
QNAME = www.example.com
QTYPE = A          # 想要的記錄型別：A / AAAA / CNAME / NS / MX ...
→ 回應 = { answer: 93.184.216.34, type: A, TTL: 300 }   # 帶 TTL 告訴你能快取多久

# 跟隨 CNAME：
QNAME=blog.example.com QTYPE=A
→ blog.example.com  CNAME  www.example.com   # 別名，resolver 自己再查一次
→ www.example.com   A      203.0.113.10       # 直到拿到 A 才算解析完成

# 管理面（網域擁有者改記錄，本 demo 對應 POST /api/record）：
PUT  zone/example.com/records
Body: { "name":"www.example.com", "type":"A", "ttl":300, "value":"203.0.113.10" }
# GeoDNS：同名多筆，依 client 區域挑一筆回
為何回 TTLTTL 是 DNS 的靈魂：它告訴每一層快取「這個答案可以重用多久」。TTL 決定了「改記錄後多久全球生效」與「快取能擋掉多少查詢」之間的取捨（§8 詳述）。

4. 資料模型

# 權威區域檔（zone file）：某網域的所有記錄
zone example.com
  example.com.      A      300   93.184.216.34
  www.example.com.  A      60    203.0.113.10     # 短 TTL（常變/要 Geo）
  blog.example.com. CNAME  300   www.example.com. # 別名 → 再跟一次
  example.com.      NS      ...   ns1.example.com. # 委派：誰是這網域的權威

# GeoDNS：同一個名字，依區域不同筆
  www.example.com (region=tw) A 203.0.113.10   # 亞太節點
  www.example.com (region=us) A 198.51.100.20  # 北美節點
  www.example.com (region=eu) A 192.0.2.30     # 歐洲節點

# Recursive resolver 的快取（靠 TTL 自動回收）
  KEY  name|type|region  ->  { answer, expire_at }   # 過期即重查
為何分「權威」與「快取」兩份權威（authoritative）是真相來源，由網域擁有者維護；快取（recursive resolver 內）是就近的副本，靠 TTL 與真相收斂。讀路徑幾乎只碰快取，這就是 DNS 能扛千萬 QPS 的祕密。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 Client(stub) ──查 www.example.com──▶ Recursive Resolver ──先查快取(命中即回)
                                            │ (未命中 → 遞迴)
                                            ├──①──▶ Root (.)       回：去找 .com 權威
                                            ├──②──▶ TLD (.com)     回：example.com 的 NS
                                            └──③──▶ Authoritative  回：A / CNAME
                                                       │
                                          GeoDNS/anycast：依區域回最近的 IP
                                            │
                       結果依 TTL 寫回快取 ◀┘  下次同名查詢直接命中
  1. Stub resolver（作業系統內）：把查詢丟給設定好的 recursive resolver。
  2. Recursive resolver（如 8.8.8.8）：有快取；命中直接回，未命中才代替你跑完整遞迴。這是讀路徑的主力。
  3. Root → TLD → Authoritative：三層權威階層，逐級把你「導向下一層」，最後由權威給出 A/CNAME。
  4. GeoDNS / anycast：權威依使用者區域回最近的 IP；anycast 讓「同一個 IP」在全球多地廣播，封包自動走到最近節點。

6. 核心元件深入

遞迴 vs 迭代解析

  • 遞迴解析（recursive）：client 只問一句「給我 IP」，recursive resolver 幫你跑完所有層（root→TLD→權威）再把最終答案交給你。客戶端輕鬆。
  • 迭代解析（iterative）：每一層只回「下一步去問誰」，由發問方自己一層層追下去。recursive resolver 對上游權威用的就是迭代（root/TLD 只給 referral，不幫你跑完）。
  • 一句話：client→resolver 是遞迴，resolver→權威階層是迭代。

快取與 TTL（讀路徑的命脈）

  • 每筆記錄帶 TTL，resolver 把答案快取到 TTL 過期。命中時整段遞迴都省掉（demo 的解析鏈會從 5 步縮到 1 步）。
  • 快取分層：瀏覽器 → 作業系統 → recursive resolver → 權威，越靠近使用者命中越省。
  • TTL 過期即失效，下次查詢重新遞迴並更新——這就是「改記錄後靠 TTL 自然生效」的機制。

記錄型別：A 與 CNAME

  • A：名字直接對應一個 IPv4（AAAA 則對 IPv6）。解析的終點。
  • CNAME：名字是另一個名字的別名。拿到 CNAME 不算結束，要再跟一次去解析目標名，直到拿到 A。常用於把 www 指到 CDN 的網域。
  • NS：委派——告訴你「這個子網域/網域的權威伺服器是誰」，是階層階接的關鍵。

GeoDNS / anycast：就近

  • GeoDNS：權威依 client 來源（IP 或 EDNS Client Subnet）判斷區域，同一個名字回不同 IP，把使用者導向最近的資料中心 / CDN 邊緣節點。
  • anycast：把同一個 IP 在全球多個節點同時宣告，路由（BGP）自動把封包送到網路上最近的節點。root server 與大型 resolver 都靠它就近 + 抗 DDoS。
  • 兩者常並用：GeoDNS 在「答案內容」就近，anycast 在「封包路徑」就近。

負面快取（negative caching）

查無此名（NXDOMAIN）也要快取一小段時間（由 SOA 的 minimum TTL 決定），避免「不存在的名字」被反覆查詢一直打穿到權威——這也是抗（誤）放大的一環。

常見陷阱把 TTL 設超長以為「省快取」，結果改記錄後數小時/數天全球才生效，緊急切換（如故障轉移）做不到。對「會變/要做容錯切換」的記錄（如 www 指向 CDN）通常用短 TTL（30~60s）。

7. 擴展與瓶頸

階段瓶頸對策
單一權威查詢全打權威、單點故障多副本 + recursive resolver 多層快取擋掉 90%+ 查詢
熱門網域少數爆紅網域查詢集中高命中率快取 + anycast 把負載攤到全球節點
全球延遲跨洲問權威 RTT 高anycast（就近節點）+ GeoDNS（就近答案）+ 邊緣快取
DDoS / 放大攻擊偽造來源 IP 用 DNS 做放大反射打別人anycast 吸收 + 限速 + Response Rate Limiting + 關閉開放遞迴
記錄變更生效長 TTL 下改不動對需變更的記錄用短 TTL；切換前先預先調低 TTL

8. 取捨與面試重點

  • TTL 長 vs 短：長 TTL → 快取命中高、權威負載低，但改記錄要等很久才生效；短 TTL → 變更快生效、容錯切換靈活，但快取命中低、權威更忙。核心折衷，看記錄「多常變 / 要不要快速切換」決定。
  • anycast vs unicast：anycast 就近 + 抗 DDoS + 故障自動繞路，但路由收斂與「同一連線可能換節點」需處理（DNS 多為單包查詢，影響小，所以特別適合 anycast）。
  • 遞迴 vs 迭代：把遞迴集中在 resolver（客戶端簡單、可共用快取）；權威階層只做迭代 referral（權威不替別人跑遞迴，否則易被當放大器）。
  • GeoDNS 的快取難題：依區域回不同答案 → 不能全球共用一份快取，需靠 EDNS Client Subnet 把區域資訊帶進去，分區快取（本 demo 的快取鍵就帶了區域）。
  • 常見追問：
    • 解析慢在哪？→ 多半是未命中時的遞迴 RTT；靠快取 + anycast 就近壓低。
    • 改了 IP 為何還連到舊的？→ TTL 還沒過期，各層快取仍是舊答案。
    • CNAME 與 A 差在哪？→ CNAME 是別名要再跟一次，A 是終點。
    • DNS 放大攻擊？→ 小查詢觸發大回應 + 偽造來源 → 反射打受害者；用 RRL/限速/anycast 緩解。

9. PHP Demo 說明

本題 demo 聚焦 DNS 的核心解析邏輯（也是面試真正考點），可實際用 PHP 內建伺服器跑起來：

  • src/Resolver.php　核心：階層遞迴解析（root→TLD→authoritative，回傳完整解析鏈）、解析快取含 TTL（命中省去遞迴、過期重查）、A/CNAME（CNAME 再跟一次）、GeoDNS（依 client 區域回最近 IP，分區快取），用 flock 原子更新狀態。
  • public/index.php　路由：解析（看解析鏈 + 快取命中）/ 加記錄 / 區域切換 / 快取狀態，附深色測試頁。
# 啟動
cd demo
php -S localhost:8050 -t public
# 瀏覽器開 http://localhost:8050
# 解析 www.example.com（tw）→ 看完整遞迴鏈；立刻再解析一次 → 快取命中(省遞迴)
# 區域切 us/eu 再解析 → GeoDNS 回不同 IP；解析 blog.example.com → CNAME 再跟一次
誠實聲明 Demo 用記憶體 + JSON 檔模擬 root / TLD / authoritative 三層區域表，沒有真正的 DNS 封包與網路傳輸（單機示意）。 但階層遞迴解析（完整解析鏈）、解析快取與 TTL、A/CNAME 再跟一次、GeoDNS 依區域回不同 IP、改記錄即失效快取等 DNS 邏輯為真實可執行。要上線只需把記憶體查表換成真正向 root/TLD/權威發送 DNS 查詢，解析與快取邏輯不變。
