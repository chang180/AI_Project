1. 60 秒搞懂它在幹嘛

你在瀏覽器打 www.google.com 按下 Enter。但電腦其實不懂名字，它只認得一串數字（IP，像 142.250.x.x）。 那「名字 → 數字」這一步是誰做的？就是 DNS（網域名稱系統）——網路世界的電話簿。

聽起來不就查個電話簿？難的是：這本電話簿全世界共用、有幾億個名字、每秒被查幾千萬次， 而且要做到：查得超快、永遠不能掛、還要依你人在哪裡給你最近的那台伺服器。

🔑 一句話 DNS = 網路的電話簿：把好記的名字（www.example.com）翻成機器用的號碼（IP）。 而且要全球、要快、要就近——你在台灣查，給你台灣最近的那台；在美國查，給你美國的。

2. 為什麼這題的難點不是「存一張對照表」？

很多人第一次想：「不就一張表，名字配 IP，查一下就好？」——這想法漏掉了三個會把系統壓垮的真正難關。

📌 比喻：問路要問好幾關 你到一個陌生城市找「example 公司 3 樓的王先生」。你不會一開口就有人知道。 你先問市政府：「這家公司在哪一區？」→ 它說「在科技區，去問科技區管理處」→ 你再問科技區：「example 公司在哪棟？」→ 它說「B 棟，去問 B 棟櫃台」→ 最後問 B 棟櫃台：「王先生在幾樓？」→ 「3 樓 305」。一層問一層，逐級導向，這就是 DNS 的解析方式。

第一個難關就是這個階層問路：沒有一台機器知道全世界所有名字，而是root → TLD → 權威三層分工，一層把你導向下一層。

第二個難關更隱形：每秒幾千萬次查詢，若每次都從頭問三關，伺服器會被問爆。所以要把問過的答案記住一陣子（快取），下次同樣的名字秒回、不用再問。

第三個難關：全世界的人都在查同一個名字，要讓每個人都連到離他最近的那台伺服器（不然台灣人連美國機房會很慢）。

💭 停一下：為什麼不乾脆做「一台超大伺服器，存全世界所有名字」就好？

看答案 因為（1）單點：一台掛了全網失明，太危險；（2）太遠：全世界都連同一台，多數人都很慢；（3）誰維護：每個網域的記錄該由它自己的擁有者管，不該塞進同一台。 所以 DNS 設計成分層 + 分散 + 大量快取：root/TLD 只負責「導向」，真正的記錄在各網域自己的權威伺服器，再靠快取與就近讓它又快又穩。

3. 把解析拆成 3 個關卡（問路三段）

別被「系統」嚇到。一次解析其實就是「問路三段」，加上一個超重要的偷懶機制（快取）：

0先偷懶：問過了嗎？（快取）

真正開始問路之前，幫你跑腿的那台機器（叫 recursive resolver）會先翻自己的小抄：「這個名字我剛剛是不是查過？」 查過而且還沒過期，就直接把上次的答案給你——三段問路全省了。

📒 比喻：跑腿小弟的小抄 你常叫同一個跑腿小弟去問同一家公司的地址。聰明的小弟會把問過的記在小抄上，下次你再問，他直接念給你聽，不用又跑一趟。 但小抄上每條都有「保存期限」（叫 TTL），過期就要重新去問一次（怕地址搬了）。

1問 root：「.com 的管理員是誰？」

小抄沒有，才開始問路。第一關問最高層的 root。root 不知道 example 在哪，但它知道「.com 這類名字該去問誰」，把你導向 .com 的管理員。

2問 TLD（.com）：「example.com 的權威是誰？」

.com 管理員（叫 TLD 伺服器）也不知道確切 IP，但它知道「example.com 這個網域，由哪台伺服器負責」（這台叫權威），把你導過去。

3問權威：「www.example.com 的 IP 是？」

終於問到權威伺服器——它是這個網域的「真相來源」，由網域擁有者維護。它直接告訴你：「IP 是 203.0.113.10」。問路結束！

🔀 兩個小狀況 ① 別名（CNAME）：權威可能說「www 其實是 blog 的小名，你去查 blog 吧」——這時要再跟一次，查到真正的 IP 為止。 ② 看人給答案（GeoDNS）：權威會看你人在哪——台灣人給台灣機房、美國人給美國機房，同一個名字回不同 IP，讓你連最近的。
✅ 三關一句話 ⓪ 先翻小抄（快取），有就秒回 → 沒有才 ① 問 root（.com 找誰）→ ② 問 TLD（example.com 找誰）→ ③ 問權威（拿到 IP）。別名要再跟一次，還會依你在哪給最近的 IP。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「掛掉」 DNS 掛了 = 全世界連名字都翻不出 IP，等於整個網路對你「失明」，比某個網站掛掉嚴重一百倍。 所以 DNS 要多台副本 + anycast（同一個地址全球多地都有，自動連最近的），做到幾乎永遠不掛。可用性是命根。
😱 怕「慢 / 被問爆」 每秒幾千萬次查詢，若每次都從頭問三關，機器跑不動、使用者也等到不耐煩。 所以靠快取（記住問過的）+ 就近（連最近的伺服器），讓絕大多數查詢秒回、根本不用問路。能不問就不問。
😱 怕「改了卻不生效」 你把網站搬到新 IP，結果有人還連到舊的——因為大家的小抄（快取）還沒過期。 這由 TTL（保存期限）控制：設太長 → 改了很久才全球生效；設太短 → 大家一直重問、伺服器更累。長短是一門取捨。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每天多少次查詢大型服務 ~1 兆次/日（千萬級 QPS）天文數字 → 不可能每次都問路，得靠快取
快取能擋掉多少命中率可達 80~90%+大多數查詢「小抄就有」，真正去問路的只剩零頭
記錄多久改一次很少改（讀寫比可達百萬:1）讀（查）才是壓力，寫（改）幾乎沒負擔
一筆答案多大一個小封包（幾百位元組）又小又快 → 很適合全球廣布（anycast）
💡 關鍵洞察：壓力全在「讀」，而讀又超好快取 DNS 幾乎只有人在「查」，很少人在「改」。而且大家都查那些熱門名字（google、facebook…）， 所以只要把熱門答案快取住，就能用很小的力氣扛住天文數字的查詢量。這就是 DNS 能撐起整個網路的祕密。

💭 停一下：查詢量這麼大（千萬級 QPS），為什麼權威伺服器沒被打爆？

看答案 因為絕大多數查詢根本沒打到權威——它們在更靠近使用者的快取（瀏覽器、作業系統、recursive resolver）就被秒回了。 真正穿透所有快取、打到權威的只是一小撮（快取沒有 / TTL 剛過期的）。多層快取像一層層的盾牌，把洪水擋在外面。

8. 系統之間怎麼溝通？（查詢介面）

🍔 比喻：超精簡的問答卡 DNS 沒有花俏的網頁 API，它的「點餐單」超簡單：你遞一張問答卡寫「我要查哪個名字、要哪種記錄」，對方回一張「答案 + 保存期限(TTL)」。就這樣。 
① 查一個名字（最常用）
   問：{ 名字: www.example.com, 要哪種: A(就是要 IP) }
   回：{ 答案: 203.0.113.10, 型別: A, 保存期限TTL: 60 秒 }

回應一定帶 TTL（保存期限）——它告訴每一層快取「這答案能重用多久」。這是 DNS 最核心的一個欄位。

② 遇到別名（CNAME）要再跟一次
   問：blog.example.com 的 IP？
   回：blog 是 www.example.com 的小名(CNAME) → 你再去查 www
   再問：www.example.com 的 IP？ → 回：203.0.113.10  ← 跟到 A 才結束
③ 網域擁有者改記錄（管理面，本 demo 的 POST /api/record）
   給我：{ 名字, 型別(A/CNAME), 值(IP 或目標名字), TTL, 可選:各區域的 IP(GeoDNS) }
   → 系統存進「權威區域表」，並把相關快取清掉，讓新值能被讀到

注意第 ③ 張是「管理面」——只有網域擁有者用來改記錄，一般查詢的人碰不到。

9. 系統要記住哪些事？（資料模型）

📒 比喻：兩本筆記本 系統要把資料記在「筆記本」裡。這題主要用兩本：一本是真相（權威區域表），一本是小抄（快取）。

📕真相簿（權威區域表）

每個網域一份，記它所有的記錄：名字、型別(A/CNAME/NS)、值、TTL。 A 記錄還可以是「看區域給不同 IP」（GeoDNS）。這本由網域擁有者維護，是最終真相。

📒小抄簿（快取——讀路徑的命脈）

跑腿小弟把問過的答案記在這：名字 → { 答案, 何時過期 }。 命中而且沒過期 → 秒回，三關全省；過期 → 丟掉重問。

為什麼小抄要帶「過期時間」？ 因為地址會變（網站搬家）。如果小抄永不過期，網站搬了你還一直連舊地址。 帶上「保存期限(TTL)」，過期就重問一次，新地址才有機會被讀到。這就是「改記錄靠 TTL 自然生效」。

🌍GeoDNS 的小聰明

同一個名字 www.example.com，在真相簿裡可以對應好幾個 IP，標好「tw 用這個、us 用那個」。 解析時看你人在哪，挑最近的給你。小抄也要分區存（台灣人的答案不能拿給美國人用）。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
查詢全打權威，權威被問爆多層快取（瀏覽器→OS→resolver）把 90%+ 查詢擋在外面
全世界連同一台，跨洲很慢anycast（同一個 IP 全球多地）+ GeoDNS（給你最近的 IP）
一台掛了全網失明多副本 + anycast 自動繞到還活著的節點
壞人用 DNS 放大攻擊打別人限速 + 關閉開放遞迴 + anycast 吸收洪水
改了記錄很久才生效對「會變的記錄」用短 TTL；要搬家前先把 TTL 調短

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ TTL 設長 vs 設短 長 → 快取命中高、伺服器輕鬆，但改記錄要等很久才全球生效（搬家、故障切換很卡）； 短 → 改了很快生效、切換靈活，但大家一直重問、伺服器更忙。看這條記錄「多常變」來決定。
⚖️ 遞迴 vs 迭代 讓「跑腿小弟（resolver）」幫你跑完整段（遞迴），客戶端最輕鬆、又能共用小抄； 但上層的 root/TLD 只做指路（迭代），不幫你跑完——否則它們會被當成「免費跑腿」拖垮，也容易被壞人利用。
⚖️ GeoDNS 的小抄難題 依區域給不同答案，小抄就不能全球共用一份（不然台灣人拿到美國的 IP）。 解法是把「你在哪一區」也記進小抄的鑰匙裡，分區各存一份（demo 的快取鑰匙就帶了區域）。
💭 追問：我把網站換了新 IP，為什麼有人還連到舊的？ 因為他那邊的小抄（快取）還沒過期——TTL 設多久，舊答案就可能被重用多久。等 TTL 過期、重新問一次，才會拿到新 IP。 所以要搬家前，通常會先把 TTL 調短（例如從 1 天改成 1 分鐘），等舊的短 TTL 全球生效後再動手，這樣切換才快。
💭 追問：CNAME 和 A 到底差在哪？ A 是終點：名字直接給你 IP。CNAME 是小名／別名：它說「我其實是另一個名字」，你得再跟一次去查那個名字，直到拿到 A 為止。常用來把 www 指到 CDN 的網域。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「三關問路 + 快取 + GeoDNS」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8050 -t public
# 然後瀏覽器打開 http://localhost:8050
  1. 解析 www.example.com（區域 tw）：看它一步步走完三關問路（root→TLD→權威）。
  2. 立刻再解析同一個：這次會顯示「快取命中」——三關全省，只剩 1 步！
  3. 把區域切成 us / eu 再解析：看 GeoDNS 回不同 IP。
  4. 解析 blog.example.com：看權威回 CNAME → 再跟一次拿到 IP。
玩的時候想一下解析一次後立刻再解析 → 看步驟從 5 步變 1 步（快取命中省遞迴）。 按「清空快取」再解析 → 又變回完整三關（小抄被擦掉了）。切換區域看同一個名字回不同 IP（GeoDNS 就近）——這就是 DNS 的精髓。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心邏輯都在一個檔（Resolver.php）裡。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 一個檔，四段邏輯 查快取（先翻小抄）→ 三關遞迴（root→TLD→權威）→ 跟隨 CNAME（別名再跟一次）→ GeoDNS + 寫回快取。名字一進來，就照這個順序跑一遍。

1先翻小抄：有就秒回（查快取）

解析的第一件事，不是急著問路，而是先問「這個名字我剛查過嗎？還沒過期嗎？」。有就直接回，三關全省。

function 解析(名字, 區域) {
    答案 = 翻小抄(名字, 區域)
    if (答案存在 且 還沒過期)
        return { 答案, 來源:"快取命中", 步驟:1 }   // ← 秒回！三關全省
    // 小抄沒有 → 才開始問路（往下）
    ...
}

就這一段，把絕大多數查詢擋在「問路」之前。少了它，每次都要從頭問三關，伺服器分分鐘被問爆。

2三關問路：root → TLD → 權威（遞迴）

小抄沒有，才一層層問下去。每問一關就在「解析鏈」上記一筆（demo 畫面看到的就是這個）：

function 問路(名字, 區域) {
    tld = 取出結尾(名字)            // 如 example.com → "com"
    記一筆鏈："① 問 root：.{tld} 的權威是誰？"
    if (root 不認得這個 tld) return 查無此名(NXDOMAIN)

    記一筆鏈："② 問 TLD(.{tld})：{網域} 的權威(NS)是誰？"
    if (TLD 沒有這個網域) return 查無此名(NXDOMAIN)

    記一筆鏈："③ 問權威：{名字} 的記錄是什麼？"
    記錄 = 權威區域表[名字]
    if (沒這筆) return 查無此名(NXDOMAIN)
    ...（依記錄型別處理，往下）
}

這就是前面講的「市政府 → 區管理處 → 大樓櫃台」。每一關只負責把你導向下一關，最後權威才給真答案。

3遇到別名就再跟一次（CNAME）

權威給的不一定是 IP，可能是個「小名」。是小名就拿著小名再問一次，直到拿到真 IP：

if (記錄型別 == CNAME) {
    記一筆鏈："權威回 CNAME → " + 目標名字 + "（要再跟一次）"
    return 問路(目標名字, 區域)      // ← 拿著別名再跑一次（直到拿到 A）
}

就是這一句「return 問路(目標名字)」做到「再跟一次」。所以你在 demo 查 blog 會看到鏈變長（它先回 CNAME 指向 www，再去解析 www）。

4看人給答案 + 記進小抄（GeoDNS + 快取）

拿到 A 記錄了。如果這筆是 GeoDNS，就看你在哪區挑最近的 IP；最後把答案記進小抄（依 TTL）：

if (記錄型別 == A) {
    if (這筆有分區設定)              // GeoDNS
        IP = 分區表[區域] ?? 預設    // ← 台灣給台灣的、美國給美國的
    else
        IP = 記錄的 IP
    記一筆鏈："權威回 A：" + IP
}
寫進小抄(名字, 區域, IP, TTL)        // ← 下次同名(同區)直接秒回
return { 答案: IP, TTL }

注意寫小抄時連「區域」也當成鑰匙的一部分——這樣台灣人的答案才不會被美國人拿去用（GeoDNS 的分區快取）。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「問路」解釋，一次 DNS 解析經過哪幾關？ 先翻小抄（快取），有就秒回。沒有才問三關：① root（.com 找誰）→ ② TLD（example.com 找誰）→ ③ 權威（拿到 IP）。像問市政府→區管理處→大樓櫃台，一層導向一層。
Q2：查詢量這麼大，為什麼權威伺服器沒被打爆？ 因為多層快取（瀏覽器→OS→resolver）把 90%+ 查詢擋在外面，大多數查詢「小抄就有」直接秒回，真正穿透打到權威的只剩一小撮。
Q3：TTL 是什麼？設太長或太短各有什麼問題？ TTL 是答案在小抄上的「保存期限」。太長 → 快取命中高但改記錄很久才生效（搬家/切換很卡）；太短 → 改得快但大家一直重問、伺服器更累。是核心取捨。
Q4：我把網站換 IP，為什麼有人還連到舊的？怎麼讓切換更快？ 因為他的快取還沒過期，還在用舊答案。要搬家前先把 TTL 調短（如改成 1 分鐘），等短 TTL 全球生效後再切換，這樣大家很快就會重問拿到新 IP。
Q5（工程）：CNAME 和 A 差在哪？解析時怎麼處理 CNAME？ A 是終點（名字→IP）；CNAME 是別名（名字→另一個名字）。拿到 CNAME 不算結束，要拿著目標名字「再跟一次」去解析，直到拿到 A。
Q6（工程）：GeoDNS 為什麼讓快取變麻煩？怎麼解？ 因為同一個名字依區域回不同 IP，不能全球共用一份快取（不然會給錯）。解法是把「區域」也放進快取鑰匙裡，分區各存一份（demo 的快取鑰匙就帶了區域）。
Q7（工程）：遞迴和迭代解析的分工是？ client→resolver 是遞迴（小弟幫你跑完整段、又能共用快取）；resolver→root/TLD 是迭代（每關只回「下一關去問誰」，不幫你跑完，避免被當免費跑腿/放大器）。

15. 30 秒回顧

DNS = 網路的電話簿，把名字翻成 IP，要全球、要快、要就近。

一次解析：⓪ 先翻小抄（快取）有就秒回 → 沒有才 ① 問 root → ② 問 TLD → ③ 問權威拿 IP。別名(CNAME)要再跟一次，還會依你在哪(GeoDNS)給最近的 IP。

3 個怕：怕掛掉（多副本＋anycast）、怕慢/被問爆（多層快取＋就近）、怕改了不生效（TTL 長短取捨）。

工程細節一句話：壓力全在「讀」、而讀又超好快取 → 多層快取＋TTL 把洪水擋在權威之外；anycast/GeoDNS 讓全球就近；CNAME 再跟一次、GeoDNS 分區快取，是兩個最常被追問的細節。
