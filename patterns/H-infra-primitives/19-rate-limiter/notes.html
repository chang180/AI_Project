⑲ 分散式限流器：保護後端不被打爆

H 基礎設施 建議 35 分鐘 Design a Distributed Rate Limiter

一句話定位 本質是「在每個 key（user / IP / API key）的時間窗上，原子地計數並決定放行或拒絕」的問題。 單機限流是一行 if (count++ > limit)；面試真正考的是 多台機器共享同一份計數 如何做到既原子、又低延遲、還能大規模水平擴展。

1. 需求釐清

功能性需求

  • 對每個 key（user id / IP / API key / endpoint）施加速率上限，如「每 10 秒 5 次」。
  • 超出上限時拒絕請求，回 429 Too Many Requests 並附 Retry-After。
  • 規則可設定且可分層：全域、租戶、單一 API、單一使用者各有不同上限。
  • 限流決策需快，且本身不能成為新的瓶頸或單點故障。

非功能性需求

  • 低延遲：限流檢查在關鍵路徑上，每請求只能加 < 1~2ms。
  • 高可用：限流器掛掉時要有 fail-open / fail-close 策略，不能拖垮整站。
  • 可擴展：限流節點通常有數十到數百台，需共享同一份計數狀態。
  • 準確度可協商：多數場景近似即可（少放行幾個沒事），不需強一致。
面試開場必問 「要精確還是近似？」與「限流放在哪一層（API Gateway / Sidecar / 應用內）？」這兩問決定整題走向。 多數真實系統選「近似 + 放在 Gateway」，因為精確全域計數的協調成本極高、且通常不值得。

2. 規模估算

指標假設推算
總流量1 億請求/日~1.2k QPS（峰值 ×5 ≈ 6k QPS）
限流節點每節點 2k QPS峰值需 ~3~5 個節點，全要看到同一份計數
計數狀態每 key 一個計數器 + TTL1 千萬活躍 key × ~100B ≈ 1 GB，輕鬆放進 Redis 記憶體
Redis 負載每請求 1 次原子操作6k QPS 對單一 Redis 主節點是小菜；熱點 key 才需要分散
延遲預算同機房 Redis RTT~0.2~0.5ms，符合 < 1~2ms 的關鍵路徑要求
關鍵洞察狀態量很小（GB 級）、但存取頻率極高且必須原子 → 設計圍繞「原子計數操作」與「熱點 key 分散」，而非容量。

3. API 設計

# 限流器對外其實沒有「業務 API」，它是中介層（middleware）。
# 對上游業務服務暴露的是一個決策介面：

allow(key, rule) -> { allowed: bool, remaining: int, retry_after: int }

# 被限流時，Gateway 直接回給客戶端：
HTTP/1.1 429 Too Many Requests
Retry-After: 3                      # 幾秒後可重試
X-RateLimit-Limit: 5                # 視窗上限
X-RateLimit-Remaining: 0            # 本視窗剩餘
X-RateLimit-Reset: 1717142400      # 視窗重置的 epoch 秒

# 規則設定（管理面 API）
PUT /admin/rules/{scope}
Body: { "key_by": "api_key", "limit": 1000, "window_sec": 60, "algo": "token_bucket" }
為何回 Retry-After讓行為良好的客戶端知道確切該等多久，避免它無腦重試把你打更慘（驚群效應）。這是限流器「友善」與否的分水嶺。

4. 資料模型

# Redis 中的計數狀態（per-key），靠 TTL 自動回收
# ---- 固定/滑動視窗計數器 ----
KEY  ratelimit:{key}:{window_id}    -> INTEGER   (該視窗的請求數)
     TTL = window_sec * 2           # 兩個視窗後自動過期

# ---- 令牌桶（token bucket）----
HASH ratelimit:{key}                -> { tokens: 4.2, last_refill: 1717142399.812 }
     # tokens 為浮點：依「現在 - last_refill」乘補充速率算出新增的 token

規則設定表（變動少，可全量快取在各節點記憶體）
  scope, key_by(user|ip|api_key), limit, window_sec, algo, fail_mode
為何用 TTL限流計數是有時效的，過期即無意義。靠 Redis 的 TTL 自動回收，省去掃描清理；key 設計把 window_id 編進去，舊視窗自然被淘汰。

5. 高階架構

🖼️ 開啟 Excalidraw 架構圖（diagram.excalidraw） 
                              ┌──────────────────┐
                              │  限流規則設定 (黃) │
                              └─────────┬────────┘
                                        │ 載入規則
                                        ▼
 客戶端 ──請求(藍)──▶ API Gateway (限流中介) ──原子計數──▶ Redis (令牌桶/計數)
                            │                                   │
            放行(綠) ◀──────┤◀─────────── allowed? ────────────┘
                            │
                            ├──放行(綠)──▶ 後端服務
                            │
                            └──超限──▶ 429 Too Many Requests + Retry-After
  1. API Gateway（限流中介）：每請求先解出 key，套用規則，呼叫 Redis 原子操作拿到放行決策。
  2. Redis（共享狀態）：所有 Gateway 節點共用，用 INCR/EXPIRE 或 Lua 腳本做原子計數，避免多節點各算各的。
  3. 決策分流：allowed → 轉發後端；超限 → 立刻回 429，不打到後端（這就是限流的價值）。

6. 核心元件深入（演算法對比）

演算法原理取捨 / 邊界突刺
固定視窗計數
Fixed Window		把時間切成固定窗（如每分鐘），窗內計數，過窗歸零。最省（一個整數）；但視窗邊界突刺：窗末與下窗初各打滿可在 1 秒內過 2 倍量。
滑動視窗日誌
Sliding Log		記下每次請求的時間戳，算「最近 N 秒」內有幾筆。最精確、無邊界突刺；但要存每筆時間戳，記憶體與計算成本高，高 QPS 下昂貴。
滑動視窗計數器
Sliding Window Counter		當前窗計數 + 前一窗計數 按重疊比例加權 估算。近似但大幅平滑邊界突刺，記憶體僅兩個計數器 → 業界最常用的折衷。
令牌桶
Token Bucket		桶容量固定，依速率持續補 token；每請求耗 1 token，無 token 則拒。允許突發（攢著的 token 可一次用掉），平均速率受控 → API 限流首選。
漏桶
Leaky Bucket		請求進佇列，以固定速率「漏出」處理；滿則丟棄。輸出絕對平滑（整流），但不允許突發、且請求被排隊會增加延遲。

令牌桶 vs 漏桶：一句話分清

令牌桶限制的是「平均速率，但容許短時突發」；漏桶限制的是「瞬時輸出速率，絕不突發」。對外 API 通常要對用戶友善 → 令牌桶；對下游脆弱服務做整流（如寫 DB） → 漏桶。

邊界突刺問題（面試愛問）

固定視窗（每分鐘 100 次）：
  10:00:59  打 100 次  ✅（這分鐘的窗）
  10:01:00  打 100 次  ✅（下一分鐘的窗）
  → 在 1 秒內實際放行 200 次，是上限的 2 倍！

滑動視窗計數器修正：
  estimate = 當前窗計數 + 前一窗計數 × (前一窗未過去的比例)
  例：剛進新窗 10%，前窗 100 → 100×0.9 + 當前 10 ≈ 100，超限即擋。

分散式下如何共享狀態？

  • Redis 原子操作：INCR key 回傳遞增後的值，搭配第一次 EXPIRE 設窗 TTL。INCR 本身原子，多節點併發無 race。
  • Lua 腳本原子化：令牌桶要「讀 token + 算補充 + 扣減 + 寫回」多步，必須包成 Lua 腳本由 Redis 單執行緒原子執行，杜絕 read-modify-write 競態。
  • 近似而非精確：節點本地先攢一小批再回報，或容忍少量誤差，換取更低的 Redis 往返。多數場景「偶爾多放行幾個」完全可接受。
常見陷阱用 GET → 判斷 → SET 三步做計數，在多節點/多連線下會有 race condition 導致超賣放行。一定要用 INCR 或 Lua 把它變成單一原子操作。

7. 擴展與瓶頸（分散式共享狀態）

階段瓶頸對策
單機限流多台 Gateway 各算各的，總量失控把計數移到共享 Redis，所有節點看同一份
Redis 往返每請求一次 RTT 加延遲放同機房；本地攢批近似上報；pipeline 合併
熱點 key單一爆紅 key 打爆單一 Redis 分片對熱點 key 分桶（key#0..N 分散到多分片，各分 limit/N），或本地令牌桶降級
Redis 單點Redis 掛了限流失效主從 + 哨兵/Cluster；明確 fail-open（放行保可用）或 fail-close（拒絕保後端）
全球多區跨洲同步計數延遲高各區獨立限流（區域配額），不追求全球精確一致

8. 取捨與面試重點

  • 精確 vs 近似：精確全域計數要強一致協調，成本高；近似（滑動視窗計數器 / 本地攢批）便宜且夠用 → 絕大多數選近似。
  • 放哪一層：API Gateway 集中管理、易觀測但多一跳；Sidecar（每服務旁）延遲低、隔離好但運維重；應用內最省但難統一治理。
  • per-key 維度：按 user 防單用戶濫用、按 IP 防爬蟲/DDoS（但 NAT 會誤傷）、按 API key 做計費分級。常多維疊加。
  • fail-open vs fail-close：限流器/Redis 故障時——對外 API 多選 fail-open（寧可放行也別全站 503），對脆弱寫入路徑可能 fail-close。
  • 常見追問：
    • 怎麼防邊界突刺？→ 滑動視窗計數器或令牌桶，別用裸固定視窗。
    • 多節點計數怎麼不打架？→ Redis INCR / Lua 腳本原子化。
    • Redis 掛了怎麼辦？→ 預先定義 fail 模式 + 本地降級令牌桶兜底。
    • 怎麼告訴客戶端？→ 429 + Retry-After + X-RateLimit-* 標頭。

9. PHP Demo 說明

本題 demo 聚焦限流的核心放行/拒絕邏輯（也是面試真正考點），可實際用 PHP 內建伺服器跑起來：

  • src/TokenBucket.php　令牌桶：容量 + 補充速率，依經過時間補 token，tryConsume 成功放行否則拒。
  • src/SlidingWindowCounter.php　滑動視窗計數器：當前視窗 + 前一視窗加權估計，平滑邊界突刺。
  • src/RateLimiter.php　per-key 狀態管理，提供 allow(key) 介面，可切換演算法；用檔案鎖原子更新模擬 Redis 的原子操作。
  • public/index.php　路由：設定限制 / 連打測試 / 重置，並附深色測試頁。
# 啟動
cd demo
php -S localhost:8019 -t public
# 瀏覽器開 http://localhost:8019
# 設定「每 10 秒 5 次」→ 連續快打看前幾次放行、超出被拒(429)
# 等待數秒後再打 → 看 token 補回 / 視窗滑動恢復放行
誠實聲明 Demo 用 檔案鎖（flock）做原子讀-改-寫，模擬真實系統裡 Redis INCR / Lua 腳本 的原子性；單機示意、非真正分散式。令牌桶補充、滑動視窗加權、per-key 隔離、429 + Retry-After 等限流邏輯為真實可執行。要上線只需把檔案鎖換成 Redis 原子操作，演算法邏輯完全不變。
