# 🅷 基礎設施 / 平台元件

## 共通模式
面試常單獨考的「**積木題**」——不是完整產品，而是被無數系統共用的底層元件（限流、發號、佇列、鎖、設定）。考點集中在**正確性、原子性與分散式一致**。

## 可複用心法
- **原子操作**：共享狀態的「讀-改-寫」要原子化（Redis INCR / Lua 腳本 / CAS / 分散式鎖），避免並發競態。
- **演算法取捨**：同一目標常有多種演算法（限流的固定視窗 / 滑動視窗 / 令牌桶 / 漏桶），要能講清楚邊界突刺、精確 vs 近似的取捨。
- **無中心化擴展**：發號器（Snowflake）用時間戳 + 機器 id + 序號避免協調；注意時鐘回撥。
- **放在哪一層**：閘道 / sidecar / 服務內嵌；就近決策降延遲。
- **降級與容錯**：元件失效時 fail-open 還是 fail-close？要明確。
