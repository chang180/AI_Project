㊾ API Gateway / 負載均衡器：微服務的統一入口

H 基礎設施 建議 35 分鐘 Design an API Gateway / Load Balancer

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把一個進來的請求，依路由規則導到對的服務池，再在池內挑一台健康的後端」的問題。 難點不在「轉發」本身，而在 當後端會壞、會慢、會變多變少時，如何穩定地挑對機器：負載均衡、健康檢查、熔斷三件事缺一不可。

1. 需求釐清

功能性需求

  • 統一入口：所有客戶端請求先打 Gateway，由它處理橫切關注點（認證、路由、限流、熔斷），後端服務專心做業務。
  • 路由：依路徑前綴（/users/*、/orders/*）/ Host / Header 把請求導到對應的上游服務池（upstream pool）。
  • 負載均衡：在池內多台後端間分散流量（round-robin / 最少連線 / 一致性雜湊），避免某台過載。
  • 健康檢查：偵測後端死活，自動把流量繞過故障節點，恢復後再加回。
  • 熔斷：某後端連續失敗就暫時停止導流，避免把請求一直丟給已知壞掉的機器（快速失敗，保護自己與下游）。

非功能性需求

  • 低延遲：Gateway 在每個請求的關鍵路徑上，自身只能加極少的 overhead（次毫秒級）。
  • 高可用：Gateway 是全站咽喉，絕不能是單點；本身要能水平擴展、無狀態。
  • 可擴展：後端可動態增減（自動擴縮容），路由表與健康狀態要能即時更新。
  • 可觀測：集中收集每條路由的 QPS、延遲、錯誤率、熔斷狀態，是 Gateway 的天然優勢。
面試開場必問 「這是 L4（傳輸層，看 IP:port） 還是 L7（應用層，看 HTTP 路徑/標頭） 的 LB？」與「集中式 Gateway 還是 服務網格 sidecar？」這兩問決定整題走向。多數面試講的「API Gateway」是 L7、集中式。

2. 規模估算

指標假設推算
總流量1 億請求/日~1.2k QPS（峰值 ×5 ≈ 6k QPS）
Gateway 節點每節點 ~5k QPS（L7 解析有成本）峰值需 ~2~3 台 + 冗餘，前面再放一層 L4 LB 分流
後端服務數十個微服務，每服務 2~10 個實例路由表規模小（百量級規則），可全量快取在每台 Gateway 記憶體
健康狀態每後端一個 up/down + 熔斷器狀態幾百個 entry，KB 級；關鍵是更新要快且各 Gateway 看法一致
延遲預算Gateway 自身 overhead路由+LB 決策 < 1ms；瓶頸通常在後端與網路 RTT，不在 Gateway 算法
關鍵洞察狀態量很小（路由表、健康狀態都 KB~MB 級），設計重點不是容量，而是「每請求都要在關鍵路徑上做決策」與「Gateway 自身的高可用/水平擴展」。

3. API 設計

# Gateway 對「外」沒有自己的業務 API，它是反向代理（reverse proxy）。
# 客戶端只是正常打業務路徑，Gateway 透明轉發：
GET /users/42        → 路由到 users 池 → LB 挑 users-3 → 轉發
GET /orders/777      → 路由到 orders 池 → LB 挑 orders-1 → 轉發

# Gateway 對「內」的控制面 API（管理路由與後端）：
PUT  /admin/routes/{prefix}        # 設定/更新一條路由
  Body: { "prefix":"/users", "upstream":"users-pool", "lb":"round_robin",
          "timeout_ms":800, "retries":1, "rate_limit":"1000/min" }

PUT  /admin/upstreams/{pool}       # 設定服務池的後端清單
  Body: { "backends":["10.0.1.5:8080","10.0.1.6:8080"], "health":"/healthz" }

# 後端被熔斷 / 全部不可用時，Gateway 回給客戶端：
HTTP/1.1 503 Service Unavailable   # 池內無健康後端
HTTP/1.1 502 Bad Gateway           # 後端回錯/逾時
HTTP/1.1 429 Too Many Requests     # 觸發 per-route 限流（呼應第⑲題）
為何分「資料面 / 控制面」資料面（data plane）是高頻轉發請求的熱路徑（要極快）；控制面（control plane）是低頻的設定下發（路由表、後端清單）。兩者分離，讓設定變更不影響轉發效能，也是服務網格的核心觀念。

4. 資料模型

# 路由表（route table）：前綴 → 服務池 + 策略
ROUTE  prefix("/users") -> { upstream:"users-pool", lb:"round_robin",
                             timeout_ms, retries, rate_limit }

# 服務池（upstream pool）：一組後端
POOL   "users-pool" -> [ backend, backend, ... ]

# 後端（backend）的執行期狀態（健康 + 熔斷 + 負載）
BACKEND {
  id:"users-3", addr:"10.0.1.7:8080",
  up: true,                         # 健康檢查結果
  conns: 12,                        # 目前在途連線數（least-conn 用）
  cb_state: "closed",               # closed | open | half_open（熔斷器）
  consecutive_fails: 0,             # 連續失敗計數
  open_until: 1717142410            # open 狀態的冷卻到期時間
}
為何後端狀態要獨立記路由表（前綴→池）變動少、可長快取；但後端的 up/down、連線數、熔斷器是高頻變動的執行期狀態。把兩者分開，路由穩定、健康狀態即時，是 Gateway 設計的關鍵分層。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
                                   ┌────────────────────┐
                                   │ 控制面：路由表 /     │
                                   │ 後端池 / 健康狀態(黃)│
                                   └─────────┬──────────┘
                                             │ 下發設定
                                             ▼
 客戶端 ─請求(藍)→ L4 LB ─→ API Gateway (L7) ──路由+LB挑一台──▶ users 池 (綠)
                  (挑GW)      │ 認證·路由·限流              ├──▶ orders 池
                             │ ·LB·健康·熔斷               └──▶ pay 池(熔斷中→繞過)
                             │
              健康探測 /healthz ◀─────────────────────────────── 各後端
  1. L4 LB（前置）：在 Gateway 群前面，純看 IP:port 把連線分到某一台 Gateway。Gateway 自己要無狀態才能這樣水平擴展。
  2. API Gateway（L7）：解析 HTTP，做認證 → 路由（前綴比對選池）→ 限流 → 池內 LB 挑後端 → 轉發；後端壞了套用健康檢查與熔斷。
  3. 控制面：維護路由表與後端清單，透過健康探測更新 up/down，把設定下發到所有 Gateway 實例。

6. 核心元件深入

L4 vs L7 負載均衡

L4（傳輸層）L7（應用層）
看什麼IP / port（TCP/UDP）HTTP 路徑 / Header / Cookie / method
能力快、低 overhead，但看不懂內容能依路徑路由、改寫 Header、認證、限流、熔斷
定位常放最前面分流到 Gateway 群就是「API Gateway」本身

典型部署是兩層疊加：L4 把連線打散到多台 L7 Gateway，L7 再依內容做精細路由與治理。

負載均衡演算法（面試核心）

演算法原理取捨
輪詢 Round-Robin依序輪流挑下一台。最簡單、分布最均勻；但不看後端真實負載，慢機器一樣被分到等量。
加權輪詢 Weighted RR依機器規格給權重，強的多分。處理異質後端；權重要人工或自動調。
最少連線 Least-Connection挑目前在途連線最少的後端。對長連線/處理時間不一的場景更公平；要維護即時連線數。
一致性雜湊 Consistent Hash把 key（user/session）雜湊到環上，固定落到同一台。提供會話親和（sticky）/ 快取命中；後端增減時只重洗少量 key（vs 取模全洗）。
一致性雜湊為何重要若用 hash(key) % N，後端數 N 一變（擴縮容），幾乎所有 key 都重新對應，快取全失效。一致性雜湊把後端放到雜湊環上，增減一台只影響環上相鄰的一段 key → 是有狀態親和場景（會話、分片快取）的標準解。

健康檢查（health check）

  • 主動式：Gateway 定期打後端的 /healthz，連續失敗 → 標 down → 從候選池移除。恢復後連續成功 → 標 up 加回。
  • 被動式：直接觀察真實流量的成敗（逾時、5xx），不用額外探測。常與主動式並用。
  • 繞過：down 的後端不進入 LB 的候選清單，流量天然只導到健康節點——這就是「kill 一台自動繞過」的本質。

熔斷器（circuit breaker）三態

closed（正常）  ──連續失敗達門檻 N──▶  open（熔斷，快速失敗，不導流）
   ▲                                        │ 冷卻時間到
   │ 試探成功                                ▼
   └──────────  half_open（半開，放「一個」試探請求）
                      │ 試探失敗 → 退回 open（再等一輪冷卻）
  • closed：正常導流，累計連續失敗；成功就歸零。
  • open：已知這台壞了，直接不導流（快速失敗），避免把請求白白丟給壞機器、也給它喘息恢復。
  • half_open：冷卻後放一個試探；成功 → 回 closed 恢復，失敗 → 退回 open。
熔斷 vs 健康檢查兩者互補：健康檢查是「主動探測死活」（後端可能完全沒回應）；熔斷是「從真實請求的失敗率快速反應」（後端活著但在吐錯/變慢）。健康檢查管「能不能用」，熔斷管「最近表現好不好」。

重試與逾時

  • 逾時（timeout）：每條路由設定後端最大等待時間，避免慢請求拖垮 Gateway 連線池。
  • 重試（retry）：失敗時可換另一台健康後端重試，但只對冪等請求重試（GET 安全，POST 要小心），且要限重試次數 + 退避，否則重試風暴會放大故障。

前置治理：認證 / 限流

Gateway 是放認證（驗 JWT / API key）與限流（呼應 第⑲題）的天然位置——在轉發前統一處理，後端服務就不必各自重做。

7. 擴展與瓶頸

階段瓶頸對策
單台 Gateway全站咽喉，掛了全站死Gateway 無狀態 + 多實例 + 前置 L4 LB 分流；任一掛掉自動繞過
健康狀態同步各 Gateway 對後端死活看法不一主動探測 + 控制面集中下發；容忍短暫不一致（被動式即時補位）
長連線（WebSocket/gRPC stream）連線黏在某台 Gateway，難重均衡連線層親和 + 優雅排空（drain）；擴容對新連線生效
熱路由 / 熱 key某條路徑或某 key 爆量打爆單一後端per-route 限流 + 加權/最少連線分散；一致性雜湊熱點再加虛擬節點/分桶
設定下發路由/後端變更要即時生效又不抖動控制面推送 + 本地快取 + 原子切換（不中斷在途請求）

8. 取捨與面試重點

  • 集中式 Gateway vs 服務網格 sidecar：集中式好治理、好觀測、但多一跳且自己要高可用；sidecar（每服務旁掛一個代理，如 Envoy）延遲低、east-west 流量也能治理，但運維複雜、資源開銷大。多數團隊「南北向用 Gateway、東西向用 mesh」。
  • LB 演算法取捨：無狀態短請求 → 輪詢就夠；處理時間差異大/長連線 → 最少連線；需要會話親和或快取命中 → 一致性雜湊。沒有萬用解，看流量形狀。
  • 熔斷閾值：門檻太低（如失敗 1 次就斷）→ 偶發抖動就誤斷、可用性反而降；太高 → 壞機器拖很久才被隔離。要配合滑動窗錯誤率而非裸計數，並設半開試探避免一直斷。
  • 重試的雙面刃：重試提升成功率，但故障時重試風暴會把流量放大數倍、加速雪崩 → 必須限次數 + 退避 + 只重試冪等請求 + 配合熔斷。
  • 常見追問：
    • kill 一台後端怎麼自動繞過？→ 健康檢查標 down，不進 LB 候選清單。
    • 為什麼要熔斷而不只靠健康檢查？→ 後端「活著但在吐錯/變慢」時，熔斷從真實失敗率更快反應。
    • 後端擴縮容，一致性雜湊比取模好在哪？→ 只重洗環上相鄰一段 key，快取/會話不全失效。
    • Gateway 自己怎麼不成單點？→ 無狀態 + 多實例 + 前置 L4 LB。

9. PHP Demo 說明

本題 demo 聚焦 Gateway 的決策鏈（也是面試真正考點），可實際用 PHP 內建伺服器跑起來：

  • src/Gateway.php　核心引擎：路由（最長前綴比對選池）、負載均衡（round-robin / least-connection / consistent-hash 可切換）、健康檢查（kill 的後端不進候選、自動繞過）、熔斷器（連續失敗 3 次 → open，冷卻 10 秒 → half_open 試探 → 成功 closed）。狀態用 flock 原子讀-改-寫。
  • public/index.php　路由 + 深色測試頁（註冊服務池、送請求看挑到哪台、kill 看繞過、連續失敗看熔斷、看流量分布長條圖）。
# 啟動
cd demo
php -S localhost:8049 -t public
# 瀏覽器開 http://localhost:8049
# ① 一鍵建範例（/users 3 台 + /orders 2 台）
# ② 連送 12 次 → 看 round-robin 平均輪流；切「最少連線/一致性雜湊」看分布差異
# ③ kill 一台 → 再送，看自動繞過只導到健康的
# ④ 對某台連續失敗 ×4 → 熔斷打開自動繞過；等 10 秒看它轉半開試探
誠實聲明 Demo 為單機模擬：後端不是真的服務，而是記在 JSON 裡的計數器，用 flock(LOCK_EX) 把整段「讀狀態 → 跑 LB/熔斷 → 寫回」鎖成單一原子操作，模擬真實 Gateway 對共享狀態（通常在 Redis/etcd）的原子更新。但路由（最長前綴）、三種 LB 演算法、健康檢查繞過、熔斷器 closed→open→half_open→closed 的狀態轉移皆為真實可執行的邏輯。要上線只需把「JSON 計數器」換成真正轉發 HTTP + 主動健康探測，決策邏輯完全不變。
