1. 60 秒搞懂它在幹嘛

你用一個手機 App，點「我的訂單」「個人資料」「付款」。其實背後是好幾十個不同的小程式（微服務）分頭在做這些事，而且每種還開了好幾台機器一起扛。

問題來了：App 怎麼知道「個人資料要找哪台機器」？某台機器掛了它會不會就壞掉？這時就需要一個大門兼接待員——這就是 API Gateway。

🔑 一句話 API Gateway = 大樓的櫃台接待員：所有訪客都先到櫃台，櫃台看你要找哪個部門（路由）、那個部門有好幾個人就挑一個現在比較閒的（負載均衡）、那個人請假了就別把人帶過去（健康檢查）、那個人最近一直出包就暫時別找他、讓他休息（熔斷）。

2. 為什麼不能讓客戶端直接打後端？

很多人第一次想：「App 直接連到那台機器不就好了，幹嘛中間多一個櫃台？」——這想法在只有一台機器、永遠不會壞時成立，但真實世界不是這樣。

📌 比喻：沒有櫃台的大樓 想像一棟有 50 個部門、每部門好幾個座位的大樓，沒有櫃台。訪客得自己記住「會計王小姐在 7 樓 12 號座位」。 結果：王小姐請假換人坐了 → 訪客撲空；公司加開兩個座位分攤 → 訪客根本不知道有新座位；某個座位的人一直擺爛 → 訪客還是一直去找他生氣。整棟樓亂成一團。

有了櫃台（Gateway），訪客只要記住「我要找會計」，剩下「現在誰在、誰閒、誰能用」全部櫃台幫你搞定。後端機器愛怎麼增減、怎麼壞，客戶端完全不用知道。

💭 停一下：為什麼「讓 App 自己記住每台後端的地址」在大系統行不通？

看答案 因為後端機器會一直變：自動擴容多開幾台、壞掉下線、換 IP……如果每個 App 都記死地址，機器一變所有 App 就都要更新，根本管不動。 讓櫃台（Gateway）統一知道現在有哪些機器、誰能用，App 只認「我要找哪個服務」，這層「中間人」把變動全吸收掉了。

3. 把工作拆成 4 個關卡

櫃台接待員其實就做 4 件事，一關一關來：

1路由：你要找哪個部門？

請求進來，櫃台先看「你要找誰」。看的是網址路徑：/users/... 是要找會員部門、/orders/... 是要找訂單部門。

🏢 比喻：大樓樓層導覽 櫃台牆上有張表：「會員 → 3 樓」「訂單 → 5 樓」「付款 → 7 樓」。你說要找會員，櫃台就知道帶你往 3 樓那群人去。這張表就叫路由表。

👉 面試小撇步：路由可以看路徑（最常見）、也可以看網域（api.x.com vs img.x.com）或標頭。重點是「看請求的某個特徵，決定它該去哪一群後端」。

2負載均衡：這部門有好幾個人，找哪個？

「會員部門」不是只有一個人，是一群人（好幾台機器）。櫃台得挑一個把你帶過去——而且要挑得公平，別老是累死同一個。

🏦 比喻：銀行叫號 銀行有好幾個櫃員窗口。怎麼分客人？① 輪流（1 號、2 號、3 號、再回 1 號）；② 看誰手上客人最少就去那；③ VIP 固定找同一個專員（他熟你的狀況、辦得快）。

這三種挑法，工程上就叫三種負載均衡演算法：

  • 輪流（round-robin）：最公平簡單，一台一台輪。
  • 最少連線（least-connection）：誰現在手上最閒就給誰，適合「每個請求處理時間差很多」的場景。
  • 一致性雜湊（consistent-hash）：同一個客人固定找同一台（像 VIP 找固定專員），這台機器會「記得」你、快取你的資料，辦起來更快。

3健康檢查：那個人今天請假了！

挑人之前要先確認：這個人今天到底有沒有來上班？請假的（機器掛了）就別把客人帶過去，自動跳過他、帶去別的還在的人那邊。

📋 比喻：櫃台的出勤板 櫃台牆上有塊出勤板，誰在誰請假一目了然。櫃台每隔一陣子去敲敲每個人的門問「還在嗎？」（這叫健康探測）。沒回應的就在板上標「請假」，之後叫號直接跳過他，等他回來再標「上班」加回去。

這就是為什麼「kill 掉一台機器，系統不會壞」——櫃台發現它沒回應，自動把它從候選名單拿掉，客人全導到還活著的機器。

4熔斷：那個人最近一直出包，先讓他休息

有種狀況比「請假」更麻煩：那個人有來上班，但最近一直辦錯、或慢到不行。每次帶客人過去都搞砸，客人白等一場還氣得要死。怎麼辦？

🔌 比喻：家裡的保險絲（斷路器） 家裡電線短路時，保險絲會「啪」一聲跳掉斷電，不是壞掉，是故意切斷免得燒掉整間房子。過一陣子你去試著合上看看，沒事就恢復供電，還是有問題就再跳掉。

櫃台對「一直出包的人」也這樣：連續搞砸 3 次 → 暫時別再把客人帶給他（這叫熔斷打開，客人直接帶去別人那，不用白等）。過 10 秒後，試著帶「一個」客人去探探看（這叫半開試探）——他辦好了就恢復正常，又搞砸就再讓他多休息一輪。

✅ 四關一句話 ① 看你要找哪部門（路由）→ ② 那部門挑個閒的人（負載均衡）→ ③ 請假的跳過（健康檢查）→ ④ 一直出包的先讓他休息、過陣子再試（熔斷）。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「櫃台自己倒了」 所有人都得先經過櫃台——萬一櫃台本身掛了，整棟樓癱瘓。這是 Gateway 最致命的怕。 所以櫃台要開好幾個（多台 Gateway），前面再放一個更簡單的分流器（L4 LB）把訪客分散到不同櫃台。任一個櫃台倒了，還有別的頂上。咽喉絕不能是單點。
😱 怕「把客人帶給壞掉的人」 帶客人去找請假的人（撲空）、或一直出包的人（白等），都是災難。 所以才需要健康檢查（請假的跳過）+ 熔斷（出包的先休息）。兩個一起，確保客人只會被帶到「現在真的能用」的人那。只導到健康的。
😱 怕「越救越亂」 某台慢了，櫃台想「那我重試、換一台」——聽起來貼心，但如果大家同時一直重試，等於把流量加倍灌進本來就快撐不住的系統，越救越糟（重試風暴）。 所以重試要限次數、要等一下再試、只對「重做也沒關係」的請求重試（查資料可以重試，扣款可不能重試兩次！）。救援也要節制。

round-robin（輪詢）一台一台輪流挑
least-connection（最少連線）誰手上客人最少就給誰
consistent-hash（一致性雜湊）同一個客人固定找同一台（像 VIP 找固定專員）

L4 / L7L4 只看門牌號(IP)分流；L7 看得懂「你要找哪個部門」

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定開幾個櫃台、租多大場地。系統設計也一樣：先估數字，才知道要準備多大的「料」。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每秒多少請求平常 ~1200，尖峰 ~6000一個櫃台扛不完 → 要開好幾個櫃台分攤
要開幾個櫃台(Gateway)2~3 個 + 備援前面再放一個分流器把人散到各櫃台
後端有多少機器幾十個服務 × 每個 2~10 台「路由表 + 出勤板」其實很小，記在記憶體就好
櫃台自己慢不慢每個請求加 < 1 毫秒櫃台不是瓶頸，慢通常慢在後端本身
💡 關鍵洞察：要存的東西很少，難的是「不能倒」 這題跟「存很多資料」的題目不一樣——路由表、出勤板都小到能放進記憶體。 真正難的是：櫃台在每一個請求的必經之路上，所以它① 要夠快（別拖慢大家）② 絕不能倒（倒了全站癱瘓）。設計重點全在這兩點。

💭 停一下：這題要存的資料明明很少，為什麼還是很難？

看答案 因為難點不在「存多少」，在「它是全站的咽喉」。每個請求都得先過櫃台，所以櫃台慢一點 → 全站都慢；倒一個 → 全站都連不上。 所以要開多個櫃台（高可用）、決策要極快（低延遲）、還要即時知道後端誰能用（健康/熔斷）。是「可靠度」難，不是「容量」難。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」。這題特別的是：櫃台對客人「沒有自己的點餐單」——客人照常點餐（打業務網址），櫃台只是把單子轉交給對的廚房。 
① 客人正常打業務網址，櫃台「透明」轉發（客人感覺不到中間有櫃台）
   GET /users/42    → 櫃台路由到「會員部門」→ 挑一台 → 轉過去
   GET /orders/777  → 櫃台路由到「訂單部門」→ 挑一台 → 轉過去

客人不用知道「會員部門現在有哪幾台機器」，只要打 /users/...，櫃台自然帶到對的地方。這就是 Gateway 最大的價值：把後端的複雜全藏起來。

② 給管理員用的「設定點餐單」（控制面，客人碰不到）
   設定一條路由：  /users → 會員部門，用「輪流」挑機器，逾時 0.8 秒
   設定部門有哪些機器：  會員部門 = [機器A, 機器B, 機器C]，健康檢查打 /healthz

這些是管理員在用的，平常不太變動。把「客人用的轉發」和「管理員用的設定」分開，是 Gateway 的重要設計（叫資料面 vs 控制面）。

③ 出事時，櫃台回給客人的「壞消息」
   503 = 這個部門「沒有人能用了」（全請假或全熔斷）
   502 = 帶過去了但那台機器吐錯/超時
   429 = 你太頻繁了，被限流擋下（呼應第⑲題限流器）

9. 系統要記住哪些事？（資料模型）

📒 比喻：櫃台桌上的三本本子 櫃台要把資料記在「本子」（資料表）裡。這題主要三本：

🗺️樓層導覽表（路由表）

記「哪個路徑 → 哪個部門 + 用什麼方式挑人」：/users → 會員部門、用輪流、逾時 0.8 秒。 這本很少變動，可以一直放著。

👥部門名冊（服務池）

記「每個部門有哪些人（機器）」：會員部門 = [機器A, 機器B, 機器C]。 機器擴容/下線時改這本。

📋出勤板（每台機器的即時狀態——核心！）

這本變動最頻繁，記每台機器的： 現在在不在(健康)、手上幾個客人(連線數)、最近連續出包幾次、保險絲狀態(正常/跳開/試探中)。

為什麼這本要單獨記、還要即時？ 因為「樓層表、部門名冊」幾乎不變，但「誰現在能用」每秒都在變（有人剛掛、有人剛出包、有人剛恢復）。 把這本即時狀態單獨拉出來、隨時更新，櫃台每次挑人前看一眼，就能只挑現在真的能用的。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
只有一個櫃台 → 它倒了全站癱瘓櫃台不存私人狀態(無狀態) → 就能開好幾個 + 前面放分流器
各櫃台對「誰請假」看法不一致有人專門去敲門更新出勤板，再統一發給每個櫃台
視訊/長連線一直黏在某個櫃台讓連線黏住沒關係，新連線才重新分配；舊的慢慢排空
某條超熱門路徑打爆某部門對那條路徑限流 + 用「最少連線」分散 + 機器再加幾台
大家同時重試把系統壓垮重試限次數 + 等一下再試 + 配合熔斷快速失敗

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 一個大櫃台 vs 每個部門配一個小秘書 「一個大櫃台」（集中式 Gateway）好管理、好統計，但中間多一站、自己要顧好別倒； 「每個部門門口配小秘書」（服務網格 sidecar）反應更快、部門之間互打也能管，但要養超多小秘書、很耗資源、難維護。大公司常兩者並用。
⚖️ 挑人方式怎麼選？ 請求都差不多快 → 輪流最省事；每個請求快慢差很多 → 最少連線更公平； 需要「同一個客人固定找同一台」(用到快取/記住狀態) → 一致性雜湊。沒有萬用解，看你的請求長什麼樣。
⚖️ 保險絲要多敏感？ 太敏感（出包一次就跳）→ 偶爾抖一下就把好機器關掉，反而更不可用； 太遲鈍（出包很多次才跳）→ 壞機器拖很久才被隔離，客人白等一堆。 折衷：看「最近一段時間的出包比例」而不是死數次數，再加「過陣子放一個試探」避免一直關著。
💭 追問：為什麼有了「健康檢查」還要「熔斷」？不是重複嗎？ 不重複，它們抓不同狀況：健康檢查抓「人請假了/完全沒回應」（主動敲門問）；熔斷抓「人有來但一直辦錯/變超慢」（從真實客人的成敗快速反應）。 一個管「在不在」，一個管「最近表現好不好」。兩個一起，才能完整避開「把客人帶給不能用的人」。
💭 追問：後端從 3 台變 4 台，為什麼「一致性雜湊」比「除以台數取餘數」好？ 如果用「客人編號 ÷ 台數 取餘數」決定找哪台，台數從 3 變 4，幾乎每個客人對應的機器全變了——之前記住你的快取全部白費。 一致性雜湊則是把機器排在一個圈圈上，加一台只影響圈圈上它旁邊那一小段的客人，其他人照舊找原本那台，快取大多還在。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「路由 + 負載均衡 + 健康檢查 + 熔斷」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8049 -t public
# 然後瀏覽器打開 http://localhost:8049
  1. 一鍵建範例：建好「會員部門 3 台 + 訂單部門 2 台」。
  2. 連送 12 次請求到 /users/42：看長條圖——「輪流」會三台平均；切「最少連線」「一致性雜湊」看分布怎麼變。
  3. kill 一台：把某台標成壞掉，再送請求 → 看它自動被跳過，流量只進活著的。
  4. 對某台連續失敗 ×4：看它的保險絲從 closed 跳成 open（熔斷），請求自動繞過；等 10 秒看它變 half_open（放一個試探）。
玩的時候想一下把一台 kill 掉後，剩下兩台分到的請求是不是變多了？這就是「自動繞過」。再讓某台連續失敗觸發熔斷，看它明明還「在」(UP) 但保險絲跳了 → 一樣不導流——這就是健康檢查與熔斷的差別。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那四關到底是哪幾段程式做的」。 好消息：核心都在一個檔 Gateway.php裡，剛好對應前面講的四關。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 一個檔 = 櫃台接待員的腦袋 Gateway.php 就是櫃台的大腦：請求一進來，依序跑「路由 → 篩掉不能用的 → 挑一台 → 餵給保險絲」。我們一段一段看。

1路由：你要找哪個部門？（最長前綴）

請求進來第一件事：看路徑屬於哪個部門。比對「哪條路由前綴最符合」：

function 找部門(請求路徑) {
    在所有路由規則裡，找「請求路徑開頭剛好是這個前綴」的，
    若有好幾個都符合，挑「最長那個」（最精確）
    例：/users/vip 同時符合 /users 和 /users/vip → 選 /users/vip
}

用「最長前綴」是為了讓更精確的規則優先。找不到任何部門就回 404（這路徑沒人接）。

2篩掉不能用的：請假的、保險絲跳開的（健康檢查 + 熔斷）

挑人之前，先把「現在不能用的」從候選名單拿掉：

候選名單 = []
for (部門裡每一台 機器) {
    if (機器.健康 == false) continue;      // ← 請假的(被 kill)：跳過
    if (機器.保險絲 == "跳開")   continue;   // ← 熔斷中：跳過
    候選名單.加入(機器)                      // 只有「能用的」才進候選
}
if (候選名單 是空的) return 503;            // 整個部門都不能用了

這就是「kill 一台自動繞過」「熔斷後不導流」的本質——不能用的根本不進候選，後面挑人自然不會挑到它。

3挑一台：三種挑法（負載均衡）

從候選名單裡挑一台，看設定用哪種方式：

if (方式 == "最少連線")
    挑 候選名單裡「手上連線數最小」的那台
else if (方式 == "一致性雜湊")
    把這個客人(key)放到圈圈上，順時針找到第一台候選機器（同 key 永遠同一台）
else  // 預設「輪流」
    游標往後走，停在「下一個還在候選名單裡」的機器（天然跳過請假/熔斷的）

注意「輪流」的游標只會停在候選名單裡的機器——所以就算中間有幾台被 kill 或熔斷，它會自動跳過，不會挑到壞的。

4餵給保險絲：成功歸零、連續失敗就跳開（熔斷狀態機）

這是全題的心臟。請求處理完（成功或失敗），更新這台機器的保險絲：

if (這次成功) {
    機器.連續失敗 = 0                        // 歸零
    if (機器.保險絲 == "試探中") 機器.保險絲 = "正常"  // 試探成功 → 恢復
} else {  // 這次失敗
    機器.連續失敗++
    if (機器.連續失敗 >= 3) {                 // ← 連續失敗 3 次
        機器.保險絲 = "跳開"                  // 熔斷打開！
        機器.冷卻到 = 現在 + 10 秒            // 休息 10 秒
    }
}

// 另外，每次處理前先看：跳開的保險絲冷卻夠了沒？
if (機器.保險絲 == "跳開" 且 已過冷卻時間)
    機器.保險絲 = "試探中"                    // ← 半開：放一個試探看看好了沒

前面講的「正常 → 跳開 → 試探中 → 回正常」整套保險絲循環就是這段。連續失敗 3 次跳開、休息 10 秒、放一個試探、成功就恢復。這也是面試官最愛追問的一段。

💭 停一下：保險絲「跳開」後，為什麼不直接永遠關著，要設一個「試探中」狀態？

看答案 因為機器可能只是暫時出問題，後來自己好了。如果永遠關著，它好了也永遠用不到，浪費一台。 所以休息一陣後放「一個」客人去試探——好了就恢復全部流量，還沒好就再關一輪。只放一個，是怕它還沒好就一次湧進一堆客人又全失敗。
🧭 把程式對回前面的關卡
關卡（前面講的）哪段程式
① 找哪個部門找部門()：最長前綴比對
② 請假/熔斷的跳過組「候選名單」時的兩個 continue
③ 挑一台三種挑法（輪流 / 最少連線 / 一致性雜湊）
④ 出包就跳保險絲處理後更新「連續失敗 / 保險絲狀態」那段

👉 重點：真實系統會把「JSON 計數器」換成「真的轉發 HTTP 到機器 + 定時敲門健康探測 + 狀態放 Redis」， 但上面這幾段判斷邏輯一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「大樓櫃台」比喻，API Gateway 做哪四件事？ ① 看你要找哪個部門（路由）② 那部門挑一個閒的人帶你過去（負載均衡）③ 請假的人跳過（健康檢查）④ 最近一直出包的人先讓他休息、過陣子再試（熔斷）。
Q2：為什麼不讓 App 直接連後端機器？ 因為後端機器會一直變（擴容、下線、換 IP、壞掉）。讓 App 記死地址，機器一變所有 App 都要改、根本管不動。櫃台(Gateway)當中間人，把這些變動全吸收，App 只認「我要找哪個服務」。
Q3：三種「挑機器」的方式各適合什麼場景？ 輪流(round-robin)：請求都差不多 → 最省事；最少連線：請求快慢差很多 → 給手上最閒的更公平；一致性雜湊：要「同客人固定同一台」(快取/記住狀態) → 像 VIP 找固定專員。
Q4：有了健康檢查為什麼還要熔斷？ 健康檢查抓「完全沒回應/請假了」（主動敲門問）；熔斷抓「有在但一直出錯或變超慢」（從真實請求成敗快速反應）。一個管「在不在」，一個管「最近表現好不好」，互補。
Q5：保險絲（熔斷器）的三個狀態怎麼轉？ 正常(closed)：累計連續失敗，成功就歸零 → 連續失敗達 3 次 → 跳開(open)：直接不導流、休息 → 冷卻 10 秒後 → 試探中(half_open)：放一個請求試 → 成功回正常、失敗退回跳開再休息一輪。
Q6：為什麼 API Gateway「絕不能是單點」？怎麼解？ 因為它是全站咽喉，每個請求都得經過它，倒一個就全站連不上。解法：讓 Gateway「不存私人狀態」(無狀態)，就能開好幾個，前面再放一個 L4 分流器把流量散到各台，任一台倒了還有別的頂上。
Q7：後端從 3 台變 4 台，一致性雜湊好在哪？ 若用「÷台數取餘數」，台數一變幾乎所有客人對應的機器全變、快取全失效；一致性雜湊把機器排在圈圈上，加一台只影響旁邊一小段客人，其他人照舊找原機器，快取大多保留。

15. 30 秒回顧

API Gateway = 大樓的櫃台接待員：所有請求的統一入口。

4 個關卡：① 看路徑找部門（路由）→ ② 部門裡挑個閒的（負載均衡：輪流/最少連線/一致性雜湊）→ ③ 請假的跳過（健康檢查）→ ④ 一直出包的先休息、過陣子試探（熔斷，像保險絲）。

3 個怕：怕櫃台自己倒（多開櫃台 + 前置分流器）、怕把客人帶給壞掉的人（健康檢查 + 熔斷）、怕越救越亂（重試要限次數+退避+只重試安全的）。

工程細節一句話：要存的東西很少（路由表/部門名冊/出勤板都進記憶體），難的是「它是咽喉」→ 必須又快又不能倒；用「點餐單(API)」透明轉發、把「客人用的轉發」與「管理員用的設定」分開、出勤板即時更新只挑能用的、找出塞車環節各自拓寬。
