㉓ 唯一 ID 產生器 Snowflake：分散式、無協調、可排序

H 基礎設施 建議 35 分鐘 Design a Unique ID Generator (Snowflake)

一句話定位 本質是「在多台機器上、不互相協調，就能各自產生全域唯一且大致時間有序的 64-bit ID」的問題。 單機自增是一行 id++；面試真正考的是 分散式 下如何同時做到唯一、趨勢遞增（可排序）、高吞吐、又不依賴中心發號者。

1. 需求釐清

功能性需求

  • 產生全域唯一的 ID，供分散式系統當主鍵 / 訊息 id / 訂單號。
  • 趨勢遞增（roughly time-ordered）：後產生的 ID 數值通常較大，便於依時間排序、建索引。
  • ID 為數字且能塞進 64-bit 整數（DB BIGINT、語言 int64）。
  • 可由 ID 反解出產生時間、機器來源（除錯、分片）。

非功能性需求

  • 高吞吐 / 低延遲：取號是熱路徑，每次 < 1ms、本機記憶體運算，不該每次都打網路或 DB。
  • 高可用：發號掛了等於全站不能寫入 → 不能有單點。
  • 水平擴展：加機器就能線性提高總產量。
為何不用 DB auto-increment？
  • 協調瓶頸 / 單點：單一自增序列要嘛走單庫（吞吐天花板、單點），要嘛多庫設不同步長（運維脆弱、難擴）。
  • 跨分片不唯一：分庫分表後各自 auto-increment 會撞號。
  • 每次取號一次往返：高頻寫入時 DB 變熱點。
為何不用 UUIDv4？
  • 無序：隨機 128-bit，插入 B+Tree 主鍵造成頁分裂、索引碎裂、寫放大，快取局部性差。
  • 太長：128-bit / 36 字元字串，比 64-bit 整數占空間、比較慢。
Snowflake 用「時間在高位」換到趨勢遞增，同時保留分散式無協調的優點。（UUIDv7 把時間放高位也解了無序問題，見 §8 對比。）

2. 規模估算

指標佈局推算
單機每毫秒12 bit 序號2¹² = 4096 個 / ms
單機每秒4096 × 1000≈ 409 萬 個 / 秒
叢集每秒10 bit 機器 id1024 機 × 409 萬 ≈ 41.9 億 個 / 秒
時間可用年限41 bit 毫秒2⁴¹ ms ÷ (1000·60·60·24·365) ≈ 69.7 年
數值範圍63 bit 正整數最大 ~9.2×10¹⁸，落在 BIGINT / int64 內
關鍵洞察瓶頸不在「能不能產生夠多」（41.9 億/秒綽綽有餘），而在時鐘（回撥、NTP 跳變）與機器 id 的分配與唯一性。整題的深度都在這兩點。

3. API 設計

# 產生一個 ID（本機記憶體運算，極低延遲）
POST /api/id
→ 200 { "id": 319639026040573952,
        "decoded": { "timestamp": 1780275082414, "machineId": 1, "seq": 0 } }

# 批次產生（一次取一段，攤平 RPC / 鎖成本）
POST /api/id/batch   Body: { "n": 50 }
→ 200 { "count": 50, "ids": [ ... ] }

# 解碼（除錯用：由 ID 反推時間 / 機器 / 序號）
GET /api/decode/{id}
→ 200 { "timestamp": ..., "datetime": "2026-06-01 00:51:22.414 UTC",
        "machineId": 1, "seq": 0 }
部署形態取號器通常以函式庫內嵌在每個應用程式（如本 demo），或做成本機 sidecar / 同機服務，避免每次取號跨網路。上面的 HTTP API 主要是為了 demo 與跨語言共用。

4. ID 位元佈局（核心）

Snowflake 把 64-bit 切成四段，時間放最高位是「趨勢遞增」的關鍵：時間大的 ID 數值一定大。

欄位位元數範圍作用
sign（保留）1固定 0保持正整數，避免有號數變負
時間戳（ms）410 .. 2⁴¹−1自訂 epoch 起的毫秒；放高位 → 可排序、可用 ~69 年
機器 id100 .. 1023區分不同產生器；保證跨機唯一（不必協調）
序號 seq120 .. 4095同一毫秒內遞增，解決單機同毫秒撞號
 0 | 41 bits 時間戳(ms) | 10 bits 機器 id | 12 bits 序號
 ^   ^                    ^                 ^
sign 自訂 epoch 起的毫秒   0..1023           0..4095（同毫秒遞增）

組合：  id = ((ts - EPOCH) << 22) | (machineId << 12) | seq
EPOCH = 1704067200000   // 2024-01-01 00:00:00 UTC（毫秒）
為何自訂 epoch用接近「上線時間」的 epoch 取代 1970，可讓 41 bit 的 ~69 年從上線那刻才開始算，最大化可用年限。位元分配可依需求調整（如機器多就多給機器 id、單機產量大就多給 seq），但總和固定 63 bit。

5. 高階架構

🖼️ 開啟 Excalidraw 架構圖（diagram.excalidraw） 
 App Server #1 ─┐  (內嵌 Snowflake，machineId=1)
 App Server #2 ─┼─▶ 各自於本機記憶體取號，無中心協調 ──▶ 全域唯一 ID
 App Server #N ─┘  (machineId=N)
        ▲
        │ 開機時註冊 / 領取唯一 machineId（黃色基建）
   ┌──────────┐
   │ ZooKeeper │  或 etcd / Redis
   │  / etcd   │
   └──────────┘
  1. 無中心發號者：每台機器內嵌產生器，取號全在本機記憶體完成，不跨網路 → 無單點、極低延遲、線性擴展。
  2. 唯一性怎麼來：靠 (時間戳 + 機器 id + 序號) 三者組合。只要機器 id 不重複、且同機時間單調，全域就唯一。
  3. 唯一的協調點：只有「機器 id 的分配」需要一次性協調（開機時向 ZooKeeper/etcd 領一個），之後取號完全自治。

6. 核心元件深入

同毫秒序號遞增

同一毫秒內多次取號 → seq = (seq + 1) & 4095 遞增；新的毫秒到來 → seq 歸零。

if (now == lastTs) {
    seq = (seq + 1) & MAX_SEQ;        // 0..4095
    if (seq == 0) now = waitNextMs(lastTs); // 序號用滿 → 自旋等到下一毫秒
} else {
    seq = 0;                          // 進入新毫秒，歸零
}
lastTs = now;
序號用滿（4096/ms）極端高頻下單毫秒 4096 個用完 → 自旋忙等到下一毫秒再發。這是 Snowflake 的硬上限，也是「為何單機產量 = 4096×1000」的由來。

時鐘回撥（clock skew / NTP 校時）

Snowflake 的單調性建立在「本機時鐘不倒退」上。但 NTP 校時、VM 暫停、閏秒都可能讓 now < lastTs，若不處理會產生重複 ID。標準對策：

  • 小幅回撥（如 ≤ 數毫秒）：自旋等待時鐘追上 lastTs 再發號，犧牲一點延遲換正確性。
  • 大幅回撥：拒絕發號並丟例外 / 告警，把該節點摘除，絕不冒重複風險。
  • 進階：把 lastTs 週期性持久化，重啟時若發現系統時間早於它則拒絕啟動。

機器 id 分配（ZooKeeper / etcd）

  • 靜態設定：寫死在設定檔 / 環境變數。簡單，但容易人為設重複、擴縮容麻煩。
  • ZooKeeper 順序節點：開機時在某路徑建立 ephemeral sequential 節點，取得的序號就是 machineId；節點是 ephemeral，機器掛掉 session 過期就釋放，可回收重用（Twitter 原版做法）。
  • etcd / Redis 租約：用 lease/INCR 領一個唯一 id，定期續租；失聯則釋放。
  • 關鍵不變式：同一時刻不得有兩台機器持有同一 machineId，否則同毫秒同 seq 會撞號。

7. 擴展與瓶頸

面向瓶頸 / 風險對策
產量單機 4096/ms 上限加機器（10 bit → 1024 台）；或調整位元分配多給 seq
機器數10 bit 只夠 1024 台重新分配位元（犧牲年限或 seq）；或機器 id 拆「資料中心 5 + worker 5」
時間年限41 bit ≈ 69 年用上線時間當 epoch；到期前升級佈局
時鐘回撥 → 重複/亂序回撥偵測（等待 / 拒絕）；可靠 NTP；禁用 step 大跳變
機器 id 唯一性設定重複 → 撞號ZooKeeper/etcd 集中分配 + 心跳回收
嚴格遞增 vs 趨勢遞增跨機 ID 非全序接受「趨勢遞增」即可；要全序得回到中心發號（犧牲擴展）

8. 取捨與面試重點

方案唯一性排序性協調 / 依賴適用
Snowflake強（機器id+時間+seq）趨勢遞增僅機器 id 一次性分配高吞吐、要可排序數字主鍵
UUIDv4強（隨機）無序零協調、無狀態不在乎排序、純去重
UUIDv7強時間有序（高位含 ms 時間）零協調要有序又不想管機器 id；128-bit 較長
DB ticket server強（單一序列）嚴格遞增中心化、單點量不大、要絕對全序
號段模式 (segment)強趨勢遞增定期向 DB 領一段（step）美團 Leaf；攤平 DB 往返，DB 短暫不可用也能撐
  • Snowflake vs UUIDv7：兩者都「時間在高位 → 可排序」。Snowflake 是 64-bit、更省、能反解機器來源，但要管機器 id 與時鐘；UUIDv7 是 128-bit、零協調、標準化，但較長。現代新系統很多直接選 UUIDv7。
  • Snowflake vs 號段模式：Snowflake 純本機、不碰 DB，但怕時鐘；號段(Leaf-segment)向 DB 批次領號、不怕時鐘，但依賴 DB 可用性、且重啟可能跳號。Leaf 也提供 snowflake 模式。
  • 常見追問：時鐘回撥怎麼辦？（§6）為何不用 DB 自增？（§1）機器 id 怎麼保證唯一且可回收？（ZooKeeper ephemeral）ID 會洩漏什麼？（產生速率、時間、機器數 → 可被推估業務量，敏感場景可加擾動或改用不可預測方案）。

9. PHP Demo 說明

本題 demo 著重Snowflake 的核心取號邏輯（也是面試真正考點），可用 PHP 內建伺服器實跑：

  • src/Snowflake.php　64-bit 位元組合 / 同毫秒 seq 遞增 / 序號溢位自旋 / 時鐘回撥（小幅等待、大幅丟例外）/ decode() 雙向還原。
  • public/index.php　路由：POST /api/id、POST /api/id/batch、GET /api/decode/{id}，及附 bit 佈局示意的測試頁。
  • data/snowflake_state.json　持久化的 {last_ts, seq}（自動建立、已 gitignore）。
# 啟動
cd demo
php -S localhost:8023 -t public
# 瀏覽器開 http://localhost:8023：產生 ID、批次看 seq 遞增、貼 ID 解碼
誠實聲明 真實 Snowflake 的 {lastTimestamp, sequence} 常駐單機行程記憶體、以 synchronized/CAS 推進，不落盤。本 demo 因 PHP 內建伺服器每個請求是獨立 process、記憶體不共享，故改用檔案 + flock 鎖把 state 持久化到 data/，以模擬單機記憶體中的臨界區、維持跨請求單調——這條檔案鎖路徑是教學替身，正式環境不該每次取號都讀寫檔案。nextIds(n) 批次只取一次鎖、在記憶體連續推進，才是最貼近真實 Snowflake 的路徑，也是測試頁能看到同毫秒 seq 連續遞增的原因。位元佈局、同毫秒遞增、時鐘回撥、雙向解碼等核心邏輯皆為真實可執行。純原生 PHP 8.4、零依賴，只用 json 擴充（無 mbstring / gmp / openssl / sqlite）。
