51 內容傳遞網路 CDN：把內容放到離使用者最近的地方

H 基礎設施 建議 40 分鐘 Design a CDN

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把靜態內容複製到分散全球的邊緣節點，讓使用者就近取得、降低延遲、減少回源」的問題。 抓一份檔案放伺服器很簡單；面試真正考的是 快取一致性（怎麼失效）、就近路由（怎麼導到最近邊緣）、與 回源洪峰（怎麼不被打爆）。

1. 需求釐清

功能性需求

  • 把靜態資產（圖片、JS、CSS、影片片段、檔案下載）放到離使用者近的邊緣節點（PoP）提供，降低延遲。
  • 邊緣命中（hit）直接回；未命中（miss）則回源（origin fetch）抓取，並依 Cache-Control 的 TTL 存進邊緣快取。
  • 支援快取失效（purge）：內容更新時主動清除各邊緣的舊版本。
  • 依使用者就近選擇邊緣（GeoDNS / anycast）。
  • 提供命中率、回源比等運營統計。

非功能性需求

  • 低延遲：邊緣就近，命中時 RTT 應遠小於回源（跨洲）。
  • 高命中率：命中率直接決定省了多少回源頻寬與延遲，是 CDN 的核心 KPI。
  • 高可用：單一邊緣或 origin 故障時要能容錯（其他邊緣 / 多 origin）。
  • 可擴展：邊緣節點數十到數千個，分布全球。
  • 一致性可協商：多數靜態內容容忍「短時間舊版」，靠 TTL + purge + 版本化 URL 控制。
面試開場必問 「快取的是靜態還是動態內容？」與「能容忍多久的陳舊（staleness）？」這兩問決定走 TTL 強快取還是即時失效、push 還是 pull。

2. 規模估算

指標假設推算
總請求100 億次/日~115k QPS（峰值 ×3 ≈ 350k）
命中率熱門內容集中目標 95%+ → 只有 5% 請求需回源
回源比= miss / 總請求5% → ~5.75k 回源 QPS（其餘由邊緣吸收）
頻寬節省每次回應平均 200KB95% 命中即 origin 出口頻寬降到 1/20，這就是 CDN 最大的省
邊緣容量熱資產 fit 進邊緣 SSD長尾冷內容靠 TTL 淘汰 / LRU，不必全存
關鍵洞察CDN 的價值幾乎全在命中率：命中率每提升一點，省下的回源頻寬與延遲都是放大效應。設計圍繞「把熱內容留在邊緣、把失效做對」。

3. API 設計

# 取得資產（讀，超高頻，關鍵路徑）——對使用者其實就是一般 HTTP GET
GET https://cdn.example.com/assets/app.js
→ 200 + 內容
  Cache-Control: public, max-age=3600        # origin 告訴邊緣可快取多久
  X-Cache: HIT | MISS                         # 命中標頭（觀測用）
  Age: 142                                    # 此回應在邊緣已存在幾秒

# 失效（管理面）
POST /admin/purge   Body: { "asset": "/assets/app.js" }     # 單一資產
POST /admin/purge   Body: { "tag": "release-2024" }         # 依標籤批次

# 運營統計
GET /admin/stats → { "hit_rate": 0.96, "origin_ratio": 0.04, "by_region": {...} }
為何 Cache-Control 在 origin快取多久由內容擁有者（origin）決定最合理：圖片可 max-age=86400，常變的 JSON 設短 TTL，邊緣只是遵守這個指示。

4. 資料模型

# origin 的資產（真實來源）
asset
  path              VARCHAR  PRIMARY KEY   -- /assets/app.js
  body              BLOB
  cache_control_ttl INT                    -- max-age 秒

# 邊緣快取項（每個 PoP 一份，記憶體/SSD）
edge_cache_entry  (key = cache_key)
  cache_key   STRING        -- 通常 = method + host + path + 關鍵 query/標頭
  body        BLOB
  expires_at  EPOCH         -- = 存入時間 + TTL；過期即視為 miss

# 統計（每區累計）
stats  region -> { hit, miss }
cache key 是關鍵同一資產若 query string / 語系 / 裝置不同要不要當不同快取？cache key 設計決定命中率：把無關參數（如追蹤碼）正規化掉，避免快取碎裂（cache fragmentation）。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
                         ┌─────────────────────┐
                         │ DNS / anycast (黃)   │  依使用者位置導向最近邊緣
                         └──────────┬──────────┘
                                    ▼
 Client ──請求(綠)──▶ 就近邊緣 PoP ──(hit)──▶ 直接回 Client
                          │
                          └──(miss)──回源(藍)──▶ Origin 來源 ──▶ 依 Cache-Control TTL 存回邊緣

 Purge 控制器(黃) ──失效──▶ 所有邊緣 PoP（清掉舊版本，下次必 miss 回源拿新版）
  1. DNS / anycast：把使用者導到最近的邊緣 PoP（GeoDNS 看解析來源、anycast 看網路拓樸）。
  2. 邊緣 PoP：查本地快取 → 命中直接回；未命中回源抓，依 TTL 存下後再回。
  3. Origin：真實內容源（物件儲存 / 應用伺服器），只在 miss 時被打到。
  4. Purge 控制器：內容更新時，把失效指令傳播到所有邊緣清掉舊版本。

6. 核心元件深入

邊緣快取與 TTL

  • 邊緣存資產時記下 expires_at = now + TTL。請求進來：存在且未過期 → hit；否則 miss → 回源。
  • 過期不必立刻刪：可用 stale-while-revalidate（先回舊的、背景去更新），兼顧低延遲與新鮮度。

回源（origin fetch）

miss 時邊緣向 origin 抓取。回源是昂貴的（跨洲 RTT + origin 負載），因此整個設計都在「盡量別回源」：高命中率、合理 TTL、回源請求合併。

快取失效：purge vs 版本化 URL

做法原理取捨
purge（主動失效）內容變了 → 發指令清掉各邊緣的該 key。即時生效；但要把指令傳播到全球所有邊緣，有延遲與一致性挑戰。
版本化 URL
（cache busting）		檔名帶版本/雜湊，如 app.abc123.js；換內容＝換 URL。新 URL 天然就是 miss，不必失效舊的（讓它自然過期）；最乾淨，靜態資產首選。
短 TTL設很短的 max-age，靠頻繁過期換新。實作簡單；但回源變多、命中率下降，只適合本來就常變的內容。

cache key 設計

決定「什麼算同一份快取」。把無關 query 正規化（如丟掉 utm_*）、依需要才把語系/裝置納入 key，避免同一內容因參數差異被存成多份（快取碎裂、命中率掉）。

push CDN vs pull CDN

  • pull（lazy / on-demand）：邊緣第一次被請求才回源抓（本 demo 即此模型）。簡單、自動只快取被用到的內容；代價是第一個使用者承受 miss 延遲（冷啟動）。
  • push（pre-warm）：發布時主動把內容推到各邊緣。第一個使用者就 hit；代價是要管理推送、可能推了沒人用的內容。大型發布 / 影片首播常用 push 預熱。

anycast 就近

同一個 IP 在全球多處宣告（BGP anycast），使用者封包自動走到網路上最近的 PoP；搭配 GeoDNS 做粗粒度地理導向。這是「就近」的底層機制。

7. 擴展與瓶頸

瓶頸成因對策
冷啟動回源洪峰
cache stampede		熱資產過期/被 purge 的瞬間，大量並發請求同時 miss，一起回源打爆 origin。回源請求合併（request coalescing / single-flight）：同一 key 只放一個請求回源，其餘等它；加 stale-while-revalidate 先回舊的。
熱資產（hot object）單一爆紅資產把單一邊緣打爆。邊緣內多副本 / 一致性雜湊把資產分散到多台邊緣機；分層快取（邊緣 → 區域中間層 → origin）。
失效傳播purge 要送到全球所有邊緣，慢且可能漏。用版本化 URL 盡量免 purge；purge 走可靠的 pub/sub 廣播 + 對帳。
節點增減擾動邊緣機加減時，若用 hash % N 會讓快取大規模失效。一致性雜湊：只搬相鄰一段資產，擾動最小。
命中率偏低cache key 碎裂、TTL 太短、長尾內容多。正規化 cache key、合理 TTL、分層快取吸收長尾。
常見陷阱熱資產一過期，成千上萬請求同時 miss 同時回源（cache stampede），origin 瞬間被打爆。務必用回源合併讓同一 key 只回源一次。

8. 取捨與面試重點

  • TTL 長 vs 短：長 → 命中率高、回源少，但內容更新後陳舊久；短 → 新鮮但回源多、命中率掉。靜態資產配長 TTL + 版本化 URL，常變內容配短 TTL。
  • 強快取 vs 即時失效：強快取（長 TTL）省最多但難即時更新；要即時就靠 purge 或版本化 URL。多數選「長 TTL + 版本化 URL」——既強快取又能瞬間換版。
  • push vs pull：pull 簡單自動、但有冷啟動 miss；push 預熱免冷啟動、但要管理推送。日常用 pull，大型發布用 push 預熱。
  • 一致性：CDN 是最終一致——容忍各邊緣短時間版本不同。要強一致就別快取（或極短 TTL），但那就失去 CDN 的意義。
  • 常見追問：
    • 怎麼讓內容更新立刻生效？→ 版本化 URL（最乾淨）或 purge。
    • 熱資產過期瞬間怎麼不打爆 origin？→ 回源合併 + stale-while-revalidate。
    • 加減邊緣節點為何不能用 hash % N？→ 會大規模失效，改用一致性雜湊。
    • 怎麼導到最近邊緣？→ anycast + GeoDNS。

9. PHP Demo 說明

本題 demo 聚焦 CDN 的邊緣快取核心路徑（也是面試真正考點），可實際用 PHP 內建伺服器跑起來：

  • src/HashRing.php　一致性雜湊環（含虛擬節點）：把資產穩定對應到邊緣節點，加減節點時擾動最小。
  • src/Origin.php　來源伺服器：回源抓資產，每資產帶 Cache-Control TTL，並統計實際回源次數。
  • src/Cdn.php　核心：就近選邊緣 → 邊緣 hit/miss → miss 回源依 TTL 存 → purge 失效 → 命中率統計，全程 flock 原子讀-改-寫。
  • public/index.php　路由：GET /（測試頁）、GET /api/fetch、POST /api/purge、GET /api/stats，並附深色測試頁。
# 啟動
cd demo
php -S localhost:8051 -t public
# 瀏覽器開 http://localhost:8051
# 從某區域請求資產 → 第一次 MISS（回源 origin、依 TTL 存邊緣）
# 同區再請求 → HIT（邊緣命中）
# purge 該資產 → 再請求變 MISS（回源拿新版）
# 看 GET /api/stats：各邊緣命中率、總命中率、回源比、回源次數
誠實聲明 Demo 用記憶體模擬邊緣節點（JSON 檔 + flock 原子讀-改-寫），單機示意、非真正全球分散。但 hit/miss 判定、回源、TTL 過期、purge 失效、就近路由、一致性雜湊對應、命中率統計皆為真實可執行邏輯。要上線只需把 JSON 換成各 PoP 的本地快取（Varnish / nginx）、origin 換成物件儲存、就近改用 anycast/GeoDNS，演算法邏輯不變。
