1. 60 秒搞懂它在幹嘛

你在台灣打開一個網站，網站的圖片和影片其實放在美國的伺服器。如果每次都從美國拉過來， 訊號要漂洋過海、繞地球半圈，又慢又卡。

CDN（內容傳遞網路）做的事就是：事先把這些檔案複製一份，放到離你最近的機房（例如台北就有一台）。 你要圖片時，從台北那台直接拿，又快又不打擾遠在美國的「總部」。

🔑 一句話 CDN = 遍布全球的「便利商店」：總倉庫（origin）在很遠的地方，但每個社區都有一家便利商店（邊緣節點） 囤著熱賣商品。你要東西，走幾步到巷口便利商店就有，不用大老遠跑去總倉庫。

2. 為什麼這題的難點不是「存檔案」？

很多人第一次看這題會想：「不就是把檔案多複製幾份放各地嗎？」——這想法漏掉了兩個會把系統搞砸的真正難關。

📌 比喻：便利商店的兩個頭痛問題 開連鎖便利商店，真正難的不是「進貨」，而是：①架上的牛奶過期了怎麼辦？（總倉庫換了新配方，但各分店架上還是舊的） ②新品上市，全國分店同時擠爆總倉庫叫貨，總倉庫當場被叫貨電話打爆。

CDN 的第一個難關就是 ①：總部的內容更新了（圖片改了、程式換新版）， 但各地便利商店架上還是舊的——怎麼讓它們「換掉舊貨」？這叫快取失效（purge）。

第二個難關是 ②：某個熱門檔案剛好同時過期，那一瞬間全球的便利商店都發現「架上沒貨了」， 同時衝去總倉庫叫貨，把總倉庫打爆。這叫回源洪峰（cache stampede）。

💭 停一下：為什麼「每次都直接從美國總部拿」這麼簡單的做法不行？

看答案 因為太慢（訊號繞半個地球）、而且總部會被全世界的人同時打爆。CDN 的核心就是「就近放一份」—— 讓 99% 的請求在巷口便利商店就解決，只有極少數需要回總部。剩下的所有難題（怎麼換新貨、怎麼別擠爆總部）都是為了維持這個「就近」。

3. 把系統拆成 4 個關卡

別被「系統」嚇到。整件事其實就是闖 4 關，一關一關來：

1就近：找到離你最近的那家店

你要拿檔案，系統得先把你導到最近的便利商店（邊緣節點），而不是大老遠的總倉庫。

🗺️ 比喻：導航帶你去最近的分店 你打開地圖搜「便利商店」，它自動列出離你最近的那家，不會叫你開三小時去隔壁城市。 CDN 靠 anycast / GeoDNS 做同一件事：看你從哪裡上網，把你導到最近的邊緣節點。

2命中或回源：店裡有沒有貨？

到了最近的店，問一句：「架上有沒有我要的東西？」

  • ✅ 有（命中 hit）：直接拿走，超快！
  • ❌ 沒有（未命中 miss）：店員只好打電話跟總倉庫叫貨（回源），拿到後給你，順便放一份在架上，下一個人來就有了。
🥛 比喻：第一個買的人要等，後面的人就快了 你是今天第一個要這款牛奶的人 → 店裡沒貨 → 店員叫貨（你等一下，這叫 miss）。 貨到後放上架，後面來買的人直接拿（hit）。所以「第一個使用者」承擔了那次慢。

3保鮮期：架上的貨放多久要換？

架上的貨不能放到天荒地老。每樣東西都有一個保鮮期（TTL），這是總倉庫決定的： 圖片之類不太會變的，保鮮期很長（放一天）；常變的設定檔，保鮮期很短（放 10 秒就要重新叫貨）。

📅 比喻：架上的有效期限標籤 每件商品貼一張「到期日」。過了期，店員就把它當「沒貨」，下一個人來買時重新跟總倉庫叫一份新的。 保鮮期長 = 少叫貨（省事）但可能有點舊；保鮮期短 = 常保新鮮但常打擾總倉庫。

4換貨：總部出新版，怎麼把舊的全清掉？

總部突然改了內容（圖片換了、程式修了 bug）。但各地店裡架上還是舊的、而且保鮮期還沒到， 怎麼辦？總部發一道命令：「所有分店，把這個商品從架上拿掉！」這就是 purge（失效）。

📢 比喻：總部下令全國下架 就像食品被驗出問題，總部一聲令下「全國分店立刻下架」。下架後，下一個來買的人就會「沒貨 → 重新叫貨」， 拿到的自然是新版本。還有個更聰明的招：直接給新商品換個新名字（版本化），這樣根本不用下架舊的。
✅ 四關一句話 ① 就近導到最近的店 → ② 架上有就拿（hit）、沒有就叫貨（miss 回源）→ ③ 每件貨有保鮮期（TTL） → ④ 總部出新版就下架舊的（purge）。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「給了過期的舊貨」 總部明明換了新版，使用者卻一直拿到舊的（因為店裡架上還是舊貨、保鮮期沒到）。 所以才需要 purge（下架）+ 版本化命名 + 合理的保鮮期，讓「該換的時候真的換掉」。
😱 怕「擠爆總倉庫」（回源洪峰） 一個熱門檔案剛好同時過期，全球分店同時衝去叫貨，把總倉庫打到當機。 所以要做叫貨合併：同一個東西大家排隊，只派一個人去叫，叫回來大家一起用。
😱 怕「慢 / 沒就近」 如果導錯店（叫台灣人去美國拿）或命中率太低（架上老是沒貨、一直叫貨），CDN 就失去意義了。 所以要靠 anycast 就近 + 高命中率，讓絕大多數請求在巷口就解決。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定要準備多少料。系統設計也一樣：先估數字，才知道要準備多大。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每天多少次請求~100 億次（每秒十幾萬筆）不可能全打總倉庫 → 要靠就近的邊緣吸收
命中率（架上就有的比例）目標 95% 以上只剩 5% 要叫貨 → 總倉庫壓力小 20 倍
頻寬省多少命中 95% → 總倉庫出貨量降到 1/20這就是 CDN 最值錢的地方
💡 關鍵洞察：一切都看「命中率」 命中率每高一點，省下的時間和頻寬都是放大效應。所以整個系統的目標就是： 把熱門的東西留在架上（高命中）+ 該換的時候正確換掉（失效做對）。

💭 停一下：為什麼命中率從 90% 升到 95%，省的不只是「5%」？

看答案 因為要看的是「需要叫貨的比例」：90% 命中 = 10% 要叫貨；95% 命中 = 5% 要叫貨。 叫貨量從 10% 直接砍半到 5%！對總倉庫來說壓力少一半，這就是命中率的放大效應。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」。CDN 的點餐單超簡單，因為對使用者來說就是普通的拿檔案。 
① 拿檔案（最常用，就是一般 GET）
   GET https://cdn.example.com/app.js
   我回：檔案內容 + 一張標籤「X-Cache: HIT 或 MISS」（告訴你是不是架上就有）
        還有「Cache-Control: max-age=3600」（這東西的保鮮期是 3600 秒）
② 下架（管理用）
   POST /admin/purge   給我：{ 要下架哪個檔案 }
   → 命令所有分店把它從架上清掉，下次有人要就重新叫貨（拿到新版）
③ 看營運數字
   GET /admin/stats
   我回：{ 命中率, 回源比, 各區域分別多少 }

注意第 ① 張的「保鮮期是總倉庫貼的」——快取多久不是分店自己決定，是內容擁有者（origin）說了算。

9. 系統要記住哪些事？（資料模型）

📒 比喻：三本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題主要用三本：

🏭總倉庫清單（origin 有哪些貨）

每件貨一列：檔案路徑、內容、保鮮期 TTL。保鮮期就是貼在貨上的標籤，分店叫貨時會一起拿到。

🏪各分店的架上（邊緣快取）

每個分店一本，記架上現在有哪些貨、各自的「到期時間」：檔案、內容、expires_at（什麼時候過期）。 來客人時就翻這本：有且沒過期 → 命中；否則 → 叫貨。

📊記帳本（命中率統計）

每個區域記命中幾次、沒命中幾次，就能算出命中率和回源比，知道哪家分店表現好不好。

那「哪個貨放哪家店」誰決定？ 如果一個區域有好幾台機器，要決定某個檔案歸哪一台管，用的是一致性雜湊—— 一種聰明的分配法，好處是加減機器時，只有一小部分貨要搬家，不會整批大風吹（那會讓快取大規模失效）。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
熱門貨同時過期 → 全國同時叫貨打爆總倉庫叫貨合併：同一個東西只派一個人去叫；還能「先給舊的、背景偷偷換新」
某個爆紅檔案把單一分店擠爆分店裡放多份 + 用一致性雜湊把熱門貨分散到多台機器；中間再加一層「區域倉庫」
下架命令要送到全球所有分店，慢又可能漏盡量改用版本化命名（換新名字就不用下架）；真要下架就用可靠廣播 + 對帳
加減分店時快取大規模失效用一致性雜湊，只搬相鄰一小段
命中率太低（架上老是沒貨）把「同一個東西」正規化成同一個格子別存成好幾份、保鮮期設合理、加區域中間層吸長尾

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 保鮮期長 vs 短 長 → 很少叫貨（省、快），但內容更新後可能舊一陣子；短 → 常保新鮮，但一直叫貨、命中率掉。 常見折衷：保鮮期設很長 + 用版本化命名——既很少叫貨，又能瞬間換新版。
⚖️ 開賣前先鋪貨(push) vs 有人問才叫貨(pull) pull 簡單、自動只囤被買過的貨，缺點是「第一個人要等」；push 開賣前先把貨鋪到各分店，第一個人就有，但要管理鋪貨。 日常用 pull，大型發布 / 影片首播用 push 先鋪好。
⚖️ 要不要追求「全國同步」？ CDN 接受各分店短時間版本不太一樣（最終一致）。硬要全國分秒同步代價極高、也失去就近快取的意義。
💭 追問：內容更新了，要讓使用者「立刻」拿到新版，有哪兩招？ ① purge（下架）：命令各分店清掉舊的，下次叫貨拿新版；② 版本化命名：新內容換個新檔名， 新名字天然就「架上沒有 → 叫貨」，連下架都不用。後者最乾淨，靜態檔首選。
💭 追問：為什麼加減分店不能用「檔名 ÷ 分店數 取餘數」來分配？ 因為分店數一變，餘數全變，幾乎每個檔案的歸屬都改了 → 等於全部重新叫貨，瞬間打爆總倉庫。 用一致性雜湊就只動相鄰一小段，其他不變。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到 hit/miss/回源/purge：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8051 -t public
# 然後瀏覽器打開 http://localhost:8051
  1. 選一個資產 + 區域，按「請求」：第一次會是 MISS（架上沒貨 → 去總倉庫叫貨）。
  2. 同一區再請求一次：變成 HIT（架上已有，直接拿）。
  3. 對它按 purge（下架），再請求一次：又變回 MISS（被清掉了，重新叫貨）。
  4. 看下方各邊緣命中率和目前各分店架上有什麼（剩多少保鮮期）。
玩的時候想一下 換不同區域請求同一個資產，看是不是各區「第一次都要各自叫一次貨」（每家分店各自囤貨）——這就是「就近」的代價與好處。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 HashRing（決定哪個貨歸哪家店的分配器）→ Origin（總倉庫，負責被叫貨）→ Cdn（店長大腦：查架上、叫貨、下架、記帳）。

1查架上：有就拿、沒有就叫貨（Cdn 的 fetch）

這是全題的心臟。一個請求進來，先就近選店，再問「架上有沒有、過期沒」：

function 拿資產(資產, 區域) {
    店 = 依「區域」選最近的那家           // ← 就近路由
    架上 = 店的快取[資產]
    if (架上存在 且 還沒過期) {
        記一筆 命中(hit)
        return 直接從架上給             // ← HIT！超快
    }
    // 架上沒有 / 過期了 → 叫貨
    貨 = 總倉庫.叫貨(資產)              // ← 回源（昂貴、慢）
    保鮮期 = 貨帶的 Cache-Control TTL
    店的快取[資產] = { 內容:貨, 到期時間: 現在 + 保鮮期 }   // ← 放上架，記到期日
    記一筆 未命中(miss)
    return 從剛叫到的貨給
}

「存在且沒過期 → HIT；否則叫貨 → MISS」這短短幾行就是 CDN 的核心。注意叫貨後順手放上架並記下到期時間，下一個人就能 HIT。

2下架：總部一聲令下，全國清貨（Cdn 的 purge）

內容更新時，要把所有分店架上的舊版本清掉：

function 下架(資產) {
    for (每一家分店) {
        if (這家架上有 資產)
            從架上拿掉(資產)           // ← 清掉舊版本
    }
    // 之後任何人來要，都會「架上沒有 → 叫貨」拿到新版
}

下架不是去「換成新的」，而是直接拿掉。拿掉後下一次請求自然 MISS → 叫貨，拿到的就是總倉庫的最新版——簡單又可靠。

3分配器：哪個貨歸哪家店（HashRing）

如果一個區域有好幾台機器，要決定某個資產歸哪一台。最笨的做法是「編號 ÷ 機器數 取餘數」， 但機器一加減，餘數全變、貨全部要搬家。一致性雜湊用一個「環」來避免這件事：

function 找節點(資產) {
    h = 把資產雜湊成環上的一個座標
    // 在環上「順時針」找第一個機器點
    return 第一個 >= h 的機器（繞一圈回到開頭也算）
}

妙處在於：加一台機器，只有它「身後相鄰那一小段」的貨要搬，其他全不動。 比起「取餘數」會全部大風吹，這讓加減節點時的快取擾動降到最小。demo 還給每台機器在環上放很多個虛擬點，讓貨分得更平均。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「便利商店」解釋什麼是 CDN？ 總倉庫（origin）很遠，CDN 在每個社區開便利商店（邊緣節點）囤熱賣商品。你要東西走到巷口就有（就近、低延遲）， 不用大老遠跑去總倉庫，也不會大家一起把總倉庫擠爆。
Q2：hit 和 miss 差在哪？miss 時系統做了什麼？ hit＝架上就有，直接拿（快）；miss＝架上沒有，店員去總倉庫「叫貨」（回源，要等），拿到後順手放上架並記下保鮮期，下一個人就能 hit。
Q3：保鮮期（TTL）是誰決定的？長短各有什麼代價？ 由總倉庫（origin）透過 Cache-Control 決定。長 → 少叫貨（省、快）但可能舊；短 → 常保新鮮但常叫貨、命中率掉。常見折衷是「長保鮮期 + 版本化命名」。
Q4：內容更新後要讓大家拿到新版，有哪兩招？ ① purge（下架）：命令各分店清掉舊的，下次叫貨拿新版；② 版本化命名：新內容換新檔名，根本不用下架。後者最乾淨。
Q5：什麼是「回源洪峰（cache stampede）」？怎麼防？ 熱門檔案剛好同時過期，全球分店同時衝去叫貨，打爆總倉庫。解法是「叫貨合併」：同一個東西只派一個人去叫，叫回來大家一起用；還能先給舊的、背景偷偷更新。
Q6（工程）：為什麼加減邊緣節點不能用「取餘數」分配？該用什麼？ 取餘數時節點數一變，幾乎所有資產的歸屬都改 → 全部重新叫貨打爆 origin。改用一致性雜湊（環），加減節點只搬相鄰一小段。
Q7（工程）：命中率從 90% 升到 95%，為什麼很有感？ 因為要看的是「要叫貨的比例」：10% → 5%，叫貨量直接砍半，總倉庫壓力少一半。命中率的好處是放大的。

15. 30 秒回顧

CDN = 遍布全球的便利商店，把內容放到離你最近的地方。

4 個關卡：① 就近導到最近的店 → ② 架上有就拿（hit）、沒有就叫貨（miss 回源、順手放上架）→ ③ 每件貨有保鮮期（TTL） → ④ 總部出新版就下架舊的（purge）。

3 個怕：怕給過期舊貨（purge + 版本化 + 合理 TTL）、怕擠爆總倉庫（叫貨合併）、怕慢/沒就近（anycast + 高命中率）。

工程細節一句話：一切看命中率 → 用 hit/miss + TTL 管快取、用 purge 或版本化做失效、用一致性雜湊分配資產不亂、用叫貨合併擋回源洪峰。
