1. 60 秒搞懂它在幹嘛

想像你開一家蛋糕店，今天限量手作蛋糕只做 100 個，明星代言，網路一公布，瞬間 100 萬人同時點開頁面狂按「我要買」。

你的任務只有一個聽起來很簡單、做起來要命的事：不多不少，剛好賣出 100 個。 賣超過 → 收了錢卻沒貨，要道歉賠償；賣不出去 → 庫存浪費、客人罵你卡頓。而且這一切要在幾秒鐘內，在百萬人同時搶的混亂中完成。

🔑 一句話 秒殺 / 搶票 = 百萬人同時搶少量庫存，而你絕不能賣超過。難點不是「賣東西」，而是「那一瞬間的瘋狂並發下，數字算得分毫不差」。

2. 為什麼這題這麼難？（超賣的真相）

很多人第一反應是：「賣東西嘛，庫存減一不就好了？」我們來看一個你一定遇過的生活情境，就懂為什麼會出事——

🛒 兩個人搶最後一罐 超市貨架上只剩最後一罐奶粉。甲、乙兩人同時看到「還有 1 罐」，兩人都心想「太好了還有貨」， 同時伸手去拿——結果只有一罐，卻有兩個人以為自己買到了。

電腦裡發生的事一模一樣，這叫「先看、再減」的race（競爭）：

  • 請求甲：讀到庫存 = 1 → 心想「>0，可以賣」
  • 請求乙：幾乎同一瞬間也讀到庫存 = 1 → 也心想「>0，可以賣」
  • 兩個各自把庫存減 1 寫回去 → 結果賣了 2 個，庫存卻只有 1 個。這就是超賣。

所以秒殺真正考的不是「會不會減法」，而是：當幾百萬人在同一瞬間搶同一個數字，怎麼保證這個數字永遠不會被算錯、絕不賣超過？

💭 停一下：問題出在哪一步？是「讀」錯了，還是「減」錯了？

看答案 兩步本身都沒錯，錯在「讀」和「減」之間有個空檔，別人趁這空檔也讀到了同一個值。解法就是把「檢查還有貨 + 扣減」綁成一個不可被打斷的動作（像把這兩步鎖在同一個小房間裡，一次只放一個人進去），這叫「原子操作」。

3. 把系統拆成 3 個關卡

別被「百萬並發」嚇到。整件事其實就是闖 3 關，一關擋掉一種災難：

1削峰：先發「號碼牌」擋住大部分人

100 萬人衝進來，但只有 100 個蛋糕——讓 100 萬人全部擠到「扣庫存」那一步，系統一定垮。 所以第一關就要狠狠擋掉絕大多數人。

🎟️ 比喻：夜店門口的管制 熱門夜店裡面只能站 100 人，門口卻來了一萬人。保鏢不會讓一萬人全擠進去（裡面會踩死人）， 而是在門口發限量入場手環：拿到手環的才放進去，沒拿到的請在外面排隊或回家。

系統也一樣：先發限量的「入場令牌」（可以略多於庫存，留點緩衝），拿到令牌才有資格進到下單環節。 這樣真正打到「扣庫存」那一步的，就只剩一小撮人——99% 的人在門口就被攔下了，根本碰不到庫存。

2防超賣：扣庫存要「原子」，只有還有貨才扣

通過第一關的人來下單了。這是整題的命脈：扣庫存這個動作，絕對不能用前面說的「先讀再減」。

🚪 比喻：一次只放一個人進的小房間 把「檢查還有沒有貨 + 扣掉一個」這兩步，鎖進一個一次只准一個人進去的小房間。 你進去看到還有貨就拿走一個再出來，門才開給下一個人。這樣永遠不會有兩個人同時看到「還有最後一個」。

關鍵口訣是：「只有『還有貨』才扣，而且檢查跟扣是同一個不可打斷的動作」。 扣成功 → 恭喜搶到；數量已經是 0 → 直接回「賣完了」，不要再往下做。這一關保證：賣出總數永遠 ≤ 庫存。

3防重複：同一個人手滑點兩次，只能算一次

網路很卡、按鈕沒反應，人類的本能就是狂點。同一個人連點五下「購買」，或網路重送了三次—— 如果每次都當成新訂單，就會一個人扣掉好幾個庫存，超賣又出現了。

🎫 比喻：演唱會門票的票根存根 每筆下單請求都帶一張獨一無二的存根編號（叫「冪等鍵」）。系統收到時先看： 「這張存根我處理過了嗎？」處理過 → 直接把上次的結果再給你一次（不重扣）；沒處理過 → 才真的去扣庫存。

這樣「重送、重試、手滑連點」都天然安全：同一張存根編號，做幾次結果都一樣，這個性質叫「冪等」。 再加上「每人限購 N 張」，就能擋住黃牛一個人掃光全部庫存。

✅ 三關一句話 ① 門口發令牌削峰（擋掉 99% 的人）→ ② 原子條件扣減（只有還有貨才扣，絕不超賣）→ ③ 冪等鍵去重（手滑點兩次只算一次）。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「超賣」 賣出比庫存多 → 收了錢卻沒貨，要道歉、退款、賠信譽，真金白銀的損失。 所以「扣庫存」這一步寧可慢、寧可序列化（一個一個來），也絕不能算錯。正確性 > 速度。
😱 怕「被流量打垮」 100 萬人同時湧入，如果全讓他們打到資料庫，整個系統當機，連那 100 個能成交的也賣不出去。 所以要層層削峰：擋掉注定失敗的 99%，讓真正打到庫存的請求數量接近庫存量。盡早擋掉，別讓無效流量打到核心。
😱 怕「重複扣」 使用者狂點、網路重送，一不小心同一個人扣掉好幾個庫存，又變成超賣，還會讓人「付兩次錢」。 所以每筆下單都要能去重——同一張存根編號只算一次。重送必須安全。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
瞬間湧入多少人熱門演唱會 ~100 萬人搶 1 萬張票不能讓所有人都打到庫存，要先削峰
開賣那一秒多忙10 秒內集中，下單 QPS 飆到 10 萬+（查庫存更高）從 0 瞬間爆量，要扛得住尖峰
真正會成功幾筆= 庫存量，只有 1 萬筆成功，其餘全是「沒搶到」99% 是無效流量，盡早擋掉
核心瓶頸在哪大家都扣同一個庫存數字單一「熱點」，是整題最會塞的地方
🔻 關鍵洞察：倒三角漏斗 流量是個倒三角：上面進來百萬請求，但底部「真正成交」只有「庫存數」那麼一點點。 所以設計重點是一層一層削峰（CDN → 限流 → 令牌 → 佇列），讓真正打到「扣庫存」的請求數量，接近庫存量就好。

💭 停一下：99% 的人注定搶不到，那把這些請求「擋在門口」跟「讓他們打到資料庫才回沒搶到」，差別有多大？

看答案 差別是系統活不活得下去。讓 100 萬人全打到資料庫去算庫存，資料庫會被壓垮、連那 1 萬個能成交的也卡死。 在門口（令牌/快取）就回「賣完了/請排隊」幾乎不花成本，把昂貴的庫存扣減留給通過篩選的一小撮人。越早擋掉越省。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住三張「點餐單」，順序剛好對應三個關卡：

① 領入場令牌（削峰第一關，限量發）
   POST /sales/{id}/token
   給我：{ 使用者 }
   我回：200 { token }      ← 拿到才有資格下單
        429 「排隊中」       ← 令牌發完了，請稍後再試

為什麼先發令牌？這就是夜店門口的手環。99% 的人卡在這關，連下單 API 都碰不到，後面的庫存才不會被打爆。

② 下單扣庫存（核心關鍵路徑，帶存根編號）
   POST /sales/{id}/buy
   標頭：Idempotency-Key: <一串獨一無二的編號>
   給我：{ 使用者, token, 數量 }
   我回：200 { 訂單編號 }          ← 搶到了
        409 「賣完了」             ← 庫存歸零
        409 「超過每人限購」        ← 黃牛擋下
        200 (同一個訂單編號)        ← 冪等：重送回原訂單，不重扣

那個 Idempotency-Key 就是票根存根編號：同一張編號重送，系統認得它、直接回上次的結果，不會重複扣庫存。

③ 查還剩幾張（超高頻讀，走快取）
   GET /sales/{id}/stock → { 剩餘: 42 }

大家會狂刷「還剩幾張」，這種讀不必每次都去問真正的庫存，回一個快取的近似值、甚至只回「有/沒有」就夠，省下大量壓力。

9. 系統要記住哪些事？（資料模型）

📒 比喻：幾本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題的筆記本，剛好對應「要防的那幾件事」：

🎂庫存簿（最重要、最燙手）

記「這場活動 → 還剩幾個（remaining）」。這就是大家搶的那一個數字，所有扣減都打它。 真實系統會把它放在Redis（很快、又能做原子扣減），就是前面說的「一次只放一人進的小房間」。

為什麼它是瓶頸？ 因為所有人搶的是同一個數字，這個數字一次只能被一個人安全地改，天生會「排隊」。 這也是擴展那一節要解的最大難題。

🧾訂單簿 + 冪等簿（防重複扣）

訂單簿記成功的下單：訂單編號、活動、使用者、數量、時間。 冪等簿記「哪張存根編號 → 處理出了什麼結果」，用存根編號當鑰匙。

冪等簿在幹嘛？ 使用者重送時，系統先翻冪等簿：「這張編號我記過了！」就把上次結果再給一次，不會再扣一次庫存。 這在金流系統是通用招式——付款也是這樣防你被扣兩次錢的。

👤每人已購簿 + 令牌池（防黃牛、削峰）

每人已購簿記 (活動 + 使用者) → 已買幾個，用來檢查「有沒有超過每人限購」，擋黃牛掃貨。 令牌池記「這場活動 → 發出去幾張 / 上限幾張」，用來做門口的限量發手環。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
大家都扣同一個庫存數字，被迫排隊庫存分桶：1000 張拆成 10 桶各 100，請求分散到不同桶各自扣；某桶賣完再跨桶借
大家狂刷「還剩幾張」，讀爆了用快取回近似值，甚至只回「有/沒有」這個布林值
瞬間湧入要建一堆訂單，寫爆資料庫扣減成功後把「建訂單」丟進佇列（MQ）慢慢寫，先回使用者「搶到了」
99% 注定失敗的請求還在打後端令牌 / 排隊盡早擋；賣完後直接回快取的「sold_out」
下單服務一台不夠下單服務做成無狀態，要多快就多開幾台（水平擴展）
分桶的小代價 分桶能讓吞吐變大，但也帶來「A 桶賣完了、B 桶還有貨」的碎片問題，所以需要「跨桶調度」把零頭湊起來。 天下沒有白吃的午餐——這也是下一節「取捨」的一個例子。

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 正確 vs 快 扣庫存必須強一致（當下就要準，不能超賣），所以這一步寧可序列化、慢一點； 但「建訂單、計數、寄通知」可以最終一致（晚幾秒對上就好，走佇列）。訣竅是把「必須當下準」的同步點壓到最小，只留扣庫存。
⚖️ 同步下單 vs 非同步 同步＝當場告訴你成不成功，體驗好但壓力大；非同步＝扣減成功後先回「處理中」，訂單排隊慢慢建，吞吐高但體驗稍複雜。
⚖️ 零容忍超賣 vs 效能 完全不准超賣 → 扣減得序列化（慢）；如果業務能容忍少量超賣（事後退款補償），就能更激進地用快取/分桶衝效能。看的是業務願不願意。
💭 追問：有人「加入購物車、進結帳卻不付款」，一直占著庫存怎麼辦？ 進結帳時先預扣庫存並設一個倒數計時（TTL）；時間到還沒付款就自動釋放庫存回去，避免有人占著茅坑不買。
💭 追問：黃牛用機器人狂搶怎麼防？ 「每人限購 N 張」+ 每人已購計數（且這個檢查要在那個「小房間」臨界區內做，避免並發繞過）；再搭配實名、裝置指紋、風控規則擋機器人。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「防超賣 + 冪等 + 令牌」：

# 啟動（在這題的 demo 資料夾裡）
cd demo

# 然後瀏覽器打開 http://localhost:8028
  1. 先設定庫存（例如 100 張）。
  2. 按「一鍵模擬」：假裝 1000 人同時來搶 100 張。
  3. 觀察：售出數恰好是 100（不多不少、不超賣），其餘 900 人收到「沒搶到」。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那三關到底是哪幾段程式做的」。 好消息：核心邏輯都擠在同一個小檔（FlashSale）裡，剛好對應前面講的三個關卡。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 一個檔 = 一個小幫手（但裡面有四個機關） 這題不像別題拆成好幾個檔，而是把全部機關裝在同一個FlashSale 小幫手裡：領令牌（門口發手環）→ 下單（裡面再依序做冪等、限購、原子扣減）。每筆搶購就照這順序跑一遍。

0那個「一次只放一人進的小房間」（atomic）

先認識全題的地基。所有會改到共享數字（庫存、令牌、已購…）的動作，都包在這個小房間裡跑，一次只准一個請求進去。

function atomic(要做的事) {
    鎖門(LOCK_EX)              // ← 上門栓，後面的人在門外等
    讀出目前狀態                // 庫存、令牌、訂單… 全在這份狀態裡
    新狀態 = 要做的事(目前狀態)  // 在門內安心地「讀-改-寫」
    寫回新狀態
    開門(LOCK_UN)              // ← 開門，放下一個人進來
}

第 2 節說的「讀和減之間的空檔被人插隊」，就靠這個鎖門/開門堵死。進門的人從讀到寫整段都不會被打斷，下一個人進來時看到的一定是最新數字——這就是「原子」。下面三關全都在這個房間裡發生。

1削峰：門口發限量手環（issueToken）

第一關，發限量入場令牌。發完了就把人擋在門外（請排隊），後面的下單根本碰不到庫存。

function 領令牌(使用者) = atomic( 狀態 => {
    if (已發數量 >= 令牌上限)
        return 回「排隊中(queued)」    // ← 手環發完 → 擋在門外
    已發數量 = 已發數量 + 1
    token = 隨機產生一張
    令牌池[token] = 使用者            // 記住這張是誰的
    return { token }                 // 拿到才有資格下單
})

這就是夜店門口的手環。令牌上限通常設得略多於庫存（留點緩衝），但仍遠小於湧入人數，所以 99% 的人卡在這關。注意整段在 atomic 房間裡，所以「已發數量」並發下也不會被灌爆。

2下單三連檢查：冪等 → 限購 → 原子扣減（buy）

這是全題的心臟。一筆下單進到房間後，照順序過三道關卡，任何一關不過就退回：

function 下單(使用者, 存根編號, token) = atomic( 狀態 => {

    // (0) 冪等：這張存根編號處理過了嗎？
    if (冪等簿有這張編號)
        return 直接回上次的訂單結果      // ← 手滑點兩次／重送 → 不重扣

    // (1) 令牌驗證：手上這張手環有效、且是你的嗎？
    if (需要令牌 且 令牌無效)
        return 回「invalid_token」

    // (2) 每人限購：你已經買的 + 這次 會不會超過上限？
    已購 = 每人已購簿[使用者]
    if (已購 + 這次數量 > 每人限購)
        return 回「limit_exceeded」      // ← 擋黃牛掃貨

    // (3) 原子條件扣減（核心！）：只有「還有貨」才扣
    if (剩餘 < 這次數量)
        return 回「sold_out」            // ← 沒貨了，絕不硬扣 → 不超賣
    剩餘 = 剩餘 - 這次數量               // 扣庫存（這一步保證售出 ≤ 庫存）
    每人已購簿[使用者] = 已購 + 這次數量

    建立訂單(...)                       // 落單（真實系統丟 MQ 慢慢做）
    消耗掉這張令牌(token)               // 手環一次性，用過作廢
    冪等簿[存根編號] = 這次的訂單結果    // (6) 記起來，之後重送回這筆
    return { 訂單編號, 剩餘 }
})

看 (3) 這幾行：先檢查 剩餘 < 數量 就回 sold_out，否則才減一——而且整段都在 atomic 房間裡，所以「檢查還有貨 + 扣減」是同一個不可打斷的動作，這就是原子條件扣減，第 2 節的超賣 race 從根上被堵死。(0) 的冪等、(2) 的限購也都因為在同一個房間裡，並發無法繞過。

💭 停一下：為什麼「每人限購」這個檢查也一定要放進 atomic 房間裡，不能在外面先檢查？

看答案 因為如果在門外檢查，同一個人同時送兩筆下單，兩筆都會讀到「已購 = 0」、都覺得沒超限，於是雙雙放行 → 一個人買了兩個，限購形同虛設。只有把「讀已購 + 比上限 + 寫回」鎖進房間一次只放一人，限購才真的擋得住——這跟防超賣是同一招。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「兩個人搶最後一罐奶粉」解釋，超賣為什麼會發生？ 兩人「同時」讀到「還有 1 罐」，都以為自己買得到，各自扣 1，結果賣了 2 個但只有 1 個庫存。問題出在「讀」和「減」之間有空檔被人插隊。解法是把「檢查還有貨＋扣減」綁成不可打斷的原子操作。
Q2：為什麼要先發「入場令牌」？用夜店的比喻說說看。 像夜店門口發限量手環：裡面只站得下 100 人，卻來了一萬人，保鏢在門口就攔掉大部分。系統發限量令牌，拿到才能下單，99% 的人卡在門口，根本打不到庫存，系統才不會被流量壓垮。
Q3：同一個人狂點五下「購買」，為什麼不會扣五個庫存？ 因為每筆下單帶一個獨一無二的「存根編號（冪等鍵）」。系統收到先查「這張編號處理過嗎？」處理過就直接回上次結果、不重扣；沒處理過才真的去扣。同一張編號做幾次結果都一樣，這叫冪等。
Q4：這系統「最怕的三件事」是什麼？ 怕超賣（收錢沒貨、真金白銀損失）、怕被流量打垮（百萬人全打到 DB 會當機）、怕重複扣（手滑/重送讓一個人扣好幾個）。三個關卡（令牌削峰、原子扣減、冪等去重）就是各自避開這三個怕。
Q5（工程）：扣庫存為什麼不能用「先讀庫存 → 判斷>0 → 減 1 → 寫回」？正確做法是什麼？ 因為並發下兩個請求會同時讀到同一個值、都判斷可賣，造成超賣（read-modify-write race）。正確做法是把「判斷>0 + 扣減」包成單一原子操作：Redis 用 Lua 腳本，資料庫用 UPDATE ... WHERE remaining>0 看影響筆數，或單機用 flock 包住臨界區。
Q6（工程）：扣減成功後，為什麼「建立訂單」可以丟去佇列慢慢做？ 因為防超賣只靠「扣庫存」這一個強一致同步點就夠了。扣成功就代表名額確定是你的，建訂單、計數、寄通知這些可以最終一致——丟進 MQ 削峰、非同步落地，先回使用者「搶到了」，吞吐更高也不影響正確性。
Q7（工程）：所有人都扣同一個庫存數字會塞車，怎麼拓寬？有什麼代價？ 用「庫存分桶 / sharding」：把 1000 張拆成 10 桶各 100，請求分散到不同桶各自原子扣，吞吐變大。代價是會出現「某桶賣完、別桶還有」的碎片，需要跨桶調度把零頭湊起來借用。

15. 30 秒回顧

秒殺 / 搶票 = 百萬人同時搶少量庫存，而你絕不能賣超過。

3 個關卡：① 門口發令牌削峰（夜店手環，擋掉 99%）→ ② 原子條件扣減（一次只放一人進的小房間，只有還有貨才扣）→ ③ 冪等鍵去重（票根存根，手滑點兩次只算一次）+ 每人限購擋黃牛。

3 個怕：怕超賣（正確性壓倒速度）、怕被流量打垮（層層削峰、盡早擋）、怕重複扣（重送必須安全）。

工程細節一句話：流量是倒三角，削峰到接近庫存量 → 用「點餐單(API)」三關溝通 → 庫存/訂單+冪等/限購+令牌幾本筆記本 → 扣減強一致、其餘走佇列最終一致 → 熱點分桶拓寬。
