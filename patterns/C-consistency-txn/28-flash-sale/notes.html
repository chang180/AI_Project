㉘ 秒殺 / 搶票 Ticketmaster：防超賣與冪等

C 一致性/交易 建議 45 分鐘 Design a Flash Sale / Ticketing System

一句話定位 本質是「極端讀寫熱點下，對單一庫存 key 做零容忍超賣的並發扣減」。 平時的 CRUD 很簡單；秒殺真正考的是 瞬間百萬人搶同一批有限庫存 時，如何 不超賣、不重複扣、又扛得住流量——這正是一致性 × 高並發的交叉點。

1. 需求釐清

功能性需求

  • 限量商品/票券在指定開賣時間瞬間開放搶購（如 1000 張票、十萬人同時湧入）。
  • 防超賣：賣出總數絕不超過初始庫存（零容忍）。
  • 每人限購 N 張：防黃牛壟斷、保障公平。
  • 冪等下單：使用者重複點擊、網路重送、前端重試，不可重複扣庫存/重複建單。
  • 排隊 / 削峰：先發入場令牌控制進入下單環節的人數，擋住絕大多數流量。

非功能性需求

  • 正確性優先：寧可少賣、慢賣，也不能超賣（超賣＝真實金錢與信譽損失）。
  • 瞬時極高並發：QPS 在開賣那一秒從 0 飆到數十萬。
  • 低延遲回饋：搶到/沒搶到要快速明確告知，避免使用者一直重試放大流量。
  • 公平性：盡量先到先得；至少不能讓機器人輕易壟斷。
  • 可用性：下單服務掛一台不能讓庫存錯亂。
面試開場必問 「超賣的容忍度是 0 還是可少量？」決定走強一致同步扣減還是可超賣再補償。 Ticketmaster 這類實名票券通常零容忍 → 本筆記主軸是原子條件扣減 + 冪等 + 削峰。

2. 規模估算

指標假設推算
湧入人數熱門演唱會瞬間 100 萬人搶 1 萬張票
峰值 QPS開賣 10 秒內集中~10 萬+ 下單 QPS（讀庫存更高）
有效寫入= 庫存量最終只有 1 萬筆成功扣減，其餘全是「沒搶到」
熱點同一商品庫存單一 key 寫熱點，是整題的核心瓶頸
無效流量99% 的人搶不到關鍵是盡早擋掉注定失敗的請求，別打到 DB
關鍵洞察 流量是個倒三角：百萬請求進來，但有效寫入只有「庫存數」這麼多。 設計重點是層層削峰（CDN→限流→令牌→佇列），讓真正打到庫存扣減的請求數量接近庫存量。

3. API 設計

# 1) 領入場令牌（削峰第一關，限量發放）
POST /api/v1/sales/{id}/token
Body: { "user": "u123" }
→ 200 { "token": "tk_xxx" }          # 拿到才有資格下單
→ 429 { "error": "queued" }          # 令牌已發完 → 排隊/稍後再試

# 2) 下單扣庫存（帶冪等鍵，核心關鍵路徑）
POST /api/v1/sales/{id}/buy
Header: Idempotency-Key: <uuid>
Body: { "user": "u123", "token": "tk_xxx", "qty": 1 }
→ 200 { "order_id": "...", "status": "ok" }      # 搶到
→ 409 { "error": "sold_out" }                    # 賣完
→ 409 { "error": "limit_exceeded" }              # 超過每人限購
→ 200 (同 order_id)                              # 冪等：重送回原訂單，不重扣

# 3) 查庫存（高頻讀，走快取）
GET  /api/v1/sales/{id}/stock → { "remaining": 42 }

# 4) 一鍵壓測模擬（demo 用）
POST /api/v1/sales/{id}/simulate
Body: { "users": 1000, "stock": 100 } → 售出/拒絕統計 + 無超賣驗證

4. 資料模型

sales（商品/場次）
  sale_id      PK
  name
  stock_total  INT            -- 初始庫存
  per_user_limit INT          -- 每人限購
  starts_at    TIMESTAMP

stock（庫存熱 key；真實系統放 Redis）
  sale_id      -> remaining INT   -- 原子扣減的對象（DECR / Lua / UPDATE WHERE）

orders（成功落單）
  order_id     PK
  sale_id, user, qty, created_at

idempotency（冪等去重表）
  idem_key     PK              -- 同鍵重送直接回原結果
  sale_id, user, order_id, result

user_purchased（每人已購計數，做限購）
  (sale_id, user) -> qty

tokens（入場令牌池）
  sale_id -> issued / capacity  -- 限量發放，削峰
為何要冪等表 分散式下「扣了庫存但回應丟失」會讓客戶端重試 → 若無去重就重複扣。 用 Idempotency-Key 當唯一鍵：第一次成功就把 (key→order) 記下，之後同鍵一律回原訂單。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 百萬 User
    │ （CDN 擋靜態 + 前端限流 + 等待頁）
    ▼
 [入場令牌 / 排隊削峰]  ← 限量發 token，多數人被擋在這裡（黃）
    │ 通過者
    ▼
 下單服務（無狀態，可水平擴展）
    │ 帶 Idempotency-Key（冪等去重）
    ▼
 [原子條件扣減 remaining>0 才扣]（藍）──▶ 庫存熱 key（Redis，黃）
    │ 扣減成功
    ▼
 非同步落單：MQ ──▶ 訂單 DB（綠，最終一致）
  1. 削峰漏斗：CDN/限流 → 入場令牌 → 佇列，逐層把無效流量擋在外面。
  2. 原子扣減：唯一不可省的同步點。Redis DECR/Lua 或 DB UPDATE ... WHERE stock>0，保證不超賣。
  3. 非同步落單：扣減成功後把建單丟 MQ，DB 慢慢寫，回應使用者「搶到了」即可。

6. 核心元件深入

① 原子條件扣減（防超賣的命脈）

  • 錯誤做法：讀 remaining → if>0 → remaining-1 → 寫回。並發下兩個請求同時讀到 1，都判斷可賣 → 超賣。這是經典的 read-modify-write race。
  • Redis 方案：DECR 是原子的；但要防扣成負數，用 Lua 腳本把「判斷>0 + 扣減」包成單一原子操作。
  • DB 方案：UPDATE stock SET remaining=remaining-1 WHERE sale_id=? AND remaining>0，看 affected_rows：1=搶到，0=賣完。資料庫的行鎖保證原子，天然防超賣。
  • Demo 對應：單機用 flock 把「讀-改-寫」包成臨界區（CAS：只有 remaining>0 才扣），語意等同上面兩者。

② 冪等鍵去重

下單入口先查冪等表：命中 → 直接回原結果（不再扣）；未命中 → 扣減成功後把 (key→order) 寫入。 讓「重試/重送」天然安全，這也是金流系統的通用模式。

③ 排隊削峰與令牌桶

  • 入場令牌：限量發 N 個（可略大於庫存，留緩衝），拿到才准進下單環節。99% 的人卡在發令牌這關，根本打不到庫存。
  • 等待頁 / 排隊號：拿不到令牌的人進虛擬等待室，前端輪詢，避免一直重打後端。
  • 令牌桶限流：對下單 API 再做一層 rate limit，平滑尖峰。

④ 預扣與 TTL 釋放

進入結帳但未付款的庫存先預扣並設 TTL；逾時未付款自動釋放回庫存，避免有人占著不買。

⑤ 防黃牛限購

每人已購計數 + 每人限購 N；搭配實名、裝置指紋、風控規則擋機器人。限購檢查也要在臨界區內，避免並發繞過。

7. 擴展與瓶頸（單一熱 key）

瓶頸問題對策
單一庫存熱 key所有扣減打同一 key，序列化成瓶頸庫存分桶 sharding：1000 張拆 10 桶各 100，請求雜湊到桶，桶內原子扣；賣完跨桶借
讀庫存爆大家狂刷「還剩幾張」快取 + 近似值；甚至只回「有/無」布林
寫尖峰瞬間湧入打垮 DBMQ 削峰填谷，扣減成功後非同步落單
無效流量99% 注定失敗仍打後端令牌/排隊盡早擋；賣完後直接回快取的 sold_out
熱點實例Redis 單分片熱分桶後 key 分散到多分片；或用本地快取扛讀

8. 取捨與面試重點

  • 強一致扣減 vs 最終一致：庫存扣減必須強一致（防超賣）；落單、計數、通知可最終一致（走 MQ）。把同步點壓到最小（只有扣減）。
  • 同步下單 vs 非同步：同步＝即時告知但壓力大；非同步＝扣減成功後排隊建單、回「處理中」，吞吐高但體驗稍複雜。
  • 超賣零容忍 vs 效能：零容忍要序列化扣減（慢）；可容忍少量超賣則能更激進快取/分桶，事後補償退款。看業務。
  • 單 key vs 分桶：分桶提升吞吐但帶來「某桶賣完別桶還有」的碎片，需跨桶調度。
  • 常見追問：如何防超賣？（原子條件扣減）如何防重複下單？（冪等鍵）如何扛流量？（令牌+排隊+MQ）如何防黃牛？（限購+風控）。

9. PHP Demo 說明

本題 demo 著重於秒殺的真實核心邏輯（防超賣 + 冪等 + 令牌削峰），可實際用 PHP 內建伺服器跑起來：

  • src/FlashSale.php　核心：原子條件扣減（flock 包讀-改-寫，只有 remaining>0 才扣）、冪等鍵去重、每人限購、入場令牌池。
  • public/index.php　路由：領令牌 / 下單 / 查庫存 / 一鍵壓測模擬，並附互動測試頁。
  • data/　執行期狀態（庫存、訂單、冪等記錄、令牌、每人已購），已 gitignore。
# 啟動
cd demo
php -S localhost:8028 -t public
# 瀏覽器開 http://localhost:8028
# 設庫存 → 一鍵模擬 1000 人搶 100 張 → 看售出恰為 100（不超賣）
# 同 user 帶相同 idem_key 重送 → 回原訂單不重複扣
誠實聲明 Demo 用 flock 檔案鎖把「讀-改-寫」包成臨界區，模擬 Redis 原子操作 / DB 行鎖的角色—— 這是單機教學版（非真正的分散式 Redis Lua / 叢集）。但防超賣（CAS 條件扣減）、冪等去重、每人限購、令牌削峰的邏輯都是真實可執行的： 模擬高並發搶購後，售出數恰等於庫存、不多不少，同 user 同 idem_key 重送不重複扣。
