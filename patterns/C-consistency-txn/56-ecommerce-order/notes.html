56 電商下單系統：狀態機 × Saga 補償 × 冪等

C 一致性/交易 建議 50 分鐘 Design an E-commerce Order System

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把下單流程拆成幾道關卡、邊讀邊自我測驗。
一句話定位 本質是「一筆下單橫跨庫存、支付、出貨多個服務，要在沒有跨服務交易（2PC）的前提下，做到不超賣、不重複扣、不丟單」的綜合題。 平時的 CRUD 很簡單；真正考的是 付款付到一半失敗 時，如何讓已扣的庫存退回、已收的款退回，整筆訂單收斂到一致的終態。 這是本組三題的集大成：秒殺㉘的庫存原子預扣＋錢包⑰的扣款/退款＋工作流㊶的Saga 編排補償。

1. 需求釐清

功能性需求

  • 使用者從購物車送出下單，系統完成下單 → 預扣庫存 → 付款 → 確認/出貨的全流程。
  • 訂單狀態清楚可查：CREATED → PAID → SHIPPED → DONE，異常則 CANCELLED；狀態只能合法轉移，不可跳級或倒退。
  • 付款失敗/逾時要能回滾：已預扣的庫存釋放、已扣的款退回，不留中間態。
  • 冪等下單：使用者重複點擊、網路重送、前端重試，不可重複建單、不可重複扣庫存/扣款。
  • 防超賣：賣出總量絕不超過庫存（庫存原子預扣）。

非功能性需求

  • 正確性優先：錢與貨的數量在任何失敗情境下都要對得上（不超賣、不重扣、不漏單）。
  • 跨服務：庫存、支付、出貨多半是獨立服務，沒有單一資料庫交易能一次包住——這是整題的難點來源。
  • 最終一致可接受：下單瞬間各服務的更新可短暫不同步，但必須收斂到一致終態。
  • 高可用 / 可水平擴展：訂單量大、促銷有尖峰，下單服務需無狀態可擴展。
面試開場必問 「下單牽涉幾個服務？它們共用一個資料庫嗎？」如果共庫 → 本地交易就解決；如果是獨立服務各自有庫（真實電商）→ 沒有便宜的跨庫交易，必須走 Saga（最終一致 + 補償），這就是本題主軸。

2. 規模估算

指標假設推算
下單量大型電商~100 萬單/日 ≈ 平均 12 單/秒，促銷峰值 ×50 ≈ 600+ QPS
讀寫比查單/查物流頻繁讀 ≫ 寫（查訂單狀態、物流追蹤）→ 讀走快取/讀副本
每單跨服務呼叫庫存+支付+出貨一筆下單 ≈ 3~5 次跨服務動作，失敗就要補償
訂單儲存每筆 ~2KB，3 年100萬×365×3×2KB ≈ 2 TB，需分庫分表（按 user_id / 時間）
熱點爆款商品庫存單一 SKU 庫存熱 key，與秒殺同源，預扣需原子
關鍵洞察 下單量本身不算驚人，真正的工程重點是「正確性」而非「吞吐」：一筆橫跨多服務的寫操作，任何一步可能失敗， 系統要保證要嘛全部生效、要嘛全部回滾——這在沒有分散式交易的世界裡，靠 Saga 達成。

3. API 設計

# 1) 下單（核心，帶冪等鍵；觸發 saga 編排）
POST /api/v1/orders
Header: Idempotency-Key: <uuid>
Body: { "user": "u123", "items": [{ "sku": "sku-1", "qty": 1 }] }
→ 201 { "order_id": "ord_001", "state": "SHIPPED" }   # 全流程成功
→ 200 (同 order_id)                                    # 冪等：重送回原訂單
→ 409 { "state": "CANCELLED", "reason": "payment_declined" }  # 失敗已補償回滾

# 2) 查訂單（含狀態與 saga 軌跡）
GET /api/v1/orders/{id} → { "state": "PAID", "saga": [...] }

# 3) 推進狀態（後台/出貨系統觸發，狀態機把關）
POST /api/v1/orders/{id}/transition  Body: { "to": "DONE" }
→ 200 ok  /  409 illegal_transition（非法轉移被拒）

# 4) 取消訂單（觸發補償）
POST /api/v1/orders/{id}/cancel → 退款 + 釋放庫存 → CANCELLED

4. 資料模型

orders（訂單主表）
  order_id     PK
  user_id, amount
  state        ENUM(CREATED,PAID,SHIPPED,DONE,CANCELLED)  -- 狀態機的當前狀態
  created_at, updated_at

order_items（訂單明細）
  order_id, sku, qty, unit_price

inventory（庫存；爆款是熱 key，真實放 Redis）
  sku -> stock INT       -- 原子條件預扣對象（UPDATE ... WHERE stock>=qty）

payments（支付流水）
  payment_id PK, order_id, amount, status(CHARGED/REFUNDED)

idempotency（冪等鍵表）
  idem_key   PK           -- 同鍵重送直接回原訂單
  order_id, created_at

saga_log（編排狀態 / 步驟軌跡，崩潰後可續跑或補償）
  order_id, step, action(forward/compensate), result, ts
為何要 saga_log Saga 是長活交易：編排器可能在第 2 步後當機。把「每一步做了什麼」落地成 saga_log， 重啟後就能判斷「該往前續跑、還是該反向補償」。這是 Saga 可靠性的根基（與冪等表一起，讓重試安全）。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 Client 下單
    │ 帶 Idempotency-Key
    ▼
 訂單服務（狀態機 + 冪等）──建單 CREATED──▶ Saga 編排器
                                              │ 依序發起 3 步正向動作
        ┌─────────────────────────────────────┼─────────────────────┐
        ▼ A 預扣庫存                            ▼ B 扣款(→PAID)         ▼ C 確認出貨(→SHIPPED)
   庫存服務(原子預扣)                      支付服務(扣款)          出貨服務(確認)
        │                                     │                       │
        └──任一步失敗→反向補償(後進先出)：釋放庫存 ◀── 退款 ◀──────────┘
                                              ▼
                                     訂單 CANCELLED（最終一致）
 （服務間用訊息佇列做非同步事件驅動，補償亦由反向事件觸發）
  1. 訂單服務：守門員。建單給初始狀態 CREATED、檢查冪等鍵、所有狀態變更都過狀態機白名單。
  2. Saga 編排器：把「預扣庫存 → 扣款 → 確認」串成有序流程，每個正向動作都登記一個對應的補償動作。
  3. 反向補償：任一步失敗 → 對已成功的步驟後進先出（LIFO）逐一補償 → 訂單轉 CANCELLED，達成最終一致。
  4. 訊息佇列：服務解耦、削峰、非同步驅動；補償也透過反向事件傳遞（at-least-once + 冪等消費）。

6. 核心元件深入

① 訂單狀態機（合法轉移）

  • 把「誰能轉到誰」寫成一張白名單表：CREATED→{PAID,CANCELLED}、PAID→{SHIPPED,CANCELLED}、SHIPPED→{DONE,CANCELLED}，DONE/CANCELLED 為終態。
  • 任何不在表內的轉移（如 DONE→PAID、CREATED→SHIPPED 跳級）一律拒絕。把無數 if/else 收斂成單一真相來源，杜絕資料錯亂。
  • 面試價值：說得出「狀態機讓非法狀態無法被表示」，比逐個 if 判斷高一個層次。

② Saga 編排與補償（本題心臟）

  • 沒有跨服務的 2PC 時，用 Saga：把大交易拆成一串本地小交易，每個都配一個補償交易（語意回滾，而非資料庫 rollback）。
  • 編排式（Orchestration）：一個中央編排器發號施令、掌握全局狀態（本 demo 採用，最好懂、好除錯）。編舞式（Choreography）：各服務聽事件自主反應，無中央點但難追蹤。
  • 補償順序＝後進先出：A→B→C 成功到 B，C 失敗 → 先補償 B（退款）再補償 A（釋放庫存）。順序錯了會留下不一致。
  • 補償要冪等且盡量不失敗：退款/釋放庫存設計成可重試；補償若也失敗，進人工介入佇列告警。

③ 庫存原子預扣 vs 防超賣

  • 錯誤做法：讀 stock → if>=qty → 扣 → 寫回，並發下兩請求同時讀到足量 → 超賣（經典 read-modify-write race）。
  • 正解：UPDATE inventory SET stock=stock-qty WHERE sku=? AND stock>=qty，看 affected_rows；或 Redis Lua「判斷+扣」一次原子完成。
  • 預扣（reserve）而非直接扣：下單先預扣並設 TTL，付款成功才確認、逾時未付自動釋放——避免有人占著庫存不買。

④ 冪等下單

下單入口先查冪等表：命中 → 直接回原訂單（不再跑 saga）；未命中 → 跑完 saga 後把 (idem_key→order_id) 寫入。 讓「重試/重送/前端重複點擊」天然安全，這是金流類系統的通用模式。

⑤ 最終一致（Saga）vs 強一致（2PC）

  • 2PC：協調者鎖住所有參與者直到全體 commit，強一致但同步阻塞、可用性差、協調者單點，跨服務幾乎不用。
  • Saga：每步本地提交、失敗用補償回滾，無全域鎖、可用性高，但有「中間可見的不一致窗口」，需業務能容忍最終一致。
  • 電商下單普遍選 Saga：寧可短暫不一致 + 自動補償，也不要長時間鎖庫存/帳戶。

7. 擴展與瓶頸

瓶頸問題對策
爆款庫存熱點單一 SKU 預扣序列化成瓶頸庫存分桶 sharding（同秒殺㉘）+ 快取扛讀；賣完跨桶借
訂單量 / 容量單庫裝不下、寫壓力大訂單分庫分表（按 user_id / 時間範圍），查單走讀副本/快取
跨服務一致同步呼叫鏈長、易雪崩Saga + 訊息佇列非同步解耦；超時與重試 + 補償
編排器可靠性編排到一半當機saga_log 落地 + 重啟後續跑/補償；步驟冪等消費
補償也失敗退款/釋放庫存掛了重試 + 退避；最終進人工介入佇列告警

8. 取捨與面試重點

  • Saga 最終一致 vs 2PC 強一致：跨服務選 Saga（無全域鎖、高可用），代價是有不一致窗口與要寫補償邏輯。共庫才考慮本地交易/2PC。
  • 編排式 vs 編舞式 Saga：編排式有中央點、好追蹤好除錯（推薦面試講這個）；編舞式去中心、鬆耦合但難觀測、易形成隱性循環依賴。
  • 同步 vs 非同步：同步呼叫鏈即時但耦合、易雪崩；非同步（MQ）解耦削峰、吞吐高，但要處理亂序、重複、補償延遲。
  • 預扣 TTL 長短：太長 → 庫存被未付款訂單占住、降低可售；太短 → 慢付款的人剛付完庫存就被釋放、下單失敗。需依支付時長調。
  • 常見追問：付款成功但確認出貨失敗怎辦？（退款 + 釋放庫存 → CANCELLED，或標記人工處理）重複扣款怎防？（冪等鍵 + 支付端冪等）補償失敗怎辦？（重試 + 人工介入）。

9. PHP Demo 說明

本題 demo 著重於下單的真實核心邏輯（狀態機 + Saga 補償 + 冪等 + 原子預扣），可實際用 PHP 內建伺服器跑起來：

  • src/OrderStateMachine.php　訂單狀態機：合法轉移白名單，非法轉移一律拒絕。
  • src/OrderSaga.php　核心：Saga 編排（3 步正向動作 + 反向補償）、庫存原子預扣（flock 包讀-改-寫，防超賣）、冪等下單。
  • public/index.php　路由：下單看狀態流轉 / 注入失敗看補償回滾 / 冪等重送，並附互動測試頁。
  • data/　執行期狀態（庫存、錢包、訂單、冪等鍵），已 gitignore。
# 啟動
cd demo
php -S localhost:8056 -t public
# 瀏覽器開 http://localhost:8056
# ① 正常下單 → 看 CREATED→PAID→SHIPPED，庫存/錢包扣減
# ② 注入付款失敗 → saga 反向補償（退款、釋放庫存）→ 訂單 CANCELLED，庫存/錢包原樣回復
# ③ 同 idem_key 重送 → 回原訂單，不重複扣
誠實聲明 Demo 在單機用 flock 把整段 Saga 的「讀-改-寫」包成臨界區，模擬多服務的編排與最終一致結果—— 不是真正的多服務、多訊息佇列（這部分是示意）。但訂單狀態機合法轉移、Saga 反向補償（退款 + 釋放庫存）、冪等下單、庫存原子預扣防超賣的邏輯都是真實可執行的： 注入失敗後庫存與錢包回到下單前數值、訂單轉 CANCELLED；同 idem_key 重送回原訂單不重複扣；非法狀態轉移被拒。 要換成真正的分散式 Saga，把各步驟改成對庫存/支付服務的呼叫、補償改成反向補償事件即可，編排與狀態機邏輯不變。
