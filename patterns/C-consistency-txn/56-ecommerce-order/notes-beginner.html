1. 60 秒搞懂它在幹嘛

你在網路商店把一支耳機加入購物車，按下「結帳」。幾秒後畫面顯示「訂單成立、已付款、準備出貨」。 這背後其實發生了一連串事：先把這支耳機從庫存「留」起來 → 從你的帳戶「扣錢」→ 通知倉庫「出貨」。這就是電商下單系統。

聽起來很順對吧？難就難在：這三件事是三個不同的部門在做（庫存部、收銀台、倉庫），中間任何一步都可能突然出包—— 錢扣了但倉庫說沒貨、或庫存留了但信用卡被拒。系統必須保證：要嘛三件事全部成功，要嘛全部當作沒發生（該退的退、該放回的放回）， 絕不能「錢扣了貨沒出」這種爛攤子。

🔑 一句話 電商下單 = 一條「留貨 → 收錢 → 出貨」的流水線：每一站都可能卡住， 一卡住就得把前面做過的事一步步倒回去（退錢、把貨放回架上），讓帳面回到乾淨狀態。

2. 為什麼這題的難點不是「存訂單」？

很多人第一次看會想：「不就是把訂單存進資料庫嗎？」——這想法漏掉了一個會把系統搞爛的真正難關：這筆操作橫跨好幾個部門，而它們各管各的帳本。

📌 比喻：訂婚宴的連環預訂 你要辦婚宴，得同時搞定三件事：訂餐廳、訂樂團、訂禮車——三家不同的店。 你先付了餐廳訂金（成功），再去訂樂團（成功），最後訂禮車時對方說「那天滿了」（失敗）。 這時你不能假裝沒事——你得回頭把樂團退掉、把餐廳訂金要回來，否則白白損失。

下單就是這個連環預訂：留貨、收錢、出貨三家店，沒有一個「萬能總管」能一次幫你三家一起訂、一起取消。 （在電腦世界，這個「萬能總管」叫跨服務交易（2PC），但它又慢又脆弱，大公司幾乎不用。）

所以真正的解法是：一步一步做，每做一步就記住「萬一後面失敗，這步該怎麼倒回去」。 這套「做不成就反著把做過的倒回去」的招數，工程上叫 Saga（補償交易）——這就是這題的靈魂。

💭 停一下：為什麼不乾脆「全部成功才一起存」就好？

看答案 因為「留貨、收錢、出貨」是三個獨立系統各自的動作，沒辦法把它們塞進同一個「一起成功或一起失敗」的盒子裡（那需要昂貴又脆弱的跨服務交易）。 現實做法是一步步做、每步都留好「反悔藥」（補償動作）：後面卡住了，就把前面吃過的步驟一顆顆吐回去。這就是 Saga。

3. 把下單拆成 3 道關卡（一條流水線）

整件事就是一條流水線，闖 3 關。每一關都有一個「反悔藥」配對著（萬一後面失敗，吃這顆把它倒回去）：

1留貨：先把商品從架上「保留」起來

下單第一件事，是去庫存部說「幫我留 1 支耳機」。庫存要當場原子地確認「還有貨才留」—— 不能兩個人同時看到「還剩 1 支」就都留走（那就超賣了，賣出比實際多）。

🎬 比喻：電影院選位 你點了某個座位，系統馬上把它變灰，別人就點不到了。不能「兩個人同時點到同一個位子」。

👉 反悔藥：如果後面收錢失敗，就把這支耳機放回架上（釋放庫存）。

2收錢：從你的帳戶扣款

留好貨，去收銀台扣錢。扣成功，訂單狀態就從「剛建立」前進到「已付款」。

💳 比喻：刷卡 刷卡成功，店員才會把商品交給你。刷卡失敗（餘額不足、卡片被拒），這筆交易就得喊停。

👉 反悔藥：如果後面出貨失敗，就把這筆錢退回去（退款）。

3出貨：通知倉庫把貨寄出

錢收了，通知倉庫確認出貨，訂單前進到「已出貨」，最後送達變「完成」。

📦 比喻：倉庫貼單寄出 倉庫貼好寄件單、交給物流，這單就算搞定了。

👉 萬一這關失敗（倉庫系統當機、地址有誤），就要倒著吃前兩顆反悔藥：先退款、再把貨放回架上，最後把訂單標成「取消」。

✅ 三關一句話 ① 留貨（電影院選位，原子保留）→ ② 收錢（刷卡 → 已付款）→ ③ 出貨（→ 已出貨 → 完成）。 任何一關失敗，就「倒著」把做過的關一顆顆退回去（退款、放回貨）→ 訂單變「取消」。乾淨收場，這就是 Saga。

4. 設計時最怕的 3 件事

這個系統最怕三件事，每個設計都是為了避開某個「怕」：

😱 怕「錢扣了，貨沒出」 這是電商最致命的怕：收了錢卻沒給貨，客訴爆炸、信譽崩盤。 所以才需要 Saga 反悔藥（補償）：出貨失敗就自動退款 + 放回貨，絕不留「錢貨不對等」的爛帳。要嘛全成，要嘛乾淨倒回。
😱 怕「按一下變成下兩單 / 扣兩次錢」 你網路卡頓多點了一次結帳、或前端自動重試——絕不能因此建兩張單、扣兩次款。 所以下單要帶一把「下單身分證」（冪等鍵）：同一把身分證重送，系統直接回原來那張單，不重做。重送也只算一次。
😱 怕「超賣」 只剩 1 支耳機卻賣給了 5 個人。所以「留貨」必須原子：當場確認「真的還有才留」， 像電影院選位一點就變灰，別人搶不走。賣出去的，絕不超過架上有的。

6. 名詞白話翻譯

工程術語白話解釋
Saga（補償交易）一步步做、每步留好「反悔藥」；後面失敗就倒著把做過的退回去
補償（compensate）那顆反悔藥：退款、把貨放回架上（不是真的時光倒流，是做一個相反的動作）
編排器（orchestrator）Saga 總管：指揮三個部門照順序做、出事就下令倒回去
狀態機訂單只能照固定路線走：剛建立→已付款→已出貨→完成（不能跳級或倒退）
合法轉移「哪個狀態能變成哪個狀態」的白名單；不在名單的（如已完成→未付款）一律拒絕
原子預扣留貨時「確認有貨＋扣下來」一口氣完成，中間沒人能插隊（像點電影座位）
超賣賣出去的比架上實際有的還多（最該避免）
冪等 / 冪等鍵同一把「下單身分證」重送只算一次，不會下成兩單、扣兩次
最終一致中間可能短暫對不上，但系統保證「最後一定收斂到對的狀態」
2PC（兩階段提交）「萬能總管一次搞定三家」的強一致做法——但又慢又脆，跨部門幾乎不用
訊息佇列部門之間傳話的「待辦清單」，非同步、不互相卡住

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 先估「大概多少單、多少查詢」，才知道要準備多大的料。
估什麼大概數字所以呢？
每天多少單~100 萬單/日 ≈ 每秒 12 單平常不多，但促銷會暴衝幾十倍
讀 vs 寫查訂單/查物流遠多於下單查單走快取、讀副本，別都打主資料庫
一筆下單牽動幾個部門留貨+收錢+出貨 ≈ 3~5 個動作每多一步，失敗時就多一顆反悔藥要準備
訂單要存多久幾 TB（存好幾年）單一資料庫裝不下 → 要拆成很多份（分庫分表）
💡 關鍵洞察：這題拼的是「正確」，不是「快」 下單量其實不算嚇人。真正難的是「橫跨好幾個部門的一筆操作，怎麼保證要嘛全成、要嘛乾淨倒回」—— 這就是為什麼 Saga（反悔藥）和狀態機（固定路線）才是核心，而不是拼每秒幾萬筆。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統講好的「點餐單格式」：你給我哪些欄位、我回你什麼。 
① 下單（最重要，帶「下單身分證」防重複）
   POST /api/v1/orders
   給我：{ 哪個使用者, 買哪個商品幾個 }＋一個 Idempotency-Key（下單身分證）
   我回：201「訂單成立，狀態：已出貨」 或 「失敗已取消，原因：刷卡被拒」

同一張「下單身分證」重送 → 直接回原來那張單，不會下成兩單（這就是冪等）。

② 查訂單（看現在到哪一步了）
   GET /api/v1/orders/{訂單編號}
   我回：{ 狀態, 走過哪些步驟（saga 軌跡） }
③ 推進狀態（出貨系統用，狀態機把關）
   POST /api/v1/orders/{訂單編號}/transition  給我：{ 想轉到的狀態 }
   → 合法就轉；非法（如「已完成」想變回「未付款」）直接拒絕

9. 系統要記住哪些事？（資料模型）

📒 比喻：幾本筆記本 系統把資料記在「筆記本」（資料表）裡：

📋訂單簿（每張單現在到哪一步）

每張單一列：訂單編號、誰下的、金額、目前狀態（剛建立/已付款/已出貨/完成/取消）。那個「狀態」就是狀態機的記憶。

📦庫存簿（每個商品還剩幾個）

商品 → 剩餘數量。留貨時「確認有貨＋扣下來」要一口氣原子完成，才不會超賣。

🪪身分證簿（防重複下單）

記「下單身分證 → 對應的訂單編號」。同一張身分證再來 → 直接回原訂單，不重做。

📝Saga 流水帳（做到哪一步了）

記「這張單做過哪些步驟、哪些已補償」。萬一總管做到一半當機，重啟後看這本帳就知道「該繼續往前、還是該倒回去」。

10. 哪裡會塞車、怎麼拓寬？（擴展）

哪裡會塞車怎麼拓寬
爆款商品：大家搶留同一個庫存把那個熱門商品的庫存切成好幾桶分開扣（跟秒殺那題同招）
訂單太多，一個資料庫裝不下分庫分表（按使用者或時間切開）；查單走快取/副本
三個部門同步互打，一個慢全卡住改用訊息佇列非同步傳話，互相不卡；失敗了再補償
Saga 總管做到一半當機把「做到哪一步」寫進流水帳，重啟後接著做或倒回去
連「反悔藥」都失敗（退款也掛了）一直重試；真的不行就丟給人工處理並告警

11. 還有哪些「魚與熊掌」？（取捨）

⚖️ 一步步補償(Saga) vs 萬能總管一次搞定(2PC) Saga 不卡住、好擴展，但中間會有「短暫對不上」的空檔，而且要自己寫每顆反悔藥； 2PC 一次到位最乾淨，但又慢又脆、把所有部門鎖住等——跨部門幾乎都選 Saga。
⚖️ 有總管(編排) vs 各自聽消息行動(編舞) 有個中央總管指揮，好追蹤、好除錯（面試講這個）；各部門自己聽消息反應雖然鬆耦合，但出事很難查「現在到底卡在哪」。
⚖️ 留貨能留多久（預扣 TTL）？ 留太久 → 一堆沒付款的人占住庫存，別人買不到；留太短 → 慢慢付款的人剛付完，貨就被放回去了，白下單。要抓一個剛好的時間。
💭 追問：付款成功了，但通知倉庫出貨時失敗，系統怎麼辦？ 倒著吃反悔藥：先退款（把剛收的錢退回）、再釋放庫存（把留的貨放回架上），最後把訂單標成取消。錢和貨都回到下單前，乾淨收場。
💭 追問：為什麼訂單要用「狀態機」管，不直接改個欄位就好？ 因為訂單只能照固定路線走（剛建立→已付款→已出貨→完成）。用狀態機把「哪個狀態能變成哪個」寫成白名單，就能擋掉所有亂跳（如「已完成」又變回「未付款」），避免資料錯亂。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「狀態流轉 + Saga 補償」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8056 -t public
# 然後瀏覽器打開 http://localhost:8056
  1. 正常下單：按「下單」，看訂單一路走 剛建立 → 已付款 → 已出貨，庫存和錢包都減少了。
  2. 注入失敗：選「扣款失敗」或「確認失敗」下單，看系統自動倒回（退款、放回貨）→ 訂單變取消，庫存和錢包原樣回復。
  3. 冪等重送：用相同的 idem_key 再下一次，看它回原來那張單，沒有重複扣。
玩的時候想一下先「查全局狀態」記下庫存和錢包數字 → 注入一次失敗 → 再查一次，會發現數字完全沒變——這就是 Saga 補償的威力：失敗了，帳面乾乾淨淨像沒發生過。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看那幾關是哪幾段程式做的。好消息：核心只有兩個小檔。不用會 PHP，我把程式翻成白話、只留關鍵那幾行。

🔧 兩個檔 = 兩個小幫手 OrderStateMachine（管訂單「能走哪條路線」的交通號誌）＋ OrderSaga（那位指揮三步、出事就倒回去的總管）。

1交通號誌：訂單只能照路線走（OrderStateMachine）

先把「哪個狀態能變成哪個」寫成一張白名單，不在名單的一律拒絕：

合法路線白名單 = {
    剛建立 → [已付款, 取消],
    已付款 → [已出貨, 取消],
    已出貨 → [完成,   取消],
    完成   → [],     // 終點，不能再動
    取消   → [],     // 終點，不能再動
}
function 能不能轉(從, 到) {
    return 到 在 白名單[從] 裡面;   // ← 一句話判斷合不合法
}

就這張表，把「已完成又變回未付款」這種亂跳全部擋掉。所有狀態變更都得先問它一句「能不能轉」。

2留貨：確認有貨才扣（防超賣）（OrderSaga）

留貨這步要原子地「先確認、再扣」，中間不能被插隊：

剩貨 = 庫存[商品]
if (剩貨 < 想買的數量)
    丟出失敗("沒貨了");        // ← 沒貨就停，觸發後面倒回
庫存[商品] = 剩貨 - 想買的數量  // 真的有貨才扣（預扣）

demo 用 flock（檔案鎖）把這段「讀-改-寫」框起來，保證同時只有一個人能扣 → 絕不超賣。真實系統換成資料庫的 UPDATE ... WHERE 剩貨>=數量 或 Redis，道理一模一樣。

3總管編排：三步順著做，出事倒著退（OrderSaga）

這是全題的心臟。總管照順序做三步，每做成一步就記在「已完成堆疊」；任何一步失敗，就倒著把堆疊裡的步驟一一吃反悔藥：

已完成 = [];
try {
    預扣庫存();        已完成.push("A");   // 第 1 步成功
    扣款();  訂單轉「已付款」;  已完成.push("B");   // 第 2 步成功
    確認出貨(); 訂單轉「已出貨」; 已完成.push("C");  // 第 3 步成功
    // 三步全成 → 完工！
} catch (某步失敗) {
    for (步驟 in 反過來(已完成)) {        // ← 後進先出，倒著退
        if (步驟=="B") 退款();           // 反悔藥：把錢退回
        if (步驟=="A") 釋放庫存();        // 反悔藥：把貨放回架上
    }
    訂單轉「取消」;                       // 收斂到乾淨的終點
}

前面講的「倒著吃反悔藥」整套就是這段 try / catch。注意 反過來(已完成) 那行——退的順序要和做的順序相反（先退款再放貨），這就是「後進先出補償」。

💭 停一下：為什麼補償要「倒著」做（先退款，再放回貨），不能隨便順序？

看答案 因為後面的步驟可能依賴前面的步驟。倒著退（後做的先退）能保證每顆反悔藥執行時，它依賴的東西都還在、狀態才一致。這跟脫衣服要反著穿回去的順序是同個道理。

4冪等：同一張「下單身分證」只算一次（OrderSaga）

下單的第一道關卡，先檢查「這張身分證是不是來過了」：

if (這個 idem_key 之前建過單)
    return 直接回那張原訂單;    // ← 冪等：不重做、不重複扣

... 正常跑完 saga 後 ...
記下(idem_key → 這張新訂單);     // 下次同 key 重送就命中上面

就算你手滑連按、或網路重試送了兩三次，系統一看「這張身分證來過」就直接回原訂單，絕不會下成兩單、扣兩次錢。

14. 考考自己

Q1：用「婚宴連環預訂」解釋，這題的核心難關是什麼？ 一筆下單橫跨三個獨立部門（留貨/收錢/出貨），沒有萬能總管能一次幫你全訂或全退。所以要一步步做、每步留好反悔藥，後面失敗就倒著把做過的退回去——這就是 Saga（補償交易）。
Q2：付款成功但出貨失敗，系統做哪些事？順序為何？ 倒著吃反悔藥：先退款（補償第 2 步），再釋放庫存（補償第 1 步），最後把訂單標成取消。順序要和執行順序相反（後進先出），錢和貨都回到下單前。
Q3：為什麼訂單要用「狀態機」？防的是什麼？ 防止狀態亂跳。把「哪個狀態能變成哪個」寫成白名單，擋掉跳級（剛建立直接出貨）和倒退（已完成又變未付款），讓非法狀態根本無法出現。
Q4：「冪等」在這題防的是什麼？怎麼做？ 防「手滑連按 / 網路重試 → 下成兩單、扣兩次」。下單帶一把 idem_key（下單身分證），系統記下 key→訂單；同 key 重送直接回原訂單，不重做。
Q5（工程）：為什麼跨服務選 Saga 而不是 2PC（萬能總管）？ 2PC 要把所有部門鎖住一起 commit，又慢又脆、可用性差、協調者是單點。Saga 每步本地提交、失敗用補償回滾，不鎖全域、高可用，代價是有「短暫不一致窗口」和要自己寫補償——電商普遍接受最終一致，所以選 Saga。
Q6（工程）：留貨為什麼要「原子」？不原子會怎樣？ 不原子（先讀剩貨、再判斷、再扣）時，兩個人可能同時讀到「還剩 1」都判斷可賣 → 超賣。要用原子的「確認＋扣」一步完成（DB 的 UPDATE...WHERE stock>=qty 或 Redis），保證賣出不超過實際庫存。

15. 30 秒回顧

電商下單 = 一條「留貨 → 收錢 → 出貨」的流水線，每步都備好反悔藥。

3 道關卡：① 留貨（原子，像選電影座位，防超賣）→ ② 收錢（→已付款）→ ③ 出貨（→已出貨→完成）。任一關失敗就倒著吃反悔藥（退款、放回貨）→ 訂單取消。

3 個怕：怕「錢扣了貨沒出」（Saga 補償最致命）、怕「按一下變兩單/扣兩次」（冪等鍵）、怕「超賣」（原子預扣）。

工程細節一句話：跨服務沒有便宜的「一起成功一起失敗」，所以用 Saga（一步步做 + 後進先出補償）達成最終一致；用狀態機把訂單鎖在固定路線；用冪等鍵擋重複；用原子預扣防超賣。
