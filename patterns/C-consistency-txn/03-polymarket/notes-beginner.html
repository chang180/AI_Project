1. 60 秒搞懂它在幹嘛

想像你經營一個線上菜市場，但賣的不是青菜，而是「對未來事件的賭注」。 比如「2028 誰當選總統？」這個攤位上，有人喊「我用 0.62 元買一張『會當選』的票」， 另一個人喊「我用 0.62 元賣一張」——你的工作就是把喊價對得上的買家賣家湊成一筆交易，而且一塊錢都不能算錯。

Polymarket（預測市場）就是在做這件事。它的難，難在兩點：怎麼又快又公平地把買賣單配對，以及錢絕對不能算錯（不能讓人花同一筆錢買兩次、餘額不能變負數）。

🔑 一句話 預測市場 = 一個「喊價配對」的交易所 + 一本「絕不出錯」的帳本：把買賣單依規則撮合成交，同時保證每一塊錢都對得上、查得到。

2. 為什麼「機率」可以拿來買賣？

很多人第一個疑問是：「未來還沒發生，怎麼能買賣？」關鍵在一個你其實懂的生活經驗——下注。

🎫 比喻：一張「兌獎券」 想像一張票，上面寫「如果 A 當選，這張票兌換 1 元；如果沒當選，這張票變廢紙」。 如果大家覺得 A 有 62% 機會當選，這張票現在大概值多少？——差不多 0.62 元。 機率越高的事，它的票就越值錢，價格自然往 1 元靠；越沒希望，價格往 0 元掉。

所以這裡的「價格」其實就是大家心中「這件事會發生的機率」。買 0.62 的人賭它會發生（將來兌 1 元、賺 0.38）；賣的人賭它不會。 每個攤位有兩種票：YES（會發生）和 NO（不會發生），事件結束後由一個公正的「裁判」（叫預言機 oracle）宣布結果，贏的那種票每張兌 1 元，輸的歸零。

💭 停一下：如果一張「YES 票」現在賣 0.9 元，這代表市場覺得這件事很可能還是很不可能發生？

看答案 很可能發生。價格越接近 1 元，代表市場認為機率越高（90%）。因為事件成真時這張票會兌 1 元，所以「願意付 0.9 元買」就等於「賭它有九成把握成真」。價格 = 隱含機率，這是預測市場最核心的直覺。

3. 把系統拆成 4 個關卡

別被「交易所」嚇到。整件事其實就是闖 4 關，一關一關來：

1下單 + 先扣住錢（凍結）

你喊「用 0.62 元買 100 張」，系統第一件事不是馬上成交，而是先把這筆錢扣住、卡在一邊（0.62 × 100 = 62 元）。

💳 比喻：刷卡的「預授權」 你去飯店 check-in 或租車，店家會先在你信用卡上「預授權」一筆錢卡住，但還沒真的請款。 確定你有這筆錢、跑不掉了，才放心讓你住、讓你開車。系統下單也一樣：先凍結、確定你付得起，才讓這張單上場。

👉 為什麼要先凍結？因為如果「掛單時有錢、輪到成交時錢卻被花掉了」，帳就爆了。先卡住錢，就杜絕這種情況——凍結不到，就不准下單。

2撮合：把買家賣家湊成一對

單子進場後，要去找對手。問題是：同時有一堆人想買、一堆人想賣，誰先成交、用誰的價？

🐟 比喻：菜市場喊價配對 菜市場裡，出價最高的買家最先買到魚（誰肯出錢誰先得）；開價最低的賣家最先賣掉（誰肯便宜賣誰先成交）。 如果兩個人出一樣的價，那就先喊的先成交（排隊先到先贏）。

這套規則叫 「價格優先、時間優先」：先比價（買家比誰出得高、賣家比誰開得低），同價再比誰先掛單。 當最高的買價 ≥ 最低的賣價，就配對成交。大單也可以吃掉好幾張小單，沒吃完的部分繼續留在攤位上等（叫「部分成交」）。

⚠️ 關鍵細節：這件事不能讓很多人同時亂搶，否則會出現「兩個買家同時吃掉同一張賣單」的鬼打牆。所以同一個攤位的所有單子，要排成一條隊、一筆一筆處理（後面 Part 2 會講為什麼這樣反而更快又更安全）。

3過帳：把錢和票真正交換（一次到位）

配對成交後，要真的銀貨兩訖：買方付錢拿到票、賣方交票拿到錢。這一步絕對不能做一半。

🤝 比喻：一手交錢一手交貨 二手交易最怕「給了錢沒拿到貨」或「給了貨沒收到錢」。所以要同時發生、要嘛全部成功、要嘛全部取消—— 不可能「買方錢扣了，票卻沒給」。系統把這一整組動作綁成一個不可分割的交易（術語叫 ACID 的原子性）。

而且每扣一筆錢都用「有才扣得動」的方式做（餘額夠才扣、不夠就整筆拒絕），這樣餘額永遠不會變成負數，也防止有人拿同一筆錢買兩次（叫「防雙花」）。

4結算：事件揭曉，發錢

事件結束（選舉開票了），由公正裁判預言機宣布結果。贏的那種票每張兌 1 元，輸的歸零。系統按大家手上的持倉，把錢發給贏家。

🎰 比喻：開獎就像彩券開獎：對中的票去櫃台換錢，沒中的就作廢。交易期間的買賣，賭的就是「我這張票最後能不能兌到 1 元」。
✅ 四關一句話 ① 下單先凍結錢（刷卡預授權）→ ② 喊價配對撮合（菜市場）→ ③ 一手交錢一手交貨過帳（絕不做一半）→ ④ 事件揭曉結算發錢。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「錢算錯」 這是錢的事，正確性壓倒一切。不能超額下單、不能讓人花同一筆錢兩次、餘額不能變負數，事件結束後總錢數必須對得上（守恆）。 所以資金一定要強一致——你剛存的 100 元，下一秒查一定看得到，絕不會「等一下才同步」。寧可慢一點，不可算錯。
😱 怕「同時搶單搶到打架」 很多人同時對同一個攤位下單，如果讓系統並行亂處理，會出現「兩個人吃掉同一張賣單」「同一筆錢被花兩次」的競態。 所以同一個市場的單子排成一條隊、一筆一筆認真處理（單執行緒序列化）——慢工出細活，反而最不會出錯。
😱 怕「系統一掛，帳就消失」 撮合在記憶體裡跑（很快），但如果機器突然斷電，記憶體一清空，剛剛成交的單豈不是全沒了？ 所以每一個動作（下單、成交、過帳）都先寫成一筆「永不塗改的流水帳」（事件日誌）。 機器掛了，把流水帳重新播一遍就能還原現場，事後查帳、對帳、處理糾紛也全靠它。

7. 要準備多大？（規模估算）

🍱 比喻：開店前先估生意量 開店前，你會先估「一天大概做幾筆生意、尖峰時段多擠」，才決定請幾個店員、櫃台開幾個。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
一天下多少單（寫）熱門事件 ~500 萬單/日，平均每秒約 58 筆，尖峰 ×10 約 580 筆寫入量不算誇張，但不能出錯才是重點
看行情的人（讀）看的人遠多於下單的人，讀:寫可達 100:1看行情要走推播 + 快取，別每次都查帳本
一個攤位撮合多快純記憶體、一條隊伍可達每秒~10 萬筆單一攤位夠快了，瓶頸不在這
同時開幾個攤位同時數千個活躍市場不同攤位分給不同機器各管各的
💡 關鍵洞察：別想「加人一起搶同一個收銀台」 一般系統「變慢就多加機器並行」。但撮合不能這樣——多人同時動同一本帳會搶到打架、算錯帳。 正解是「一個攤位配一個專屬收銀台、一條隊伍」，要擴大就多開幾個攤位（不同市場），攤位之間互不干擾。

💭 停一下：行情（最新價格）每秒有上萬人來看，為什麼不直接讓他們去查那本「真相帳本」？

看答案 因為那本帳本是錢的真相、要強一致，被上萬人狂查會拖垮它、還可能拖慢真正重要的下單與過帳。 所以把「看行情」和「動錢」分開：行情走推播 + 快取（看到的可能晚個零點幾秒沒關係），帳本只專心處理會動到錢的事。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」：

① 下一張單（最重要，撮合的入口）
   POST /api/v1/orders
   給我：{ 哪個市場, YES還是NO, 買還是賣, 價格0~1, 數量, 我的下單編號 }
   我回：201「收到，已成交 40 張，還有 60 張掛著等」

那個「我的下單編號（client_order_id）」很關鍵：如果網路不穩、你按了兩次或自動重送，系統靠這個編號認出「這是同一張單」，只算一次，不會幫你重複下單、重複凍結錢。這叫冪等——同個編號做幾次，結果都一樣。

② 撤單（把還沒成交的單收回，退還凍結的錢/票）
   DELETE /api/v1/orders/{order_id}
③ 看行情（超多人看 → 走推播 + 快取，別狂查帳本）
   GET /api/v1/markets/{id}/book   → 現在簿上的買價/賣價、各有多少量
   WS  /ws/markets/{id}            → 一有變動就「即時推」給你（不用一直問）

為什麼看行情用「推」（WebSocket）而不是「拉」（一直重新整理）？因為上萬人每秒狂問「現在價格多少」會把系統問爆；改成「有變動我才主動推給你」，省事又即時。就像群組有新訊息會跳通知，你不用每秒打開看一次。

④ 查帳戶 / 結算
   GET  /api/v1/accounts/{id}/balance     → 可用現金、凍結中、各持倉
   POST /api/v1/markets/{id}/resolve      → 裁判宣布結果，開始發錢

9. 系統要記住哪些事？（資料模型）

📒 比喻：兩種筆記本——「可塗改的」和「不可塗改的」 系統的資料分兩大類：會一直變動的「現況本」（餘額、掛著的單），和永不塗改的「流水帳本」（成交紀錄、事件日誌）。 分開的原因：錢的現況要能快速更新且強一致；歷史紀錄則要永久保存、能查帳。

💰帳戶本（誰有多少錢，可塗改）

每個人一列：可用現金、凍結中的現金、持有哪些 YES/NO 票。 這是「錢的真相」，要強一致。每次扣錢都用「有才扣得動」的條件更新——餘額不夠就整筆拒絕，所以永遠不會變負數。

為什麼要分「可用」和「凍結」兩欄？ 因為下單時錢還沒真的花掉，只是被「卡住」（預授權）。分兩欄，你才知道「哪些錢還能用、哪些已經被掛單佔住」，避免重複拿同一筆錢去下別的單。

📋掛單本 + 攤位本（現在簿上有什麼，可塗改）

掛單本記每張還沒成交完的單（價格、數量、成交了多少、狀態）；攤位本記每個市場（哪個事件、現在開市還是已結算）。這些是當下的現況，會隨成交一直變。

📜成交本 + 事件日誌（歷史，永不塗改）

每筆成交、每個關鍵動作都只增不改地記一筆（append-only）。 這本才是真正的真相來源：撮合引擎在記憶體裡跑得飛快，但萬一崩潰，就把這本流水帳從頭播一遍，現場立刻還原。查帳、對帳、處理糾紛也全靠它。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
一個攤位的撮合速度到頂它已經很快了（純記憶體、一條隊伍）；可批次寫日誌再加速
一台機器塞不下幾千個攤位不同市場分到不同機器各管各的（分片），彼此不影響
結算過帳全打同一本帳本帳戶分片 + 條件更新；成交先記事件，過帳可批次處理
看行情的人太多，把系統問爆改成推播 + 快取，把「看行情」和「動錢」分開
引擎崩潰，記憶體狀態全沒了重播事件日誌還原 + 定期存「快照」縮短重播時間

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 委託簿（喊價配對）vs 自動做市（AMM） 委託簿撮合精準、價差小，但需要有人來掛單提供流動性；自動做市（用公式算價）永遠有得買賣，但會有滑點。實務常兩種混用。
⚖️ 一條隊伍（單執行緒）vs 多人並行 一條隊伍換來「絕不算錯、可重播」，速度靠多開攤位；多人並行看似快，但要上鎖、難保證不出錯。錢的場景幾乎都選一條隊伍。
⚖️ 強一致 vs 最終一致 動錢（過帳、餘額）必須強一致，剛改完就要查得到、絕不超額；看行情/做分析可以最終一致（晚一點同步沒關係）。該嚴的地方嚴、能鬆的地方鬆。
💭 追問：同一個人「同時下兩張單、想花同一筆錢」會怎樣？ 第一張單下單時就把錢凍結了；第二張單要凍結時發現「可用餘額不夠」，當場被拒。靠「先凍結 + 有才扣得動的條件更新」防住雙花。
💭 追問：成交價到底用買家出的還是賣家開的？ 用先掛在簿上那張（maker）的價。主動來吃單的人（taker）反而可能拿到比自己出價更好的價，多凍結的保證金會退回。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 OrderBook（菜市場的喊價配對員＝委託簿）→ Ledger（那本絕不出錯的帳本，管凍結與過帳）→ MatchingEngine（總指揮：排隊、先凍結、再撮合、再過帳）。一筆單進來，就照這個順序跑一遍。

1喊價配對：價高/價低先，同價先到先（OrderBook）

這是菜市場喊價那一關。一張新買單進來，去吃簿上「最便宜、同價先掛」的賣單，能吃多少吃多少，剩下的掛回去等：

function 撮合(新買單) {
    把賣單排序：價低的在前、同價先掛的在前   // ← 價格-時間優先
    剩餘 = 新買單.數量
    for (每一張賣單 in 賣簿) {
        if (賣單.價格 > 新買單.出價) break;   // ← 最便宜的都比我出價貴 → 收手
        成交量 = min(剩餘, 賣單.數量)
        記一筆成交{ 成交價 = 賣單.價格,        // ← 成交價用「先掛的那張(maker)」的價
                   量 = 成交量, 買方, 賣方 }
        剩餘      -= 成交量
        賣單.數量 -= 成交量                    // 這張被吃掉一部分
    }
    清掉數量歸零的賣單
    if (剩餘 > 0) 把剩餘掛回買簿            // ← 沒吃完 → 部分成交，繼續掛著等
    return 這次的成交清單
}

排序那兩行就是「誰出得高/開得低誰先，同價先到先」；break 那行是「最便宜的都比我貴就收手」。成交價刻意取簿上先掛那張（maker）的價，所以主動來吃單的人有時拿到比自己出價更好的價——多凍結的錢稍後會退。這段是純算簿、完全不碰錢，職責分得很乾淨。

2先卡住錢：有才扣得動，餘額永不為負（Ledger）

下買單前，先把錢卡住（刷卡預授權）。關鍵是用「有才扣得動」的條件判斷——不夠就整筆拒絕，所以餘額永遠不會變負數：

function 凍結保證金(使用者, 金額) {
    if (可用餘額 < 金額) return false;    // ← 條件不成立 → 拒絕(防超額)，不進簿、不撮合
    可用餘額 -= 金額                       // 從「可用」搬到「凍結」
    凍結中   += 金額
    return true
}
// 賣單同理：freezeShares() 先把要賣的「票」卡住 → 防雙花

就這一行 if，把「餘額不足」「同一筆錢想花兩次」全擋在門外。對應到真實 DB 就是一句條件更新 UPDATE … SET available=available-N WHERE available >= N——受影響列數 = 0 就代表錢不夠，邏輯一模一樣。

成交時則要一手交錢一手交貨，買賣雙方一次到位（要嘛全成、要嘛全不動）：

function 成交過帳(買方, 賣方, 量, 成交價) {
    成本 = 量 × 成交價
    買方.凍結中 -= 成本;   買方.持票 += 量    // 買方：扣掉預授權的錢、拿到票
    賣方.持票   -= 量;     賣方.可用 += 成本  // 賣方：交出票、收到錢
}   // ← 這整段在「同一把鎖(＝同一個交易)」內完成，全成或全不動

四行加減必須綁成一個不可分割的動作（ACID 原子性）：絕不會「買方錢扣了、票卻沒到手」。demo 用一把檔案鎖模擬「一個 DB 交易」，真實系統換成 BEGIN … COMMIT，原子過帳的邏輯不變。

3總指揮：排一條隊，先凍結→再撮合→再過帳（MatchingEngine）

這是全題的心臟。它把上面兩個幫手串起來，而且關鍵在整段下單在「一條隊伍」裡一筆筆做，不讓人同時搶：

function 下單(使用者, 買賣, 價格, 數量) {
    搶到全域鎖();                          // ← 一次只處理一筆單(單執行緒序列化)，防搶單打架
    try {
        // 1) 先凍結 → 凍不到就直接拒，不進簿、不撮合
        if (買) { if (!凍結保證金(使用者, 價格×數量)) return 拒絕「餘額不足」 }
        else    { if (!凍結票(使用者, 數量))         return 拒絕「持倉不足」 }

        記事件('ORDER_PLACED', 這張單)        // ← 每個動作都寫進永不塗改的流水帳
        成交清單 = OrderBook.撮合(這張單)     // 2) 純算簿，產生成交

        for (每筆成交 in 成交清單) {
            Ledger.成交過帳(...)              // 3) 逐筆原子過帳
            記事件('TRADE_EXECUTED', 成交)
            if (成交價 比我出價便宜) 退回多凍結的錢  // ← 拿到更好的價 → 退差額
        }
        把委託簿寫回硬碟                       // 4) 含剩餘掛單，跨請求也記得
    } finally { 放開全域鎖() }
}

順序不能換：一定是先凍結（防超額）→ 再撮合 → 再原子過帳。「搶到全域鎖 / 放開鎖」那兩行，就是前面說的「同一攤位排一條隊、一筆筆認真處理」；中間每步都 記事件()，崩潰時把流水帳重播一遍就能還原。三個幫手在這裡合體，一題的精華全在這段。

💭 停一下：為什麼是「先凍結錢，才去撮合」？反過來先撮合再扣錢不行嗎？

看答案 不行。若先撮合再扣錢，撮合都成交了卻發現「買方其實沒錢」，這筆成交就得反悔、把對手單也退回——一團亂。先凍結（確定付得起）再讓單上場，凍不到就根本不進簿、不撮合，從源頭杜絕爛帳。這就是「刷卡預授權確認你付得起，才放心讓你開車」。
🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 下單先凍結錢Ledger凍結保證金() 的那行 if（有才扣得動）
② 喊價配對撮合OrderBook撮合()（價格-時間優先 + 部分成交）
③ 一手交錢一手交貨Ledger成交過帳()（同一交易內原子完成）

👉 重點：真實系統會把「JSON 檔 + 檔案鎖 + 迴圈撮合」換成「專業撮合引擎 + 關聯式 DB 交易（BEGIN…COMMIT）+ 事件串流」， 但上面這幾段判斷邏輯一字都不用改：撮合照樣價格-時間優先、資金照樣「先凍結→原子過帳→有才扣得動」。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：為什麼一張票的「價格」就等於「機率」？用兌獎券解釋。 因為這張票「事件成真就兌 1 元、不成真歸零」。如果市場覺得有 62% 機會成真，這張票就值約 0.62 元——願意付多少買，就等於賭它有多少把握成真。所以價格 = 隱含機率。
Q2：用「菜市場喊價」解釋撮合規則（價格-時間優先）。 出價最高的買家先買到、開價最低的賣家先賣掉（價格優先）；同價就先掛單的先成交（時間優先）。當最高買價 ≥ 最低賣價就配對，大單可吃掉多張小單，沒吃完的繼續掛著。
Q3：下單時為什麼要「先凍結錢」？像生活中的什麼？ 像刷卡的「預授權」：飯店先卡住你一筆錢、確定你付得起才讓你住。下單先凍結，確定你有錢、跑不掉，才讓單上場——杜絕「掛單時有錢、成交時沒錢」的爛帳。
Q4：這系統「最怕」哪三件事？ 怕錢算錯（正確性壓倒一切、要強一致）、怕同時搶單搶到打架（同一攤位排一條隊一筆筆處理）、怕系統一掛帳就消失（每個動作先寫永不塗改的流水帳，崩潰能重播）。
Q5（工程）：為什麼撮合要「一條隊伍、一筆筆處理」，而不是多人並行加速？ 多人同時動同一本委託簿會搶到打架（兩人吃同一張單、同筆錢花兩次），要靠大量鎖且難保證正確。一條隊伍序列化反而無鎖、確定性高、可重播；要擴大就多開攤位（按市場分片），攤位間互不干擾。
Q6（工程）：成交時「過帳」為什麼一定要綁成一個原子交易？ 因為「買方付錢拿票、賣方交票拿錢」必須同時發生、要嘛全成要嘛全退，絕不能做一半（買方錢扣了票卻沒給）。再配上「有才扣得動」的條件更新，保證餘額永不為負、不被雙花。
Q7（工程）：引擎在記憶體跑很快，但崩潰了狀態怎麼救回來？ 每個關鍵動作（下單、成交、過帳）都先寫進「永不塗改的事件日誌」。崩潰後把日誌從頭重播一遍就能還原現場；再配定期存「快照」縮短重播時間。這本日誌也是查帳、對帳、處理糾紛的真相來源。

15. 30 秒回顧

預測市場 = 喊價配對的交易所 + 絕不出錯的帳本。價格就是大家心中的「機率」。

4 個關卡：① 下單先凍結錢（刷卡預授權）→ ② 喊價配對撮合（菜市場，價格-時間優先）→ ③ 一手交錢一手交貨原子過帳 → ④ 事件揭曉結算發錢。

3 個怕：怕錢算錯（正確性壓倒一切、強一致）、怕同時搶單打架（一條隊伍一筆筆處理）、怕崩潰丟帳（永不塗改的流水帳可重播）。

工程細節一句話：讀多寫少 → 看行情走推播+快取、別查真相帳本；撮合按市場分片各跑一條隊伍；資金用「先凍結 → 原子過帳 → 有才扣得動」防超額防雙花；一切先記事件日誌，崩潰可重播、可審計。
