③ Polymarket 預測市場：order book 撮合與資金一致性

C 一致性/交易 建議 53 分鐘 Design a Prediction Market / Order-Book Exchange

一句話定位 本質是「限價單委託簿交易所 + 資金強一致結算」。預測市場用 YES/NO 股份 把「事件機率」變成可買賣的價格（0~1）； 面試真正考的是 撮合引擎如何在高並發下不出錯（不超賣、不雙花、可審計），而非畫出 CRUD。

1. 需求釐清

功能性需求

  • 市場以二元結果建模：每個市場有 YES / NO 兩種股份，市場到期由預言機（oracle）判定結果，贏的一側每股兌現 1 元、輸的一側歸零。
  • 限價單委託簿：使用者掛買/賣單（價 0~1 = 隱含機率），撮合引擎依規則成交，支援部分成交與撤單。
  • 資金管理：下單前凍結保證金/股份；成交時原子過帳；查詢餘額、持倉、歷史成交。
  • 市場生命週期：開市 → 交易 → 凍結 → 結算（resolve）→ 兌付（payout）。
  • 行情：即時推送最佳買賣價、深度、最近成交（讀多寫多但可分流）。

非功能性需求

  • 正確性凌駕一切：涉及金錢，不能超額下單、不能雙重花用、餘額不可為負，結算後資金必須守恆。
  • 強一致：撮合與資金過帳要 ACID；同一帳戶的並發下單不可產生競態。
  • 低延遲撮合：熱門市場 p99 撮合 < 10ms；行情近即時。
  • 可審計 / 可重放：每一筆下單、成交、過帳都要能回溯（監管、對帳、爭議處理）。
  • 高可用：交易引擎是關鍵路徑，崩潰後能從事件日誌重建狀態。
面試開場必問 「是 委託簿（order book / CLOB） 還是 自動做市商（AMM / LMSR）？」 Polymarket 主力是中央限價委託簿（撮合 maker/taker），本筆記主軸即委託簿撮合 + 結算一致性；AMM 可作為流動性補充作為延伸討論。

2. 規模估算

指標假設推算
下單（寫）熱門事件期間 500 萬單/日~58 寫 QPS（峰值 ×10 ≈ 580）
撮合吞吐單市場單執行緒單核可達 ~10 萬筆/秒（純記憶體無鎖）
行情讀觀看者遠多於下單者讀:寫可達 100:1，走推播 + 快照快取
事件日誌每筆事件 ~300 bytes，5 年500萬×365×5×300B ≈ 2.7 TB（可冷儲存）
市場數同時數千個活躍市場依市場 ID 分片，各自單執行緒撮合
關鍵洞察撮合不靠「加機器並行」提速（並行會破壞順序與正確性），而是按市場分片：每個市場單執行緒序列化，市場之間水平擴展。

3. API 設計

# 下限價單（寫，撮合核心入口）
POST /api/v1/orders
Body: { "market_id": "elec-2028", "outcome": "YES",
        "side": "buy", "price": 0.62, "qty": 100, "client_order_id": "uuid" }
→ 201 { "order_id": 987, "status": "partially_filled",
        "filled_qty": 40, "trades": [ ... ] }

# 撤單（釋放凍結的保證金/股份）
DELETE /api/v1/orders/{order_id}

# 委託簿快照 / 深度（讀，高頻 → 快取 + 推播）
GET  /api/v1/markets/{id}/book        → { bids:[[price,qty]...], asks:[[...]] }
WS   /ws/markets/{id}                  → 推播最佳價、增量更新、最新成交

# 帳戶（餘額 / 持倉 / 成交）
GET  /api/v1/accounts/{id}/balance     → { available, held, positions:{YES,NO} }

# 市場結算（管理 / 預言機）
POST /api/v1/markets/{id}/resolve      Body: { "winner": "YES" }
冪等下單帶 client_order_id；撮合引擎以它去重，避免網路重試造成重複下單（重複下單 = 重複凍結資金）。

4. 資料模型

accounts (帳本，強一致主庫)
  account_id     BIGINT PK
  available      DECIMAL(18,6)   -- 可用現金
  held           DECIMAL(18,6)   -- 凍結現金（掛單保證金）
  version        BIGINT          -- 樂觀鎖
positions
  account_id, market_id, outcome(YES|NO), qty, held_qty   -- held_qty=掛賣凍結中

orders (掛單，可變狀態)
  order_id PK, market_id, account_id, outcome, side, price,
  qty, filled_qty, status(open|partial|filled|cancelled), ts

trades (成交，append-only)
  trade_id PK, market_id, price, qty, buy_order, sell_order, maker, taker, ts

events (事件日誌，append-only，可重放/審計)
  seq BIGINT PK, type, payload JSONB, ts        -- 真實走 Kafka commit log
為何分開accounts/positions 是可變且強一致的資金真相；trades/events 是不可變的歷史。撮合在記憶體中跑，落地靠事件日誌；帳本過帳走 DB 交易。三者職責分離。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 交易者 ──下單──▶ API/風控(凍結保證金) ──▶ 撮合引擎(單執行緒 order book)
                        │                          │
                        │                          ├─▶ 事件日誌(append-only)
                        │                          │        │(重放)
                        │                          ▼        ▼
                        └◀─行情推播(WS,綠)◀── 結算/帳本(原子過帳, DB 交易)
                                                   ▲
                                              錢包/餘額
  1. API / 風控：驗證、冪等去重、下單前先凍結資金（買）或股份（賣），不足直接拒。
  2. 撮合引擎：每個市場一條單執行緒序列化處理，跑價格-時間優先撮合，產生成交。
  3. 事件日誌：下單/成交/過帳全寫成不可變事件 → 崩潰後重放重建、可審計。
  4. 結算/帳本：依成交做原子過帳（DB 交易）；市場到期執行兌付。
  5. 行情推播：撮合產生的增量更新經 WebSocket 即時推給觀看者。

6. 核心元件深入

為什麼撮合引擎要單執行緒？

  • 多執行緒搶同一本委託簿 → 需要大量鎖，且容易產生競態（兩張單同時吃掉同一張對手單）。
  • 業界做法（LMAX、多數交易所）：單執行緒序列化處理單一市場的所有指令，無鎖、完全確定性、輸入相同→輸出相同，天然可重放。
  • 擴展靠分片：不同市場分到不同引擎實例；單市場吞吐已足夠（純記憶體可達數十萬筆/秒）。

價格-時間優先（price-time priority）

  • 買單：價高者先，同價先到者先；賣單：價低者先，同價先到者先。
  • 當最高買價 ≥ 最低賣價即成交，成交價取被動單（maker，先掛在簿上那張）的價格；主動單（taker）可能因此拿到更好的價。
  • 支援部分成交：大單吃掉多張對手單，剩餘量留在簿上。

資金一致性：凍結 → 過帳 → 防超額（本題靈魂）

  • 先凍結：買單下單即把 price×qty 現金從 available 移到 held；賣單則凍結對應股份。沒凍結到就不進簿 → 杜絕「掛單時有錢、成交時沒錢」。
  • 原子過帳：成交時在同一個 DB 交易內：買方扣凍結現金+加持倉、賣方減凍結股份+加現金，全成功或全失敗。
  • 防超額/雙花：所有扣減走條件更新 UPDATE … SET available=available-? WHERE available >= ?，受影響列數=0 即拒絕 → 餘額永不為負。高衝突可改 SELECT … FOR UPDATE（悲觀）或樂觀鎖 version（CAS）。

事件溯源（event sourcing）

撮合引擎的記憶體狀態不是真相來源，事件日誌才是。下單、成交、過帳依序 append；引擎崩潰後從日誌重放即回到一致狀態，且整段歷史可審計/對帳/重現爭議。這也是金融系統能「跑得快又賠得起」的關鍵。

7. 擴展與瓶頸

階段瓶頸對策
單市場吞吐單執行緒撮合上限純記憶體無鎖（已極快）；批次寫日誌、群組提交
多市場一台機器裝不下所有市場依 market_id 分片，市場間水平擴展、互不影響
帳本寫熱點結算過帳打同一帳本帳戶分片 + 條件更新；成交先記事件、過帳可批次/非同步對帳
行情讀爆大量觀看者拉委託簿WebSocket 推增量 + 快照快取（Redis/CDN），讀寫解耦
崩潰恢復引擎掛了狀態遺失事件日誌重放重建 + 定期快照（snapshot）縮短重放時間

8. 取捨與面試重點

  • 委託簿 CLOB vs AMM/LMSR：CLOB 撮合精準、價差小但需流動性（掛單者）；AMM 永遠有流動性但有滑點與資金效率問題。實務常混用。
  • 單執行緒 vs 多執行緒撮合：單執行緒換來確定性與可重放，吞吐靠分片；多執行緒提速但要鎖、難證明正確。金融場景幾乎都選前者。
  • 強一致 vs 最終一致：資金過帳必須強一致（讀自己的寫、不可超額）；行情/分析可最終一致。
  • 悲觀鎖 vs 樂觀鎖：同帳戶高衝突用 FOR UPDATE；低衝突用 version CAS 提吞吐。
  • 常見追問：
    • 引擎崩潰怎麼辦？→ 事件日誌重放 + 快照。
    • 同帳戶兩張單同時下、想花同一筆錢？→ 凍結 + 條件更新，第二張凍結失敗被拒（防雙花）。
    • 成交價怎麼定？→ 取 maker 價；taker 可能得到更優價，多凍結的保證金退回。
    • 市場結算錯了怎麼辦？→ 預言機爭議期 + 可回滾事件（補償交易）。
    • 如何對帳？→ 重放事件日誌重算帳本，與真相比對。

9. PHP Demo 說明

本題 demo 著重於系統核心邏輯（也是面試真正的考點：order book 撮合 + 資金一致性），可實際用 PHP 內建伺服器跑起來：

  • src/OrderBook.php　限價單委託簿；價格-時間優先撮合、部分成交、產生成交(trade)。
  • src/Ledger.php　帳本/錢包；凍結保證金/股份、成交原子過帳、條件更新防超額/防雙花（餘額永不為負）。
  • src/MatchingEngine.php　撮合引擎；全域鎖序列化下單（單執行緒語意），串接凍結→撮合→過帳，並做市場結算。
  • src/EventLog.php　事件日誌（append-only），每個關鍵動作寫一筆不可變事件，可印出可重放的事件序列。
  • public/index.php　路由：下單 / 結算 / 重置 / 狀態，並附深色測試頁（委託簿、最近成交、帳戶餘額/持倉、事件日誌）。
# 啟動
cd demo
php -S localhost:8003 -t public
# 瀏覽器開 http://localhost:8003：下幾筆買/賣單看撮合，故意下超過餘額的單看被拒
誠實聲明 Demo 用 JSON 檔 + 檔案鎖模擬「DB 交易 / 單執行緒序列化」屬示意；而 撮合演算法、價格-時間優先、部分成交、凍結保證金、成交原子過帳、防超額/防雙花、事件溯源 等系統邏輯為真實可執行（所有操作後現金與股份皆守恆，已實測）。生產級替換方式見 demo/README.md。
