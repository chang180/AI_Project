1. 60 秒搞懂它在幹嘛

想像你開了一家記帳超嚴格的銀行。每天有上億筆「A 付錢給 B」， 你要保證一件最簡單卻最難的事：錢一塊都不能多、不能少、不能憑空消失。

數位錢包 / 支付系統就是在做這件事。它的難，難在四點：手滑點兩次不能扣兩次、餘額不夠不能硬扣、 同一筆轉帳要嘛全成要嘛全敗、而且整本帳隨時加起來都要對得上。

🔑 一句話 數位錢包 = 一本永遠對得起來的會計帳本：每筆交易都「有借必有貸」，錢只是從一個口袋搬到另一個口袋， 全系統的總額永遠守恆——絕不會憑空多出或少掉一塊錢。

2. 為什麼錢包不是「餘額加加減減」這麼簡單？

很多人第一個直覺是：「錢包不就是每個人一個 balance 數字，付錢就減、收錢就加？」 聽起來合理，但這正是會出大事的地方。關鍵在一個你早就懂的生活經驗——

📒 記帳本 vs 便利貼 把餘額當一個「可以隨手改的數字」，就像把存款記在一張便利貼上，塗掉再寫新的。 出錯了你完全查不出哪裡錯、錯多少——便利貼沒有歷史。 真正的會計用的是記帳本：每筆錢的進出都一行一行「只能往下加，不能塗改」地記下來。

而且會計記帳有個鐵律：有借必有貸、借貸金額相等。A 付給 B 100 元，記成兩行： 「A 少 100（借）」和「B 多 100（貸）」，兩行金額一樣、方向相反。這叫雙分錄。

這樣做的好處超大：因為每筆交易都是「從一個口袋搬到另一個口袋」，全系統的錢加起來永遠是同一個數字。 哪天總額對不上，你立刻知道「有 bug 或有人動了手腳」——這叫守恆，是金融系統的安全網。

💭 停一下：如果用「便利貼」式的餘額欄位，兩筆轉帳同時想花你帳戶裡同一筆錢，會發生什麼事？

看答案 兩筆交易可能同時讀到「餘額還有 100」，於是都判斷「夠扣」，結果各扣 100——你只有 100 卻被花掉 200，憑空多花了一筆（這叫「雙花」/競態）。便利貼擋不住這種事；後面講的「條件更新防透支」就是來治它的。

3. 把系統拆成 4 個關卡

別被「支付系統」嚇到。錢的事說穿了就是闖 4 關，每一關都在守護「錢不會算錯」：

1記帳：每筆交易「有借必有貸」（這是靈魂）

A 轉 B 100 元，系統不是把 A 的數字減 100、B 的數字加 100 就算了。 它會在帳本上「只往下加、永不塗改」地寫兩行：A 借 100、B 貸 100。

🏦 比喻就像銀行的存摺：你只會看到一行一行「+」「−」往下記，沒人能用立可白塗掉某一行。要更正只能再記一行「沖正」。完整保留歷史，隨時可查帳。

👉 這帶來一個威力：「餘額」其實不是存起來的，而是把帳本所有行加總「算出來」的。就算系統當機，只要帳本還在，餘額隨時能重算。

2防手滑：同一筆轉帳手滑按兩次，只扣一次（冪等）

網路很不可靠。你按下「轉帳」，畫面卡住，你又按了一次——錢不能被扣兩次。

🎫 比喻：演唱會的票根存根 你交一張票進場，工作人員撕下一半存根、記下你的座位號。如果你拿同一張票再來一次， 他一看存根「這張已經進過了」，就直接放你進去（你還是在原座位），不會幫你多開一個座位。

系統的做法：每筆轉帳你帶一張獨一無二的「冪等鍵」（idempotency key，就是那個座位號）。 系統處理前先看「這把鑰匙用過沒？」——用過就直接回上次的結果，不再扣款。 這叫冪等：同一個動作做幾次，效果都跟做一次一樣。

3防透支：餘額不夠，就不准扣

你只有 100，卻想轉 150——系統必須擋下來，餘額絕不能變負數。聽起來理所當然，難在「兩筆同時來」。

🚪 比喻：只有一道窄門，一次只放一個人 想像帳戶門口有一道只容一人通過的窄門。要扣款的人得先擠進門，門裡的人檢查完餘額、扣完款、出來，下一個才能進。 這樣第二筆扣款進門時，看到的已經是扣過的餘額，發現不夠就被擋下——不會兩筆都以為自己夠扣。

工程上的招數叫「條件更新」：扣款時下一道命令「餘額夠才扣，不夠就一毛都別動」， 由資料庫保證這個「檢查＋扣款」是一口氣做完、中間插不進別人。第二筆一看不夠，就被乾淨地拒絕。

4對帳：全部加起來要對得上（守恆）

跑得再快，也要賠得起——系統要能隨時證明「錢一塊都沒少」。

🧮 比喻：店長每晚盤點收銀機 便利商店店長每晚會數收銀機現金，跟「今天賣了多少」對一對。對得上就安心，對不上立刻查。 系統也有個「對帳 Job」定期把整本帳重新加總一遍，驗證「全部的借 = 全部的貸」、 算出來的餘額跟系統顯示的一致。一旦對不上馬上告警。

因為有了關卡 1 的「有借必有貸」，這個對帳才做得到——全系統的錢理論上永遠守恆，差一塊都會被抓出來。

✅ 四關一句話 ① 每筆交易有借必有貸記進帳本 → ② 手滑兩次只扣一次（冪等）→ ③ 餘額不夠不准扣（防透支）→ ④ 全帳加總永遠對得上（對帳守恆）。

4. 設計時最怕的 4 件事

工程上的「取捨」聽起來很抽象。換個方式記：錢的系統最怕四件事，每個設計都是為了避開某個「怕」。

😱 怕「重複扣款」 網路重試、用戶連點，同一筆付款被執行兩次——你少了一筆錢。所以每筆付款帶一把冪等鍵， 系統認鑰匙不認次數，同鑰匙只生效一次，重送只回上次結果。做幾次 = 做一次。
😱 怕「憑空多花（雙花 / 透支）」 餘額不夠卻扣成功、或兩筆同時花同一筆錢——錢憑空多出來。所以扣款一律走條件更新 （夠才扣）＋「窄門」序列化，餘額永不為負。沒這筆錢，誰都別想扣走。
😱 怕「扣一半（不原子）」 最可怕的情況：A 扣了、B 卻沒加，錢蒸發了。所以「借」和「貸」兩行綁在同一個交易裡， 要嘛兩行都成、要嘛兩行都不發生，絕不會只做一半。全成或全敗，沒有中間態。
😱 怕「算不清、賴帳」 出了糾紛卻查不出錢怎麼跑的，最致命。所以帳本只能追加、不能塗改（append-only）， 每塊錢來去都留痕，加上定期對帳驗證守恆。隨時攤開帳本，一筆都賴不掉。

雙分錄帳本
(double-entry ledger)		會計記帳法：每筆交易記兩行「有借必有貸」，金額相等方向相反
分錄 (entry)帳本上的一行：誰、借還是貸、多少錢
append-only「只能往下加、不能塗改」的帳本，像存摺
冪等 (idempotent)同一個動作做幾次，效果都跟做一次一樣
冪等鍵 (idempotency key)每筆付款獨一無二的「票根存根號」，用來認出重送
原子 (atomic)一組動作「全成或全敗」，不會只做一半
交易 / 交易 (transaction)把多個動作綁成一包，全成或全敗的那層保護
防透支餘額不夠就不准扣，餘額永不為負
條件更新「夠才扣，不夠一毛別動」的一口氣指令
雙花 (double-spend)同一筆錢被花兩次（兩筆同時扣）
對帳 (reconciliation)定期把整本帳重算、驗證「借=貸」、餘額對得上
守恆全系統的錢加總永遠是同一個數，不多不少
物化 / 快取把「算好的餘額」先存一份，查餘額才不用每次掃全帳
沖正 (reversal)記錯了不塗改，而是再記一行相反的把它抵銷
TPS / QPS每秒幾筆交易／請求（衡量「多忙」的單位）
分片 (sharding)資料太大，拆給好幾台機器分開存

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每天多少筆付款（寫）1 億筆/日 ≈ 每秒 ~1,200 筆，尖峰再 ×10寫入要扛得住，且每筆都得正確
帳本長多快每筆至少 2 行分錄 → 每天 ≥ 2 億行帳本只增不刪，一年十幾 TB，要分冷熱
查餘額（讀）多頻繁讀遠多於寫，可達 50:1查餘額要快 → 存一份算好的「餘額快取」
付款要多快回應p99 < 200 毫秒付款是關鍵路徑，不能讓人等
💰 關鍵洞察：一分都不能用浮點數 錢一律用「分」這種整數存（1500 分 = 15.00 元），絕不用小數（浮點數）。 因為電腦算小數會有微小誤差（0.1 + 0.2 居然不等於 0.3），一兆筆交易累積下來會「吃掉錢」——對銀行是大忌。

💭 停一下：查餘額比付款頻繁這麼多倍，如果每次查餘額都去「把整本帳加總一遍」，會怎樣？

看答案 會慢到爆、還把資料庫壓垮。所以系統另外存一份「算好的餘額」（物化／快取），查的時候直接讀這個數字，秒回。帳本（慢、但正確）負責「真相」，餘額快取（快）負責「給人看」——這就是「寫路徑保正確、讀路徑求飛快」。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」，最重要的是第一張（轉帳）一定要帶冪等鍵：

① 轉帳 / 付款（最重要，必帶冪等鍵）
   POST /transfers
   附上一把鑰匙：Idempotency-Key: 9f1c2e7a-...   ← 你產生的、獨一無二
   給我：{ 從誰 from, 給誰 to, 金額 amount:1500（分）, 幣別 }
   我回：201「成功，這是交易編號 txn_123，借貸兩行如下」

同一把鑰匙再送一次 → 我回 200 + 上次那筆 txn_123，不再扣款。這就是「票根存根」在運作。若同鑰匙卻帶了不同金額？我回 409「衝突」——一把鑰匙只能對應一筆交易，不准冒充。

② 儲值 / 提領（其實也是轉帳：系統金庫 ↔ 你）
   POST /topups        { to:"你", amount:10000 }   ← 充值
   POST /withdrawals   { from:"你", amount:3000 }  ← 提領

把「系統金庫」也當成一個帳戶，儲值就是「金庫 → 你」、提領就是「你 → 金庫」。這樣連充值提領都遵守「有借必有貸」，全帳照樣守恆。

③ 查餘額（很頻繁 → 讀那份算好的快取）
   GET /accounts/{id}/balance   → { available:8500, currency:"TWD" }

④ 看對帳單 / 帳本（只追加、可審計）
   GET /accounts/{id}/ledger    → { entries:[ 一行一行的借貸... ] }

9. 系統要記住哪些事？（資料模型）

📒 比喻：三本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題只要三本：

📒帳本簿（最重要：每筆借/貸都在這，只追加）

每一行記：分錄ID、這筆轉帳的編號 transfer_id、是誰的帳戶、借(D)還是貸(C)、多少錢、時間。

為什麼這本是「真相」？ 因為它只追加、不塗改，完整保留每一塊錢的來去。 鐵律：同一個 transfer_id 的所有行，借的總和 = 貸的總和（淨額為 0）。 任何違反都代表出 bug 或被竄改——這就是對帳能抓漏洞的根據。

💳帳戶簿（餘額快取，可由帳本重算）

每個帳戶一列：帳戶ID、餘額(整數分)、幣別、版本號。

餘額為什麼只是「快取」？ 因為真正的餘額是「帳本所有行加總」算出來的。這裡存一份是為了查詢飛快， 就算這份壞了或當機，拿帳本重算一遍就能補回來。那個「版本號」是給「窄門」防並發用的。

🎫鑰匙簿（冪等去重，防手滑）

每把用過的冪等鍵一列：鑰匙、請求內容的指紋、對應的交易編號、狀態(處理中/已完成)、何時過期。

就是那本「票根存根」 新付款先來這本登記、佔位；重送的同鑰匙一查「已完成」，直接回上次結果，不再扣款。 鑰匙留一段時間（涵蓋重試窗口）就能到期自動回收，不會無限長大。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
單一資料庫寫不下這麼多交易依帳戶 ID 分片：不同人的帳分到不同機器
熱門帳戶（金庫、人氣商家）被狂扣狂加把它的計數拆成好幾桶分開記，再定期合併
A、B 不在同一台機器，無法用「一個交易」綁住用 Saga（分段做＋出錯就補償沖正），靠冪等＋對帳兜底
查餘額的請求把主資料庫打爆餘額物化＋快取（Redis），讀寫分開
帳本只增不刪，越長越大冷熱分層＋定期存「快照」，重算時不用從頭加
鑰匙簿無限累積設到期時間自動回收（留夠重試窗口即可）

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的四個怕」是核心。這裡補幾個常被面試官追問的：

⚖️ 餘額欄位 vs 雙分錄帳本 便利貼式餘額簡單，但無法查帳、並發易錯；雙分錄帳本可審計、守恆、可重建，是金融標準。本題務必選帳本。
⚖️ 餘額「即時更新」 vs 「事後慢慢算」 寫的時候同步更新餘額 → 你立刻看到正確數字，但寫比較重；事後背景慢慢算 → 寫得快但餘額有點延遲。錢包通常選即時更新（用戶要馬上看到對的餘額）。
⚖️ 悲觀鎖（窄門）vs 樂觀鎖（先做後驗） 同帳戶常常搶 → 用「窄門」鎖住排隊最穩；很少搶 → 用「版本號比對（先做，提交時驗有沒有被別人改過）」吞吐更高。
💭 追問：扣了我的款，但對方系統沒收到，怎麼辦？ 投遞保證「至少送到一次」＋對方冪等消費（同一筆只認一次）＋對帳補償：對帳發現缺口，就追加一行沖正把帳補平。錢不會卡在半空中。
💭 追問：系統崩潰重開後，餘額會不會就錯了？ 不會。帳本是「真相」且只追加，重開後拿帳本重放／用快照重算就能把每個餘額還原。餘額快取只是加速用，壞了能重建。
💭 追問：怎麼向別人「證明」錢一塊都沒少？ 跑對帳：驗證全系統的借總和 = 貸總和，再把帳本重算的餘額跟系統顯示的餘額逐一比對。全對得上，就是錢沒少的鐵證。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「雙分錄＋冪等＋防透支」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8017 -t public
# 然後瀏覽器打開 http://localhost:8017
  1. 先開幾個帳戶並給初始餘額（等於開幾個錢包）。
  2. 做一筆轉帳 → 再用同一把冪等鍵重送一次，看是不是只扣一次。
  3. 故意轉超過餘額的金額 → 看它被乾淨地拒絕（防透支）。
  4. 最後列出帳本，驗證每筆都「借 = 貸」、全帳總額守恆。
玩的時候想一下同一把冪等鍵重送，餘額有沒有少第二次？這就是關卡 2 的冪等。轉一筆超過餘額被拒，餘額有沒有變負？這就是關卡 3 的防透支。最後列帳本看「借總和 = 貸總和」——這就是關卡 4 的守恆在運作。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有兩個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 兩個檔 = 兩個小幫手 Ledger（那本只能往下加的帳本＝記分錄、扣加餘額、對帳）→ TransferService（每筆轉帳的守門員＝冪等＋防透支＋原子過帳）。一筆轉帳進來，就由守門員指揮帳本，照這個順序跑一遍。

1雙分錄：一筆轉帳寫兩行（Ledger）

記帳的靈魂：A 轉 B 100 元，不是改兩個數字，而是在帳本「只往下加」地寫兩行——A 借（D）、B 貸（C），金額一樣、方向相反。

function 寫兩行分錄(轉帳編號, 出錢方, 收錢方, 金額) {
    帳本[] = { 轉帳編號, 帳戶:出錢方, 方向:'D'(借), 金額 }   // ← 第一行：出錢方「借」
    帳本[] = { 轉帳編號, 帳戶:收錢方, 方向:'C'(貸), 金額 }   // ← 第二行：收錢方「貸」
    // 兩行金額相等、方向相反 → 這筆的淨額剛好是 0
}

這就是「有借必有貸」。因為每筆都這樣寫，全帳的「借總和」永遠等於「貸總和」——錢只是從一個口袋搬到另一個口袋，總額憑空生不出也滅不掉。連開戶初始入帳都走這套（系統金庫「借」、你的帳戶「貸」），所以全帳照樣守恆。

2防透支：餘額夠才扣（Ledger）

扣款的關鍵一行。要扣之前先算「扣完會不會變負數」，會變負就一毛都別動、直接回報失敗。

function 改餘額(帳戶, 變動額) {        // 扣款時變動額是負數，例如 -150
    新餘額 = 現在餘額 + 變動額
    if (新餘額 < 0)
        return 失敗;                   // ← 防透支：會變負就拒絕，餘額永不為負
    餘額[帳戶] = 新餘額                 // 夠才真的扣
    return 成功;
}

真實系統會把這寫成一句條件式 SQL：UPDATE 帳戶 SET 餘額=餘額-150 WHERE 餘額>=150，由資料庫保證「檢查＋扣款」一口氣做完、中間插不進別人（就是前面說的「窄門」）。受影響列數 = 0 就代表餘額不足、扣款被乾淨地拒絕。

3冪等：同一把鑰匙只扣一次（TransferService）

這是防手滑的心臟。每筆轉帳帶一把獨一無二的冪等鍵，真正動錢之前先查「這把鑰匙用過沒？」

function 轉帳(出錢方, 收錢方, 金額, 冪等鍵) {
    if (鑰匙簿裡有這把冪等鍵) {
        if (這次的請求內容 != 上次登記的內容)
            return 衝突;               // ← 同鑰匙卻換了金額 → 不准冒充
        return 上次的結果;             // ← 用過了：直接回放，不再扣款！
    }
    // ↓↓ 沒用過，才真的進去動錢（接第 4 段）↓↓
}

手滑按兩次、網路自動重送，第二次一進來看到「這把鑰匙已登記」，就直接回上次那筆 txn 的結果，餘額一毛都不會再少。這就是「轉帳按兩次只扣一次」——做幾次 = 做一次。

💭 停一下：為什麼「扣款失敗（餘額不足）」時，不要把這把冪等鍵寫進鑰匙簿？

看答案 因為餘額不足是「暫時」的失敗——你晚點儲值後，同一把鑰匙應該還能重試成功。如果失敗也把鑰匙登記成「已處理」，那你補了錢再送，系統會誤以為「這筆早做過了」直接回放失敗結果，害你永遠轉不出去。所以只有真的成功過帳才寫鑰匙。

4原子過帳：扣、加、記帳綁成一包（TransferService）

確認鑰匙沒用過、餘額也夠，才真的動錢。關鍵是這三件事綁在同一道鎖裡，全成或全敗：

上鎖();   // ← 窄門：整段過帳序列化，同一帳戶一次只放一筆進來
try {
    if (改餘額(出錢方, -金額) == 失敗)
        return 餘額不足;               // 防透支擋下，這裡直接返回（不寫鑰匙）
    改餘額(收錢方, +金額)              // 對手方入帳
    寫兩行分錄(新交易編號, 出錢方, 收錢方, 金額)   // append 借/貸兩行
    鑰匙簿[冪等鍵] = 這筆結果          // ← 成功了才登記鑰匙，供日後重送回放
} finally { 解鎖(); }

「扣出錢方 → 加收錢方 → 寫借貸兩行」三步在同一把鎖內一起完成：要嘛全做、要嘛全不做，絕不會「A 扣了、B 沒加」把錢扣一半蒸發掉。這把鎖就是 Part 1 講的「窄門」，也讓兩筆同時搶同一帳戶時排好隊、不會雙花。

5對帳守恆：整本帳重算一遍（Ledger）

最後的安全網：把帳本從頭加總一遍，驗證「全部的借 = 全部的貸」，而且重算的餘額要跟「存著的餘額快取」對得上。

function 對帳() {
    借總和 = 貸總和 = 0
    foreach (帳本每一行 分錄) {
        if (分錄.方向 == 'D') { 借總和 += 金額; 重算餘額[帳戶] -= 金額 }  // 借=出錢=減
        else                 { 貸總和 += 金額; 重算餘額[帳戶] += 金額 }  // 貸=收錢=加
    }
    守恆嗎 = (借總和 == 貸總和)                    // ← 全帳守恆：借=貸
    foreach (每個帳戶) 比對(重算餘額 vs 餘額快取)   // ← 算出來的 要等於 存著的
}

因為關卡 1 每筆都「有借必有貸」，這裡的「借總和 = 貸總和」理論上永遠成立；一旦不成立，就代表有 bug 或被竄改，馬上告警。這就是「總額不憑空生滅」能被「證明」出來的根據——像店長每晚盤點收銀機。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 有借必有貸記帳Ledger寫兩行分錄()（一借一貸）
② 手滑兩次只扣一次TransferService動錢前的「查鑰匙」
③ 餘額不夠不准扣Ledger改餘額() 的那行 if < 0
③+ 全成或全敗TransferService鎖內「扣→加→記帳」一包
④ 全帳對得上Ledger對帳()（借=貸＋餘額比對）

👉 重點：真實系統會把「JSON 檔 + 檔案鎖」換成「強一致資料庫的 DB 交易 + 帳本表 + 冪等鍵索引」， 但上面這幾段判斷邏輯一字都不用改：雙分錄（有借必有貸）、冪等（按兩次只扣一次）、防透支（夠才扣）、守恆（借=貸）。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「記帳本 vs 便利貼」解釋，為什麼餘額不該是一個「可隨手改的數字」？ 便利貼式餘額塗掉重寫、沒有歷史，出錯查不出哪裡錯、也擋不住兩筆同時扣的競態。記帳本「只追加不塗改」，每塊錢來去都留痕、可查帳、可重算餘額，還能驗證全帳守恆。
Q2：什麼是「雙分錄（有借必有貸）」？它帶來什麼好處？ 每筆轉帳記成金額相等、方向相反的兩行（一借一貸），共用同一個 transfer_id。好處是錢只是從一個口袋搬到另一個口袋，全系統總額永遠守恆——對帳時驗「借總和 = 貸總和」就能抓出任何漏洞。
Q3：用「演唱會票根存根」說明「冪等」，為什麼手滑按兩次只扣一次？ 每筆付款帶一把獨一無二的冪等鍵（像票根存根號）。系統處理前先查這把鑰匙用過沒，用過就直接回上次結果、不再扣款。同一個動作做幾次，效果都跟做一次一樣。
Q4：用「窄門」說明，為什麼餘額不會被透支或雙花？ 扣款得擠過一道一次只放一人的窄門：第一筆檢查並扣完款出來，第二筆進門時看到的已是扣過的餘額，不夠就被擋下。工程上靠「條件更新（夠才扣）」＋鎖序列化，餘額永不為負。
Q5（工程）：為什麼金額一律用整數「分」，不用小數（浮點數）？ 電腦算小數有微小誤差（0.1+0.2≠0.3），大量交易累積會「吃掉錢」。用整數分（1500 分 = 15.00 元）做運算與儲存，精確不漏。
Q6（工程）：「借、貸兩行」為什麼一定要綁在「同一個交易」裡？ 為了原子性——要嘛兩行都成、要嘛都不發生。否則可能「A 扣了、B 沒加」，錢憑空蒸發。同一個 DB 交易保證全成或全敗，沒有扣一半的中間態。
Q7（工程）：餘額快取壞了或系統崩潰，餘額會不會永久算錯？ 不會。真相來源是 append-only 帳本，餘額只是由帳本加總而來的快取。崩潰後重放帳本（或用快照加速）就能把每個帳戶餘額重算還原。

15. 30 秒回顧

數位錢包 = 一本永遠對得起來的會計帳本。錢只在口袋間搬，不憑空多也不憑空少。

4 個關卡：① 有借必有貸記帳本 → ② 手滑兩次只扣一次（冪等）→ ③ 餘額不夠不准扣（防透支）→ ④ 全帳加總對得上（對帳守恆）。

4 個怕：怕重複扣款（冪等鍵）、怕憑空多花（條件更新防透支/雙花）、怕扣一半（同交易原子）、怕算不清（只追加＋對帳）。

工程細節一句話：金額用整數分、寫路徑保正確讀路徑求快 → 用「點餐單(API)」溝通、轉帳必帶冪等鍵 → 三本筆記本（帳本/帳戶/鑰匙）＋帳本是真相 → 找出塞車環節（分片/快取/Saga）各自拓寬。
