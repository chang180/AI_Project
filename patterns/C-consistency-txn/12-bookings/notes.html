一句話定位 本質是「每房每日一個庫存單位 + 跨日期區間的原子扣減 + 防重複預訂的並發控制」題。 搜尋與展示是讀路徑、易擴展；面試真正考的是 同一房同一晚在高並發下絕不能被兩個人訂走（超賣），以及 下單→保留→付款 的一致性流程。

1. 需求釐清

功能性需求

  • 搜尋可訂房源：地點 + 入住/退房日期 + 人數，回傳「該區間每晚皆可用」的房源。
  • 查看單一房源的房況日曆（哪幾天已被訂滿）。
  • 預訂（下單）：選定房源 + 入住/退房日期區間，原子扣減該區間「每一晚」的庫存。
  • 保留（hold）：下單後先暫留庫存（TTL，例如 10 分鐘），讓使用者去付款；付款成功 → 確認；逾時/失敗 → 釋放庫存可再訂。
  • 取消訂單：釋放對應日期的庫存。

非功能性需求

  • 絕不超賣：同一房同一晚只能被一個有效訂單佔用（強一致，這是首要約束）。
  • 下單低延遲：交易/鎖路徑 p99 < 300ms，避免使用者久候。
  • 讀遠多於寫：瀏覽/搜尋 QPS 遠高於下單（可達 1000:1），讀路徑可放寬一致性。
  • 高可用：搜尋掛掉影響轉換率，下單掛掉直接損失營收。
面試開場必問 「庫存的單位是什麼？」短租通常是「每房每日一個單位」（房間 × 日期 = 一格），預訂是對「連續多格」的原子佔用；連鎖飯店則可能是「房型 × 日期 = N 個單位」（同房型有多間）。先講清楚才知道扣減邏輯與鎖粒度。

2. 規模估算

指標假設推算
房源數500 萬間每間維護未來 ~365 格日曆
日曆庫存格500 萬 × 365 天~18 億格（每格 ~50B ≈ 90 GB，可分片）
搜尋（讀）1 億次/日~1150 讀 QPS（峰值 ×5 ≈ 6k）
下單（寫）500 萬筆/日~58 寫 QPS（峰值 ×5 ≈ 300）
讀寫比搜尋 : 下單約 20:1 ~ 1000:1（依熱門度）
關鍵洞察寫 QPS 不高（幾百），難點不在吞吐而在正確性——同一格在並發下的互斥。讀路徑大、可用快取/讀副本並接受短暫不一致（搜尋結果偶爾顯示「剛被訂走」的房很正常，最終在下單時用交易把關）。

3. API 設計

# 搜尋可訂房源（讀，高頻，可走讀副本/快取）
GET /api/v1/search?city=Tokyo&checkin=2026-06-01&checkout=2026-06-04&guests=2
→ 200 { "rooms": [ { "room_id": "R123", "price": 4200, "nights": 3 }, ... ] }

# 查單一房源房況日曆（讀）
GET /api/v1/rooms/{room_id}/calendar?from=2026-06-01&to=2026-06-30
→ 200 { "booked_dates": ["2026-06-02","2026-06-05"] }

# 建立保留（寫，進交易+鎖；成功則暫留庫存 + 回 TTL）
POST /api/v1/holds
Body: { "room_id": "R123", "checkin": "2026-06-01", "checkout": "2026-06-04" }
→ 201 { "hold_id": "H789", "expires_at": "2026-06-01T10:10:00Z" }
→ 409 { "error": "部分日期已被預訂" }          # 防超賣：區間有任一晚不可用即整筆拒絕

# 確認（付款成功後，把 hold 轉正式訂單）
POST /api/v1/holds/{hold_id}/confirm
Body: { "payment_token": "tok_xxx" }
→ 200 { "booking_id": "B456", "status": "CONFIRMED" }

# 取消 / 付款失敗 → 釋放
DELETE /api/v1/holds/{hold_id}     # 或逾時由背景工作自動釋放
為何拆成 hold → confirm 兩段付款是慢的外部呼叫（數秒），不能讓它整段佔著 DB 交易/行鎖。先在短交易內「原子佔位」拿到 hold（含 TTL），釋放鎖；付款在交易外進行；成功再用第二個短交易 confirm。這樣鎖只持有毫秒級。

4. 資料模型

rooms
  room_id      VARCHAR PRIMARY KEY
  host_id      BIGINT
  city         VARCHAR
  base_price   INT

-- 核心：每房每日一格庫存（room_id + date 唯一）
room_inventory
  room_id      VARCHAR
  date         DATE
  status       ENUM('OPEN','HELD','BOOKED')   -- 該晚狀態
  hold_id      VARCHAR NULL                    -- HELD 時指向哪筆保留
  version      INT                             -- 樂觀鎖用
  PRIMARY KEY (room_id, date)                  -- 唯一鍵天然防同格重複佔用

holds (暫留，有 TTL)
  hold_id      VARCHAR PRIMARY KEY
  room_id, checkin, checkout
  expires_at   TIMESTAMP                       -- 逾時由背景掃描釋放
  status       ENUM('HELD','CONFIRMED','RELEASED')

bookings (確認後的正式訂單)
  booking_id, hold_id, user_id, room_id, checkin, checkout, status, created_at
關鍵設計：把「日期區間」攤平成「每日一列」不要把可訂與否塞在一個 rooms 欄位。攤成 room_inventory(room_id, date) 後，預訂 6/1–6/4 就是鎖/更新 6/1、6/2、6/3 三列（退房日不佔），(room_id, date) 主鍵的唯一性本身就是防超賣的最後一道防線。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 搜尋(讀) ─▶ 搜尋服務 ─▶ 讀副本 / 快取(房況)      （最終一致，容忍短暫過期）
                                  ▲
                                  │ 變更事件
 下單(寫) ─▶ 預訂服務 ─(交易+鎖)─▶ 庫存日曆 DB(room_inventory)
                │                       ▲
                ▼                       │ TTL 到期釋放
            建立 hold(TTL) ─▶ 付款服務 ─┴─▶ confirm / 釋放
                                  │
                                  └─(失敗)─▶ 釋放庫存（status 回 OPEN）
  1. 搜尋服務：無狀態、查讀副本/快取的房況索引；可用性過濾允許最終一致（下單時才強校驗）。
  2. 預訂服務：關鍵路徑，開 DB 交易，鎖住區間每一格，全部可用才原子扣減為 HELD。
  3. 付款服務 + TTL 釋放器：付款在交易外；背景工作掃描逾期 hold 釋放庫存（OPEN）。

6. 核心元件深入

區間可用性檢查 + 原子扣減（防超賣核心）

在單一交易內，對區間每一晚做「檢查皆可用 → 一次扣減」，不能先全查再全寫（那中間會有競態）。兩種主流做法：

-- 悲觀鎖：先鎖住區間每一格，鎖內判斷，再更新
BEGIN;
SELECT date, status FROM room_inventory
  WHERE room_id='R123' AND date IN ('2026-06-01','2026-06-02','2026-06-03')
  FOR UPDATE;                       -- 鎖住這幾列，其他交易等待
-- 應用層確認三晚皆為 OPEN，否則 ROLLBACK 回 409
UPDATE room_inventory SET status='HELD', hold_id='H789'
  WHERE room_id='R123' AND date IN (...) AND status='OPEN';
COMMIT;

-- 樂觀鎖（條件更新 / CAS）：不先鎖，直接條件式扣減
UPDATE room_inventory SET status='HELD', hold_id='H789', version=version+1
  WHERE room_id='R123' AND date IN (...) AND status='OPEN';
-- 若 affected_rows != 區間天數 → 有人搶先，ROLLBACK 回 409

悲觀鎖 vs 樂觀鎖（本題核心取捨）

  • 悲觀鎖 SELECT FOR UPDATE：適合高衝突熱門房源；先排隊鎖住，少做白工，但持鎖期間阻塞其他人、有死鎖風險（多格要固定順序（如日期升冪）加鎖避免循環等待）。
  • 樂觀鎖 version/CAS：適合低衝突；不持鎖、吞吐高，但衝突時整筆失敗要重試；對「區間多格」要靠 affected_rows == 天數 判定，否則回滾。
  • 實務：多數情境用條件更新（樂觀）即可；極熱門房源（演唱會週末）才考慮悲觀鎖或排隊。

hold / TTL：付款不阻塞庫存

下單先把區間扣為 HELD + 記 expires_at，交易立即提交（鎖只持有毫秒）。付款慢慢做：

  • 付款成功 → confirm：HELD → BOOKED，hold 轉 booking。
  • 付款失敗 / 使用者放棄 / 逾時 → 釋放：HELD → OPEN、清掉 hold_id，該晚可再訂。
  • 釋放靠背景 TTL 掃描器（或 confirm 時惰性檢查 expires_at），避免依賴使用者主動取消。

退房日不佔庫存

6/1–6/4 佔的是 6/1、6/2、6/3 三晚，退房當天 6/4 不佔——這樣 A 訂到 6/4 退、B 從 6/4 入住可無縫銜接，是真實飯店語意（也是常見邊界 bug）。

7. 擴展與瓶頸

階段瓶頸對策
單機 / 起步下單與搜尋打同一庫讀寫分離：搜尋走讀副本，下單走主庫交易
搜尋讀爆可用性查詢量大房況索引快取 + 讀副本；接受最終一致（下單時強校驗）
下單寫熱點熱門房源同格高並發爭鎖(room_id,date) 分片、縮短持鎖時間、樂觀重試、必要時排隊/限流
容量18 億日曆格依 room_id 雜湊分片；過期日期歸檔
逾時釋放延遲大量 hold 待掃描到期時間建索引 + 分批掃描；或惰性釋放（confirm/讀取時順手判斷）

8. 取捨與面試重點

  • 悲觀 vs 樂觀：高衝突用悲觀（少白工、會阻塞）；低衝突用樂觀（高吞吐、需重試）。多格務必固定加鎖順序防死鎖。
  • 強一致 vs 最終一致的分界：下單/扣減=強一致（交易）；搜尋/房況展示=最終一致（快取、讀副本，容忍偶爾顯示已被訂走的房）。
  • hold/TTL vs 直接下單：直接下單把付款放進交易 → 鎖被慢付款卡住、吞吐崩；hold/TTL 拆兩段短交易，鎖只持毫秒，但需處理逾時釋放與「付款成功但 hold 已過期」的補償。
  • 避免熱門房源寫熱點：同一格的並發本質無法平行（必須序列化）→ 縮短臨界區、用條件更新、極端情況排隊/發號入場。
  • 常見追問：
    1. 付款成功但 hold 已 TTL 過期被釋放、且該晚已被別人訂走？→ 退款 + 補償（Saga），或 confirm 前再驗一次可用性。
    2. 跨服務（庫存 / 付款 / 訂單）如何一致？→ Saga 補償交易 + 冪等鍵，避免 2PC 的可用性代價。
    3. 怎麼防同一使用者重複提交？→ confirm 用冪等鍵（hold_id 唯一）。
    4. 為何用 (room_id,date) 主鍵？→ 唯一性是超賣的最後防線，即使應用層有 bug，DB 也擋下重複插入。

9. PHP Demo 說明

本題 demo 聚焦並發防超賣 + 區間原子扣減 + hold/TTL（不是 CRUD），可實際用 PHP 內建伺服器跑起來：

  • src/InventoryStore.php　每房每日庫存日曆（JSON 模擬），用 flock(LOCK_EX) 模擬 DB 交易/行鎖，臨界區內完成「區間檢查 + 原子扣減」。
  • src/BookingService.php　訂房（房 + 入住/退房區間）→ 鎖內檢查區間每晚皆 OPEN 才扣為 HELD（防超賣）；hold（保留 + TTL）→ confirm / 逾時自動釋放；另示意樂觀鎖條件更新。
  • public/index.php　路由：建房況 / 訂房 / hold / confirm / 釋放 / 日曆，並附深色測試頁。
# 啟動
cd demo
php -S localhost:8012 -t public
# 瀏覽器開 http://localhost:8012
# 兩個並發訂同一房同日期 → 只有一個成功，另一個被拒（409 防重複預訂）
# hold 後不付款 → 逾時自動釋放 → 可再訂
誠實聲明 Demo 用檔案鎖（flock）模擬 DB 交易與行鎖、用 JSON 檔模擬庫存日曆表；並發防超賣、區間原子扣減、hold/TTL、樂觀鎖條件更新等系統邏輯為真實可執行。真實系統會改用關聯式資料庫的交易（BEGIN … SELECT FOR UPDATE / UPDATE … WHERE status='OPEN' … COMMIT），核心演算法完全一致。
