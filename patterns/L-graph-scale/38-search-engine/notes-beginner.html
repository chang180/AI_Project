1. 60 秒搞懂它在幹嘛

你在 Google 打「分散式系統」，按下 Enter，不到半秒，它從幾百億個網頁裡，挑出最相關、最可信的幾筆排給你看。

這件事的難，難在兩點：① 網頁多到一台電腦放不下（500 億頁的索引是好幾個 PB，比你硬碟大幾百萬倍）； ② 同樣有提到「分散式系統」的頁有上百萬個，到底誰該排第一？

🔑 一句話 網頁搜尋 = 一座超大型圖書館的查詢櫃台：先把每本書做好「索引頁」（哪個字在哪幾頁）， 查詢時同時問很多分館再合併答案，最後不只看「誰提到這個字」，還看「誰比較有份量」，排出順序給你。

2. 關鍵魔法：書末的「索引頁」

很多人以為搜尋引擎是「拿到你的字，再一頁一頁翻過所有網頁找」。那樣慢到天荒地老。真正的魔法你其實早就見過——

📖 翻到書的最後一頁 拿一本厚厚的教科書，要找「光合作用」在哪。你不會從第 1 頁一頁頁翻到第 800 頁， 你會翻到書末的「索引」：「光合作用 …… 見 P.342, P.510」，直接跳過去。

搜尋引擎事先做的，就是這張超大的索引：把每個「字」對應到「哪些網頁有出現它」。這張表叫做 倒排索引。

「倒排」是因為它把關係反過來了：一般我們想「這一頁有哪些字」（順著看）；索引則是「這個字出現在哪些頁」（倒著看）。 查詢時直接從字查到頁，完全不用翻全部網頁。

💭 停一下：如果不先做索引，每次有人搜尋都要「掃過全部 500 億頁」，會發生什麼事？

看答案 每一次查詢都要讀完整個網際網路一遍，可能要算上好幾天，而且同時有幾萬人在搜尋。這完全不可行。所以一定要事先把索引建好（「預先算好，換查詢飛快」），查詢時只查那張表。這跟「書末索引」是同一招。

3. 把系統拆成 4 個關卡

別被「搜尋引擎」嚇到。整件事其實就是闖 4 關，一關一關來：

1爬：先把網頁「搬回家」

網頁分散在全世界的網站上，搜尋引擎得先派出爬蟲（一支自動程式），順著超連結一頁一頁逛、把內容抓回自己家存起來。

🕷️ 比喻像一個不眠不休的剪報員，每天逛遍所有報紙雜誌，把全部文章剪下來收進倉庫。

👉 這關（爬蟲）其實是第 ㊲ 題的主題。本題假設「文章都已經剪回來了」，我們專心做後面 3 關。

2建索引：做出那張「索引頁」

把每篇抓回來的文章切成一個個字，記下「哪個字出現在哪些網頁、出現幾次」，存成上面講的倒排索引。

📇 比喻就像圖書館員把每本新書讀過，把書裡的關鍵詞抄進卡片目錄，以後別人查這個詞就找得到這本書。

但索引大到一台電腦放不下，所以要切成很多份放在很多台機器上——這就帶到第 3 關。

3查：同時問很多分館，再合併（scatter-gather）

因為索引被拆開放在好幾百、上千台機器（每台叫一個分片 shard），你要找的那個字可能每台機器都存了一部分。怎麼辦？

🏛️ 比喻：問遍所有分館 你要找一本書，但圖書館有 1000 個分館，書可能在任何一個。你同時打電話問全部分館「你那有沒有？有的話最相關的幾本給我」， 然後把大家的回覆收齊、合併，挑出最好的幾本。

這個「同時撒出去問所有分片 → 收齊合併」的動作，工程上叫 scatter-gather（scatter＝撒出去、gather＝收回來）。 每個分片各自挑出自己手上最相關的幾筆回報，協調者再把全部合在一起。

4排序：誰該排第一？（相關 + 有份量）

合併後可能有上百萬筆都「提到了你的字」。最後一關是排順序，靠兩個分數一起算：

  • 相關性（叫 BM25）：這頁跟你查的字多相關——這個字出現幾次？是不是很罕見的關鍵字？（罕見字命中更值錢）
  • 權威 / 人氣（叫 PageRank）：這頁本身多有份量，跟你查什麼無關。下一節專門講它。

最後 總分 = 相關性 + 權威，分數高的排前面。這就是為什麼搜尋結果第一名通常又相關、又是大家信得過的網站。

✅ 四關一句話 ① 爬回網頁 → ② 建好索引頁（字 → 哪些頁）→ ③ 查詢時問遍所有分片再合併 → ④ 用相關 + 人氣排順序。就這樣。

4. 人氣分數：PageRank 是什麼？

第 4 關提到的 PageRank，是 Google 起家的招牌絕活，但概念超好懂——它就是給每個網頁打一個「人氣 / 權威分數」。

⭐ 比喻：被大咖轉發的人比較有份量 社群上，一個人有沒有「份量」，不是看他自己喊多大聲，而是看「誰轉發他」。 被很多大咖（本身也很有份量的人）轉發的人，明顯比被一堆路人帳號轉發的更可信。

網頁也一樣。在網路上，「A 頁放了一個連結指向 B 頁」就等於「A 幫 B 投了一票 / 轉發了 B」。PageRank 的核心精神：

  • 越多網頁連到你 → 你越可信。
  • 而且連到你的那些頁本身越有份量 → 這一票越值錢（大咖的轉發 > 路人的轉發）。

所以一個沒沒無聞的小部落格，就算自稱權威也沒用；維基百科被全世界幾百萬頁連入，自然分數很高。 關鍵是：PageRank 只看「連結關係」，跟你查什麼字無關——它是這頁與生俱來的「江湖地位」。

🔁 為什麼要「反覆算很多輪」？ 這裡有個雞生蛋問題：「要算 B 的份量，得先知道連到 B 的 A 有多少份量；可是要算 A 的份量，又得先知道連到 A 的人……」 解法是大家先平分，然後一輪一輪互相更新：第一輪大家分數一樣，每頁把自己的分數平分給「它連出去的頁」； 收到票的頁分數上升，再進行下一輪……反覆幾十輪後，分數就會穩定下來（不再大變動），那就是最終人氣分。這個反覆過程叫冪次迭代。

因為要掃過整張「誰連誰」的大網（幾兆條連結），這個計算很重、很慢，所以是離線事先算好存起來的， 不會在你按 Enter 那半秒裡現算。查詢時只是查一下這頁的人氣分是多少，很快。

💭 停一下：為什麼搜尋結果不能「只看人氣（PageRank）」就好？

看答案 因為人氣跟「你查的字」無關。維基百科首頁人氣超高，但你搜「我家附近的牛肉麵」，把維基首頁排第一完全沒用。所以一定要同時看「相關性（有沒有講到你要的）」跟「人氣（可不可信）」，兩個一起算才合理。光看人氣會讓萬年熱門頁霸榜；光看相關性，內容農場靠堆關鍵字也能上。

5. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「慢」 你按 Enter 後超過 1 秒沒結果，就會覺得這搜尋引擎好爛。但要同時問上千個分片、還要算分排序——很容易慢。 對策：熱門查詢的結果先快取起來（同樣的問題別人剛問過，直接給上次的答案，跳過整個流程）。能不算就不算。
😱 怕「排錯順序」 搜尋引擎的命脈就是排序好不好。第一名爛，大家就換別家用。所以「相關 + 人氣」要平衡： 太偏人氣 → 萬年大站霸榜、好的新內容被埋沒；太偏相關 → 內容農場堆關鍵字也能上。排序就是搜尋引擎的靈魂。
😱 怕「大到放不下 / 算不動」 500 億頁的索引一台機器放不下、PageRank 的兆級連結一次算不完。 對策：索引切成上千片分散放（查詢時問遍再合併）；PageRank 離線慢慢算好存起來，不擋查詢。放不下就切開、算不動就先算好。

白話導讀，分「建索引（平時準備）」跟「查詢（你按 Enter）」兩條路：

  • 🕷️ 爬蟲把網頁抓回來 → 🧩 切詞建索引，依編號分配到不同分片（shard0、shard1…）。

  • 🧮 收齊合併（gather）→ 用「相關 + 人氣」融合排序 → 把答案存進快取下次更快 → 回給你。

8. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
要索引多少網頁~500 億頁索引大到數百 TB ~ PB 級，一台機器絕對放不下
要切成幾片單片控在 ~50GB → 需要數千片每片放一台機器，再加副本備援
每秒多少查詢峰值 ~10 萬 QPS因為每次查詢都要問遍所有分片，每片承受的壓力＝外部總查詢量
查詢要多快p99 < 200 毫秒問所有分片 ~100ms + 合併 ~30ms + 排序取摘要 ~50ms，要塞進預算內
PageRank 多大500 億節點、數兆條連結太重 → 離線批次算，絕不放在你按 Enter 的那條路上
💡 關鍵洞察：兩個「太大」決定一切 這題的所有設計，都來自兩個「太大」：① 索引太大 → 切片 + 問遍合併（scatter-gather）； ② PageRank 太重 → 離線先算好，查詢時只查一個分數。抓住這兩條，整題就通了。

💭 停一下：為什麼「熱門查詢的快取」這麼划算？

看答案 因為大家搜的東西高度重複（少數熱門關鍵字佔了大半流量）。一旦命中快取，就整條 scatter-gather 全部跳過——不用問上千個分片、不用算分排序，直接回上次的答案。用一點點記憶體，換掉最貴的那段運算，當然划算。

9. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」：

① 查詢（最高頻，就是你按 Enter）
   GET /api/v1/search?q=分散式系統&op=or&alpha=0.7&beta=0.3
   給我：要查的字、AND還OR、相關性vs人氣的比重(α/β)
   我回：排好序的結果 + 各片各撈幾筆 + 是不是來自快取

那個 alpha / beta 是「相關 vs 人氣」的旋鈕：beta 調大 → 更偏好權威大站；alpha 調大 → 更純比相關。可以依查詢類型調。

② 新增 / 更新一篇文件（建索引用）
   POST /api/v1/documents
   給我：{ 網址, 標題, 內文, links:[連到哪些網址] }

特別注意 links 那欄——它記下「這頁連到誰」，就是拿來拼出連結圖、算 PageRank 的原料。

③ 重算 PageRank（離線批次，不在查詢路上）
   POST /api/v1/pagerank   { damping: 0.85 }

④ 看系統狀態（分片分布、快取命中率…）
   GET /api/v1/state

PageRank 是「平時偷偷重算」的，所以它有自己的觸發口，跟你按 Enter 的查詢完全分開兩條路。

10. 系統要記住哪些事？（資料模型）

📒 比喻：幾本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題的筆記本可以這樣理解：

📄文件簿（原始網頁）

每頁一列：編號、網址、標題、內文、連到哪些頁(out_links)。 那欄「連到哪些頁」之後會拿來拼連結圖。

📇分片索引簿（字 → 哪些頁，切成很多本）

文件依編號分到不同片：第 s 片只存「自己那批頁」的：某個字 → 哪些頁、各出現幾次。 另外存每頁的字數（算相關性時要做「長文不該佔便宜」的修正）。

⚠️ 一個關鍵小細節：「罕見度」要看全部，不能只看自己這片 相關性裡有一項是「這個字夠不夠罕見」（罕見字命中更值錢）。但「罕見度」必須以整個網路來算—— 如果每片只用自己手上的資料判斷罕見度，同一個字在不同片會算出不同分數，合併時就比不準了。 所以這個「全域罕見度」要跨所有片共用一份。

🔗連結圖簿 + 人氣分簿

連結圖：哪頁指向哪些頁；人氣分：每頁的 PageRank 分數（離線冪次迭代算好的結果）。 查詢排序時，從這本直接查一頁的人氣分，很快。

⚡快取簿（熱門答案）

(正規化後的查詢字 + α + β) → 上次算好的結果。同樣的問題再來，直接給，跳過整個 scatter-gather。

11. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
單一分片太大 → 查詢變慢把每片控在剛好的大小（如 ~50GB），語料變大就多切幾片 + 加副本分攤讀取
問遍上千片，要等最慢的那片才能合併每片只回「自己最好的幾筆」減少傳輸；對慢的片再補問一台副本，誰先回用誰
罕見查詢快取命中不了，每次都要全問一遍靠「切片平行」把工作攤平；熱門詞的索引可提前備好
同一個爆紅查詢被打爆結果快取（命中就跳過整條路）+ 在離使用者近的地方也放一份
「全域罕見度」分片後難維護分散維護，或用近似值（容許一點點誤差，換可擴展）
PageRank 兆級連結算不動離線分散式批次算，週期性重算，絕不進查詢路徑

12. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 相關 vs 人氣（α vs β 的旋鈕） 人氣調太重 → 萬年大站霸榜、好的新內容被埋；相關調太重 → 內容農場靠堆關鍵字也能上。要平衡，實務還會再加「點擊回饋、新鮮度」等更多訊號。
⚖️ 準確 vs 快 每片只回「自己最好的幾筆」是一種近似——理論上可能漏掉某片排第 K+1、但其實全域夠前的頁。換來的是傳輸與合併便宜很多。實務上「幾筆」取夠大就可忽略。
⚖️ 依「文件」分片 vs 依「字」分片 這題依文件分（每次查詢要問所有片，但寫入簡單、不會有熱點）。另一種依字分（查詢只問含該字的片，但熱門字那片會被打爆、多字查詢要跨片求交集，更複雜）。業界主流選前者。
💭 追問：剛爬到的新頁，要等下次重建索引才搜得到嗎？ 不用。新頁先進一個「增量小索引」可立刻被查到，背景再慢慢合併進主分片。PageRank 因為連結圖變動慢，可以容忍延遲、週期性重算就好。
💭 追問：為什麼相關性的「罕見度」一定要用全部資料、不能各片各算？ 因為各片各算的話，同一個字在不同片的分數標準不一樣，合併時就像「不同單位的成績硬湊在一起比」，會排錯。所以罕見度必須是全域共用的一份（或夠好的近似）。

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「分片 + 問遍合併 + 人氣排序」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8038 -t public
# 然後瀏覽器打開 http://localhost:8038
  1. 看文件連結圖與 PageRank：哪頁被連得多，人氣分就高。
  2. 看分片分布：文件被分到哪幾片（這就是「切開放」）。
  3. 同一個查詢，比「只看相關性」vs「相關 + 人氣融合」的排序差在哪。

玩的時候想一下把某一頁多加幾個「連到它」的連結，再重算 PageRank，看它的人氣分是不是上升、排名是不是往前——這就是「被越多頁連入越可信」在運作。把 β 調到很大，看是不是高人氣頁開始霸榜——這就是排序取捨。

14. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有兩個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 兩個檔 = 兩個小幫手 ShardedIndex（圖書館總櫃台：管那一堆分館索引、撒查詢、收答案、融合排序、記快取）→ PageRank（人氣評分員：在「誰連誰」的大網上反覆算，算出每頁的江湖地位）。 平時 PageRank 先把人氣分算好存著；你一查詢，總櫃台就撒給各分館、收齊、配上人氣分排好給你。

1倒排索引分片：文件分進哪個分館（ShardedIndex）

索引大到一台放不下，所以每篇文件進來時，先用編號算出「該歸到哪個分館」，只在那個分館記它的「字 → 出現幾次」。

function 分到哪片(文件編號) {
    return crc32(文件編號) % 分片數     // ← 用雜湊均勻打散到 4 個分館
}

function 加文件(標題, 內文, 連到哪些頁) {
    編號 = 拿一個新編號
    片號 = 分到哪片(編號)
    詞們 = 切詞(標題 + 內文)            // 小寫、去標點、去 the/is 這種虛字
    各分館[片號].每頁字數[編號] = 詞們的數量
    foreach (詞 → 次數 in 統計(詞們))
        各分館[片號].索引[詞][編號] = 次數   // ← 倒排：字 → 哪頁出現幾次
    清掉查詢快取                        // 內容變了，舊答案作廢
}

關鍵就是 crc32(編號) % 分片數 這招：同一個編號永遠算到同一個分館，文件就被均勻打散到各分館，每館只扛自己那批。 這正是「索引切成很多本、各放一個分館」那一關。

2scatter-gather：同時問遍所有分館再合併（ShardedIndex）

查詢進來，總櫃台撒給每個分館各自算相關性（BM25），再把大家的答案收齊合在一起。

function 查詢(字, ...) {
    // ── scatter：問遍每一個分館 ──
    foreach (片號 in 所有分館) {
        這片答案 = 在這片的索引上算 BM25(字)   // ← 各館只翻自己的書
        每片撈幾筆[片號] = 這片答案的筆數       // 看各館各貢獻多少（demo 會顯示）
        foreach (頁 → 分數 in 這片答案)
            全部收集[頁] = 分數                // ── gather：直接收進大袋子 ──
    }
    // 接著進入「融合排序」（下一段）
}

就是前面說的「同時打電話問 1000 個分館 → 收齊彙整」。 demo 為了好懂用一個 for 迴圈「依序」問每個分館；真實系統是平行把查詢送到分散在不同機器的分館——邏輯一模一樣，只是 demo 在一台機器上模擬。

💭 停一下：BM25 裡有一項「這個字夠不夠罕見」，為什麼不能讓各分館用自己手上的資料各算各的？

看答案 因為各館各算的話，同一個字在不同館會有不同的罕見度標準，合併時就像「不同單位的成績硬湊著比」，會排錯。 所以程式裡刻意先算一份全域罕見度（跨所有分館的 df / 平均字數）發給各館共用——這樣各館算出來的分數才能放在一起公平比較。

3PageRank 冪次迭代：反覆算到人氣分穩定（PageRank）

這是離線先算好的人氣分。一句話講原理：一頁的分數 = 連到它的那些頁，各分一點點給它；反覆算很多輪，直到分數不再大變動就停。

每頁先給一樣的分數 = 1 / 總頁數          // 大家先平分
重複很多輪 {
    foreach (頁 p) {
        新分數[p] = 基礎分
        foreach (連到 p 的頁 q)
            新分數[p] += q的分數 / q連出去的頁數   // ← q 把自己的分平均分給它連到的每頁
    }
    變動量 = 所有頁「新分數 − 舊分數」的總和
    舊分數 = 新分數
    if (變動量 < 很小的門檻) break          // ← 穩定了 → 收斂，停！
}

「q 把自己的分數平均分給它連到的每頁」就是「A 連到 B＝A 轉發 B、把份量分一點給 B」。 一輪一輪互相更新，被越多「有份量的頁」連入的頁分數就越爬越高，最後穩定下來不再大動——這個反覆逼近的過程就叫冪次迭代。 那個 基礎分 是給「隨機跳到任一頁」留的小機率（阻尼），避免有些頁分數歸零。

4融合排序 + 查詢快取：相關×人氣排好、答案存著下次秒回（ShardedIndex）

收齊各分館的答案後，最後一步把相關性和人氣兩個分數加權相加，排好順序；而且整個查詢結果會存進快取，同樣的問題下次直接回。

function 查詢(字, α, β) {
    鑰匙 = hash(排序後的字 + op + α + β)
    if (快取有這把鑰匙) return 快取[鑰匙]   // ← 查詢快取：命中就跳過整個 scatter-gather

    全部收集 = scatter-gather(...)          // 前面第 2 段
    最高相關 = 全部收集裡最大的 BM25
    最高人氣 = 所有頁裡最大的 PageRank
    foreach (頁 in 全部收集) {
        相關分 = 頁的BM25  / 最高相關        // 各自正規化到 0~1，才能公平相加
        人氣分 = 頁的人氣  / 最高人氣
        最終分[頁] = α × 相關分 + β × 人氣分  // ← 融合！α 偏相關、β 偏人氣
    }
    依最終分由高到低排序
    快取[鑰匙] = 結果                        // ← 存起來，下次同問題秒回
    return 結果
}

α × 相關 + β × 人氣 就是那個「相關 vs 人氣的旋鈕」：β 調大→權威大站往前；α 調大→更純比相關。 兩個分數要先各自除以最大值正規化到 0~1，否則一個是個位數、一個是 0.0001，加起來人氣會被淹沒。 最後把 (字+α+β) 當鑰匙存結果——這就是前面「熱門答案先存著、同問題直接給」那一關，省掉最貴的整段運算。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
② 建索引 + 切片ShardedIndex分到哪片()（crc32 % 分片數）
③ 問遍分館再合併ShardedIndex查詢() 裡的 scatter / gather 迴圈
人氣分（PageRank）PageRank「反覆算到收斂」的 for 迴圈
④ 融合排序 + 快取ShardedIndexα×相關 + β×人氣 + 鑰匙存快取

👉 重點：真實系統會把「JSON 檔 + 迴圈問各片」換成「分散在上千台機器的分片 + 平行 scatter-gather + 分散式批次算 PageRank」（業界就是 Google／Elasticsearch 那一套）， 但上面這幾段判斷邏輯一字都不用改——分片、scatter-gather、PageRank 冪次迭代、α/β 融合排序，全都是同一套。所以讀懂這個小 demo，你就讀懂了這題的核心。

15. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「書末索引」解釋，搜尋引擎為什麼能秒回，而不是現翻 500 億頁？ 因為它事先建好了「倒排索引」——像書末那張「字 → 在哪幾頁」的表。查詢時直接查這張表，不用掃過全部網頁。預先算好，換查詢飛快。
Q2：索引大到一台機器放不下怎麼辦？查詢時又怎麼運作？ 把索引切成很多片（shard）放在很多台機器。查詢時用 scatter-gather：同時問遍所有片（scatter），每片回報自己最相關的幾筆，再收齊合併（gather）。像同時打電話問 1000 個分館再彙整。
Q3：用「被大咖轉發」的比喻說說 PageRank 是什麼？ PageRank 是網頁的人氣 / 權威分。「A 連到 B」等於「A 轉發了 B」。被越多頁、尤其被越有份量的頁連入，分數越高——就像被大咖轉發比被路人轉發更有份量。它跟你查什麼字無關，是這頁本身的江湖地位。
Q4：為什麼最終排序要「相關 + 人氣」一起算，不能只看一個？ 只看人氣 → 維基首頁這種高人氣頁霸榜，但跟你查的字可能毫不相關。只看相關 → 內容農場堆關鍵字也能上、不可信。所以要兩個一起：又相關、又可信的頁才該排前面。
Q5（工程）：為什麼 PageRank 要「離線事先算好」，不在查詢當下現算？ 因為它要掃過整張「誰連誰」的大網（幾兆條連結）、反覆迭代幾十輪，非常重非常慢。查詢只有幾百毫秒預算，現算來不及。所以離線批次算好存成一張分數表，查詢時只查一個分數。
Q6（工程）：相關性裡的「罕見度」為什麼必須用全部資料來算，不能各片各算？ 因為各片各算的話，同一個字在不同片會有不同分數標準，合併時就比不準、排錯。所以「罕見度」要是全域共用的一份（或夠好的近似），各片算出來的相關性分數才能放在一起比較。
Q7（工程）：熱門查詢的「結果快取」為什麼是最划算的優化？ 因為查詢高度集中，少數熱門詞佔了大半流量。一旦命中快取，就整條 scatter-gather 全跳過——不用問上千片、不用算分排序，直接回上次答案。用一點記憶體換掉最貴的運算。

16. 30 秒回顧

網頁搜尋 = 超大型圖書館的查詢櫃台。（是第 ㉖ 題站內搜尋的「整個網際網路」放大版）

4 個關卡：① 爬回網頁 → ② 建「書末索引」（字 → 哪些頁）→ ③ 查詢時問遍所有分片再合併（scatter-gather）→ ④ 用「相關(BM25) + 人氣(PageRank)」融合排序。

PageRank 一句話：網頁的人氣分，被越多有份量的頁連入越可信（像被大咖轉發），跟查的字無關、離線先算好。

3 個怕：怕慢（熱門查詢用快取跳過）、怕排錯順序（相關與人氣要平衡）、怕大到放不下/算不動（索引切片、PageRank 離線算）。
