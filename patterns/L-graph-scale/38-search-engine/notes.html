一句話定位 本質是「爬 → 索引 → 排序 → 服務」的全鏈題：把第 ㉖ 題的倒排索引 + BM25 放大到「單機放不下」的規模（→ 索引分片 + scatter-gather 查詢）， 並回答 Web 特有的問題——「哪頁更權威？」（→ 在連結圖上算 PageRank）。 最終排序＝相關性（BM25）＋ 權威（PageRank）的融合。

1. 需求釐清

網頁搜尋 vs 站內搜尋

  • 站內搜尋（㉖ Twitter Search）：文件你自己擁有、結構化、無「連結圖」、排序幾乎只看相關性。
  • 網頁搜尋（本題）：文件來自外部、海量、互相超連結。 你得先爬（crawl）回來，還要回答「哪些頁更可信/權威」——同樣相關時， 被很多高品質頁連入的頁該排前面。這就是 PageRank 切入點。

功能性需求

  • 爬取網頁、建立可查詢的索引（爬蟲本身是第 ㊲ 題，本題假設文件已抓回）。
  • 輸入關鍵字 → 回傳相關網頁，依相關性 + 權威排序。
  • 支援布林查詢（AND / OR）、可調相關性與權威的權重。
  • 近即時：新爬到的頁應在可接受延遲內可被搜到。

非功能性需求

  • 低延遲：查詢 p99 < 200ms（含 scatter 到所有 shard + gather 合併）。
  • 高吞吐：查詢 QPS 高（讀遠多於寫），熱門查詢重複度高 → 快取效益大。
  • 可擴展：索引大到單機放不下 → 必須分片水平擴展。
  • 相關性品質：排序好不好是搜尋引擎的命脈（相關性 + 權威）。
面試開場必問 「是站內搜尋還是網頁搜尋？」決定了要不要連結圖與 PageRank。 若是網頁搜尋，接著問規模——大到要分片嗎？要的話 scatter-gather 與全域統計（df/avgdl）的維護就是重點。

2. 規模估算

指標假設推算
網頁規模可索引網頁 ~500 億倒排索引（含位置）約數百 TB ~ PB 級
索引分片數單 shard 控在 ~50GB 以利查詢延遲數百 TB ÷ 50GB ≈ 數千個 shard（再 ×副本）
查詢 QPS~10 萬 QPS（峰值）每查詢 scatter 到所有 shard → shard 內部 QPS = 外部 QPS（每片都被打）
延遲預算p99 200msscatter+各 shard 評分 ~100ms + gather 合併 ~30ms + 融合排序/取摘要 ~50ms
PageRank500 億節點、~數兆條邊離線批次（如 MapReduce/Pregel）算，不在查詢路徑上，數十次迭代收斂
快取命中查詢高度集中（長尾 + 熱門）熱門查詢快取命中率可 30~60%，直接省掉整個 scatter-gather
關鍵洞察 兩個量級決定設計：① 索引太大 → 分片 + scatter-gather； ② PageRank 太重 → 離線預算好，查詢時只是「查一個分數」。 相關性（BM25）即時算、權威（PageRank）離線備好，查詢時融合。

4. 資料模型

# 文件儲存
documents
  doc_id      BIGINT PRIMARY KEY
  url, title, body
  out_links   doc_id[]            -- 連到哪些頁（建連結圖用）

# 分片倒排索引：文件依 hash(doc_id) % N 分到 shard
shard[s].postings   term → { doc_id → tf }     -- 各 shard 只存自己那批
shard[s].doc_len    doc_id → 詞數              -- BM25 長度正規化用
# 全域統計（跨 shard）：df(term)、avgdl、N  -- BM25 的 idf 需全域視角

# 連結圖 + PageRank（離線算）
link_graph          doc_id → [被指向的 doc_id...]
pagerank            doc_id → 權威分數（冪次迭代收斂結果）

# 查詢結果快取
query_cache         hash(normalized_query, α, β) → results
為何 BM25 的 df/avgdl 要「全域」 BM25 的 idf 衡量「一個詞在整個語料多稀有」。若每個 shard 只用自己局部的 df， 同一個詞在不同 shard 會有不同 idf → 跨 shard 分數不可比、合併會錯。 所以 df/avgdl 必須是全域（分散維護或近似），各 shard 才能算出可合併的 BM25。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 建索引（寫）
   Crawler ──▶ Indexer(切詞) ──▶ 分片倒排索引 [shard0][shard1]...[shardN]
                                  │
                  連結圖 ──▶ PageRank(冪次迭代, 離線) ──▶ 權威分數表

 查詢（讀）
   Client ─▶ 解析(AND/OR)+查快取 ─(miss)▶ 協調者
                                            │  scatter（送到所有 shard）
                          ┌─────────┬───────┴───────┬─────────┐
                       shard0     shard1          ...        shardN
                       局部BM25   局部BM25                   局部BM25
                          └─────────┴──────gather──────┴─────────┘
                                            │
                       融合排序  score = α·BM25 + β·PageRank
                                            │
                                      寫入查詢快取 ─▶ 回 Client
  1. 建索引路徑：Crawler 抓頁 → Indexer 切詞 → 依 docId 雜湊寫進對應 shard 的倒排索引；連結另存連結圖。
  2. PageRank（離線）：在連結圖上冪次迭代算每頁權威分，存成分數表，不在查詢路徑。
  3. 查詢路徑：解析 → 查快取 →（miss）協調者 scatter 到所有 shard → 各 shard 局部 BM25 → gather 合併 → 融合 BM25+PageRank 排序 → 回填快取。

6. 核心元件深入

倒排索引分片與 scatter-gather

  • 分片：文件依 hash(doc_id) % N 散到 N 個 shard，每個 shard 只持有自己那批文件的 term → postings。好處是索引可水平擴展、寫入分散。
  • scatter（散開）：因為某個 term 的文件分散在所有 shard，協調者必須把查詢送到每一個 shard（document-partitioned 分片下，每片都要查）。
  • gather（收集）：各 shard 在局部索引上用全域 df/avgdl算 BM25，取出局部 top-K 候選回傳，協調者合併成全域候選。
  • 取捨：document-partition（依文件分片，本題採用）→ 每查詢打所有 shard、但寫入簡單；term-partition（依詞分片）→ 查詢只打含該詞的 shard、但熱門詞 shard 易成熱點、且多詞查詢要跨片交集，工程更複雜。業界主流是 document-partition。

PageRank：冪次迭代（本題演算法重點）

把 Web 看成有向圖（頁=節點、連結=邊）。一頁被越多「高權威」頁連入，自己越權威。迭代公式：

PR(p) = (1 - d)/N  +  d · Σ_{q→p}  PR(q) / outDeg(q)

 d        = 阻尼係數(0.85)：隨機衝浪者「沿連結走」的機率
 (1-d)    = 「厭倦後隨機跳到任一頁」的機率（保證收斂、處理孤島）
 q→p      = 所有指向 p 的頁
 outDeg(q)= q 的出連結數（把自己的分數平均分給連到的每頁）
  • 冪次迭代：初始全部 PR=1/N，反覆套公式，直到分數變化（L1 距離）< ε 收斂。數學上是在逼近轉移矩陣的主特徵向量。
  • dangling node（無出連結的頁，outDeg=0）：分數無處可流會「漏掉」，處理法是把所有 dangling 的質量收集起來、平均灑回全圖（等價它連向所有頁），確保總質量守恆。
  • 離線算：迭代成本高（每輪掃全圖），所以批次預算好存表，查詢時只是查一個分數。

BM25 + PageRank 融合排序

  • BM25 回答「這頁跟查詢多相關」（詞頻 tf、稀有度 idf、長度正規化）。
  • PageRank 回答「這頁本身多有份量」（與查詢無關的全域權威）。
  • 融合：兩者量綱不同，先各自正規化到 [0,1]，再 score = α·normBM25 + β·normPageRank。 調大 β → 權威頁上升（適合導航類查詢）；調大 α → 純比相關（適合長尾精準查詢）。

查詢理解

  • 解析布林運算（AND=交集、OR=聯集）、正規化（小寫、去 stopwords、切詞）。
  • 進階（本題點到為止）：拼字校正、同義詞擴展、查詢意圖分類，都在送進索引前的「查詢重寫」階段。

結果快取

以 hash(正規化查詢 + α + β) 為鍵存結果。查詢分布高度集中（熱門詞重複度高），命中即跳過整個 scatter-gather，是最便宜的延遲優化。需配合 TTL / 索引更新時失效。

近即時索引

新爬到的頁先進「增量小索引」（可立即查），背景再合併進主分片並重算受影響的統計；PageRank 則週期性批次重算（連結圖變動慢，容忍延遲）。

7. 擴展與瓶頸

瓶頸症狀對策
索引分片數單 shard 太大 → 查詢慢；太多 → gather 合併成本高把單 shard 控在可接受查詢延遲的大小（如 ~50GB），分片數隨語料線性增長 + 加副本擴讀
跨 shard 合併scatter 到上千 shard，gather 要等最慢的各 shard 只回局部 top-K（減少傳輸）；尾延遲用 hedged request / 部分結果先回
長尾查詢罕見查詢快取無法命中、每次都全 scatter提前計算/快取熱門詞 postings；長尾靠分片平行攤平
熱門查詢同一查詢被打爆查詢結果快取（命中跳過整條路徑）+ 邊緣快取
全域統計維護df/avgdl 是全域的，分片後難維護分散維護或近似全域 df（容許小誤差換可擴展）
PageRank 規模兆級邊、算不動離線分散式（Pregel/MapReduce），週期重算，不進查詢路徑

8. 取捨與面試重點

  • 相關性 vs 權威（α vs β）：β 太大 → 萬年熱門頁霸榜、長尾好內容被埋；α 太大 → 內容農場靠關鍵字堆砌也能上。實務還會加上點擊回饋、新鮮度等更多訊號，BM25+PageRank 只是骨架。
  • 精確 vs 延遲：各 shard 只回局部 top-K 是「近似」——理論上可能漏掉某 shard 第 K+1 名其實全域夠前的文件；換來的是傳輸與合併大幅變便宜。實務 K 取夠大即可忽略。
  • 分片數 vs 合併成本：分片越多越能平行、單片越快，但 gather 要協調的片越多、尾延遲風險越高。是一個 U 形權衡。
  • document-partition vs term-partition：前者每查詢打所有 shard 但簡單、無熱點；後者查詢只打相關 shard 但熱門詞成熱點、多詞查詢跨片複雜。主流選前者。
  • 常見追問：BM25 的 idf 為何要全域？（跨 shard 可比）。PageRank 為何離線？（迭代太重、連結圖變動慢）。dangling node 怎麼處理？（質量灑回全圖）。如何近即時？（增量索引 + 背景合併）。

9. PHP Demo 說明

本題 demo 聚焦全鏈核心邏輯（也是面試真正的考點），可用 PHP 內建伺服器跑起來：

  • src/PageRank.php　在連結圖上用冪次迭代算 PageRank（damping 0.85、處理 dangling node、L1 收斂判定）。
  • src/ShardedIndex.php　倒排索引分片（crc32(docId)%N）+ scatter-gather 查詢（各 shard 用全域 df/avgdl 算局部 BM25）+ BM25×PageRank 融合排序（α/β 可調）+ AND/OR 解析 + 查詢結果快取。
  • public/index.php　路由：GET /（測試頁）、POST /api/doc、POST /api/pagerank、GET /api/search?q=&alpha=&beta=、GET /api/state；預設塞數篇互相連結的範例文件並算好 PageRank。
# 啟動
cd demo
php -S localhost:8038 -t public
# 瀏覽器開 http://localhost:8038：
#   ① 看文件連結圖與 PageRank　② 看分片分布
#   ③ 同一查詢比「純 BM25」vs「BM25+PageRank 融合」的排序變化
#   ④ 自己查詢調 α/β、AND/OR，第二次同查詢看 cached:true
誠實聲明 Demo 為單機模擬多個 shard（用迴圈代替「平行送到不同機器」），規模小、用 JSON 檔當索引。 但分片、scatter-gather、BM25 評分、PageRank 冪次迭代、BM25×PageRank 融合排序、查詢快取 都是真實可執行的系統邏輯。零依賴：僅 PHP 內建 json + hash， 切詞用 ASCII strtolower + preg_split，共享狀態用 flock。
