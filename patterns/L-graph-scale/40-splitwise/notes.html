㊵ Splitwise 帳務分攤：分攤、淨額守恆與最小現金流

L 圖/演算法 建議 40 分鐘 Design Splitwise (Expense Sharing)

一句話定位 本質是「群組帳本 + 淨額聚合 + 圖上最小現金流化簡」。 記一筆支出很簡單；面試真正考的是 多人多筆欠款如何化簡成最少筆轉帳、 金額如何避免浮點誤差、淨額為何必須守恆（和=0），以及重送如何不重複入帳。

1. 需求釐清

功能性需求

  • 建立群組與成員；在群組內記錄一筆支出：誰付款、總金額、在哪些人之間分攤。
  • 三種分攤方式：均分（equal）、指定金額（exact）、百分比（percentage）。
  • 淨額查詢：每人對群組的淨餘額（正=應收 creditor、負=應付 debtor）。
  • 債務簡化（還款建議）：把錯綜複雜的多對多欠款，化簡成最少筆數的轉帳。
  • 支援多幣別（依當下匯率換算成群組基準幣記帳）。

非功能性需求

  • 金額絕對正確：帳務系統，一分錢都不能算錯 → 用整數分，禁用浮點。
  • 守恆：群組內所有人淨額相加恆為 0（任何時刻皆成立，可被驗證）。
  • 冪等：網路重送、重試不可造成重複入帳。
  • 強一致帳本：餘額是讀「自己的寫」的場景，使用者加完支出立刻要看到正確餘額。
  • 可擴展：少量超大群組（如百人旅遊團）與海量小群組並存。
面試開場必問 「分攤方式要支援到哪幾種？」「還款建議是要『最少筆數』還是『保留原始欠款關係』？」 這兩個決定了核心演算法是否要寫最小現金流化簡。

2. 規模估算

指標假設推算
使用者5000 萬多為小群組（2–10 人）
群組數人均 5 群~2.5 億群組
新增支出（寫）1000 萬/日~115 寫 QPS（峰值 ×3 ≈ 350）
餘額/簡化查詢（讀）每筆支出後多人多次查讀寫比 ~20:1 → ~2.3k 讀 QPS
儲存（支出明細）每筆 ~300 bytes，永久保留1000萬×365×5×300B ≈ 5.5 TB
大群組少數百人團 + 千筆支出淨額重算與簡化的熱點
關鍵洞察絕大多數群組很小（淨額重算與簡化都是毫秒級），瓶頸在少數超大群組與熱門群組的並發寫 → 設計圍繞「每群獨立帳本 + 餘額增量維護 + 寫入冪等」。

3. API 設計

# 建立群組
POST /api/v1/groups        { "name": "日本旅遊", "members": ["Amy","Ben","Cat","Dan"], "currency": "TWD" }

# 新增支出（寫，核心）
POST /api/v1/groups/{gid}/expenses
Header: Idempotency-Key: 7f3a...        # 冪等鍵，重送不重複入帳
Body: {
  "payer": "Amy", "amount": 12000,      # 整數分（$120.00）
  "type": "equal|exact|percentage",
  "among": ["Amy","Ben","Cat","Dan"],   # equal：人名清單
                                        # exact：{ "Cat":4000, "Dan":2000 }（總和=amount）
                                        # percentage：{ "Amy":40, "Ben":30, ... }（總和=100）
  "desc": "晚餐", "currency": "TWD"
}
→ 201 { "id": 42, "shares": { "Amy":3000, "Ben":3000, "Cat":3000, "Dan":3000 } }

# 每人淨餘額（讀）
GET /api/v1/groups/{gid}/balances
→ { "net": { "Amy":-3000, "Cat":17000, ... }, "sum": 0, "conserved": true }

# 債務簡化：最少筆數還款建議（讀）
GET /api/v1/groups/{gid}/settle
→ { "transfers": [ {"from":"Dan","to":"Cat","amount":8000}, ... ], "before": 8, "after": 3 }

4. 資料模型

groups
  gid          BIGINT PK
  name         TEXT
  base_currency CHAR(3)              -- 群組基準幣

group_members
  gid, user_id                       -- 多對多

expenses (append-only 帳本，永久保留)
  id           BIGINT PK
  gid          BIGINT
  payer        user_id
  amount_cents BIGINT                -- 整數分，絕不用 float
  currency     CHAR(3)
  fx_rate      BIGINT                -- 換算成基準幣的匯率（定點，如 ×10^6）
  type         ENUM(equal,exact,percentage)
  desc         TEXT
  idem_key     VARCHAR(64) UNIQUE    -- 冪等鍵，DB 唯一索引擋重複
  created_at   TIMESTAMP

expense_splits (每筆支出拆出的每人分攤)
  expense_id, user_id, share_cents   -- sum(share)=amount_cents（守恆）

balances (聚合快取，可由 expenses 重算)
  gid, user_id, net_cents            -- 正=應收 負=應付；增量維護
為何拆 splits 表支出與「每人分攤」拆開，讓淨額聚合（付款額 − 分攤額）可用 SQL 聚合或增量維護；balances 是可重算的快取，真相永遠是 append-only 的 expenses + splits。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
           ┌────────────┐      ┌──────────────┐     ┌──────────────┐
 加支出 ───▶│ 交易服務     │────▶│ 帳本(append) │────▶│ 淨額聚合      │
 (寫,冪等)  │ 算分攤/匯率  │     │ expenses     │     │ creditor/    │
           └────────────┘      │ +splits      │     │ debtor (sum=0)│
                                └──────────────┘     └──────┬───────┘
                                                            ▼
 查還款 ───────────────────────────────────────▶  ┌──────────────┐
 (讀)                                              │ 債務簡化      │
                                                   │ 最小現金流    │
                                                   │ → 最少轉帳    │
                                                   └──────────────┘
  1. 交易服務：驗證分攤（總和=金額 / 百分比=100）、多幣別換算、產生每人分攤，靠冪等鍵去重後寫入帳本。
  2. 帳本：append-only 的 expenses + splits，是唯一真相來源（source of truth）。
  3. 淨額聚合：每人 淨額 = Σ墊付 − Σ分攤；可增量維護或重算，恆守恆（sum=0）。
  4. 債務簡化：讀時把淨額圖跑最小現金流，回最少筆數轉帳建議。

6. 核心元件深入

三種分攤方式

  • 均分 equal：金額 ÷ 人數；不能整除的餘數（分）逐分配給前面的人，保證 Σ分攤 = 總額，不丟也不多一分。
  • 指定金額 exact：直接給每人應分攤的分；驗證 Σ = 總額，否則拒絕。
  • 百分比 percentage：先按比例下取整，再用最大餘數法把因下取整少掉的分補回，仍保證 Σ = 總額。

淨額守恆（sum = 0）

每筆支出對「付款人」加上整筆金額、對每個「分攤者」減去其分攤額。 因為 Σ分攤 = 總額，每筆支出對全體淨額的貢獻是 +總額 − Σ分攤 = 0。 所以任意筆數支出後，全體淨額和恆為 0。這是帳務系統的不變式（invariant）， 每次計算都應主動驗證 sum===0；一旦不為 0 代表分攤計算有 bug（典型是浮點或餘數沒處理好）。

債務簡化 — 最小現金流（核心考點）

多對多欠款（Amy 欠 Ben、Ben 欠 Cat、Cat 欠 Amy…）可化簡。先把每人壓成一個淨額， 原始的個別欠款邊就不重要了——重要的只有「誰最後該收多少、誰該付多少」。

貪婪法（每輪結算最大債主 ↔ 最大欠款人）：
  while 還有非零淨額:
    c = 淨額最正者(最大債主)，d = 淨額最負者(最大欠款人)
    t = min(c, -d)
    記一筆轉帳 d → c，金額 t
    c -= t; d += t            # 至少有一人歸零
  N 人 → 最多 N-1 筆轉帳

Demo 中 4 人原始 8 條欠款邊 → 簡化成 3 筆轉帳（皆轉給唯一債主 Cat）。 注意：「最少筆數還款」嚴格來說是 NP-hard（可化簡為子集和分組）， 貪婪法不保證理論最優，但實務上接近最優且 O(N log N)，是業界常用做法； 面試講清楚「貪婪近似 + N-1 上界 + 為何不追求絕對最優」即可。

整數分避免浮點

0.1 + 0.2 !== 0.3 在浮點下成立，帳務系統絕不能用 float。 一律以整數分（cents）儲存與運算，顯示時才除以 100。分攤的餘數用整數邏輯逐分配， 確保「加總無誤差、守恆成立」。

冪等去重

客戶端重送/重試在分散式下無法避免。每筆寫入帶 Idempotency-Key， 服務端用 DB 唯一索引或已見鍵集合判斷：同鍵第二次到達 → 回傳第一次的結果（replay），不重複入帳。

多幣別匯率

支出可用任意幣別，記帳時依當下匯率換算成群組基準幣，並把原幣別與匯率一起存進帳本（保留可追溯）。 匯率用定點數（如 ×10⁶ 的整數）避免浮點；換算後仍以整數分入帳。

7. 擴展與瓶頸

瓶頸情境對策
大群組淨額重算百人團、千筆支出，每次全表掃描增量維護 balances：每筆支出只改動相關幾人的淨額，不全重算
熱門群組並發寫同群多人同時加支出，餘額互相覆蓋每群獨立帳本 + 行鎖 / 樂觀鎖；append-only 寫不衝突，聚合分開做
歷史明細查詢長期群組支出很多、分頁查expenses 依 gid 分片 + 游標分頁；冷資料下沉
簡化即時性每次查 settle 都重跑演算法淨額變動才重算並快取 settle 結果；O(N log N) 對小群組可即時
跨群分片2.5 億群組單庫裝不下依 gid 雜湊分片；同群所有資料同片，聚合不跨片

8. 取捨與面試重點

  • 即時簡化 vs 顯示原始欠款：最小現金流轉帳數最少（少跑幾趟銀行），但「Amy 直接付 Cat」可能看不出原本是「Amy 欠 Ben、Ben 欠 Cat」→ 有些 App 同時提供「原始明細」與「簡化建議」。
  • 精確 vs 效能：絕對最少筆數是 NP-hard；貪婪近似 O(N log N) 換來實務上夠好的結果，正確取捨。
  • 強一致帳本 vs 最終一致：餘額是「讀自己的寫」→ 帳本與聚合需強一致（加完立刻看到）；跨群統計類報表可最終一致。
  • balances 快取 vs 永遠重算：balances 是可丟棄的快取，真相永遠是 append-only 帳本；任何時候都能由帳本重算驗證守恆。
  • 常見追問：負金額/退款怎麼記？→ 反向支出。改一筆舊支出？→ 不可變帳本用「沖銷 + 重記」而非就地改。匯率波動？→ 記帳當下鎖定匯率。

9. PHP Demo 說明

本題 demo 著重帳務系統的真正考點：三種分攤、淨額守恆、債務簡化，皆可實際用 PHP 內建伺服器跑起來：

  • src/Ledger.php　帳本核心：三種分攤計算（整數分、餘數逐分配）、淨額聚合（守恆驗證）、債務簡化（最小現金流貪婪）、冪等去重；JSON 檔 + flock 保證並發原子性。
  • public/index.php　路由 + 測試頁：建成員、加支出（選分攤方式）、看各人淨餘額（sum=0）、一鍵看最少轉帳。
# 啟動
cd demo
php -S localhost:8040 -t public
# 瀏覽器開 http://localhost:8040 加幾筆不同分法的支出，看淨額和=0、債務簡化的最少轉帳筆數
誠實聲明 Demo 是單機帳本（JSON 檔模擬 DB、檔案鎖模擬交易），不含分片、多區域、真正的多幣別匯率源。 但分攤計算、淨額守恆（sum=0）、債務簡化（最小現金流）、冪等去重、整數分皆為真實可執行邏輯， 這些正是面試的核心考點。
