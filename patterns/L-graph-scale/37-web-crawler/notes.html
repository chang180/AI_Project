㊲ 網頁爬蟲 Web Crawler：從種子到十億頁的抓取管線

L 圖/演算法 建議 48 分鐘 Design a Web Crawler

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「一個跑在巨大圖（網頁超連結圖）上的分散式 BFS」： 從種子出發、不斷抓頁→抽連結→去重→排程再抓。面試真正考的不是「怎麼下載一個網頁」，而是 幾十億 URL 如何去重（記憶體）、如何當個有禮貌的訪客（politeness）、 如何分散式分片又不互相打架、以及頁面會變，何時重抓（freshness）。

1. 需求釐清

功能性需求

  • 給一組種子 URL，沿超連結遞迴抓取網頁，下載並儲存內容（供搜尋引擎建索引 / 監控頁面變動）。
  • 從抓到的 HTML 抽出外連 link，加入待爬清單（frontier）。
  • URL 去重：同一頁不重複抓（網頁圖充滿回連與環）。
  • 遵守 robots.txt 與 crawl-delay（禮貌性，不打垮對方站台）。
  • 重抓：頁面會變，依新鮮度（freshness）排程重新抓取。

非功能性需求

  • 可擴展：從幾千頁到十億級頁面，水平擴展抓取與去重。
  • 禮貌：對單一 domain 的抓取速率受控（這是法律/道德與封 IP 風險的核心）。
  • 強健：能容忍壞頁、超時、重複、爬蟲陷阱（無限 URL）。
  • 新鮮：重要/常變頁面要更勤重抓。
  • 可去重 / 不漏：去重要省記憶體又不能漏抓太多。
面試開場必問 「爬來做什麼？」決定取捨。建搜尋索引＝廣度優先、覆蓋率與新鮮度重要； 監控特定站＝深度/針對性、重抓頻率高；訓練語料＝量大、去重與品質過濾重要。 再問「範圍（全網 vs 特定站）、深度上限、要不要尊重 robots、單機還是分散式」。

2. 規模估算

指標假設推算
目標頁數10 億頁，1 個月爬完10⁹ / (30×86400) ≈ ~400 頁/秒（峰值 ×3 ≈ 1200）
頻寬每頁平均 100 KB400 × 100KB ≈ 40 MB/s ≈ 320 Mbps 下載
內容儲存每頁壓縮後 ~30 KB10⁹ × 30KB ≈ 30 TB（單輪）
URL 去重結構10 億 URL，精確 set 每筆 ~64 B≈ 64 GB（放不下單機）→ bloom：1% 偽陽性僅需 ~1.2 GB
Frontier 規模外連分支因子 ~10frontier 可瞬間爆量 → 需可溢寫到磁碟/分散式佇列
重抓頻率新聞頁每天、靜態頁每月依頁面變更率分層排程，攤平重抓流量
關鍵洞察 兩個數字主導整題：去重結構 64 GB 放不進單機記憶體（→ bloom filter / 分散式去重）、 對單一 domain 不能太快（→ politeness / 分片）。其餘都是工程化。

3. API / 介面設計

爬蟲多半是內部批次/常駐系統而非對外 API，介面以「元件間契約」為主：

# 提交種子 / 控制（管理面）
POST /crawler/seeds          Body: { "urls": ["https://a.com"], "max_depth": 5 }
POST /crawler/jobs/{id}/pause | resume
GET  /crawler/stats          → { fetched, frontier_size, dedup_size, qps, per_domain }

# 內部元件契約（概念）
Frontier.enqueue(url, priority)      # 加入待爬（依優先序/分片）
Frontier.dequeue() -> url            # 取出下一個可抓的 url（已過 politeness 檢查）
Dedup.seen(url) -> bool              # bloom 先擋，命中再查精確集合
Fetcher.fetch(url) -> (status, html) # 受 per-domain 速率限制
Parser.extractLinks(html) -> url[]   # 正規化後回傳外連
Scheduler.scheduleRecrawl(url, freshness)

4. 資料模型

frontier（待爬佇列，依 domain 分片；可溢寫磁碟）
  shard_id, url, depth, priority, enqueued_at

url_seen（去重；兩層）
  bloom_filter        -- 記憶體中，省空間的近似集合（可能偽陽性）
  url_fingerprints    -- 精確集合（hash(url)），bloom 命中時二次確認

pages（已抓內容中繼）
  url_hash PRIMARY KEY, url, fetched_at, content_hash,
  next_recrawl_at, fetch_count, http_status

domains（per-domain 禮貌狀態）
  domain PRIMARY KEY, last_fetch_at, crawl_delay, robots_rules, shard_id
為何 per-domain 狀態要集中 politeness 是「對同一 host 的速率限制」。若同一 domain 的 URL 散落在多台抓取機，就會多台同時猛打它。 所以依 domain 雜湊分片，把同一 domain 的所有 URL 與其 last_fetch 狀態集中到同一個 shard/worker， 速率限制才有意義。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
  種子 ──▶ ┌──────────┐   取出   ┌─────────┐  HTML  ┌────────┐
           │ Frontier │ ───────▶ │ Fetcher │ ─────▶ │ Parser │
           │（分片佇列）│ ◀─新URL── │（限速）  │        │（抽link）│
           └────┬─────┘          └────┬────┘        └───┬────┘
                │                      │ politeness       │ links
          回填  │                per-domain last_fetch    │
                │                                          ▼
                │            ┌───────────────┐      ┌──────────┐
                └───(新)──── │ URL 去重        │ ◀─── │ 正規化    │
                  bloom+精確  │ bloom + 精確set │      └──────────┘
                             └───────────────┘
   內容 ──▶ 內容儲存（壓縮/物件儲存）；已抓頁 ──▶ 重抓排程（freshness）──▶ 回 Frontier
  1. Frontier：待爬 URL 佇列，依 domain 分片；BFS 用 FIFO，重要性爬蟲用優先佇列。
  2. Fetcher：下載頁面，受 per-domain crawl-delay 速率限制；無狀態可大量水平擴展。
  3. Parser：解析 HTML、抽出 <a href>、做 URL 正規化（去 fragment、排序 query、補相對路徑）。
  4. URL 去重：正規化後的 URL 先過 bloom（省記憶體），命中再查精確集合，沒見過才回填 Frontier。
  5. 內容儲存：壓縮後存物件儲存；中繼（fetched_at / content_hash）進 DB 供重抓判斷。
  6. 重抓排程：依 freshness（頁面變更率）排 next_recrawl_at，到期重新入 Frontier。

6. 核心元件深入

Frontier：BFS vs 優先佇列

  • BFS（FIFO）：天然廣度優先，覆蓋面廣、實作簡單，適合通用爬全網。
  • 優先佇列：依 PageRank / 更新頻率 / 重要性排序，先爬高價值頁；但需估價與更複雜的佇列。
  • Frontier 會瞬間爆量（分支因子 ~10），必須能溢寫到磁碟或分散式佇列（如 Kafka），不能只靠記憶體。

URL 去重：bloom filter 省記憶體（本題重點）

  • 精確集合（hash set）10 億 URL ≈ 64 GB，單機放不下。
  • Bloom filter：位元陣列 + k 個雜湊。add 把 k 個位元設 1；mightContain 檢查那 k 位是否全 1。
  • 無偽陰性（說沒見過就一定沒見過）、有偽陽性（說可能見過，其實沒有）。1% 偽陽性 10 億 URL 僅需 ~1.2 GB。
  • 偽陽性的後果＝把某個沒爬過的新 URL 誤判成爬過 → 少爬一頁，通常可接受。
  • 要近乎不漏：bloom 命中時再查一個精確集合 / DB二次確認（demo 採此策略）。

Politeness / robots.txt / crawl-delay

  • 抓每個 domain 前先讀並快取 robots.txt，遵守 Disallow 與 Crawl-delay。
  • per-domain 記 last_fetch_at，同 domain 兩次抓取間隔須 ≥ crawl-delay；太密則該 URL 被延後重排。
  • 用 per-host token bucket 控速；超速會被對方封 IP，也不道德。

分散式 frontier：依 domain 分片

  • 把 URL 依 domain 雜湊分到 N 個 shard（hash(domain) % N），同 domain → 同 shard。
  • 好處：每個 domain 的 politeness 狀態集中在單一 shard/worker，速率限制才生效，不會多機同時打同站。
  • 壞處：熱門大站（如 youtube.com）會讓某 shard 過載 → 需對巨站再做子分片或專屬 worker。

陷阱：無限 URL / 爬蟲陷阱、重抓 freshness

  • 爬蟲陷阱：動態產生無限連結（如萬年月曆 ?date=…、無限分頁）會把 frontier 撐爆 → 用最大深度、每站 URL 數上限、URL 正規化、模式偵測防護。
  • Freshness：頁面會變，依變更率分層重抓（新聞每天、靜態每月）；用 content_hash 比對判斷是否真的變了，攤平重抓流量。

7. 擴展與瓶頸

瓶頸說明對策
DNS 解析每次抓取要解析 host，DNS 易成熱點/瓶頸自建 DNS 快取 / 本地解析器，批次預解析
頻寬下載量巨大（數十～數百 Gbps）多區域抓取節點就近抓、壓縮、只抓需要的內容類型
去重結構規模10 億+ URL 的集合超出單機bloom filter（近似）+ 分散式分片去重；可分層 bloom
熱門站集中大站連結極多，單 shard 過載且受 politeness 限速巨站子分片 / 專屬 worker；按站台容量調整 crawl-delay
Frontier 爆量分支因子高，記憶體裝不下分散式佇列（Kafka）+ 溢寫磁碟 + 優先序丟棄低價值 URL
協調與容錯worker 掛掉、重複抓、進度遺失frontier/去重持久化、at-least-once + 去重保證冪等

8. 取捨與面試重點

  • 廣度 vs 深度：BFS 覆蓋廣、適合建索引；DFS/深度限制適合針對性抓站。多數通用爬蟲走 BFS + 深度上限。
  • 即時 vs 批次：即時爬（如新聞/監控）延遲低但持續耗資源；批次爬吞吐高、排程友善但新鮮度差。常混合：重要頁即時、長尾批次。
  • bloom 偽陽性 vs 精確去重記憶體：bloom 省記憶體但偶爾漏抓（偽陽性）；精確集合不漏但 64 GB 放不下。折衷＝bloom 先擋 + 命中再查精確。
  • 禮貌 vs 速度：crawl-delay 拉高抓得慢但安全；拉低抓得快但風險封 IP/被告。依站台規模動態調整。
  • 常見追問：如何避免重複抓相同內容不同 URL？→ 內容指紋（content_hash / simhash）去重。如何處理 JS 渲染頁？→ headless 瀏覽器抓（貴，選擇性用）。如何分散式協調 frontier？→ 依 domain 分片 + 持久化佇列。

9. PHP Demo 說明

本題 demo 著重爬蟲的系統邏輯（也是十億級頁面真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/BloomFilter.php　位元陣列 + k 雜湊（double hashing）；add/mightContain，並估算偽陽性率，示範 URL 去重的省記憶體概念。
  • src/Crawler.php　內建確定性「假網路圖」+ BFS frontier + bloom + 精確 seen 二次確認去重 + per-domain politeness/crawl-delay + 依 domain 分片的分散式 frontier + freshness 重抓排程 + 深度上限防爬蟲陷阱。
  • public/index.php　路由：seed / crawl（步進）/ state / reset，附測試頁可看 frontier、visited、bloom 統計、politeness 延後、shard 指派。
# 啟動
cd demo
php -S localhost:8037 -t public
# 瀏覽器開 http://localhost:8037：先 seed，再反覆按 crawl 看 frontier/visited 成長、去重、politeness、分片
誠實聲明 Demo 不連真網路，用 process 內建的假網路圖（url→內容+外連 links）做確定性爬取，方便觀察與測試； 但 BFS frontier、bloom filter 去重（含偽陽性二次確認）、politeness/crawl-delay、依 domain 分片的分散式 frontier、freshness 重抓排程、爬蟲陷阱深度限制等核心邏輯皆為真實可執行。 要換成真實抓取，只需把 Crawler::sampleWeb() 換成「對 url 發 HTTP GET、解析 HTML 抽 link」，其餘邏輯不變。
