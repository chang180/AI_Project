1. 60 秒搞懂它在幹嘛

想像你是個超勤奮的剪報員。老闆給你一份網址清單（叫「種子」），要你：打開每一頁、把整頁影印存檔， 順便把那頁上提到的每個連結都抄下來，接著一個一個再去開、再影印、再抄連結……一直滾下去， 直到把整個網路都剪報完。

網頁爬蟲（Google、Bing 背後那隻）就是在做這件事——把全世界的網頁自動抓回來存好，搜尋引擎才有東西可以查。

🔑 一句話 網頁爬蟲 = 順著連結一直加好友的剪報機器人：從幾個種子網址出發 → 抓頁面 → 把頁面上的連結加進「待爬清單」 → 一層層往外爬， 過程中要不重複、有禮貌、還能分給很多台機器一起做。

2. 為什麼「順著連結爬」就能逛遍整個網路？

關鍵在一個你天天在用的生活經驗——加好友。

👥 一層層加好友 想認識全校的人？你不用拿到全校名冊。你只要從幾個朋友（種子）開始， 看他們的好友清單，把上面的人都加進來；再看這些新朋友的好友清單，繼續加…… 只要大家彼此都有連到，遲早能透過「朋友的朋友」連到全校每一個人。

網路就是這樣一張大網：每個網頁是一個「人」，頁面上的每個連結就是一條「好友關係」。 爬蟲只要從種子出發、順著連結一直走，就能走遍整張網。工程上把「先把眼前這層的好友全加完，再往下一層」這種爬法叫 BFS（廣度優先）——一圈一圈往外擴，而不是一頭栽進某條路走到黑。

💭 停一下：你只拿到「一個」朋友的名字，最後卻能連到全校的人。為什麼不必一開始就拿到全校名冊？

看答案 因為每個人的好友清單會「接力」帶出更多人——朋友的朋友的朋友……只要這張關係網是連通的，從一個點出發就能擴散到整張網。網頁也一樣：你不必擁有「全網網址表」，順著連結爬，連結會自己帶你去到下一頁。種子只是「起點」，不是「全部」。

3. 把系統拆成 4 個關卡

別被「分散式爬蟲」嚇到。整件事就是闖 4 關，一關一關來：

1待爬清單：手上那疊「還沒去開的網址」

爬蟲手上永遠有一疊「還沒去開的網址」，工程上叫 frontier（待爬清單）。 每抓完一頁、抽出新連結，就把新網址排到清單後面；要爬時就從前面拿一個出來開。

📋 比喻：待辦清單 就像你的待辦事項清單：做完一件事常常會冒出新的待辦，你把新的寫到清單下面，再從上面挑下一件來做。先進先出、一圈圈往外，就是 BFS。

👉 這疊清單會爆炸性變長（一頁就帶出十幾個新連結），所以它不能只放在腦袋裡（記憶體），裝不下時要能溢出去寫到硬碟或排隊系統。

2去重：「這頁我是不是爬過了？」

網路到處是互相連回去的環（A 連 B、B 又連回 A）。如果不檢查，爬蟲會同一頁爬到天荒地老、原地打轉。 所以每拿到一個網址，要先問：「這個我爬過了嗎？」

📓 比喻：超省紙的「打勾備忘」 最直覺的做法是準備一本大冊子，爬過的網址全抄進去，每次翻查。但網址有幾十億個，這本冊子大到桌上擺不下（單機記憶體裝不下）。 於是改用一種超省空間的快速備忘叫 bloom filter：它不抄完整網址，只在一張小卡上「打幾個勾」來記住「這個大概看過了」。

bloom filter 的脾氣很特別，記住這兩句就好：

  • 它說「沒看過」→ 一定沒看過（很可靠，絕不會把新頁誤當舊頁而漏抓……反過來才會）。
  • 它說「可能看過」→ 有時候會認錯（偶爾把沒爬過的新頁誤判成爬過 → 少爬那一頁，通常可接受）。

想幾乎不漏？bloom 說「可能看過」時，再去翻一次真正的精確名冊確認一下就好。省空間的近似先擋，命中再精查。

3禮貌：別對同一個網站「狂敲門」

如果你 1 秒鐘對同一個網站開幾百頁，等於瘋狂按它家門鈴——對方伺服器會被你打趴，輕則封鎖你的 IP，重則惹上麻煩。

🔔 比喻：當個有禮貌的訪客 去別人家拜訪，你不會對著門鈴猛按不放。你會按一下、等人來應門、聊完再走。 而且門口常貼張紙條（robots.txt）寫「哪些房間不歡迎參觀、來訪請間隔幾秒」——有禮貌的客人會讀紙條、照規矩來。

這就叫 politeness（禮貌）：抓每個網站前先讀它的 robots.txt（規矩），同一個網站兩次抓取之間要隔一段時間（crawl-delay），太密就把這個網址延後再排。

4分工：待爬清單分給很多台機器

幾十億頁，一台機器爬到下輩子。要很多台機器一起爬。但這裡有個陷阱：如果隨便分，可能三台機器同時去敲同一個網站——禮貌就破功了。

📮 比喻：依「地址」分郵差 郵局分信不會亂分，而是依地址分區：同一個社區的信全交給同一個郵差。 這樣每個郵差才知道「我今天去過這社區了，等等再來」，不會三個郵差同一小時擠在同一個門口。

所以爬蟲依「網站（domain）」分片：同一個網站的所有網址，都交給同一台機器。 這樣那台機器才管得住「我剛剛敲過這站，要隔幾秒再敲」——禮貌規則才生效。這就是分散式 frontier。

✅ 四關一句話 ① 從種子出發、把連結加進待爬清單一層層爬 → ② 用 bloom filter 去重不重爬 → ③ 對每個網站講禮貌別狂敲 → ④ 依網站分給很多台機器一起做。就這樣。

4. 設計時最怕的 4 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕四件事，每個設計都是為了避開某個「怕」。

😱 怕「原地鬼打牆」 網路全是互連的環，不去重就會同一頁爬到天荒地老，永遠爬不完。所以去重是命脈——但完整名冊太大裝不下，只好用 bloom filter 這種省空間的近似備忘，偶爾少爬一頁換來「裝得下」。記得住 > 記得完美。
😱 怕「被網站封鎖」 對同一個網站抓太快 → 被封 IP、甚至挨告。所以「禮貌」是硬規定：讀 robots.txt、控速、同站慢慢來。抓得慢一點，總比被擋在門外抓不到好。
😱 怕「掉進陷阱爬不出來」 有些網站會自動生出無限多個假連結（像萬年月曆「下一天、下一天……」永遠按不完），會把待爬清單撐爆。 所以要設最大深度、每站網址上限，並把長得像的網址正規化合併，免得被拖進無底洞。
😱 怕「抓回來的是過期舊聞」 網頁會變（新聞每天更新、商品改價）。只抓一次就再也不回頭，存的就是過期資料。 所以要安排重抓（freshness）：常變的頁勤快回去看，靜態頁久久看一次。新鮮 vs 省力，要拿捏。

5. 一張圖看懂

白話導讀，跟著「一頁的旅程」走一圈：

  • 📋 待爬清單（Frontier）吐出一個還沒爬的網址 →
  • ⬇️ 抓取器（Fetcher）把那頁下載回來（抓之前先看「對這站禮貌嗎？間隔夠了嗎？」）→
  • ✂️ 解析器（Parser）把整頁讀過，抽出上面所有連結，並把網址洗乾淨（正規化）→
  • 🔁 每個新連結問去重：「爬過沒？」沒爬過的才放回待爬清單（清單就這樣一圈圈長大）→
  • 💾 抓回的整頁內容壓縮存進倉庫；同時記下「何時該回來重抓」交給重抓排程。

工程術語白話解釋

去重（dedup）「這頁爬過沒？」避免原地鬼打牆
bloom filter超省空間的「打勾備忘」：說沒看過一定準，說看過偶爾認錯

robots.txt網站門口貼的紙條：哪裡別爬、來訪請間隔幾秒

正規化（normalize）把長得不一樣其實同一頁的網址洗成同一個樣子
freshness（新鮮度）頁面會變，安排「多久回來重抓一次」
API系統之間講好的「點餐單」格式：你給我什麼、我回你什麼

7. 要準備多大？（規模估算）

🍱 比喻：搬家前先估幾箱 搬家前你會先估「大概幾箱東西」，才決定叫多大的貨車、租多大倉庫。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
要爬多少頁、多久爬完10 億頁，1 個月爬完≈ 每秒爬 ~400 頁（尖峰再 ×3），單台辦不到 → 要很多台
要多大的網路頻寬每頁平均 100 KB≈ 每秒下載 40 MB（320 Mbps），下載量很可觀
抓回來要存多少每頁壓縮後 ~30 KB10 億頁 ≈ 30 TB，一輪就這麼多
「爬過名冊」要多大記憶體10 億網址、每筆完整存 ~64 B≈ 64 GB，單機塞不下 → 改用 bloom，1% 認錯率只要 ~1.2 GB
💡 關鍵洞察：兩個數字決定整題 其實整題就被兩件事卡住：(1)「爬過名冊」64 GB 桌上擺不下 → 所以要 bloom filter／分散式去重； (2) 對同一個網站不能抓太快 → 所以要禮貌＋依網站分片。其餘都是把這兩件事工程化而已。

💭 停一下：「精確名冊 64 GB」對比「bloom filter 1.2 GB」差了 50 倍。我們用什麼換到這個便宜？

看答案 用「偶爾認錯」換到的。bloom filter 不存完整網址、只打幾個勾，所以超省空間；代價是它有時會把「沒爬過的新頁」誤判成「爬過」（偽陽性），讓我們少爬那一頁。對搜尋引擎來說，幾十億頁裡偶爾少抓一頁影響很小，用 50 倍的記憶體節省去換，非常划算。真要補救，bloom 說「可能看過」時再翻一次精確名冊即可。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

爬蟲多半是關起門來自己跑的內部系統，不太對外開放。所以它的「點餐單」分兩種：一種給管理員下指令，一種是內部零件彼此講話。

① 管理員：丟種子 / 看進度
   POST /crawler/seeds   給我：{ 網址清單, 最大深度 } ← 開工！
   GET  /crawler/stats   我回：{ 爬了幾頁, 待爬清單多長, 每秒幾頁, 各網站速率 }

就像跟剪報員說「這份清單開始剪」，隨時可以問他「剪到哪了、還剩幾張」。

② 內部零件互相講話（概念）
   待爬清單.放進去(網址)      把新連結排進待辦
   待爬清單.拿一個()          取出下一個「已通過禮貌檢查」的網址
   去重.看過了嗎(網址) → 是/否  先問 bloom，命中再查精確名冊
   抓取器.抓(網址)            下載頁面（受該網站速率限制）
   解析器.抽連結(HTML)        洗乾淨後回傳頁面上的外連
   排程.安排重抓(網址, 新鮮度)  決定多久後回來再抓一次

注意「待爬清單.拿一個()」拿出來的，一定是已經通過禮貌檢查的網址——還沒到 crawl-delay 的會先被擋著、延後。這樣抓取器只管抓，不用自己操心禮貌。

9. 系統要記住哪些事？（資料模型）

📒 比喻：四本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題剛好四本，正對應前面四個關卡：

📋待爬簿（還沒爬的網址）

每個待爬網址一列：分片編號、網址、深度、優先序、排進來的時間。 依網站分片，而且裝不下時要能溢寫到硬碟——因為這本簿子會爆炸性變厚。

✅爬過簿（去重，兩層）

分兩層：bloom filter（放記憶體、超省空間、會偶爾認錯）＋ 精確指紋名冊（bloom 說「可能爬過」時，再翻這本確認）。

為什麼要兩層？ 光靠 bloom 會偶爾少爬一頁（認錯）；光靠精確名冊又裝不下（64 GB）。 所以bloom 先擋（快又省），命中時再查精確名冊（準）——兩層搭配，既省空間又幾乎不漏。

💾頁面簿（抓回來的內容中繼）

每抓過的頁一列：網址雜湊（主鍵）、網址、抓取時間、內容指紋、下次重抓時間、抓過幾次、HTTP 狀態。 存內容指紋是為了下次回來時能比對「這頁真的變了嗎？」沒變就不用重存。

🔔網站簿（每個網站的禮貌狀態）

每個網站一列：網站名（主鍵）、上次抓取時間、crawl-delay、robots 規則、分片編號。 把同一網站的禮貌狀態集中在一處，速率限制才管得住——這也是「為什麼要依網站分片」的根本原因。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
每次抓頁都要先「查網址→IP」（DNS），查到爆自己做一份DNS 快取、批次預先查好
下載量太大，網路頻寬塞滿多區域節點就近抓、壓縮、只抓需要的類型
「爬過名冊」太大，單機裝不下用 bloom filter（近似）＋ 分散式分片去重
熱門大站（如 YouTube）連結爆多，把某台機器壓垮對巨站再細分或給它專屬機器
待爬清單爆量、記憶體裝不下用排隊系統（如 Kafka）＋ 溢寫硬碟，低價值網址先丟
機器掛掉、重複抓、進度遺失清單與去重存到硬碟、用「至少抓一次＋去重」保證不亂

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的四個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 廣度 vs 深度 BFS（一圈圈往外）覆蓋面廣，適合建搜尋索引；DFS／深度限制適合針對性抓某個站。多數通用爬蟲走 BFS ＋ 設個深度上限。
⚖️ bloom 認錯 vs 精確名冊太大 bloom 省記憶體但偶爾少爬一頁；精確名冊不漏但 64 GB 塞不下。折衷＝bloom 先擋 ＋ 命中再查精確，兩全其美。
⚖️ 禮貌 vs 速度 crawl-delay 拉長 → 抓得慢但安全；拉短 → 抓得快但可能被封 IP。依站台規模動態調整，大站可稍快、小站要溫柔。
⚖️ 即時 vs 批次 新聞/監控要即時爬（延遲低但一直耗資源）；長尾頁面批次爬（吞吐高但新鮮度差）。常混合用：重要頁即時、其餘批次。
💭 追問：兩個不同網址其實是同一頁內容，怎麼避免重複存？ 對「網址」去重擋不住這種（網址不同）。要對「內容」算一個指紋（content_hash / simhash），內容一樣就視為同一頁，不重複存。
💭 追問：有些頁要靠 JavaScript 跑完才看得到內容，怎麼抓？ 一般 Fetcher 只下載原始 HTML，抓不到 JS 跑出來的東西。要用「無頭瀏覽器」（headless browser）真的把頁面跑一遍再抓——但很耗資源，只對需要的頁選擇性使用。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「待爬清單長大、去重、禮貌、分片」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8037 -t public
# 然後瀏覽器打開 http://localhost:8037
  1. 先按「seed」（丟入種子網址，等於給剪報員第一份清單）。
  2. 反覆按「crawl」（一步一步爬），觀察待爬清單（frontier）和已爬（visited）怎麼長大。
  3. 注意看：bloom 的去重統計、被 politeness 延後的網址、每個網址被指派到哪個 shard（分片）。
玩的時候想一下連按 crawl，待爬清單是不是一下子變很長？這就是「一頁帶出十幾個連結」的分支爆炸。看看有沒有網址因為「剛抓過同站」而被延後——那就是禮貌（politeness）在運作。同一個網站的網址是不是都落在同一個 shard？那就是依網站分片。
誠實提醒 這個 demo 不連真網路，用程式內建的「假網路圖」做確定性爬取，方便你觀察。但 BFS 待爬清單、bloom 去重（含認錯後二次確認）、politeness、依網站分片、重抓排程這些核心邏輯都是真的可執行——要換成真網路，只要把「查假網路圖」換成「對網址發 HTTP GET、解析 HTML 抽連結」，其餘不變。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心其實就幾段小邏輯，剛好對應前面講的關卡。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 兩個檔 = 兩個小幫手 Crawler（順著連結爬的剪報機器人本體：管待爬清單、禮貌、分片）＋ BloomFilter（超省空間的「打勾備忘」＝去重小卡）。 一個網址進來，就照「分片入列 → 輪到它 → 禮貌檢查 → 抓取抽連結 → 去重後再入列」跑一圈。

1待爬清單：一圈圈往外加好友（BFS frontier）

整題的心跳就是這個迴圈：從清單前面拿一個、抓它、把它頁上的新連結排到清單後面。「先進先出」就讓爬法自動變成一圈圈往外的 BFS。

function 爬一步() {
    網址 = 待爬清單.從前面拿一個()      // 先進先出 → 自然就是 BFS
    if (沒東西可拿) return 爬完了
    頁面 = 抓取(網址)                  // 下載這頁、讀出內容
    for (連結 in 頁面.所有外連) {       // 把這頁提到的每個連結…
        if (太深) 跳過                 // ← 防陷阱：限制最大深度
        if (去重.看過了嗎(連結)) 跳過    // ← 沒看過的才要
        去重.記下(連結)
        待爬清單.排到後面(連結)         // ← 新發現排到隊尾，下一圈才輪到
    }
}

關鍵是「新連結排到隊尾」：這層的連結會先被處理完，下一層才開始——這就是「先把眼前這層好友加完，再加朋友的朋友」的 BFS。demo 用一個陣列當佇列；真實系統把它換成排隊系統（Kafka）＋ 溢寫硬碟，邏輯一樣。

2去重：指紋設位元，可能爬過再精查（BloomFilter）

每個新連結都要先問「爬過沒？」。但完整名冊太大裝不下，所以用 bloom filter：把網址算出幾個「指紋位置」，在一張小卡上把那幾個位元設成 1。查的時候，看那幾個位元是不是全都是 1。

function 記下(網址) {                  // = bloom 的 add
    for (位置 in 指紋位置(網址))         // 同一網址算出固定的幾個位置
        小卡[位置] = 1                  // 把那幾個位元設成 1
}
function 可能爬過嗎(網址) {             // = bloom 的 mightContain
    for (位置 in 指紋位置(網址))
        if (小卡[位置] == 0)
            return 一定沒爬過            // ← 有一位是 0 → 鐵定沒加過（不會漏抓）
    return 可能爬過                    // ← 全是 1 → 可能爬過（但也可能是別人剛好把這幾位設成 1 = 認錯）
}

「指紋位置」怎麼算？用兩個雜湊（crc32＋FNV）湊出 k 個位置（double hashing），同一個網址永遠算到同一組位置，所以記過的查得到。

但 bloom 說「可能爬過」偶爾會認錯（別的網址剛好把同樣那幾位也設成 1）。怕漏爬？很簡單——bloom 說「可能」時，再去翻一次真正的精確名冊確認：

function 看過了嗎(網址) {
    if (!可能爬過嗎(網址)) return 否     // bloom 說「一定沒」→ 直接當新網址（不會錯）
    return 精確名冊.有這個網址嗎(網址)   // bloom 說「可能」→ 再精查一次擋掉認錯
}

這就是「省空間的近似先擋，命中再精查」：bloom 用 1.2 GB 擋掉絕大多數，只有它說「可能」時才去碰那本貴的精確名冊——又省空間、又幾乎不漏爬。

💭 停一下：為什麼 bloom 說「一定沒爬過」可以完全相信、不用再精查，但說「可能爬過」就得再查一次？

看答案 因為記下一個網址時，會把它的那幾個位元全設成 1。所以只要有任一位是 0，就代表這網址鐵定沒被記下過（沒有偽陰性）→ 一定沒爬過，完全可信。反過來「全是 1」可能是這網址真的記過，也可能是別的網址剛好把這幾位都設成 1（偽陽性／認錯）——分不出來，所以要再翻精確名冊確認。

3禮貌：別狂敲同個網站的門（per-domain crawl-delay）

輪到一個網址時，抓之前先看一句：「這個網站，我上次抓是多久前？」太近就先別抓，把它延後重排。

網站 = 取出網址的網站名(網址)            // 例：news.example
上次抓 = 各網站狀態[網站].上次抓取時間
if (上次抓存在 且 現在 - 上次抓 < crawl-delay) {
    待爬清單.排到後面(網址, 標記=延後)    // ← 太密！先讓它回隊尾等等再說
    return 這步「延後」
}
// 間隔夠了 → 真的抓
各網站狀態[網站].上次抓取時間 = 現在      // 記下「我這次敲門的時間」
頁面 = 抓取(網址)

關鍵是每個網站各記一個「上次敲門時間」，同一站兩次抓取必須隔 ≥ crawl-delay，否則延後。就像當有禮貌的訪客：按一下門鈴、等一會兒，不對著門鈴猛按不放。真實系統還會先讀網站門口的 robots.txt 拿到它指定的 crawl-delay，邏輯一樣。

4分片：同一個網站交給同一台機器（依 domain 分 frontier）

第 3 關那個「上次敲門時間」只有在同網站集中管時才有用。所以待爬清單依網站雜湊分成 N 份（shard），同一個網站永遠落在同一份：

function 該去哪個分片(網址) {
    網站 = 取出網址的網站名(網址)
    return 雜湊(網站) % 分片數            // ← 用「網站」算，不是用整個網址！
}
// 入列時用它決定 shard：同網站 → 同 shard
分片編號 = 該去哪個分片(網址)
待爬清單[分片編號].排到後面(網址)

用「網站」而不是「整個網址」來算分片，是整段的靈魂——這樣 news.example 的所有頁都進同一個 shard，那台機器才管得住「我剛敲過 news.example，要隔幾秒」。就像郵局依地址分郵差，同社區的信永遠同一個郵差送，不會三個郵差擠在同一個門口。

💭 停一下：如果改成「用整個網址算分片」（而不是只用網站名），禮貌會發生什麼事？

看答案 同一個網站的不同網址會被雜湊到不同 shard、散到不同機器。於是每台機器各自記著「我這份的上次敲門時間」，彼此不知道對方剛敲過——結果好幾台機器同一秒一起去敲同一個網站，crawl-delay 完全失效，網站被打趴、IP 被封。所以一定要用「網站」算分片，把同站的禮貌狀態集中在同一處。
🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 待爬清單一圈圈爬Crawler爬一步()：拿一個 → 抓 → 新連結排到隊尾
② 去重不鬼打牆BloomFilter看過了嗎()：bloom 先擋、命中再精查
③ 對網站講禮貌Crawler抓之前的 crawl-delay 檢查
④ 依網站分片Crawler該去哪個分片()：用網站名雜湊

👉 重點：真實系統是一整群機器組成的分散式爬蟲叢集，會把「陣列佇列 + JSON 檔 + 假網路圖」換成「Kafka 排隊 + 分散式 KV/DB + 真的對網址發 HTTP GET 抓 HTML」， 但上面這幾段BFS、bloom 去重、politeness、依網站分片的判斷邏輯一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「加好友」的比喻解釋，為什麼只給幾個種子網址就能爬遍整個網路？ 每個網頁的連結就像一個人的好友清單。從幾個種子（朋友）出發，看他們連到的頁（好友），再看那些頁連到的頁（朋友的朋友）……只要這張網是連通的，一層層往外就能走遍。種子是起點，不是全部。先把這層爬完再下一層，就是 BFS。
Q2：為什麼不能拿一本「完整名冊」記哪些頁爬過？改用什麼？ 幾十億網址完整記下來約 64 GB，單機記憶體裝不下。改用 bloom filter：超省空間的「打勾備忘」，說「沒看過」一定準、說「可能看過」偶爾認錯（誤判→少爬一頁，可接受）。想幾乎不漏，bloom 命中時再翻一次精確名冊確認。
Q3：politeness（禮貌）是什麼？不做會怎樣？ 別對同一個網站抓太快，要讀它的 robots.txt、同站兩次抓取隔一段時間（crawl-delay）。像當有禮貌的訪客、不對門鈴猛按。不做的話會把對方伺服器打趴、被封 IP 甚至挨告。
Q4：為什麼待爬清單要「依網站」分給不同機器，而不是隨便分？ 因為禮貌規則是「對同一個網站別太快」。如果同一網站的網址散在好幾台機器，就會多台同時去敲它，禮貌破功。依網站分片（同站→同機），那台機器才管得住「我剛敲過這站，要隔幾秒再敲」。像郵局依地址分郵差。
Q5（工程）：bloom filter 用 1.2 GB 取代精確名冊的 64 GB，我們付出了什麼代價？ 代價是「偶爾認錯」（偽陽性）：把沒爬過的新頁誤判成爬過，少爬那一頁。對搜尋引擎幾十億頁來說影響很小，用 50 倍記憶體節省去換很划算。要補救就在 bloom 命中時再查精確名冊。
Q6（工程）：待爬清單（frontier）為什麼不能只放記憶體？怎麼處理？ 因為一頁就帶出十幾個新連結，分支爆炸，清單會瞬間爆量、記憶體裝不下。要能溢寫到硬碟或用分散式排隊系統（如 Kafka），並可丟棄低價值網址、設深度與每站上限防爬蟲陷阱。
Q7（工程）：頁面簿為什麼要存「內容指紋」？重抓時有什麼用？ 重抓回來時，拿新內容的指紋跟舊的比對，就能知道「這頁真的變了嗎」。沒變就不用重新存、也能據此調整下次重抓的間隔，攤平重抓流量、省儲存。

15. 30 秒回顧

網頁爬蟲 = 順著連結一直加好友的剪報機器人。

4 個關卡：① 從種子出發、把連結加進待爬清單一層層爬（BFS）→ ② 用 bloom filter 去重不鬼打牆 → ③ 對每個網站講禮貌（robots ＋ crawl-delay）→ ④ 依網站分片給很多台機器。

4 個怕：怕原地鬼打牆（要去重）、怕被封鎖（要禮貌）、怕掉進無限連結陷阱（設深度上限）、怕抓回舊聞（要重抓）。

工程細節一句話：整題被兩個數字卡住——「爬過名冊 64 GB 裝不下」（用 bloom）和「對同站不能太快」（用禮貌＋分片）；其餘是四本筆記本（待爬/爬過/頁面/網站）＋找出塞車環節各自拓寬。
