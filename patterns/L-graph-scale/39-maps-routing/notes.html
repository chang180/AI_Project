㊴ 地圖路徑規劃 (Google Maps)：毫秒級回應的全球最短路

L 圖/演算法 建議 50 分鐘 Design Maps & Route Planning

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把道路網建成加權圖 → 在圖上求最短/最快路徑 → 預處理讓查詢達毫秒級」。 天真的 Dijkstra 在一張大陸級的圖上要掃描上千萬節點、慢到無法上線；面試真正考的是 A* 用啟發式剪枝 與 Contraction Hierarchies 預處理捷徑 如何把查詢從秒級壓到毫秒級， 以及 即時路況 如何持續改寫邊權、重算 ETA。

1. 需求釐清

功能性需求

  • 給定起點與終點，回傳一條路徑（座標序列）。
  • 支援兩種目標：最短距離（公尺）與最快時間（ETA，受限速與路況影響）。
  • 即時路況：壅塞/事故會即時改寫路段通行時間 → 重算 ETA、必要時改道。
  • 多模式：開車 / 步行 / 大眾運輸 / 騎乘（每種模式一張子圖或不同邊權集合）。
  • 回傳 ETA 與轉彎指示；地圖顯示需要瓦片（map tiles）。

非功能性需求

  • 低延遲：路徑查詢 p99 < 100ms（含跨大陸長程）。
  • 讀遠多於寫：查路徑極多、底圖更新少（道路幾週才變一次）；路況則高頻更新。
  • 高可用：導航是即時關鍵路徑。
  • 正確性：路徑須是真正的最短/最快（或可控的近似）。
  • 可擴展：全球數十億節點、百萬級查詢 QPS。
面試開場必問 「要最短距離還是最快時間？要不要吃即時路況？」
純距離 = 靜態邊權，可全力預處理；最快時間 + 即時路況 = 邊權會變，預處理結構必須能容忍部分動態更新（這是難點）。

2. 規模估算

指標假設推算
圖規模全球道路網節點 ~數億個路口、邊 ~10 億+ 路段
查詢量（讀）10 億次/日~11.5k QPS（峰值 ×3 ≈ 35k）
單次天真 Dijkstra掃描成本與節點數成正比大陸級可達 千萬節點/查詢 → 秒級，不可接受
CH 預處理後沿捷徑只展開數百~數千節點< 1ms ~ 數 ms，可上線
預處理成本一次性離線大陸級數十分鐘~數小時，產出捷徑索引
路況更新（寫）路段級壅塞係數數百萬路段、秒~分鐘級刷新 → 需可增量重算
瓦片z0~z20 金字塔海量靜態圖片 → 全部丟 CDN
關鍵洞察 查詢量大且要毫秒回應 → 不能每次都全圖搜尋。整題圍繞「離線預處理換線上查詢速度」與「啟發式剪枝」。

3. API 設計

# 路徑查詢（讀，超高頻）
GET /api/v1/route?from=25.04,121.53&to=25.06,121.57&mode=drive&metric=time
→ 200 {
    "path": [[25.04,121.53], ...],          # 座標序列
    "eta_seconds": 842, "distance_m": 7310,
    "steps": [{ "instruction": "左轉中山路", "dist_m": 320 }, ...]
  }

# 道路圖更新（寫，低頻，觸發重建/增量預處理）
POST /api/v1/graph        Body: { "nodes":[...], "edges":[...] }

# 即時路況（寫，高頻，改邊權 → 影響 ETA）
POST /api/v1/traffic      Body: { "segments":[{ "a":1,"b":2,"speed_kmh":12 }] }

# 地圖瓦片（讀，海量，走 CDN）
GET /tiles/{z}/{x}/{y}.png

# 偵錯/教學：比較演算法探索節點數
GET /api/v1/route/compare?from=&to=&metric=

4. 資料模型

nodes（路口）
  id        BIGINT PRIMARY KEY
  lat, lng  DOUBLE                       -- 用於啟發式（直線距離）與顯示

edges（路段，有向；雙向路存兩筆）
  from_id, to_id   BIGINT
  distance_m       DOUBLE                -- 固定（由座標算）
  base_time_s      DOUBLE               -- 自由流時間 = distance / 限速
  road_class       SMALLINT              -- 高速/大道/巷弄（多模式用）

traffic（即時路況，與 edges 解耦的高頻寫）
  edge_key   (from_id,to_id)
  congestion DOUBLE                      -- 壅塞係數，live_time = base_time × congestion

ch_shortcuts（CH 預處理產物）
  u, v, weight, via                      -- 跳過中間節點的捷徑邊
  node_order INT                         -- 收縮順序 / 階層等級

tiles（底圖瓦片金字塔，物件儲存 + CDN）
  z/x/y → png/vector
為何分開 distance_m 永不變、base_time_s 幾乎不變、congestion 每分鐘變。 把「會變的路況」與「不變的拓樸」分層，才能讓昂貴的預處理只在拓樸變動時重做，路況更新只乘一個係數。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 地圖資料(OSM/供應商) ──▶ 圖建構 ──▶ 預處理(CH) ──▶ 捷徑索引
                                                      │
 查詢(from,to,metric) ─▶ 路由服務 ─(Dijkstra / A*)──▶ 路徑 + ETA ─▶ 回應
                              ▲
 即時路況串流 ──▶ 邊權更新 ────┘（live_time = base × congestion）

 顯示底圖 ─▶ 瓦片服務 ─▶ CDN（z/x/y 靜態圖片，與路由解耦）
  1. 圖建構：把原始地圖資料正規化成節點/邊加權圖（距離、限速、路別）。
  2. 預處理（CH）：離線收縮節點、產生捷徑，讓線上查詢只走少數階層。
  3. 路由服務：無狀態、可水平擴展；收查詢 → 跑 A*/CH 雙向搜尋 → 回路徑與 ETA。
  4. 路況串流：持續把壅塞係數推進邊權，ETA 隨之變化（不需重做整套 CH，只調權重或用可動態化的 CRP）。
  5. 瓦片服務：底圖是靜態金字塔，全進 CDN，與路由計算完全分離。

6. 核心元件深入

Dijkstra vs A*：差在「先展開誰」

  • Dijkstra：依「已走成本 g」由小到大展開 → 像水波四面八方均勻擴張，與終點方向無關，展開節點多。
  • A*：依 f = g + h 展開，h 是「到終點的樂觀估計」（啟發式）→ 朝終點方向擴張，展開節點少。
  • 兩者找到的路徑與成本相同（皆最優），差別只在「探索/定案的節點數」—— 這正是 demo 直接量給你看的數字。

啟發式的「可採納性」（admissibility）

  • 可採納 = h 永不高估真實剩餘成本 → A* 保證最優。
  • 距離度量：用直線（haversine/歐式）距離當 h，直線必不長於沿路走的距離 → 天然可採納。
  • 時間度量：把直線距離除以地圖最高限速得「最樂觀通行秒數」當 h → 不高估、可採納。
  • 若 h 高估（不可採納），A* 變快但可能回傳非最優路徑（這是某些「導航夠好就好」的刻意取捨）。

Contraction Hierarchies（CH）：預處理捷徑達毫秒級

  • 離線：依某種順序逐一「收縮」節點；移除一個節點時，若它是兩鄰居間最短路的必經點，就加一條直接連兩鄰居的捷徑邊（shortcut），把它「跳過」。
  • 線上：查詢時做雙向搜尋，且只往階層更高的節點走。因為有捷徑，一條跨城市的路只需展開數百~數千節點，而非數百萬 → 毫秒級。
  • 代價：一次性預處理（大陸級數十分鐘~數小時）＋ 額外捷徑儲存；邊權若大幅改變需重做。
  • 應對動態路況的進階版：CRP（Customizable Route Planning） 把預處理切成「拓樸層（慢、少做）」與「權重客製層（快、可隨路況重跑）」，兼顧速度與即時性。

即時路況更新邊權 → ETA

每條路段維護 live_time = base_time × congestion。路況串流把壅塞係數推進來，最快路徑的搜尋就會自動避開塞車段、改道， ETA 也跟著上升或下降。距離度量不受路況影響（距離是幾何固定值），只有時間度量會變 —— demo 兩種都可切換對照。

地圖分塊 / 瓦片（slippy map）

底圖用 Web Mercator 把世界切成 z/x/y 金字塔瓦片，縮放越深瓦片越多。瓦片是靜態、可永久快取，全部丟 CDN， 與路由計算完全解耦。demo 的 GET /api/tile?z=&x=&y= 用標準瓦片公式把瓦片座標換回經緯邊界，回傳落在框內的節點/邊（簡化版）。

7. 擴展與瓶頸

階段瓶頸對策
單張大圖全球圖記憶體裝不下、單次搜尋太久地理分區/分層（partition）；CH/CRP 預處理；只在分區邊界做跨區拼接
查詢延遲長程 Dijkstra 掃太多節點A* 啟發式剪枝 + CH 捷徑雙向搜尋 → 毫秒級
查詢 QPS路由服務 CPU無狀態服務水平擴展 + 結果快取（熱門 OD 對）
預處理成本拓樸更新就要重算道路變動低頻、離線批次；用 CRP 分層只重跑客製層
即時路況邊權高頻變動、CH 不易增量權重與拓樸分離（CRP）；或在小範圍做即時 A*，遠程仍用預處理
底圖流量海量瓦片請求靜態瓦片全進 CDN，與路由完全分離

8. 取捨與面試重點

  • 預處理 vs 查詢時間：CH 用昂貴的離線預處理換毫秒級線上查詢；純 Dijkstra/A* 無預處理但長程慢。查詢量越大越該預處理。
  • 最短 vs 最快 vs 即時：距離是靜態、可全力預處理；最快時間吃即時路況、邊權會變，預處理結構要能容忍動態（CRP）。
  • 精確 vs 啟發（近似）：可採納啟發式保證最優但剪枝有限；放寬成不可採納（高估）或用地標/A* landmark 加速，速度更快但可能非最優 —— 導航實務常接受「夠好的近似」。
  • 常見追問：CH 如何處理單行道/轉彎限制？→ 有向圖 + 轉彎成本建模。如何做大眾運輸時刻表路由？→ 時間相依圖 / RAPTOR 演算法。如何避開收費/特定路段？→ 邊權加懲罰或過濾子圖。

9. PHP Demo 說明

本題 demo 著重於路由演算法的核心邏輯（也是這題真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/Graph.php　道路圖：節點含座標、邊有 distance/time 權重；haversine 距離、壅塞係數、瓦片切分。
  • src/Router.php　Dijkstra 與 A*（共用同一二元堆積），各自回報探索節點數；A* 啟發式用直線距離。
  • public/index.php　路由 + 測試頁：選起訖 → 比較兩演算法探索節點數、調路況看 ETA/改道，並提供 /api/route /api/compare /api/traffic /api/tile /api/state。
  • 預設塞一個 6×5 格狀城市道路網（30 路口、49 路段）。
# 啟動
cd demo
php -S localhost:8039 -t public
# 瀏覽器開 http://localhost:8039：選起點/終點 → 比較 Dijkstra vs A*；
# 對路徑上某段套壅塞 → 看最快路徑改道、ETA 變化。
誠實聲明 Demo 的道路圖為小型示意（30 路口），Dijkstra、A*（含探索節點數比較）、即時路況改 ETA、瓦片切分皆為真實可執行邏輯； Contraction Hierarchies 在此僅作觀念說明（§6）未實作——在這種小圖上 CH 不會帶來可感差異，它的價值要在大陸級圖才顯現。 要看 A* 明顯少探索節點，選格網中段的終點即可（角落對角是 A* 的最壞情況）。
