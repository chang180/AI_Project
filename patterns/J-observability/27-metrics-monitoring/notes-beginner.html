1. 60 秒搞懂它在幹嘛

想像你是一間大醫院的加護病房護理長。病房裡躺著幾千台機器（每台都是一台伺服器）。 你最怕的就是某台突然「不行了」卻沒人發現。怎麼辦？

你會在每張病床旁裝一台生理監測儀：它每隔幾秒就自動量一次心跳、血壓、血氧， 畫成一條一條會走動的曲線。數值正常就安靜；一旦某條曲線連續好幾分鐘超過警戒線， 監測儀就嗶嗶叫，把護士叫過來。

🔑 一句話 指標監控系統（Prometheus）= 幫成千上萬台機器裝生理監測儀：定時去每台「量數據」、把數值記成會走動的曲線、 一旦曲線異常持續一段時間就「嗶嗶叫」通知工程師。

這就是 Prometheus 在做的事。它的難，不難在「記數字」（誰都會），難在三件事： 要照顧的機器超多、曲線暴增會把自己撐爆、還有不能一抖動就誤報害大家半夜被吵醒。

2. 「指標」到底是什麼？跟日誌差在哪？

很多人一開始會搞混三個詞：指標（metrics）、日誌（logs）、追蹤（traces）。 用一個你熟悉的經驗就懂了——看醫生：

🏥 體溫表 vs 病歷本 vs 轉診單
  • 指標 = 床邊的體溫表：只記「數字＋時間」，例如 37.5°C、38.0°C…… 很省空間、適合畫成曲線看趨勢、適合設警戒線。
  • 日誌 = 詳細的病歷文字：「下午三點病人咳嗽、開了什麼藥」——資訊豐富但又多又雜，存起來貴。
  • 追蹤 = 一張轉診單：記「這位病人從掛號→看診→抽血→拿藥」一路經過哪些科。

這一題只做指標（體溫表那種）：純數字、能畫曲線、能聚合、最適合「嗶嗶叫的告警」。 日誌跟追蹤是另外兩套系統，這裡先不管。

💭 停一下：如果你想知道「過去一小時這台伺服器的記憶體是不是一直在漲」，你該看「體溫表」還是「病歷本」？

看答案 看「體溫表」（指標）。因為你要的是一條隨時間走動的數字曲線，指標天生就是這個形狀，畫出來一眼就看到「往上爬」。 病歷本（日誌）是一堆文字，要看趨勢得自己慢慢數，又慢又貴。

3. 把系統拆成 4 個關卡

別被「監控系統」嚇到。整件事其實就是闖 4 關，一關一關來：

1量數據：護士主動去「巡房」（scrape 拉取）

怎麼拿到每台機器的數據？有兩種做法：機器主動回報，或監控系統主動去問。Prometheus 選後者。

🚶 比喻：護士定時巡房量體溫 不是叫病人「想到就自己回報體溫」（會忘、會亂報），而是護士每隔 15 秒固定去每張病床量一次。 這個「主動去量」的動作就叫 scrape（拉取）。

每台機器只要在門口貼一張「現在數值看板」（一個叫 /metrics 的網址），上面列著目前的數字。 Prometheus 定時來「抄這張看板」就好。

👉 這招有個隱藏好處：護士走到床邊發現「人不見了／看板打不開」，就知道這台機器掛了——連「死亡偵測」都免費附送，不用另外做。

2記曲線：每個數字都要「貼標籤」分得清

抄回來的數字要存成一條一條會走動的曲線（這個倉庫叫時序資料庫 TSDB，「時序」就是「按時間排好的數字」）。 問題是：同樣是「請求數」，GET 的、POST 的、成功的、失敗的……怎麼分得清是哪一條？

🏷️ 比喻：行李吊牌 機場每件行李都掛一張吊牌，上面寫「航班 + 艙等 + 目的地」。就算外觀一樣的兩個黑箱子，吊牌不同就是不同件行李。 Prometheus 也一樣：每條曲線 = 一個名稱 + 一組標籤（label）。 
http_requests_total{method="GET",  code="200"}   ← 一條曲線
http_requests_total{method="POST", code="200"}   ← 另一條（標籤不同）
http_requests_total{method="GET",  code="500"}   ← 又一條（這條在數失敗）

名稱相同、只要任何一個標籤不一樣，就是另一條獨立曲線。之後查詢時，你就能說「把所有 code=500 的曲線加起來」，秒看錯誤總量。

3看趨勢：兩種數字、一個關鍵動作 rate()

曲線分兩種個性，先分清楚，後面才不會算錯：

🚗 比喻：里程表 vs 油量表
  • counter（計數器）= 汽車里程表：只會往上加，永遠不會倒退。例：網站開站以來「總共處理過幾筆請求」。
  • gauge（量表）= 油量表 / 室溫：會上上下下。例：目前用了多少記憶體、佇列裡還有幾個待辦。

這裡有個超關鍵的觀念：里程表上的「總公里數」本身沒什麼用，你真正想知道的是「現在開多快」。

⏱️ rate() = 算「現在跑多快」 里程表從 10000 變成 10060，過了 60 秒 → 你就知道「每秒跑 1 公里」。 rate() 做的就是這件事：拿一條只增的 counter 曲線，算出它「每秒增加多少」。 「每秒新增幾筆請求」「每秒新增幾個錯誤」——這些才是你真正想盯的。

💭 停一下：為什麼「總請求數 = 8,000,000」這個數字，對「現在系統忙不忙」幾乎沒幫助？

看答案 因為那是累積一整年的總和，看不出「此刻」的狀況。就像里程表顯示 20 萬公里，沒告訴你現在是停著還是飆 120。 你要的是變化速度——用 rate() 把「總數曲線」轉成「每秒增加多少」，才看得出當下忙不忙。

4告警：異常「持續一段時間」才嗶嗶叫

最後一關：什麼時候該把工程師叫起來？很簡單——某條曲線超過警戒線就告警。但這裡藏著一個大坑。

🌡️ 比喻：量到一次高溫先別急 你剛運動完量體溫，瞬間 37.8°C，難道馬上送急診？當然不會——你會等幾分鐘再量一次， 如果連續好幾次都偏高才當真。一瞬間的高點很可能只是剛好抖一下。

告警也一樣：規則不是「超過閾值馬上響」，而是「超過閾值並且持續一段時間（例如 5 分鐘）」才真的響。 這段等待叫 for。它的狀態是這樣走的：

沒事(inactive) → 剛超標、觀察中(pending) → 持續超標夠久、真的響了(firing) → 恢復正常(resolved)

👉 這一步就是為了避免「一瞬間抖動就誤報」。少了它，半夜每個小尖刺都把工程師吵醒，沒幾天大家就把告警全關掉了。

✅ 四關一句話 ① 護士定時巡房量數據（scrape）→ ② 每個數字掛吊牌記成曲線（name+label）→ ③ 用 rate() 看變化速度（里程表→時速）→ ④ 異常持續夠久才嗶嗶叫（避免誤報）。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「曲線爆炸」（高基數） 每多一種「吊牌組合」就多一條曲線、多吃一份記憶體。如果有人手賤把「使用者編號」或「完整網址」也當成標籤， 那幾千條曲線會瞬間暴增成幾百萬條，直接把監控系統自己撐爆。 所以鐵則：標籤只放「種類有限」的東西（方法、狀態碼、區域），不放「每個都不一樣」的東西。這是頭號事故來源。
😱 怕「監控自己先掛」 監控系統是「看門狗」，如果它比被它看守的機器還容易掛，那就毫無意義。 所以原則是「可靠 > 完整」：漏抄一兩次數據可以接受，但這隻看門狗絕對不能倒。 它刻意做得簡單、單機就能跑、不依賴一堆別的服務。
😱 怕「狼來了（告警疲勞）」 如果三天兩頭誤報、半夜沒事亂叫，工程師會直接把告警靜音——等真的出事時就沒人理了（這叫「告警疲勞」）。 所以才要用「持續一段時間才響」過濾抖動，再交給專門的整理員（Alertmanager）把同類告警合併、去重，別一次轟炸 100 則。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
總共幾條曲線1 萬台機器 × 每台 100 條 ≈ 100 萬條曲線數（基數）才是成本主角，不是數字多寡
多久量一次每 15 秒巡房一次100 萬 ÷ 15 ≈ 每秒寫進 6.7 萬個數字
一個數字佔多大原本 16 位元組，壓縮後 ~2相鄰時間/數值差很小，壓縮能省到 ~1/10
存 15 天要多大磁碟≈ 170 GB本機磁碟放得下，所以預設只留近期
💡 關鍵洞察：怕的不是「量很多次」，是「曲線太多種」 很多人以為「量越頻繁越花錢」。其實主要成本來自曲線的「種類數」（基數）。 像衣櫃：你不是怕「同一件衣服穿很多次」，而是怕「款式爆增到塞不下」—— 一個高基數標籤就能把 100 萬條曲線變成上千萬條，記憶體直接爆。

💭 停一下：A 同事把 method（GET/POST，就幾種）當標籤；B 同事把 user_id（幾百萬個都不同）當標籤。誰會把系統撐爆？

看答案 B 會撐爆。 method 種類有限（就那幾個），曲線數可控；但 user_id 有幾百萬個值， 每個值都會生出一條新曲線，瞬間爆出幾百萬條，吃光記憶體與索引。 所以標籤只放「種類有限」的東西，user_id 這種應該丟去日誌系統，不要放進指標標籤。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」：

① 機器門口的「現在數值看板」（被巡房抄的對象，最重要）
   GET /metrics
   我回（純文字，一行一個數字＋吊牌）：
     # TYPE http_requests_total counter   ← 先宣告：這條是里程表型
     http_requests_total{method="GET",code="200"} 1027
     # TYPE process_resident_memory_bytes gauge  ← 這條是油量表型
     process_resident_memory_bytes 5.1e7

為什麼要先寫 # TYPE … counter？因為查詢端要知道「這是里程表還是油量表」，才知道能不能對它用 rate()（只有里程表型才該算時速）。

② 查詢曲線（用簡化版 PromQL）
   GET /api/query?expr=instant 記憶體      ← 取「此刻最新值」（看油量表）
   GET /api/query?expr=rate 總請求數[60]   ← 算「最近 60 秒每秒增加多少」（里程表→時速）
   GET /api/query?expr=sum 總請求數 by(method)  ← 把同類曲線「依方法分組加總」

三種最常用的問法：instant＝「現在多少」、rate＝「變多快」、sum by＝「分組加總看全貌」。

③ 問現在有哪些告警在響
   GET /api/alerts
   → { "HighRequestRate": { "state":"firing", ... } }

state 就是前面那台「監測儀」的燈號：沒事(inactive)／觀察中(pending)／響了(firing)。

9. 系統要記住哪些事？（資料模型）

📒 比喻：曲線的「身分證」與「成長紀錄」 倉庫要存兩種東西：每條曲線「是誰」（身分），以及它「一路走來的每個數值」（紀錄）。

🆔曲線身分 = 名稱 + 排序後的標籤

一條曲線（series）的身分 = 名稱 + 一組標籤。例：http_requests_total{method="GET",code="200"}。

為什麼標籤要「先排序」再當鑰匙？ {a=1,b=2} 跟 {b=2,a=1} 其實是同一條曲線，只是寫的順序不同。 所以系統會把標籤固定照字母排好再組成鑰匙，這樣兩種寫法都對到同一條，不會誤生兩條。

📈每個樣本 = (時間, 數值)，一直往曲線尾巴貼

每量一次就產生一個樣本（sample）＝ (時間戳, 數值)，append 接到對應曲線的尾端。

曲線A：(10:00, 1000) (10:15, 1027) (10:30, 1100) …  ← 越接越長

🔍怎麼快速找到「所有 code=500 的曲線」？（倒排索引）

查詢常常是「給我所有 code=500 的曲線」。如果一條條翻太慢，所以多存一本「標籤 → 哪些曲線」的對照表。

就像書末的「索引頁」 你不會為了找「Prometheus」這個詞，從第一頁一頁頁翻到最後。你翻書末索引：「Prometheus → 第 12、45、78 頁」。 系統存的就是「code=500 → 第 3、9、20 條曲線」這種對照表，依標籤篩選秒查。這叫倒排索引。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
曲線太多種，吃光單機記憶體嚴管標籤基數、把高基數曲線直接丟掉不收（relabel）
一台 Prometheus 巡不完那麼多床多請幾台，每台各巡一部分機器（水平分片）
想看「全公司跨機房的總覽」聯邦 (federation)：上層那台只去抄各下層「已加總過」的少量曲線
本機磁碟只留 15 天，想看歷史遠端儲存：把數據再寫一份到雲端大倉庫（Thanos/Mimir 等）長期保存
儀表板每次都算一大段，查很慢預先算好（recording rules）：常用的聚合先算成新曲線存著，查時直接取

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 主動去量(pull) vs 等人回報(push) Prometheus 選「護士主動巡房(pull)」：天然附送死亡偵測、好除錯（自己 curl 看板就行）、機器不必知道後端在哪。 代價是「閃一下就結束的短命任務」可能還沒被巡到就跑掉了 → 用 Pushgateway 讓它先把結果「寄存」一下再走。
⚖️ 存本機 vs 存雲端 存本機：讀寫快、好維護，但只能留近期、看不到跨機房全貌。寄到雲端遠端倉庫：能長期＋全域查，代價是更複雜。
⚖️ 量很密 vs 省成本 每 5 秒量一次比每 30 秒清楚，但儲存成本也跟著線性上升。多數服務 15~30 秒就夠，只有最關鍵的服務才縮短間隔。
💭 追問：機器重開機，里程表(counter)歸零了，rate() 會不會算出「負的時速」？ 這正是最容易出錯的地方。counter 重啟會從一個大數字突然掉回 0，天真地相減會得到負值或嚴重低估。 所以 rate() 會偵測到「後面的值比前面小」就判定是重啟，把掉下去的那段補回來（當作從 0 重新累計），結果永遠不會是負的。
💭 追問：怎麼快速判斷「一台服務還活著嗎」？ Prometheus 每次巡房都會自動記一條 up 曲線：抄到看板就是 1、抄不到就是 0。 所以只要看 up == 0 就知道哪台死了——不用自己另外做心跳偵測。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「量數據 → rate → 告警」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8027 -t public
# 然後瀏覽器打開 http://localhost:8027
  1. 連按幾次「Scrape」（等於護士去巡房，每次 counter 的值會調大一點）。
  2. 跑 rate(...)，看它把「總數曲線」算成「每秒增加多少」（里程表→時速）。
  3. 跑 sum ... by(...)，看同類曲線怎麼分組加總；再看告警從 pending 變 firing。
玩的時候想一下 多按幾次 Scrape 後再算 rate，是不是「時速」就出來了？這就是關卡 3。 把 counter 的值故意調小（模擬機器重開、里程表歸零），看 rate 會不會變負——不會，因為它補償了重啟，這就是 Part 2 的那個追問。

13. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「醫院監測儀」的比喻，一句話說 Prometheus 在幹嘛？ 幫成千上萬台機器裝生理監測儀：定時去每台「量數據」、記成會走動的曲線、異常持續一段時間就「嗶嗶叫」通知工程師。
Q2：指標、日誌、追蹤差在哪？用看醫生的比喻說說看。 指標＝床邊體溫表（只記數字＋時間、適合畫曲線設警戒線）；日誌＝詳細病歷文字（豐富但又多又貴）；追蹤＝轉診單（記一個請求一路經過哪些服務）。這題只做指標。
Q3：counter 和 gauge 的差別？各舉一個例子。 counter＝里程表，只增不減（總請求數、總錯誤數）；gauge＝油量表/室溫，會上下（記憶體用量、佇列長度）。
Q4：為什麼「總請求數」要用 rate() 轉一下才有意義？ 總數是累積值，看不出此刻忙不忙（像里程表 20 萬公里不代表現在飆車）。rate() 把它算成「每秒增加多少」＝當下的時速，才看得出當下負載。
Q5：告警為什麼要「持續一段時間（for）」才響？ 避免一瞬間的抖動就誤報。像量到一次高溫先別急，等連續幾次都偏高才當真。少了它會狼來了、害大家把告警全關掉（告警疲勞）。
Q6（工程）：把 user_id 當標籤會發生什麼事？為什麼是頭號事故？ 每個不同的 user_id 都會生出一條新曲線（高基數），幾百萬使用者＝幾百萬條曲線，吃光記憶體與索引把系統撐爆。標籤只能放「種類有限」的列舉值；user_id 這種該丟去日誌系統。
Q7（工程）：機器重開機、counter 歸零，rate() 為什麼不會算出負數？ rate() 偵測到「後值比前值小」就判定為重啟，把掉下去的那段補回（視為從 0 重新累計），所以結果永遠非負，不會因重啟而算出負速率或嚴重低估。

14. 30 秒回顧

指標監控（Prometheus）= 幫成千上萬台機器裝生理監測儀。

4 個關卡：① 護士定時巡房量數據（scrape）→ ② 每個數字掛吊牌記成曲線（name+label）→ ③ 用 rate() 把里程表算成時速（看變化速度）→ ④ 異常持續夠久才嗶嗶叫（避免一抖就誤報）。

3 個怕：怕曲線爆炸（標籤別放高基數的 user_id）、怕看門狗自己先掛（可靠＞完整）、怕狼來了（持續才響＋整理員去重）。

工程細節一句話：成本由「曲線種類數」決定 → 用「點餐單(API)」溝通，看板先宣告型別才知能不能 rate → 曲線身分＝名稱＋排序標籤、配倒排索引秒查 → 找出塞車環節（基數/分片/聯邦/遠端/預算）各自拓寬。
