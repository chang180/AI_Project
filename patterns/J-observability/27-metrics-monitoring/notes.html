J 可觀測性 建議 42 分鐘 Design a Metrics & Monitoring System (Prometheus)

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「pull 模型的時序資料庫（TSDB）+ 多維度 label 模型 + PromQL 查詢 + 告警狀態機」。 難點不在「存數字」，而在 series 的高基數（cardinality）爆炸、rate() 對 counter reset 的正確處理、以及 pull vs push 的取捨。

1. 需求釐清

功能性需求

  • 從大量 target 收集指標（metrics）：每個 target 暴露 /metrics（Prometheus 曝露格式），系統定時拉取。
  • 支援多維度 label：http_requests_total{method="GET",code="200"}，可依任意 label 切片聚合。
  • 支援查詢語言：取最新值（instant）、rate()/increase()、sum/avg by(label) 聚合。
  • 支援告警規則：表達式（如 rate > 閾值）持續 for 一段時間後觸發。

非功能性需求

  • 監控四黃金訊號：延遲（latency）、流量（traffic）、錯誤（errors）、飽和度（saturation）。
  • 高寫入吞吐：每秒數百萬樣本（samples/s）。
  • 查詢低延遲：儀表板（Grafana）需秒級回應。
  • 可靠性優先於完整性：監控系統本身必須比被監控者更穩；漏一兩個樣本可接受，但不能掛。
面試開場必問 「metrics、logs、traces 三者要哪個？」本題是 metrics（數值型、可聚合、低成本、適合告警）。logs 是離散事件文字、traces 是單請求跨服務鏈路——三者互補但儲存與查詢模型完全不同。接著問「pull 還是 push？」Prometheus 是 pull（主動 scrape），這決定了 service discovery 與 target 健康判斷的設計。

2. 規模估算

指標假設推算
active series 數（基數）1 萬 target × 每台 100 series~100 萬 active series
scrape 間隔每 15 秒拉一次樣本率 ≈ 100萬 / 15 ≈ 6.7 萬 samples/s
單樣本儲存delta 壓縮後 ~1.5~2 bytes/sample未壓縮 16B（8B 時間+8B 值）→ 壓縮率 ~10×
儲存（15 天保留）6.7萬 × 86400 × 15 × 2B≈ 170 GB（壓縮後本地磁碟）
查詢Grafana 多面板每 10s 刷新查詢命中近期 block / 記憶體 head，需索引加速
關鍵洞察成本與規模幾乎完全由 cardinality（series 數）決定，而非樣本率。一個高基數 label（如把 user_id、request_id 放進 label）就能讓 series 從幾千暴增到數百萬，拖垮記憶體與索引——這是 Prometheus 的頭號運維事故來源。

3. API 設計

# target 暴露指標（被 scrape，曝露格式 exposition format）
GET /metrics
→ 200 text/plain
  # TYPE http_requests_total counter
  http_requests_total{method="GET",code="200"} 1027
  # TYPE process_resident_memory_bytes gauge
  process_resident_memory_bytes 5.1e7

# 餵入 / 模擬 scrape 一批樣本（本 demo）
POST /api/scrape    Body: 曝露格式文字  或  application/json { "samples":[{name,labels,value,type}] }
→ 200 { "ingested": 3, "ts": 1717200000000 }

# 查詢（PromQL 簡化語法）
GET /api/query?expr=instant process_resident_memory_bytes
GET /api/query?expr=rate http_requests_total[60]          # 60 秒窗每秒變化率
GET /api/query?expr=sum http_requests_total by(method)    # 依 method 分組加總

# 告警狀態
GET /api/alerts → { "alerts": { "HighRequestRate": { "state":"firing", ... } } }

4. 資料模型

series（時間序列）= metric name + 一組 labels，唯一識別一條曲線
  http_requests_total{method="GET",code="200"}   ← 一條 series
  http_requests_total{method="POST",code="200"}  ← 另一條（labels 不同）

sample（樣本）= (timestamp 毫秒, float value)，append 到所屬 series
  series A: (t0, 1000) (t1, 1027) (t2, 1100) ...

型別：
  counter  單調遞增（總請求/總錯誤）；重啟會 reset 歸零 → rate() 必須補回
  gauge    可上可下（記憶體、佇列長度、溫度）
  （延伸：histogram / summary 用多條 series 表示分位數）

曝露格式（exposition format）：每行一個樣本，# TYPE 宣告型別
為何是「name + 排序後 labels」當 key{a="1",b="2"} 與 {b="2",a="1"} 必須對到同一條 series。實作上把 labels 依 key 排序再序列化成穩定字串當索引鍵；真實 Prometheus 用倒排索引（label→series 集合）加速「依 label 篩選」的查詢。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 Targets(/metrics) ──┐
 Targets(/metrics) ──┤   Service Discovery（k8s/consul/檔案）
 Targets(/metrics) ──┘            │ 提供 target 清單
                                  ▼
                       ┌──────────────────────┐
                       │  Prometheus Server    │ 定時 scrape（pull, 藍）
                       │  ┌────────────────┐   │
                       │  │ 本地 TSDB      │   │ series 表 + WAL + block
                       │  └────────────────┘   │
                       │  PromQL 引擎(rate/sum) │ ← 查詢（綠）
                       └──────────┬────────────┘
                       告警規則評估 │
                                  ▼
                          Alertmanager（去重/分組/靜默/路由 → Slack/PagerDuty）

 Grafana ──查詢──▶ Prometheus（視覺化儀表板）
  1. Service Discovery：動態取得要 scrape 的 target 清單（k8s pod、consul、靜態檔），target 來去自動納入/移除。
  2. Prometheus Server：定時 pull 每個 target 的 /metrics，打上 scrape 時間戳，寫入本地 TSDB。
  3. PromQL 引擎：對 TSDB 跑 instant / range 查詢（rate、increase、aggregation）。
  4. Alertmanager：接收 firing 告警，做去重、分組、靜默、抑制與路由通知（與評估解耦）。

6. 核心元件深入

pull 模型與 service discovery

  • Prometheus 主動拉取，好處：target 不需知道後端在哪、天然得到「能否 scrape 成功」當作健康訊號（up 指標）、易於本機測試（直接 curl /metrics）。
  • 缺點：短命任務（batch job）還沒被 scrape 就結束 → 用 Pushgateway 暫存；防火牆內的 target 需可被 Prometheus 連到。

TSDB 與時間窗

  • 近期樣本進記憶體 head block + WAL（防當機遺失），定期切成不可變 block 落盤並壓縮（delta-of-delta 時間 + XOR 值編碼，~1.5B/sample）。
  • 查詢是時間窗操作：range query 取一段時間的樣本陣列再套函式。

PromQL：rate / increase / aggregation 與 counter reset

  • rate(counter[5m]) = 窗內（最後值 − 第一值）/ 時間差，得「每秒平均增速」。
  • counter reset 處理（重點）：counter 應只增不減；若相鄰樣本後值 < 前值，代表 target 重啟歸零。rate 必須把下降的那段補回（視後值為從 0 重新累計），否則會算出負速率或嚴重低估。本 demo 的 rate() 真實實作了這個補償。
  • sum by(label)：跨多條 series 取最新值加總，可依某 label 分組（如 sum by(code)(rate(...)) 看各狀態碼速率）。

告警 for 持續時間與 firing/resolved 狀態機

  • 規則 expr 成立後不立即告警，先進 pending；持續成立達 for 時長才升為 firing（過濾瞬時抖動，降低噪音）。
  • 條件解除 → resolved（回 inactive）。狀態流轉：inactive → pending → firing → resolved。

label cardinality 爆炸風險

  • 每個獨特 label 值組合都是一條新 series，吃記憶體與索引。把高基數欄位（user_id、email、完整 URL、trace_id）放進 label = 自爆。
  • 守則：label 值集合要有界且低基數（method、status code、region 這類列舉值）。高基數資訊應放 logs/traces，而非 metrics label。

7. 擴展與瓶頸（高基數 / 聯邦 / 遠端儲存）

階段瓶頸對策
單機記憶體active series 太多（高基數）限制 label 基數、relabel 丟棄高基數 series、metric_relabel_configs
單機抓不完target 數超過單 Prometheus 負荷水平分片：多台 Prometheus 各抓一部分 target（依 hash/標籤分配）
跨叢集彙總要全域視圖聯邦 federation：上層 Prometheus scrape 下層的彙總指標（只抓聚合後少量 series）
長期保留本地磁碟只留 15 天remote write 到遠端儲存（Thanos / Cortex / Mimir / VictoriaMetrics），長期 + 全域查詢
查詢爆量大範圍 range query 慢recording rules 預先算好常用聚合，存成新 series

8. 取捨與面試重點

  • pull vs push：pull 天然有健康檢查（up）、易除錯、target 解耦；push 適合短命任務與防火牆穿透。Prometheus 選 pull，短命任務用 Pushgateway 補。
  • 本地 TSDB vs 遠端儲存：本地讀寫快、運維簡單、但保留期與全域視圖受限；遠端（remote write）換來長期 + 跨叢集，代價是複雜度與一致性。
  • 取樣精度 vs 成本：scrape 間隔越短（如 5s）解析度越高但樣本量與儲存成本線性上升；多數場景 15~30s 足夠，關鍵服務才縮短。
  • metrics vs logs vs traces：metrics 便宜可告警但維度有限；別把高基數塞進 label，該用 logs/traces 的就用它們。
  • 常見追問：counter 重啟 rate 為何不會變負？→ reset 補償。如何避免告警風暴？→ for 抑制抖動 + Alertmanager 分組/去重/抑制。如何看「服務有沒有在」？→ up == 0。

9. PHP Demo 說明

本題 demo 用 PHP 內建伺服器把 TSDB 核心邏輯做成真實可跑：

  • src/Tsdb.php　series 識別（name+排序 labels 序列化）、scrape 解析曝露格式、append 樣本、instant/rate（含 counter reset 補償）/sumBy、告警狀態機（inactive→pending→firing→resolved），以 flock 持久化到 JSON。
  • public/index.php　路由 POST /api/scrape、GET /metrics、GET /api/query?expr=、GET /api/alerts，並附測試頁。
# 啟動
cd demo
php -S localhost:8027 -t public
# 瀏覽器開 http://localhost:8027：按幾次 Scrape（counter 值逐次調大）→ 跑 rate/sum → 看告警 firing
誠實聲明 Demo 為單機 JSON 模擬 TSDB；series 識別、scrape 解析、樣本追加、instant/rate（含 counter reset）/sum 查詢、告警狀態機皆為真實可執行邏輯。未做樣本壓縮、高基數最佳化、保留期回收、remote write 與聯邦。要接真 Prometheus，把本系統 /metrics 當作一個 exporter target 設進 scrape_configs 即可。
