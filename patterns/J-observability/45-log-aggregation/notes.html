㊺ 日誌聚合系統 ELK：從幾百台機器的 log 到一處可搜尋

J 可觀測性 建議 42 分鐘 Design a Log Aggregation System (ELK)

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「分散式日誌的攝取管線 + 倒排索引全文搜尋 + 多維度（等級/時間/服務）過濾」。 難點不在「存文字」，而在 每秒數十萬筆寫入的攝取吞吐、倒排索引的時間分片與保留分層、以及 高基數欄位與索引爆炸的控制。
與第㉖題（全文搜尋）的關係 本題共用「tokenize + 倒排索引」這個核心引擎，但多了三件日誌特有的事：攝取管線（agent → 緩衝 → 處理 → 索引）、等級維度（debug/info/warn/error 過濾）、時間範圍查詢（log 永遠是時序的）。把搜尋引擎放進「時間 + 等級」的座標系裡，就是日誌聚合。

1. 需求釐清

功能性需求

  • 從幾百台機器、幾十個服務收集 log，集中到一處可全文搜尋。
  • 每筆 log 是結構化文件：時間戳、等級（level）、來源服務（service）、訊息（message）+ 任意欄位。
  • 查詢支援：關鍵字（全文，多詞 AND/OR）+ 等級過濾（只看 error）+ 時間範圍（最近 1 小時）。
  • 支援聚合：依服務/等級分組計數、畫趨勢圖（Kibana 儀表板）。

非功能性需求

  • 高寫入吞吐：日誌是高頻 append-only，峰值每秒數十萬筆。
  • 查詢低延遲：「最近 15 分鐘所有 error」要秒級回應。
  • 可靠攝取：應用不能因為日誌系統慢就被拖垮 → 攝取要解耦、可緩衝、可丟（背壓）。
  • 成本可控：log 量極大，需保留分層（hot/warm/cold）與過期回收。
面試開場必問 「三支柱（pillars）要哪個？」可觀測性 = metrics / logs / traces。本題是 logs：離散的、帶上下文的文字事件，適合「事後追查某次請求到底發生什麼」。metrics 是數值聚合（第㉗題）、traces 是跨服務鏈路（第㊹題）。logs 最貴（量大、要全文索引），所以攝取吞吐與索引成本是本題主軸。

2. 規模估算

指標假設推算
機器數 × 每台日誌率500 台 × 每台 200 行/秒~10 萬 logs/s（峰值 ×3 ≈ 30 萬）
單筆 log 大小結構化 + 訊息 ~500 bytes原始寫入 ≈ 10萬 × 500B ≈ 50 MB/s
每日原始量50MB/s × 86400≈ 4.3 TB/日（原始）
索引膨脹倒排索引 + 欄位 ≈ 原始的 1.1~1.5×含索引 ≈ 5~6 TB/日
保留 30 天6TB × 30≈ 180 TB（須分層降本）
關鍵洞察成本由「量 × 保留期 × 索引膨脹」三者相乘決定。降本三招：①取樣/丟棄低價值 log（debug 不長存）②時間分片 + 保留分層（舊資料移到便宜冷儲存或不索引）③控制高基數欄位（別把每個 request_id 都當索引欄位）。

3. API 設計

# 批次攝取（內部，來自收集 agent / Logstash）
POST /api/logs    Body: application/json
  { "logs": [ { "ts":1717200000000, "level":"error",
                "service":"auth-service", "message":"login failed uid=99" }, ... ] }
→ 200 { "ingested": 3, "ids": [101,102,103] }

# 查詢（關鍵字 + 等級 + 時間範圍）
GET /api/search?q=login%20failed&level=error&from=1717200000000&to=1717203600000&limit=50
→ 200 { "total": 12, "returned": 12, "usedIndex": true,
        "hits": [ { id, ts, level, service, message }, ... ] }

# 聚合（畫趨勢/分組計數，真實系統如 ES aggregations）
GET /api/aggregate?group_by=service&level=error&from=&to=
→ 200 { "auth-service": 120, "api-gateway": 33, ... }
設計重點查詢是多維度交集：先用倒排索引把關鍵字命中的文件取出（縮小候選），再套 level + 時間範圍過濾。把「最便宜、最能縮小範圍」的條件先用（通常是時間分片 + 關鍵字 posting list），是查詢效能的關鍵。

4. 資料模型

log 文件（document）= 一筆結構化日誌
  {
    id:      自增/雪花 ID
    ts:      事件時間戳（毫秒）        ← 時序維度，決定落在哪個時間分片
    level:   debug | info | warn | error   ← 低基數，適合當 filter 欄位
    service: "auth-service"          ← 來源，分組/過濾用
    message: "login failed uid=99"   ← 全文欄位，會被 tokenize 進倒排索引
    （+ host, trace_id, ... 任意結構化欄位）
  }

倒排索引（inverted index）= 全文搜尋的心臟
  "login"  -> [101, 102, 205, ...]   ← posting list：這個詞出現在哪幾筆 log
  "failed" -> [102, 205, 318, ...]
  查 "login failed" = 兩個 posting list 取「交集」(AND)

時間分桶（time-based index）= 按時間切成多個小索引
  logs-2026.06.01-h14   logs-2026.06.01-h15  ...
  好處：查時間範圍只碰相關分桶；過期整桶刪除（O(1) 回收）；舊桶可降溫到冷儲存
為何 level 當 filter、message 當全文level 只有 4 種值（低基數），用 term filter（精確比對）即可；message 是自由文字（高基數），必須 tokenize 建倒排索引才能「含某詞就命中」。把欄位分成「過濾用（精確、低基數）」與「搜尋用（全文、高基數）」兩類，是日誌資料建模的核心判斷。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 應用伺服器(數百台) ──▶ 收集 Agent(Filebeat) ──▶ 緩衝佇列(Kafka) ──▶ 處理(Logstash)
   寫 log 到檔案/stdout      在每台機器上 tail        削峰、解耦、可重播       解析/結構化/打標
                                                                              │
                                                                              ▼
                                                          索引/儲存(Elasticsearch)
                                                          倒排索引 + 時間分片 + 分片副本
                                                                              │
                                          查詢 ◀── 視覺化/查詢(Kibana) ◀───────┘
                                          關鍵字 + 等級 + 時間範圍 + 聚合
  1. 收集 Agent（Filebeat）：每台機器上的輕量 agent，tail 應用 log 檔，往後送；應用本身不需知道後端在哪。
  2. 緩衝（Kafka）：把「產 log」與「處理/索引」解耦——Elasticsearch 慢或重啟時，log 暫存在 Kafka 不遺失，可重播；同時對突發洪峰削峰。
  3. 處理（Logstash）：解析非結構化文字（grok）、抽欄位、打 tag、丟棄雜訊、敏感資料遮罩。
  4. 索引（Elasticsearch）：建倒排索引 + 結構化欄位，按時間分片（index per hour/day），分片 + 副本水平擴展。
  5. 查詢（Kibana）：全文 + filter + 時間範圍 + 聚合儀表板。

6. 核心元件深入

結構化日誌（structured logging）

  • 應用直接輸出 JSON log（含 level、service、trace_id 等欄位），省去下游用 regex/grok 硬解析的脆弱與成本。
  • 「先結構化、再傳輸」是現代最佳實務；舊系統的純文字 log 才靠 Logstash grok 補救。

倒排索引（inverted index）

  • 對 message 做 tokenize（切詞、轉小寫、去標點）→ 每個詞建一條 posting list（詞 → 文件 id 清單）。
  • 關鍵字查詢 = 取各詞 posting list 的交集（AND）/聯集（OR）；posting list 已排序，交集是線性合併，遠快於逐筆掃描全部 log。
  • 真實 Elasticsearch 底層是 Lucene；本 demo 用 PHP 實作同樣的「詞 → posting list → 取交集」邏輯。

時間分片與 index rollover

  • 不要把所有 log 塞進一個巨大索引。按時間切成 logs-2026.06.01 這類分桶，達到大小/時間門檻就滾動（rollover）出新桶。
  • 查時間範圍只需掃相關分桶；過期資料整桶刪除（O(1)），不必逐筆 delete（昂貴）。

過濾與聚合

  • filter（過濾）：level/service 等低基數欄位用精確比對，可快取、不算分數，成本低。
  • aggregation（聚合）：依欄位分組計數/求趨勢（如「各服務每分鐘 error 數」），是儀表板的基礎。

保留分層 hot / warm / cold

  • hot：最近資料，放 SSD、多副本、可寫可查（最貴）。
  • warm：稍舊，唯讀、降副本、放較慢磁碟。
  • cold / frozen：很舊，移到物件儲存（S3）、按需載入，甚至只存原始不建索引；最後到期刪除。
  • 這套 ILM（Index Lifecycle Management）是把 180TB 成本壓下來的關鍵。

7. 擴展與瓶頸（寫入吞吐 / 高基數 / 索引爆炸）

階段瓶頸對策
攝取洪峰突發 log 灌爆 ES，寫入拒絕Kafka 緩衝削峰 + 背壓 + 批次寫入（bulk）
寫入吞吐單節點索引速度上限分片（sharding）：一個時間索引切多個 shard 分散到多節點並行寫
高基數（cardinality）把 request_id/user_id 當索引欄位 → 詞典爆炸、記憶體吃光高基數欄位不索引或只存不分析（keyword doc_values / 不建倒排）
索引爆炸分片過多（每小時一索引 × 多 shard × 多服務）合理 rollover 門檻、合併小分片（force merge）、ILM 自動管理
查詢慢大範圍 + 全文跨大量分片時間分片裁剪（只查相關桶）、filter 快取、限制回傳量
長期成本全量長保全索引太貴hot/warm/cold 分層 + 取樣 + debug log 短期保留

8. 取捨與面試重點

  • 全文索引 vs 成本：全部欄位都建索引查得快但儲存/記憶體爆炸；只索引必要欄位（message 全文 + 少數 filter 欄位），其餘只存不索引。索引是用空間換查詢速度，要選擇性地花。
  • 即時 vs 批次：Kafka + 近即時索引（near real-time，refresh 間隔通常 1s）兼顧吞吐與可見性；要求嚴格即時則代價是吞吐與成本。多數場景「秒級可見」足夠。
  • 保留期：留越久越好查歷史，但成本線性上升。折衷：error 長保、info 中期、debug 短期；舊資料降溫到冷儲存而非直接刪。
  • 結構化 vs 純文字：應用直接輸出 JSON log 最省下游解析；遺留系統才靠 grok 補救（脆弱、耗 CPU）。
  • logs vs metrics vs traces：高基數、要「看細節原文」用 logs；可聚合的數值告警用 metrics；跨服務單請求鏈路用 traces。別把高基數塞進 metrics label，也別用 logs 做高頻數值告警。
  • 常見追問：為何要 Kafka 在中間？→ 解耦 + 削峰 + 可重播。如何回收舊 log？→ 時間分桶整桶刪 + ILM。索引為何會爆？→ 高基數欄位 + 分片過多。

9. PHP Demo 說明

本題 demo 用 PHP 內建伺服器把攝取管線 + 倒排索引查詢做成真實可跑：

  • src/LogIndex.php　ingest（結構化文件 + 自增 id）、tokenize（切詞）、倒排索引（詞 → posting list）、search（關鍵字多詞 AND 取交集 + 等級過濾 + 時間範圍 [from,to] + 時間新到舊排序），以 flock 持久化到 JSON。
  • public/index.php　路由 POST /api/logs（批次攝取）、GET /api/search?q=&level=&from=&to=、GET / 測試頁（餵 log、用關鍵字+等級+時間查、看命中與「候選 id 數」如何被索引縮小）。
# 啟動
cd demo
php -S localhost:8045 -t public
# 瀏覽器開 http://localhost:8045：攝取幾筆 log → 用關鍵字+等級查 → 看倒排索引內容與命中
誠實聲明 Demo 為單機 JSON 模擬；攝取管線、tokenize、倒排索引（詞→posting list）、關鍵字 AND 交集、等級 + 時間範圍過濾、排序皆為真實可執行邏輯。未做索引分片、時間分桶 rollover、保留分層（hot/warm/cold）、壓縮、相關性評分（TF-IDF/BM25）與分散式合併；tokenize 採 ASCII 規則（不依賴 mb_*），中文整段視為一詞。要接真 ELK，把攝取端換成 Filebeat → Logstash → Elasticsearch，倒排索引與查詢語意一致。
