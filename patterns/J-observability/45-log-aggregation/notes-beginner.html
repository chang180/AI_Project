1. 60 秒搞懂它在幹嘛

半夜兩點，網站掛了。你想知道「到底哪裡出錯」。錯誤的線索藏在 log（日誌）裡——但你有幾百台機器，每台都在不停吐 log。 你總不能一台一台 SSH 進去用肉眼翻吧？

日誌聚合就是解這個的：把所有機器的 log 集中到一個地方，讓你能像用 Google 一樣， 打個關鍵字「login failed」、勾「只看 error」、選「最近 15 分鐘」，一秒就撈出所有相關的 log。

🔑 一句話 日誌聚合 = 幾百台機器 log 的「Google 搜尋引擎」：全部集中、建好索引，讓你用「關鍵字 + 等級 + 時間」一查就到，不用再一台台翻。

2. 為什麼這題的難點不是「把 log 存起來」？

很多人第一次看會想：「不就把每台的 log 抄到一個大檔案嗎？」——這想法漏掉了兩個會把系統壓垮的真正難關。

📌 比喻：一座沒有目錄的圖書館 想像把全世界的書都堆進一個倉庫，但沒有目錄、沒有書架編號。你要找「某本提到『恐龍』的書」，只能一本一本翻——一億本書翻到天荒地老。 log 也一樣：堆成一個大檔案容易，但「找出含某個字的那幾筆」如果要整個檔案掃一遍，幾 TB 的 log 你會等到當機。

第一個難關就是這個：要能飛快地全文搜尋。解法是先幫每個字建一張「哪幾筆 log 提到這個字」的目錄——這叫倒排索引（就是圖書館的目錄卡）。

第二個難關更隱形：log 來得超快又超多（每秒幾十萬筆）。如果接收的人來不及處理，要嘛把產 log 的應用拖慢（很糟），要嘛 log 直接掉了（查不到）。 怎麼穩穩地接住這股洪流，是第二個難關。

💭 停一下：為什麼「把 log 全堆進一個大檔案，要找就整個掃一遍」在這題行不通？

看答案 因為 log 量是幾 TB 等級，整個掃一遍要很久、很耗資源，根本做不到「秒級回應」。 我們需要先建「目錄」：每個字對應「哪幾筆 log 有它」。要查的時候翻目錄就好，只碰真正相關的那幾筆——這就是倒排索引。

3. 把系統拆成 4 個關卡

別被「系統」嚇到。整件事其實就是闖 4 關，一關一關來：

1收集：把每台機器的 log 撈出來

幾百台機器各自在吐 log。我們在每台機器上放一個小幫手（叫 Filebeat），它專門盯著 log 檔，一有新的就往外送。

📮 比喻：每棟樓的收信員 每棟大樓門口都站一個收信員（Filebeat），住戶（應用）只要把信丟進信箱（寫 log 檔），收信員就自動收走、往總部送。住戶完全不用管信怎麼送、送去哪。

👉 好處：應用程式不用知道後端在哪，只管乖乖寫 log；要換後端、後端掛了，應用都不受影響。

2緩衝：先排隊，別把後台衝垮

收信員把信源源不絕送來，但總部（建索引的地方）處理速度有限。尖峰時信來得比處理得快怎麼辦？先丟進一個超大的排隊區（叫 Kafka）。

🏪 比喻：郵局的暫存倉 雙十一包裹爆量，郵局不會叫快遞員停在門口乾等，而是先全部收進暫存倉，後面的人慢慢分。 就算分信的機器當機重啟，包裹還好好待在倉裡，一件都不會掉，修好了接著分。

這個排隊區做兩件大事：削峰（洪峰先存著慢慢消化）+ 解耦（產 log 跟處理 log 互不拖累，後台掛了 log 也不會掉）。

3處理 + 建索引：整理乾淨、做目錄

從排隊區拿出 log，先整理乾淨（拆出時間、等級、服務、訊息等欄位），再幫每個字建目錄（倒排索引）。

📇 比喻：圖書館編目員 新書進館，編目員先登記基本資料（書名、作者、分類＝結構化欄位），再做關鍵字索引卡：「提到『恐龍』的書 → 第 3、17、102 本」。 以後讀者查「恐龍」，翻索引卡就知道去拿哪幾本，不用翻遍整館。

這就是 Elasticsearch 在做的事：把 log 變成「可以一秒查到」的形狀。

4查詢：像 Google 一樣搜 log

最後給你一個搜尋框（叫 Kibana）：打關鍵字、勾等級、選時間範圍，一秒看到結果，還能畫成趨勢圖。

🔍 比喻：圖書館的查詢電腦 你不用進書庫，站在查詢電腦前打「恐龍 + 兒童區 + 2024 年後出版」，螢幕馬上列出符合的書與位置。 查 log 也一樣：「login failed + 只看 error + 最近 15 分鐘」 → 秒出結果。

注意這裡有三個條件疊在一起：關鍵字（翻目錄）、等級（只要 error）、時間（只要這段）。三個一交集，就精準命中你要的那幾筆。

✅ 四關一句話 ① 每台機器放收信員收集 → ② 丟進排隊區緩衝（削峰、不掉信）→ ③ 整理 + 建目錄（倒排索引）→ ④ 像 Google 一樣關鍵字+等級+時間查。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「查不動 / 太慢」 幾 TB 的 log，如果每次查都整個掃一遍，等到天荒地老。 所以一定要先建倒排索引（目錄），查的時候只翻目錄、只碰相關的那幾筆。先做目錄，別硬翻。
😱 怕「掉 log / 拖垮應用」 log 來得又快又多，接收端一忙不過來，要嘛把產 log 的應用拖慢，要嘛 log 直接掉了（出事卻查不到）。 所以中間放一個排隊區（Kafka）削峰、解耦，洪峰先存著、後台掛了也不掉信。先接住，再慢慢消化。
😱 怕「太貴 / 存爆」 log 量大到 30 天就幾百 TB，全部長期保留 + 全部建索引，錢直接燒光。 所以要分層保留（新的放快又貴的、舊的搬去慢又便宜的、太舊就刪）+ 不重要的 log 少留。該省的地方要狠心省。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每秒進來多少 log500 台 × 200 行/秒 ≈ 10 萬/秒洪流很大 → 一定要排隊區（Kafka）削峰
每天多大每筆 ~500 bytes → 約 4 TB/日量驚人 → 必須建目錄才查得動、要分層省成本
建了索引後多大原始的 1.1~1.5 倍 ≈ 5~6 TB/日目錄本身也佔空間 → 別什麼欄位都建索引
留 30 天要多少≈ 180 TB不分層會燒爆預算 → hot/warm/cold + 該刪就刪
💡 關鍵洞察：成本 = 量 × 保留期 × 索引膨脹 這三個相乘決定你要花多少錢。降本就從這三項下手：少留沒用的 log（debug 不長存）、舊的搬冷庫或不建索引、別把每筆都不一樣的欄位拿去建目錄。

💭 停一下：每天才 4 TB 聽起來還好，為什麼成本會失控？

看答案 因為要乘上保留天數，還要加上索引膨脹。4TB × 30 天 × 1.3 倍索引 ≈ 180TB，而且這還是「放在快又貴的 SSD 上」的話。 所以才需要分層（新的放貴的、舊的搬便宜的）和「不重要的 log 短期保留」，把這個乘積壓下來。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你給我哪些欄位、我回你什麼。講好格式雙方就能合作。

這題只要記住兩張「點餐單」：

① 攝取 log（內部，收集 agent 在用）
   POST /api/logs
   給我：{ logs: [ { 時間, 等級, 服務, 訊息 }, ... ] }   ← 可一次送一批
   我回：200「收了 3 筆，編號 101~103」

一次送「一批」而不是一筆一筆，是因為 log 量太大，批次（bulk）才有效率（少了很多次來回的開銷）。

② 查 log（你在搜尋框打的）
   GET /api/search?q=login failed&level=error&from=...&to=...
   給我：關鍵字 + 等級 + 時間範圍
   我回：{ 命中幾筆, [ {時間,等級,服務,訊息}, ... ] }

注意這張單同時帶三種條件：關鍵字（翻目錄）、等級（精確過濾）、時間範圍。系統會先用最能「縮小範圍」的條件，再套其他過濾。

9. 系統要記住哪些事？（資料模型）

📒 比喻：兩本筆記本 系統把資料記在「筆記本」裡。這題主要兩本：一本記每筆 log 長什麼樣，一本是那本目錄（倒排索引）。

📝log 簿（每筆 log 的結構化內容）

每筆 log 一列：編號、時間、等級(debug/info/warn/error)、來源服務、訊息。 把欄位拆好（結構化）的好處是：等級和時間可以精準過濾，訊息可以全文搜尋。

📇目錄簿（倒排索引——核心！）

給定一個字，要能瞬間說出「它出現在哪幾筆 log」。長這樣：

"login"  → [101, 102, 205]
"failed" → [102, 205, 318]
為什麼要這本？ 因為查「login failed」時，只要把這兩個字的清單取交集（兩邊都有的：102、205）就是答案， 完全不用翻過每一筆 log。這就是把「整館翻書」變成「翻目錄卡」的精髓。

🗂️時間分箱（讓查時間範圍變快、過期好清）

不要把所有 log 塞進一個大箱子。按時間切成一個個小箱子（如「6/1 14 點這箱」）。 查「最近 15 分鐘」只要開相關的那幾箱；某天過期了，整箱丟掉就好（不用一筆一筆刪，超快）。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
洪峰：log 一下湧太多，建索引的來不及排隊區（Kafka）削峰：先全收進來慢慢消化；批次寫入
寫太慢：單一台建索引有上限分片：把一個索引切成好幾塊，分給多台一起建
目錄爆炸：拿「每筆都不一樣」的欄位建索引那種高基數欄位別建索引（只存、不做目錄）
箱子太多：每小時×每服務都開新箱合理設定多久開一箱、把小箱合併，用自動管理（ILM）
查太慢：一次查超大時間範圍只開相關時間箱、過濾條件做快取、限制回傳筆數
太貴：全部長期保留 + 全建索引分層（新貴舊便宜）+ 不重要的 log 短期保留 + 到期刪

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 全部欄位都建索引 vs 只建必要的 全建查得快但存爆、記憶體吃光；只建必要的（訊息全文 + 少數過濾欄位），其餘只存不建目錄。索引是「用空間換速度」，要選擇性地花。
⚖️ 即時看到 vs 高吞吐 要 log 一進來「立刻」查得到，代價是吞吐和成本變高。多數場景「秒級可見」就夠了，用排隊區 + 近即時索引兼顧兩者。
⚖️ 保留期要多長？ 留越久越好查歷史，但成本線性上升。折衷：error 長保、info 中期、debug 短期；舊的搬冷庫而不是直接刪。
💭 追問：為什麼中間一定要放 Kafka 這個排隊區？ 三個理由：① 削峰——洪峰先存著，後台慢慢消化不會被衝垮；② 解耦——產 log 跟建索引互不拖累；③ 不掉信——後台當機重啟期間，log 還在 Kafka 裡，修好接著處理（甚至能重播）。
💭 追問：為什麼 level（等級）用「精確過濾」，message（訊息）卻要「建目錄」？ 因為 level 只有 4 種值（種類少），直接精確比對「是不是 error」又快又省；message 是自由文字（種類無限多），必須切詞建倒排索引，才能做到「含某個字就命中」。欄位種類少的拿來過濾、自由文字的拿來全文搜尋，是建模的關鍵判斷。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「攝取 + 倒排索引查詢」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8045 -t public
# 然後瀏覽器打開 http://localhost:8045
  1. 先攝取幾筆 log：選等級、填服務與訊息（或用批次 JSON 一次餵一批）。
  2. 看下面「倒排索引內容」：每個字出現在幾筆 log，目錄就是這樣長出來的。
  3. 查詢：打關鍵字（試試兩個字，例如 login failed）、勾等級 error，看命中哪幾筆。
玩的時候想一下查兩個字 login failed 時，注意結果是「兩個字都有」的那幾筆（取交集）。再加上等級過濾 error，看候選怎麼被進一步縮小——這就是「翻目錄 → 再過濾」的全過程。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心邏輯都在一個小檔 LogIndex.php 裡，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 一個檔 = 三個小幫手 tokenize（切詞員）→ ingest（建目錄的編目員）→ search（翻目錄 + 過濾的查詢員）。log 進來就照這個順序跑。

1切詞：把訊息剁成一個個字（tokenize）

要建目錄，得先把訊息切成一個個字，順便轉小寫（這樣 Login 和 login 算同一個字）。

function tokenize(訊息) {
    小寫 = 把訊息全轉小寫
    詞陣列 = 用「非字母數字」當刀，把它切開    // "GET /api 500" → [get, api, 500]
    回傳 去掉重複後的詞陣列                    // 同一字在一筆只記一次
}

就這幾行，把一句話變成「可以逐字建目錄」的零件。轉小寫和去重是兩個小但重要的細節。

2建目錄：記下「這個字在哪筆」（ingest）

每攝取一筆 log，先存好它本身，再把它的每個字登記進目錄：

function ingest(等級, 服務, 訊息, 時間) {
    編號 = 下一個流水號
    log簿[編號] = { 編號, 時間, 等級, 服務, 訊息 }   // ← 先把整筆存好

    for (每個字 in tokenize(訊息 + " " + 服務)) {     // 服務名也可被搜到
        目錄[字].加入(編號)                          // ← 「這個字 → 加一筆它出現的 log」
    }
    回傳 編號
}

關鍵就是那行 目錄[字].加入(編號)：每個字底下掛一串「它出現在哪幾筆」的清單（posting list）。目錄就是這樣一筆一筆長出來的。

3查詢：翻目錄取交集，再套過濾（search）

這是全題的心臟。分三步：先翻目錄縮小範圍，再套等級和時間過濾，最後排序。

function search(關鍵字, 等級, 起始時間, 結束時間) {
    // 第 1 步：翻目錄，多個字「取交集」(AND)
    候選 = 全部 log
    if (有關鍵字) {
        for (每個字 in tokenize(關鍵字))
            清單們.加入( 目錄[字] )        // 每個字一張目錄卡
        候選 = 取所有清單的交集            // ← 每個字都出現的才留（login∩failed）
    }

    // 第 2 步：對候選套「等級 + 時間範圍」過濾
    命中 = []
    for (編號 in 候選) {
        log = log簿[編號]
        if (有指定等級 且 log.等級 != 等級) 跳過       // 等級不符 → 丟
        if (log.時間 < 起始時間 或 > 結束時間) 跳過    // 不在時間範圍 → 丟
        命中.加入(log)
    }

    // 第 3 步：時間新到舊排序
    命中.依時間由新到舊排序()
    回傳 命中
}

看出順序了嗎？先用目錄（最能縮小範圍）取出候選，再套等級和時間過濾——而不是反過來把全部 log 拿來逐筆比。這個順序就是查詢能「秒級回應」的關鍵。

💭 停一下：為什麼要「先翻目錄取候選，再過濾等級/時間」，而不是先過濾等級/時間？

看答案 因為翻目錄（倒排索引）能一步把範圍從幾百萬筆縮到幾十筆，後面只要在這幾十筆裡比等級和時間，很快。 如果先逐筆檢查等級/時間，等於還是把每一筆都碰一遍，就失去建目錄的意義了。先用最強的篩子。

4交集怎麼算：兩張清單都有才算數

「取交集」是多關鍵字 AND 的核心。做法很直覺：數每個編號被幾張清單提到，被全部清單都提到的才留。

function 取交集(清單們) {
    需要 = 清單的張數                  // 例如查 2 個字 → 需要 = 2
    for (清單 in 清單們)
        for (編號 in 清單)
            計數[編號]++              // 這個編號又被一張清單提到
    回傳 所有「計數 == 需要」的編號     // ← 每張清單都有它 = 兩個字都出現
}

查 login failed：login 的清單 [101,102,205]、failed 的清單 [102,205,318]，數一數只有 102、205 被兩張都提到 → 它們就是答案。簡單但這正是搜尋引擎的底層做法。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「圖書館」解釋，這題在解什麼問題？ 幾百台機器的 log 像堆滿一倉庫卻沒目錄的書，肉眼一台台翻找不可能。日誌聚合就是把書全部集中、做好「字 → 哪幾本書有它」的目錄（倒排索引），讓你打關鍵字一查就到。
Q2：為什麼不能把 log 堆成一個大檔案、要找就整個掃？ 因為量是幾 TB，整個掃要很久、做不到秒級回應。要先建倒排索引（目錄），查時只翻目錄、只碰相關那幾筆。
Q3：中間那個「排隊區（Kafka）」是幹嘛的？ 削峰（洪峰先存著慢慢消化，不衝垮後台）、解耦（產 log 跟建索引互不拖累）、不掉信（後台當機重啟期間 log 還在，修好接著處理）。
Q4：每天才 4TB 不算多，為什麼成本會失控？ 因為成本 = 量 × 保留天數 × 索引膨脹。乘上 30 天和索引膨脹就變約 180TB，還可能全放在貴的 SSD。所以要分層（新貴舊便宜）+ 不重要的 log 短期保留。
Q5（工程）：查「login failed」時，倒排索引怎麼算出答案？ 取 login 的清單和 failed 的清單，做「交集」——兩張清單都出現的編號才留。做法是數每個編號被幾張清單提到，被全部清單都提到的就是命中。
Q6（工程）：為什麼要「先翻目錄取候選，再套等級/時間過濾」？ 因為倒排索引一步就能把範圍從幾百萬縮到幾十筆，後面只在這幾十筆比等級和時間很快。先逐筆過濾等於每筆都碰一遍，失去建目錄的意義。先用最強的篩子。
Q7（工程）：為什麼 level 用精確過濾、message 用全文索引？ level 只有 4 種值（種類少），直接精確比對又快又省；message 是自由文字（種類無限），必須切詞建倒排索引才能「含某字就命中」。種類少的拿來過濾、自由文字的拿來搜尋。

15. 30 秒回顧

日誌聚合 = 幾百台機器 log 的 Google 搜尋引擎。

4 個關卡：① 每台收集（收信員 Filebeat）→ ② 緩衝排隊（Kafka 削峰、不掉信）→ ③ 整理 + 建目錄（倒排索引）→ ④ 關鍵字+等級+時間查（Kibana）。

3 個怕：怕查不動（先建倒排索引別硬掃）、怕掉 log/拖垮應用（Kafka 削峰解耦）、怕太貴存爆（分層保留 + 該刪就刪）。

工程細節一句話：成本 = 量 × 保留期 × 索引膨脹 → 用「點餐單(API)」批次攝取、查詢帶三條件 → 兩本筆記本（log 簿 + 目錄）+ 時間分箱 → 倒排索引先取候選再過濾等級/時間 → 找出塞車環節各自拓寬。
