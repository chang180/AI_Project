1. 60 秒搞懂它在幹嘛

你在 App 上按「結帳」，轉了 3 秒才成功。你覺得「好慢喔」——但到底慢在哪？

問題是：這一次按下去，背後其實跑過好幾個不同的小程式（服務）：先到「門口服務」、再到「訂單服務」、訂單又去找「付款服務」、付款又打電話給「銀行」、順便去查「庫存」…… 這 3 秒是大家加起來的。沒有工具的話，你只知道「總共 3 秒」，完全不知道是哪個環節拖慢的。

🔑 一句話 分散式追蹤 = 幫一個請求裝上 GPS 行車紀錄器：它經過哪幾個服務、每一站待多久、哪一站塞車、哪一站出車禍， 全部畫成一張「旅程地圖」讓你一眼看懂。

2. 為什麼這題的難點不是「存紀錄」？

很多人第一次看會想：「不就是每個服務各記一筆『我花了幾秒』，存起來嗎？」——這想法漏掉了最難的那一步。

📦 比喻：包裹的轉運紀錄 一個包裹從台北寄到高雄，會經過好幾個轉運站。每個站都會各自蓋一個章：「我這站收到了、我這站幾點寄出」。 但問題來了——這些章是分開蓋在不同站的本子上。你要怎麼把它們串成同一個包裹的完整旅程，還要知道誰是誰的下一站？

這就是分散式追蹤真正的難關：每個服務只知道自己這一段，而且它們是各跑各的、各自記。 系統要做到：(1) 讓每段紀錄都帶著同一個「包裹編號」（這樣才知道是同一次請求）， (2) 還要記住「我的上一站是誰」（這樣才能把零散的章重新排成一條旅程）。

💭 停一下：如果每個服務都只記「我花了幾秒」、卻沒有共同的編號，會發生什麼事？

看答案 你會得到一大堆「某服務花了 X 秒」的零散紀錄，但完全沒辦法知道哪幾筆屬於同一次請求—— 就像一地散落的轉運章，卻不知道哪些章蓋的是同一個包裹。所以「同一個編號 + 記住上一站」才是整題的命脈。

3. 把系統拆成 4 個關卡

別被「系統」嚇到。整件事其實就是闖 4 關，一關一關來：

1發編號 + 傳下去：每段都帶著「同一個包裹號」

請求一進門，第一個服務就發一個獨一無二的編號（叫 trace_id），代表「這一整趟旅程」。 接著最關鍵的動作：每次呼叫下一個服務，都要把這個編號一起帶過去。

🎽 比喻：接力賽的接力棒 跑接力賽，棒子要一棒接一棒傳下去，中間掉了就失格。 這個編號就是接力棒——每個服務呼叫下一個時都要交棒（塞在請求的「夾帶資訊」裡）。一旦哪一棒忘了傳，旅程就在那裡斷掉。

👉 工程上這叫 context propagation（情境傳播）。它是整題最容易被追問、也最容易出包的地方。

2每段記一筆：我是誰、我的上一站是誰、我花多久

每個服務做事時，都記下一張小卡片（叫 span，一段工作）。卡片上寫： 我的編號、我屬於哪個包裹(trace_id)、我的上一站是誰(parent)、我做了什麼、花了多久。

🧾 比喻：每站的轉運單 每個轉運站都填一張單：「包裹號 #123、我是新竹站、我上一站是桃園站、我處理花了 30 分鐘」。 重點是那句「我上一站是誰」——有了它，之後才能把單子排回正確順序。

3重建旅程：把零散卡片拼回一棵樹

請求結束後，系統把這個包裹號底下所有的卡片收集起來，照「上一站是誰」一張一張接回去， 拼成一張旅程地圖（其實是一棵樹：門口 → 訂單 →（付款、庫存）→ 銀行……）。

🌳 比喻：拼家族樹 每張卡片寫了「我爸是誰」，你就能把一堆人拼成一棵家族樹。 追蹤系統也一樣：每個 span 寫了「我的父 span 是誰」，就能把一堆零散 span 拼回呼叫樹。

拼好之後，畫成火焰圖：每段是一條橫條，越長代表越慢，一眼就看出「哪一條最寬 = 哪裡塞車」。

4取樣：別把每一趟都存下來（會爆）

大網站每秒幾百萬段卡片，全部存下來貴到吐血、也根本看不完。 所以系統只抽一部分來存（例如每 100 趟留 1 趟）。這叫 取樣。

🗳️ 比喻：民調抽樣 要知道全國民意，不用問每個人，抽幾千人問就夠準。 追蹤也一樣：不用留每一趟，抽一部分就能看出系統健康狀況、又省下大把儲存。

但抽樣有個重點：同一趟旅程的所有卡片，要嘛全留、要嘛全丟，不能只留半截（半截旅程沒意義）。怎麼保證一致，後面 Part 2 會講。

✅ 四關一句話 ① 發編號 + 一路傳下去（接力棒）→ ② 每段記一張卡片（含「上一站是誰」）→ ③ 把卡片拼回旅程樹、畫火焰圖找塞車 → ④ 取樣只留一部分（省錢，但同趟要一致）。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「接力棒掉了（斷鏈）」 只要有一個服務忘了把編號傳下去，旅程就在那裡斷掉，後面全變成「不知道屬於誰」的孤兒卡片。 所以才要用標準化的傳棒方式（大家都照同一個規矩塞在請求 header 裡），最好讓工具自動傳，別靠人記得。棒子絕不能掉。
😱 怕「太多、存不下也看不完」 每秒幾百萬段，全存會把錢和硬碟燒光，而且資料多到根本找不到重點。 所以要取樣只留一部分。但又怕「剛好把出問題那趟丟了」——所以聰明做法是對「慢的、出錯的」那幾趟特別保留。該省的省，該留的留。
😱 怕「拖慢正事」 追蹤是來幫忙的，不能反而讓使用者的請求變慢。如果每記一張卡片都要等它存好才繼續，那就本末倒置了。 所以記卡片要超輕量，而且非同步送出去（先把卡片丟進旁邊的籃子，背景慢慢送，不耽誤正事）。幫忙不添亂。

5. 一張圖看懂

白話導讀，由左到右一路看：

  • 🧩 各個服務一邊做事一邊產生卡片(span)，並把編號(接力棒)一路傳下去 →
  • 📮 收集小幫手(Agent)在每台機器旁邊，把卡片湊一批批非同步送走（不耽誤正事）→
  • 🧹 收集器(Collector)做取樣決定哪幾趟要留 →
  • 🗄️ 留下的卡片寫進儲存（同一趟放一起，方便整趟撈出來）→
  • 🔍 查詢服務把同一趟的卡片拼回旅程樹，算總時間、找最慢、標關鍵路徑 →
  • 🔥 最後在 UI 火焰圖上呈現，你一眼看出「哪一段最寬 = 卡在哪」。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每秒幾個請求~10 萬個還好，但每個會「放大」——往下看
一個請求跨幾個服務平均 ~20 段(span)10萬 × 20 = 每秒 200 萬張卡片！
全部存下來要多大每張卡 ~500 bytes不取樣的話一天上看好幾十 TB → 燒錢
取樣後（只留 1%）每秒剩 ~2 萬張瞬間省 100 倍 → 取樣是生存必需，不是選配
💡 關鍵洞察：壓力來自「放大」 每秒 10 萬請求聽起來還好，但每個請求會炸成 20 張卡片，一下變成每秒 200 萬張。 所以這題的核心張力是：怎麼把這 200 萬張壓到存得起（取樣）、又別剛好丟掉出問題的那一趟。

💭 停一下：請求量沒變（10 萬/秒），為什麼要存的卡片會多到 200 萬/秒？

看答案 因為一個請求 = 一整趟旅程 = 跨好多服務 = 好多張卡片。一個請求平均產生 20 張 span 卡片， 10 萬請求就變 200 萬張。這就是「放大」——真正的成本壓力來自每請求的 span 數，不是請求數本身。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」：

① 服務上報卡片（最常用，由埋點工具自動送）
   POST /v1/traces
   給我：{ 一批 span 卡片 }
   我回：202「收到了！」（非同步，盡量別拖慢你）

注意是「一批」一起送，不是一張一張——湊批省得頻繁打擾後端，也減少對正事的干擾。

② 查一整趟旅程（重建旅程樹）
   GET /api/trace/{包裹編號}
   我回：{ 總時間, 最慢的那段, 關鍵路徑, 旅程樹 }
③ 找「可疑」的旅程（慢的、出錯的）
   GET /api/search?service=payment&min_duration_ms=500&error=true
   我回：[ 符合條件的旅程清單 ]
④ 取樣決策（教學示範）
   POST /api/sample   給我：{ 包裹編號, 取樣率N }
   我回：{ 這趟留不留, 算出來的桶號 }

第 ④ 張就是「每 N 趟留 1 趟」的判定——同一個編號永遠得到同一個答案，這樣同趟才會一致地全留或全丟。

9. 系統要記住哪些事？（資料模型）

📒 比喻：一疊轉運單 系統主要就記一種東西：一張一張的卡片(span)。每張長這樣：

🧾一張卡片(span)上有什麼

包裹編號(trace_id)、我的編號(span_id)、我的上一站(parent_span_id)、哪個服務、做什麼操作、幾點開始、花多久、成功還失敗。

為什麼「上一站」這欄最重要？ 因為它就是把零散卡片拼回旅程樹的鑰匙。每張卡片寫「我爸是誰」，系統才能照父子關係接回去。 少了這一欄，你就只有一堆排不出順序的卡片。

🗂️怎麼存才好查？「同一趟放一起」

查的時候單位永遠是「一整趟」，所以把同一個包裹編號的卡片全放在同一個櫃子(分片)。 這樣查一趟時一次撈完，不用翻好幾個櫃子拼湊。

⚠️一個常見地雷：別在卡片上亂貼標籤

卡片可以貼「標籤(tag)」描述細節。但別把「每次都不一樣」的東西當標籤（像每個使用者的 ID、完整網址）， 否則標籤種類爆炸、索引被撐爆。這叫高基數地雷——有界的、種類少的（像 HTTP 方法、狀態碼）才適合當標籤。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
記卡片拖慢使用者的正事記卡片超輕量 + 非同步批次送（先丟旁邊籃子，背景慢慢送）
每秒 200 萬張卡片灌爆後端取樣只留一部分 + 收集器湊批、限流
標籤種類爆炸（高基數）限制標籤只放「種類少」的；細節資料別當索引
全部留太貴保留期縮短（如 7 天）+ 取樣後才落地
查一趟要翻好多櫃子同一趟放同一個櫃子(依編號分片)，一次撈完
接力棒被某服務搞丟（斷鏈）用統一標準傳棒 + 工具自動注入，別靠人記得

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 開頭就決定留不留(head) vs 看完整趟再決定(tail) 開頭決定(head)便宜、簡單、同趟一致，但是「盲抽」——可能剛好把出問題那趟丟了。 看完再決定(tail)能專門留住慢的、出錯的，但要先把整趟全暫存，貴又複雜。實務常兩個混用。
⚖️ 取樣率調多低？ 抽太少 → 省錢但可能看不到問題；抽太多 → 看得清楚但貴。 折衷：平常低取樣，但對「慢的、出錯的」那幾趟強制保留，兩全其美。
⚖️ 自動埋點 vs 手動埋點 自動（工具幫你在框架上自動記卡片）覆蓋廣、省人力，但細節少； 手動（工程師自己加 span）精準但花人力。通常自動打底、關鍵環節再手動補。
💭 追問：最慢的那一段，就一定是該優化的瓶頸嗎？ 不一定！如果那段慢、卻是跟別人同時並行跑、而且不是最後完成的那個， 優化它不會讓整趟變快。真正要看的是「關鍵路徑」（決定整趟何時結束的那一串）和「self-time」（某段自己花的時間，扣掉它在等別人的時間）。
💭 追問：接力棒(編號)要怎麼跨「訊息佇列」這種非同步的場景傳？ 把編號塞進訊息的 header / metadata 一起送進佇列；下游從訊息裡讀回編號，繼續接力。 原理跟 HTTP header 一樣，只是換個地方夾帶。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心都在一個小檔 TraceStore.php 裡，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個動作 = 三段程式 拼旅程樹（buildTree）→ 找最慢 + 算關鍵路徑（analyze）→ 取樣決定留不留（sampleDecision）。卡片進來，就照這個順序處理。

1拼旅程樹：照「上一站是誰」接回去（buildTree）

把一堆零散卡片，照每張寫的「父是誰」掛回父親底下，沒有父親的就是起點(根)。

// 第一輪：每張卡片先變成一個「節點」，準備一個空的 children 籃子
for (每張卡片 s)  節點[s.span_id] = s + { children: [] }

// 第二輪：照「上一站(parent)」把自己掛到父親的籃子裡
for (每張卡片 s) {
    if (s 有父親 且 父親在這批裡)
        節點[s.parent].children.push( 自己 )   // ← 接回去！
    else
        roots.push( 自己 )                      // 沒父親 → 它是起點
}

就這兩輪迴圈，一堆扁平卡片就拼回一棵樹。父親剛好被取樣丟掉的「孤兒卡片」，就當成起點接上，不讓旅程斷掉。

2找最慢 + 走關鍵路徑（analyze）

先講一個小聰明：找瓶頸不能只看誰總時間最長（那永遠是最上面的起點，它涵蓋全部）。要看 self-time（自己花的時間）：

// 每段「自己」花的時間 = 自己總耗時 − 它底下子段的耗時總和
self_time(段) = 段.耗時 − (它所有直接子段的耗時加起來)
// → 這樣「在等銀行」的 payment 不會被誤判，真正慢的 call_bank_api 才會浮出來

最慢 span 就挑 self-time 最大的那個——這才是「真正在幹活幹很久」的瓶頸。

再來是關鍵路徑：從起點出發，每一層都挑「結束時間最晚」的那個子段往下走，一路走到底：

function 走關鍵路徑(節點) {
    路徑 = [節點]
    while (節點還有子段) {
        下一個 = 子段裡「結束時間最晚」的那個   // ← 它決定了這層何時結束
        路徑.push(下一個);  節點 = 下一個
    }
    return 路徑    // 這一串就是「決定整趟何時完成」的環節
}

為什麼挑「結束最晚」？因為一層裡有好幾段並行跑，最後收工的那個才卡住整層。把每層的「最後收工者」串起來，就是整趟的關鍵路徑——要加速就從這串下手。

3取樣：同一趟永遠同一個答案（sampleDecision）

這是降量的關鍵。重點是「同一個編號要永遠得到同樣的留/丟決定」，這樣同趟的所有卡片才會一致：

function 要不要留(包裹編號, 取樣率N) {
    桶號 = crc32(包裹編號) % N        // ← 把編號變成 0..N-1 的一個固定桶號
    return (桶號 == 0)               // 落在 0 號桶就留 → 平均每 N 趟留 1 趟
}

用 crc32 把編號雜湊成一個穩定又均勻的數字，再 % N 取桶號。同一個編號算出來永遠一樣，所以整趟要嘛全留、要嘛全丟——不會出現「半截旅程」。這就是「head-based 取樣」：在開頭看到編號就定生死。

💭 停一下：為什麼取樣要「依編號算」，而不是「每張卡片各自擲骰子」？

看答案 如果每張卡片各自擲骰子，同一趟的 20 張卡片就會有的留有的丟，拼出來是殘缺的半截旅程，毫無用處。 改成「依編號算」，同一趟的卡片算出來的桶號完全相同，就能保證整趟一致地全留或全丟。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「包裹轉運」解釋，這題的核心難關是什麼？ 每個服務（轉運站）只知道自己這一段、各自記。難在要讓每段紀錄帶著同一個包裹編號(trace_id)、還要記住「上一站是誰」(parent)，才能把零散的章重新排成一條完整旅程。沒有共同編號就拼不起來。
Q2：為什麼「接力棒掉了」這麼可怕？怎麼避免？ 只要一個服務忘了把編號傳下去，旅程就在那裡斷掉，後面全變孤兒卡片，拼不回完整樹。避免方法：用統一標準傳棒（如 W3C traceparent header）+ 讓工具自動注入，別靠人記得。
Q3：請求才 10 萬/秒，為什麼要存的卡片會到 200 萬/秒？ 因為一個請求會「放大」——它跨好多服務，平均產生 20 張 span 卡片，10 萬 × 20 = 200 萬。真正的成本壓力來自每請求的 span 數，所以取樣是生存必需。
Q4：取樣為什麼要「整趟一致」？怎麼做到？ 因為半截旅程沒意義。做法是依編號雜湊決定（crc32(trace_id) % N == 0），同一個編號永遠算出同一個答案，所以同趟所有卡片一致地全留或全丟。
Q5（工程）：最慢的 span 一定是要優化的瓶頸嗎？怎麼正確找瓶頸？ 不一定。要看關鍵路徑（決定整趟何時結束的那一串）和 self-time（某段自己花的時間，扣掉它在等子服務的時間）。涵蓋全部的根 span 總時間最長卻不是瓶頸；並行又非最後完成的慢段，優化它也沒用。
Q6（工程）：buildTree 怎麼把扁平卡片拼回樹？ 兩輪掃描：第一輪把每張卡片變成帶 children 籃子的節點；第二輪照每張的 parent_span_id 把自己掛到父節點的 children 裡，沒有父親（或父親不在這批）的當成根。
Q7（工程）：head 取樣和 tail 取樣差在哪？ head 在旅程開頭就依編號決定留不留——便宜、同趟一致，但「盲抽」可能丟掉出問題的趟。tail 先把整趟全暫存、看完再決定——能留住慢/錯的趟，但貴且複雜。實務常混用。

15. 30 秒回顧

分散式追蹤 = 幫一個請求裝 GPS 行車紀錄器，畫出它跨服務的旅程地圖。

4 個關卡：① 發編號 + 一路傳下去（接力棒）→ ② 每段記卡片（含「上一站是誰」）→ ③ 把卡片拼回旅程樹、火焰圖找塞車 → ④ 取樣只留一部分（同趟要一致）。

3 個怕：怕接力棒掉了（斷鏈 → 標準化 + 自動傳棒）、怕太多存不下（取樣，但慢/錯的強制留）、怕拖慢正事（輕量 + 非同步批次）。

工程細節一句話：壓力來自「一個請求放大成幾十張卡片」→ 用「點餐單(API)」批次上報、卡片含 parent 欄位 → 同趟放同櫃子(依編號分片)、別貼高基數標籤 → buildTree 兩輪拼樹、analyze 用 self-time + 關鍵路徑找真瓶頸、sampleDecision 依編號雜湊整趟一致取樣。
