J 可觀測性 建議 45 分鐘 Design a Distributed Tracing System

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把跨服務的因果鏈，用同一個 trace_id 串起來、靠 parent 關係重建成呼叫樹」的系統。 難點不在「存 span」，而在 context 如何跨網路傳播、由扁平 span 正確重建樹並找出關鍵路徑/瓶頸、以及 用取樣把海量資料壓到負擔得起。

1. 需求釐清

功能性需求

  • 一個使用者請求會跨 N 個微服務（gateway → order → payment → DB…）。要能回答：這條請求在每個服務各花多久？哪一段慢？哪裡出錯？
  • 每個工作單元產生一個 span（含 trace_id / span_id / parent_span_id / 服務 / 操作名 / 起始 / 耗時）。
  • 查詢：給 trace_id 取回整條 trace，重建成呼叫樹，以火焰圖呈現各 span 的時間軸。
  • 能算整條 trace 總時間、標出最慢的 span 與關鍵路徑。

非功能性需求

  • 低埋點開銷：trace 不能拖慢被追蹤的業務請求（加 span 的成本要極低、非同步上報）。
  • 高寫入吞吐：每秒可能數百萬 span。
  • 可承受漏失：監控系統的可靠性優先於完整性——掉幾條 trace 可接受（這正是取樣存在的理由）。
  • 查詢可用：依 trace_id 秒級取回；能依服務 / 操作 / 延遲區間搜尋。
面試開場必問 「metrics、logs、traces 三者要哪個？」本題是 trace（可觀測性三支柱中專門回答「單一請求跨服務的因果鏈與時間分佈」）。metrics 是聚合數值（適合告警）、logs 是離散事件文字；trace 把一個請求在各服務的足跡用因果關係串起來。接著問「head 還是 tail 取樣？」這決定收集器設計。

2. 規模估算

指標假設推算
請求量10 萬 req/s每請求平均跨 20 個 span
原始 span 率10萬 × 20~200 萬 spans/s（不取樣會爆）
取樣後head 取樣 1%~2 萬 spans/s 真正落地儲存
單 span 大小~500 bytes（含 tags）2萬 × 500B ≈ 10 MB/s ≈ ~860 GB/日
保留7 天≈ 6 TB（取樣後；不取樣是 100 倍）
關鍵洞察原始 span 量是請求量 × 每請求 span 數的乘積，輕鬆破百萬/秒——所以取樣不是選配，是生存必需。設計的核心張力是「取樣降量」與「別漏掉出問題的那條 trace」之間的拉鋸。

3. API 設計

# 上報 span（被埋點 SDK 批次送，走 OTLP）
POST /v1/traces            # OpenTelemetry OTLP
Body: { "resourceSpans": [ ... 一批 span ... ] }
→ 202 Accepted（非同步，盡量不阻塞）

# 取回整條 trace（重建樹）
GET /api/trace/{trace_id}
→ { trace_id, total_ms, slowest, critical_path, tree:[...] }

# 搜尋 trace（找慢/錯的代表）
GET /api/search?service=payment&min_duration_ms=500&error=true&limit=20

# 取樣決策（教學示範用）
POST /api/sample   Body: { trace_id, rate }  → { sampled, bucket, rate }

4. 資料模型

span（一個工作單元；trace = 同一 trace_id 的 span 集合）
  trace_id        BINARY(16)   -- 一整條請求鏈共用（全域唯一）
  span_id         BINARY(8)    -- 單一 span 唯一
  parent_span_id  BINARY(8)    -- 指向上一層 span；根 span 為空
  service         VARCHAR      -- 服務名
  name            VARCHAR      -- 操作名（GET /checkout、SELECT…）
  start_us        BIGINT       -- 起始時間（微秒）
  duration_us     BIGINT       -- 耗時
  status          ENUM(ok,error)
  tags / attrs    JSON         -- http.method、db.statement…（小心高基數）

-- trace 不是一張表，而是「同 trace_id 的所有 span」。
-- parent_span_id 把扁平 span 連成一棵「呼叫樹」（DAG，通常是樹）。
為何依 trace_id 分片查詢的單位是「整條 trace」，把同一 trace_id 的所有 span 放在同一個分片，取回時一次撈完、不用跨分片 join。寫入則是高頻 append-only，適合寬欄/列式儲存（Cassandra / ClickHouse / ES）。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 應用服務(埋點 OTel SDK) ──OTLP──▶ 收集 Agent(每台主機, batch) ──▶ 收集器 Collector
   產 span / 傳 context                                            (取樣 head/tail)
                                                                      │ 寫入
                                                                      ▼
   UI/火焰圖 ◀── 查詢服務(重建樹/總時間/最慢/關鍵路徑) ◀── Trace 儲存(依 trace_id 分片)
  1. 埋點 SDK：在每個服務產生 span、把 trace_id 與 span_id 透過 context propagation（如 W3C traceparent header）傳到下游。
  2. 收集 Agent / Collector：批次收 span、做取樣決策、轉送儲存（解耦應用與後端）。
  3. 查詢服務：依 trace_id 撈出所有 span，重建呼叫樹，算總時間 / 最慢 / 關鍵路徑，餵給 UI 畫火焰圖。
  4. OpenTelemetry 是這套的事實標準：統一 SDK、語意慣例與 OTLP 協定。

6. 核心元件深入

trace_id / span_id 傳播（context propagation）

  • 請求進入第一個服務 → 產生 trace_id 與根 span_id。呼叫下游時，把這兩者塞進 HTTP header（traceparent）或訊息 metadata。
  • 下游服務讀到 header → 用同一 trace_id、把自己的 span 的 parent_span_id 設為上游的 span_id。這一步是整個系統的命脈：漏傳就斷鏈，trace 變成一截一截的孤兒。

parent-child 重建呼叫樹

  • 儲存裡的 span 是扁平的。查詢時用 parent_span_id 把每個 span 掛到父 span 底下，重建成樹（兩輪掃描：先建節點索引，再依 parent 掛載）。
  • 父 span 不在這批（被取樣掉/掉包）→ 出現孤兒 span，實務上接到根上、不讓樹斷。

關鍵路徑（critical path）vs 最慢 span

  • 關鍵路徑：從根出發，每層挑「結束時間最晚」的子 span 一路走到葉——這串 span 決定整條 trace 何時完成，是「該優化哪裡」的依據。
  • 最慢 span ≠ 一定在關鍵路徑上：若某個慢 span 與別人並行、卻不是最後完成的那個，優化它不會縮短總時間。找瓶頸常看 self-time（獨佔時間 = 自身耗時 − 子 span 耗時），避免把「涵蓋全部的根 span」誤判為瓶頸。

head vs tail 取樣

  • head-based：在 trace 開頭就決定整條留不留（如 hash(trace_id) % N == 0）。優點：便宜、各服務決策一致（同 trace_id 一致）、不用先全收。缺點：盲取樣——出問題的慢/錯 trace 也可能被丟。
  • tail-based：先把整條 trace 全部緩衝，看完再決定（留住「慢的、有錯的、罕見的」）。優點：留下最有價值的 trace。缺點：收集器要暫存整條、記憶體與複雜度高。

火焰圖（flame graph）

把每個 span 畫成一條水平長條，左端 = 起始時間、長度 = 耗時，依父子垂直堆疊。一眼看出「哪段最寬（最慢）、哪些並行、時間花在哪」。

7. 擴展與瓶頸（高基數 / 取樣 / 儲存）

階段瓶頸對策
埋點開銷產 span 拖慢業務請求非同步 batch 上報、span 本地緩衝、低開銷 SDK
原始 span 量200 萬 spans/s 灌爆收集/儲存取樣（head 便宜 / tail 留價值）+ Collector 批次/限流
高基數 tags把 user_id / 完整 URL 塞進 tag → 索引爆炸tag 值要有界；高基數放 span body 不建索引
儲存成本全量保留太貴短保留期 + 冷熱分層 + 取樣後才落地
查詢「整條 trace」span 散在多分片要 join依 trace_id 分片，同 trace 同片，一次撈完
跨服務斷鏈某服務沒傳 context標準化傳播（W3C traceparent）+ SDK 自動注入

8. 取捨與面試重點

  • 取樣率 vs 完整性：取樣越低越省，但越可能漏掉「剛好出問題」的 trace。常見折衷：基礎 head 取樣 + 對 error/慢請求強制保留（或上 tail-based）。
  • head vs tail sampling：head 便宜、決策一致、但盲取；tail 留住有價值 trace、但要緩衝整條、貴且複雜。實務常混用（邊緣 head 粗篩 + 後端 tail 精選）。
  • 埋點成本 vs 覆蓋率：自動埋點（框架 hook）覆蓋廣但細節少；手動埋點精準但人力貴。先靠自動埋點打底，關鍵路徑再手動補。
  • trace vs metrics vs logs：三支柱互補。trace 回答「這一條請求發生什麼」；別把高基數塞進 metrics label，該用 trace/log 的就用它們。三者可用 trace_id 互相關聯。
  • 常見追問：context 怎麼跨非同步/訊息佇列傳？→ 把 traceparent 放進訊息 header。時鐘不同步怎麼辦？→ 用相對時間/父子關係而非絕對牆鐘排序。如何找瓶頸？→ 關鍵路徑 + self-time，而非單看最大 duration。

9. PHP Demo 說明

本題 demo 用 PHP 內建伺服器把追蹤系統的核心邏輯做成真實可跑：

  • src/TraceStore.php　由扁平 span 依 parent_span_id 重建呼叫樹、算總時間、用 self-time 找最慢 span、走關鍵路徑，並做 head-based 取樣（crc32(trace_id) % rate），以 flock 持久化到 JSON。
  • public/index.php　路由 GET /（測試頁：內建一條跨 4 服務的 trace，看重建樹 + 火焰圖長條 + 標最慢 + 取樣示範）、POST /api/spans（批次收 span）、GET /api/trace/{trace_id}（回樹 + 總時間 + 最慢 + 關鍵路徑）、POST /api/sample（取樣決策）。
# 啟動
cd demo
php -S localhost:8044 -t public
# 瀏覽器開 http://localhost:8044：看 /checkout 的呼叫樹被重建、火焰圖標出最慢的 call_bank_api、取樣 1/10 的決策
誠實聲明 Demo 為單機 JSON 模擬；由扁平 span 重建呼叫樹、總時間 / 最慢 span(self-time) / 關鍵路徑、head-based 取樣皆為真實可執行邏輯。未做真正的 OTLP 收集、context 跨網路傳播、tail-based 取樣緩衝、列式/寬欄儲存與索引。要接真系統，把 span 改由 OpenTelemetry SDK 產生、用 OTLP 上報到 Collector 即可，分析邏輯一字不改。
