㉖ 全文搜尋引擎 Twitter Search：倒排索引與 BM25 相關性排序

G 搜尋/索引 建議 46 分鐘 Design a Full-text Search Engine

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把文件拆成詞、建倒排索引（term → 哪些文件含此詞），查詢時用 BM25 算相關性並排序」的題目。 難點不在「找到含某詞的文件」（那是倒排索引的基本功），而在 怎麼排出最相關的、怎麼把索引擴展到每分鐘數百萬則新貼文、怎麼做到近即時可搜。

1. 需求釐清

功能性需求

  • 輸入關鍵字，回傳相關文件（貼文）並依相關性排序，而不是隨意列出。
  • 布林查詢：AND（含全部詞）、OR（含任一詞）、片語查詢 "exact phrase"。
  • 分頁（每頁 N 筆、可翻頁）、可選過濾（時間、作者、語言）。
  • 近即時：新貼文發出後數秒內可被搜到。

非功能性需求

  • 低延遲：查詢 p99 < 200ms。
  • 高吞吐：搜尋 QPS 高，且寫入（新貼文）也高。
  • 可擴展：文件數可到數十億，索引需分片。
  • 相關性：排序品質是核心體驗（爛排序＝沒用）。
面試開場必問 「全文搜尋 vs LIKE '%word%' 差在哪？」 — LIKE 要掃全表（O(N) 且無法用一般索引）、不能依相關性排序、不能處理大小寫/分詞/多詞。全文搜尋靠倒排索引把「詞 → 文件」預先建好，查詢變成 O(命中數)，並能用 TF-IDF / BM25 排序。

2. 規模估算

指標假設推算
新貼文（寫）5 億/日~5,800 寫 QPS（峰值 ×3 ≈ 1.7 萬）
搜尋（讀）2 億次查詢/日~2,300 讀 QPS（峰值 ≈ 7,000）
單篇大小貼文 ~300 bytes5 億×300B ≈ 150 GB/日 原文
倒排索引大小索引約原文 0.3～0.5×~50～75 GB/日新增索引
詞典（term dictionary）不同詞數有上限（語言）數百萬～數千萬個 term，可常駐記憶體
關鍵洞察寫入與索引大小都很大 → 必須分片（shard）；查詢要 scatter-gather 到各分片再合併排序。詞典相對小，可放記憶體加速。

4. 資料模型

# 倒排索引（核心）：term → postings list
postings:
  "index"  → [ {docId:1, tf:3, pos:[5,12,40]}, {docId:7, tf:1, pos:[2]} ]
  "search" → [ {docId:1, tf:2}, {docId:3, tf:5}, {docId:7, tf:1} ]
  #            ↑ tf=term frequency（該詞在該文件出現幾次）；pos 供片語查詢

# 詞典 term dictionary：term → df（含此詞的文件數）+ postings 位置指標
dict:
  "index"  → { df: 2, ptr: ... }

# 文件儲存 doc store：docId → 原文與 metadata（供回顯/highlight）
docs:
  1 → { title, body, author, ts, len }   # len=文件詞數，BM25 要用

# 全集統計
stats: { N: 文件總數, avgdl: 平均文件長度 }
為何要存 doc 長度與 avgdlBM25 會用「文件長度 / 平均長度」做長度正規化——長文件天生會命中較多詞，需要懲罰，避免長文一律排前面。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 寫(新貼文) ─▶ Ingest ─▶ 分析器(tokenize) ─▶ 建倒排索引 ─▶ Index Shard 1..M（含副本）
                                                              ▲
 讀(查詢) ──▶ Query 解析 ─▶ scatter 到各 shard ─▶ 各 shard 取 postings + BM25 評分
                                          │
                                          └─▶ gather 合併、全域排序 ─▶ Top-K 回傳
  1. Ingest → 分析器：tokenize（小寫化、切詞、去 stopwords、可選 stemming），把文件變成詞流。
  2. 建倒排索引：每個詞寫進對應 postings（docId → tf、位置）；更新 df、文件長度、avgdl。
  3. Query 解析：把查詢字串也 tokenize，解析 AND/OR/片語。
  4. 評分排序：各分片用 BM25 算分，scatter-gather 合併出全域 Top-K。
  5. 分片 + 副本：分片解決容量與寫入吞吐，副本解決讀吞吐與高可用。

6. 核心元件深入

倒排索引結構

核心是 term → postings list。postings 依 docId 排序，便於做交集（AND）時雙指標歸併；常用 delta 編碼 + 壓縮（如 VByte）省空間。詞典（term dictionary）通常用 Trie / FST 存，可常駐記憶體。

TF-IDF 與 BM25

  • 直覺：一個詞在某文件出現越多次（TF↑）→ 越相關；但這個詞若在所有文件都出現（如 the）→ 沒鑑別度（IDF↓），該降權。
  • BM25 公式（業界標準，k1=1.2, b=0.75）： 
    idf(t)   = log( (N - df + 0.5) / (df + 0.5) + 1 )
score(t) = idf(t) · ( tf·(k1+1) ) / ( tf + k1·(1 - b + b·dl/avgdl) )
查詢總分 = Σ score(t)   （t 為各 query term）
  • BM25 比純 TF-IDF 好在：① TF 飽和（出現 100 次不會比 10 次線性高 10 倍，邊際遞減）；② 用 dl/avgdl 做文件長度正規化，公平對待長短文。

布林查詢：交集與聯集

AND = 各 query term 的 postings 取交集（雙指標歸併，因 postings 已排序）；OR = 取聯集。命中候選集出來後，一律用 BM25 排序。

片語 / 位置索引

"full text search" 要連續出現 → postings 需記每個詞的位置(position)，比對時檢查位置是否相鄰（postext = posfull+1…）。代價是索引變大。

分析器：stemming 與 stopwords

stemming（running→run、searches→search）讓查詢與文件詞形一致；stopwords（the/a/is）幾乎無鑑別度，可在分析階段移除以省索引與雜訊（但片語查詢時要小心被去掉）。

7. 擴展與瓶頸

階段瓶頸對策
單機索引記憶體 / 容量裝不下依 docId 或 term 雜湊分片(shard)
查詢延遲分片多、要查全部scatter-gather：併發查各 shard，gather 合併 Top-K
新貼文要快被搜到整段重建索引太慢近即時 refresh：寫入小段 segment、定期 refresh（如 1s）；背景 merge
熱門查詢重複算同 query 一直重算快取熱門 query 的結果（注意新文件造成失效）
讀吞吐查詢 QPS 高每 shard 多副本(replica)分攤讀，兼顧高可用
Lucene 的做法索引是一段段不可變的 segment，寫入只追加新 segment（近即時可搜），背景把小 segment merge 成大 segment。不可變讓快取、複製都簡單。

8. 取捨與面試重點

  • 精確 vs 相關性：資料庫 LIKE 精確子字串但無排序、且慢；全文搜尋給「最相關」的排序，是模糊但有用的結果。
  • 即時索引 vs 批次：近即時 refresh 讓新文件秒級可搜，但頻繁 refresh 產生大量小 segment（讀變慢，要 merge）；批次建索引吞吐高但延遲大。
  • 位置索引 vs 不存位置：存位置才能片語/鄰近查詢，但索引顯著變大、寫入變慢——按需求取捨。
  • stopwords / stemming：提升召回與省空間，但可能傷害精確片語（"to be or not to be" 全是 stopwords）。
  • 自建 vs Elasticsearch：面試講清楚倒排索引 + BM25 + 分片副本後，務實答案常是「用 Elasticsearch / Lucene」，把心力放在 schema、分片策略、相關性調校。

9. PHP Demo 說明

本題 demo 著重於搜尋引擎的核心邏輯（也是面試真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/InvertedIndex.php　分析器 tokenize（小寫化 + 切詞 + 可選 stopwords）、倒排索引 term → (docId → tf)、BM25 排序（k1=1.2, b=0.75）、布林 AND/OR 查詢、索引 JSON 持久化。
  • public/index.php　路由：GET /（搜尋框 + 選 AND/OR + 列出依分數排序的結果與命中詞）、POST /api/doc（加文件）、GET /api/search?q=&op=and|or；預設塞幾筆範例貼文。
# 啟動
cd demo
php -S localhost:8026 -t public
# 瀏覽器開 http://localhost:8026 搜尋「inverted index」，切 AND/OR 看排序差異
誠實聲明 Demo 為單機、記憶體 + JSON 檔的教學索引：tokenize、倒排索引、BM25 評分、AND/OR 布林查詢、持久化為真實可執行；未實作索引分片 / 副本、位置索引與片語查詢、stemming、scatter-gather、近即時 refresh、查詢快取。要上正式環境，把倒排索引與評分換成 Elasticsearch / Lucene，查詢與排序語意一致。
