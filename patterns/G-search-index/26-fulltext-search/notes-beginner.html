1. 60 秒搞懂它在幹嘛

想像 Twitter 上有 幾十億則貼文。你在搜尋框打「地震」，按下 Enter—— 系統要在不到 0.2 秒內，從那幾十億則裡，找出最相關的幾則排在最前面給你看。

全文搜尋引擎就是在做這件事。它的難，難在三點：文件多到爆、要夠快、而且找到的還要「照相關程度排好順序」——不是隨便列一堆含這個詞的就交差。

🔑 一句話 全文搜尋 = 幫幾十億篇文章建一本「超級索引頁」：你查一個詞，它馬上告訴你「這詞出現在哪幾篇」，再依「誰最相關」排好順序給你。

2. 為什麼不能「一篇篇翻」？

很多人第一個想法是：「搜尋不就是把每篇文章打開來看有沒有這個詞嗎？」——這想法沒錯，但慢到不能用。關鍵在一個你早就懂的生活經驗——

📖 查書末的「索引頁」 你想在一本 800 頁的書裡找「光合作用」這四個字。你不會從第 1 頁翻到第 800 頁一頁頁找（太慢）。 你會翻到書末的索引頁，那裡寫著「光合作用…… 第 142、288、501 頁」，你直接翻過去就好。

「一篇篇打開來找」在程式裡就是資料庫的 LIKE '%地震%'：要掃過每一篇，文件越多越慢，而且找到一堆之後不知道哪篇最相關。

搜尋引擎用的就是「書末索引頁」這招，名字叫 倒排索引。系統事先把每篇文章拆成一個個詞，做成一張表：「某個詞 → 出現在哪幾篇文章」。 之後你查「地震」，它直接翻到那一格，馬上拿到「含地震的文章清單」，不用碰其他幾十億篇。

💭 停一下：為什麼叫「倒排」索引？跟一般的「文章 → 內含哪些詞」差在哪？

看答案 一般我們想的是「打開一篇文章 → 看它有哪些詞」（文章在前、詞在後）。倒排索引把方向反過來：「給一個詞 → 列出含它的文章」（詞在前、文章在後）。因為你搜尋時手上拿著的是「詞」，想要的是「文章」，所以反過來建表才查得快。就像書末索引是「詞 → 頁碼」，不是「頁碼 → 詞」。

3. 把系統拆成 4 個關卡

別被「搜尋引擎」嚇到。整件事其實就是闖 4 關，一關一關來：

1切詞：把文章剁成一個個「詞」

要建索引，得先把文章拆成一個個詞。這一步叫 tokenize（切詞）。順手還會做幾件小事：

  • 統一小寫：讓「Apple」和「apple」算同一個詞。
  • 丟掉廢話詞：像 the、a、is、的、了 這種每篇都有、毫無辨識度的字，可以丟掉省空間（這些叫 stopwords）。
🍳 比喻就像做菜前把食材切好洗好。文章是整顆高麗菜，切詞就是切成一絲一絲，後面才好處理。

2建索引：寫出那本「索引頁」

把每個詞登記進「索引頁」：這個詞，出現在哪幾篇文章。順便多記兩個小數字，後面排序要用：

  • 這個詞在那篇文章出現幾次（出現越多次，通常越相關）。
  • 這個詞總共出現在幾篇文章（出現在越少篇，代表越「稀有」、越有辨識度）。
📇 比喻就是真的在編一本書末的索引頁：把整本書讀過一遍，把每個關鍵詞和它出現的頁碼都記下來。編好後，查任何詞都是秒查。

3查詢：多個詞要「都含」還是「含其一」？

你打的常常不只一個詞，比如「地震 預警」。系統要先問：你是要兩個詞都含的，還是含任一個就好？

🎯 比喻：兩張名單對一對 索引頁查「地震」給你一張文章清單 A，查「預警」給你另一張清單 B。
  • AND（都要含）＝ 兩張清單的共同部分（A 和 B 都出現的文章）→ 結果少而精。
  • OR（含其一）＝ 兩張清單合起來（A 或 B 任一出現的）→ 結果多而廣。

還有一種「片語查詢」：你加引號搜 "地震 預警"，要這兩個詞緊鄰、照順序出現，不能東一個西一個。這就要在索引裡多記每個詞的位置（在第幾個字），比對時看它們是不是排在隔壁。

4排序：把「最相關」的排到最前面

找到一堆含這些詞的文章後，最關鍵的一步：誰排第一？ 排爛了，搜尋就等於沒用。系統用一個叫 BM25 的打分公式。它的直覺其實很符合常識——

⭐ 比喻：哪篇最「切題」？三個常識
  • 🔁 這個詞在某篇出現越多次 → 那篇越可能在講這件事 → 加分。
  • 💎 這個詞本身越罕見（很少文章用到）→ 一旦出現就越有辨識度 → 加重分。（「的」到處都有，加分趨近於 0）
  • 📏 同樣含 5 次，短文章比長文章更切題（長文塞一堆詞本來就容易碰到）→ 短文加分、長文要打折。

把每個查詢詞的分數加起來，分數高的排前面，取前幾名（Top-K）回給你。這就是你 Google 一下、最相關的排最上面的祕密。

✅ 四關一句話 ① 切詞把文章剁成詞 → ② 建索引頁記「詞 → 哪幾篇」 → ③ 查詢時做 AND/OR清單對一對 → ④ 用 BM25 把最相關的排最前面。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「排錯順序」 找到含某詞的文章是基本功（索引頁一翻就有），真正的考驗是排序。如果把不相關的爛文章排在前面，使用者翻兩頁就走了，搜尋形同虛設。所以 BM25 排序品質是這題的核心，排得準 > 找得到。
😱 怕「慢」 搜尋要每次都在 0.2 秒內回應，慢了使用者就跑了。這就是為什麼絕對不能「一篇篇翻」（那是 O(全部文章)），而要靠事先建好的索引頁（查詢只碰「命中的那幾篇」）。事先多花力氣建索引，換查詢飛快。
😱 怕「裝不下 / 跟不上」 幾十億篇文章的索引，一台機器記憶體裝不下；而且每分鐘還有幾百萬則新貼文要馬上能被搜到。所以索引要切成很多片分給多台機器（分片），新貼文要能幾秒內進索引（近即時）。太大就分開放、太慢就邊寫邊讓人搜。

5. 一張圖看懂

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw）

白話導讀，分成「寫」和「讀」兩條路：

  • ✍️ 寫（有新貼文）：新貼文進來 → 🔪 切詞剁成詞 → 📇 寫進索引頁（更新「詞 → 哪幾篇」）→ 分散存到好幾片索引分片。
  • 🔍 讀（有人搜尋）：查詢字串也先切詞 → 同時問每一片索引「你那有沒有？」（scatter，撒出去）→ 各片各自用 BM25 打分 → 把各片結果收回來合併、全域排序（gather，收回來）→ 回傳最相關的前幾名給你。
記這個畫面「撒出去問每一片、再收回來合併排序」就是 scatter-gather。像老師出一題，叫每一組各自找答案（撒），再把各組答案收齊、選出最好的（收）。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每天多少新貼文（寫）~5 億則/日 → 平均每秒約 6 千、尖峰約 1.7 萬寫得很兇，索引要邊寫邊扛
每天多少次搜尋（讀）~2 億次/日 → 尖峰約每秒 7 千讀也很兇，要靠副本分攤
原文有多大5 億 × 每則 300 位元組 ≈ 每天 150 GB累積很快，一台機器裝不下
索引頁有多大大約是原文的 0.3～0.5 倍 → 每天約 50～75 GB索引也很大 → 一定要分片
不同的詞有幾個（詞典）數百萬～數千萬個相對小，可整本放記憶體，加速查找
💡 關鍵洞察：兩頭都很重 這系統「寫」和「讀」都很大：新貼文一直灌進來（要建索引），搜尋也一直進來（要查索引）。 所以不像有些系統只擔心讀或只擔心寫——這題兩頭都要顧，這也是為什麼非分片＋副本不可。

💭 停一下：為什麼「不同的詞」雖然有上千萬個，卻可以整本放進記憶體，但「索引頁全部內容」卻不行？

看答案 「不同的詞」只是詞的清單本身（像目錄的詞條），上千萬個也不過幾百 MB，記憶體放得下。但「索引頁全部內容」是每個詞 + 它出現在哪幾億篇文章的清單，那是好幾百 GB，遠超一台機器的記憶體——所以詞典可常駐記憶體加速，但完整索引必須分片存到很多台機器（甚至落地到磁碟）。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住兩張「點餐單」：一張放東西進去（寫），一張搜東西出來（讀）。

① 放一篇新文章進索引（寫）
   POST /api/v1/documents
   給我：{ 標題, 內文, 作者, 時間 }
   我回：201「收進來了，編號 12345」

系統收到後，會默默把這篇文章切詞、寫進索引頁，讓它之後搜得到。

② 搜尋（讀，最常用）
   GET /api/v1/search?q=倒排+索引&op=and&page=1&size=10
   ─ q：要搜的字   ─ op=and/or：都要含 / 含其一
   ─ page/size：第幾頁、每頁幾筆（分頁）
   我回：{ 共幾筆, 結果:[ {編號, 分數, 標題, 命中的關鍵字} ... ] }

注意回傳裡每筆都有一個 分數（score）——那就是 BM25 算出來的相關性，分數高的排前面。還會把命中的關鍵字標亮（highlight）讓你一眼看到為什麼這篇被選上。

9. 系統要記住哪些事？（資料模型）

📒 比喻：四本筆記本 系統要把資料記在「筆記本」裡。這題的核心就四本，全部都是為了「查得快、排得準」：

📇索引頁（最核心：詞 → 哪幾篇）

每個詞一格，裡面放含它的文章清單，每篇記上「出現幾次、在第幾個字（位置）」。例如：

"索引" → [ 第1篇(出現3次, 位置5,12,40), 第7篇(出現1次, 位置2) ]
"搜尋" → [ 第1篇(出現2次), 第3篇(出現5次), 第7篇(出現1次) ]

記「位置」是為了片語查詢（看兩個詞是不是排隔壁）；不需要片語就能省下這部分。

📕詞典（所有不同的詞 + 指向它在索引頁的位置）

像書的目錄：列出所有出現過的詞，每個詞記「總共出現在幾篇」（df，BM25 算稀有度要用）和「去哪找它的清單」。因為相對小，可整本放記憶體。

📗原文簿（編號 → 原本的文章內容）

索引頁只放「詞」，但搜尋結果要顯示原本的標題內文給你看，所以要另外存原文。這裡還會記每篇的長度（幾個詞）——BM25 要拿它跟「平均長度」比，幫長文打折。

📘全集統計（總共幾篇、平均多長）

只有兩個數字：總文章數 N 和 平均文章長度。BM25 公式裡稀有度要用 N、長度打折要用平均長度，所以全域記著這兩個值。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
索引太大，一台機器記憶體裝不下把索引切成好幾片（分片）分給多台機器
分片多了，一次查詢要問很多片scatter-gather：同時問所有片，再把結果收回來合併排序
新貼文要馬上搜得到，但重建索引很慢新貼文先寫成小段索引，每隔約 1 秒讓它可被搜（近即時），背景再慢慢把小段合併成大段
熱門關鍵字一直被重複搜、重算把熱門查詢的結果快取起來（但有新文章時要記得更新）
搜尋的人太多（讀很兇）每片索引多放幾個副本分攤查詢，順便防一台壞掉
業界做法：Lucene／Elasticsearch 真實世界的搜尋引擎（Lucene，以及包它的 Elasticsearch）把索引存成一段段「寫好就不再改」的小段（segment）。新貼文只是追加一段新的（所以幾秒內就能搜到），背景再把很多小段合併成大段。因為每段「不會再改」，所以快取、複製都變得超簡單。

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 精確 vs 相關 資料庫 LIKE 能找「一字不差的子字串」但慢且不會排序；全文搜尋給的是「最相關」的結果——可能不是一字不差，但更有用。搜尋要的是「有用」，不是「死板地精確」。
⚖️ 即時 vs 省力 要新貼文秒級可搜 → 得頻繁產生小段索引（之後讀變慢、要花力氣合併）；要省力 → 累積一批再一起建（吞吐高，但新貼文要等比較久才搜得到）。
⚖️ 存位置 vs 省空間 在索引裡多記每個詞的「位置」 → 能做片語查詢（"完全 這樣"）；但索引會明顯變大、寫入變慢。用不到片語就別存。
💭 追問：丟掉廢話詞（stopwords）很省空間，那為什麼有時又不能丟？ 因為有些片語整句都是廢話詞，例如英文名句 "to be or not to be"（to/be/or/not 幾乎都是 stopwords）。全丟掉就搜不到這個片語了。所以省空間的同時，要小心別傷到片語查詢——這就是取捨。
💭 追問：那要不要自己從頭刻一個搜尋引擎？ 面試時要講得清楚倒排索引 + BM25 + 分片副本的原理（證明你懂）；但務實的答案常是「直接用 Elasticsearch / Lucene」，把力氣花在設計欄位、分片策略、調校相關性，而不是重造輪子。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小搜尋引擎，讓你親眼看到「索引 + BM25 排序」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8026 -t public
# 然後瀏覽器打開 http://localhost:8026
  1. 它預設已經塞了幾筆範例貼文（等於已經建好索引頁）。
  2. 在搜尋框打「inverted index」，看它列出依分數排序的結果和命中的關鍵字。
  3. 切換 AND / OR，看結果數量怎麼變（AND 變少而精，OR 變多而廣）。
玩的時候想一下看每筆結果旁邊的分數（score）——那就是 BM25。試著找一篇很短、又重複出現那個詞的貼文，看它是不是排很前面（短＋出現多次＝高分）。再切 AND/OR，體會「兩張清單取交集 vs 聯集」的差別。
誠實聲明 這個 demo 是單機、用記憶體＋JSON 檔的教學版：切詞、倒排索引、BM25 排序、AND/OR 查詢、存檔都是真的能跑；但沒有做分片／副本、位置索引與片語查詢、詞形還原、scatter-gather、近即時 refresh、查詢快取。要上線就把核心換成 Elasticsearch / Lucene，原理一模一樣。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心其實就一個小檔（InvertedIndex），裡面剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 一個檔 = 四個小幫手分工 雖然只有一個檔，但裡面住著四個小幫手：切詞員 tokenize（把文章剁成詞）→ 登記員 addDocument（寫那本索引頁：詞→哪幾篇→出現幾次）→ 對清單員 candidateDocs（AND 取交集／OR 取聯集）→ 評審 bm25（幫每篇打相關性分數）。一篇文章進來建索引、一個查詢進來找答案，都靠這四個輪流上場。

1切詞員：把文章剁成一個個詞（tokenize）

第一關，把一段文字剁成乾淨的詞：統一小寫、遇到非英數字就切一刀、丟掉廢話詞。

function 切詞(文字) {
    小寫文字 = 全部轉小寫           // Apple 和 apple 算同一個詞
    詞陣列  = 用「非英數字」切開    // 標點、空白都當成切點
    foreach (詞 in 詞陣列)
        if (詞 不是廢話詞)         // the/is/of… 沒辨識度，丟掉
            收進結果
    return 結果
}

這就是前面「做菜前把食材切好洗好」那一步。建索引和查詢都會先呼叫它——這很重要：兩邊用同一把刀切，存進去的詞和查的詞才對得上（不然你存 apple、查 Apple 就永遠搜不到）。

2登記員：寫那本索引頁（addDocument）

新文章進來，先切詞，數一數這篇有幾個詞（文章長度），再把每個詞 → 這篇 → 出現幾次登記進索引頁。

function 加入文章(標題, 內文) {
    詞們 = 切詞(標題 + " " + 內文)
    文章長度[這篇] = 詞們的數量          // 之後 BM25 幫長文打折要用

    // 先數這篇裡每個詞出現幾次（tf＝詞頻）
    foreach (詞 in 詞們)
        次數[詞] = 次數[詞] + 1

    // 登記進索引頁：詞 → (這篇 → 出現幾次)
    foreach (詞, 幾次 in 次數)
        索引頁[詞][這篇] = 幾次
}

索引頁[詞][這篇] = 幾次 這一行就是整本「書末索引頁」的本體——術語叫 倒排索引（term → docId → tf）。建好後，查任何詞都是「翻到那一格」秒拿清單，不用碰其他文章。

💭 停一下：為什麼要先在這篇文章內把每個詞「數成 tf（出現幾次）」，而不是看到一個詞就往索引頁塞一次？

看答案 因為索引頁要存的是「這個詞在這篇出現幾次」這個數字（tf），不是出現一次就記一筆。先在這篇內把次數加總好，一篇一個詞只寫一格，索引才乾淨、又剛好存到 BM25 要用的 tf。

3對清單員：AND 取交集、OR 取聯集（candidateDocs）

查「inverted index」時，先各自從索引頁拿出兩張清單，再看你要 AND 還是 OR：

// 先拿每個查詢詞的「文章清單」
清單["inverted"] = 索引頁["inverted"] 的所有文章
清單["index"]    = 索引頁["index"]    的所有文章

if (op == "and") {                  // 都要含 → 取交集
    候選 = 清單[第一個詞]
    foreach (其餘的詞)
        候選 = 候選 ∩ 這個詞的清單     // 一路取交集，空了就停
} else {                            // op == "or" → 含其一 → 取聯集
    候選 = 把所有清單「合起來去重」
}

這就是前面「兩張名單對一對」：AND＝共同部分（交集，少而精）、OR＝合起來（聯集，多而廣）。注意 AND 時只要某個詞在索引頁裡完全不存在，它的清單是空的，交集就直接變空——所以「含一個沒人用過的詞」會查無結果，很合理。

4評審：BM25 幫每篇打相關性分數（bm25TermScore）

這是全題的心臟。候選文章拿到後，對每個查詢詞算一個分數加起來。別被公式嚇到，它就是把三個常識翻成數字：

function 算分(詞, 這篇) {
    tf  = 索引頁[詞][這篇]            // 這詞在這篇出現幾次
    df  = 這詞總共出現在幾篇
    dl  = 這篇有多長；  avgdl = 全部文章平均多長

    稀有度 = 看 df 算（df 越小→這詞越罕見→稀有度越高）   // 「的」到處都有→趨近 0
    加分   = 稀有度 ×（tf 越高→加越多，但會「越加越緩」）
                     ×（這篇 dl 比 avgdl 長→打折；短→加分）
    return 加分
}
// 一篇的總分 = 把每個查詢詞的「加分」全部加起來；分數高的排前面

對著前面「⭐ 三個常識」一句一句看：① 詞頻越高加分（tf）、② 詞越常見扣分／越罕見加重分（稀有度，看 df）、③ 文章越長打折（dl ÷ avgdl）。BM25 真正的公式只是把這三句寫成數學式（還加了 k1、b 兩個旋鈕微調力道），方向跟常識完全一致，背不背公式都不影響你懂它。

💭 停一下：兩篇都含「index」3 次，一篇 20 個詞、一篇 200 個詞，BM25 會給誰較高分？為什麼？

看答案 20 個詞的短文較高分。因為長文章天生塞更多詞、本來就比較容易碰到「index」3 次，沒什麼了不起；短文裡同樣出現 3 次，代表它「整篇都在講這個」、更切題。所以 BM25 用「這篇長度 ÷ 平均長度」幫長文打折、短文加分，這就是常識③。
🧭 把程式對回前面的關卡
關卡（前面講的）哪個函式關鍵那段
① 切詞tokenize()小寫化 + 非英數字切開 + 去廢話詞
② 建索引頁addDocument()索引頁[詞][這篇] = 幾次（倒排索引）
③ AND / OR 對清單candidateDocs()交集 ∩ ／ 聯集 ∪

👉 重點：真實系統會把「一個 PHP 檔 + JSON 存檔 + 迴圈過濾」換成 Lucene / Elasticsearch（分片、副本、近即時 refresh、scatter-gather）。 但上面這四段的核心邏輯——倒排索引（詞→哪幾篇→幾次）＋ BM25（罕見＋出現多＋文章短＝排前面）——一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「書末索引頁」解釋，為什麼搜尋不能「一篇篇翻」？ 一篇篇翻（資料庫 LIKE）要掃過每一篇，文件越多越慢，且不知道哪篇最相關。搜尋引擎像查書末索引頁：事先做好「詞 → 出現在哪幾篇」的倒排索引，查一個詞直接翻到那一格拿清單，不用碰其他幾十億篇——又快又能附帶相關性資訊。
Q2：「倒排」是反過來的什麼？ 一般想法是「文章 → 內含哪些詞」；倒排索引反過來成「詞 → 含它的文章清單」。因為搜尋時手上拿的是詞、想要的是文章，反過來建表才查得快。
Q3：BM25 憑哪三個常識決定誰排前面？ ① 詞在某篇出現越多次→那篇越相關（加分）；② 詞本身越罕見（很少文章用到）→越有辨識度（加重分），常見的「的」幾乎不加分；③ 同樣出現次數下，短文章比長文章更切題→短文加分、長文打折（用文章長度 ÷ 平均長度做正規化）。
Q4：搜「地震 預警」，AND 和 OR 結果差在哪？ 把「地震」的文章清單和「預警」的文章清單對一對：AND＝兩張清單的交集（兩個詞都含），結果少而精；OR＝兩張清單的聯集（含任一個），結果多而廣。兩者拿到候選後都用 BM25 排序。
Q5（工程）：為什麼索引要「分片」，又為什麼查詢要 scatter-gather？ 幾十億篇的索引一台機器裝不下，所以切成很多片分給多台機器（分片）。但這樣一次查詢就得問所有片：同時撒出去問每一片（scatter），各片各自用 BM25 打分，再把結果收回來合併、做全域排序取前幾名（gather）。
Q6（工程）：新貼文每分鐘幾百萬則，怎麼做到「幾秒內就搜得到」又不拖慢系統？ 用近即時做法：新貼文先寫成一個小段索引（segment），每隔約 1 秒就讓它可被搜到，不必整個索引重建。背景再慢慢把很多小段合併成大段。因為每段「寫好就不再改」，所以快取與複製都很簡單（這正是 Lucene 的做法）。
Q7（工程）：為什麼資料模型裡要特別存「文章長度」和「平均文章長度」？ BM25 做長度正規化要用：長文章天生塞更多詞、容易碰到查詢詞，若不處理會一律排前面（不公平）。拿「這篇長度 ÷ 全集平均長度」幫長文打折、短文加分，才能公平比較相關性。

15. 30 秒回顧

全文搜尋 = 幫幾十億篇文章建一本「超級索引頁」，查得快又排得準。

4 個關卡：① 切詞把文章剁成詞 → ② 建索引頁（詞→哪幾篇）→ ③ 查詢做 AND/OR 對清單 → ④ 用 BM25 把最相關的排最前面。

BM25 三常識：詞越罕見、在文章出現越多次、文章越短 → 越相關、越該排前面。

3 個怕：怕排錯順序（排序是核心）、怕慢（靠索引別硬翻）、怕裝不下／跟不上（分片＋近即時）。
