⑯ Typeahead 自動完成：每個前綴的 Top-K 熱門建議

G 搜尋/索引 建議 40 分鐘 Design a Search Autocomplete / Typeahead

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「每個前綴預先算好 Top-K 熱門建議，查詢時用 O(前綴長度) 直接讀出」的題目。 難點不在「找到所有以前綴開頭的詞」（那是 Trie 的基本功），而在 怎麼排序、怎麼把排序預先算好、怎麼近即時更新， 讓每次敲鍵都能在個位數毫秒內回應。

1. 需求釐清

功能性需求

  • 使用者每敲一個字，回傳前綴對應的 Top-K 建議（通常 K = 5~10），依熱門程度排序。
  • 建議來源是歷史查詢頻率（大家搜過什麼、搜了幾次），不是隨機字典詞。
  • 新的熱門查詢要能「近即時」進榜（例如突發新聞、節慶關鍵字）。
  • 支援多語言（含中文）；前綴可能是位元組層級也可能是字元層級。

非功能性需求

  • 超低延遲：每次敲鍵都是一次查詢，p99 必須 < 100ms（含網路），伺服端通常要 < 10ms。
  • 讀遠多於寫：查詢（讀）量是更新（寫）量的數百倍，是極端讀重型。
  • 高可用：自動完成掛了不能讓主搜尋框卡住（須優雅降級）。
  • 最終一致即可：建議榜慢幾分鐘更新可接受，不需強一致。
面試開場必問 「建議是固定字典（如商品名）還是由查詢頻率動態產生？」 前者退化成純 Trie 前綴查詢；後者才有真正的系統挑戰：如何把「依頻率排序的 Top-K」預先算好並近即時更新。本筆記主軸是後者。

2. 規模估算

指標假設推算
搜尋查詢量5 億次搜尋/日~6k 搜尋 QPS
敲鍵查詢量每次搜尋平均敲 6 鍵、每鍵一次請求~35k 敲鍵 QPS（峰值 ×3 ≈ 100k）
唯一查詢詞長尾很長，去重後~1 億條獨特查詢字串
Trie 規模每詞平均 20 bytes + 節點開銷索引壓縮後約數 GB ~ 數十 GB（可全放記憶體）
更新頻率離線重建每小時批次重建，或近即時增量補丁
關鍵洞察敲鍵 QPS 是搜尋 QPS 的數倍 → 讀路徑必須記憶體化、預算化；查詢時不能當場掃描所有候選詞排序，否則延遲爆掉。

3. API 設計

# 取得前綴的 Top-K 建議（讀，超高頻，關鍵路徑）
GET /api/v1/suggest?q=自&limit=10
→ 200 {
    "prefix": "自",
    "suggestions": [
      { "term": "自動完成", "score": 98213 },
      { "term": "自助餐",   "score": 41020 },
      { "term": "自行車",   "score": 33715 }
    ]
  }

# 回報一次「被選中」事件（寫，用來更新頻率，可批次/非同步）
POST /api/v1/select   Body: { "term": "自動完成" }
→ 202 Accepted   （丟進查詢日誌，由離線管線聚合，不即時改索引）
設計重點讀 API 必須無狀態、可大量水平擴展、回應可短 TTL 快取；寫（被選中 / 被搜尋）只是「記一筆日誌」，絕不在關鍵路徑上同步改 Trie。

4. 資料模型

// 線上服務：Trie 節點（駐記憶體）
TrieNode {
  children: map<char, TrieNode>   // 子節點（中文以字元、英文以位元組）
  isWord:   bool                   // 此處是否為一個完整查詢詞
  topK:     [(term, score)] × K    // ★ 預存：以此節點為前綴的 Top-K 建議
}

// 離線資料（用來重建 Trie）
query_log      (append-only 高頻寫 → Kafka / 日誌系統)
  term, ts, user_region, ...
term_frequency (聚合結果，離線批次算)
  term  VARCHAR  PRIMARY KEY
  freq  BIGINT                     // 該查詢詞的累計/時間衰減後頻率
為何在節點預存 Top-K 若查詢時才從前綴節點向下蒐集所有完成詞再排序，最壞要走遍整棵子樹（可能上百萬個詞）。 把每個節點的 Top-K 預先算好存在節點上，查詢就退化成「沿著前綴走 N 步、回傳該節點的 topK 陣列」，複雜度 O(前綴長度)，與候選詞數量無關。這是本題的靈魂。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 讀(敲鍵) ──▶ Suggest Service ──▶ Trie + 前綴 Top-K 快取（駐記憶體）
                    │                         ▲
                    │ 被選中/被搜尋事件         │ 載入最新索引
                    ▼                         │
              查詢日誌(Kafka) ──▶ 聚合計頻 ──▶ 離線重建 Trie（批次/近即時）
  1. Suggest Service：無狀態、記憶體放整棵 Trie（或分片），查詢 O(前綴長度) 直接回 Top-K。
  2. 查詢日誌 → 聚合：所有「被搜尋 / 被選中」事件 append 進 Kafka，下游批次聚合出每個詞的頻率（可做時間衰減，讓近期熱詞加權）。
  3. 離線重建：用最新頻率重建 Trie 並在每個節點算好 Top-K，產出不可變的索引檔，再熱載入到服務（藍綠切換 / 影子載入）。

6. 核心元件深入

Trie 怎麼回前綴的所有完成詞？

  • 沿前綴每個字元往下走，走到前綴對應節點；若中途斷掉表示沒有任何詞以此前綴開頭。
  • 從該節點做 DFS/BFS，收集所有 isWord=true 的路徑 → 即為所有完成詞。但這是基本款，量大時太慢。

Top-K 排序：查詢時排 vs 預先算好

  • 方案 A：查詢時排序　走到前綴節點 → 蒐集全部完成詞 → 依頻率排序取前 K。實作簡單，但最壞 O(子樹大小 × log)，熱門短前綴（如「a」）會炸。
  • 方案 B：節點預存 Top-K（面試首選）　建索引時，自底向上把每個節點的 Top-K 算好存在節點裡（子節點 Top-K 合併 + 自己的詞，取前 K）。查詢只需 O(前綴長度) 讀出，與候選數無關。代價是索引變大、更新需重算。
  • 進階：對超熱前綴再加一層 Redis/記憶體 LRU 快取（前綴 → Top-K JSON），命中率極高。

頻率更新與重建

線上服務的 Trie 是唯讀的；「被選中 / 被搜尋」只增加日誌中的計數。離線管線定期（如每小時，或近即時的增量串流）重新聚合頻率、重建 Trie、重算各節點 Top-K，再整批熱載入。這樣讀路徑永遠不被寫鎖卡住。

近即時 vs 離線批次 離線批次（每小時重建全量）簡單穩定但延遲高；近即時（串流增量更新節點頻率與 Top-K）可讓突發熱詞數分鐘進榜，但要處理併發更新與一致性。實務常兩者並用：大盤靠批次重建，熱點靠串流補丁。

分片（Sharding）

當 Trie 大到單機記憶體裝不下，依前綴第一/前幾個字元分片（如 a–f 一台、g–m 一台）。查詢路由依首字元導到對應分片。注意分片要均衡——某些首字母（中文常用字、英文 s/c）特別熱，需再細分或加副本。

7. 擴展與瓶頸（每秒數十萬次敲鍵）

階段瓶頸對策
單機查詢慢查詢時掃子樹排序節點預存 Top-K，查詢降到 O(前綴長度)
敲鍵 QPS 暴衝每鍵一次請求無狀態服務水平擴展 + 前端防抖(debounce) + 熱前綴快取
記憶體裝不下 Trie單機放不下上億詞依前綴分片 + 索引壓縮（如 ternary/radix tree、DAWG）
熱前綴傾斜短前綴（a、自）流量集中該分片加副本 + 邊緣/CDN 快取建議結果
更新延遲熱詞太慢進榜離線全量重建 + 串流增量補丁雙軌

8. 取捨與面試重點

  • 查詢時排序 vs 節點預存 Top-K：前者省記憶體、更新即時但查詢慢；後者查詢極快但索引大、更新要重算 → 讀重型系統選後者。
  • Trie vs 反向索引/前綴 hash：Trie 天然支援前綴遍歷且能在節點掛 Top-K；用 (prefix → list) 的 hash 表也行但前綴爆炸（每個詞要存所有前綴），記憶體更兇。
  • 批次 vs 近即時更新：批次簡單一致；近即時複雜但能追熱點。多數系統雙軌並用。
  • 常見追問：
    • 怎麼做拼字糾錯（打錯也給對的建議）？→ 編輯距離 / 模糊 Trie / n-gram 索引，先糾錯再查前綴。
    • 怎麼做個人化？→ 全域 Top-K 之上，混入該使用者/地區的歷史，重新排序。
    • 怎麼時間衰減讓舊熱詞退場？→ 頻率用指數衰減或滑動視窗計數，重建時套用。
    • 中文 / 多語言怎麼切？→ 中文沒有空白分詞，前綴以字元為單位（見 demo：用 PCRE /u 切字元，因本機 PHP 未載入 mbstring）。

9. PHP Demo 說明

本題 demo 聚焦系統核心邏輯（也是面試真正的考點）：Trie + 前綴 Top-K 排序 + 頻率更新影響排名，可實際用 PHP 內建伺服器跑起來：

  • src/Trie.php　Trie 字典樹：insert(詞, 權重)、給定前綴蒐集所有完成詞。中文以 PCRE preg_split('//u') 取字元（本機未載入 mbstring）。
  • src/Suggester.php　給定前綴回傳依頻率排序的 Top-K；在節點預存 Top-K 加速（並註明真實系統如此做），選中某建議時頻率 +1 並重算受影響節點的 Top-K，排名隨之變化。
  • public/index.php　路由 + 深色測試頁，輸入框即時顯示建議、可點選建議使其頻率上升。
# 啟動
cd demo
php -S localhost:8016 -t public
# 瀏覽器開 http://localhost:8016 ：輸入前綴看 Top-K → 點建議使其頻率上升 → 再查同前綴看排名變化
誠實聲明 Demo 用記憶體 + JSON 檔模擬「駐記憶體 Trie 索引」與「查詢日誌」；單機、單棵 Trie，沒有真正的分片、Kafka、離線批次管線與 CDN 快取。 但 Trie 結構、前綴遍歷、節點預存 Top-K、選中事件使頻率上升並改變排名等核心邏輯為真實可執行，正是擴展時要水平複製/分片的那一塊。
