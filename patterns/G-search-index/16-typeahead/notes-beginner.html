1. 60 秒搞懂它在幹嘛

你在 Google、YouTube、蝦皮的搜尋框打字，還沒打完，下面就「啪」地跳出一排建議： 你打「自」，它馬上列出「自動完成」「自助餐」「自行車」。而且這排建議，是大家最常搜的那幾個， 排在最上面的就是最熱門的。

這就是 Typeahead（自動完成）。它的難，難在三點：你每按一個鍵就要回一次（超高頻）、 要在幾毫秒內回完（不然你已經打下一個字了）、而且要照熱門程度排好（最多人搜的排前面）。

🔑 一句話 自動完成 = 「你打前幾個字，我秒猜出大家最常搜的那幾個」。 秘訣是：答案事先就算好擺著，你一打字就直接拿，不是當場現算。

2. 為什麼一打字就能「秒回」建議？

很多人第一個疑問是：「全世界有上億條搜尋詞，我打一個字，它怎麼可能瞬間就從上億條裡挑出最熱門那幾個？」 關鍵在兩個你早就懂的生活經驗——

📖 經驗一：查英文字典是「按字母分層翻」 你查 "apple" 不會從第一頁一頁頁翻。你先翻到 a 區，再進 ap，再 app…… 一層一層往下走，幾步就到。這種「按字母一層一層分下去」的結構，就是 Trie（字典樹）： 打前綴就等於「走到字典裡那個位置」。
🍱 經驗二：便利商店的「熱銷排行架」 便利商店不會等你問「最近大家最常買什麼」才當場去翻一整年的發票統計。 它事先就把熱銷前幾名做成一張排行榜貼在架上，你一看就知道。 自動完成也一樣：每個前綴位置，都事先把「最熱門前幾名（Top-K）」算好擺在那。

把這兩個合起來，就是這題的靈魂：打字 → 沿字典樹走到那個前綴節點（很快，只走幾步）→ 直接讀出那個節點上事先貼好的熱門排行榜（不用現算）。所以才能秒回。

💭 停一下：為什麼一定要「事先算好」？當場現算不行嗎？

看答案 因為你每按一個鍵就是一次查詢，量超大；而且要在你打下一個字之前回完（幾毫秒）。 如果你打「自」，系統當場去把所有「自開頭」的詞（可能上百萬個）全撈出來、排序、取前幾名——太慢了，會卡。 所以改成事先（離線時）就把每個前綴的熱門榜算好擺著，打字當下只要「走過去、讀出來」。這叫「預先算好，換查詢飛快」。

3. 把系統拆成 3 個關卡

別被「系統」嚇到。整件事其實就是闖 3 關，一關一關來：

1建字典樹：把詞「按字母分層擺好」

第一步，把所有人搜過的詞，照字典樹（Trie）的方式擺好——就像英文字典按字母分層。

🌳 比喻：一棵分岔的樹 想像一棵樹，從樹根出發。要存「貓」就走「ㄇ→ㄠ」這條路；要存「貓咪」就在「貓」後面再接一個「咪」。 共用開頭的詞會共用同一段路（「貓」「貓咪」「貓砂」都先走過同一個「貓」節點）。

👉 重點：打前綴就是「從樹根照著字往下走」。打「自動」就走到「自→動」那個節點。走到哪，就代表你打了哪個前綴。

2每個前綴「預先貼好熱門排行榜」（Top-K）

光有字典樹還不夠——走到「自」這個節點，下面可能有上百萬個「自開頭」的詞。當場排序太慢。

🏆 比喻：每個路口都掛一塊「附近熱門榜」 想像字典樹的每個節點，都事先掛一塊小牌子，上面寫「從這裡往下走，最多人搜的前 5 名是這些」。 你走到「自」就看「自」那塊牌子，走到「自動」就看「自動」那塊。看牌子，不用自己數。

這個「每個前綴最熱門的前幾名」就叫 Top-K（K 通常是 5 到 10）。 它是事先算好的：依照「大家搜過幾次」（歷史頻率）排出來，存在每個節點上。

⚠️ 小細節：這些榜怎麼算？由下往上合併——先算好底層每個詞的熱度， 再把子節點的榜往上合，每往上一層就取「合起來後最熱的前 K 名」。建好後，查詢就只是「走過去、讀牌子」。

3查詢：打字 → 走過去 → 讀榜，秒回

前面兩關都是事先準備。真正使用者打字時，做的事極簡單：

🚶 比喻：照地址走到門口，直接拿櫃台早備好的東西 你打「自動」，系統就照「自→動」走到那個節點（走 2 步），直接讀出那節點貼好的 Top-K 榜回給你。 不用翻整棵樹、不用現場排序——所以無論底下有 100 個還是 100 萬個詞，都一樣快。

這就是為什麼工程上說它的查詢速度是「O(前綴長度)」： 白話講就是「只跟你打了幾個字有關，跟總共有幾百萬個詞完全無關」。打 2 個字就走 2 步，這麼簡單。

✅ 三關一句話 ① 把詞照字典樹分層擺好 → ② 每個前綴節點事先貼好熱門榜（Top-K） → ③ 打字就走過去讀榜，秒回。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「慢」 你每按一個鍵就要回一次，要在你打下個字之前（幾毫秒）回完。 所以絕不能查詢時當場掃描排序——一定要「事先算好榜、查詢只讀」。把功夫花在離線，換取線上飛快。
😱 怕「榜跟不上熱點」 突發新聞、節慶關鍵字一爆紅，大家狂搜，但你的榜還是舊的，建議就「過時」了。 所以要能近即時更新：一邊定時整批重算（穩），一邊對爆紅的詞做「小補丁」快速插隊（快）。新鮮 vs 穩定都要顧。
😱 怕「裝不下 / 擠在同一格」 上億條詞的字典樹很大，可能一台機器的記憶體裝不下；而且像「a」「自」這種超短前綴大家都在打，流量全擠一處。 所以要切開放到多台機器（分片），並對最熱的前綴多放幾份副本分攤。

5. 一張圖看懂

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw）

白話導讀，分成「平常線上秒回」和「背後默默更新」兩條線：

  • ⌨️ 你打字（每按一鍵一次請求）→
  • ⚡ 建議服務（記憶體裡放著整棵字典樹）照前綴走過去、讀出該節點的 Top-K 榜→ 秒回給你 →
  • 👀 你選了一個建議 / 真的搜了 → 系統只記一筆日誌（不馬上改榜）→
  • 📊 背後的聚合管線把日誌統計成「每個詞被搜幾次」→
  • 🏗️ 離線重建：用最新的熱度重蓋一棵新字典樹、重貼每個節點的 Top-K 榜 → 整批熱載入換上線。

關鍵：線上那棵樹是唯讀的、只負責秒回；更新榜的髒活全在背後離線做，永遠不拖慢你打字。

前綴（prefix）你打了一半的字，例如「自動」

分片（Sharding）樹太大裝不下，就切開放到多台機器
快取 (Cache)把超熱前綴的答案另外存一份，命中就更快
QPS每秒幾筆請求（衡量「多忙」的單位）
API系統之間講好的「點餐單」格式：你給我什麼、我回你什麼
debounce（防抖）你連打很快時，前端「等你停半拍」才送一次，少打擾後端

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
每天多少次搜尋~5 億次/日（約每秒 6000 次）本身就不少
每秒多少次「打字查詢」每次搜尋平均敲 6 鍵，每鍵一查 → 每秒約 3.5 萬次（尖峰再 ×3 ≈ 10 萬）打字查詢是搜尋的好幾倍，極度繁忙
總共多少種詞去重後約 1 億條獨特查詢詞字典樹會很大
字典樹多大壓縮後數 GB ~ 數十 GB還塞得進記憶體（這很重要）
📈 關鍵洞察：讀爆多、寫很少 這系統是「讀重型」的極端：大家一直在打字查（讀），但「榜要更新（寫）」相對很少。 就像一張貼在牆上的熱銷排行榜，每天被幾百萬人看，但一天只重印一次。 所以設計重心全押在「讓『讀』飛快」，更新慢一點沒關係。

💭 停一下：既然「讀」這麼多、「寫」這麼少，這給了我們什麼設計上的好處？

看答案 可以放心把所有功夫花在「事先準備」上：離線時慢慢把每個節點的 Top-K 算到好， 換取線上每次查詢都「只讀不算」。寫很少，所以「離線重算整棵樹」的成本攤下來划算。 這正是「用空間和離線時間，換查詢速度」的經典取捨。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住兩張「點餐單」：

① 取得建議（讀，超高頻，最關鍵的那條路）
   GET /api/v1/suggest?q=自&limit=10
   給我：你打的前綴「自」、想要幾筆
   我回：{ 前綴:"自", 建議:[ {詞:"自動完成", 熱度:98213},
                            {詞:"自助餐",   熱度:41020}, ... ] }

這條路每秒被打幾萬次，所以它必須超快、可以「多開幾台一起扛」（無狀態），答案還能短暫快取一下。它就是整個系統的命脈。

② 回報「被選中」（寫，只是記一筆，慢慢處理）
   POST /api/v1/select   Body: { 詞:"自動完成" }
   我回：202「收到了！」← 立刻回，不馬上改榜

為什麼「立刻回 202、不馬上改榜」？因為改榜是背後離線慢慢做的髒活。 使用者選了什麼，我們只先記一筆日誌，絕不在你打字的關鍵路徑上去動那棵樹——就像餐廳收下你的意見卡，但不會當場停下來重排菜單。

9. 系統要記住哪些事？（資料模型）

📒 比喻：一棵活樹 + 兩本帳 系統的資料分兩種：線上「那棵秒回用的字典樹」（放記憶體），和背後離線用的「兩本帳」（用來重蓋樹）。

🌳字典樹節點（放記憶體，負責秒回）

每個節點記三樣：往下的分岔（children）、這裡是不是一個完整的詞（isWord）、這個前綴的 Top-K 榜。

為什麼節點要存 Top-K？ 如果不存，查詢時就得從這個節點往下把整棵子樹翻一遍再排序——最熱門的短前綴（像「a」）底下可能上百萬個詞，會直接卡爆。 把榜事先算好貼在節點上，查詢就退化成「走過去、讀陣列」，跟底下有幾個詞無關。這是本題的靈魂。

📝查詢日誌（一直往後追加的流水帳）

每次「被搜 / 被選中」就append 一筆：詞、時間、地區…。 它只負責「忠實記下發生了什麼」，不負責即時算榜。寫起來超快，因為只是往後面加一行。

📊詞頻表（離線統計出來的熱度）

把上面的流水帳聚合成：詞（主鍵）、頻率（被搜幾次）。 可以做「時間衰減」——讓最近的搜尋加重、老掉的搜尋慢慢退場，這樣榜才會反映「現在的熱門」。重蓋字典樹時就照這張表排榜。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
查詢時當場翻子樹、排序節點預存 Top-K 榜，查詢降到「走過去讀榜」（O(前綴長度)）
每按一鍵一次請求，量暴衝建議服務無狀態、多開幾台分攤 + 前端 debounce（停半拍才送）
上億詞的樹一台記憶體裝不下依前綴首字分片到多台（a–f 一台、g–m 一台…）+ 索引壓縮
「a」「自」等超短前綴流量全擠一處那台多放幾份副本 + 在邊緣/CDN 快取熱門建議
爆紅熱詞太慢進榜離線整批重建（穩）＋ 串流小補丁（快）雙軌並用

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 查詢時排序 vs 事先存好榜 當場排：省記憶體、改完馬上反映，但查詢慢。事先存榜：查詢飛快，但樹變大、更新要重算。 這題是讀重型，選「事先存好榜」——犧牲一點記憶體和更新成本，換每次打字都快。
⚖️ 整批重建 vs 近即時補丁 整批重建（如每小時重蓋整棵樹）：簡單、穩定、一致，但熱詞進榜慢。 近即時補丁：熱詞幾分鐘進榜，但要處理併發更新、比較複雜。多數系統兩者並用：大盤靠重建，熱點靠補丁。
⚖️ Trie 字典樹 vs 直接用 (前綴→清單) 的對照表 用 hash 表「前綴 → 建議清單」也能查，但每個詞要把它所有前綴都存一份（「自動完成」要存「自」「自動」「自動完」…），記憶體爆炸。 Trie 天然共用開頭、又能在節點直接掛榜，所以是首選。
💭 追問：打錯字（拼字錯誤）也想給對的建議怎麼辦？ 用「編輯距離 / 模糊比對 / n-gram」：先把可能打錯的字猜回正確的，再去查前綴。等於「先糾錯、再走字典樹」。
💭 追問：中文沒有空白，前綴怎麼切？ 中文以「字元」為單位一個一個往下走（不像英文用空白分詞）。 本題 demo 因為本機 PHP 沒載入 mbstring，是用 PCRE 的 /u 旗標把字串切成一個個字元來處理。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「字典樹 + 熱門榜 + 頻率改變排名」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8016 -t public
# 然後瀏覽器打開 http://localhost:8016
  1. 在輸入框打前綴（例如「自」），看下面即時跳出依熱度排好的 Top-K 建議。
  2. 點一個建議，等於幫它「被選一次」，它的頻率會 +1。
  3. 再查同一個前綴，看看剛才那個詞是不是往上爬了——這就是「選中事件改變排名」。
玩的時候想一下當你打第 2、第 3 個字，建議是不是「越來越少、越來越精準」？ 那就是你沿著字典樹往下又走了一步，讀到更深節點上那塊更專一的榜。 反覆點同一個建議幾次，看它怎麼一路爬到榜首——這就是頻率（熱度）決定排名。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那三關到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 Trie（那棵按字母分層的字典樹，負責「走過去」）→ Suggester（在每個路口掛熱銷排行榜＝預存 Top-K，並負責頻率更新）→ Store（背後那本「詞→被搜幾次」的帳，重蓋樹時拿來排榜）。打字一來，就照「走過去 → 讀榜」跑一遍。

1建字典樹：照著字一個一個往下接（Trie）

第一關，把每個詞照字元一個一個往下走擺進樹裡。走到沒有的岔路就長一個新節點，走到底就在那個節點蓋章「這裡是一個完整的詞」。

function 插入(詞, 熱度) {
    節點 = 樹根
    for (詞裡的每一個字 之 字) {       // ★ 中文用「字元」為單位切，不是位元組
        if (節點底下沒有 字 這條岔路)
            節點.分岔[字] = 新節點()    // ← 沒路就長一條新的
        節點 = 節點.分岔[字]            // 往下走一步
    }
    節點.是一個詞 = true               // 走到底：這裡是完整的詞
    節點.熱度    = 熱度
}

「貓」「貓咪」「貓砂」都先走過同一個「貓」節點——共用開頭就共用同一段路，這就是 Trie 省空間的關鍵。打前綴時也是用同一招「照字往下走」，走到哪就代表你打了哪個前綴。

2每個路口預先掛好熱銷榜：由下往上合（Suggester）

第二關是全題的靈魂：在每一個節點事先算好「從這往下走，最熱門的前 K 名」，貼在節點上。算法是由下往上合併——先算好子節點的榜，再把它們合起來取前 K 名：

function 算這個節點的榜(節點) {
    候選 = []
    if (節點.是一個詞) 候選.加入({ 節點.詞, 節點.熱度 })
    for (節點的每個子節點 之 子) {
        子榜 = 算這個節點的榜(子)        // ← 先把底下的榜算好（後序：先子後父）
        候選.加入(子榜全部)
    }
    節點.榜 = 候選照熱度由高到低排序後.取前K名   // ★ 合併後只留前 K 名貼在節點上
    return 節點.榜
}

⚠️ 注意是後序（先把子節點的榜都算好，再合出自己的）。每個節點只留 K 名（K 約 5~10），不會把整棵子樹都背在身上。建好後，那塊牌子就一直掛在路口等人來讀。

💭 停一下：為什麼要「由下往上」算，而不是每個節點各自從頭把子樹翻一遍？

看答案 因為「由下往上」每個節點只需合併幾個子節點已經算好的小榜（各 K 名），算一次就好；如果每個節點都自己從頭翻整棵子樹，最熱門的短前綴（像「自」）會被重複翻無數次，慢到爆。這叫「算過的別重算」。

3查詢：照前綴走過去、直接讀榜（Suggester）

前面兩關都是事先準備。使用者真正打字時，做的事極簡單：照前綴走到那個節點，把貼在上面的榜直接讀出來就回。

function 給建議(前綴) {
    節點 = 照前綴一個字一個字走到對應節點   // 走不到 → 沒有詞以此開頭，回空
    if (節點 不存在) return []
    return 節點.榜                          // ★ 直接讀預存的 Top-K，不現算、不排序
}

關鍵就在最後那行：不翻子樹、不排序，直接讀。所以無論底下有 100 個還是 100 萬個詞都一樣快——這就是前面說的 O(前綴長度)：只跟你打幾個字有關，跟總詞數無關。

4頻率更新：被選一次 → 熱度 +1 → 順手重算這條路上的榜（Suggester）

你點了一個建議，等於它「被選一次」。它的熱度 +1，而且要讓下次查同前綴時排名馬上反映——所以只重算這個詞會影響到的那幾個榜：

function 被選中(詞) {
    沿路收集 = 照詞一個字一個字走，把沿途節點記成一條鏈
    if (走不到 或 終點不是一個詞) return null
    終點.熱度 = 終點.熱度 + 1               // ← 熱度 +1
    for (這條鏈 由下往上 之 節點)
        算這個節點的榜(節點)               // ★ 只重算「這個詞的每個前綴」那幾塊榜
    return 終點.熱度
}

為什麼只重算這一條鏈？因為「自動完成」變熱，只會影響「自」「自動」「自動完」…這幾個前綴的榜，其他路口完全沒被碰到，不用動。改完，再查「自」就會看到它往上爬了——這正是 demo 裡「點一下就爬榜」的真相。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 建字典樹（按字母分層）Trie插入() 的「照字往下走」
② 每個前綴預貼熱門榜（Top-K）Suggester算這個節點的榜()（由下往上合）
③ 打字秒回（走過去讀榜）Suggester給建議() 直接讀 節點.榜

👉 重點：真實系統會把「JSON 檔 + 駐記憶體 Trie」換成壓縮 Trie（FST/Radix Tree）放分片伺服器、把「就地重算」換成背後離線管線重建 + 串流補丁、把詞頻表搬到外部儲存， 但上面這幾段邏輯——「照前綴走過去」＋「節點預存 Top-K」——一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「英文字典」解釋，打前綴怎麼瞬間找到位置？ 詞照字典樹（Trie）按字母一層一層分下去擺好。打「自動」就像查字典：先到「自」區、再到「自動」，照著字往下走幾步就到那個節點。走到哪，就代表你打了哪個前綴。
Q2：為什麼不「當場現算」最熱門那幾名？該怎麼做？ 因為每按一鍵就要回一次、量超大又要幾毫秒回完；當場把上百萬個同前綴的詞撈出來排序太慢。改成事先（離線）在每個前綴節點貼好「最熱門前 K 名」的榜，打字時只要「走過去、讀榜」。
Q3：Top-K 是什麼？用「便利商店排行架」說說看。 Top-K = 每個前綴「最熱門的前 K 名（約 5~10）」。像便利商店事先把熱銷排行榜貼在架上，你一看就知道，不用當場翻一整年發票。系統把這榜事先貼在字典樹每個節點上。
Q4：為什麼說查詢速度跟「總共有幾百萬個詞」無關？ 因為查詢只是「照前綴走到對應節點、讀出那節點貼好的榜」。你打 2 個字就走 2 步，跟底下有 100 個還是 100 萬個詞無關——這就是 O(前綴長度)。
Q5（工程）：使用者「選了一個建議」，為什麼我們立刻回 202、不馬上改榜？ 因為改榜是背後離線慢慢做的髒活，不能卡在你打字的關鍵路徑上。所以選中事件只先「記一筆日誌」，由離線管線聚合頻率、重建字典樹、重算各節點 Top-K，再整批換上線。線上那棵樹永遠唯讀、只負責秒回。
Q6（工程）：字典樹一台機器裝不下、或「自」這種短前綴流量爆量，怎麼辦？ 裝不下就依「前綴首字」分片到多台機器（a–f 一台、g–m 一台…），查詢照首字導到對應分片；某些超熱前綴流量集中，就對那台多放幾份副本，並在邊緣/CDN 快取它的建議結果分攤。
Q7（工程）：為什麼用 Trie 而不是直接拿「前綴→清單」的 hash 對照表？ hash 對照表要把每個詞的「所有前綴」各存一份（「自動完成」要存「自」「自動」「自動完」…），記憶體爆炸。Trie 天然共用開頭、又能直接在節點掛 Top-K 榜，省空間又好查。
