1. 60 秒搞懂它在幹嘛

你打開交友 App，螢幕上出現一張張附近的人的卡片。喜歡就往右滑（Like），沒興趣就往左滑（Pass）。 如果你 Like 的那個人，剛好也 Like 了你——「叮！你們配對了 💞」，這時才能開始聊天。

聽起來很簡單對吧？難的是：使用者有好幾千萬，每個人每天滑上百張卡， 系統要做到：① 快速找出「你附近、又符合你喜好」的人；② 每張滑卡都記下來；③ 一旦兩個人互相喜歡，立刻配對；④ 滑過的人別再出現。

🔑 一句話 交友配對 = 幫你從附近的人裡挑出對的人發牌，等你和對方「互相按讚」才牽線。 重點不是存資料，而是「怎麼又快又準地發牌」和「怎麼判斷兩人互相喜歡」。

2. 為什麼這題的難點不是「存使用者」？

很多人第一次看這題會想：「不就是存一堆使用者，然後互相喜歡就配對嗎？」——這想法漏掉了三個真正的難關。

📌 比喻：相親派對的發牌員 想像一個超大相親派對，現場有幾千萬人。你不可能把每個人的資料卡都翻一遍找對象（太慢）。 發牌員得先只從你旁邊幾桌（地理就近）挑人，再濾掉你不會喜歡的（性別/年齡/距離不合）， 最後把剩下的排個順序遞給你——而且你看過的人不再遞第二次。

所以難關有三個：

  • 找附近的人要快：幾千萬人裡，怎麼瞬間挑出「你附近的」？（這就是地理候選，和「找附近餐廳」同一招）
  • 配對要雙向：不是你喜歡就配，要兩個人互相喜歡。怎麼判斷「他之前是不是也喜歡過我」？
  • 滑過的別再出現：你滑掉的人，明天不該又冒出來。怎麼記住「看過誰」？

💭 停一下：為什麼不能「把全部幾千萬人都拿來比一遍」再挑出附近又合適的？

看答案 因為人太多，每次滑卡都掃幾千萬人，計算量會爆炸、而且很慢。 所以要先用「地理就近」快速縮小到附近這一小撮（用 geohash，下面會講），再做細的偏好過濾——這叫「召回 → 過濾」，是推薦系統的標準套路。

3. 把系統拆成 4 個關卡

別被「系統」嚇到。整件事其實就是闖 4 關，一關一關來：

1發牌：找出「你附近又合適」的人

第一關是產生候選——把該遞給你的卡找出來。分兩步：

🗺️ 比喻：先看你旁邊幾桌 發牌員不會翻全場名冊，而是先圈出你座位周圍幾桌的人。系統用一個小技巧把地圖切成一格格（叫 geohash）， 你在哪一格，就先撈你那格 + 周圍 8 格的人（共 9 格）。這跟「找附近的餐廳」是一模一樣的招式。

撈出附近的人之後，再濾掉不合的：性別不對、年齡不在你設定範圍、距離太遠的，通通拿掉。而且是雙向濾——你要喜歡他、他也要可能喜歡你（不然你滑了也是白滑）。

2滑卡：喜歡往右、不喜歡往左

卡發到你手上，你就滑：右滑 = Like、左滑 = Pass。每一滑都要記下來（誰對誰、Like 還是 Pass）。

📝 比喻：在名冊上打勾打叉 你看一個人，喜歡就在他名字旁打勾（Like），不喜歡就打叉（Pass）。 這本「誰對誰打了勾或叉」的紀錄，是後面判斷配對的關鍵——而且量超大（每人每天上百筆）。

3配對：兩個人互相打勾才牽線

重點來了：不是你喜歡就配。系統要看「對方之前是不是也對你打過勾」。

💞 比喻：兩張勾才成對 你對 Bob 打了勾。系統翻一下名冊：「Bob 之前有對你打勾嗎？」 —— 沒有：先記著，還不配。
—— 有！那就是兩張勾湊成一對 → 配對成立 💞，現在你們可以聊天了。

所以判斷配對，就是查「你對他」和「他對你」這兩筆紀錄是不是都打了勾。簡單又可靠。

4去重：滑過的別再發給你

你滑掉的人，明天不該又出現在牌堆裡。系統要記住「你看過誰」，發牌時把這些人剔掉。

🗑️ 比喻：看過的丟進「已看箱」 你看過的每張卡都丟進一個「已看箱」。下次發牌前，發牌員先對一下已看箱，看過的就不再遞。 這跟網路爬蟲「爬過的網址不再爬」是同一招——用一個集合擋掉重複。
✅ 四關一句話 ① 用 geohash 找附近的人＋雙向過濾偏好（發牌）→ ② 滑卡記下 Like/Pass → ③ 兩人互相 Like 才配對 → ④ 看過的去重不再出現。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「發牌太慢 / 算不動」 幾千萬人，如果每次發牌都從頭挑，使用者滑一張要等半天。 所以要先用 geohash 快速圈出附近的人（只看你附近 9 格），再做偏好過濾——只算必要的那一小撮。先縮小範圍，再精挑。
😱 怕「配錯 / 漏配」 明明兩人互相喜歡卻沒牽線（漏配）→ 使用者失望；或單方喜歡就硬湊（配錯）→ 違反「互相喜歡才配」的規則。 所以配對一定要查兩筆紀錄（你對他 + 他對你）都是 Like 才成立，一筆都不能少。雙向勾才算數。
😱 怕「重複 / 騷擾」 滑過的人又冒出來（重複）→ 體驗很差；沒配對的人能私訊你（騷擾）→ 隱私災難。 所以要用「已看箱」去重，並嚴守「沒配對不能聊天」的隱私底線。看過不重發、沒配不能聊。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。系統設計也一樣：先估數字，才知道要準備多大的「料」。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
有多少活躍使用者每天 ~1000 萬人在滑不可能每次都掃全部人 → 要先用 geohash 找附近
每人每天滑幾張~100 張1000萬 × 100 = 每天 10 億筆滑卡，寫量超大
換算每秒幾筆滑卡~11,500 筆/秒（尖峰更高）滑卡記錄要能扛高頻寫 → 分片 + 緩衝
多少 Like 真的配成大約只有 ~1%配對事件其實很稀疏，真正的壓力在「滑卡」與「發牌」
💡 關鍵洞察：壓力在「發牌」和「滑卡」，不在「配對」 配對成立其實很少（百中取一）。真正要扛的是每秒上萬筆滑卡的寫入，和每次發牌都要又快又準地找附近+過濾。 所以設計重心是「geohash 快速召回 + 候選快取」和「滑卡高吞吐寫入 + 互查配對」。

💭 停一下：每天才 1% 的 Like 配成，配對量不大，為什麼系統還是很忙？

看答案 因為「滑卡」這個動作本身量超大（每天 10 億筆）。每一滑都要記錄，而且每滑完一疊就要再發一疊新牌（又一次找附近+過濾+去重）。 配對雖少，但它前面的「發牌 + 滑卡」是高頻動作——忙的是這兩件事。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼、我回你什麼。講好格式，雙方就能合作。

這題只要記住幾張「點餐單」：

① 取候選堆疊（發牌，最常用）
   GET /api/v1/candidates?user=我&limit=20
   我回：[ { 對象, 距離, 他是否已喜歡我 }, ... ]（已濾掉看過的、不合偏好的）

注意回的是距離不是精確座標（隱私）。「他是否已喜歡我」會被拿來排序——已喜歡你的排前面，因為你一滑就配上。

② 滑卡（超高頻）
   POST /api/v1/swipe   給我：{ 誰, 滑誰, like 還是 pass }
   我回：{ 記下了, 是否配對成功, 配對對象 }

如果是 like，系統當場查「對方是否也 like 過我」→ 是就回「配對成功！」。

③ 看我的配對
   GET /api/v1/matches?user=我  →  [ { 配對對象 }, ... ]
④ 改偏好 / 位置（低頻）
   PATCH /api/v1/users/{我}  給我：{ 新位置, 想看性別, 年齡範圍, 最大距離 }
   → 系統要重算我的 geohash 格 + 把舊的候選快取作廢

9. 系統要記住哪些事？（資料模型）

📒 比喻：四本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題用四本：

👤使用者簿（每個人是誰、在哪、喜歡誰）

每人一列：id、性別、年齡、位置(lat/lng)、geohash 格、想看的性別、年齡範圍、最大距離。 多存一欄 geohash，是為了發牌時一查就知道你在哪一格，能快速找附近。

📝滑卡簿（誰對誰打了勾或叉——量最大！）

每滑一筆：(誰 → 滑誰, like/pass, 時間)，用 (誰, 滑誰) 當鑰匙——同一張只記一次（冪等）。 這本是高頻寫，量超大，工程上會獨立出去、分片存放，不擠在主資料庫。

💞配對簿（哪兩個人互相喜歡）

每對一列：用排好序的「配對鑰匙」（把兩個 id 由小到大串起來，如 alice|bob），這樣「A 對 B」和「B 對 A」就是同一把鑰匙，不會存成兩筆。

🗑️已看箱（你看過誰——去重用）

記「你滑過的所有人」。發牌前先對一下，看過的剔掉。其實它可以直接從滑卡簿推導出來（凡是「你滑的」對象都算看過）。規模大時用 bloom filter 省記憶體。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
熱門的人被滑爆（大家都滑同幾個帥哥美女）給他們曝光配額、降權，把曝光分一些給長尾與新用戶（公平）
滑卡寫入量爆（每秒上萬筆）滑卡簿按人分片 + 用佇列緩衝；Pass 可批次慢慢存，Like 才即時處理
滑完附近就沒人了 / 新用戶沒人看到放大 geohash 範圍（格子變大）+ 混入新用戶 + 候選快取設短效期
「已看箱」越來越大用 bloom filter 壓記憶體，或只留最近幾天
每次發牌都重算很慢背景先算好一疊候選快取起來，滑到底再即時補

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ geohash 格子要切多細？ 切太細（格子小）→ 候選精準但人少，市區好、郊區一下就滑完；切太粗（格子大）→ 候選多但要算更多距離。 折衷：市區用細格、郊區自動放大格子湊夠候選。
⚖️ 候選即時算 vs 先算好快取 即時算最新鮮（剛上線的新用戶馬上看得到）但慢；先算好快取最快但可能過時。 折衷：背景先算 + 短效期，滑到底再即時補一批。
⚖️ 公平曝光 vs 配對率 只把「最容易配上的」往前推 → 熱門的人被滑爆、長尾永遠沒曝光。 要刻意留一些版面給新用戶和長尾，犧牲一點短期配對率，換整個生態健康。
💭 追問：怎麼判斷兩個人配對成功？為什麼用「兩筆滑卡」？ A 對 B 打勾時，去查滑卡簿裡有沒有「B 對 A 也打過勾」。有就配成。用兩筆滑卡互查，比另外維護一張「誰喜歡誰」的雙向關係表更簡單，而且滑卡本來就要記，順手就能查。
💭 追問：使用者改了位置或偏好，系統要做什麼？ 兩件事：① 重算他的 geohash 格（位置變了，他屬於哪格也變了）；② 把他舊的候選快取作廢，下次重新按新位置/新偏好發牌。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「發牌 + 滑卡 + 配對」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8058 -t public
# 然後瀏覽器打開 http://localhost:8058
  1. 先按「塞入示範使用者」（台北市區 8 位，性別/年齡/偏好都不同）。
  2. 選一位使用者（例如 Alice），看她的候選堆疊——只會出現附近、符合她偏好、她沒滑過的人。
  3. 讓 Alice 對 Bob 按 Like：還不會配（Bob 還沒喜歡她）。再讓 Bob 對 Alice 按 Like：💞 配對成立！
  4. 再看 Alice 的候選：Bob 不見了（已經滑過，去重）。
玩的時候想一下注意候選裡有人標著「❤ 已喜歡你」——那是對方先 Like 了你，系統把他排到最前面，因為你一 Like 就立刻配上。這就是排序提升配對率。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有幾個小檔，剛好對應前面講的關卡。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 GeoHash（把地圖切格、找附近 9 格的人）→ MatchEngine（發牌 + 判斷配對的大腦）→ Store（記滑卡/配對/已看的本子）。發牌或滑卡時，就照這個順序跑。

1找附近：撈你那格 + 周圍 8 格（GeoHash）

把你的座標編成一格（geohash），再算出周圍 8 格，一起當作「附近」。為什麼要連周圍 8 格？因為你可能就站在格子邊緣，真正最近的人在隔壁格。

找附近的人(我):
    我的格 = 把(我的緯度,經度)編成 geohash
    9 格 = 我的格 + 周圍 8 格    // ← 連鄰格一起撈，才不會漏掉貼邊的人
    回傳 這 9 格裡的所有人

這跟「找附近餐廳」用的是一模一樣的招式，只是格子裡裝的是「人」。

2發牌：召回 → 雙向過濾 → 去重 → 排序（MatchEngine）

這是發牌的大腦，把上面撈到的附近的人，加工成一疊「該滑的卡」：

產生候選(我):
    候選 = 找附近的人(我)               // ① 召回：先用 geohash 撈附近
    對每個 對方 in 候選:
        if 對方 in 我的已看箱: 跳過      // ④ 去重：滑過的不再出現
        if 不符合「我要看的偏好」: 跳過    // ② 我要喜歡他（性別/年齡/距離）
        if 不符合「對方的偏好」: 跳過      //    他也要可能喜歡我（雙向！）
        對方.已喜歡我 = 查(對方是否曾 like 我)
        留下 對方
    把候選排序：已喜歡我的排最前，再距離近的優先   // 排序提升配對率
    回傳 候選

看到沒？「召回 → 過濾 → 排序」就是推薦系統的標準三步。雙向過濾（兩個 if）很關鍵：少了「對方也要喜歡我」這條，你會滑到一堆根本不會喜歡你的人，白費力氣。

3配對偵測：兩張勾才成對（MatchEngine + Store）

這是全題的心臟。你滑一張 Like 時：

滑卡(我 → 對方, like):
    if 我已經滑過對方: return         // 冪等：同一張只記一次
    記錄 (我, 對方, like)              // 寫進滑卡簿
    if 對方之前也 like 過我:           // ← 查「他對我」那一筆
        配對 = 建立配對(我, 對方)       //    兩張勾湊成一對！
        通知雙方「配對成功 💞」

關鍵就是那句「對方之前也 like 過我嗎？」——它去滑卡簿查「對方 → 我, like」這筆在不在。在，就配成。「我對他」「他對我」兩筆都 like，雙向才算數。這也是面試官最愛追問的一段。

💭 停一下：建立配對時，為什麼要把兩個 id「由小到大」串成鑰匙（如 alice|bob）？

看答案 因為配對沒有方向（A 和 B 配對 = B 和 A 配對）。如果不排序，「alice|bob」和「bob|alice」會被當成兩筆，重複了。先由小到大排再串，同一對永遠是同一把鑰匙，只會存一筆。

4已看箱：從滑卡簿推導去重（Store）

「你看過誰」其實不用另外記——凡是你滑過的對象，就是你看過的：

已看箱(我):
    集合 = { 我自己 }                  // 不會發自己給自己
    對每筆滑卡 in 滑卡簿:
        if 這筆是「我滑的」: 把對象加進集合
    回傳 集合                          // 發牌時拿它剔除候選

真實系統人多時，會把這個集合換成 Redis set 或 bloom filter（超省記憶體），但邏輯一模一樣：用一個集合擋掉「已經看過」的——和爬蟲「不重爬」同一招。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔關鍵那段
① 找附近的人GeoHashneighbors()（你那格 + 8 鄰格）
① 發牌：召回→過濾→排序MatchEnginecandidates()
③ 配對：兩張勾才成對MatchEngine/Storeswipe() 裡的「對方也 like 過我？」
④ 去重StoreseenSet()

👉 重點：真實系統會把「JSON 檔 + 迴圈過濾」換成「分片資料庫 + Redis GEO + 向量召回 + bloom 去重」， 但上面這幾段判斷邏輯一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「相親派對發牌員」解釋，這題的核心難關是什麼？ 幾千萬人裡，要又快又準地挑出「你附近、又符合你喜好、你還沒看過」的人發給你。不能掃全部人，要先用 geohash 撈附近，再雙向過濾、去重、排序——就是「召回 → 過濾 → 排序」。
Q2：為什麼配對要「雙向」？怎麼判斷？ 因為交友要兩情相悅，不能你喜歡就硬湊。判斷方法：你 Like 某人時，查滑卡簿裡「對方是否也 Like 過你」，兩筆都是 Like 才配成。
Q3：geohash 是做什麼的？為什麼要連「周圍 8 格」一起撈？ geohash 把地圖切成一格格，你在哪格就先撈那格的人（找附近）。連周圍 8 格一起撈，是因為你可能站在格子邊緣，真正最近的人落在隔壁格——只撈中心格會漏掉。
Q4：每天才 1% 的 Like 配成，配對量不大，為什麼系統還很忙？ 因為「滑卡」這個動作量超大（每天 10 億筆），而且每滑完一疊就要再發一疊新牌（又一次找附近+過濾+去重）。忙的是高頻的「發牌 + 滑卡」，不是稀疏的配對。
Q5（工程）：「已看箱」為什麼可以從滑卡簿推導？大規模怎麼存？ 因為凡是你滑過的對象就是你看過的，所以「你滑的所有 to」就是已看集合。大規模時用 Redis set 或 bloom filter 省記憶體（bloom 容許極小誤判換空間）。
Q6（工程）：配對表為什麼用「排序後的 pair key」？ 因為配對沒方向，A-B 和 B-A 是同一對。把兩個 id 由小到大串成一把鑰匙（如 alice|bob），同一對永遠是同一鑰匙，避免存成兩筆。
Q7（工程）：「雙向偏好過濾」少了「對方也要喜歡我」這條會怎樣？ 你會滑到一堆根本不會喜歡你的人（你在他們眼裡不符偏好），白費滑卡、永遠配不成，也浪費系統算力。雙向過濾把不可能配成的提前剔除。

15. 30 秒回顧

交友配對 = 從附近的人裡挑出對的人發牌，互相按讚才牽線。

4 個關卡：① 用 geohash 找附近＋雙向過濾偏好（發牌）→ ② 滑卡記 Like/Pass → ③ 兩人互相 Like 才配對（查兩筆滑卡）→ ④ 看過的去重不再出現。

3 個怕：怕發牌太慢（geohash 先縮小範圍）、怕配錯/漏配（雙向勾才算）、怕重複/騷擾（已看箱去重 + 沒配不能聊）。

工程細節一句話：壓力在「發牌」和「每秒上萬筆滑卡」，配對本身很稀疏 → 用「點餐單(API)」溝通、四本筆記本（使用者/滑卡/配對/已看箱）→ 召回(geohash)→雙向過濾→去重(bloom)→排序提升配對率 → 各環節分片/快取/曝光配額拓寬。
