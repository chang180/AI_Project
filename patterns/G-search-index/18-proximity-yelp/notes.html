⑱ 附近的人 / 地點：地理空間搜尋（Proximity / Yelp Nearby）

G 搜尋/索引 建議 42 分鐘 Design a Proximity Service

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把二維地理座標變成可索引的一維鍵，做半徑 / 最近 K 查詢」。 難點不是查詢本身，而是如何索引連續的經緯度——B-Tree 對單一欄位有效，但「同時靠近某經度 且 某緯度」是二維問題， 這就是 Geohash / QuadTree / S2 / PostGIS 存在的原因。讀多寫少（商家座標幾乎不變）讓快取與讀副本成為主力。

1. 需求釐清

功能性需求

  • 給定座標 (lat, lng) 與半徑，回傳半徑內的地點，依距離由近到遠排序。
  • 「最近 K 個」查詢（即使半徑外也要找到最近的幾個）。
  • 結合屬性過濾：類別（餐廳/景點…）、最低評分、營業中等。
  • 分頁（前端地圖捲動 / 下拉載入更多）。
  • 地點 CRUD（新增/更新商家座標、屬性）——但這是低頻寫。

非功能性需求

  • 讀多寫少：商家地點是近乎靜態資料（座標開店後幾乎不變），查詢量遠大於更新量。
  • 低延遲：附近搜尋 < 100ms（p99），使用者在地圖上即時互動。
  • 高可用：搜尋是核心體驗，需可水平擴展。
  • 結果正確：不能因索引邊界而漏掉明明很近的地點（見 §6 邊界問題）。
面試開場必問 「地點是靜態（商家，本題）還是高頻移動（即時找附近的人 / 叫車）？」 靜態 → 重點在搜尋索引、快取、讀副本（本筆記主軸）；移動 → 重點在高頻位置更新與 TTL（那是 Robotaxi / 即時媒合題的走向）。 再問：要精確半徑還是近似即可？地圖類通常近似（先填滿畫面）即可。

2. 規模估算

指標假設推算
地點總數全球商家~2 億筆
每筆中繼資料名稱/類別/評分/座標 ~1KB2億 × 1KB ≈ 200 GB（單機記憶體可放索引）
搜尋（讀）5000 萬 DAU，每人 10 次附近搜尋5 億/日 ≈ ~5.8k QPS（峰值 ×5 ≈ 30k）
地點更新（寫）商家偶爾改資料< 100 寫 QPS（讀寫比 > 100:1）
geohash 精度 6格子 ≈ 1.2km × 0.6km市區一格內數十～數百家，適合粗篩
關鍵洞察資料量小（索引可全進記憶體）、讀寫極不對稱 → 設計圍繞「記憶體內空間索引 + 快取熱門查詢 + 讀副本擴展讀」，寫路徑相對輕鬆。

3. API 設計

# 半徑查詢（讀，超高頻）
GET /api/v1/places/nearby?lat=25.0339&lng=121.5645&radius=2km
        &category=restaurant&min_rating=4.0&limit=20&offset=0
→ 200 { "results": [ { "id":"p1", "name":"...", "distance_km":0.11, ... } ],
        "total_matched": 37, "next_offset": 20 }

# 最近 K 個（不限半徑）
GET /api/v1/places/nearest?lat=..&lng=..&k=5&category=cafe

# 新增 / 更新地點（寫，低頻）
POST  /api/v1/places   Body: { "name":"...", "category":"...", "rating":4.5, "lat":.., "lng":.. }
PATCH /api/v1/places/{id}   Body: { "rating": 4.7 }     # 更新需重算 geohash 桶 + 失效快取
設計細節半徑用 radius 帶單位；分頁用 offset/limit（或地圖類常用 bounding-box ?sw=&ne= 直接給可視範圍四角）。回傳 distance_km 讓前端標距離。

4. 資料模型

places  (主庫：讀多寫少，可大量讀副本 + Redis 快取)
  id          BIGINT PRIMARY KEY
  name        TEXT
  category    VARCHAR        -- 屬性過濾
  rating      DECIMAL(2,1)   -- 屬性過濾
  lat, lng    DOUBLE
  geohash     CHAR(8)        -- 預先算好的索引鍵（精度可多存幾層）
  INDEX (geohash)            -- 前綴查詢靠它；或交給 PostGIS GiST / S2

# 空間索引（記憶體 / Redis）
geohash 桶 → [place_id, ...]      # 同前綴 = 地理鄰近
# Redis GEO：ZADD places lng lat member（底層即 geohash 排序集）
為何預存 geohash把座標在寫入時就編成 geohash 字串並建索引，查詢就退化成「字串前綴比對」——可直接用一般 B-Tree / 排序集，不需要特殊空間資料庫。屬性（評分/類別）留在同一列，過濾時順手做掉。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
 寫(低頻) ─▶ 地點寫入服務 ─▶ 算 geohash ─▶ 地點DB(主) ─┐
                                  │                      ├─▶ 索引更新(桶)
                                  └──────────────────────┘
 讀(高頻) ─▶ 查詢服務 ─(查快取命中?)─▶ 快取(熱門查詢) ─┐
               │ (miss)                                  │
               ▼                                         ▼
       空間索引(geohash 分桶: 中心格+8鄰格) ─▶ 地點DB讀副本 ─▶ haversine 精算+過濾+排序
  1. 查詢服務：無狀態、可水平擴展；先打快取，未命中走空間索引粗篩 + 精算。
  2. 空間索引：以 geohash 前綴分桶；查詢取中心格 + 周圍 8 鄰格共 9 桶（解決邊界）。
  3. 地點 DB：讀多寫少 → 多個讀副本擴展讀；寫入時算 geohash 並更新索引。
  4. 快取：熱門查詢（熱門商圈座標 + 常見半徑/類別）結果快取，命中率高（查詢集中在熱點區）。

6. 核心元件深入

空間索引方案對比

方案原理優點缺點 / 適用
Geohash 前綴交錯二分經緯度編成 base32 字串，共同前綴越長越近實作簡單、可用普通字串索引 / Redis GEO；前綴查詢快邊界問題（需查 8 鄰格）；格子大小固定不隨密度調整
QuadTree遞迴把區域切四象限，密集處切更細密度自適應（市區切細、海上不切）；查詢效率高需建樹、再平衡較麻煩；多在記憶體
Google S2把球面投到立方體再用 Hilbert 曲線編 cell id球面失真小、cell 大小均勻、鄰格運算精確概念複雜、需函式庫；Google Maps / Foursquare 採用
PostGIS關聯式 DB 的地理擴充，GiST 索引 + ST_DWithin直接 SQL、精確、功能完整（多邊形/路網）單庫擴展受限；超大規模需分片

Geohash 邊界問題（最常被追問）

geohash 把連續座標離散成格子，共同前綴越長 = 越接近。但格子有硬邊界：兩個物理上只差幾公尺的點，可能剛好落在相鄰格子（前綴不同）。 若只查中心格，會漏掉貼著邊界的鄰近地點。

          ┌──────┬──────┬──────┐
          │ NW   │  N   │  NE  │   查詢點若靠近中心格右緣，
          ├──────┼──────┼──────┤   真正最近的店可能在 E 格！
          │  W   │ 中心 │  E   │   → 一律取「中心格 + 8 鄰格」共 9 桶
          ├──────┼──────┼──────┤      做粗篩，再用 haversine 精算過濾。
          │ SW   │  S   │  SE  │
          └──────┴──────┴──────┘
面試金句「我會查中心格加周圍 8 個鄰格，避免邊界遺漏；這 9 桶是粗篩（candidate set），縮小範圍後再用 haversine 球面距離精算過濾掉半徑外的、並依距離排序。」

geohash 精度 vs 格子大小

精度（字元數）格子寬 × 高（約）適用
4~39km × 19km城市級
5~4.9km × 4.9km行政區 / 大商圈
6~1.2km × 0.6km市區附近搜尋（本 demo 採用）
7~153m × 153m街區級 / 步行範圍
8~38m × 19m建物級

精度要配合查詢半徑：半徑遠大於格子，要查的鄰格就不只 8 個（得擴大環形掃描）；半徑遠小於格子，9 桶內候選太多浪費精算。讓格子大小 ≈ 常見查詢半徑最省。

最近 K 個：候選不足時放大範圍

若 9 桶內地點數 < K（例如郊區很稀疏），就縮短 geohash 前綴（放大格子）重查，逐步擴大搜尋環，直到湊滿 K 個。實作即「先精度 6，不夠降到 5、4…」。

讀多寫少 → 快取與讀副本

  • 商家座標近乎靜態 → 查詢結果可大膽快取（熱門商圈 + 常見半徑/類別組合，命中率高）。
  • 讀流量靠讀副本水平擴展；空間索引可整份放記憶體（資料量僅數百 GB）。
  • 寫入（改評分/座標）時：重算 geohash 桶歸屬 + 失效相關快取（讀自己的寫）。

7. 擴展與瓶頸

階段瓶頸對策
單機CPU / 記憶體無狀態查詢服務 + 負載均衡；索引整份進記憶體
讀爆熱門商圈查詢打 DB快取熱門查詢結果 + 多讀副本
熱點分布不均市區桶超大、海上桶空改用 QuadTree / S2 等密度自適應索引；或市區用更高精度
全球延遲跨洲 RTT依地理分區部署，就近服務 + 就近快取
索引更新批量匯入新地點低頻寫，可離線重建索引 + 灰度切換

8. 取捨與面試重點

  • Geohash vs QuadTree/S2：geohash 簡單（普通索引就能跑）但格子固定、有邊界問題；QuadTree/S2 密度自適應、邊界處理好，但複雜。面試先講 geohash（好寫好答），再加一句「資料密度極不均時改 QuadTree/S2」。
  • 精確 vs 近似：地圖類通常先填滿畫面即可（近似），不必嚴格半徑；嚴格半徑才要 haversine 精算每一筆。
  • 讀路徑快取一致性：商家更新少 → 快取 TTL 可長；更新時主動失效。評分/排序若摻即時人氣，快取需更短 TTL。
  • 距離公式：haversine（球面）夠用；要更準用 Vincenty（橢球），但成本高、附近搜尋無必要。
  • 常見追問：① 半徑遠大於格子怎辦？→ 擴大環形掃描或降精度。② 如何排序？→ 距離為主，可混合評分/人氣加權。③ 移動的「附近的人」？→ 改高頻位置更新 + 短 TTL（轉成即時題）。④ 為何不用兩個 B-Tree（lat 一個、lng 一個）？→ 二維交集效率差，無法直接「同時靠近」。

9. PHP Demo 說明

本題 demo 著重於空間索引的核心邏輯（也是面試真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/GeoHash.php　經緯度 ↔ base32 geohash 編解碼，並提供 neighbors() 算出中心格 + 8 鄰格（解決邊界）。
  • src/SpatialIndex.php　以 geohash 前綴分桶；半徑查詢取 9 桶粗篩 → haversine 精算 → 屬性過濾（類別/評分）→ 排序 → 分頁；含最近 K（候選不足自動降精度放大範圍）。
  • src/PlaceStore.php　地點資料（JSON 檔模擬讀多寫少的 DB）+ 行內快取與命中統計。
  • public/index.php　路由：建立/塞資料、半徑查詢、最近 K、桶檢視，並附深色測試頁。
# 啟動
cd demo
php -S localhost:8018 -t public
# 瀏覽器開 http://localhost:8018 → 先「塞入示範地點」→ 做半徑 / 最近 K 查詢
# 回應的 buckets_scanned 即實際掃描的 9 個 geohash 桶（含 8 鄰格）
誠實聲明 Demo 的「DB」是 JSON 檔、「快取」是 process-local 陣列、索引每次請求重建（示意簡化）； 但 geohash 編解碼、鄰格計算、前綴分桶、半徑/最近K查詢、haversine 精算、類別/評分過濾、分頁等空間索引邏輯為真實可執行。 本機 PHP 未載入 mbstring，程式全程只用 strlen/substr（geohash 皆 ASCII）。 要換成生產級索引（PostGIS / Redis GEO / S2），只需替換 SpatialIndex 內部實作，介面與呼叫端不變。
