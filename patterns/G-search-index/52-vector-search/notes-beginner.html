1. 60 秒搞懂它在幹嘛

你在 Google 相簿打「海邊的狗」，它就把一堆海邊的狗照片找出來——就算照片檔名根本沒寫「狗」也找得到。 或者你用 ChatGPT 問公司內部文件，它會先找出最相關的幾段再回答你。這背後都是向量搜尋。

關鍵點：它找的不是「字一模一樣」，而是「意思相近」。怎麼做到？先把每樣東西（一句話、一張圖）變成一串數字——叫向量——意思越像的東西，數字串的「方向」就越接近。搜尋就變成「找方向最接近的那幾串數字」。

🔑 一句話 向量搜尋 = 把萬物變成「座標上的點」，意思相近的點就靠在一起；查詢時把問題也變成一個點，找離它最近的幾個點就是答案。

2. 為什麼這題的難點不是「算誰最像」？

很多人第一次看會想：「不就是把每個點跟查詢點算距離，挑最近的嗎？」——這想法對，但會把系統壓垮。

📌 比喻：在一座億人城市找「長得最像你的人」 要找全城長相最接近你的 10 個人。最笨的辦法：把全城一億人一個一個拉來跟你比對。一定找得到最像的，但比一億次，慢到天荒地老。

向量搜尋的核心難關就是這個：點有幾億個、每個點又是幾百維（幾百個數字）。逐一比對全部（叫暴力搜尋）保證正確，但慢到不能用。

所以真正的招數是：事先把相近的點分好「區」。查詢時先看「你大概屬於哪一區」，只在那幾區裡比，跳過其他區。這叫 ANN（近似最近鄰）——用「可能會漏掉一兩個」換「快上幾百倍」。

💭 停一下：為什麼「跟全部一億人逐一比對」在這題行不通？

看答案 因為點有幾億個、每個還是幾百維，每次查詢都把全部比一遍，計算量爆炸、慢到不能線上用。 我們需要「先分區、只比附近那幾區」的聰明辦法（ANN），雖然偶爾會漏掉，但快非常多——這就是後面要講的「分群」。

3. 把系統拆成 4 個關卡

別被「系統」嚇到。整件事其實就是闖 4 關，一關一關來：

1變成向量：把東西翻成「座標上的點」

第一關，要把每樣東西（句子、圖片）變成一串數字（向量）。負責這件事的叫 嵌入模型——你可以想成一台「翻譯機」，把意思翻成座標。

🗺️ 比喻：把詞放上地圖 想像一張地圖：「貓」「狗」放得很近（都是寵物），「汽車」放在很遠的另一區。 嵌入模型做的就是這件事——意思相近的，座標就靠近。重點：建資料庫和查詢，要用同一台翻譯機，不然座標對不上。

2分群：先把相近的點分好「區」

幾億個點全擠在一起，查詢時一個個比太慢。所以事先把相近的點圈成一區一區，每區選一個「代表點」（中心）。

🏙️ 比喻：把城市分成行政區 與其在全城一億人裡亂找，不如先把城市分成幾百個行政區，每區記一個「市中心位置」。 以後找人，先看「這個人大概在哪一區」，直接去那區找就好。這個「分區」的動作，工程上叫分群（建索引）。

3查詢：只比「最近的那幾區」

查詢點來了，先跟各區的「市中心」比一下（區只有幾百個，很快），挑出最近的 1~幾個區，只在這幾區裡的點逐一精算，挑出最近的 k 個。

🎯 比喻：先鎖定行政區，再進去找 找「長得最像你的人」：先看你屬於哪個區（比幾百個市中心）→ 進那一兩個區，把區裡的人比一比 → 挑最像的 10 個。 其他幾百個區，看都不用看。一億次比對 → 變成幾千次，這就是 ANN 的威力。

代價：萬一「最像你的人」剛好住在隔壁區、而你沒進去找，就漏掉了。漏掉的比例，就是下一關要管的「召回率」。

4取捨：要找多準？（召回率 vs 速度）

你可以決定「多看幾個區」。看的區越多 → 越不會漏掉最像的（準）、但越慢；看的區越少 → 越快、但越可能漏。

🔍 比喻：搜山找人 失蹤者可能在某座山。只搜「最可能的那座」最快，但他若在隔壁山就漏了；多搜幾座山更不會漏，但更花時間。 要搜幾座（工程上叫 nprobe）就是你手上的「準 vs 快」旋鈕。

用「暴力搜尋（全部比一遍）」當標準答案，看 ANN 找到的跟標準答案重疊多少，這個比例就叫召回率。召回率 1.0＝一個都沒漏。

✅ 四關一句話 ① 用翻譯機變成向量 → ② 事先分群（分區） → ③ 查詢時只比最近的幾區 → ④ 用「看幾區」旋鈕在「準（召回率）vs 快」之間調。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「慢 / 算不動」 幾億個點 × 幾百維，每次查詢都全部比一遍，機器直接跑不動。 所以才要分群、只比最近幾區（ANN），把「比一億次」變成「比幾千次」。只比必要的，不做白工。
😱 怕「漏掉最像的」（召回率太低） 為了快只看一個區，結果「最像的」住隔壁區，沒找到 → 使用者覺得「這搜尋好爛」。 所以要能調「看幾區」，在可接受的速度下把召回率拉到夠高（如 95%+）。快，但不能漏太多。
😱 怕「裝不下記憶體」 幾億個幾百維向量可能要好幾 TB，全塞進記憶體太貴。 所以要壓縮（量化）——把每個向量用更少的位元近似存，省記憶體；還可切成好幾台機器分開放（分片）。裝得下、又夠快。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
有多少點（向量）~10 億個不可能每次都全部比一遍 → 一定要 ANN（分群只比附近）
每個點多少維~768 個數字每個點約 3 KB
全部向量多大10 億 × 3KB ≈ 3 TB塞不進一台機器的記憶體 → 要壓縮 + 切多台
每秒多少查詢數千～數萬次每次都要在 50 毫秒內回 → 非 ANN 不可
💡 關鍵洞察：兩件事決定一切 ① 點太多 → 用分群只比附近（ANN）把比對量壓下來； ② 資料太大 → 用壓縮 + 分片把 3 TB 裝進負擔得起的機器。其他都是繞著這兩件事轉。

💭 停一下：每秒「才」幾千次查詢聽起來不多，為什麼還是非用 ANN 不可？

看答案 因為每一次查詢如果用暴力，就要跟 10 億個點、每個 768 維比一遍——光一次就要算上千億次乘法。 幾千個查詢同時來，機器瞬間燒掉。ANN 把「一次比 10 億」變成「一次比幾千」，才撐得住。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」：

① 新增一個點（建資料庫時最常用）
   POST /api/v1/vectors
   給我：{ 向量[一串數字], 標籤(這是什麼、哪來的) }
   我回：201「存好了，編號 100123」

向量通常是嵌入模型先翻好再丟進來；標籤（metadata）之後可以拿來過濾（例如「只在中文文件裡找」）。

② 搜尋最像的 k 個（查詢，最高頻）
   POST /api/v1/search
   給我：{ 查詢向量 q, 要幾個 k, 看幾區 nprobe, (可選)過濾條件 }
   我回：[{ 編號, 相似度分數, 標籤 }, ...]   ← 由最像排到次像

注意查詢也用 POST：因為查詢向量是一長串數字，塞不進網址列。nprobe 就是那個「準 vs 快」旋鈕，交給呼叫端調。

③ 刪除 / 更新一個點
   DELETE /api/v1/vectors/{編號}

刪改之後，那個點在「分區索引」裡的歸屬也要跟著整理（通常先做記號、之後背景重建），這是後面「擴展」會談的麻煩事。

9. 系統要記住哪些事？（資料模型）

📒 比喻：兩本筆記本 系統要把資料記在「筆記本」（資料表）裡。這題主要用兩本：一本記原始的點，一本記分好的區。

📍點簿（每個點的座標 + 標籤）

每個點一列：編號、向量[一串數字]、標籤(這是什麼、哪來的、什麼時候)。 向量本身要留著，因為最後排名次時要用原始座標精算；標籤拿來過濾和顯示。

🗂️分區簿（索引——核心！）

記兩件事：① 每個區的市中心位置（centroid）；② 每個區有哪些點（成員名單）。 這就是讓查詢能「只進最近那幾區」的關鍵——先用市中心判斷該進哪區，再翻那區的成員名單。

另一種分區法：HNSW（連連看的圖） 除了「分行政區」，還有一種把點連成一張網的做法：相近的點互相牽線，查詢時從一個入口沿著線一路往更近的鄰居走，很快走到目標附近。它通常更準，但「記住所有連線」更吃記憶體。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
跟全部點逐一比，太慢分群（IVF）或連圖（HNSW），查詢只比最近幾區/沿圖跳近
3 TB 向量塞不進記憶體量化壓縮（PQ）把每個點用更少位元近似存；不夠就切多台（分片）
切多台之後，一個查詢要問每台每台各找自己的 top-k，再合併成總名次（scatter-gather）
維度太高，「分區」效果變差先降維（把幾百維壓成幾十維）+ 查詢時多看幾區補回召回
新增/刪除點，分區索引不好改新點先丟暫存區線性比，定期合併重建進主索引；刪除先打記號

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 看幾區（nprobe）：準 vs 快 看的區多 → 不容易漏掉最像的（召回率高），但慢；看的區少 → 快，但可能漏。沒有免費的午餐，要看業務能接受多少誤差。
⚖️ 暴力 vs 近似（ANN） 暴力 100% 正確但慢到不能線上用；ANN 用一點召回率換數百倍速度。線上幾乎都選 ANN，用暴力的結果當「標準答案」來檢查 ANN 夠不夠準。
⚖️ 記憶體 vs 磁碟 索引放記憶體最快但貴；壓縮後塞記憶體、或把冷門資料放硬碟分層。看你願意花多少錢換多少速度。
⚖️ 嵌入模型好不好，是天花板 翻譯機（嵌入模型）翻得爛，意思相近的點根本沒靠在一起，後面 ANN 再強也白搭。選對/微調嵌入模型，往往比調索引更重要。
💭 追問：為什麼建資料庫和查詢一定要用「同一台翻譯機」？ 因為不同模型把東西翻到不同的座標系。用 A 翻資料、用 B 翻查詢，就像一個用經緯度、一個用門牌號，根本對不上，算出來的「最近」毫無意義。
💭 追問：召回率不是越高越好嗎，為什麼不直接追求 100%？ 100% 召回＝幾乎等於暴力搜尋（看遍所有區），那就失去 ANN 「快」的意義了。實務是在可接受的延遲下，把召回拉到夠用（如 95%）就好。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「暴力 vs ANN 各掃幾個點、會不會漏」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8052 -t public
# 然後瀏覽器打開 http://localhost:8052
  1. 裡面預設有 9 個 4 維的點，刻意分成水果 / 交通 / 天氣三群。
  2. 在查詢框輸入一個查詢向量（例如 0.9,0.1,0,0，靠近「水果」那群）。
  3. 按 「兩者比較」：看 暴力 kNN 掃了 9 個（全部）、ANN 只掃了 3 個（一個群），以及召回率。
玩的時候想一下把查詢改成 0.5,0.5,0,0（剛好卡在水果和交通兩群中間），用 ANN（nprobe=1）查 → 你會看到召回率掉到 0.75（漏了一個落在隔壁群的近點）！再把 nprobe 調成 2 或 3（多看幾區），召回率就補回 1.0——這就親眼看到了「準 vs 快」的旋鈕。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有三個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 Similarity（量「兩點多像」的尺）→ VectorIndex（暴力 vs 分群 ANN 的大腦）→ VectorStore（存點的倉庫）。查詢一來，就靠這幾段跑一遍。

1量相似度：兩點方向多接近（Similarity）

一切的基礎：給兩個點，算「它們有多像」。最常用餘弦相似度——只看方向、不管長短。

function 相似度(a, b) {
    分子 = a 和 b 逐項相乘再相加     // 內積：方向一致 → 大
    分母 = a 的長度 × b 的長度       // 除掉長度，只比方向
    return 分子 / 分母              // 越接近 1 → 越像
}

這就是那把「尺」。整個搜尋無數次呼叫它——所以它要簡單、要快。

2暴力 kNN：跟全部比一遍，當標準答案（VectorIndex）

最直白的搜尋：把查詢點 q 跟每一個點都算一次相似度，排序，挑最像的 k 個。

function 暴力找最近(q, k) {
    對「每一個點」算 相似度(q, 點)    // ← 掃 N 個（全部）
    照分數由高到低排序
    return 前 k 個
}                                  // 一定正確，但 N 大就慢

它保證找到真正最像的，所以拿來當「標準答案」檢查 ANN 漏了沒。缺點就一個字：慢。

3ANN 分群：只比最近幾區（VectorIndex 的核心）

這是全題的心臟。先把點分群（事先做好），查詢時只進最近的 nprobe 個群找：

// 事先：把所有點分成 nlist 群，每群算一個「中心(centroid)」

function ANN找最近(q, k, nprobe) {
    對「每個群的中心」算 相似度(q, 中心)   // ← 只比幾個中心，很快
    挑出最近的 nprobe 個群

    候選 = 這幾個群裡的所有點            // ← 只掃 M 個（M 遠小於 N）
    對「候選」算 相似度(q, 點)、排序
    return 前 k 個
}

關鍵就在 挑最近的 nprobe 個群＋只掃這幾群的點：把「掃 N 個」變成「掃 M 個」。nprobe 調大 → 掃更多群 → 更準更慢；調小 → 更快但可能漏。這正是面試官最愛追問的一段。

💭 停一下：ANN 為什麼可能「漏掉」真正最近的點？

看答案 因為它只進最近的幾個群找。萬一「真正最像的點」剛好落在一個沒被選中的群裡（例如它離查詢點近、卻被分到隔壁群），就被跳過了。多看幾個群（調大 nprobe）能降低這種漏失。

4算召回率：ANN 跟標準答案差多少（VectorIndex）

最後一步，誠實地量「ANN 漏了沒」：拿 ANN 的答案跟暴力的標準答案比交集。

function 召回率(暴力答案, ANN答案) {
    命中 = 兩邊都有的編號數量          // 交集
    return 命中 / 暴力答案的數量       // 1.0 = 一個沒漏
}

這個數字就是「準」的量尺。demo 裡你調 nprobe 看著它變動，就能親手感受「準 vs 快」的取捨。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「億人城市找最像的人」解釋，這題的核心難關是什麼？ 要在幾億個點裡找「離查詢點最近的 k 個」。逐一比對全部（暴力）保證正確但慢到不能用。解法是事先把相近的點分區，查詢只進最近的幾區比——這就是 ANN。
Q2：暴力搜尋和 ANN 差在哪？各有什麼優缺點？ 暴力＝跟全部點比一遍，一定正確但慢；ANN＝只比最近幾區，快很多但可能漏掉一兩個。線上用 ANN，拿暴力結果當「標準答案」檢查 ANN 夠不夠準。
Q3：「召回率」是什麼？怎麼用一句話說明？ 召回率＝ANN 找到的，跟「標準答案（暴力結果）」重疊多少的比例。1.0 表示一個都沒漏。它是衡量 ANN「準不準」的尺。
Q4：nprobe（看幾區）調大調小各有什麼後果？ 調大＝多看幾區 → 更不容易漏掉最像的（召回率高）但更慢；調小＝少看幾區 → 更快但更可能漏。它就是「準 vs 快」的旋鈕。
Q5（工程）：為什麼建資料庫和查詢要用同一個嵌入模型？ 因為不同模型把東西翻到不同座標系，混用就像一邊用經緯度、一邊用門牌號，根本對不上，算出的「最近」沒有意義。
Q6（工程）：3 TB 向量裝不進記憶體，有哪兩招？ ① 量化壓縮（PQ）：把每個向量用更少位元近似存，省記憶體；② 分片：切成好幾台機器分開放，查詢時各台找各自的 top-k 再合併。
Q7（工程）：為什麼說「嵌入模型的品質是天花板」？ 如果翻譯機翻得爛，意思相近的東西根本沒被放在一起，那後面 ANN 再快再準也只是在錯的座標上找最近——選對/微調嵌入模型往往比調索引更關鍵。

15. 30 秒回顧

向量搜尋 = 把萬物變成座標上的點，找離查詢點最近的幾個。

4 個關卡：① 用翻譯機變成向量 → ② 事先分群（分區） → ③ 查詢只比最近的幾區（ANN） → ④ 用「看幾區」旋鈕在召回率 vs 速度之間調。

3 個怕：怕慢/算不動（分群只比近區）、怕漏掉最像的（調 nprobe 拉召回）、怕裝不下記憶體（壓縮 + 分片）。

工程細節一句話：暴力是「標準答案」但 O(N) 不能線上 → ANN（IVF 分群只搜近群 / HNSW 沿圖跳近）把比對量壓成 M；3 TB 用量化壓縮 + 分片裝下；嵌入模型品質是天花板；召回 vs 速度沒有免費午餐。
