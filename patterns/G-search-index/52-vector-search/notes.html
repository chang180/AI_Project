52 向量資料庫 / ANN 搜尋：用「語意相近」找，而非關鍵字精準比對

G 搜尋/索引 建議 48 分鐘 Design a Vector Database / ANN Search

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把資料變成向量（嵌入），查詢時找『方向最相近』的 k 筆」的題目。 難點不在「算誰最像」（那是相似度的基本功），而在 幾億支高維向量，怎麼不用逐筆掃就找到最近鄰——用 ANN（近似最近鄰）換速度，並接受一點召回率損失。它是 ⑬RAG、㊸推薦、以圖搜圖 的共同地基。

1. 需求釐清

功能性需求

  • 給一個查詢（文字 / 圖片），回傳 「語意最相近」的 top-k 筆，而非關鍵字精準比對。
  • 支援 新增 / 更新 / 刪除向量，並可附帶 metadata（標籤、來源、時間）。
  • 過濾查詢：在相似度搜尋上再加 metadata 條件（如「只在 2024 年的文件裡找最像的」）。
  • 可選擇相似度度量：cosine（看方向） 或 內積 / 歐氏距離。

非功能性需求

  • 低延遲：查詢 p99 < 50ms（RAG / 推薦的關鍵路徑）。
  • 高召回率：ANN 是近似，但要儘量逼近暴力的「真正最近鄰」（如 recall@10 > 95%）。
  • 可擴展：向量數可到數十億、維度數百～上千 → 索引需分片、可能放不進單機記憶體。
  • 可更新：新向量要能近即時被搜到（增量索引，不必整段重建）。
面試開場必問 「向量搜尋 vs 傳統關鍵字 / B-Tree 索引差在哪？」 — 傳統索引找的是「值相等 / 範圍」，向量搜尋找的是「語意相近」（方向接近）。而且高維空間裡沒有能像 B-Tree 那樣排序的單一鍵，所以最近鄰本質上要嘛暴力掃全部、要嘛用 ANN（近似最近鄰） 用一點精度換大幅速度。

2. 規模估算

指標假設推算
向量數 N10 億筆每筆一個 metadata，索引需分片
維度 d768 維（典型 BERT 類嵌入）每筆 768 × 4B(float32) ≈ 3 KB
原始向量總量10 億 × 3KB≈ 3 TB（純向量，未含索引/metadata）
暴力 kNN 成本每查詢比 N 筆 × d 維10 億 × 768 ≈ 7,680 億次乘加 / 查詢 → 不可行
查詢 QPSRAG / 推薦線上查詢數千～數萬 QPS，必須 ANN + 索引常駐記憶體
關鍵洞察「3 TB 向量 × 數萬 QPS」→ 暴力掃全部完全不可行。設計圍繞兩件事：① 用 ANN 把每次查詢的比對量從 N 降到 M（M << N）；② 用量化壓縮把 3 TB 壓進可負擔的記憶體。

3. API 設計

# 新增向量（寫）
POST /api/v1/vectors
Body: { "vec": [0.12, -0.03, ...], "meta": { "doc_id": 42, "lang": "zh", "ts": 1717200000 } }
→ 201 { "id": 100123 }

# 相似度搜尋（讀，高頻）
POST /api/v1/search
Body: { "vec": [...], "k": 10, "metric": "cosine",
        "filter": { "lang": "zh" }, "nprobe": 16 }
→ 200 {
    "results": [
      { "id": 100123, "score": 0.92, "meta": {...} },
      ...
    ]
  }

# 刪除 / 更新（重建該向量的索引歸屬）
DELETE /api/v1/vectors/{id}
為何查詢也用 POST查詢向量是幾百～上千維的陣列，放不進 query string，通常用 POST body 傳。nprobe（要探查幾個群）是讓呼叫端調速度/召回率的旋鈕。

4. 資料模型

# 向量 + metadata（核心儲存）
vectors:
  id  → { vec: float[d], meta: { doc_id, lang, ts, ... } }
  #       ↑ 原始向量（供精算/精排）    ↑ 過濾與回顯用

# 索引結構（二選一或混用）
# ── IVF（倒排檔）：先把向量分成 nlist 群，每群一個 centroid
ivf:
  centroids: [ c0, c1, ..., c_{nlist-1} ]      # 每群的中心向量
  invlists:  { c0 → [id, id, ...], c1 → [...] } # 倒排檔：群 → 成員 id
# ── HNSW（分層小世界圖）：點之間連邊，分多層，上層稀疏供「快速跳近」
hnsw:
  nodes: { id → { neighbors_by_layer: [[...],[...]] } }

# 全集統計
stats: { N: 向量總數, d: 維度, nlist, metric }
為何要保留原始向量索引（IVF 群 / HNSW 圖）只負責「快速縮小候選集」；最後仍要用原始向量精算相似度排出 top-k。若用了量化壓縮（PQ），常見做法是「用壓縮向量粗篩 → 用原始向量精排」。

                                                                          ▲
 查詢 ──▶ 查詢(文字/圖) ─▶ 同一嵌入模型 ─▶ 查詢向量 q ─▶ ANN 搜尋(只比近群) ─┘
                                                       │

  1. 嵌入模型：把文字/圖片轉成同一空間的向量。建索引與查詢必須用同一個模型，否則向量不在同一座標系。
  2. 建索引：對全部向量分群（IVF）或建圖（HNSW），讓查詢能「只比一小部分」。
  3. ANN 搜尋：q 先定位到最近的幾個群 / 從圖的入口往近處跳，只精算這一小撮候選。
  4. (可選) 精排：用原始向量或更貴的模型對候選 top-k 重新排序，提升品質。
  5. 呼應 ⑬RAG（檢索增強生成：先向量檢索相關片段再餵給 LLM）與 ㊸推薦（用使用者/物品向量找最相近的物品）。

6. 核心元件深入

相似度度量：cosine vs 內積

  • cosine（餘弦相似度）＝看兩向量方向是否一致，忽略長度：dot(a,b) / (|a|·|b|)，範圍 [-1,1]，越大越像。
  • 內積（dot product）＝方向 + 長度都算。若向量先做 L2 正規化，內積與 cosine 等價（業界常見做法，省一次除法）。
  • 歐氏距離（L2）則是另一種選擇；三者在正規化後常可互換，重點是建索引與查詢用同一度量。

暴力 kNN（基準）

算 q 對全部 N 筆的相似度，排序取前 k。結果保證最正確（召回率 100%），是評估 ANN 的「正確答案」。代價是每查詢 O(N·d)，N 上億就跑不動——所以它只當基準，不上線。

ANN 近似：IVF（倒排檔）只搜近群

  • 建索引：用 k-means 把所有向量分成 nlist 群，每群算一個 centroid（中心向量），並建「群 → 成員」的倒排檔。
  • 查詢：q 先比 nlist 個 centroid（很少），挑最近的 nprobe 個群，只在這幾個群裡的向量做精算。
  • 掃描量：從 N 降到「那幾個群的成員數 M」（≈ N·nprobe/nlist）。nprobe 越大，掃越多、越準、越慢——這就是旋鈕。

ANN 近似：HNSW（分層小世界圖）

把向量當圖的節點，相近的連邊；分多層，上層稀疏（大跳）、下層密集（精修）。查詢從上層入口貪婪地往「離 q 更近」的鄰居走，逐層下沉到最底層收斂。查詢複雜度約 O(log N)，召回率通常比 IVF 高，但記憶體（存圖的邊）與建置成本較高。

召回率 vs 速度

ANN 可能漏掉「真正最近卻落在沒被探查的群 / 沒走到的圖區域」的向量 → 召回率 < 100%。召回率＝ANN 的 top-k 命中暴力答案 top-k 的比例。調 nprobe（IVF）或 efSearch（HNSW）就是在「快」與「準」之間滑動。

量化壓縮（讓 3 TB 裝得下）

PQ（Product Quantization，乘積量化）把向量切成幾段、每段用「碼本」近似成一個小整數 → 可壓到原本的 1/8～1/32。代價是相似度變成「近似的近似」，常用「PQ 粗篩 → 原始向量精排」補回精度。

7. 擴展與瓶頸

階段瓶頸對策
單機索引3 TB 向量裝不進記憶體量化壓縮（PQ） + 依 id 雜湊分片(shard)，每片一台
查詢延遲分片多、要查全部scatter-gather：併發查各 shard，gather 合併 top-k
維度太高維度詛咒：高維下「最近」與「最遠」差距變小、分群效果差先降維（如 PCA）+ 用對的度量；提高 nprobe/efSearch 補召回
建索引成本k-means 分群 / 建 HNSW 圖很貴批次離線建、增量更新；新向量先進「暫存區」線性掃，定期 merge 進主索引
更新 / 刪除IVF/HNSW 不易就地改刪除先打 tombstone，背景重建；近即時靠小段索引 + merge
Faiss / HNSW 的實務業界多用 Faiss（IVF+PQ） 或 HNSW（Milvus / pgvector / Elasticsearch kNN 底層）。面試講清楚「分群只搜近群 / 分層圖跳近、召回 vs 速度旋鈕、量化壓縮」後，務實答案常是「用 Faiss / Milvus / pgvector」，把心力放在嵌入品質、分片與召回調校。

8. 取捨與面試重點

  • 召回 vs 延遲：nprobe / efSearch 調大 → 召回率高但慢；調小 → 快但可能漏掉真正最近鄰。沒有免費的午餐。
  • 精確 vs 近似：暴力 kNN 100% 正確但 O(N) 不可行；ANN 用一點召回率換數量級的速度，線上幾乎都用 ANN。
  • 記憶體 vs 磁碟：索引常駐記憶體最快，但 3 TB 太貴 → 量化壓縮進記憶體、或冷資料放磁碟/SSD 分層。
  • IVF vs HNSW：IVF 建置快、記憶體省、好分片；HNSW 召回/延遲通常更佳但圖很吃記憶體、更新較麻煩。
  • 嵌入品質是天花板：再好的 ANN 也救不了爛嵌入。常見追問：怎麼選/微調嵌入模型、query 與 doc 是否用同一模型。
  • 自建 vs 託管：講清楚原理後，務實常選 pgvector（資料量小、想跟既有 SQL 共存）或 Milvus/Faiss（十億級）。

9. PHP Demo 說明

本題 demo 著重於向量搜尋的核心邏輯（也是面試真正的考點），可實際用 PHP 內建伺服器跑起來：

  • src/Similarity.php　cosine / 內積相似度。
  • src/VectorIndex.php　暴力 kNN（掃全部 N，當正確答案基準）＋ ANN（IVF 倒排檔）：簡化 k-means 分群 → 查詢只比最近 nprobe 個群 → 回報掃描數與召回率。
  • src/VectorStore.php　向量 + metadata 的 JSON 持久化（檔案鎖）。
  • public/index.php　路由：GET /（加向量、kNN 查詢看「暴力掃 N 個 vs ANN 只掃 M 個」與結果差異）、POST /api/vectors、GET /api/search?q=&k=&mode=brute|ann&nprobe=、GET /api/state；預設塞 9 筆 4 維「假嵌入」（分水果/交通/天氣 3 群）。
# 啟動
cd demo
php -S localhost:8052 -t public
# 瀏覽器開 http://localhost:8052，輸入查詢向量 q，按「兩者比較」：
#   暴力 kNN 掃 9 個；ANN(nprobe=1) 只掃 3 個（一個群）、看召回率
# 試 q=0.5,0.5,0,0（落在群邊界）→ ANN 召回率會掉到 0.75，調大 nprobe 補回
