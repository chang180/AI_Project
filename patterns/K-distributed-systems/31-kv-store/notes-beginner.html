1. 60 秒搞懂它在幹嘛

想像你開了一家超大型的「寄物櫃中心」。客人拿一張號碼牌（key）來，你就把對應的包裹（value）還他；他也可以拿號碼牌來存新包裹。就這麼簡單——這就是「鍵值儲存」：給我一把鑰匙（key），我還你一個值（value）。

難的不是「存跟取」。難的是：包裹多到一個倉庫裝不下，而且這個中心絕對不能關門（哪怕半夜一個倉庫失火）。於是你被迫做兩件麻煩事：把包裹分散到很多個倉庫、而且每個包裹都多印幾份放不同倉庫。一旦這樣做，三個頭痛問題就冒出來了——這題就是在解這三個問題。

🔑 一句話 分散式鍵值儲存 = 「永不打烊、裝得下無限多包裹的連鎖寄物櫃」。三個核心機制：① 包裹放哪個倉庫（一致性雜湊環）② 每個包裹存幾份、要幾份點頭才算數（Quorum）③ 兩個人同時改同一個包裹怎麼辦（向量時鐘）。

2. 為什麼一台資料庫不夠用？

很多人想：「不就是存資料嗎？買一台超強的資料庫不就好了？」問題有兩個，都是生活中你早就懂的道理——

📦 道理一：一個倉庫會「裝不下」 包裹一直變多（10 億個號碼牌），總有一天一個倉庫塞爆。買更大的倉庫（更貴的機器）有極限，而且很貴。 比較聰明的做法是多開幾間小倉庫，把包裹分散放——要更多空間就再開一間，這叫水平擴展。
🔥 道理二：一個倉庫會「失火（當機）」 只有一間倉庫，它一失火，所有包裹都拿不到了，整個中心關門。 所以每個包裹都要影印好幾份，放在不同倉庫。就算一間燒了，別間還有——這叫副本，是「高可用（永不打烊）」的關鍵。

所以這套系統的本質就是：把資料切碎分散（裝得下）＋ 每筆存好幾份（不怕掛）。聽起來簡單，但一這樣做，就會生出三個必須解決的難題，我們一關一關過。

💭 停一下：如果每個包裹都要影印 3 份放 3 間倉庫，那「存一個包裹」這件事，要等到幾間倉庫都收好才能跟客人說「存好了」？

看答案 這正是關卡 3（Quorum）要回答的問題！如果堅持等 3 間全收好才算數，那只要有 1 間倉庫在維修，客人就存不了（不夠「永不打烊」）。如果只要 1 間收到就說存好，又怕那 1 間剛好失火、另外 2 份還沒印。答案是取中間：「至少 2 間點頭就算數」——這個「要幾間點頭」就是後面要講的旋鈕。

3. 把系統拆成 4 個關卡

別被「分散式」嚇到。整件事就是闖 4 關，每一關解一個倉庫管理問題：

1包裹放哪間倉庫？（一致性雜湊環）

第一個問題：客人拿號碼牌「12345」來，你怎麼快速、固定地決定它該放哪間倉庫？而且加開一間新倉庫時，不要把所有包裹大搬風。

🕐 比喻：把倉庫和包裹都排在一個「圓圈時鐘」上 想像一個大時鐘的圓圈。先把每間倉庫釘在圈上某幾個鐘點位置。每個包裹也照它的號碼牌算出一個鐘點位置。 規則超簡單：包裹從自己的位置順時針走，遇到的第一間倉庫就是它的家。

這招的神奇之處在「加新倉庫」時。假設你在 3 點鐘方向新開一間倉庫 D：

  • 只有「D 前面那一小段」（本來要順時針跑去更遠倉庫的包裹）需要搬進 D。
  • 圈上其他所有包裹完全不用動。
😱 對比：笨方法「號碼 ÷ 倉庫數 取餘數」 如果你用「號碼牌 % 倉庫數」來分（3 間就 %3），一旦從 3 間變 4 間，幾乎每個包裹的餘數都變了 → 全部要重搬，整個中心癱瘓。圓圈時鐘法只搬一小段，這就是它叫「一致性」雜湊的原因——加減倉庫時，分配「不會大變」。
👯 虛擬節點：一間倉庫放很多「分身」在圈上 如果每間倉庫只在圈上釘 1 個點，運氣不好會「有的倉庫管超大一段、塞爆，有的管一小段、很空」。 解法：讓每間倉庫在圈上放幾百個分身（叫虛擬節點 vnode），像把一間倉庫的名牌灑滿整個圈。這樣每間倉庫平均分到的量就很平均，不會有人累死有人閒死。大倉庫還能多放幾個分身，多扛一點。

2每個包裹存幾份？（副本 N）

為了不怕失火，每個包裹要影印好幾份。慣例是 N = 3 份，放在圈上「順時針連續的 3 間不同倉庫」。

📄 比喻：重要文件影印 3 份放 3 個地點 重要合約你不會只放一個抽屜。你會影印 3 份：家裡、公司、銀行保險箱各一份。任何一個地方出事，另外兩份還在。

「從包裹位置順時針數的前 3 間不同倉庫」這份名單，工程上叫 preference list（偏好清單）——就是「這個包裹的 3 個指定存放點」。

3幾間點頭才算數？（Quorum：N / R / W）

有 3 份副本後，新問題來了：存包裹時要等幾間收好？取包裹時要問幾間？這是整題最重要的旋鈕。

✋ 比喻：要幾個人點頭 N=3：總共 3 份。W=2（Write）：存的時候，至少 2 間倉庫回報「我收好了」就跟客人說「存成功」。R=2（Read）：取的時候，至少問 2 間倉庫，拿它們的版本來比。

為什麼要這樣設計？因為有一條神奇的數學保證：

🎯 只要 W + R > N，讀的時候一定碰得到「最新那份」 W=2、R=2、N=3，2+2=4 > 3。意思是：你寫進去的那 2 間，和你讀的時候問的那 2 間，至少會有 1 間是同一間（鴿籠原理：3 個位子塞 2+2=4 個人，必有重疊）。那間就握有最新版本，所以你絕不會只讀到舊資料。

這組旋鈕就是在「快」和「不讀到舊資料」之間調：

  • W 調小（如 W=1）：存得飛快（1 間收到就好），但比較容易讀到舊的。
  • R 調小（如 R=1）：取得飛快（問 1 間就好），適合「讀超多、寫很少」的場景。
  • W=R=2：平衡，而且還能容忍「1 間倉庫在維修」照樣讀寫。

💭 停一下：如果某間該存的倉庫剛好失火了，湊不到 W=2 怎麼辦？難道就不給客人存了嗎？（這就違反「永不打烊」了！）

看答案 不行讓客人撲空。解法叫 hinted handoff（暫存代收）：那份本來要給「失火倉庫 C」的包裹，先暫存到旁邊一間健康倉庫，並貼張紙條「這份是要還給 C 的」。等 C 修好復活，代收的倉庫再把包裹和紙條交還給 C。這樣存包裹永遠成功，副本數事後也補齊了。

4兩個人同時改同一個包裹？（向量時鐘）

最燒腦的一關。因為資料有好幾份、又允許「分區時照寫」，就會發生：甲在 A 倉庫改了包裹，乙同時在 B 倉庫也改了同一個包裹。等網路恢復，兩邊版本不一樣，到底誰對？

🏷️ 比喻：每個包裹貼一張「版本便利貼」 每次有人改包裹，就在便利貼上記一筆「誰改過幾次」，像這樣：{A 改過 2 次, B 改過 1 次}。 要判斷兩個版本誰新誰舊，就比這兩張便利貼。

比對結果只有兩種：

  • 一張完全「罩得住」另一張（每個欄位都 ≥，且至少一個 >）：例如 {A:2,B:1} vs {A:1,B:1}，前者明顯是後者的「後續版本」→ 直接用新的，丟掉舊的。沒有衝突。
  • 互不相讓：例如 {A:2,B:0} vs {A:0,B:2}，A 多改了、B 也多改了，誰也不包含誰 → 這就是「兩個人同時各改各的」＝衝突。系統兩個版本都先留著（叫 sibling 兄弟版本），交給上層（App）決定怎麼合併。
🛒 為什麼留兩份、不自動挑一個？以購物車為例 甲在手機加了「牛奶」，乙在平板同時加了「麵包」。如果硬挑一個版本贏，另一個就默默消失了（客人會氣炸：我加的東西不見了）。聰明做法是兩份都留，合併時取聯集＝「牛奶＋麵包」，誰的東西都不丟。
😱 對比：簡單粗暴的 LWW（看時間戳，誰晚誰贏） 另一種做法叫 LWW（last-write-wins）：不貼便利貼，只看「哪筆時間比較晚」，晚的贏。超級簡單，但缺點是真正同時改的時候，會默默丟掉一筆（牛奶或麵包不見一個）。而且各倉庫的時鐘還可能對不準。所以：怕丟資料 → 用向量時鐘；要簡單、能容忍偶爾丟一筆 → 用 LWW。這是個取捨。
✅ 四關一句話 ① 用圓圈時鐘決定包裹放哪間倉庫（加倉庫只搬一小段）→ ② 每個包裹存 3 份（不怕失火）→ ③ 存問 2 間、取問 2 間（W+R>N 保證讀到新的）→ ④ 同時改用版本便利貼分辨衝突、留兩份不丟資料。

還有兩個小幫手機制，順手認識一下：

  • 協調者任一台都能當：客人走進任何一間倉庫都能辦事，那間倉庫會自己算出包裹該去哪 3 間、幫你轉發。沒有「總店」這種單點，所以不怕總店倒。
  • read repair（讀的時候順手修）：取包裹問了 2 間，發現其中一間的版本落後了，系統就順手把它更新成最新版——一邊讀一邊默默補洞。

4. 設計時最怕的 4 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕四件事，每個設計都是為了避開某個「怕」。

😱 怕「關門」（停止服務） 這類系統（Dynamo / Cassandra）最高原則是永遠能寫——亞馬遜購物車「加入」絕不能失敗。所以寧可暫時資料有點不一致，也要先讓客人存進去（事後再用便利貼和反熵慢慢喬一致）。可用 > 完美一致。
😱 怕「加倉庫要大搬家」 如果加一台機器就得把幾乎所有資料重搬一遍，系統根本沒辦法長大。所以用圓圈時鐘（一致性雜湊）只搬一小段，而不是「取餘數」那種全部重排的笨法。
😱 怕「讀到舊資料」 多份副本最怕你讀到還沒更新的那份。靠 W + R > N 這條不等式保證讀寫副本集合一定重疊，讀的時候總會碰到最新那份。這是用「多問幾間」換「不讀到舊的」。
😱 怕「默默丟掉一筆寫入」 兩人同時改，如果隨便挑一個贏，另一個就無聲消失，使用者會抓狂。用向量時鐘認出「這是同時改的衝突」並保留兩份，把「要丟誰」的決定權交給最懂業務的上層，而不是系統亂猜。

7. 系統怎麼用？（API）

🎫 比喻：寄物櫃的存取單 API 就是客人和寄物中心之間講好的「存取單格式」。這題只有三張單，超簡單： 
① 存包裹
   PUT /kv/{號碼牌}
   給我：{ value(包裹內容), context(你上次拿到的版本便利貼) }
   我回：新的版本便利貼，例如 { "A":2, "B":1 }

為什麼存的時候要附上「上次的便利貼(context)」？因為這樣系統才知道「你這次改，是基於哪個版本改的」——是接著舊版往下寫（蓋掉），還是跟別人同時改（要留 sibling）。這就是購物車能正確合併的關鍵。

② 取包裹
   GET /kv/{號碼牌}
   我回：正常時 → 一個版本
        衝突時 → 好幾個 sibling（請你合併後再 PUT 回來）

取的時候可能拿到兩份並存版本，這不是 bug，是系統老實告訴你「這兩份是同時改的，我不敢幫你亂選，你來決定」。

③ 刪包裹
   DELETE /kv/{號碼牌}   → 其實是放一張「墓碑」標記

刪除為什麼不是真的擦掉、而是放一張「墓碑(tombstone)」？看下面取捨那段，這是個有趣的坑。

8. 系統要記住哪些事？（資料模型）

📒 比喻：要記的三樣東西 系統腦袋裡要記三樣：包裹本身、圓圈時鐘的座位表、還有每間倉庫怎麼把包裹存到硬碟上。

📦包裹本體（key → 值 + 版本便利貼）

每個號碼牌對應「一個或多個版本」：key → [ { value, 版本便利貼 }, ... ]。 平常只有一個版本；同時改造成衝突時，就會有好幾個 sibling 並存。

🕐圓圈時鐘座位表（環 + preference list）

記著「每間倉庫（含它的分身 vnode）釘在圈上哪些鐘點」。任一間倉庫只要查這張表，就能自己算出「某號碼牌該放的 3 間倉庫」，不必問總店。

💾單間倉庫怎麼把包裹放上架（LSM-tree）

每間倉庫內部，因為存包裹的動作超級頻繁，用一種叫 LSM-tree 的方式：新包裹先丟記憶體 + 排隊一筆筆順序寫到硬碟末端（順序寫超快），背景再慢慢整理。

為什麼順序寫比較快？ 就像你整理收件，與其每來一封就跑去檔案櫃正確位置插入（到處跑、慢），不如先全部疊在桌上一疊（按來的順序），有空再一次整理歸檔。本題重點在「分散」這層，單間倉庫的存法知道個大概就好。

9. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
加倉庫時搬資料，影響正在辦事的客人圓圈時鐘只搬一小段(~1/N)；分身(vnode)讓搬遷來源分散；背景搬遷限速別擠到正常流量
熱門包裹：某個號碼牌被狂存狂取，那 3 間累死在 key 後面加隨機後綴把它打散成多份；前面加快取；或單獨給它更多副本
有倉庫在維修，湊不到 W 間點頭hinted handoff暫存代收，維持「永遠能存」，事後補回
衝突的 sibling 版本越堆越多上層讀到就合併再寫回；便利貼太長就剪枝
墓碑(刪除標記)堆積，拖慢掃描放一陣子(GC grace)後背景清掉——但不能太快，原因見下方取捨

10. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的四個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 強一致 vs 最終一致（CAP 怎麼選？） 用白話想：停電（網路分區）時，你的寄物中心要選哪一種脾氣？ （A）「繼續營業，就算各倉庫資料可能短暫對不上，事後再喬」——這是本題走的 AP（最終一致），Dynamo / Cassandra 這派，換來「永不打烊」。 （C）「寧可暫時拒絕服務，也不能讓你拿到可能不對的資料」——這是 CP（強一致），像銀行轉帳那種，設計完全不同（用 Raft/Paxos）。面試一開場就要先問清楚要哪種！
⚖️ 向量時鐘 vs LWW（誰晚誰贏） 向量時鐘不丟資料但複雜（要處理 sibling、便利貼會變長）；LWW 超簡單但同時改會默默丟一筆。看你的業務「能不能容忍偶爾丟一筆寫入」來選。
⚖️ W / R 怎麼配？ 這就是一致性旋鈕。讀多寫少 → R 調小（讀快）；要寫最高可用 → W 調小（存快）；要平衡又容錯 → W=R=2、N=3。能說出「為什麼 W+R>N」並依場景給組合，是這題的得分點。
💭 追問：刪除為什麼不直接擦掉、要放「墓碑」？ 因為資料有好幾份副本，可能有一份正在維修、沒收到「刪除」。如果你真的擦掉其他份，等那份落後的副本醒來，背景同步(反熵)會看到「咦你怎麼少了這筆」，好心地把已刪的資料又補回來＝復活了！所以刪除要留一張「墓碑」明確說「這筆是被刪的」，讓落後副本同步到「它已死」，過一段安全期(GC grace)才真的清掉。
💭 追問：怎麼知道哪間倉庫死了？沒有總店誰來宣布？ 靠 gossip（八卦傳話）+ 心跳：每間倉庫定期跟幾個隨機鄰居交換「我還活著、我知道誰活著」的消息，像八卦一傳十十傳百，整個叢集很快就都知道某間沒回應 = 大概掛了。去中心化，不需要總店宣布。

11. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，用一支 PHP 程式模擬一整個多倉庫叢集，讓你親眼看到圓圈時鐘、Quorum、向量時鐘衝突：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8031 -t public
# 然後瀏覽器打開 http://localhost:8031
  1. 先 put 幾筆資料，看看每筆被分到圈上的哪 3 間倉庫（preference list）。
  2. get 出來（走 R quorum），看它怎麼比版本挑最新。
  3. 把某間倉庫設成 down，再 put → 觀察 hinted handoff（暫存代收）怎麼幫你撐住「還是存得進去」。
  4. 製造並發寫（兩邊各改），看系統怎麼留下 sibling 兩個版本而不是默默丟一筆。
  5. 加一間新倉庫 D，觀察是不是只有一小段資料被搬過去（而不是大搬家）。
玩的時候想一下把同一個 key 在「兩間互相看不到對方」的情況下各改一次，是不是就生出了 sibling？這就是 關卡 4 向量時鐘在運作。把一間倉庫關掉再 put，能不能成功？這就是 關卡 3 的 hinted handoff 在撐住「永不打烊」。

12. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「前面那幾關到底是哪幾段程式做的」。 好消息：整個叢集核心就三個小檔，剛好對應前面講的機制。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 三個檔 = 三個小幫手 ConsistentHashRing（圓圈時鐘座位表＋順時針找倉庫＋算 preference list）→ VectorClock（比兩張版本便利貼、判斷誰新誰舊或衝突）→ Cluster（把上面兩個串起來：存問 W 間、取問 R 間、hinted handoff 暫存代收、read repair 順手補）。一次 put 或 get，就照這個順序跑一遍。

1圓圈時鐘：順時針找倉庫＋算出存哪 3 間（ConsistentHashRing）

每間倉庫（含它的分身 vnode）都算出一個「鐘點座標」釘在圈上、由小到大排好。包裹要找家，就是「從自己的座標順時針走，遇到第一間倉庫」：

function 找這包裹的家(號碼牌) {
    座標 = hash(號碼牌)                    // 包裹在圈上的鐘點
    找出排好序的座標裡「第一個 >= 座標」的點   // ← 順時針第一間
    if (走到底還沒找到) 繞回圈上第 0 個點       // ← 圓圈是會繞回來的
    return 那個點所屬的倉庫
}

關鍵就是「座標排好序 + 找第一個 ≥ 它的」＝順時針最近。座標排好序後用二分搜尋找，超快。每間倉庫放幾百個分身(vnode) 灑滿整圈，分配才平均。

而「存 3 份要存哪 3 間」＝preference list，就是從這個點繼續順時針數，取前 N 間「不同」倉庫（同一間倉庫的多個分身要跳過，確保 3 份落在 3 台真的不同機器）：

function 算preference_list(號碼牌, N) {
    從順時針第一個點開始，一路往下走
    每遇到一間「還沒收過的」倉庫就收進名單   // ← 跳過同一間的其它分身
    收滿 N 間不同倉庫就停
    return 名單   // 例：[A, B, C]
}

這就是前面說的「順時針連續的 3 間不同倉庫」。加一間新倉庫時，因為只動到圈上一小段，只有少數包裹的家會變——這就是一致性雜湊「加倉庫不大搬家」的由來。

2存包裹：問 W 間點頭，掛掉的走 hinted handoff（Cluster.put）

存的時候，先算出 preference list 那 3 間，再逐一去送。倉庫活著就直接存、算一個 ack；倉庫掛了就找旁邊一間健康倉庫暫存代收並貼紙條（hinted handoff）。湊滿 W 個 ack 就跟客人說「成功」：

function put(號碼牌, 包裹) {
    名單 = 算preference_list(號碼牌, N)   // 例：[A, B, C]
    新便利貼 = 在舊版本上，把「協調者這間」改過次數 +1   // ← 讀-改-寫

    ack = 0
    foreach (倉庫 in 名單) {
        if (倉庫活著) {
            把包裹存進這間倉庫; ack++          // ← 正常存
        } else {
            找一間圈外健康倉庫，暫存代收 + 貼紙條「這份是要還給它的」
            ack++                            // ← hinted handoff 也算一次 ack
        }
    }
    return (ack >= W) ? "存成功" : "存失敗"   // ← W quorum
}

注意那行「把協調者改過次數 +1」——這就是更新版本便利貼。還有「掛掉的倉庫也照算 ack」：因為包裹有暫存代收、事後會交還，所以「永遠存得進去」。等倉庫復活，掃所有紙條把屬於它的交還回去，副本數就補齊了。

3版本便利貼：比兩張紙條判斷誰新、還是衝突（VectorClock）

這是全題最燒腦、也最核心的一段。比兩張便利貼 {A:2, B:1} 和 {A:1, B:1}，逐個欄位掃一遍，看「誰在某欄位比對方大」：

function 比兩張便利貼(甲, 乙) {
    甲有贏的欄位 = false;  乙有贏的欄位 = false
    foreach (倉庫 in 甲和乙出現過的所有欄位) {
        if (甲[倉庫] > 乙[倉庫]) 甲有贏的欄位 = true
        if (乙[倉庫] > 甲[倉庫]) 乙有贏的欄位 = true
    }
    if (兩邊都沒贏)        return 相同        // 同一版本
    if (只有甲有贏)        return 甲比較新     // 甲罩得住乙 → 用甲、丟乙
    if (只有乙有贏)        return 乙比較新     // 乙罩得住甲 → 用乙、丟甲
    return 同時改的衝突                       // ← 互不相讓 = 各改各的 → 留兩份 sibling
}

白話就是：一張「包含」另一張（每欄都 ≥）＝那張比較新，丟舊的；互不包含（甲多改了某欄、乙也多改了某欄）＝同時改，誰也不能丟，兩份都留成 sibling。這正是前面「版本便利貼」比喻在程式裡的樣子。

💭 停一下：{A:2,B:0} 和 {A:0,B:2} 比起來是哪一種？

看答案 A 欄甲贏（2>0）、B 欄乙贏（2>0），兩邊都有贏的欄位 → 互不相讓 → 同時改的衝突。系統不會硬挑一個，而是兩份都留成 sibling，交給上層（像購物車取聯集）合併，誰加的東西都不丟。

4取包裹：問 R 間、調和版本、順手 read repair（Cluster.get）

取的時候問 R 間健康倉庫，把它們回來的所有版本攤平，用上面的「比便利貼」做調和（丟掉被別人罩住的舊版，只留下互為衝突的 sibling）。然後順手把落後的副本補成最新（read repair）：

function get(號碼牌) {
    問到 R 間健康倉庫的版本           // ← R quorum
    調和：把所有版本兩兩比便利貼
         － 被別人罩住的舊版 → 丟掉
         － 互不相讓的 → 都留著(sibling)
    foreach (剛問過的倉庫) {
        if (這間的版本落後了)
            把它默默覆寫成調和後的最新版   // ← read repair：一邊讀一邊補洞
    }
    return 調和後的版本(可能一個，也可能多個 sibling)
}

「調和」用的就是 §3 那個比便利貼的函式。read repair 就是那行「發現落後就默默覆寫」——讀的時候順手把不一致補平，副本之間慢慢就一致了（最終一致）。回傳多個 sibling 不是 bug，是老實告訴你「這兩份同時改的，你來合併」。

13. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「圓圈時鐘」解釋，為什麼加一間新倉庫不會引起大搬家？ 倉庫和包裹都排在一個圓圈上，包裹順時針找第一間倉庫當家。新加一間倉庫，只有「它前面那一小段」原本要跑去更遠倉庫的包裹要搬進來，其他包裹完全不動。對比「號碼 % 倉庫數」一改倉庫數就全部重算、要全搬。
Q2：虛擬節點(vnode)是要解決什麼問題？ 如果每間倉庫只在圈上放 1 個點，分配容易不均（有的管超大段塞爆、有的很空）。讓每間倉庫放幾百個「分身」灑滿整圈，平均分到的量就很平均；大倉庫還能多放幾個分身多扛點。
Q3：N=3、W=2、R=2，為什麼能保證「讀不到舊資料」？ 因為 W+R=4 > N=3。寫進去的 2 間和讀的時候問的 2 間，3 個位子塞 4 個人必有重疊 → 至少 1 間同時是「被寫的」和「被讀的」，那間握有最新版本，所以讀一定碰得到最新的。
Q4：兩個人同時改同一筆，向量時鐘怎麼處理？為什麼不直接挑一個？ 比兩張「版本便利貼」：若一張罩得住另一張就用新的；若互不相讓＝同時改的衝突，兩份都先留著(sibling)交給上層合併。不自動挑是因為硬挑會默默丟掉另一筆（購物車有人加的東西會不見），讓最懂業務的上層合併（如取聯集）才不丟資料。
Q5（工程）：put 的時候為什麼要附上「上次讀到的版本便利貼(context)」？ 讓系統知道「你這次改是基於哪個版本」。據此判斷：是接著那版往下寫（蓋掉舊的），還是跟別人同時改（要保留成 sibling）。這是「讀-改-寫」因果鏈的關鍵，購物車正確合併就靠它。
Q6（工程）：刪除為什麼要放「墓碑」而不是直接擦掉？ 因為有多份副本，可能某份正維修沒收到刪除指令。若真的擦掉，背景同步(反熵)看到那份落後副本「還有這筆」會好心把它復活。放墓碑明確標「已刪」，讓所有副本都同步到它已死，過安全期才真清。
Q7（工程）：CAP 在這題怎麼用一句白話帶出來？ 停電（網路分區）時你的系統選哪種脾氣：AP=「繼續營業、資料短暫可能不一致、事後喬」（本題、Dynamo/Cassandra，換永不打烊）；CP=「寧可拒絕服務也要保證資料正確」（像銀行，用 Raft/Paxos）。一開場先確認要哪種。

14. 30 秒回顧

分散式鍵值儲存 = 永不打烊、裝得下無限多包裹的連鎖寄物櫃。

4 個關卡：① 圓圈時鐘決定包裹放哪間倉庫（加倉庫只搬一小段、分身讓負載平均）→ ② 每筆存 3 份(N) → ③ 存問 2 間、取問 2 間（W+R>N 保證讀到新的）→ ④ 同時改用「版本便利貼」分辨衝突、留兩份(sibling)不丟資料。

4 個怕：怕關門（可用優先，AP）、怕加倉庫大搬家（一致性雜湊）、怕讀到舊的（W+R>N）、怕默默丟一筆（向量時鐘留 sibling）。

工程細節一句話：三張存取單(put/get/delete)＋附 context 做讀-改-寫 → 記三樣（包裹/座位表/單間 LSM 存法）→ 找塞車環節各自拓寬（限速搬遷、打散熱 key、hinted handoff、合併 sibling、墓碑別太早清）→ CAP 一開場先選 AP 還 CP。
