㉛ 分散式鍵值儲存：一致性雜湊 × Quorum × 向量時鐘

K 分散式系統 建議 55 分鐘 Design a Distributed Key-Value Store

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用「圓圈時鐘排座位、影印三份、版本便利貼」的比喻把核心機制講清楚，再回來看這份工程速查。
一句話定位 本質是「去中心化、可水平擴展、高可用（AP）的鍵值儲存」——也就是 Amazon Dynamo / Cassandra 那一類。 面試真正考的不是 get/put API，而是三件事如何咬合： 一致性雜湊（資料放哪、加機器怎麼搬）、quorum N/R/W（容錯與一致性旋鈕）、 向量時鐘（並發寫怎麼調和而不丟資料）。

1. 需求釐清

功能性需求

  • put(key, value) / get(key)：簡單的鍵值介面（value 是不透明的 blob）。
  • 資料自動分散到多台節點，水平擴展到數百~數千節點。
  • 每筆資料有 多副本（典型 N=3），單點故障不丟資料、不中斷服務。
  • 加 / 移除節點時，只搬動少量資料，且服務不中斷（線上重平衡）。

非功能性需求

  • 高可用（always writeable）：Dynamo 的核心需求——購物車「加入」永遠要成功，哪怕分區也照寫。
  • 低延遲：p99 個位數毫秒；任一節點都能當協調者，就近處理。
  • 可調一致性：用 N/R/W 旋鈕在「一致性」與「可用性/延遲」之間滑動。
  • 去中心化：無單一 master，避免單點瓶頸與單點故障。
面試開場必問：CAP 取捨 網路分區（P）無法避免，必須在 C（一致性） 與 A（可用性） 之間選。 本題走 AP（如 Dynamo / Cassandra）：分區時仍可讀寫，事後再用向量時鐘 / read repair / 反熵讓資料「最終一致」。 若面試要的是強一致（CP），那是另一類系統（如 Spanner / ZooKeeper / etcd 用 Paxos/Raft），設計完全不同——務必先確認。

2. 規模估算

指標假設推算
資料量10 億 key × 每筆 1 KB原始 ~1 TB；N=3 副本 → ~3 TB
吞吐50 萬 ops/s（讀寫各半）分散到 100 節點 → 每節點 ~5k ops/s
單節點容量每台 ~1 TB SSD3 TB / 1 TB ≈ 至少 3~5 台起；留成長空間取 10+ 台
vnode 數每實體節點 100~256 個 vnode環上點夠密 → 負載偏差 < 幾個 %
副本與 quorumN=3, R=2, W=2W+R>N 成立；可容忍 1 節點故障仍讀寫
關鍵洞察 儲存與吞吐都靠「把 key 空間切碎、均勻散佈到很多節點」來線性擴展； 難點不在單機效能，而在「切碎的策略」（一致性雜湊）與「副本一致性」（quorum + 衝突調和）。

3. API 設計

# 寫入（協調者可為任一節點）
PUT /kv/{key}
Body: { "value": "...", "context": "<上次讀到的向量時鐘>" }
→ 200 { "version": { "A": 2, "B": 1 } }     # 回傳新的向量時鐘

# 讀取
GET /kv/{key}
→ 200 { "values": [ { "value": "...", "version": {"A":2} } ],   # 正常：單一版本
        "context": "..." }
→ 200 { "values": [ {"value":"red","version":{"A":1}},          # 衝突：多個 sibling
                    {"value":"blue","version":{"B":1}} ] }       # 由應用層合併後再 put 回去

# 刪除（其實是寫一個 tombstone 墓碑標記）
DELETE /kv/{key}
為何 get 要回 context 這是「讀-修改-寫」的因果鏈：客戶端把上次讀到的向量時鐘（context）在 put 時帶回， 系統才知道這次寫是「基於哪個版本」，據此判斷是覆蓋舊版還是產生新 sibling。Dynamo 購物車就是這樣合併的。

4. 資料模型

每個 key 對應一個或多個「版本」（並發時會有多個 sibling）：
  key → [ { value, vector_clock }, ... ]
        vector_clock = { node_id: counter }    # 如 {A:2, B:1}

一致性雜湊環（每節點放多個 vnode）：
  ring: 環座標(0~2^32-1) → 實體節點
  preference list(key) = 從 hash(key) 順時針，前 N 個「不同實體節點」

節點本地儲存（真實系統）：
  memtable（記憶體） + SSTable（磁碟，LSM-tree）+ commit log
  Merkle tree（每段 key range 一棵）→ 副本間快速比對差異做反熵
為何是 LSM-tree KV 寫入極重，LSM-tree（先寫記憶體 + 順序追加，背景 compaction）把隨機寫轉成順序寫，吞吐遠勝 B-tree。 Cassandra / RocksDB / LevelDB 皆此架構。本題重點在分散層，單機儲存引擎點到為止即可。

5. 高階架構（去中心化）

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
           hash(key) 落在環上某點
 client ──▶ 任一節點(協調者) ──▶ 一致性雜湊環 ──▶ preference list = [A, B, C]
                  │                                    │
                  ├── put：寫 N 個副本，收到 W 個 ack 即回成功 ─────┐
                  └── get：讀 R 個副本，比向量時鐘調和 + read repair │
                                                                    ▼
                          A ◀── gossip ──▶ B ◀── gossip ──▶ C   (節點互相同步成員/狀態)
                          每節點: memtable+SSTable, Merkle tree(反熵)
  1. 無 master：任一節點都能當協調者，用一致性雜湊自行算出 key 該寫去哪 N 台。
  2. gossip 協定：節點間週期性交換「誰活著、環長怎樣」的成員資訊，去中心化地維護叢集視圖。
  3. 協調者轉發：協調者把 put/get 轉發給 preference list 上的副本，依 W/R 收集回應。

6. 核心元件深入

① 一致性雜湊 + 虛擬節點（vnode）

  • 問題：hash(key) % N 一旦 N 變動，幾乎所有 key 重映射 → 雪崩、大搬遷。
  • 一致性雜湊：節點與 key 都映射到同一個環，key 順時針找第一個節點。加/移除節點只影響「環上相鄰一段」的 key（約 1/N）。
  • vnode 解資料傾斜：每個實體節點只放 1 點時分佈不均，少數節點扛太多。讓每節點放 V 個 vnode（hash("A#0"), hash("A#1")…）均勻散佈，負載偏差大幅下降；移除節點時負擔也較平均地分給其它節點，異質硬體還能按容量配不同數量的 vnode。

② Quorum N / R / W

  • N＝副本數（如 3）；W＝寫要幾個 ack 才算成功；R＝讀要讀幾個副本。
  • 核心不等式 W + R > N：保證「寫的副本集合」與「讀的副本集合」必定重疊至少一個節點 → 讀一定看得到最新的寫（在向量時鐘調和後）。
  • 旋鈕怎麼調：W=N, R=1 讀快寫慢（讀多寫少場景）；W=1, R=N 寫快讀慢、寫最高可用；W=R=2, N=3 平衡且容忍 1 故障。
  • 寬鬆 quorum（sloppy quorum）：當 preference list 上有節點故障，就寫到環上下一個健康節點（搭配 hinted handoff），維持「永遠可寫」。

③ 向量時鐘 vs LWW

  • 向量時鐘＝{node: counter}，協調者寫入時把自己分量 +1。比較兩版本：每分量都 ≤ 且至少一個 < → happens-before（舊，丟掉）；互有大小 → 並發衝突，保留為 sibling 交給應用層合併（如購物車取聯集）。
  • LWW（last-write-wins）：只比時間戳，誰新誰贏。簡單，但時鐘不同步或真正並發時會靜默丟寫。Cassandra 預設用 LWW（cell 級時間戳）換取簡單；Dynamo 用向量時鐘換取不丟寫。

④ Read Repair 與 Hinted Handoff

  • read repair：get 時若發現某些副本落後，順手把它們補成最新（調和後）版本——讀路徑上的被動修復。
  • hinted handoff：目標副本當機時，協調者把資料暫存到一個「代收」健康節點並打上「這是給 C 的」提示；C 復活後，代收節點把暫存交還給 C——保障寫可用性與最終副本數。

⑤ Merkle Tree 反熵（anti-entropy）

副本長期偏差怎麼補齊？每個 key range 建一棵 Merkle tree（葉子是資料雜湊，往上逐層雜湊）。兩副本只要從根往下比對雜湊，相同的子樹整段跳過，快速定位差異段並同步，省去逐筆掃描。這是背景持續跑的「補洞」機制。

7. 擴展與瓶頸

瓶頸症狀對策
加節點重平衡搬資料影響線上流量一致性雜湊只搬 ~1/N；vnode 讓搬遷來源分散；限流背景遷移
熱 key某 key 被狂讀寫，單副本組過載應用層加隨機後綴打散；前置快取；該 key 提高 N
分區/故障時的可用性嚴格 quorum 湊不到 Wsloppy quorum + hinted handoff，維持可寫，事後修復
sibling 爆炸長期不合併，版本越堆越多應用層讀到就合併再寫回；向量時鐘加時間戳剪枝（Dynamo 的做法）
墓碑（tombstone）堆積刪除標記拖慢掃描設定 GC grace 後 compaction 清除（注意別讓已刪資料復活）

8. 取捨與面試重點

  • 強一致 vs 最終一致：本題選最終一致（AP）換高可用低延遲；要強一致就改用 Raft/Paxos 複製（CP），但會犧牲分區時的可用性與部分延遲。
  • 向量時鐘 vs LWW：向量時鐘不丟寫但複雜（sibling 要應用層處理、版本會膨脹）；LWW 簡單但會丟並發寫。依業務能否容忍丟寫來選。
  • W/R 的選擇即一致性旋鈕：講得出「為什麼 W+R>N」並能依讀寫比例給出 N/R/W 組合，是這題的得分點。
  • 常見追問：怎麼偵測節點死活？（gossip + 心跳）怎麼避免遷移風暴？（vnode + 限流）刪除為何要墓碑？（否則反熵會把已刪資料從落後副本「復活」）。

9. PHP Demo 說明

本題 demo 在單一 PHP process 內模擬一個多節點叢集，把分散式 KV 真正的考點跑起來：

  • src/ConsistentHashRing.php　一致性雜湊環 + vnode：hash(key)（crc32）順時針查找、preference list（前 N 個不同實體節點）、加/移除節點只重映射少量 key。
  • src/VectorClock.php　向量時鐘：increment / compare（happens-before vs 並發）/ merge。
  • src/Cluster.php　叢集核心：put（quorum W + hinted handoff）、get（quorum R + 向量時鐘調和 + read repair + sibling）、節點 up/down/add/remove；各節點 KV 存 data/node_*.json，以 flock 原子化。
  • public/index.php　路由 + 測試頁：看環/vnode 分佈、查 preference list、put/get、切換節點 up/down 看 hinted handoff、製造並發寫看向量時鐘 sibling。
# 啟動
cd demo
php -S localhost:8031 -t public
# 瀏覽器開 http://localhost:8031
# 試：put 幾筆 → get（走 R quorum）→ 用 down 製造分割做並發寫看 sibling → 加節點 D、down C 看 hinted handoff
誠實聲明 Demo 是單一 process 模擬多節點：沒有真實網路、沒有 gossip 定時心跳、沒有 Merkle tree 反熵背景程序。 但一致性雜湊 + vnode、quorum N/R/W、向量時鐘衝突偵測、read repair、hinted handoff 的邏輯皆為真實可執行——這正是本題的核心考點。 要變成真叢集，把「各節點一個 JSON 檔」換成「各節點一支 RPC 服務」即可，核心演算法不變。僅用 json + hash(crc32) 擴充，每檔 declare(strict_types=1);。
