㊱ 分散式鎖 + 服務發現：用協調服務管住分散式系統的「真相來源」

K 分散式系統 建議 45 分鐘 Design a Distributed Lock & Service Discovery

一句話定位 本質是「一個小而強一致（CP）的協調服務（etcd / ZooKeeper），對外提供互斥鎖、選主、設定、服務名冊四種原語」。 單機 synchronized 很簡單；面試真正考的是 當持鎖者 GC 暫停 / 網路分割 / 時鐘漂移時，分散式鎖如何還能「安全」—— 答案的核心是 fencing token。

1. 需求釐清

為何需要分散式協調？四種剛需原語

  • 互斥（分散式鎖）：多台機器同時想動同一份資源（同一筆訂單、同一個檔案、同一個 cron 任務），需要保證「同一時刻只有一個」。
  • 選主（leader election）：一批對等節點要選出一個 leader 來做排程 / 寫入協調，掉線後要能自動換人。
  • 設定中心（config）：feature flag、限流參數要能集中改、即時推給所有節點。
  • 服務發現（service registry）：實例會擴縮、會掛，呼叫端要能查到「目前還活著的實例清單」。

非功能性需求

  • 強一致 / 線性一致（CP）：鎖與選主絕不能「同時兩個人都覺得自己拿到了」。寧可在分割時拒絕服務，也不能授予衝突。
  • 高可用：協調服務是整個系統的「真相來源」，自身掛了影響面極大 → 用奇數節點（3/5）多數決容錯。
  • 低延遲：鎖 acquire / 服務發現查詢在毫秒級。
  • 安全性 > 活性：鎖的「互斥（safety）」不可破；偶爾因租約過期早放鎖（活性受損）可接受。
面試開場必問 「這把鎖是為了效率（避免重複做白工，偶爾兩個人做也只是浪費）還是為了正確性（兩個人同時做會壞資料）？」 若是正確性，光有鎖不夠，一定要談 fencing token，否則直接出局。

2. 規模估算

指標假設推算
協調節點數容忍 1 台掛3 節點（quorum=2）；容忍 2 台掛 → 5 節點
鎖 / 寫操作 QPS需經 Raft 多數決落盤單叢集約 1萬～數萬寫 QPS（受 fsync + RTT 限制，不是無限）
watch 連線每個微服務實例監看設定 / 名冊數萬～數十萬長連線 → 注意 watch 風暴
單筆 key 大小鎖 / 註冊資訊小建議 < 1KB；etcd 預設整庫上限約 8GB（協調服務不是資料庫）
租約 TTL服務心跳常見 5～30 秒；TTL 越短偵測掉線越快但心跳壓力越大
關鍵洞察協調服務寫吞吐有限且不該存大資料。它的價值在「強一致的少量關鍵元資料」，不是高吞吐儲存。把它當資料庫用是最常見的災難。

3. API 設計

# 分散式鎖（租約）
POST /v3/lock        { key, owner, ttl }      → { fencingToken, expireAt }
POST /v3/unlock      { key, owner }
POST /v3/renew       { key, owner, ttl }      # heartbeat 續租

# 下游受保護資源：寫入必須帶 fencing token
PUT  /resource/{id}  Header: X-Fencing-Token: 42   # 資源只接受 > 上次見過的 token

# leader election
POST /v3/campaign    { election, candidate, ttl } → { isLeader, leader, epoch }
GET  /v3/leader?election=scheduler

# watch（變更通知，通常是長連線 / streaming）
WATCH key=config:flag        → 推送該 key 的每次變更事件

# 設定中心
PUT  /v3/kv  { key, value }   GET /v3/kv?key=

# 服務發現
POST /v3/register   { service, instance, addr, ttl }   # 帶租約，須續租
GET  /v3/discover?service=payment   → [ 存活實例清單 ]

4. 資料模型

# 協調服務內部是一棵「有版本的 KV 樹」+ 租約表 + watch 表
locks        key → { owner, expireAt(lease), fencingToken, version }
                   -- fencingToken 來自全域單調遞增計數器，每次「成功授予」+1
fenced       resource → lastSeenToken           -- 下游資源各自記「見過的最大 token」
leases       leaseId → { ttl, lastHeartbeat, boundKeys[] }  -- 一個 session 綁多個 key
services     service → { instanceId → { addr, leaseId, meta } }  -- ephemeral：lease 死則消失
configs      key → { value, version }            -- 版本用於 watch / CAS
watches      key → [ watcherId... ]              -- 變更時逐一推送
為何 fencing token 必須單調遞增且全域唯一 它是「邏輯時鐘 / 鎖世代（epoch）」。下游資源靠「比大小」就能擋掉任何過期持有者的延遲寫入， 不需要信任時鐘、不需要知道鎖的存在。etcd 用 key 的 mod_revision、ZooKeeper 用 znode 的 zxid / czxid 當這個 token。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
        ┌───────────────── 協調服務叢集（CP，Raft 複製）─────────────────┐
        │   Node1(Leader) ⇄ Node2(Follower) ⇄ Node3(Follower)            │
        │   每筆寫(acquire/put/register) 經 Raft 多數決落盤後才回應        │
        └───────────────────────────────────────────────────────────────┘
            ▲ acquire→拿 fencingToken    ▲ watch 變更   ▲ register/discover
            │                            │              │
        ┌───┴────┐                  ┌────┴───┐     ┌────┴────┐
        │Client A│ ──帶 token 寫──▶ │ 下游資源│     │ 呼叫端  │ discover→存活實例
        └────────┘   (資源 fence    └────────┘     └─────────┘
        ┌────────┐    掉小 token)
        │Client B│
        └────────┘
  1. 協調服務本身是 CP 系統：3/5 節點用 Raft（etcd）或 ZAB（ZooKeeper）複製同一份狀態機；只有 leader 處理寫，多數派確認後才提交 → 線性一致。
  2. Client：透過協調服務 acquire 鎖 / 註冊 / 監看；自己不互相協調，全部委託給這個「單一真相來源」。
  3. 下游資源：是 fencing 的執行者——它記住見過的最大 token，過期持有者的 stale 寫入在這一層被擋下。

6. 核心元件深入

租約鎖 vs 永久鎖

永久鎖（持有者不釋放就永遠鎖著）在分散式下是死局：持有者當機 → 鎖永遠卡死。 因此分散式鎖一律是租約（lease）：鎖帶 TTL，持有者要週期性 renew 續租； 一旦漏續租（當機 / 卡住），租約到期鎖自動釋放，他人可接手。代價是引入了下面這個經典不安全場景。

⭐ 為什麼一定要 fencing token？（本題最重要考點）

經典不安全場景（Martin Kleppmann 的圖）
  1. Client A 取得鎖（token=33），準備寫資源。
  2. A 發生 STW GC 暫停（或被換出 CPU）卡了好幾秒——它不知道自己卡住了。
  3. A 的租約在這期間過期，協調服務把鎖釋放。
  4. Client B 取得鎖（token=34），寫入資源。
  5. A 終於醒來，以為自己還持有鎖，拿著 token=33 也來寫資源 → 若無防護就覆蓋了 B 的寫入，資料壞掉。
解法：資源端記住「見過的最大 token」（已是 34），A 帶 token=33 來寫 → 33 < 34 → 拒絕。 這就是 fencing：即使鎖會誤判、時鐘不可信、GC 暫停無法預測，下游仍然安全。

關鍵點：fencing token 必須由協調服務在授予鎖時單調遞增地配發，且下游資源必須有能力檢查並拒絕舊 token。 只有「拿到鎖就放心寫」而沒有 fence，這把鎖只能用在「效率型」場景，不能用在「正確性型」場景。

watch 機制

client 監看某 key（鎖 / 設定 / 服務名冊），值一變更，協調服務主動推事件，避免 client 忙輪詢（busy polling）。 典型用法：落選者 watch 現任 leader 的 key，leader 租約一過期 → 收到 deleted 事件 → 立刻發起競選。

ephemeral node：選主與服務註冊的共同底層

ZooKeeper 的 ephemeral（臨時）節點與 session 綁定：session 一斷（心跳停），節點自動刪除。 服務註冊就是建一個 ephemeral node（實例掛了自動下線）；選主就是大家搶建同一個 ephemeral node，建到的就是 leader（建不到的 watch 它）。 etcd 用 lease + key 綁定達到等價效果。

心跳與 session 過期

TTL 設定是個取捨：太短→ 一點網路抖動就誤判掉線、頻繁換主、心跳壓力大；太長→ 真掉線時偵測慢、鎖卡很久。常見 5～30 秒。

Redlock 爭議（必知）

Redis 作者提的 Redlock（向 N 個獨立 Redis 多數派取鎖）被 Kleppmann 批評：它依賴時鐘不漂移與無長暫停的假設， 在 GC / 時鐘跳變下仍可能讓兩個 client 同時持鎖，且本身不提供 fencing token。 結論：要「正確性」用 etcd / ZooKeeper（CP）+ fencing token；Redis 鎖只適合「效率型」場景。

7. 擴展與瓶頸

瓶頸成因對策
協調服務寫吞吐每筆寫要 Raft 多數決 + fsync別把它當資料庫；只放關鍵元資料；批次 / 合併寫；分多個 namespace 叢集
watch 風暴一個熱門 key 變更要推給數萬 watcher事件批次化、分層代理（proxy 扇出）、client 端去抖動
herd effect（驚群）leader 掉線 → 所有落選者同時搶鎖ZooKeeper「有序排隊」：每人只 watch 排在自己前一位的節點，避免全體驚群
讀放大大量 client 反覆查名冊 / 設定follower 提供唯讀（線性讀需 readIndex）、client 端快取 + watch 失效
租約風暴數十萬實例各自一個短 TTL lease 心跳多實例共用一個 session/lease、拉長 TTL

8. 取捨與面試重點

  • 鎖安全性 vs 效能：每次寫都檢查 fencing token 有成本，但這是「正確性型」鎖的不可省步驟；「效率型」可放寬。
  • CP 協調 vs AP：鎖 / 選主必須 CP（etcd/ZK/Consul），分割時寧可不可用也不能腦裂；若用 AP 系統（如純 Redis）做鎖，必須接受可能雙持鎖並靠 fencing 補救。
  • 自旋（spin）vs 阻塞 watch：拿不到鎖時不要忙輪詢（浪費 + 驚群），改用 watch 在前一位釋放時被喚醒。
  • 租約 TTL 長短：短→快偵測但易誤判；長→穩定但故障恢復慢。依「可容忍多久卡住」決定。
  • 常見追問：協調服務自己掛了怎麼辦？→ 奇數節點多數決容錯 + 它掛了應「fail-safe」（拿不到鎖就不動，而非硬幹）。為何不直接用資料庫的 SELECT ... FOR UPDATE？→ 可以，但缺租約自動釋放與 watch，且把協調壓力壓在主庫上。

9. PHP Demo 說明

本題 demo 用單機檔案模擬一個 etcd / ZooKeeper 風格的協調服務，所有協調原語邏輯為真、可實際執行：

  • src/CoordService.php　協調服務本體：租約鎖 acquire/release/renew、fencing token 配發、fenceWrite 下游防護、watch 佇列、leader election、config、服務 register/discover。所有操作都在一次 flock(LOCK_EX) 臨界區內讀-改-寫狀態檔，模擬「線性一致的複製狀態機」。
  • public/index.php　路由 + 測試頁（六個區塊對應六種原語）。
# 啟動
cd demo
php -S localhost:8036 -t public
# 瀏覽器開 http://localhost:8036
誠實聲明 這是單機模擬的協調服務：用一個狀態檔 + flock 模擬「被 Raft 複製、線性一致的單一狀態機」， 沒有真正的多節點共識 / 網路 / 故障容錯。但租約鎖、fencing token 防護、watch、leader election、服務發現的協調邏輯都是真實可執行的—— 尤其可實際演示「舊 token 被 fence 拒絕」這個分散式鎖最重要的考點。要做成真叢集，把狀態機換成 etcd/ZooKeeper 的 client 即可，上層原語不變。
