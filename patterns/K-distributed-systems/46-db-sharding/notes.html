㊻ 資料庫分片：分庫分表 × scatter-gather × 全域 ID × re-shard

K 分散式系統 建議 55 分鐘 Design Database Sharding (Vitess)

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用「圖書館分館、找書 vs 數全館的書、館員號碼牌、開新分館要搬書」的比喻把核心講清楚，再回來看這份工程速查。
一句話定位 本質是「單機關聯式資料庫（MySQL）裝不下 / 扛不住，把一張大表水平切到多台」——也就是 Vitess / ProxySQL / Citus 那一類。 面試真正考的不是「怎麼切」，而是切完之後冒出的四個痛： 分片鍵選擇與熱點、沒帶分片鍵的查詢只能 scatter-gather（跨片 JOIN / 聚合很貴）、 跨片不能用單一 auto_increment（全域 ID）、加機器要搬資料（re-shard）。
與第 ㉛ 題（分散式 KV）的差異——別答混了 ㉛ 是 NoSQL KV：用一致性雜湊環 + vnode，天生無 JOIN / 無交易，加機器只搬 ~1/N。 本題是 關聯式分片：保留 SQL（JOIN、交易、二級索引、ORDER BY LIMIT），所以痛點完全不同—— 取模分片加機器幾乎全搬、沒帶分片鍵就退化成 scatter-gather、自增主鍵跨片會撞號。 面試官常用這題測你「有沒有意識到關聯式分片放棄了單機 SQL 的哪些好處」。

1. 需求釐清

功能性需求

  • 把單一邏輯大表（如 orders、users）水平切到 N 台 MySQL，對應用盡量透明。
  • 帶分片鍵的查詢（如 WHERE user_id=?）能直達單一分片。
  • 支援不帶分片鍵的查詢（範圍、聚合、排序）——退化為跨片合併但仍要能跑。
  • 主鍵全域唯一（跨片不撞號）。
  • 能線上加分片（re-shard）並搬遷資料，不停機。

非功能性需求

  • 容量 / 吞吐線性擴展：加機器就能增加總容量與 QPS。
  • 低延遲：單片查詢 p99 個位數毫秒；避免「每個查詢都打全部片」。
  • 保留關聯式語意：交易、二級索引、JOIN（至少同片內）仍可用。
  • 可運維：搬遷可控（限流、可回退），分片鍵選擇可演進。
面試開場必問：為什麼不先用副本 / 垂直分？ 分片是最後手段，先窮盡便宜招：① 讀副本（讀寫分離，解讀爆）② 垂直分庫（把不同表 / 大欄位拆到不同庫）③ 快取。 只有當單表寫入 / 單庫容量成為瓶頸（一台 MySQL 裝不下、單主寫不動）才水平分片——因為分片會犧牲跨片 JOIN / 跨片交易 / 全域排序，代價很高。

2. 規模估算

指標假設推算
資料量20 億筆訂單 × 每筆 1 KB~2 TB；加索引 ×2 → ~4 TB，單機 MySQL 吃緊
寫 QPS尖峰 8 萬寫/s單主 MySQL ~幾千~上萬寫/s → 必須切多主
分片數每片目標 < 500 GB、< 1 萬寫/s4 TB / 500 GB ≈ 8 片起；預留成長取 16~32 片
讀寫比讀:寫 ≈ 10:1每片再掛 2 讀副本分擔讀（讀寫分離）
分片鍵基數user_id 上億、分佈均勻基數高、均勻 → 適合當分片鍵（避熱點）
關鍵洞察 分片數要一開始就預切足夠多（甚至遠多於機器數，多個 shard 擺一台，之後再搬到新機器）。 因為 re-shard 很痛——預切多 shard 讓「擴容＝搬整個 shard」而非「重算每個 key 落點」，把搬遷量從「幾乎全搬」降成「搬幾個 shard」。

3. API 設計

# 應用面對的仍是「一張邏輯表」，分片中介層負責路由
# 帶分片鍵 → 單片
GET  /orders?user_id=1001            → 直達 shard = hash(1001) % N
POST /orders  { user_id, amount }    → 路由到該片 insert，回傳全域 id

# 不帶分片鍵 → scatter-gather（中介層打所有片再合併）
GET  /orders?status=PAID&sort=amount&limit=20   → 每片各查 top-20 → 合併再取 top-20
GET  /orders/stats?metric=sum_amount            → 每片各算 sum → 相加

# 內部：全域發號器
POST /idgen/next  { table:"orders" }            → 回傳全域唯一 id（號段批發）

# 運維：re-shard
POST /admin/reshard  { from:16, to:32 }         → 線上複製 + 切流（VReplication）
設計重點 API 表面看不出分片，但每個端點都要標註「帶不帶分片鍵」——這決定它是單片直達還是 scatter-gather。 面試時主動指出「這個查詢沒帶分片鍵，會掃所有片，要嘛加二級路由、要嘛用搜尋引擎旁路」是加分點。

4. 資料模型

邏輯表 orders（被水平切到 N 片）
  id          BIGINT  PRIMARY KEY     -- 全域唯一（非各片 auto_increment！）
  user_id     BIGINT                  -- ★ 分片鍵：shard = hash(user_id) % N
  amount      DECIMAL
  status      VARCHAR
  created_at  TIMESTAMP

分片路由（中介層的 vschema / 路由表）
  keyspace: orders_ks
  shard_key: user_id
  策略: hash(user_id) % N   （或：範圍分片 / 目錄分片 directory-based）

全域 ID 三種做法
  ① 號段(segment)：全域發號器批量發「不重疊區段」給各片，本地遞增（高效，Vitess/Leaf）
  ② Snowflake：時間戳 + 機器號 + 序號，趨勢遞增、無中心瓶頸
  ③ 各片 auto_increment 設不同 step/offset（片0:1,1+N；片1:2,2+N…）簡單但加片難調
分片鍵一旦選錯，幾乎要重建 分片鍵決定「資料落哪片」與「哪些查詢能單片直達」。選 user_id：查某用戶的訂單超快，但「查某商品的所有訂單」要 scatter-gather。 沒有完美分片鍵——選最高頻查詢的條件欄位，且基數高、分佈均勻、寫入不集中（避熱點）。改分片鍵＝幾乎重灌，務必慎選。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
                       ┌──────────────────────────┐
 應用 ──(一張邏輯表)──▶ │  分片中介層 (Vitess/ProxySQL) │
                       │  shard = hash(key) % N      │
                       │  帶 key→單片；不帶→scatter   │──┬─▶ MySQL 分片0 (主) ─▶ 讀副本
                       └──────────────────────────┘  ├─▶ MySQL 分片1 (主) ─▶ 讀副本
                                  │                    └─▶ MySQL 分片2…(主) ─▶ 讀副本
                                  ▼
                          全域發號器 (號段 / snowflake)
                          re-shard: 線上複製 + 切流 (VReplication)
  1. 分片中介層：應用只看到「一張表」；中介層依分片鍵算落點、做路由、scatter-gather 合併、跨片排序/聚合的二次計算。
  2. 每片一主多副本：主寫、副本讀（讀寫分離），分擔該片的讀流量。
  3. 全域發號器：保證跨片主鍵唯一。
  4. re-shard 管線：Vitess 用 VReplication 把舊片資料串流複製到新片，追上後切流，無停機。

6. 核心元件深入

① 分片鍵選擇與熱點

  • 準則：選最高頻查詢的等值條件欄位（讓多數查詢單片直達）+ 基數高 + 分佈均勻 + 寫入分散。
  • 熱點：用「時間」當分片鍵 → 最新那片被狂寫（時間熱點）；用「大客戶 id」→ 該片被打爆（資料傾斜）。對策：複合分片鍵（如 hash(user_id, order_id)）、加鹽打散大客戶、或對熱點再切。
  • 取模 vs 範圍 vs 目錄：取模均勻但 re-shard 痛、不支援範圍查；範圍支援範圍查但易熱點；目錄（查路由表）最靈活但多一跳 + 路由表自己要高可用。

② 跨片查詢 scatter-gather 與 JOIN 之痛

  • scatter-gather：沒帶分片鍵的查詢 → 中介層把同一條 SQL 廣播到所有片，各片回部分結果，中介層合併。延遲 = 最慢那片（長尾放大）。
  • ORDER BY ... LIMIT k：不能只各片取 k 就好——要各片取 k、彙整後再排序取 k（中介層二次排序）；OFFSET 大時更糟（要拉回大量列）。
  • 聚合：COUNT/SUM 可下推到各片再相加；COUNT(DISTINCT)、AVG、GROUP BY 跨片要小心（AVG 要拆成 sum/count 分別下推）。
  • 跨片 JOIN 之痛：兩張表分片鍵不同 → JOIN 要跨片拉資料到中介層做（昂貴）。對策：同分片鍵共置（co-location）讓相關表落同片可本地 JOIN；或反正規化把要 JOIN 的欄位冗餘進來；或對小表做全片複製（reference table）。

③ 分散式自增 ID（全域唯一）

  • 問題：各片各自 auto_increment 會撞號（兩片都產生 id=1）。
  • 號段(segment)：全域發號器一次批發一段（如 1~1000）給某片，片內本地遞增；用完再領下一段。避免每筆寫都搶全域計數器（那會變熱點），是 Vitess/美團 Leaf 的主流做法。
  • Snowflake：時間戳(41b) + 機器號(10b) + 序號(12b)，趨勢遞增、無中心瓶頸，但依賴時鐘（時鐘回撥要處理）。
  • 權衡：號段更緊湊、可去重；snowflake 無中心、含時間資訊但較長且可被推測產生速率。

④ re-shard 搬遷

  • 取模分片的致命傷：N → N+1 時幾乎每個 key 的 hash % N 都變 → 大部分資料要換家（本題 demo 可實測到接近 100%）。
  • 對策一：預切多 shard：一開始就切成 32/64 個邏輯 shard（多個擺一台），擴容＝把整個 shard 搬到新機器（shard 數不變、key 落點不變），搬遷量只是「幾個 shard」而非「重算所有 key」。
  • 對策二：一致性雜湊 / 範圍分片：加機器只影響相鄰一段（~1/N），如同第 ㉛ 題。
  • 線上搬遷：Vitess VReplication——新片串流複製舊片資料 + 追 binlog 增量，追上後切讀、再切寫，雙寫驗證一致再下線舊路由，全程不停機、可回退。

⑤ 讀寫分離與副本

  • 每片一主多從：寫走主、讀走副本，分擔讀流量。
  • 複製延遲：副本落後主 → 可能讀到舊值。需要「讀自己的寫」時，把該讀強制走主（read-your-writes），或用 GTID 等副本追上再讀。
  • 中介層可做讀流量負載均衡與副本健康剔除。

7. 擴展與瓶頸

瓶頸症狀對策
熱點分片某片被狂讀/寫（大客戶、時間熱點）複合分片鍵 / 加鹽打散 / 熱點再切 / 前置快取
跨片查詢爆大量不帶分片鍵的查詢做 scatter-gather共置相關表、反正規化、二級路由表、搜尋引擎(ES)旁路做檢索
跨片交易一筆交易跨多片，要 2PC（慢且易卡）盡量讓交易落單片（用同分片鍵共置）；真要跨片用 Saga / 最終一致
re-shard 成本擴容要搬大量資料、影響線上預切多 shard + 線上 VReplication 限流搬遷 + 可回退
跨片排序分頁ORDER BY + 大 OFFSET 拉回海量列改游標分頁(seek/keyset)、限制深分頁、把排序欄位下推
分散式唯一約束跨片的 UNIQUE / 外鍵無法強制把唯一欄位也當分片鍵、或用獨立去重服務

8. 取捨與面試重點

  • 分片鍵選錯很痛：它同時決定「落點」與「哪些查詢單片直達」，改它幾乎要重灌。先盤點最高頻查詢，選能讓它們單片直達且分佈均勻的鍵——這是這題最大的得分點。
  • 垂直 vs 水平分：垂直分（不同表/欄位拆庫）先做、便宜、不犧牲單表 SQL；水平分（同表切多片）是擴展上限但犧牲跨片 JOIN/交易。順序：副本 → 垂直 → 水平。
  • 範圍 vs 雜湊分片：雜湊均勻、避熱點，但不支援範圍查、re-shard 痛；範圍支援範圍查與「就近搬」，但易產生時間/字母熱點。常見折衷：雜湊 + 預切多 shard。
  • 分片 = 放棄了什麼：跨片 JOIN、跨片交易、全域唯一約束、全域 ORDER BY/AVG/DISTINCT 都變貴或不可用。能講清楚「放棄了什麼、怎麼補（共置/反正規化/Saga/ES）」才算真懂。
  • 常見追問：怎麼不停機 re-shard？（VReplication 複製+切流）跨片唯一怎麼辦？（唯一欄當分片鍵 / 去重服務）熱點大客戶？（加鹽 / 單獨成片）跨片分頁？（keyset 游標）。

9. PHP Demo 說明

本題 demo 在單一 PHP process 內，用「每個分片一個 JSON 檔」假裝成一台獨立 MySQL，把關聯式分片真正的考點跑起來：

  • src/ShardRouter.php　分片核心：shardOf()＝crc32(key) % N 路由、insert/getByKey 單片直達、aggregate() scatter-gather（count/sum/top 跨片合併排序）、nextId() 號段全域 ID、addShard() re-shard 並算出哪些 key 換家、搬遷比例。
  • public/index.php　路由 + 測試頁：設分片數、插入看落哪片、依 key 單片點查、跨片聚合、加分片看 re-shard。
# 啟動
cd demo
php -S localhost:8046 -t public
# 瀏覽器開 http://localhost:8046
# 試：一鍵塞 12 筆 → 依 key 點查(shards_scanned=1) → 跨片聚合(shards_scanned=N)
#     → 加 1 片看 moved_pct 接近 100%（取模分片 re-shard 之痛）
誠實聲明 Demo 是單一 process、JSON 檔模擬多台 MySQL：沒有真實 MySQL、沒有跨片 2PC 分散式交易、沒有線上不停機搬遷（VReplication）。 但取模分片路由、scatter-gather 跨片聚合、號段全域 ID、re-shard 換家計算與搬遷的邏輯皆為真實可執行——這正是關聯式分片的核心考點。 要變成真系統，把「各分片一個 JSON 檔」換成「各分片一台 MySQL + 分片中介層」即可，路由與合併演算法不變。僅用 json + hash(crc32)，每檔 declare(strict_types=1);，以 flock 原子化。
