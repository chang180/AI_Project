1. 60 秒搞懂它在幹嘛

想像一間超大圖書館，書多到一棟樓塞不下，櫃台也忙到排隊排到門口。怎麼辦？ 蓋好幾間分館，把書分散放——這就是資料庫分片（Sharding）：把原本擠在「一台資料庫」裡的資料，水平切開、分散到很多台。

聽起來只是「多買幾台機器」對吧？難的是：書一旦分散到好幾間分館，「怎麼快速找到某本書在哪間館」、「要統計全館總共幾本書」、「每本書怎麼編一個全館不重複的編號」、以及「開新分館時要搬多少書」——這四件事，全都變麻煩了。

🔑 一句話 分片 = 一台資料庫裝不下/忙不過來時，把資料切開分到很多台。 好處是「能無限長大」，代價是「跨館的事情都變貴」（找遍全館、統計全館、全館不撞號、開館要搬書）。

2. 為什麼難點不是「把資料存起來」？

很多人第一次看會想：「不就是多開幾個資料庫，把資料丟進去嗎？」——這想法漏掉了「切開之後」才冒出來的麻煩。

📌 比喻：連鎖書店 vs 一間大店 一間大店時，店員要找書、要盤點、要編書號，在同一個地方做都很簡單。 一旦變成好幾間分店：你問「《哈利波特》在哪？」如果事先約好「H 開頭的書都在 3 號店」，一通電話就找到（快）； 但你問「全公司總共幾本書？」就得打給每一間店各自盤點、再加起來（慢）。

這就是分片的核心難點：知道「東西放哪間」就快；不知道、要問遍全部就慢。而且每間店各自編書號會撞號（兩間店都有「1 號書」），開新店還得把舊店的書搬一部分過去。

💭 停一下：為什麼「問全公司總共幾本書」會比「問某本書在哪」慢這麼多？

看答案 因為「某本書在哪」事先用書名約好了規則（H 開頭→3 號店），一步直達一間店； 而「總共幾本」沒有任何線索能縮小範圍，只能每一間店都問一遍、再把答案加起來——店越多越慢，還要等最慢的那間回覆。這就是後面要講的 scatter-gather。

3. 把系統拆成 4 個關卡

分片要處理的，其實就是這 4 關，一關一關來：

1路由：這筆資料該放哪一台？

資料進來，要先決定放第幾台。做法是挑一個欄位當分片鍵（例如「使用者編號」），算一個數字、除以台數取餘數，餘數幾就放第幾台。

🗂️ 比喻：依姓氏分櫃 公司收信，依「姓氏第一個字」決定丟哪個信箱。下次要找王先生的信，直接走到「王」那一櫃，不用翻遍所有信箱。 分片鍵就是這個「姓氏」——選對了，多數查詢一步直達。

👉 面試最愛問：「分片鍵要選哪個欄位？」因為它同時決定「資料放哪」和「哪些查詢會快」。

2找遍全部：沒有線索的查詢（scatter-gather）

如果查詢沒帶分片鍵（例如「全公司總共幾筆訂單？」「金額前 10 大的訂單？」），就沒辦法一步直達，只能問遍每一台、再把答案合起來。

📞 比喻：打給每間分店再加總 「全公司總共幾本書？」→ 打給每一間店各自盤點 →（scatter，撒出去）→ 收回每間的數字 →（gather，收回來）→ 加總。 要排「全公司最貴的前 10 本書」更麻煩：每間先報自己的前 10，你再把這些湊起來重新排一次取前 10。

這個「撒出去問所有台、再收回來合併」就叫 scatter-gather。它慢，而且要等最慢的那台。所以好的分片設計，會盡量讓查詢都帶分片鍵、走第 1 關的「直達」。

3全域編號：別兩台都叫「1 號」

每筆資料要有一個不重複的編號（主鍵）。但如果每台各自從 1 開始編，第 0 台和第 1 台就都會有「1 號」——撞號了！

🎫 比喻：銀行抽號碼牌 銀行不會讓每個櫃台各自從 1 號發牌（會一堆人都拿到「3 號」）。而是有一台總發號機： 「1~1000 號給 A 櫃、1001~2000 給 B 櫃」，各櫃在自己的號段裡發，全行永遠不撞號。

系統也一樣：用一個全域發號器，把號碼切成不重疊的「號段」分配給各台，各台在自己的號段裡遞增。這樣全系統的編號保證唯一。

4開新分館：加一台要搬多少書？

資料又長大了，要再加一台。問題來了：原本「除以 2 取餘數」的規則，變成「除以 3 取餘數」——幾乎每筆資料算出來的台號都變了，意思是幾乎全部都要搬家！

🚚 比喻：改分櫃規則就要大搬家 原本「姓氏除以 2」分兩櫃，現在改「除以 3」分三櫃——絕大多數信的櫃號都變了，等於要把信幾乎全部重搬一次。累死人。

這就是取模分片（除以台數取餘數）最大的痛：加機器＝大搬家。工程上的解法：一開始就先切很多「邏輯小櫃」（例如先切 32 櫃，雖然只有 2 台機器，一台先放 16 櫃），擴容時整櫃搬到新機器就好，規則不變、不用重算每封信。這個「重新切、搬資料」的動作就叫 re-shard。

✅ 四關一句話 ① 用分片鍵路由（這筆放哪台）→ ② 沒線索就找遍全部再合併（scatter-gather）→ ③ 全域發號（別撞號）→ ④ 加機器要搬家（re-shard，取模分片幾乎全搬）。就這樣。

4. 設計時最怕的 3 件事

把取捨換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「分片鍵選錯」 分片鍵一旦選定，資料就照它分散了。選錯（例如多數查詢其實是用另一個欄位查）→ 大量查詢都得找遍全部（scatter-gather），慢到爆。 而且改分片鍵幾乎等於整個重灌搬家。所以選之前要先想清楚「最常用什麼欄位查」。選對鍵，半題就解了。
😱 怕「跨片的事情」 跨好幾台的查詢（找遍全部、全域排序、跨台 JOIN、跨台轉帳交易）又慢又難保證正確。 所以設計時要盡量讓相關的資料落在同一台（用同一個分片鍵），能在一台內搞定就別跨台。能不跨台，就不跨台。
😱 怕「熱點：某一台被打爆」 如果分片鍵分得不均（例如用「日期」當鍵 → 大家都在寫「今天」那一台；或某個大客戶資料特別多 → 他那台特別累），就會一台累死、其他台閒著。 對策：選分佈均勻的分片鍵、把大客戶打散。雨露均霑，別偏心。

6. 名詞白話翻譯

看完上面，這些詞你其實都已經懂了，這裡只是把「工程術語 ↔ 白話」對起來，Part 2 會用到：

工程術語白話解釋
分片 / Sharding一台裝不下時，把資料水平切開分到很多台
分片鍵 (shard key)決定「這筆放哪台」的那個欄位（像分櫃的姓氏）
取模分片 (hash % N)算個數字除以台數取餘數，餘數幾就放第幾台
路由依分片鍵算出「該去哪一台」的動作
單片直達帶分片鍵的查詢，一步走到一台（最快）
scatter-gather沒帶分片鍵 → 撒給所有台問、再收回來合併（慢，等最慢那台）
跨片 JOIN / 交易要橫跨好幾台才能完成的關聯/轉帳，又慢又難保證對
全域 ID / 發號器全系統不重複的編號；用「號段」切給各台各自發
號段 (segment)把號碼切成不重疊的區間（1~1000 給A、1001~2000 給B）
re-shard加機器時重新切、搬資料；取模分片幾乎要全搬
熱點 / 傾斜某一台被打爆（分片鍵分不均，如「日期」或大客戶）
讀寫分離 / 副本寫走主、讀走副本，分擔讀流量
分片中介層夾在應用和多台 MySQL 之間的「總機」(Vitess/ProxySQL)，負責路由與合併
垂直分 vs 水平分垂直＝不同表/欄位拆庫；水平＝同一張表切多片

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要切幾台。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
資料量20 億筆 × 1KB ≈ 2TB（加索引再 ×2）一台 MySQL 吃緊 → 要切多台
每秒寫入尖峰 8 萬寫/秒單台主庫寫不動（頂多幾千~上萬）→ 要切多主
切幾片每片目標 <500GB、<1萬寫/秒大約 8 片起跳，預留成長切 16~32 片
讀寫比讀:寫 ≈ 10:1每片再掛 2 台讀副本分擔讀
💡 關鍵洞察：一開始就「多切一點」 與其之後痛苦搬家，不如一開始就切很多邏輯小片（如 32 片，先幾片擺一台機器）。 之後擴容只要把整片搬到新機器，規則不變、不用重算每筆資料的落點——把「加機器＝大搬家」變成「加機器＝搬幾片」。

💭 停一下：明明只有 2 台機器，為什麼要先切成 32 片？

看答案 因為「片數」一變，每筆資料的落點規則就全變了（要大搬家）。但如果片數固定是 32、只是改變「哪幾片擺哪台機器」， 那擴容時只要把某幾片整片搬去新機器，每筆資料還是屬於原本那片、落點規則沒變——搬遷量小很多。先多切片＝預留擴容空間。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」。這題的重點是：每張點餐單有沒有寫分片鍵，決定它快還是慢。 
① 帶分片鍵 → 單片直達（快）
   GET /orders?user_id=1001
   有 user_id（分片鍵）→ 一步走到那一台 → 超快

② 沒帶分片鍵 → 找遍全部（慢）
   GET /orders?status=PAID&sort=amount&limit=20
   沒有分片鍵 → 撒給所有台 → 各自查 → 收回來重新排序取前 20

看出差別了嗎？同樣是查訂單，帶不帶 user_id 速度天差地遠。設計 API 時要心裡有數：哪些是單片、哪些會 scatter-gather。

③ 寫入 → 路由到對的台 + 拿全域編號
   POST /orders { user_id:1001, amount:120 }
   → 算 user_id 落哪台 → 跟發號器要一個全域不重複的 id → 寫進去

④ 運維：加一台分片（re-shard）
   POST /admin/reshard { from:16, to:32 }
   → 在背景把資料慢慢複製到新片，追上後再切換（不停機）

9. 系統要記住哪些事？（資料模型）

📒 比喻：每間分館一本帳，外加一張「總規則表」 每台 MySQL 各自存自己那份資料；另外要有一張「規則表」記住「用哪個欄位當分片鍵、現在切幾片」。

🗄️被切開的大表（每台各存一份）

例如訂單表：id（全域不重複）、user_id（分片鍵）、金額、狀態、時間。 同一個 user_id 的訂單都落在同一台，所以「查某人的訂單」一步直達。

🧭路由規則表（中介層的腦袋）

記住：分片鍵是哪個欄位、用什麼規則（除以幾取餘數）、目前切幾片。 中介層每次拿到查詢，就翻這張表決定「直達哪台」或「要不要找遍全部」。

🎫全域發號的「水位表」

記住每台目前領到的號段用到哪了（像「A 櫃發到 1538 號了」）。 用完一段就再領下一段，保證全系統永不撞號。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
某一台被打爆（熱點 / 大客戶）選分佈均勻的分片鍵；把大客戶加鹽打散或單獨成一片
太多查詢沒帶分片鍵，一直找遍全部讓相關資料落同一台、把欄位冗餘進來、或交給搜尋引擎做檢索
跨好幾台的轉帳交易盡量讓交易落在同一台；真要跨台用「分步驟補償」(Saga)
加機器要搬好多資料一開始多切片 + 背景限速搬遷 + 可隨時退回
讀太多打爆主庫讀走副本（讀寫分離），每台多掛幾台讀副本

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 先做哪一步？副本 → 垂直分 → 水平分片 分片很痛，是最後手段。先加讀副本（解讀爆）、再垂直分（不同表拆到不同庫，便宜又不犧牲 SQL），真的還不夠才水平分片。
⚖️ 取餘數分 vs 照範圍分 取餘數（hash）分得均勻、不易熱點，但不能做範圍查（「3 月所有訂單」會散在各台）、而且加機器要大搬家； 照範圍分（如按日期）能範圍查、搬家方便，但容易熱點（大家都寫今天那台）。常見折衷：取餘數 + 一開始多切片。
⚖️ 全域編號用「號段」還是「Snowflake」？ 號段（總發號機批號段）緊湊、好管，但發號機要夠穩；Snowflake（時間+機器號+序號）沒有中心、各台自己產，但編號較長、靠時鐘。看你怕「中心掛掉」還是怕「編號太長」。
💭 追問：怎麼「不停機」加一台分片？ 背景先把要搬的資料慢慢複製到新片，同時持續追上後續的新變動；等新片資料追平了，先把讀切過去、再把寫切過去， 確認一致後才把舊路由下線。全程服務不中斷，出問題還能切回去。（Vitess 的 VReplication 就是這樣做的。）
💭 追問：分片鍵選 user_id，那「查某商品的所有訂單」怎麼辦？ 這查詢沒帶 user_id（分片鍵）→ 只能 scatter-gather 找遍全部，慢。解法：另外建一張「商品→訂單」的對照索引（用商品當鍵存一份）、 或把這類檢索丟給搜尋引擎（如 Elasticsearch）處理，不要硬用分片庫做。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「路由 + 找遍全部 + 全域編號 + 搬家」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8046 -t public
# 然後瀏覽器打開 http://localhost:8046
  1. 先按「一鍵塞 12 筆樣本」，看每筆資料被算到哪一片（routed_to）。
  2. 依 key 點查：輸入某個 shard_key 查詢，看 shards_scanned=1（只打一台，超快）。
  3. 跨片聚合：按 count / sum / top，看 shards_scanned=N（打了所有片再合併）。
  4. 加 1 片：按下去，看 moved_pct——你會看到接近 100% 的資料要換家！這就是取模分片 re-shard 的痛。
玩的時候想一下對比「依 key 點查只打 1 片」和「跨片聚合打 N 片」——這就是「帶不帶分片鍵」的天壤之別。再看加片時的 moved_pct，體會為什麼工程師要「一開始就多切片」來避開大搬家。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心都在一個小檔 ShardRouter.php 裡，剛好對應前面講的四關。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 一個檔 = 四個小幫手 路由（算這筆放哪台）→ 單片點查（帶鍵只打一台）→ scatter-gather（沒帶鍵打全部再合）→ 發號 + 搬家（全域編號、加片搬資料）。

1路由：算這筆該放哪台（shardOf）

核心就一行：把分片鍵算成數字，除以台數取餘數。

function 算落哪片(分片鍵) {
    return crc32(分片鍵) % 台數;   // 餘數幾 → 放第幾台
}
// 插入時：先算落哪片 → 跟發號器要一個全域 id → 寫進那一片

就這一行 % 台數，決定了每筆資料的家。簡單、均勻——但也正因為「台數」在分母，台數一變，幾乎每筆的餘數都變，這就埋下了第 4 關「大搬家」的伏筆。

2單片點查：帶分片鍵，只打一台（getByKey）

查詢帶了分片鍵，就先算出落哪片，只翻那一片：

function 依鍵查詢(分片鍵) {
    片 = 算落哪片(分片鍵);       // 一步算出在哪台
    回傳 該片裡 shard_key 等於它的所有列;
    // shards_scanned = 1 ← 只碰一台！
}

這就是「走到『王』那一櫃」——只打一台，最快。回傳的 shards_scanned=1 就是證據。

3找遍全部：沒帶鍵，打全部再合併（aggregate）

這是關聯式分片最痛的一段。沒有分片鍵當線索，只能每一片都跑一次、再把結果合起來：

function 跨片聚合(操作) {
    對 每一片 s {                  // ← scatter：撒給所有片
        rows = 讀第 s 片;
        累加 count、累加 sum、收集所有列;
    }
    // ← gather：把各片的部分結果合成最終答案
    if (操作 == 'sum')  return 各片 sum 相加;
    if (操作 == 'top')  { 把所有列重新排序; 取前 k; }   // 中介層二次排序
    note = "打了全部 N 片，延遲取決於最慢那片";
}

注意 對 每一片 這個迴圈——它就是 scatter（撒出去問每一台）。最後的相加 / 重新排序就是 gather（收回來合併）。 台數越多越慢，還要等最慢那台。這就是為什麼設計時要拼命讓查詢「帶分片鍵」走第 2 關。

4全域編號 + 加片搬家（nextId / addShard）

編號用「號段」讓各片不撞號；加片時老老實實算「誰要換家」：

// 全域編號：第 s 片的號 = s × 號段 + 該片本地序號 → 跨片永不重複
function 發號(片 s) { return s × 號段大小 + 該片下一個序號++; }

// 加一台片：N → N+1，幾乎每筆都要重算落點
function 加一片() {
    對 每一筆資料 {
        新片 = crc32(分片鍵) % (N+1);    // 用新台數重算
        if (新片 != 舊片) 記下「這筆要換家：舊片→新片」;
    }
    回報 換家比例;   // 取模分片 → 往往接近 100%！
}

看到 % (N+1) 了嗎？分母從 N 變 N+1，餘數幾乎全變 → 幾乎全部換家。 這段就是 demo 裡讓你看到 moved_pct 接近 100% 的真相，也是「為什麼要一開始多切片」最有力的證據。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「連鎖書店」解釋，分片的核心難點是什麼？ 一間大店變很多分店後，「某本書在哪」如果事先用書名約好規則就能一步直達（快）；但「全公司總共幾本書」沒線索，只能打給每間店各自盤點再加總（慢）。難點就是「跨店的事情都變貴」。
Q2：分片鍵是什麼？為什麼選錯很痛？ 分片鍵是決定「這筆資料放哪台」的欄位。它同時決定「哪些查詢能一步直達」。選錯（多數查詢其實用別的欄位查）→ 大量查詢要找遍全部（scatter-gather），而且改分片鍵幾乎要整個重灌搬家。
Q3：scatter-gather 是什麼？為什麼慢？ 沒帶分片鍵的查詢，只能撒給所有台各自查（scatter）、再收回來合併（gather）。慢的原因：台數越多要問越多台，而且要等最慢的那台回覆（長尾）。全域排序還要在中介層二次排序。
Q4：每台各自從 1 開始編號會怎樣？怎麼解決？ 會撞號（兩台都有「1 號」）。解法是全域發號器用「號段」把號碼切成不重疊的區間分給各台（1~1000 給A、1001~2000 給B），各台在自己區間裡遞增，全系統不撞號。
Q5：為什麼「取模分片」加一台機器會幾乎全搬家？怎麼緩解？ 因為落點規則是「除以台數取餘數」，台數從 N 變 N+1 時，幾乎每筆資料算出來的台號都變了 → 幾乎全部要搬。緩解：一開始就多切邏輯片（如 32 片），擴容時整片搬到新機器，規則與落點不變。
Q6（工程）：分片是不是第一步就該做？順序是什麼？ 不是，分片很痛是最後手段。順序：先加讀副本（解讀爆）→ 垂直分（不同表拆庫，便宜又保留 SQL）→ 真的不夠才水平分片（犧牲跨片 JOIN/交易換無限擴展）。
Q7（工程）：分片鍵選 user_id，「查某商品的所有訂單」怎麼快？ 那查詢沒帶 user_id 會 scatter-gather。解法：另外維護一張「商品→訂單」的對照索引（用商品當鍵），或把這類檢索交給搜尋引擎（Elasticsearch）做，不硬用分片庫。

15. 30 秒回顧

資料庫分片 = 一台裝不下/忙不過來時，把資料水平切到很多台。

4 個關卡：① 用分片鍵路由（放哪台）→ ② 沒線索就找遍全部再合併（scatter-gather）→ ③ 全域發號（號段，別撞號）→ ④ 加機器要搬家（re-shard，取模分片幾乎全搬）。

3 個怕：怕分片鍵選錯（改它＝重灌）、怕跨片的事（JOIN/交易/排序都貴，能不跨就不跨）、怕熱點（某台被打爆，要分得均勻+打散大客戶）。

工程細節一句話：先副本→垂直分→才水平分片；分片鍵選最高頻查詢的均勻欄位；一開始多切片避開大搬家；跨片查詢能共置就共置、不行就交給搜尋引擎；全域編號用號段或 snowflake；擴容用線上複製+切流不停機。
