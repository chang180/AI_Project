一句話定位 本質是「跨多個分片 / 多個服務的『一次全做或全不做』強一致交易」——也就是把單機 DB 的 ACID 原子性延伸到分散式環境。 面試真正考的不是「怎麼下 SQL」，而是三件事如何咬合： 兩階段提交（prepare/commit 怎麼保證原子）、阻塞與協調者單點（2PC 的致命傷與緩解）、 2PC vs Saga（強一致 vs 最終一致補償）怎麼取捨。

1. 需求釐清

功能性需求

  • 跨分片 / 跨服務的原子交易：一筆業務操作橫跨多個資料節點（如「A 分片扣款、B 分片入帳」），必須全部成功或全部失敗，不能只做一半。
  • 強一致：交易提交後，所有參與者立即看到一致結果（沒有「A 扣了但 B 沒收到」的中間態被外部觀察到）。
  • 隔離性：並發交易彼此不互相污染（典型用鎖或 MVCC）。
  • 持久性：提交後即使崩潰也不丟（預寫日誌 WAL）。

非功能性需求

  • 正確性優先：金流類場景寧可拒絕交易，也不能讓錢憑空增減。
  • 可接受的延遲：強一致交易天生比單機慢（多一輪網路往返），要能量化代價。
  • 故障可恢復：協調者或參與者崩潰後，系統能恢復到一致狀態（不能永久卡住）。
面試開場必問：要強一致(2PC) 還是最終一致(Saga)？ 這決定整題走向。2PC（本題主軸）給你真正的原子性 + 隔離性，代價是提交期間鎖定、阻塞、協調者單點，且難跨組織。 Saga（㊶ 工作流／補償式）把長交易拆成一串本地交易，失敗時用補償動作反向回滾，可用性高、不長鎖，但只有最終一致、會有中間態被看到、要自己寫補償邏輯。 金額帳務、庫存扣減常用 2PC / NewSQL；跨微服務的長流程（訂單→付款→出貨）常用 Saga。務必先確認。

2. 規模估算

指標假設推算 / 含義
跨分片交易佔比多數交易落在單分片單分片走本地交易即可；只有少數跨分片才走 2PC——設計上要盡量讓資料同分片（依使用者/帳戶 colocate）
一筆 2PC 往返prepare 1 RTT + commit 1 RTT≈ 2 倍單機交易延遲（同區內 RTT ~0.5ms → 多 ~1ms；跨區則放大）
鎖定持有時間從 prepare 到 commit/abort= 兩階段全程；協調者越慢、參與者越多 → 鎖越久、阻塞越嚴重
參與者數每筆交易 N 個分片任一參與者掛掉都可能讓整筆阻塞；N 越大失敗機率越高 → 盡量小
協調者日誌寫每筆交易 ≥1 次決議落盤決議日誌是恢復的命脈，必須先落盤再通知 → 多一次 fsync 成本
關鍵洞察 分散式交易的成本不在「算」，在「協調的往返 + 提交期間的鎖定」。所以第一原則是 能不跨分片就不跨（資料 colocation）；真的要跨，才用 2PC，並盡力縮短「prepare→commit」這段鎖定窗口。

3. API 設計

# 對外：一筆跨分片業務交易（協調者內部展開成 2PC）
POST /api/transfer
Body: { "from": "A:alice", "to": "B:bob", "amount": 200 }
→ 200 { "txId": "tx_ab12", "decision": "COMMIT",
        "phase1": [{"shard":"A","vote":"YES"},{"shard":"B","vote":"YES"}] }
→ 200 { "txId": "tx_cd34", "decision": "ABORT", ... }    # 任一投 NO → 全回滾

# 協調者 ↔ 參與者（內部 RPC，2PC 兩階段）
prepare(txId, ops)  → vote: YES | NO     # 第一階段：能不能提交?（預寫+鎖定）
commit(txId)        → ok                  # 第二階段：真的提交
abort(txId)         → ok                  # 第二階段：丟棄預寫、回滾

# 查交易狀態 / 恢復用
GET /api/state      → 各參與者餘額、鎖定、總額、協調者決議日誌
為何要有獨立的 prepare 動作 prepare 是 2PC 的精髓：它讓參與者承諾「我能提交，且把資源預留好了」但還沒真的提交。 協調者收齊所有承諾後，才有資格做出「全 commit」的最終決定——這就是原子性的來源。

4. 資料模型

協調者（Transaction Manager）
  transactions
    tx_id        PK
    state        ENUM(PREPARING, COMMITTED, ABORTED)   -- 決議日誌（WAL，恢復命脈）
    participants JSON   -- 這筆牽涉哪些分片
    decided_at   TIMESTAMP

參與者（每個分片 / Resource Manager）
  accounts
    account      PK
    balance      BIGINT          -- 正式餘額（commit 後才變）
  prepared (WAL：已 prepare、未 commit/abort 的預寫記錄)
    tx_id        PK
    ops          JSON            -- 這筆要套用的變更（如 alice -200）
    locked       JSON            -- 被鎖定的資源（提交前不讓別人動）
為何 prepared 要獨立記錄 prepare 階段把變更寫進 prepared（WAL）並鎖定資源，但不動正式餘額。 這樣：①commit 就是把 prepared 套用到 balance；②abort 就是丟棄 prepared；③崩潰恢復時，參與者讀 prepared 就知道「我有一筆懸而未決的交易」，去問協調者最終結果。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
                    ┌──────────────────────────┐
  client ──轉帳──▶  │  Coordinator (TM)          │
                    │  決議日誌(WAL): COMMIT/ABORT│
                    └──────────┬───────┬─────────┘
       ① prepare(能提交?)      │       │   ① prepare(能提交?)
                               ▼       ▼
                  ┌──────────────┐  ┌──────────────┐
                  │ 參與者 A(分片)│  │ 參與者 B(分片)│
                  │ alice -200    │  │ bob +200      │
                  │ WAL預寫+鎖定  │  │ WAL預寫+鎖定  │
                  └──────────────┘  └──────────────┘
       ② 全 YES → commit（套用）／任一 NO → abort（回滾）  ← 原子性：要嘛都做、要嘛都不做
  1. 協調者(TM)：發起交易、跑兩階段、把最終決議先落盤再通知各參與者，並負責恢復。
  2. 參與者(分片 RM)：各自持有一部分資料，回應 prepare（投票 + 預寫鎖定）/ commit / abort。
  3. NewSQL（如 Google Spanner / CockroachDB / TiDB）：把這套內建——以 2PC 跨分片、用 Paxos/Raft 複製每個分片保證單片高可用、用 TrueTime / HLC 給全域時間戳達成外部一致性，讓你「像用單機 SQL 一樣寫分散式交易」。

6. 核心元件深入

① 兩階段提交 2PC：prepare / commit

  • 第一階段 prepare（投票）：協調者問所有參與者「能不能提交?」。每個參與者各自檢查可行性（如餘額夠不夠）、預寫 WAL、鎖定資源，回 YES（vote-commit）或 NO（vote-abort）。
  • 第二階段 commit / abort（決議）：全部 YES → 協調者把決議寫 COMMIT 落盤，叫大家真的 commit；任一 NO 或逾時 → 寫 ABORT，叫大家 abort 回滾。
  • 原子性從哪來：因為「真的提交」延後到第二階段、且所有人都承諾過能提交，協調者才下令，所以不會出現「A 提交了、B 卻失敗」的撕裂。
  • 一旦決議不能反悔：協調者寫下 COMMIT 後，即使某參與者 commit 時暫時失敗，也不能改成 abort——只能不斷重試 / 等該參與者恢復後依 WAL 重放，直到全部 commit。

② 阻塞問題與協調者單點（2PC 的致命傷）

  • 同步阻塞：參與者一旦投了 YES 進入 prepared 狀態，必須鎖住資源等協調者下令，期間不能釋放。協調者越慢，鎖越久，並發吞吐越差。
  • 協調者單點 → 無限阻塞：若協調者在「收齊投票後、下令前」崩潰，所有投 YES 的參與者會卡在 prepared 永遠等不到決議（資源一直鎖著）。這是 2PC 最被詬病的問題。
  • 緩解：協調者決議先落盤（恢復後可重發決議）＋協調者高可用（用 Raft 把協調者本身做成多副本，如 Spanner 的做法）＋參與者設逾時並可互相詢問。

③ 3PC（三階段提交）：想解阻塞，但代價大

3PC 在 prepare 與 commit 之間插入一個 pre-commit 階段，讓參與者在協調者掛掉時能用逾時自行做出較安全的決定，理論上非阻塞。代價是多一輪往返（更慢），且在網路分區下仍可能不一致。實務上少用——大家寧可用「2PC + 把協調者做成 Raft 高可用」來解單點，而不是 3PC。

④ Percolator：用主鍵當鎖、無中央協調者的 2PC

Google Percolator（TiDB 採用其模型）把 2PC 建在 BigTable 之上：選一個 primary lock（主鎖），prepare 時把鎖寫到各筆資料；commit 點就是「主鎖被清掉並寫上 commit 時間戳」這一個原子操作。其他筆是 secondary，順著主鎖判斷狀態。好處是不需要常駐協調者，崩潰後任何客戶端都能依主鎖把交易推進到結論（自我清理），大幅緩解阻塞。

⑤ Spanner TrueTime 與外部一致性

  • TrueTime：Spanner 用 GPS + 原子鐘給每台機器一個有界誤差的時間區間 [earliest, latest]（誤差 ε，通常幾毫秒）。
  • commit wait：交易提交時，Spanner 取一個 commit 時間戳，然後故意等待 ε，確保「這個時間戳一定已成為過去」。
  • 外部一致性（線性一致）：如此保證「交易 T1 在 T2 開始前提交，則 T1 的時間戳一定小於 T2」——全球任何地方看到的交易順序都與真實時間一致。這是比一般序列化更強的保證，也是 Spanner 的招牌。
  • 底層仍是 2PC 跨分片 + Paxos 複製每分片，TrueTime 解的是「全域時間戳排序」這個分散式特有的難題。

⑥ 2PC vs Saga：何時用哪個

  • 2PC：真原子 + 隔離（提交前鎖定，外部看不到中間態）；但同步阻塞、長鎖、協調者單點、難跨組織。適合短交易、同一資料庫/NewSQL 內、強一致剛需（轉帳、扣庫存）。
  • Saga（㊶）：每步是本地已提交的交易，失敗用補償反向修正；不長鎖、高可用，但只有最終一致、有中間態、要寫補償、補償也可能失敗。適合長流程、跨微服務、可容忍短暫不一致（訂單→付款→出貨）。

7. 擴展與瓶頸

瓶頸症狀對策
提交期間鎖定阻塞prepare→commit 全程鎖資源，並發吞吐下降盡量單分片（colocation）避開 2PC；縮短窗口；參與者數量壓小
協調者單點故障協調者掛 → 參與者卡 prepared 永遠等不到決議決議先落盤可重發；協調者用 Raft 做高可用；Percolator 式自我清理
跨區延遲放大每階段一個 RTT，跨洲 RTT 大 → 2PC 很慢資料就近分片；跨區交易改用非同步/Saga；Spanner 用 commit wait 換一致性
熱點資源死鎖多筆 2PC 互鎖同一帳戶固定加鎖順序、逾時偵測死鎖、降低交易粒度
參與者越多越脆弱N 個參與者任一掛都可能阻塞減少參與者數；可恢復的 WAL；冪等 commit/abort 便於重試

8. 取捨與面試重點

  • 強一致 2PC vs 最終一致 Saga：2PC 給原子+隔離但阻塞長鎖；Saga 高可用不長鎖但最終一致、要補償。能講出依「是否容忍中間態 / 是否跨組織 / 交易長短」來選是得分點。
  • 可用性代價：2PC 在 CAP 下偏 C——分區或協調者故障時寧可阻塞（犧牲可用性）也不破壞一致性。要清楚這正是它「慢、會卡」的根因。
  • 延遲代價：每筆跨分片交易至少多兩個 RTT + 一次決議落盤；所以第一優先是讓資料同分片，把 2PC 留給少數真跨分片的交易。
  • 協調者單點怎麼解：能答出「決議落盤 + 協調者 Raft 高可用 / Percolator 主鎖自我清理」就比只會背「2PC 會阻塞」高一個層次。
  • NewSQL 怎麼把這些藏起來：Spanner/CockroachDB/TiDB = 2PC（跨分片原子）+ Paxos/Raft（每分片高可用）+ TrueTime/HLC（全域排序），讓你用單機 SQL 體驗寫分散式強一致交易。
  • 常見追問：協調者在下 commit 前掛了怎辦？（參與者卡 prepared，靠決議日誌恢復後重發）2PC 為何不用 3PC 解阻塞？（3PC 更慢、分區仍不一致，不如把協調者做高可用）為何說 2PC 不適合跨公司？（要對方長時間鎖資源並信任你的協調者，現實不可行 → 改 Saga）。

9. PHP Demo 說明

本題 demo 在單一 PHP process 內模擬一個協調者 + 兩個參與者（分片），把 2PC 的核心考點跑起來：

  • src/Coordinator.php　協調者(TM)：transfer() 跑完整 2PC——prepare 收集投票、全 YES → commit / 任一 NO → abort、決議寫入日誌（恢復命脈）。
  • src/Participant.php　參與者(分片 RM)：prepare()（檢查餘額、預寫 WAL、鎖定，投 YES/NO）、commit()（套用變更、解鎖）、abort()（丟棄預寫、回滾）；各分片狀態存 data/shard_*.json，以 flock 原子化。
  • public/index.php　路由 + 測試頁：跑跨分片轉帳看 prepare/commit 兩階段、注入某參與者失敗看 abort 回滾、查餘額與總額守恆。
# 啟動
cd demo
php -S localhost:8047 -t public
# 瀏覽器開 http://localhost:8047
# 試：① 正常轉帳（看兩邊 prepare→YES、決議 COMMIT、總額不變）
#     ② 注入分片 B prepare 投 NO → 再轉一次（A 投 YES、B 投 NO → 決議 ABORT → A 回滾，錢不會少）
誠實聲明 Demo 是單一 process 模擬協調者 + 參與者：沒有真實網路 RPC、沒有多機、沒有 Paxos/Raft 複製與 TrueTime。 但 2PC 的 prepare/commit/abort 兩階段、預寫日誌(WAL)+鎖定、投票決議、原子回滾皆為真實可執行——這正是本題核心考點。 注入失敗時，協調者確實收到 NO → 決議 ABORT → 參與者丟棄預寫回滾，總額永遠守恆。 要變成真叢集，把「各參與者一個 JSON 檔」換成「各參與者一支 RPC 服務」即可，2PC 演算法不變。僅用 json + flock，每檔 declare(strict_types=1);。
