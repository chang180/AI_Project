1. 60 秒搞懂它在解什麼

你要從銀行 A 的帳戶轉 200 元到銀行 B 的帳戶。這件事有兩步：A 扣 200、B 加 200。

聽起來很簡單，但魔鬼藏在「萬一只做了一半」：A 扣了 200，B 卻因為當機沒收到——錢就憑空消失了。反過來 B 加了、A 沒扣，錢就憑空多出來。

所以這題要保證的就一句話：「兩步要嘛都成功、要嘛都不做，絕不能只做一半。」這在資料庫叫原子性。當這兩步落在不同機器上時，要做到原子性就變得很難——這就是分散式交易。

🔑 一句話 分散式交易 = 讓散在好幾台機器上的多個步驟，像一個動作一樣「全做或全不做」。最經典的做法叫兩階段提交（2PC）：先一個一個問「你準備好了嗎」，全都說好，才一起真的下手。

2. 為什麼「跨兩家轉帳」這麼難？

很多人想：「不就 A 減、B 加嗎？寫兩行就好。」——這想法漏掉了一個會出大事的真相：機器會在任何時刻當機、網路會斷。

📌 比喻：兩個人同時簽合約 你和遠方一個夥伴要簽同一份合約，必須兩個人都簽才生效。 如果你先簽了、把合約寄出去，結果對方臨時反悔不簽——你已經簽的那份就尷尬地懸在半空。 聰明的做法是：先各自說「我願意簽」（口頭承諾），等雙方都點頭，再同時落筆。這樣就不會有人單方面被綁住。

分散式交易的難關正是這個：你不能「叫 A 先扣、再叫 B 加」，因為中間任何一步都可能掛掉，留下「做一半」的爛攤子。

💭 停一下：為什麼不能「先讓 A 扣款，成功後再叫 B 入帳」就好？

看答案 因為 A 扣完之後、輪到 B 入帳之前，B 可能剛好當機或餘額/規則不允許。這時 A 的錢已經扣了，B 卻沒加——錢憑空消失。 而且你也沒法簡單把 A 的扣款收回（萬一收回的指令也掛了呢？）。所以要先問清楚大家都行，再一起動手，這就是 2PC。

3. 兩階段提交：拆成 2 個關卡

2PC 的名字就告訴你它分兩關。中間有個「主持人」叫協調者，被問的各方叫參與者（這裡就是銀行 A 和 B）。

1第一關 prepare：先問「你準備好了嗎？」

協調者一個一個問每個參與者：「我要做這筆交易，你能不能提交？」

🍽️ 比喻：訂位前先確認 辦一場聚餐要訂三家店配合。你先打電話一家一家問：「這天這時段，你能不能保留位子？」 店家會先把位子圈起來（預留、先別給別人），然後回你「可以」或「不行」。

每個參與者收到 prepare 會做三件事：① 檢查能不能做（A 餘額夠嗎？）② 把這筆變更先記在小抄上、但還沒真的改餘額（這張小抄叫「預寫日誌」）③ 把相關帳戶鎖起來（先別讓別人動），然後回答 YES（我準備好了） 或 NO（我不行）。

2第二關 commit / abort：全說好才下手，有人不行就全退

協調者收集所有人的回答，然後只有兩種結局：

  • 大家都說 YES → 協調者下令「commit（真的提交）」：每個參與者把小抄上的變更真正套用（A 真的扣、B 真的加），然後解鎖。✅ 全部成功。
  • 有任何一個說 NO（或不回應/逾時） → 協調者下令「abort（取消回滾）」：每個參與者把小抄撕掉、解鎖，餘額從頭到尾沒被動過。❌ 全部當沒發生。
🍽️ 比喻：訂位的第二通電話 三家店都說「可以保留」→ 你再打第二通：「那就訂了！」三家確認成交。 只要有一家說「抱歉客滿」→ 你打第二通跟另外兩家說：「不好意思取消」，他們把圈起來的位子放開。沒有人被白白綁住。

這就是原子性的祕密：「真的下手」永遠延到第二關，而且要全員點過頭。所以絕不會出現「A 扣了、B 沒加」。

✅ 兩關一句話 ① prepare：協調者問大家「能提交嗎？」，各自預留+鎖定並投票 → ② 全 YES 就 commit（真做）、任一 NO 就 abort（全退）。要嘛都成功、要嘛都回滾。

4. 設計時最怕的 3 件事

😱 怕「做一半」 這是最致命的怕：A 扣了 B 沒加，錢就不見了。2PC 就是為了避開它——先問再做、全員同意才動手，讓「做一半」變成不可能。 寧可整筆取消，也不要做一半。
😱 怕「主持人突然消失」 大家都投了 YES、把帳戶鎖起來等指令，結果協調者（主持人）突然當機，再也沒人下「commit 還是 abort」。 參與者就卡在那裡、帳戶一直鎖著動不了——這叫阻塞，是 2PC 最有名的痛點。 解法：協調者先把決定寫在紙上（落盤），這樣它重開機後能照紙上重發指令；更進階是把協調者做好幾個備份，掛一個還有別的頂上。
😱 怕「鎖太久、大家排隊」 從 prepare 到 commit 這段，帳戶是鎖著的，別人想動同一個帳戶只能排隊。參與者越多、協調者越慢，鎖越久、塞車越嚴重。 所以工程上會盡量讓相關資料放在同一台機器（不用跨機就不用 2PC），真要跨才用，而且盡量把這段鎖定時間縮到最短。

6. 名詞白話翻譯

看完上面，這些詞你其實都懂了，這裡把「工程術語 ↔ 白話」對起來，Part 2 會用到：

分散式交易散在多台機器的好幾步，要像一個動作一樣「全做或全不做」
原子性 (Atomicity)「全做或全不做」這個保證本身（A 的 ACID 之首）
2PC（兩階段提交）先問大家「能提交嗎」（prepare），全說好才一起真做（commit），否則全退（abort）
協調者 (Coordinator/TM)主持人：發起交易、收集投票、下最終命令、負責當機後恢復
參與者 (Participant/分片)被問的各方：各管一份資料，回答 prepare、執行 commit/abort
prepare（第一階段）「你能不能提交？」——各自檢查、預留、鎖定、投票 YES/NO
commit / abort（第二階段）真的做 / 取消回滾
預寫日誌 (WAL)那張「先記下要改什麼、但還沒真改」的小抄；當機後靠它恢復
鎖定 (lock)prepare 後先把帳戶圈起來，別讓別人同時動
阻塞 (blocking)協調者掛了，參與者卡著鎖等指令、動不了
Saga另一種做法：每步先各自做完，出錯再用「補償動作」反向修正（最終一致）
NewSQL（Spanner 等）把這一整套內建好的資料庫，讓你像用單機一樣寫分散式強一致交易

7. 要付出多少代價？（估算）

📞 比喻：每次都要打兩輪電話 2PC 每筆交易都得「打一輪確認電話（prepare）+ 打一輪拍板電話（commit）」。 電話來回是要花時間的——這就是 2PC 的主要代價：比單機慢、而且講電話期間位子一直圈著。
估什麼大概感覺所以呢？
一筆跨機交易要幾輪來回至少 2 輪（prepare + commit）比單機交易慢約一倍 → 能不跨機就不跨
帳戶鎖多久整個 prepare→commit 期間主持人越慢、參與者越多 → 鎖越久、別人越塞
多少交易其實要跨機通常只有少數把相關資料放同一台（colocation），大多數就走快速單機路
主持人要不要存紀錄每筆都要把決定落盤多一次寫硬碟，但這是當機能救回來的命根子
💡 關鍵洞察：最省的招是「根本不跨機」 2PC 再聰明也有代價。所以第一原則是盡量把要一起改的資料放在同一台機器，這樣就用普通的單機交易、又快又不卡。 只有真的非跨不可，才動用 2PC，而且把參與者數量壓到最少、鎖定時間壓到最短。

💭 停一下：既然 2PC 又慢又會卡，為什麼還要用它？

看答案 因為在錢、庫存這種場景，「做一半」是絕對不能發生的。2PC 用「慢一點、會卡一點」換來鐵一般的原子性。 如果你的場景能容忍短暫不一致（例如電商下單後慢慢出貨），就改用Saga（後面取捨會講），用補償換高可用。選哪個看你「最不能忍受什麼」。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是系統之間講好的「點餐單格式」。這題的點餐單分兩種：對外（你發起轉帳）和對內（協調者跟參與者講的兩階段悄悄話）。 
① 你發起一筆轉帳（對外）
   POST /api/transfer
   給我：{ 從哪(A:alice), 到哪(B:bob), 金額 }
   我回：{ 結果: COMMIT 或 ABORT, 兩邊各投了 YES/NO }
② 協調者跟參與者的兩階段悄悄話（對內）
   prepare(交易編號, 要改什麼)  → 回 YES / NO   ← 第一關：能提交嗎
   commit(交易編號)             → 真的改
   abort(交易編號)              → 撕小抄、回滾

注意：你只碰得到第①張（發起轉帳）。第②組是協調者在背後跟各參與者跑的兩階段，你看不到，但結果（COMMIT/ABORT）會回給你。

9. 系統要記住哪些事？（資料模型）

📒 比喻：主持人一本帳、每個參與者各兩本 主持人要記「每筆交易最後決定是成交還是取消」；每個參與者要記「我的正式餘額」和「還沒拍板的小抄」。

🧑‍⚖️主持人的「決議日誌」

每筆交易一列：交易編號、最終決定(COMMITTED / ABORTED)、牽涉哪些參與者。 這本最重要：主持人當機重開後，靠它知道「這筆我之前決定成交了沒」，才能重發正確指令、不讓參與者卡死。

💰參與者的「正式餘額」

每個帳戶一列：帳戶、餘額。只有 commit 那一刻才會被改；prepare 階段碰都不碰它。

📝參與者的「小抄（預寫日誌 WAL）」

每筆還沒拍板的交易一列：交易編號、要改什麼(alice -200)、鎖了哪些帳戶。 commit 就是把小抄套到正式餘額；abort 就是把小抄撕了；當機重開看到小抄還在，就知道「我有一筆懸著的交易」，去問主持人結果。

10. 哪裡會卡住、怎麼救？（擴展）

🚗 比喻：找出最容易塞車的路口，先救它 系統變大時總有地方先卡。2PC 的卡點很集中，一個一個看：
哪裡會卡怎麼救
鎖定期間別人排隊（最常見）盡量把相關資料放同一台、別跨機；真跨也把鎖定時間縮到最短、參與者壓最少
主持人當機 → 參與者卡死主持人決定先落盤（重開能重發）；更進階：把主持人做好幾個備份（Raft 高可用）
跨城市/跨國，電話來回特別久資料就近放；跨國交易改用非同步 / Saga；Spanner 用特殊時鐘技巧換一致性
兩筆交易互相鎖住對方（死結）規定固定的加鎖順序、設逾時自動放棄重來

11. 還有哪些「魚與熊掌」？（取捨）

⚖️ 2PC（先問再做） vs Saga（先做再補償） 2PC：先問大家、全同意才動手，真原子、別人看不到做一半的中間態；但會鎖、會卡、主持人是單點。 Saga：每步先各自做完（先扣款、再出貨…），哪步失敗就跑反向補償（退款）；不鎖、高可用，但只有最終一致、看得到中間態、要自己寫補償。 錢/庫存這種「絕不能做一半」用 2PC；長流程、跨好多服務、能忍一下不一致用 Saga。
⚖️ 一致性 vs 可用性 2PC 選的是一致性：遇到當機或網路斷，它寧可卡住（不可用）也不讓資料錯。這就是它「會卡」的本質原因，不是 bug，是它的選擇。
⚖️ 自己刻 vs 用 NewSQL 自己用 2PC 拼很容易出錯（恢復、死結都難）。現在多半直接用 NewSQL（Google Spanner、CockroachDB、TiDB）——它把 2PC + 多副本 + 全球時鐘都內建好了，你就像寫單機 SQL 一樣，背後自動幫你跑分散式強一致交易。
💭 追問：協調者在「收齊投票後、還沒下令前」當機，會怎樣？ 投了 YES 的參與者會卡在「鎖著、等指令」的狀態（這就是阻塞）。救法：協調者重開後讀決議日誌，知道這筆該 commit 還是 abort，重發一次指令把大家解開。所以「決定先落盤」才那麼關鍵。
💭 追問：那為什麼不乾脆用「三階段提交(3PC)」來解卡死？ 3PC 多加一關，理論上參與者能在主持人掛掉時自己逾時做決定、比較不卡。但代價是每筆更慢（多一輪電話），而且網路一斷照樣可能不一致。所以實務上大家寧可用「2PC + 把主持人做好幾個備份」來解單點，而不是 3PC。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「prepare → commit / abort」兩階段：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8047 -t public
# 然後瀏覽器打開 http://localhost:8047
  1. 正常轉帳：alice(A) 轉 200 給 bob(B)。看兩邊都 prepare → YES、主持人決議 COMMIT、兩邊 commit。總額還是 2000。
  2. 注入失敗看回滾：把分片 B 設成「prepare 投 NO」，再轉一次。看 A 投 YES、B 投 NO → 主持人決議 ABORT → A 把小抄撕掉回滾。alice 的錢沒少、總額還是 2000！
  3. 試試轉一筆超過 alice 餘額的金額：A 在 prepare 就投 NO，整筆 abort。
玩的時候想一下第 2 步是這題的精髓：明明 A 已經願意（投 YES、鎖好、寫了小抄），只因為 B 不行，整筆就全部退回，A 一毛都沒被扣。這就是原子性——全做或全不做，總額永遠守恆。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「兩階段到底是哪幾段程式做的」。 好消息：核心就兩個角色——主持人（Coordinator）和參與者（Participant）。不用會 PHP，我把程式翻成白話、只留關鍵幾行，看註解就懂。

🔧 兩個角色 = 兩個檔 Coordinator（主持人，跑兩階段、下最終決定）→ Participant（參與者，回答 prepare、執行 commit/abort）。一筆交易就是主持人指揮所有參與者跑一遍。

1第一關 prepare：參與者「能不能提交？」（Participant）

參與者收到 prepare，先檢查 → 寫小抄 → 鎖定 → 投票：

function prepare(交易編號, 要改什麼) {
    // ① 檢查可不可行（例如：扣完會不會變負的？）
    if (餘額 + 變動 < 0) return NO;        // ← 不行 → 投 NO
    // ② 寫小抄(WAL)：記下要改什麼，但『還沒』真的改餘額
    小抄[交易編號] = 要改什麼
    // ③ 鎖定這些帳戶，先別讓別人動
    鎖定(相關帳戶)
    return YES                              // ← 我準備好了 → 投 YES
}

重點：prepare 完，正式餘額一點都沒變，只是寫了小抄 + 鎖起來。這就是為什麼後面要 abort 時，只要撕掉小抄就乾乾淨淨回滾了。

2第二關：主持人收票，全 YES 才 commit（Coordinator）

這是全題的心臟。主持人問完所有人，只有全票通過才下 commit：

function transfer(從, 到, 金額) {
    交易編號 = 產生一個
    要改的 = { A: alice -金額, B: bob +金額 }

    // ===== 第一關：一個一個問 prepare =====
    全部YES = true
    for (每個參與者 in 要改的) {
        if (participant.prepare(交易編號, 它要改的) == NO) {
            全部YES = false; break       // ← 有人投 NO，不用再問了
        }
    }

    // ===== 第二關：依投票結果決定 =====
    if (全部YES) {
        寫決議日誌(交易編號, "COMMIT")    // ← 先落盤！當機才救得回
        for (每個參與者) participant.commit(交易編號)   // 真的改
    } else {
        寫決議日誌(交易編號, "ABORT")
        for (投過票的參與者) participant.abort(交易編號) // 撕小抄、回滾
    }
}

看到了嗎？「真的改」(commit) 永遠在第二關、而且要先確認全員 YES。這一個 if (全部YES) 就是原子性的來源。還有那行「先寫決議日誌再下令」——少了它，主持人一當機大家就卡死了。

3commit 真的改、abort 乾淨退（Participant）

主持人下令後，參與者執行——這兩個動作簡單到不行，因為前面 prepare 已經把功課做好了：

function commit(交易編號) {
    把小抄[交易編號]的變更『真的套用』到餘額   // alice 真的 -200
    刪掉小抄、解鎖                              // 收工
}
function abort(交易編號) {
    刪掉小抄、解鎖                              // ← 餘額從頭沒被動過，撕了就回滾完成
}

abort 之所以這麼乾淨，正是因為 prepare 階段只寫小抄、沒動正式餘額。要回滾？把小抄撕了就好，餘額自然像沒發生過。這是 2PC 設計的巧妙之處。

💭 停一下：為什麼 commit 之前要「先把決定寫進日誌」，而不是直接叫大家 commit？

看答案 因為主持人可能在叫到一半時當機。如果決定沒落盤，重開後它就不知道這筆到底該 commit 還是 abort，參與者就永遠卡著。先把「COMMIT」寫進日誌，重開後一看就知道「我承諾過要成交」，重發 commit 把大家推完。日誌是恢復的命根子。

14. 考考自己

先別看答案，自己用講給朋友聽的方式回答看看：

Q1：用「兩人簽合約」解釋，2PC 為什麼要先 prepare 再 commit？ 因為不能讓任何一方「單方面先做」而被綁住。先各自口頭承諾「我願意簽（prepare 投 YES、預留鎖定）」，等雙方都點頭，才同時落筆（commit）。任何一方不肯，就大家都不簽（abort）。這樣絕不會「一方簽了、另一方沒簽」。
Q2：prepare 階段參與者會做哪三件事？ ① 檢查能不能做（如餘額夠不夠）；② 把要改的內容寫進「小抄」(預寫日誌 WAL)，但還沒真的改餘額；③ 鎖定相關帳戶。然後回 YES 或 NO。
Q3：為什麼 abort 回滾可以那麼乾淨？ 因為 prepare 階段只寫了小抄、沒動正式餘額。要回滾只要把小抄撕掉、解鎖，餘額自然像沒發生過——不需要「反向再扣一次」那種麻煩補償。
Q4：2PC 最有名的痛點是什麼？怎麼緩解？ 阻塞：協調者在「收齊投票後、下令前」當機，投了 YES 的參與者會卡著鎖等指令、動不了。緩解：協調者把最終決定先落盤（重開能重發），再進階就把協調者做成多副本（Raft 高可用）。
Q5：什麼時候該用 2PC、什麼時候該用 Saga？ 錢/庫存這種「絕不能做一半、不能讓人看到中間態」用 2PC（強一致，代價是鎖+卡）。長流程、跨好多微服務、能容忍短暫不一致用 Saga（每步先做、出錯用補償反向修正，高可用但最終一致）。
Q6（工程）：協調者下 commit 前一定要先做什麼？為什麼？ 先把決議（COMMIT/ABORT）寫進日誌並落盤。因為它可能下令到一半當機，重開後得靠日誌知道「這筆該成交還是取消」，才能重發正確指令把參與者解開，不然大家永遠卡著。
Q7（工程）：NewSQL（如 Spanner）是怎麼幫你把這些藏起來的？ 它把 2PC（跨分片原子）+ Paxos/Raft（每個分片做多副本高可用）+ TrueTime（全球一致的時間戳排序） 全部內建。你就像寫單機 SQL 一樣下交易，背後自動跑分散式強一致，連恢復、排序都幫你處理好。

15. 30 秒回顧

分散式交易 = 讓散在多台機器的好幾步，像一個動作一樣「全做或全不做」。

2 個關卡（2PC）：① prepare：主持人問大家「能提交嗎」，各自檢查+寫小抄+鎖定+投票 → ② 全 YES 就 commit（真做）、任一 NO 就 abort（撕小抄回滾）。

3 個怕：怕做一半（2PC 先問再做避開它）、怕主持人消失（決議先落盤 + 做多副本）、怕鎖太久（盡量不跨機、縮短鎖定）。

工程細節一句話：真正下手永遠延到第二關、且要全員點頭 → 用「點餐單(API)」溝通、主持人記決議日誌、參與者記正式餘額+小抄(WAL) → 找出最會卡的「鎖定」與「主持人單點」各自救 → 真要又快又穩，直接用 NewSQL（Spanner/CockroachDB/TiDB）把整套內建。
