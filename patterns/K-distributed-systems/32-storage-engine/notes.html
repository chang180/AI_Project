㉜ LSM-tree 儲存引擎：為寫密集而生的儲存結構

K 分散式系統 建議 50 分鐘 Design an LSM-tree Storage Engine

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「把隨機寫變成順序寫」的儲存引擎：寫只 append 到日誌與記憶體， 再批次刷成不可變檔，讀時靠 bloom filter 與 compaction 控制讀放大。 這是 RocksDB / LevelDB / Cassandra / HBase 的共同底座，也是「寫優化 vs 讀優化」取捨的經典題。

1. 需求釐清

功能性需求

  • 提供 KV 介面：put(key, value)、get(key)、delete(key)。
  • 持久化：寫入後即使行程崩潰/斷電也不丟資料。
  • 支援範圍掃描（key 有序，可 scan）。
  • 刪除：能標記某 key 為已刪除，之後讀不到。

非功能性需求

  • 高寫入吞吐：寫密集場景（時序、日誌、訊息、計數器、Cassandra 類工作負載）。
  • 可接受的讀延遲：讀比 B-tree 稍慢可接受，但要可控（靠 bloom + compaction）。
  • 空間可控：過期/被刪版本要能回收（compaction）。
  • 崩潰可復原：未落盤的寫能由 WAL 重播。
面試開場必問 「這是寫密集還是讀密集？」決定選 LSM 還是 B-tree。寫多 → LSM（順序寫、寫放大低）； 讀多、要單筆低延遲 → B-tree（一次樹查找）。把這句講清楚，整題基調就定了。

2. 規模估算

指標假設推算 / 意義
寫入吞吐每筆 200 bytes，5 萬寫/秒~10 MB/s 進 WAL（順序寫，HDD 也扛得住）
memtable 大小64 MB 滿即 flush每 ~6 秒產生一個 SSTable
寫放大leveled，每層放大 ~10×，4 層同一筆資料一生會被 compaction 重寫多次 → 寫放大 ~10～30×
讀放大無 compaction 時，N 個 SSTable最壞要查 N 個檔 + memtable；bloom 把多數降到 ~1 次磁碟讀
空間放大舊版本/tombstone 未回收compaction 前可能存多份；壓實後回收
關鍵洞察 LSM 把成本從「寫入當下」延後到背景的 compaction。寫變快，但代價是 寫放大（資料被重寫多次）與 讀放大（要查多層）。整題就是在調這三角。

3. API 設計

# 寫入 / 覆寫
put(key, value)        # WAL append → memtable[key] = value

# 刪除（不是真的清掉，而是寫 tombstone）
delete(key)            # WAL append → memtable[key] = TOMBSTONE

# 讀取
get(key) -> value | NOT_FOUND
  先查 memtable；未命中則由新到舊掃 SSTable（bloom 先過濾）；
  第一個命中版本即答案（若是 tombstone → NOT_FOUND）。

# 範圍掃描（key 有序才做得到）
scan(start, end) -> 合併 memtable 與各 SSTable 的有序串流（取每個 key 最新版本）

# 維運操作
flush()    # 手動把 memtable 落成 SSTable
compact()  # 手動觸發 compaction（合併 SSTable、回收舊版本）

4. 資料模型

WAL（write-ahead log，順序 append）
  每行一筆：{ op: P|D, key, value }     -- 崩潰時重播回 memtable

memtable（記憶體有序表，常用跳表 SkipList / 紅黑樹）
  key -> value 或 TOMBSTONE             -- 可變，最新的寫都在這

SSTable（Sorted String Table，落盤後不可變）
  ├─ data：key 有序的 (key, value) 區塊（可分 block + 壓縮）
  ├─ sparse index：每隔幾個 key 記一個偏移，用來定位 block（不必每個 key 都索引）
  └─ bloom filter：此 SSTable 含哪些 key 的機率型摘要（常駐記憶體）

manifest：目前有哪些 SSTable、屬於哪一層、key 範圍
為何 SSTable 不可變 不可變 = 寫成後永不就地修改 → 沒有隨機寫、好快取、好並行讀、compaction 只是「讀舊檔寫新檔再換掉」。 「更新」與「刪除」都靠寫新版本（含 tombstone）表達，舊版本留待 compaction 回收。

5. 高階架構（寫路徑 vs 讀路徑）

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw） 
寫路徑（藍）
  put/delete ─▶ ① WAL append（順序寫，持久化）
                 └▶ ② memtable（記憶體有序）
                       └▶（滿門檻）③ flush ─▶ SSTable_n（不可變）+ index + bloom
                                              清空 memtable、截斷 WAL

讀路徑（綠）
  get(key) ─▶ memtable 命中？─是─▶ 回值/NOT_FOUND(tombstone)
                 └─否─▶ 由新到舊：SSTable_n → … → SSTable_1
                          每個先問 bloom：mightContain?
                            false → 跳過（不碰磁碟）
                            true  → 讀該檔；命中即回（含 tombstone）

背景（黃）
  compaction：合併多個 SSTable ─▶ 丟棄被覆寫/被刪版本 ─▶ 更少更大的檔
  1. 寫：只碰「append 一行日誌 + 改記憶體」，沒有隨機寫 → 寫吞吐高。
  2. 讀：memtable 最新；SSTable 由新到舊，bloom 過濾掉絕大多數不含此 key 的檔。
  3. 背景：compaction 持續整理，控制 SSTable 數量（讀放大）與空間。

6. 核心元件深入

WAL 與崩潰復原

每次寫先 append 到 WAL 才改 memtable。memtable 在記憶體，崩潰會丟；但只要 WAL 在， 重啟時把 WAL 重播回 memtable 即可復原。flush 成功後該批資料已在不可變 SSTable，WAL 即可截斷。 （生產上 WAL 寫入要不要 fsync 是延遲 vs 持久性的取捨。）

memtable → SSTable（flush）

memtable 超過門檻（如 64MB）→ 整批排序後一次性順序寫成一個 SSTable， 同時建稀疏索引與該檔的 bloom filter。期間可先切一個新的 memtable 繼續接寫，不阻塞寫入。

bloom filter 省磁碟 I/O

每個 SSTable 配一個常駐記憶體的 bloom filter。讀一個 key 時，先問每個 SSTable 的 bloom 「可能有嗎」：false = 一定沒有，直接跳過、不開檔（省一次磁碟 I/O）； true = 可能有（含偽陽性），才真的去讀。沒有 bloom 的話，查一個不存在的 key 得把每個 SSTable 都翻一遍，讀放大會爆炸。bloom 不會偽陰性，所以安全。

compaction 策略：size-tiered vs leveled

策略做法偏向代表
size-tiered把大小相近的多個 SSTable 合成更大的一個；同層可有多個重疊 key 範圍寫放大低、空間放大較高、讀放大較高Cassandra 預設
leveled分層 L0/L1/…；每層內 SSTable key 範圍不重疊，每層容量約上層 10×讀放大低、空間放大低、寫放大較高LevelDB / RocksDB
tombstone 何時真的消失 刪除只寫一個 tombstone（標記）。compaction 把含此 key 的所有 SSTable 一起合併、且確定沒有更舊副本時， 才能把該 key 與 tombstone 一併丟棄。過早丟 tombstone 會讓「已刪的舊值」復活，所以要小心。

讀放大 / 寫放大 / 空間放大：三角取捨

  • 寫放大：一筆資料被 compaction 重寫的總量倍數。leveled 高、size-tiered 低。
  • 讀放大：一次 get 平均要碰幾個 SSTable。SSTable 越多越糟；compaction 與 bloom 來壓。
  • 空間放大：實際佔用 / 有效資料。舊版本與 tombstone 未回收則高；compaction 來回收。
  • 三者不可能同時最佳，compaction 策略就是在這三角上選位置。

7. 擴展與瓶頸

瓶頸症狀對策
compaction 吃 IO背景合併與前台寫搶磁碟頻寬，寫入卡頓（write stall）限流 compaction、分離壓實執行緒、用更快磁碟、調整層級觸發門檻
讀放大SSTable 太多，單次 get 要查很多檔加大/更積極 compaction、用 leveled、bloom 調參
bloom 調參偽陽率太高 → 白讀檔；位元太多 → 吃記憶體依每 key bits 與雜湊數 k 平衡偽陽率與記憶體（典型 ~10 bits/key, 1% 偽陽）
大 valuevalue 大時 compaction 重寫成本高KV 分離（如 WiscKey，value 另存 log，LSM 只放 key+指標）
單機容量資料超過單機上層按 key range / hash 分片，每分片各自一棵 LSM（如 Cassandra/HBase）

8. 取捨與面試重點

面向LSM-treeB-tree
寫入順序 append → 快、寫放大低就地更新 → 隨機寫、頁分裂，寫放大高
讀取可能查多層 SSTable（讀放大），靠 bloom 緩解一次樹查找 → 讀延遲低且穩定
空間舊版本/tombstone 需 compaction 回收就地更新，空間放大低
背景成本compaction 持續吃 IO/CPU幾乎無背景重寫
適用寫密集、時序、計數、訊息讀密集、交易型、需穩定點查
  • 誰在用 LSM：RocksDB、LevelDB、Cassandra、HBase、ScyllaDB、InfluxDB、Kafka 索引等。
  • 誰在用 B-tree：MySQL InnoDB、PostgreSQL、多數傳統關聯式 DB 的索引。
  • 常見追問：① 為何不用 fsync 每筆？（延遲 vs 持久性）② tombstone 何時可丟？（確定沒更舊副本） ③ 讀放大怎麼壓？（bloom + compaction + leveled）④ 範圍掃描怎麼做？（多路有序歸併取最新版本）。

9. PHP Demo 說明

本題 demo 把 LSM-tree 的核心邏輯用零依賴 PHP 實作，可直接跑：

  • src/Bloom.php　Bloom filter（位元陣列 + 雙重雜湊 crc32/fnv1a32）；add / mightContain，示範「查不到的 key 不用碰磁碟」與偽陽性。
  • src/LsmEngine.php　WAL（崩潰復原）、memtable（有序、tombstone 刪除）、flush（→ 不可變 SSTable + 稀疏索引 + bloom）、get（memtable → 由新到舊掃 SSTable，bloom 先過濾）、compaction（size-tiered 簡化版：合併並丟棄被刪/被覆寫版本）；以 flock 保護共享狀態。
  • public/index.php　路由與測試頁：put/get/delete、看 memtable 內容、SSTable 清單、手動 flush 與 compaction、bloom 命中/跳過統計。
# 啟動
cd demo
php -S localhost:8032 -t public
# 連續 put 多筆觸發自動 flush；del 一個 key；get 不存在的 key 看 bloom 跳過；compact 後再 get
誠實聲明 Demo 的 SSTable 以 JSON 檔模擬（真實系統是排序後的二進位 block + 索引）；但 WAL、memtable、flush、bloom filter、compaction、tombstone 的邏輯皆為真實可執行， 為單機教學示意，未實作真實的 leveled 分層 compaction 與 block 壓縮。
