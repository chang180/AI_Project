1. 60 秒搞懂它在幹嘛

資料庫底層有個小元件，專門負責「把資料真的存到硬碟、再找回來」——這就是儲存引擎。 你 put 一筆、get 一筆、delete 一筆，背後幫你跑腿的就是它。

同一件事有兩種做法，這題講的 LSM-tree 是其中一種，它特別擅長一件事：寫得超快。 它的秘訣，其實是一個你每天都在用的生活習慣——

🔑 一句話 LSM-tree = 「先記便利貼，之後再整理歸檔」的存法。 來一筆就快速貼張便利貼（記憶體），貼滿了整批排好序歸成一本冊子（檔案）；之後背景慢慢把舊冊子併一併。 重點：把「翻到哪頁改哪頁」的隨機寫，換成「從頭一路寫下去」的順序寫，所以快。

2. 為什麼「亂改」會那麼慢？

很多人第一個疑問是：「存資料就存啊，為什麼還要分兩種做法？」關鍵在一個你早就懂的生活經驗——

📖 翻書改字 vs 抽張便利貼 想像你管一本已經排好序、裝訂好的厚字典。今天要新增一個詞，你得：翻到該插入的那一頁、 把後面的字往後擠、可能那頁塞不下還得拆頁重排——每改一筆都要翻書、翻到對的地方、就地動手術。這很慢。 反過來，如果你只是隨手抽張便利貼寫下「新增：XXX」往桌上一貼，是不是快得多？

硬碟（尤其傳統硬碟）最怕的就是「翻到哪、改哪」這種跳來跳去的隨機寫——磁頭要一直移動定位，慢。 反之「從頭一路往後寫」的順序寫快非常多。

  • B-tree（翻書派）：資料一直保持排好序、裝訂好。讀很快（翻一次就到），但每次改都要就地翻書改頁＝隨機寫，寫較慢。
  • LSM-tree（便利貼派）：來一筆就貼便利貼，從不就地改舊資料。寫超快（永遠只是往後加），代價是讀的時候可能要多翻幾本冊子。

所以這題的靈魂只有一句：用「順序寫」取代「隨機寫」，把寫入變快。其他所有設計，都是為了讓「便利貼派」的讀取別太慢、別爆掉。

💭 停一下：如果你開一家店，客人一直來下訂單，你會選「每來一單就翻總帳本改一頁」，還是「先把單子串在釘子上、打烊後一次結算」？哪個讓你在尖峰時段更快？

看答案 當然是「先串釘子、之後一次結算」。尖峰時你需要「收單動作越輕越好」，把整理的重活延後到不忙時做。 LSM-tree 就是這個思路：寫入當下只做最輕的事（貼便利貼），把整理（compaction）丟到背景。

3. 把儲存引擎拆成 4 個關卡

別被「儲存引擎」嚇到。整件事其實就是闖 4 關，一關一關來：

1寫入：貼便利貼 + 先記流水帳

來一筆 put，系統不急著動硬碟上排好序的資料。它只做兩件超輕的事：

  • 📝 貼便利貼：把這筆寫進記憶體裡一張暫存表（叫 memtable）。記憶體很快，瞬間完成。
  • 🧾 先記一筆流水帳：同時在硬碟的一個「流水帳本」尾巴 append 一行（叫 WAL）。
🧾 比喻：為什麼要記流水帳？ 便利貼貼在桌上（記憶體），一陣風（當機/斷電）就吹光了。所以每貼一張，先在一本「絕不擦掉、只往後寫」的流水帳本上抄一行。 萬一當機，重開機時把流水帳從頭照念一遍，桌上的便利貼就全部復原了。這就是 WAL（Write-Ahead Log，先寫日誌）。

👉 重點：寫入只碰「記憶體 + 流水帳尾巴」，兩者都超快，所以 LSM 寫得飛快。

2歸檔：便利貼貼滿了，整批排序歸成一本冊子

桌上的便利貼（memtable）越貼越多，總會滿（例如到 64MB）。這時系統把整桌便利貼一次排好序， 寫成硬碟上一本排好序、裝訂好、之後永不修改的冊子，叫 SSTable。

📚 比喻：整理歸檔 就像你桌上便利貼貼太多了，找一個下午把它們按字母排好、裝訂成一本冊子歸到書架上。 歸好的冊子從此不再改——要更新？貼新便利貼就好，舊冊子放著不動。

歸檔後便利貼清空、流水帳也可以截斷（因為資料已安穩躺在硬碟的冊子裡）。然後桌子清空，繼續貼新便利貼。

⚠️ 注意：「刪除」也不是真的去舊冊子裡擦掉，而是貼一張便利貼寫「這個 key 已刪」（叫 tombstone 墓碑），讀的時候看到墓碑就當作沒有。

3查詢：先看目錄判斷「這本一定沒有」就跳過

問題來了：資料散在「桌上便利貼 + 書架上一堆冊子」裡。要 get 一個 key，難道每本冊子都翻一遍？太慢。

🔎 比喻：先看冊子封面的目錄 每本冊子封面都貼一張「本冊收錄了哪些詞」的快速目錄（叫 bloom filter）。 要找某個詞時，先瞄一眼目錄：目錄說「沒有」＝這本一定沒有，直接跳過不用翻（省下翻書＝省一次磁碟讀）； 目錄說「可能有」＝才真的翻開這本找。

查詢順序是：先看桌上便利貼（最新）→ 再從最新的冊子往舊翻，每本先問 bloom 目錄，「一定沒有」就跳過。第一個找到的版本就是答案。

⚠️ 小細節：bloom 目錄可能「誤報有」（說可能有結果翻開沒有），但絕不會誤報沒有——所以跳過的都是真的安全，不會漏資料。

4整理：定期把舊冊子合併、丟掉過時資料（compaction）

一直貼便利貼、一直歸檔，書架上冊子會越來越多——查詢要翻的冊子也越來越多（讀變慢），而且同一個 key 可能在好幾本冊子裡有舊版本，浪費空間。

🗂️ 比喻：合併幾本舊冊子成一本新的 趁系統不忙時，把幾本舊冊子拿出來合併成一本新的：同一個 key 只留最新版本， 被刪掉的（墓碑）和被覆寫的舊值趁機丟掉，然後用新冊子換掉那幾本舊的。這叫 compaction（壓實/合併整理）。

compaction 就是 LSM 的「家務」：冊子變少又變大 → 查詢要翻的本數變少（讀變快）、過時資料被回收（省空間）。它默默在背景跑。

✅ 四關一句話 ① 寫＝貼便利貼+記流水帳 → ② 歸檔＝貼滿了排序成冊子 → ③ 查＝先看目錄跳過沒有的冊子 → ④ 整理＝合併舊冊子丟過時資料。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。 而且這三件事像翹翹板，不可能同時都滿分，調設計就是在它們之間選位置。

😱 怕「當機丟資料」 便利貼在記憶體，一斷電就沒了。所以每貼一張先記一筆流水帳（WAL），當機後照念復原。 （細節取捨：流水帳要不要每筆都「確實壓進硬碟」(fsync)？要 → 更安全但每筆更慢；不要 → 更快但當機可能丟最後幾筆。）
😱 怕「讀太慢」（讀放大） 便利貼派的代價：資料散在好多本冊子，查一個 key 最壞要翻很多本。 靠兩招壓住：bloom 目錄跳過「一定沒有」的冊子、compaction 把冊子數量壓低。
😱 怕「同一筆被重寫太多次」（寫放大）+ 占空間 每次 compaction 都把資料「搬一次家」，同一筆資料一輩子可能被搬好幾次（叫寫放大），很吃硬碟。 整理得太勤 → 寫放大大、吃資源；整理得太懶 → 冊子一堆讀變慢、舊版本占空間（空間放大）。要拿捏「多久整理一次」。
⚖️ 一句話總結三角 寫放大、讀放大、空間放大三者不可能同時最佳——這就是 LSM 的核心取捨，所有調校都繞著這個三角轉。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」、瓶頸大概在哪。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？（白話）
寫入有多快每筆 200 bytes、每秒 5 萬筆 ≈ 10 MB/秒因為是「順序寫」，連傳統硬碟都扛得住
多久歸檔一本冊子便利貼貼滿 64MB 就歸檔 → 約每 6 秒一本冊子會持續一本一本長出來，所以才需要定期合併
同一筆被重寫幾次（寫放大）約 10～30 倍compaction 一直搬家，很吃硬碟頻寬——這是寫快的代價
查一個 key 要翻幾本（讀放大）最壞＝全部冊子；有 bloom 多半降到約 1 次磁碟讀bloom 目錄是讀取的救命稻草
💡 關鍵洞察：成本被「延後」了 LSM 並沒有讓總工作量變少，它只是把成本從「寫入當下」延後到「背景的整理」。 就像「先串釘子、打烊後再結算」——尖峰當下變輕鬆，但帳總是要算的，只是換個時間算。

💭 停一下：如果你的應用「寫超多、讀很少」（例如日誌、感測器數據），LSM 和 B-tree 你選哪個？為什麼？

看答案 選 LSM。寫多時，「每筆寫都輕（順序寫）」的好處被放大；讀少時，「讀可能要翻多本」的缺點影響很小。 反過來如果是讀多寫少、又要單筆穩定低延遲（例如交易型），就選 B-tree。

8. 怎麼跟它對話？（API）

🍔 比喻：點餐單 API 就是你跟儲存引擎之間講好的「點餐單格式」：你要它做什麼、它回你什麼。 儲存引擎的點餐單超簡單，就是「存 / 取 / 刪」三件事（這叫 KV 介面，key-value，鑰匙對應值）。 
① 存一筆（新增或覆寫都用它）
   put(key, value)
   做的事：先在流水帳(WAL)抄一行 → 再貼進便利貼(memtable)

② 刪一筆（注意：不是真的擦掉）
   delete(key)
   做的事：貼一張「這個 key 已刪」的墓碑(tombstone)便利貼

③ 取一筆
   get(key) → 值 或「查無此 key」
   做的事：先看便利貼 → 再從新到舊翻冊子(bloom 先過濾)，
           第一個命中就回；若命中的是墓碑 → 回「查無此 key」

為什麼 delete 不真的擦掉？因為冊子排好序又裝訂好、不可修改，回去擦舊冊子＝又變成慢吞吞的隨機寫。所以「刪除」也用「貼一張新便利貼」表達，真正的清除留給背景的 compaction 做。

④ 進階：範圍查詢（因為 key 是排好序的，才做得到）
   scan(起點, 終點) → 依序吐出這段範圍內每個 key 的最新版本

⑤ 維運：手動催它做家務
   flush()   → 手動把便利貼歸檔成冊子
   compact() → 手動觸發合併整理

能做 scan（範圍查詢）是因為每本冊子內部都按 key 排好序——把幾本冊子「邊讀邊歸併」就能依序吐出結果，每個 key 取最新版本。

9. 它把東西記在哪？（資料模型）

📒 比喻：一桌便利貼 + 一本流水帳 + 一櫃冊子 LSM 的資料就放在這幾個地方，名字聽起來嚇人，其實你都認得：

🧾WAL 流水帳本（防當機）

硬碟上一個只往後 append 的檔，每行記一筆：{ 動作:存/刪, key, value }。 便利貼會被風吹走，流水帳不會——當機重開機時照念一遍就把便利貼全復原。

📝memtable 桌上便利貼（記憶體、可改、最新）

記憶體裡一張自動保持排序的暫存表（常用跳表 SkipList 實作），key → 值 或 墓碑。 最新的寫都在這，所以查詢一律先看它。

📚SSTable 冊子（硬碟、排好序、永不改）

每本冊子（檔案）裡有三樣東西：

  • 資料本體：按 key 排好序的 (key, value)。
  • 稀疏索引：「每隔幾個 key 記一個位置」的小目錄。不用每個 key 都記，找的時候先跳到附近、再往下掃一小段就好——省記憶體。
  • bloom 目錄：「本冊大概收了哪些 key」的快速判斷器，常駐記憶體，讓你不用翻就知道「一定沒有」。
為什麼冊子要「不可修改」？ 永不就地改 → 就沒有慢吞吞的隨機寫，而且可以放心快取、多人同時讀也不打架，連 compaction 都只是「讀舊冊子、寫新冊子、再換掉」這麼單純。 代價就是「更新/刪除」都得靠寫新便利貼表達，舊版本留給 compaction 慢慢回收。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車白話症狀怎麼拓寬
compaction 搶硬碟背景一直搬家，跟前台寫入搶硬碟，寫入忽然卡住（write stall）幫 compaction 限速、分到不同執行緒、換更快的硬碟、調整觸發門檻
冊子太多（讀放大）一次 get 要翻好多本更積極地 compaction、用 leveled 策略、調 bloom 參數
bloom 目錄調不好太鬆 → 常常「白翻一本」；太準 → 吃記憶體調「每個 key 用幾個位元」平衡誤報率與記憶體（典型約 10 bits/key、1% 誤報）
value 很大大 value 讓 compaction 搬家成本爆高把 value 另外存（KV 分離，如 WiscKey），冊子裡只放 key + 指標
資料超過一台機器單機塞不下上層按 key 範圍或雜湊分片，每片各自一棵 LSM（如 Cassandra/HBase）
🧹 compaction 兩種掃地法（順帶認識） size-tiered（堆相近大小一起併）：搬家次數少（寫放大低），但冊子重疊多、讀較累、占空間（Cassandra 預設）。 leveled（分層、每層 key 不重疊）：讀很順、省空間，但搬家更勤（寫放大高）（LevelDB/RocksDB）。又是三角取捨。

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心。這裡用最常被面試官追問的 LSM vs B-tree 收束一次：

⚖️ 寫快 vs 讀快 LSM（便利貼派）寫快，但讀可能翻多本冊子；B-tree（翻書派）讀快又穩（翻一次就到），但每改一筆都要就地翻書改頁＝隨機寫，寫較慢。
⚖️ 整理得勤 vs 懶 勤整理（leveled）→ 讀順、省空間，但同一筆一直搬家＝寫放大高、吃資源；懶整理（size-tiered）→ 寫放大低，但冊子一堆、讀慢又占空間。
⚖️ 流水帳要不要每筆都壓進硬碟（fsync） 每筆都壓 → 最安全（當機保證不丟），但每筆變慢；批次壓 → 快，但當機可能丟最後極少數幾筆。看你多怕丟資料。
💭 追問：墓碑(tombstone)什麼時候才能真的丟掉？ 要等 compaction 把「含這個 key 的所有冊子」都合併在一起、確定沒有更舊的副本還藏在別本時，才能把 key 和墓碑一起丟。 太早丟墓碑，會讓某本舊冊子裡「已刪的舊值」復活——所以要小心。
💭 追問：誰在用 LSM、誰在用 B-tree？ LSM：RocksDB、LevelDB、Cassandra、HBase、InfluxDB 等（寫密集、時序、計數、訊息）。 B-tree：MySQL InnoDB、PostgreSQL 等傳統關聯式資料庫的索引（讀密集、交易型、要穩定點查）。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心只有兩個小檔，剛好對應前面講的工作。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 兩個檔 = 兩個小幫手 LsmEngine（整套流程的管家：put／get／flush／compact 都它在跑）＋ Bloom（冊子封面那張「一定沒有就跳過」的快速目錄）。 一筆 put 進來、一次 get 出去，就照下面四段順序跑。

1寫入：先記流水帳，再貼便利貼（put）

來一筆 put，順序很重要：一定先在硬碟流水帳(WAL)抄一行，之後才動記憶體的便利貼。這樣萬一中間斷電，流水帳照念就能復原。

function put(key, 值) {
    寫流水帳(「存」, key, 值)        // ← WAL 先記！抄在「絕不擦掉」的帳本尾巴
    便利貼[key] = 值                // 再貼進記憶體的 memtable
    把便利貼依 key 重新排好序        // 維持有序，之後歸檔才好排
    if (便利貼數量 >= 上限)
        歸檔()                     // 貼滿了 → 觸發 flush（下一段）
}

function delete(key) {
    寫流水帳(「刪」, key, 墓碑)       // 刪除也只是「再貼一張便利貼」
    便利貼[key] = 墓碑              // tombstone：標記已刪，不去舊冊子擦
}

注意「先記流水帳、再改記憶體」這個順序就是 Write-Ahead Log（先寫日誌） 的字面意思。delete 不是真的擦資料，而是貼一張寫著「墓碑」的便利貼——對應前面「便利貼貼桌上、刪除靠貼新便利貼」的比喻。

2歸檔：便利貼貼滿了，整批排序寫成一本冊子（flush）

便利貼一滿就把整桌排好序寫成一本永不再改的冊子(SSTable)，順便替它做一張 bloom 目錄；寫完就清空桌子、截斷流水帳。

function 歸檔() {
    if (便利貼是空的) return;

    目錄 = 新的 Bloom()
    for (每個 key in 便利貼)
        目錄.放入(key)              // 把這本冊子收的 key 都登記進 bloom 目錄

    把便利貼整批寫成新冊子 sstable_000N.json   // 已排序、從此不可修改
    冊子清單.加入({ 檔名, 最小key, 最大key, 目錄 })  // 記進 manifest

    便利貼 = 空              // ← 桌子清空，繼續貼新的
    流水帳 = 空             // ← 資料已安穩落盤，流水帳可截斷
}

對應比喻「便利貼貼滿了 → 找個下午排好序、裝訂成冊歸到書架」。歸檔後便利貼與流水帳都清空，因為這批資料已經安穩躺在硬碟的冊子裡了。每本冊子封面都附一張 bloom 目錄，下一段查詢就靠它。

3查詢：先看便利貼，再由新到舊翻冊子，bloom 說沒有就跳過（get）

這是讀路徑的心臟。順序是：先看桌上便利貼（最新）→ 再從最新冊子往舊翻，每本先問 bloom 目錄，「一定沒有」直接跳過、不碰硬碟；第一個命中的版本就是答案。

function get(key) {
    if (便利貼有 key)                       // 1) 先看最新的便利貼
        回傳 (是墓碑 ? 查無此 key : 值)

    冊子們 = 冊子清單.由新到舊()             // 2) 新 → 舊 逐本看
    for (冊子 in 冊子們) {
        if (!冊子.目錄.可能有(key)) {
            跳過數++; continue;             // ← bloom 說「一定沒有」→ 不開檔！
        }
        實際讀檔數++
        資料 = 讀這本冊子(冊子.檔名)         // 真的開檔讀（一次磁碟 I/O）
        if (資料有 key)
            回傳 (是墓碑 ? 查無此 key : 值)   // ← 第一個命中即答案（由新到舊保證最新）
        // bloom 說可能有、翻開卻沒有 = 偽陽性，繼續找更舊的
    }
    回傳 查無此 key
}

「由新到舊、第一個命中即答案」是為什麼同一個 key 即使散在多本冊子，也一定拿到最新那筆。bloom 那行 continue 就是「看封面目錄、一定沒有就跳過」，幫你省下大量沒必要的開檔。

那張 bloom 目錄怎麼判斷的？白話講：放入一個 key 時，用幾個雜湊函式算出幾個位置，把那幾格位元設成 1；查詢時同樣算出那幾格，只要有一格是 0 → 一定沒有（放入過就不可能還是 0），全都是 1 → 可能有（也可能剛好被別的 key 湊出來＝偽陽性）。

function 放入(key):                          // add
    for (每個 雜湊 in 幾個雜湊)
        位元[ 雜湊(key) ] = 1                 // 把對應幾格設 1

function 可能有(key):                         // mightContain
    for (每個 雜湊 in 幾個雜湊)
        if (位元[ 雜湊(key) ] == 0)
            return false                     // ← 有任一格是 0 → 100% 沒有
    return true                              // 全是 1 → 可能有（含偽陽性）

所以 bloom 絕不會誤報「沒有」（放入時那幾格一定被設 1 了）、只會偶爾誤報「有」。因此「跳過」永遠安全、不會漏資料——這正是前面說「目錄判斷免翻每本」的原理。

💭 停一下：為什麼 get 一定要「由新到舊」翻冊子，不能隨便挑一本先翻？

看答案 因為同一個 key 可能在多本冊子裡留下不同版本（每次覆寫都產生新的）。只有「由新到舊、第一個命中就回」才保證拿到最新那筆；若隨便翻，可能先撞到舊版本，回出過期資料。便利貼（記憶體最新）排第一個看，也是同個道理。

4整理：把多本舊冊子合併成一本、丟掉過時資料（compact）

冊子越堆越多 → 查詢要翻的本數越多（讀變慢）。compaction 趁背景把多本合併成一本：同一 key 只留最新、墓碑與被覆寫的舊值趁機丟掉。

function compact() {
    if (冊子數 < 2) return 「不用整理」;

    合併 = {}
    for (冊子 in 冊子清單.由舊到新())          // ← 由舊到新處理
        for (key, 值 in 冊子)
            合併[key] = 值                    // 後處理的(較新)覆蓋先處理的(較舊)

    丟掉 合併 裡所有「墓碑」                     // ← 壓實後，已刪的 key 徹底消失
    把 合併 排序後寫成「一本」新冊子
    刪掉 剛剛那些舊冊子檔                       // 用一本新的換掉好幾本舊的
    冊子清單 = [ 那本新冊子 ]
}

「由舊到新、後者覆蓋前者」讓最新值勝出；接著一次把墓碑與舊版本清掉。結果：冊子又少又乾淨 → 讀放大、空間放大一起降。對應比喻「幾本舊冊子合併歸成一冊、舊版本趁機丟掉」。

🧭 把程式對回前面的關卡
關卡（前面講的）哪個檔／函式關鍵那段
① 寫＝WAL 先記 + 貼便利貼LsmEngine put()先 寫流水帳 再改 memtable
② 滿了排序歸成冊子LsmEngine flush()整批寫 SSTable + 建 bloom + 清空
③ 查＝由新到舊 + bloom 跳過LsmEngine get() ＋ Bloom新→舊迴圈裡的 可能有() continue
④ 整理＝合併丟舊LsmEngine compact()合併覆蓋 + 丟墓碑 + 換成一本

👉 重點：這個 demo 為了好懂，把冊子用 JSON 檔、便利貼用記憶體陣列、compaction 用「全部併成一本」的簡化版。 真實系統是 RocksDB / LevelDB（Cassandra、HBase 也用同一套），把這些換成 SkipList、二進位 SSTable、分層 compaction——但 WAL 先記、memtable→SSTable flush、bloom 跳過、get 由新到舊、compaction 合併丟舊這幾段邏輯一字都不用改。讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「翻書 vs 便利貼」解釋，為什麼 LSM 寫得比 B-tree 快？ B-tree 要就地翻到對的頁去改＝跳來跳去的隨機寫，慢。LSM 不改舊資料，只往記憶體貼便利貼 + 在流水帳尾巴 append＝順序寫，快。核心就是「把隨機寫變順序寫」。
Q2：bloom filter 是什麼？用「冊子目錄」說說看，它為什麼安全？ bloom 是每本冊子封面的快速目錄。查 key 時先問它：說「沒有」就一定沒有，可直接跳過不翻（省磁碟讀）。它只會「誤報有」、絕不會「誤報沒有」，所以跳過的都是真的安全、不會漏資料。
Q3：刪除一筆資料，LSM 實際上做了什麼？ 不是去舊冊子裡擦掉（那會變慢的隨機寫），而是貼一張「這個 key 已刪」的墓碑(tombstone)便利貼。讀到墓碑就當作查無此 key，真正的清除留給背景 compaction。
Q4：compaction 在做什麼？不做會怎樣？ 把多本舊冊子合併成新的一本，同一 key 只留最新版本、丟掉墓碑與被覆寫的舊值。不做的話冊子越堆越多 → 查詢要翻很多本（讀變慢）、舊版本占空間（空間放大）。
Q5（工程）：WAL（流水帳）的用途是什麼？為什麼「先寫日誌」？ memtable 在記憶體，當機會丟。所以每筆寫先 append 到硬碟上的 WAL，當機重開時照念 WAL 把 memtable 復原。flush 成冊子後該批已安穩落盤，WAL 就能截斷。
Q6（工程）：寫放大、讀放大、空間放大是什麼？它們的關係？ 寫放大＝同一筆被 compaction 重寫的倍數；讀放大＝一次 get 要翻幾本冊子；空間放大＝實際占用/有效資料。三者是翹翹板，不可能同時最佳——compaction 策略就是在這三角上選位置。
Q7（工程）：什麼情況該選 LSM、什麼情況該選 B-tree？ 寫密集（日誌、時序、計數、訊息）→ LSM（順序寫、寫放大低）。讀密集、交易型、要單筆穩定低延遲 → B-tree（一次樹查找、讀快又穩）。面試開場先問「寫多還讀多」定調。

15. 30 秒回顧

LSM-tree = 「先記便利貼、之後再整理歸檔」的存法，靠把隨機寫變順序寫來換取極快的寫入。

4 個關卡：① 寫＝貼便利貼(memtable)+記流水帳(WAL) → ② 滿了排序歸檔成不可改的冊子(SSTable) → ③ 查＝先看 bloom 目錄跳過「一定沒有」的冊子 → ④ 背景 compaction 合併舊冊子、丟過時資料。

3 個怕：怕當機丟資料（WAL 復原）、怕讀太慢（bloom + compaction）、怕重寫太多次/占空間（拿捏整理頻率）。

對比：B-tree 是「翻書派」——讀快又穩，但就地改頁＝隨機寫所以寫較慢。寫多選 LSM，讀多／交易型選 B-tree。三角取捨：寫放大、讀放大、空間放大不可能同時最佳。
