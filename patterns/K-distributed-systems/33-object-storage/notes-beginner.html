1. 60 秒搞懂它在幹嘛

想像你經營一間超大型連鎖倉儲，幫全世界的人「寄放東西」。你不在乎客人放的是照片、影片、還是備份檔——對你來說都只是「一箱不知道裝什麼的貨」。客人交貨時你給一張取件單號，之後憑單號就能把那箱貨原封不動拿回去。

這就是物件儲存（S3 是最有名的那一家）：你把任何大小的檔案丟上雲端，它幫你可靠地存著，之後用一個名字（bucket/key，等於「哪間倉庫／哪個單號」）原封不動取回。

🔑 一句話 物件儲存 = 全世界共用的超大連鎖倉儲：你寄一箱貨（檔案）拿一張單號，它保證「幾乎永遠不會弄丟」，要拿時憑單號還你一模一樣的東西。

2. 為什麼不直接「丟進一台大硬碟」就好？

很多人第一個疑問是：「存檔案而已，買顆大硬碟不就好了？」問題在這個生活經驗——

📦 把所有家當塞同一個房間 如果你把全部重要東西堆在同一個房間，那房間一淹水、一失火，全部一起沒了。 硬碟也一樣：硬碟一定會壞（只是早晚），全押一顆 = 賭命。

所以物件儲存不靠「一顆很猛的硬碟」，而是靠「一大群普通、便宜、會壞的硬碟」，加上一套聰明的擺法，讓就算隨時有硬碟在壞，你的東西也不會不見。S3 號稱可靠到「11 個 9」（99.999999999%）——白話說：你存一千萬個檔案進去，放一萬年都不一定會掉一個。

它怎麼做到的？答案就藏在後面的關卡裡：把東西拆小、分散放、還多存幾份備份。

💭 停一下：為什麼「東西不能改、只能整個重存」反而讓這套系統更好做？

看答案 因為物件儲存規定「檔案一旦存進去就不能就地修改，要改就用同名字重存一份新的」。這叫不可變（immutable）。好處是：既然永遠不會有人「邊存邊改」，系統就能放心大膽地複製、快取、分散——不用煩惱「複製到一半別人又改了」這種同時動手的亂局。把問題簡化掉，海量又便宜才做得到。

3. 把系統拆成 4 個關卡

別被「分散式儲存」嚇到。整件事其實就是闖 4 關，一關一關來：

1收件：超大行李，拆箱寄（分段上傳）

客人要寄一個50 GB 的影片。如果硬要「一口氣傳完」，網路傳到一半斷線——整個重來，氣死人。

🧳 比喻：超大行李拆成好幾箱寄 行李太大塞不進一個箱子，你會拆成好幾箱、分批寄。哪一箱寄丟了，只補寄那一箱就好，不用全部重來。 傳完了再說「我寄完了，請組裝」，倉庫才把幾箱拼回成原本那件行李。

這就是 multipart（分段上傳）：大檔切成小塊、分批甚至同時傳（更快），斷了只續傳那一塊，全部到齊再喊一聲「組裝」。

2放置：這箱貨放哪間倉庫？（一致性雜湊）

連鎖倉儲有幾千間分倉。每一箱貨該放哪間？總不能擲骰子，不然之後要拿時根本不知道去哪找。

🕐 比喻：圓圈時鐘指位置 想像把所有倉庫排成一個圓圈時鐘。每箱貨用名字算出一個「時間點」，順著時針往前走，遇到的第一間倉庫就是它的家。 之後要拿，再算一次同樣的時間點，一定走到同一間——不用問人，自己就找得到。

這套「圓圈時鐘」叫 一致性雜湊（consistent hashing）。它最大的好處：新增或關掉一間倉庫，只有那附近的一小撮貨要搬家，其他統統不動。要是用笨方法（例如「箱號除以倉庫數量取餘數」），一旦倉庫數量改變，幾乎全部都要重搬，天下大亂。

3防丟：多存幾份 or 拼圖備份（容錯）

硬碟（倉庫）一定會壞。怎麼讓「壞了也不會掉東西」？有兩招，是本題最核心的取捨：

📑 招式 A：影印 3 份（副本） 最直覺：同一份東西完整複製 3 份，放在 3 間不同倉庫。掉了一份，直接拿另一份，超快。缺點：佔 3 倍空間、3 倍成本。
🧩 招式 B：同位拼圖（抹除碼 / Erasure Coding） 更省空間的魔術：把資料切成幾塊，另外算出一塊「拼圖校驗片」。神奇之處是——就算掉了其中任一塊，都能用剩下的塊「算」回來。 最簡單版本：校驗片 = 資料片1 ⊕ 資料片2（互斥或），缺哪片就拿另兩片反推回去。這樣只多存約 0.5 倍就能扛掉一片損失（共 1.5 倍），比影印 3 份省很多。

👉 怎麼選？常用、要拿很快的熱門資料 → 用副本（讀最快）；很少動、又大又冷的資料 → 用抹除碼（省錢省空間）。背景還會有巡邏隊，發現某份東西備份不夠了就默默補回——這才是「11 個 9」真正的來源。

4取件：給外人一張「有期限的取件碼」（預簽 URL）

你想把一個檔案分享給朋友下載，但不可能把倉庫的萬能鑰匙（金鑰）交給他——那他不就能亂翻你所有東西？

🎟️ 比喻：超商有期限的取件碼 就像超商寄件給你一組取件碼：只能領「那一件」、而且過期就失效。你不用把店家保險箱鑰匙給人，對方也能順利取貨。

這叫 預簽 URL（presigned URL）：伺服器用自己的祕密簽出一條限定這個檔案、限定時間內有效的連結。對方拿去用，伺服器一驗「簽章對不對、過期了沒」就放行；竄改網址或超時，立刻擋下。

✅ 四關一句話 ① 大檔拆箱寄（multipart）→ ② 圓圈時鐘決定放哪間倉庫（一致性雜湊）→ ③ 多存幾份或拼圖備份防丟（副本／抹除碼）→ ④ 給外人有期限取件碼（預簽 URL）。就這樣。

4. 設計時最怕的 3 件事

工程上的「取捨」聽起來很抽象。換個方式記：這個系統最怕三件事，每個設計都是為了避開某個「怕」。

😱 怕「弄丟東西」 存東西的最高指令就是絕對不能掉。硬碟、倉庫、甚至整個機房都可能壞，所以東西要分散放在好幾個不同地點（不能全押一處），還要多存備份＋巡邏隊持續補。耐久度壓倒一切。
😱 怕「太貴」 要存的是 EB 等級（億 GB）的海量資料。如果每份都影印 3 倍，電費機器費爆炸。所以能用抹除碼（1.5 倍）就別用副本（3 倍），冷門大檔尤其要省——耐久度要高，但錢包也要顧。
😱 怕「找不到 / 拿到舊的」 幾兆個檔案，單號（key）對照表本身就大到一台機器裝不下，要拆開分散管理；而且「我剛存好，馬上去拿」必須拿到最新版，不能拿到還沒寫完或舊的。所以「記東西放哪的帳本」要又快又準。

7. 要準備多大？（規模估算）

🍱 比喻：辦活動前先估人數 辦活動前，你會先估「大概來幾個人」，才決定訂幾個便當、租多大場地。 系統設計也一樣：先估數字，才知道要準備多大的「料」。這一步叫規模估算。

我們估幾個關鍵數字（數字不必背，重點是感受它的「形狀」）：

估什麼大概數字所以呢？
要存多少箱貨10 億用戶 × 每人 1000 個檔 ≈ 1 兆個「帳本」筆數多到爆，一台機器裝不下
總共多少資料量平均每個 ~1 MB → 約 1 EB（億 GB）只能放在「一大群便宜硬碟」上
備份後實際佔多少抹除碼 ≈ 1.4 倍 → ~1.4 EB（若影印 3 份則 3 EB）選哪種備份法，差「兩倍空間 / 成本」
帳本本身多大1 兆筆 × 每筆 ~1 KB ≈ 1 PB連「目錄」都大到必須拆開分散管理
🧠 關鍵洞察：兩個世界，分開蓋 其實有兩種完全不同的東西要管：一是「貨」（真正的位元組，又大又笨重）——丟便宜硬碟、靠分塊＋備份；二是「帳本」（記哪箱貨放哪的小紙條，又小又要查得飛快）——放快速的小型資料庫。 把這兩個分開蓋、各用各的技術，是物件儲存最重要的架構決定。

💭 停一下：為什麼「帳本」要跟「貨」分開存，不放同一個地方？

看答案 因為它們需求剛好相反。「貨」又大又重、寫一次就不太動，要的是便宜＋耐放；「帳本」每筆超小、但要每秒被查超多次、而且要查得快又準。硬塞同一種技術，必然有一邊很慘。分開後，貨放便宜大倉、帳本放高速小型資料庫，各自用最適合的工具、各自擴展。

8. 系統之間怎麼溝通？（API）

🍔 比喻：點餐單 API 就是兩個系統之間講好的「點餐單格式」：你要點什麼（給我哪些欄位）、我回你什麼。 講好格式，雙方就能合作，不用知道對方廚房怎麼運作。

這題只要記住幾張「點餐單」：

① 小檔，一口氣寄（單次上傳）
   PUT /{倉庫}/{單號}   Body: <檔案內容>
   我回：200，附一張「指紋」ETag（核對沒傳壞）

ETag 是內容的「指紋」。你下次拿回來時比對指紋，就知道跟當初存進去的一模一樣、沒壞沒被動過。

② 大檔，拆箱寄（分段上傳 multipart）
   POST /{倉庫}/{單號}?uploads         → 先拿一個「批次號 uploadId」
   PUT  ...?partNumber=N&uploadId=..   → 第 N 箱（可同時傳、可單獨重傳）
   POST ...?uploadId=.. (complete)     → 「我寄完了，請組裝」

關鍵在「能單獨重傳某一箱」：第 7 箱傳到一半斷線，只補第 7 箱，不用整個 50 GB 重來。小撇步：分段上傳組裝後的 ETag 結尾會有 -箱數（例如 -12），看 ETag 尾巴就知道這是不是拆箱寄的。

③ 取件 + 給外人取件碼
   GET  /{倉庫}/{單號}                 → 拿回檔案
   GET  /{倉庫}/{單號}?Expires=..&Signature=..  → 預簽 URL（有期限、免鑰匙）

預簽 URL 就是把「取件碼」直接寫進網址：裡面包含「准你拿這個檔、有效到幾點、以及一段防偽簽章」。改網址或過期，簽章對不上，直接 403 擋下。

9. 系統要記住哪些事？（資料模型）

📒 比喻：兩種帳冊 系統要把資料記在「帳冊」裡。這題重點是分清楚「小紙條（帳本）」和「真正的貨（資料塊）」是兩套東西：

📒帳本：每箱貨的「分裝清單」（小、要查得快）

每個檔案一張清單：單號(bucket/key)、大小、指紋ETag、用哪種備份法(副本/抹除碼)、每一塊放在哪幾間倉庫。

為什麼要記「放在哪幾間倉庫」？ 因為貨被拆成好幾塊、分散各地。沒這張清單，之後根本不知道去哪把塊撿回來拼。所以帳本＝尋寶圖，又小又必須超準。

📦倉庫裡的「塊」：真正的貨（大、不可改）

每間倉庫（儲存節點）裡，存的是一塊塊不可變的位元組：塊編號 → 內容。它可能是「一份副本」或「一塊抹除碼拼圖片」。

為什麼塊「只能新增、不能改」？ 因為不可變，倉庫只要會「放好、找到」就行，永遠不用煩惱「有人邊讀我邊改」。簡單，所以可以塞超便宜的硬碟、堆到海量。
🔑 一個關鍵順序 存東西時，一定先把所有塊＋備份都寫好放穩，最後才在帳本上蓋章「完成」。 這樣就不會發生「帳本說有貨，去倉庫卻找到半箱」的慘劇——帳本指到的，一定是已經放穩的貨。

10. 哪裡會塞車、怎麼拓寬？（擴展）

🚗 比喻：找出塞車路段，拓寬它 系統變大時，總有某個環節會先「塞車」。工程師的工作就是找出最會塞的地方，把它拓寬。一個一個看：
哪裡會塞車怎麼拓寬
太多小檔：每個都要一張帳本＋翻硬碟一次，帳本和硬碟都累垮把很多小檔打包進一個大塊（packing）、帳本多用快取
某間熱門倉庫被擠爆（大家的貨都算到同一間）用雜湊把單號打散（別照字母排）、熱門的自動再切分
一台倉庫掛掉，巡邏隊瘋狂補備份，把網路頻寬吃光幫補備份的速度踩煞車（節流）、備份本來就分散多地降低衝擊
單一連線拉大檔太慢，跑不滿網速大檔本來就分散在多間倉庫 → 同時從多間平行拿
整個機房（區域）出事，全區拿不到背景跨區域再複製一份，一區垮了另一區頂上

11. 還有哪些「魚與熊掌」？（取捨）

「Part 1 的三個怕」是核心取捨。這裡補幾個常被面試官追問的：

⚖️ 副本 vs 抹除碼 副本：讀很快、小檔友善，但佔 3 倍空間。抹除碼：省空間（1.5 倍），但拿的時候要「算」、小檔不划算。一句話收尾：熱資料用副本、冷資料／大檔用抹除碼。
⚖️ 扁平命名 vs 真的資料夾 S3 沒有真的資料夾，單號 a/b/c.jpg 只是看起來像路徑，其實是一整串字。好處是簡單、好水平擴展；代價是「重新命名一個資料夾」其實要搬一大堆檔案，很貴。
⚖️ 帳本要超準 vs 貨可以慢慢同步 「帳本（指到貨在哪）」必須強一致——剛存完馬上要查得到、且是最新。但「貨本身複製到其他區域」可以慢慢來（最終一致）。把「指標」和「資料」的一致性需求分開談，是高分關鍵。
💭 追問：「11 個 9」這麼誇張的耐久度，到底怎麼來的？ 三件事相乘：① 多份冗餘（副本或抹除碼，扛得住掉幾份）＋② 分散在不同故障域（不同機架／機房，不會一次全壞）＋③ 巡邏隊持續修復（一發現備份變少就立刻補回）。三者一起，把「同時壞掉超過容忍數量」的機率壓到極低極低。
💭 追問：怎麼防止「帳本說好了，貨其實只寫一半」？ 靠寫入順序：先把所有塊和備份都寫好放穩，最後才在帳本上原子性地蓋章「完成」。在那之前帳本根本看不到這筆，所以使用者絕不會查到指向半成品的紀錄。

12. 動手玩玩看

讀十遍不如玩一遍。這題有一個真的可以跑的小程式，讓你親眼看到「拆塊存放、製造倉庫故障、再還原」：

# 啟動（在這題的 demo 資料夾裡）
cd demo
php -S localhost:8033 -t public
# 然後瀏覽器打開 http://localhost:8033
  1. 先建一個 bucket（倉庫），再一鍵分段上傳一個檔案。
  2. 看它被切成哪幾塊、各自落在哪些「倉庫」（demo 用資料夾模擬倉庫）。
  3. 故意弄壞一個倉庫（製造節點失效）→ 再下載，看它怎麼用副本或拼圖（抹除碼）把缺的塊算回來，並核對指紋一致。
  4. 產生一條預簽 URL，試試改網址或等過期會不會被擋下。
玩的時候想一下把同一個檔案分別用「副本」和「抹除碼」存，比較佔的空間差多少（3 倍 vs 1.5 倍）——這就是關卡 3 的核心取捨。再弄壞一個倉庫，看抹除碼怎麼用 剩下的塊 ⊕ 校驗片 把缺的那塊變回來——這就是同位拼圖的魔術。

13. 掀開引擎蓋：核心程式怎麼寫的

玩過 demo 了，來掀開引擎蓋看看「那幾關到底是哪幾段程式做的」。 好消息：核心就幾個小檔，剛好對應前面講的四關。不用會 PHP，我把程式翻成白話、只留關鍵那幾行，看註解就懂。

🔧 幾個檔 = 幾個小幫手 組裝員（multipart 把好幾箱拼回原檔、蓋指紋）→ 圓圈時鐘（一致性雜湊決定放哪間倉庫）→ 拼圖師（抹除碼算校驗片、缺一塊算回來）→ 發碼員（預簽 URL 簽出有期限取件碼）。一箱貨進來，就照這個順序跑一遍。

1組裝員：好幾箱拼回原檔、蓋指紋（multipart + ETag）

對應關卡 1（拆箱寄）。每箱（part）上傳時先各自算一枚「指紋」（這箱的 ETag＝md5）；喊「組裝」時，照箱號排好 → 一箱接一箱黏成完整檔，再算出整顆物件的指紋。

function 組裝(這批所有 part) {
    把箱號 由小到大排好          // ← 順序錯就拼成亂碼，這步最關鍵
    完整檔 = ""
    指紋串 = ""
    for (每一箱 依序) {
        完整檔 = 完整檔 + 這箱內容          // 一箱接一箱黏起來
        指紋串 = 指紋串 + 這箱的指紋(二進位)  // 把每箱指紋接起來備用
    }
    // S3 風格的「分段指紋」：把各箱指紋串起來再算一次，尾巴附箱數
    物件ETag = md5(指紋串) + "-" + 箱數     // 例："a1b2...-2"
    return { 完整檔, 物件ETag }
}

看出那個 -2 了嗎？這就是前面說的「ETag 尾巴帶箱數」——光看指紋尾巴就知道這檔是不是拆箱寄的、拆成幾箱。它不是整檔直接算 md5，而是「把每箱的指紋接起來再算一次」，這樣每箱可以各傳各的、最後才湊指紋。

2圓圈時鐘：這箱貨走到哪間倉庫（一致性雜湊）

對應關卡 2（放置）。把每間倉庫（含分身）算出一個「時間點」釘在圓圈上、排好序；一塊貨用名字算出自己的時間點，順時針走，遇到的第一間倉庫就是家。要放幾份就繼續往前撿幾間不同的。

function 選倉庫(這塊的名字, 要幾間) {
    圓圈 = 每間倉庫算出時間點、由小到大排好   // 先把鐘面排好
    我的點 = 雜湊(這塊的名字)
    從圓圈上「第一個 >= 我的點」的位置開始
    一路順時針撿，撿到不重複的倉庫，撿滿「要幾間」為止
    （走到底就繞回開頭 → 所以叫「圓圈」）
    return 撿到的那幾間倉庫
}

關鍵在「順時針撿第一個」這個規則固定不變：所以存的時候跟之後拿的時候，用同一個名字一定走到同一間倉庫，不用另外問人。增減一間倉庫時，也只有那附近一小撮貨的「下一站」會變，其他都不動。

3拼圖師：算校驗片、缺一塊算回來（抹除碼 XOR）

對應關卡 3（防丟）的抹除碼那招，也是本題最魔術的一段。把檔切成兩塊資料片 A、B，多算一塊校驗片 parity：

// 存的時候：算出校驗片
parity = A XOR B          // ⊕ 互斥或：兩片「逐位元相加不進位」

於是倉庫裡放著三塊：A、B、parity，分散在三間不同倉庫。比影印 3 份省很多——只多存半塊（共 1.5 倍）。

魔術在「缺一塊怎麼算回來」。XOR 有個自反特性：掉了 B，就用 A XOR parity 還原：

function 還原(剩下的塊) {
    找出「缺了哪一塊」
    缺的那塊 = 把「剩下的所有塊」全部 XOR 起來
    // 例：parity = A XOR B
    //     掉了 B → A XOR parity = A XOR (A XOR B) = B  ✓ 變回來了！
    return 拼回 A + B（截掉補齊的尾巴）
}

A XOR (A XOR B) 裡兩個 A 互相抵消，剩下的剛好就是 B——這就是同位拼圖的核心。掉的若是 parity？那資料 A、B 本來就還在，重算一次 parity 即可。所以「k=2 + m=1」能容忍任一塊遺失。

💭 停一下：如果同時掉了 A 和 B 兩塊（只剩 parity），還救得回來嗎？

看答案 救不回來。只多算一塊 parity（m=1），就只能扛一塊遺失；一次掉兩塊就超過容忍上限。想扛更多塊同時壞，就要算更多校驗片（真實系統用 Reed-Solomon 做到 k+m 任意配置），代價是多存一點空間、還原時多算一點。原理一樣：多算的校驗片越多，能容忍同時壞的塊就越多。

4發碼員：簽一張有期限的取件碼（預簽 URL HMAC）

對應關卡 4（取件）。要給外人一條「只能拿這個檔、過期就失效」的連結。做法是用伺服器的祕密金鑰，把「動作＋倉庫＋單號＋到期時間」一起簽出一段防偽簽章：

function 發取件碼(倉庫, 單號, 有效幾秒) {
    到期時間 = 現在 + 有效幾秒
    要簽的內容 = "GET" + 倉庫 + 單號 + 到期時間
    簽章 = HMAC(要簽的內容, 伺服器祕密金鑰)   // ← 沒金鑰就偽造不出來
    return 網址 + "?expires=" + 到期時間 + "&sig=" + 簽章
}

對方拿這條網址來取貨時，伺服器用同一把金鑰重算一次簽章，兩關都過才放行：

function 驗碼(倉庫, 單號, 到期時間, 對方帶來的簽章) {
    應有簽章 = HMAC("GET" + 倉庫 + 單號 + 到期時間, 祕密金鑰)
    if (應有簽章 != 對方帶來的簽章) return 擋下;  // 網址被竄改 → 簽章對不上
    if (現在 > 到期時間)            return 擋下;  // 過期了
    return 放行;
}

妙處：到期時間是被一起簽進去的，對方想偷改成「永不過期」？一改，簽章立刻對不上 → 403。所以不用把金鑰交出去，也能發一張限定檔案、限定時間的取件碼。（小細節：比對簽章要用「定時比較」，避免從「對到第幾個字」反推祕密。）

🧭 把程式對回前面的關卡
關卡（前面講的）哪段邏輯關鍵那幾行
① 拆箱寄 + 指紋組裝員排好箱號 → 黏成檔 → md5(指紋串)+"-箱數"
② 圓圈時鐘選倉庫一致性雜湊「第一個 >= 我的點」順時針撿
③ 拼圖備份／還原拼圖師parity=A XOR B；缺塊＝剩下全 XOR
④ 有期限取件碼發碼員HMAC(動作+倉庫+單號+到期) + 驗章

👉 重點：真實系統是 S3 / Ceph / MinIO 那種海量分散式儲存，會把「資料夾模擬倉庫 + JSON 帳本」換成「幾千台儲存節點 + 獨立的 metadata 服務 + Reed-Solomon 抹除碼」， 但上面這幾段邏輯——multipart 組裝＋ETag、一致性雜湊放置、抹除碼 XOR 還原、預簽 URL 簽章——一字都不用改。所以讀懂這個小 demo，你就讀懂了這題的核心。

14. 考考自己

先不要看答案，自己用講給朋友聽的方式回答看看（講得出來才是真的懂）：

Q1：用「連鎖倉儲」的比喻說說看，物件儲存在幹嘛？ 它像幫全世界寄放東西的超大連鎖倉儲：你丟一箱貨（檔案）進去拿一張單號（bucket/key），它保證幾乎永遠不弄丟，要拿時憑單號還你一模一樣的東西。重點不是「存檔案」，而是「用一大群會壞的便宜硬碟，做到幾乎永不掉資料」。
Q2：為什麼大檔要「拆箱寄」（multipart）？好處是什麼？ 一口氣傳大檔，網路一斷就整個重來。拆成好幾塊分批／同時傳，斷了只補那一塊，還能平行加速、組裝後算整體指紋。就像超大行李拆成好幾箱寄。
Q3：副本和抹除碼差在哪？什麼時候用哪個？ 副本＝影印好幾份（如 3 份），讀很快但佔 3 倍空間；抹除碼＝多算一塊「同位拼圖校驗片」，掉一塊能算回來，只佔約 1.5 倍但拿的時候要算。熱門／小檔用副本，冷門／大檔用抹除碼。
Q4：預簽 URL 是什麼？為什麼不直接把鑰匙給人？ 它是「有期限的取件碼」：伺服器簽出一條限定某檔案、限定時間有效的連結。給金鑰等於把保險箱鑰匙交出去，對方能亂翻所有東西；預簽 URL 只准拿那一件、過期失效，安全多了。
Q5（工程）：為什麼「帳本」要跟「貨」分開存？ 需求相反：貨又大又重、寫一次不太動，要便宜耐放；帳本超小但要被查超多次、又快又準。分開後貨放便宜大倉、帳本放高速小型資料庫，各用最適合的工具、各自擴展。這是物件儲存最核心的架構決定。
Q6（工程）：「一致性雜湊（圓圈時鐘）」比「箱號除以倉庫數取餘數」好在哪？ 用取餘數的笨方法，一旦倉庫數量變了，幾乎全部的貨都要重搬，天下大亂。一致性雜湊把倉庫排成圓圈，新增／關掉一間只影響附近一小撮貨要搬家，其他都不動——增減機器的成本極低。
Q7（工程）：怎麼防止「帳本說有貨、倉庫卻只有半箱」？「11 個 9」又怎麼來的？ 防半箱：先把所有塊和備份寫好放穩，最後才在帳本上原子蓋章「完成」，之前查不到。11 個 9：多份冗餘 ＋ 分散不同故障域 ＋ 巡邏隊持續修復，三者相乘把「同時壞超過容忍數」的機率壓到極低。

15. 30 秒回顧

物件儲存 = 全世界共用的超大連鎖倉儲：寄一箱貨拿一張單號，幾乎永遠不弄丟。

4 個關卡：① 大檔拆箱寄（multipart）→ ② 圓圈時鐘選倉庫（一致性雜湊）→ ③ 多存幾份或拼圖備份（副本 3 倍／抹除碼 1.5 倍）→ ④ 給外人有期限取件碼（預簽 URL）。

3 個怕：怕弄丟（分散放＋多備份＋巡邏隊）、怕太貴（能用抹除碼就別影印 3 份）、怕找不到／拿到舊的（帳本要快又準、剛寫完馬上讀得到）。

工程細節一句話：把「貨（海量位元組）」和「帳本（小紙條尋寶圖）」分開蓋 → 用點餐單(API)溝通、看 ETag 指紋核對 → 先寫穩塊與備份再原子蓋章 → 找出小檔／熱點／修復風暴等塞車環節各自拓寬。
