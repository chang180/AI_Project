㉞ 分散式快取 (Redis Cluster)：用 16384 個 slot 把快取攤到整個叢集

K 分散式系統 建議 45 分鐘 Design a Distributed Cache (Redis Cluster)

🌱 第一次接觸這題？ 覺得這份太硬，先看 初學者友善版（看故事就懂），用生活比喻把系統拆成幾個關卡、邊讀邊自我測驗。
一句話定位 本質是「一個容量與吞吐都超過單機的 KV 快取，靠固定 16384 個 hash slot 把 key 攤到多個 shard，每個 shard 主從複製保高可用」。 單機 Redis 很簡單；面試真正考的是 怎麼分片（slot 路由）、怎麼重分片（加減節點只搬受影響的 slot）、怎麼在記憶體滿時淘汰（LRU/LFU），以及怎麼處理穿透／擊穿／雪崩三大快取問題。

1. 需求釐清

快取用途、為何要分散式

  • 用途：擋在 DB 前面的記憶體層，吸收讀流量（讀多寫少）、降延遲（記憶體 ns/µs 級 vs DB ms 級）、削峰。也可做 session、排行榜、限流計數、分散式鎖。
  • 為何分散式：單機 Redis 受限於①單機記憶體（資料量超過幾十 GB 裝不下）②單執行緒吞吐（單核 ~10 萬 QPS 上限）③單點故障。把資料切片到多個 shard，就能水平擴容容量與吞吐，並用主從副本扛故障。

功能性需求

  • SET key value [EX ttl] / GET key，支援 TTL 過期。
  • 資料自動分片到多個節點；client 不需知道 key 在哪台，由路由決定。
  • 加 / 移除節點時可重分片（resharding），且只搬移受影響的 slot。
  • 記憶體達上限（maxmemory）時依淘汰策略（LRU / LFU）逐出。
  • 每個 shard 有主從副本，master 故障時 replica 自動升主（故障轉移）。

非功能性需求

  • 低延遲：p99 < 1ms（命中時）。
  • 高吞吐：百萬級 QPS（靠分片把流量攤到多 shard）。
  • 高可用：單一節點掛掉不影響整體（主從 + 故障轉移）。
  • 可擴展：線上加節點不中斷服務。
  • 一致性可妥協：快取允許短暫不一致（最終一致），優先保可用與低延遲（AP 取向）。
面試開場必問 「資料量與 QPS 多大？」決定要不要分散式。幾 GB / 幾萬 QPS → 單機 Redis + 副本就夠；幾百 GB / 百萬 QPS → 才需要 Redis Cluster 這種分片方案。別一上來就上叢集。

2. 規模估算

指標假設推算
資料量5 億個 key，每個含值平均 400 bytes≈ 200 GB（含 Redis 額外開銷實際更多）
單機裝得下嗎單機可用 ~64 GB裝不下 → 需 ≥ 4 個 shard 分攤
讀 QPS峰值 100 萬讀/秒單 shard ~10 萬 → 需 ~10 個 shard 分流
命中率熱資料集中（長尾）目標 > 95%；命中率每降 1% 回源壓力大增
每 shard slot 數16384 / N 個 shard10 shard → 每個約 1638 個 slot
副本每 master 配 1 replica節點數 = shard 數 × 2
關鍵洞察分片數由「容量」與「吞吐」兩條線取較大者決定。命中率是快取的命脈：算清楚記憶體夠不夠裝熱資料集，裝不下就會頻繁淘汰、命中率崩、回源打垮 DB。

3. API 設計

# 基本讀寫（client 端算 slot 後直連對應 shard）
SET user:42 "{...}" EX 300        # 寫 master，非同步複製到 replica
GET user:42                       # 命中回值；未命中由應用回源 DB 後再 SET 回填

# 叢集管理（運維面）
CLUSTER SLOTS                     # 查 slot → 節點 的映射表
CLUSTER ADDSLOTS / SETSLOT        # 重分片：把某段 slot 指派/搬移到節點
CLUSTER FAILOVER                  # 手動觸發 replica 升主

# 本 demo 對應 HTTP API
POST /api/set      { key, value, ttl }      → 回 slot / shard / 是否淘汰誰
GET  /api/get      ?key=                     → 回 slot / shard / 命中
POST /api/policy   { policy:lru|lfu, maxKeys }
POST /api/node     { action:add|remove|failover, node }
GET  /api/slots    [?key=]                   → slot→節點映射 + 各 shard 概觀
POST /api/scenario { type:penetration|breakdown|avalanche, mitigate, n }
重點Redis Cluster 是 smart client 模式：client 本地持有 slot→節點 映射，自己算 slot 直連目標 shard，不經過中央 proxy。若打錯節點，節點回 MOVED/ASK 重導向並讓 client 更新映射。

4. 資料模型（slot / shard / 主從）

固定 16384 個 hash slot（0 .. 16383）
  slot = CRC16(key) % 16384            # 同 key 必同 slot

slot → shard 映射（每個 master 負責一段連續 slot 區間）
  shardA: slots 0     – 5460
  shardB: slots 5461  – 10921
  shardC: slots 10922 – 16383

每個 shard = 1 master + N replica
  master   : 接受寫，持有該段 slot 的全部 key
  replica  : 非同步複製 master，唯讀；master 掛掉時被選為新 master

key 內 {tag}：hash tag
  user:{42}:profile 與 user:{42}:orders → 只對 "42" 算 slot
  → 落同一 slot/shard，可做同 shard 內的多鍵操作（如 MGET、交易）
為何是 16384？① 節點間用 gossip 交換「我負責哪些 slot」的點陣圖，16384 bits = 2KB，夠小可常傳。② 對一般叢集規模（< 1000 節點）解析度足夠（每節點仍分到十幾個 slot）。③ 是 CRC16 輸出空間（65536）的 1/4，取模均勻。

5. 高階架構

✏️ 用 Excalidraw 開啟編輯（diagram.excalidraw）

 Client ──▶ │ slot = CRC16(key) % 16384     │  (smart client 本地路由)
           │ 查本地 slot→節點 映射          │
           └───────────────┬──────────────┘
            ┌──────────────┼──────────────┐
            ▼              ▼              ▼
        ┌────────┐     ┌────────┐     ┌────────┐
        │ shardA │     │ shardB │     │ shardC │   每 shard：
        │master  │     │master  │     │master  │     master 接寫
        │  │repl │     │  │repl │     │  │repl │     replica 複製/升主
        └──┴─────┘     └──┴─────┘     └──┴─────┘
            └── 命中回值；未命中 → 應用回源 DB → SET 回填 ──▶ DB

對比：一致性雜湊（Cassandra/DynamoDB 風格）= client 端把 key 雜湊到「環」上，
      順時針找節點；加減節點只影響環上相鄰區段。Redis 用「固定 16384 slot」
      取代「連續環」，本質都是把 key 空間切片、只搬受影響片段。
  1. 路由：client 算 slot → 查映射 → 直連 shard 的 master（寫）或 replica（讀，可選）。
  2. 回源：未命中時由應用層走 cache-aside：查 DB → SET 回填。
  3. 故障轉移：master 掛 → 叢集其他 master 投票 → 該 shard 的 replica 升為新 master。

6. 核心元件深入

① 16384 slot 與重分片（resharding）

  • slot = CRC16(key) % 16384 把 key 攤到固定 slot；slot 再指派給 shard。
  • 加節點：從現有節點各勻出一批 slot 給新節點，只搬被勻走的那些 slot 的 key，其餘不動。對比「hash(key) % N」一改 N 幾乎全表要重排，slot 法只動受影響片段。
  • 移除節點：把它負責的 slot 併回其他節點。搬移期間 slot 處於 migrating/importing 狀態，老節點對該 slot 回 ASK 把請求導到新節點。

② 主從複製與故障轉移

  • master 寫入後非同步把指令串流給 replica（預設非同步 → 故障時可能丟最後幾筆寫）。
  • master 失聯 → 叢集多數 master 認定其下線 → 該 shard 的 replica 發起選舉、升為新 master，slot 歸屬轉到它。
  • 快取容忍少量丟失，故多採非同步複製換取低延遲（要強保證可用 WAIT 等待副本確認，但會拉高延遲）。

③ 淘汰策略：LRU vs LFU

  • LRU（Least Recently Used）：淘汰最久沒被存取的 key。適合「近期用過的近期會再用」的時間局部性。缺點：一次掃描（如批次任務遍歷全表）會把熱資料擠掉。
  • LFU（Least Frequently Used）：淘汰使用次數最少的 key。抗「一次性掃描污染」，更貼近真實熱度；Redis 的 LFU 用帶衰減的計數器避免老熱 key 永遠不被淘汰。
  • 其他：volatile-ttl（優先淘汰快到期的）、noeviction（滿了直接拒寫，回錯誤）。實務上 allkeys-lru 或 allkeys-lfu 最常用。

④ 三大快取問題與解

問題成因緩解
穿透 penetration查「根本不存在」的 key（含惡意掃描），快取永遠 miss，每次都打 DB① 空值快取：把「查無」也快取一小段時間 ② 布隆過濾器：一定不存在的 key 直接擋掉不回源
擊穿 breakdown單一熱 key 過期瞬間，大量併發同時回源打 DB（cache stampede）① 互斥鎖 / singleflight：同 key 同時只放一個回源，其餘等結果 ② 熱 key 邏輯不過期 / 後台續期
雪崩 avalanche大量 key 同一時間過期（或整個快取宕機），回源洪峰打垮 DB① TTL 加隨機抖動打散過期時間 ② 多級快取 / 限流降級 ③ 快取高可用避免整體宕機

⑤ Hot key（熱點 key）

  • 單一超熱 key（如爆款商品）會把它所在的單一 shard 打爆（slot 不能再細分到別台）。
  • 解法：本地快取（client 端 / 應用層多級快取，熱 key 不打 Redis）、key 拆分（hotkey#1..#N 散到多 slot 再聚合）、讀走 replica 分流。

7. 擴展與瓶頸

瓶頸症狀對策
熱 slot某 slot 的 key 過熱，單 shard CPU/網卡打滿本地快取擋熱 key、讀分流到 replica、熱 key 拆分散 slot
大 key單一 key 值過大（大 hash/大 list），讀寫慢、複製/搬移卡住、阻塞單執行緒拆小、用 HSCAN 漸進處理、避免一次取整包
重分片搬移加減節點時搬 slot，搬移期間延遲抖動、可能 MOVED/ASK 重導低峰期搬、限速搬移、客戶端正確處理重導向
命中率下滑記憶體不足頻繁淘汰，回源暴增加 shard 擴容、調淘汰策略、縮短/分層 TTL
複製延遲replica 落後，讀 replica 讀到舊值關鍵讀走 master、或 WAIT 強制等待

8. 取捨與面試重點

  • 固定 slot vs 一致性雜湊：slot（Redis）映射明確、重分片可精準控制搬哪些 slot、運維直觀；一致性雜湊（Dynamo/Cassandra）無中央 slot 表、加減節點只動環上相鄰段、更去中心。兩者都解決「取模 N 全表重排」的痛點，差在「離散 slot」vs「連續環 + 虛擬節點」。
  • 強一致 vs 可用：快取幾乎都選可用 + 最終一致（AP）。非同步複製 → master 掛可能丟最後幾筆寫，但換來低延遲與高可用；要強一致就 WAIT 等副本，犧牲延遲。
  • 本地快取 vs 集中快取：本地（程序內，如 Caffeine）最快、零網路，但各機不一致、容量小；集中（Redis）共享一致、容量大，但有網路 RTT。實務常多級：本地擋熱 key，Redis 擋其餘，DB 兜底。
  • cache-aside vs write-through：
    • cache-aside（旁路）：應用負責「讀 miss 回源回填、寫時刪快取」。最常用、實作簡單，但有「寫後讀到舊值」的競態（用刪除而非更新、延遲雙刪緩解）。
    • write-through（寫穿）：寫先寫快取再由快取同步寫 DB，讀永遠命中、一致性好，但寫延遲高、冷資料也佔快取。另有 write-back（先寫快取非同步刷 DB）最快但可能丟資料。
  • 常見追問：MOVED 與 ASK 差別？（前者 slot 已永久搬、client 要更新映射；後者搬移中的臨時重導）。為何 16384 不是 65536？（gossip 點陣圖大小與解析度的取捨）。如何防大 key 阻塞？（拆分 + 漸進式刪除 UNLINK）。

9. PHP Demo 說明

本題 demo 把分散式快取的系統核心邏輯用單機檔案模擬出來，可實際用 PHP 內建伺服器跑：

  • src/SlotMap.php　16384 個 hash slot 與 slot→節點映射；slot = crc32(key) % 16384；加/移除節點時只搬受影響的 slot 區段並回報哪些 slot 換主。支援 {tag} hash tag。
  • src/CacheNode.php　單一 shard（master + replica），SET/GET with TTL，maxmemory 滿時依 LRU / LFU 淘汰並回報淘汰誰，replica 可升主（故障轉移）。
  • src/Cluster.php　把 SlotMap + 多個 CacheNode 組起來路由，並示範穿透／擊穿／雪崩三大問題各一個可觸發場景 + 緩解開關（空值快取、布隆過濾器、singleflight 互斥鎖、TTL 抖動），用「回源 DB 次數」量化緩解效果。
  • public/index.php　路由 + 測試頁：set/get 看 slot 落點、加/移除節點看 slot 搬移、切 LRU/LFU + maxmemory 看淘汰、一鍵觸發三大問題比較緩解前後。

cd demo
php -S localhost:8034 -t public

誠實聲明 Demo 為單機用檔案模擬多個 shard，無真正的網路、叢集 gossip 或自動故障偵測；slot 雜湊用 crc32 % 16384 近似 Redis 真正的 crc16 % 16384（本環境禁用相關擴充，但兩者皆為「同 key 必同 slot、分佈均勻」的雜湊取模，路由邏輯完全相同）。其餘——16384 slot 路由、重分片只搬受影響 slot、主從複製與升主、LRU/LFU 淘汰、穿透/擊穿/雪崩三問題與緩解——皆為真實可執行的邏輯。
