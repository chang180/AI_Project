🎓 系統設計框架小抄

每次練習前看一遍。目標不是背架構，是練出「需求 → 約束 → 有理由的取捨」的反射。

心法零：你被考的是 signal，不是答案 面試官評分勾的是：你能不能從需求推導出約束，再推導出有理由的取捨。 「我選 X 因為這題讀寫比 100:1」比「我會用 Kafka+Redis」值錢十倍。

⏱️ 45 分鐘時間盒

時間階段大師關鍵動作
0–5需求釐清砍範圍；講出假設；指出哪個非功能需求主宰全局
5–8規模估算只算「會改變設計的數字」（QPS、讀寫比、儲存）
8–13API + 資料模型定合約：實體、存取模式
13–25高階架構先畫能跑通的最簡 happy path，別一開始就分散式
25–40⭐ 深入 deep dive挑 2–3 個被主宰需求逼出的難點，深挖 + 取捨（signal 全在這）
40–45收尾主動講瓶頸、故障模式、「更多時間我會…」

🧩 5 個模式（14 題的壓縮）

模式本質題目
A 通知 / Fan-out事件 → 大規模可靠投遞地震 · 價格 · Webhook
B 即時連線長連線 + 狀態同步Messenger · Docs · RoboTaxi
C 一致性 / 交易不能超賣 / 雙花Polymarket · Bookings
D 串流 / 儲存分發讀爆 + 大檔分發QR · Spotify · YouTube
E AI / LLM非同步工作流 + GPU 吞吐ChatGPT · RAG · 推論

🔧 積木 × 故障模式（要能講出「它何時害死你」）

  • 快取 → 失效策略、雪崩、讀到舊值
  • 訊息佇列 → at-least-once ⇒ 必須冪等；失敗 ⇒ DLQ
  • DB 分片 → 跨片交易痛、熱點 key
  • 讀複本 → 主從延遲 ⇒「讀自己的寫」會失敗
  • WebSocket → 連線狀態放哪？哪台 server？⇒ Redis Pub/Sub 轉發
  • CDN / 物件儲存 → 失效、回源風暴、冷啟動

⚖️ 取捨句型（每個決策都套）

「我會選 X 而不是 Y，因為這題 ___需求 主宰； 代價是 ___，但可接受，因為 ___。」

常見取捨軸：一致性 ↔ 可用性 · 延遲 ↔ 成本 · 讀優化 ↔ 寫優化 · 強一致 ↔ 最終一致

💀 5 個致命破綻（自我檢查）

  1. 還沒問需求就開始畫架構
  2. 算了估算卻沒拿來用
  3. 只給唯一正解，講不出替代方案
  4. 深挖簡單部分炫技，略過真正難點
  5. 完全不提故障與規模成長
